### Post by MoLE on 2006-07-27
I'd like to document an issue I've had some difficulty resolving with dapper VMware virtual machines.

After installing various flavours of ubuntu 6.06 LTS on different virtual machines (created using both easyvmx.com and the qemu method), I have noticed the following problem:

1. The virtual machine works well in either NAT or bridged networking mode as long as it is running on the same host OS that the virtual machine was created under.

2.  If the virtual machine is run from a different host OS (I have tested vmware player 1.0.1 running on both WinXP and Ubuntu 6.06), the ubuntu guest OS fails to recognise the virtual ethernet card.  Each virtual machine used is configured with the ```
ethernet0.address = "generated"
``` option, rather than a static MAC address. 

3. If the virtual machine is used under the original host OS again, networking begins functioning again after a reboot without any further intervention.

4.  After much searching, I tried the solution suggested on [The VMware community forums]("http://www.vmware.com/community/thread.jspa?threadID=44925&tstart=0"), which involved editing the /etc/iftab config file, which links a MAC address to a network device.

5.  Solving the problem simply involved commenting out the existing /etc/iftab entry and rebooting.  Thus:

```
# This file assigns persistent names to network interfaces.
# See iftab(5) for syntax.

# eth0 mac 00:0c:2f:55:18:e1 arp 1
```

I'm not sure if this is a reproducible problem by others, however this fix worked for me, and given that I couldn't find it well documented elsewhere, I opted to share the information gleaned here.

### Post by rernst on 2006-11-10
Thanks, your workaround helps.

I think the source of the problem is as follows:

easyvmx (and possibly others) just leave the type="autogenerated" field as-is, but vmware player actually modifies the VMX file to include a generated MAC address when the machine is turned off. 

Somehow this is related to the 'disappearing eth0' problem -- I think it's because when the VM is moved, a new MAC address is generated for it, and the old mac address in /etc/iftab doesn't match up (and hence, the OS can't find the 'new' eth0.

I've had this problem not just when transferring between computers, but after leaving the VM shut down for any length of time on the same host.

Hi,
today I faced the same problem one year later :-) with Ubuntu 8.04 and OpenSUSE 10.3 VM-appliances on ESX 3.5 Server.
There is no more /dev/iftab file which configures the relation ethX <-> MAC but instead there is now a rule file which can be modifíed so that the address match is generic for example the last two goups of the MAC. 
It is called [FONT="Courier New"]/etc/udev/rules.d/70-persistent-net.rules[/FONT]
Change its content to something like this:

```

# This file was automatically generated by the /lib/udev/write_net_rules
# program run by the persistent-net-generator.rules rules file.
#
# You can modify it, as long as you keep each rule on a single line.

# PCI device 0x1022:0x2000 (pcnet32)
SUBSYSTEM=="net", DRIVERS=="?*", ATTR{address}=="00:50:56:85:??:??", NAME="eth0"

```

Important are the question marks as they can be replaced by any number by the file's parser (regular expressions). The sequence 00:50:56:85 is a prefix from VMWare which is added for "automatically" generated MAC addresses. In fact only the last two groups are changed for new virtual devices. If you like, you can place a "*" instead of the whole address...

With this change the VM-Ware Linux Clone-Base can be cloned and immediately started with a new MAC address for eth0 without touching anything else.

Attention! if you add a new network device and then reconfigure the Linux box with Yast, the sript file above is re-generated and the question-marks will disappear.

Enjoy VMWare!
