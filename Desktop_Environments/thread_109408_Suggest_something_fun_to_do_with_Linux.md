### Post by N6546R on 2005-12-28
I've been doing Linux since almost the beginning ('94 or so), and to me the 'real' power of Linux is that you don't even know it's there. I've done a lot of fun things over the years... [http://groups.google.com/group/comp.os.linux.misc/browse_thread/thread/6dfbb52eb3c20090/bf544724c2ee9fe8?lnk=st&q=email+led+box+kitchen&rnum=1#bf544724c2ee9fe8](http://groups.google.com/group/comp.os.linux.misc/browse_thread/thread/6dfbb52eb3c20090/bf544724c2ee9fe8?lnk=st&q=email+led+box+kitchen&rnum=1#bf544724c2ee9fe8) is a project that displays a light for each member of the family when they receive email, for example...lots of web-related stuff, some audio projects, a popular MUD, and so on. 

It isn't really Linux that drives any of this. Certainly all of my projects could have been done on a Mac, or a Wintel box, or an Atari (or a Xerox word processor running CP/M on a Z80, that was a text-to-speech project). I think that computer hobbyists tend to gravitate to Linux because it's got so many knobs to tweak. Self-selection of a sort.

The most fun I've had have been projects that automate some mundane task. For example, I like to do crosswords, so I set up a script that would log in to the NY Times site, pull down a copy of the paper in Postscript (it was free back then), parse the .ps file to find the page with the daily crossword puzzle, then print it so that it'd be waiting for me in the tray when I got home from work. I learned a lot of things doing all that, including teaching myself how to program in Postscript (how many folks even know that it's a language?).

I guess the thing that ties all these together is that I always look at tasks in Linux as an opportunity to learn something. So, that's my advice. If you need to solve a problem, don't just copy-and-paste someone else's solution, instead dig in and find out what the problem -really- is and how it could be solved, and you'll learn something along the way.
