### Post by Natume on 2007-06-16
Similar to the post I made previously about gedit, I'm having a slight problem with nautilus, though nothing to the same degree.  Whenever I go to edit a file name (or create a new one) in the browser window (from the shell, everything works fine), the program often hangs for several seconds on pressing the shift key, or any of the other keys of that sort (alt, super, num lock, etc.).  Unlike gedit, it recovers after a few seconds, but often the input until then isn't registered (and it will often hang up the next time one of those keys is pressed).

This is a little annoying, considering I like to make folder names with capital letters, and occassionally use japanese ; I have uim set to toggle to japanese input with shift + space, which has always worked fine in the past.  In gedit, I solved my problem by switching the input method from xim to uim by right clicking in the program and navigating through the menu, and I have a feeling that it's the same sort of thing in nautilus.  (As an aside, it should be noted that shift and the other keys still work fine in nautilus when I'm not inputting text - ctrl + h still hides / shows hidden files, for example, without any hangup.)

How does one change the default input method for programs like that?  I didn't see anything very useful in the --help option for either gedit or nautilus.  At the very least, I'd like to know how to change nautilus' input method from within the program (though that's only a temporary solution.)

Anything you know or any suggestions would help.

~Natume

(Oh, I just realized that I can change the input method (temporarily only, of course, which is annoying, but helps nevertheless) in nautilus by right clicking inside the field when I'm inputting text.  The question about changing defaults still holds.)
