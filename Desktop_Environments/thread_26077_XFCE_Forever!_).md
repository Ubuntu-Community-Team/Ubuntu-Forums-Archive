 I have done this many times on many distros for myself and others, both with KDE and GNOME (dumping). In Synaptic, be sure to take things slowly and check residual configuration files afterwards (under 'status' in the lower left interface), but not until you are satisfied that you still have a workable environment. This means logging in and out from time to time. Quite a few GNOME applications and utilities are system helpers, like Networking interfaces, Gedit which is handy to have around, and many others. But as far as I am concerned, Nautilus suX, has always, and alwayswill. It is one of the crashiest things that has been inflicted on Linux., although in Ubuntu, as everything else in the distro, it does the best it can do.

Be careful with your Firefox and Mozilla with GNOME support. And whatever you do, don't begin by dumping libraries indiscriminately. As I stated, take your time and go slowly, noting all the associated apps and utilities that will be dumped with each selection you wish to purge. Remember that purging an entry, does not purge all the associated 'uninstalls' that go with it. This is where the status button is useful also.
Whatever you do, don't dump GDM by mistake.

Don't dump metapagages to do a full sweep. One or a few at a time is best.

GNOME is pretty good at sharing libraries. In other words, I have often had pure Xfce installs with a few GNOME apps and associated GNOME libraries.

As far as I am concerned, Xfce is the best. I have been using it since the beginning of the project, many years ago.

### Post by Glanz on 2005-04-16
For those who have been having xscreensaver error messages upon the launching of applications in XFce.

You may be interested to know that I eliminated the spurious generation of 
~/.xsession-errors in my install of xfce4. According to my messages, it 
concerned the xscreensaver. An "xscreensaver" error message was generated at 
every single launch of any application in my system. I traced the prob to /etc/xdg/xfce4/xinitrc (normally around lines 64-65)::
# Launch xscreensaver (if available), but only as non-root user
test $UID -gt 0 -a -z "$VNCSESSION" && xscreensaver -no-splash &
======================================
There is some kind of error in that line..., or elsewhere in the mcs-manager-
plugins script or another script concerning the xscreensaver. So for now, I commented  line out thusly:
# test $UID -gt 0 -a -z "$VNCSESSION" && xscreensaver -no-splash &
and that stopped the generation of error msgs and also the use of the sxscreensaver in xfce, of course.

### Post by rosslaird on 2005-05-09
Happy to oblige.
As far as I know, panel transparency is enabled only in the packaged version of xfce from os-works (the "graphical installer" packages) and not in the Debian version from the Ubuntu sources. And even with the os-works packages, the panel is only partly transparent (the icons and the background are both affected, so the effect is not that great). And it's not configurable. Apparently the next version of xfce will have full panel transparency options, and since xfce has its own composting manager, this won't require xcompmgr or transset (the xorg packages that are so buggy). You can use the gnome panel with xfce if you really want the transparency effect (gnome-panel).

There are quite a few posts over at forum.xfce.org about this issue (and other related things such as using pypanel instead of the default panel). Here's what the [xfce documentation](http://xfce.org/index.php?page=documentation&lang=en#xcomposite)  has to say about the issue:

_________

Xfwm4 includes its own compositing manager, which allows transparency for the panel and shadows for the windows. The compositing manager uses the X.org >= X11R6.8.0 Xcomposite extension, so you first have to be running this version of the graphical server.

The composite extension has to be activated in you /etc/X11/xorg.conf using the following section:

Section "Extensions"
    Option "Composite" "Enable"
EndSection

Xfwm4 has to be built with the --enable-compositor configure option, since it's not enabled by default. If you installed Xfce using binaries provided for your distribution, you'll probably have to rebuild xfwm4.

Please be aware that the X.org Composite extension is still under development, so it includes bugs. Using the compositing manager can slow down your environment if your video card does not support RENDER in hardware. It's known to work well with NVidia cards and NVidia drivers, especially if you add the Option "RenderAccel" "true" option in the Device section of your xorg.conf file. But note that Xinerama and TwinView don't play well with Composite, so, for now, you can either use dual-head or Composite.

___________

Despite what the above says, you should know that adding RenderAccel "true" to your xorg.conf file will cause your x session to be unstable if you are using an nvidia card and the most recent nvidia apps from ubuntu.

Cheers.

Ross
