### Post by ajgreeny on 2023-10-07
I've not heard of this happening to an ssd but it has certainly been seen several times in external flash thumb drives. It is the result of the filesystem on the disk being an ISO-9660, the type used by optical disks, ie, CDs and DVDs.

I would first try creating a new partition table on the disk using gparted -> Device.  This should remove everything currently on the disk and allow you to start again with a totally clear empty disk.
After creating the partition table you will have to create a new partition or partitions of whichever filesystem you need in order to make any use of it. ext4 would be my recommendation unless the disk also has to be read/used by a Windows system.

### Post by sudodus on 2023-10-07
Edit:
You can [check the SMART status](https://askubuntu.com/questions/972978/fsck-reports-that-filesystem-still-has-errors/972983#972983) of the drive. If it is 'passed', you should be able to edit its partition table.
end of edit
[hr][/hr]
You should unmount all partitions on a drive before you edit its partition table.

If you use a tool in Ubuntu, for example **[FONT=Courier New]gparted[/FONT]** to create a clean parition table, the drive will not be called G: but **[FONT=Courier New]/dev/sdx[/FONT]** (where x is the drive letter, for example a or b), or **[FONT=Courier New]/dev/nvme...[/FONT]**).

If it is still difficult to create a clean partition table, you can install and use [mkusb](https://help.ubuntu.com/community/mkusb) and 'wipe the first mibibyte'. After that most partitioning tools should manage to create a partition table.

[hr][/hr]
Are you using Windows? I think that partitioning tools in Windows can also create a fresh partition table on the SSD drive.
