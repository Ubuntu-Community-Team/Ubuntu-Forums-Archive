### Post by dino99 on 2013-02-06
I'm also concerned by that issue time to time; it mainly fail while it create the boot ram disk on some configs.
This is not a new issue, as it has been reported many times since a while; but is very hard to fix as there is no log for helping in such case.
It is either a race condition or an initramfs issue with some chipsets.

All you can do is to test the latest kernel, and/or test with a new install on  fresh formatted partitions (/, swap)

### Post by tgalati4 on 2013-02-09
You might have a power supply problem.  When booting, if the disks don't come up to speed (due to undervoltage) they won't register.  It's also possible that your motherboard is failing.  I had a client's desktop that lost the sound chip.  I put in a $10 sound card.  Two months later the USB ports (including USB mouse and keyboard) stopped working.  Put in a $10 USB card.  Two months later, trouble booting.  Put in a $50 power supply.  Two months later, failed to POST.  Replaced the motherboard.

The PS2 port is powered differently than the USB subsystem, so yes, it's possible to lose USB and still have PS2 working.

Have you cleaned the machine and reseated everything?  Check your power supply voltages with a meter or in BIOS Health Monitoring.

### Post by melon73 on 2013-03-22
I had a Kernel Panic unable to mount VFS problem too. It took me a week to figure it out with a lot of Google not leading anywhere. 

Symptoms: After a kernel update to 3.2.0-39 in Kubuntu I hit Kernel Panic after reboot. I tried 

- to boot using an older kernel
- recovery mode
- Linux Boot Repair
- changing grub config to use /dev/sdx instead of UUID
- installing other kernels using chroot by command line

-- ALL TO NO AVAIL.

My system is a dual boot - I had an older Ubuntu 10.04 on top which I kept just in case the new Kubuntu with quite a few horrific KDE related bugs doesn't come up. The later always booted just fine. When I installed another distro Linux Mint 14 as 3rd OS I managed to get into the older Kubuntu 12.04 for 2 logins then kernel panic again. Also the newly installed Linux Mint 14 suddenly had the same kernel panic. Since I had the old 10.04 on HDD1 and the other 2 on HDD2 I suspected the HDD at first but then I figured that when in 10.04 access to that drive was flawless, SMART showed no errors and FS integrity was OK too.

So I was more than sure that it was a HW error related to the initial RAM disk FS not coming up which would be the RAM then right?

Just as so I removed the second 8GB Corsair Vengeance module from the 16GB config leaving a single RAM module with 8GB in and - voila - everything worked fine. I tested this and it is repeatable.

Now this is to show that a defect RAM module can cause this error just as much as a corrupt kernel or GRUB. Now the difference between the initramfs of 10.04 (=old) and 12.04 (post kernel 3.2.0-39) and 12.10 seems to be the reason that one boots and the other one doesn't. It looks to me that 10.04 was just fine not accessing the upper 8GB on the second module while 12.04 did. 

So if you are unlucky and have a defect RAM module you will see a kernel panic after the second reboot on newer kernels - not being able to boot anything but will get a hang / freeze on olde kernels if you ever access the upper defect RAM of the second module. 

Interestingly, RAM testers show the defect but BIOS RAM tests don't give you any indication that the module isn't OK.

I just thought I share another reason for this omninous problem so one can test if it's not a problem with crappy RAM rather than anythign else.
