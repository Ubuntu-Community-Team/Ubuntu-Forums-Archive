### Post by TheFu on 2022-08-23
If you want to control what and how things are mounted, use autofs or the /etc/fstab to control mounts.  Don't allow the system to determine mount stuff automatically.

If you want help, we need to know how the file systems are connected, what samba has to do with it, and if it is a less than great connection, like USB. 

Usually, the automatic mounts done by Gnome are connected to the user's login and LABEL put on the file system.  The **blkid** command should show the labels for locally connected storage.

### Post by TheFu on 2022-08-24
Loop devices mean nothing for storage.  They don't count and are never useful. They just cruft up output in any storage commands. Best to remove them.

```
/dev/sdd: LABEL="Media-03" UUID="f92bc148-1231-44a2-968c-c6749742c1c7" BLOCK_SIZE="4096" TYPE="ext4"
/dev/sdb: LABEL="Media-01" UUID="f07a99da-ee5d-4742-8f93-daf3f0ea7810" BLOCK_SIZE="4096" TYPE="ext4"
/dev/sde: LABEL="Media-04" UUID="5183fcd9-8643-4367-beab-e84de1e2d9c7" BLOCK_SIZE="4096" TYPE="ext4"
/dev/sdc1: LABEL="Media-02" UUID="3a693d11-36cf-492d-880a-38752a42064a" BLOCK_SIZE="4096" TYPE="ext4" PARTUUID="e8c1ada7-927c-4dda-b503-ee7db83155b1"
```

So, that shows me that 3 drives didn't get partitioned and that can cause issues.  HDDs should always, always, always, get a partition table and at least 1, if not more, partitions.

USB is a terrible way to connect storage. The USB-Storage command set isn't the same as SATA or SCSI commands, plus USB connections are prone to vibration and disconnections.  It will happen, eventually, if it doesn't already happen.  External storage connections really need a cable with screws (SCSI/Infiniband) or at least a clip (eSATA).

Unix systems talk best with NFS, that includes Macs.  It is the native network file system for all Unix systems.

But you have what you have and there is little that can easily be done to fix the less-than-great choices.  My setup has issues too, BTW. As we move forward, we learn and try to avoid making the same mistakes again.

With USB connected storage, using autofs would be my recommendation.  I'd also recommend AGAINST mounting anything under /media/, which is for DVDs and temporary mounted storage needs, not permanent storage.  Having 2 different subsystems trying to control the same directory for mounts is asking for failures.

[https://help.ubuntu.com/community/Autofs](https://help.ubuntu.com/community/Autofs) is the how-to for autofs.  I place storage under /d/ - like this, using autofs:
```
Filesystem                        Type  Size  Used Avail Use% Mounted on
istar:/d/D1                       nfs4  3.5T  3.5T   23G 100% /d/D1
istar:/d/D2                       nfs4  3.6T  3.6T  9.1G 100% /d/D2
istar:/d/D3                       nfs4  3.6T  3.6T   13G 100% /d/D3
```
That's the client view.  

On the server, there is more than double that storage and it is mounted under /d/ as well.
```
Filesystem                                  Type  Size  Used Avail Use% Mounted on
/dev/mapper/istar--vg-lv_media              ext4  3.5T  3.5T   23G 100% /d/D1
/dev/mapper/istar--vg2-lv_media2            ext4  3.6T  3.6T  9.1G 100% /d/D2
/dev/mapper/istar--vg3--back-lv_media3_back ext4  3.6T  3.6T   13G 100% /d/D3
```

The autofs config file for this storage looks like:
```
 /d/D1   -nodev,nosuid,timeout=2,fstype=ext4 /dev/istar-vg/lv_media            
 /d/D2   -nodev,nosuid,timeout=2,fstype=ext4 /dev/istar-vg2/lv_media2          
 /d/D3   -nodev,nosuid,timeout=2,fstype=ext4 /dev/istar-vg3-back/lv_media3_back
```
 
NFS makes mounts on Unix clients trivial and since it is system-to-system, not userid-to-system, the mounts happen at boot, not because some user logs in.  For media servers, I really don't want to have to login to gain access to storage.  I really don't use NFS to access media files, that's what DLNA and Jellyfin is for.  Those are better at serving media to all sorts of clients than NFS and 3x better than CIFS.

I've also been burned by having 2 different subsystems think they control storage mounts and sharing in the same directory.  I've never been impressed by the Gnome storage used for end-user mounting.  That stuff is junk.  That goes for Gnome Disks too, BTW.  Just because something is the default, doesn't mean we should use it.  

Shorter paths means less typing and easier scripting.  What really brought that idea to me was some abusive software - Plex Media Server.  The highest directory for that is in:
 /var/lib/plexmediaserver/Library/Application Support/Plex Media Server
I'm not shitting.  71 freakin' characters!  How is that better than:
 /var/lib/plexms
Hint, it isn't.  It is stupid to waste so many characters.  There is a limit for the number of characters allowed in a path. In an old job, I had to modify those limits for every system in the kernel on the air-gapped network.  There were over 2000 systems involved.  The plex guys (the company) would have been rejected over wasting so many characters in that environment.  I won't waste characters, even if they seem free.

BTW, I also don't allow spaces in any directories or filenames, if it can be avoided.  Spaces in filespecs break scripts. It is easier to remove/replace them than to make scripts handle spaces. This  is because a space is a natural word separator and when passing in a glob of filenames to a script, perhaps to handle hundreds of files for processing, spaces break the globbing. Sure, there are ways around it, but they are just more hassle.  A script that works on files without spaces is trivial to create. Add in spaces and all sorts of extra stuff is needed. It is a pain.

> **roguegeek said:**
> That was my understanding of RAID1 as well, so I'm actually doing nightly rsync for all of my media drives as well as off-site backup.
  For really important media, you may want to add par2 files, say 10%, to catch and be able to recover from small bitrot.  Or use ZFS, which proactively validates the bits sent to be written actually got there.  I use par2 on a very small subset of the media files.

> **roguegeek said:**
> From a high level, how does autofs know when to mount and unmount? Is it simply when that volume is pinged that it mounts? Also, do you happen to have a good resource for fstab as well? I'd love to see a quick sample line of both autofs and fstab so I can cross-ref it against some sort of resource.
Whenever a file object is requested [fopen() or open()] and it isn't currently mounted, it will mount it. As long as the file object is open, it will remain mounted.  When the last file object is closed, there's a time out period - I think 2 minutes or so, then it umounts.  The supported mount options for autofs and fstab are the same.  autofs grew out of the amd/automountd daemons which have been used for decades on Unix.  There is no performance difference once the storage is mounted.
