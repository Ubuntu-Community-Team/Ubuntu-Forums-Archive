### Post by srs5694 on 2011-04-30
I'm not positive, since I'm not 100% sure that I understand how the Boot Info Script reports the data, but it looks to me as if your partition type codes may have gotten messed up. To perform further diagnostics, please follow these steps:


[list=1]
[*]Boot Ubuntu.
[*]Open a Terminal window.
[*]Type "sudo apt-get install gdisk" to install the gdisk package.
[*]Type "sudo gdisk -l /dev/sda". This will display the partition table using gdisk.
[*]Post the results.
[/list]


It's not clear to me what partition 1 should be, since the Boot Info Script didn't identify the filesystem type from its partition scan, but partitions 2 and 3 are both NTFS; I suspect that 2 should be type 2700 and 3 should be 0700 (under gdisk's "Code" column). If I'm right, they're currently set to something else -- perhaps FFFF or some other value. If that's what you see, you can try launching gdisk without the "-l" option, using the "t" option to change the type codes, and then using "w" to save the changes.

As to partition #1, it's too small to be anything useful except for a BIOS Boot Partition, so perhaps that's what it was originally, but you've got another BIOS Boot Partition, so it *might* be safe to delete the partition. OTOH, it's probably safer to leave it in place, in case it's important to Windows.

There's a bigger question here, though: Your partitions seem to be set up for booting with a BIOS rather than with a UEFI -- you've got a BIOS Boot Partition, as required by BIOS, but no EFI System Partition (ESP), as required by UEFI. Windows won't boot from a GPT disk on a BIOS-based computer, though. It's possible you used to have a [hybrid MBR,](http://www.rodsbooks.com/gdisk/hybrid.html) which would enable Windows to boot from such a disk. If so, you'll need to either restore a new hybrid MBR (which gdisk can do; you'd use the "h" option on the recovery & transformation menu, as described on the hybrid MBR page I just referenced) or convert the disk from GPT to MBR form (which you can do with gdisk using the "g" option on the recovery & transformation menu). The latter option is riskier in the short run but safer in the long run because hybrid MBRs are so flaky and dangerous. Unfortunately, your partitions have no gaps between them, which means you can't convert all of them to MBR form. You'd need to delete /dev/sda4 (which you'd no longer need) and /dev/sda6 (which you could later re-create) to do the conversion, and you might as well axe /dev/sda1 while you're at it. If you convert from GPT to MBR, be aware that you'll need to re-install GRUB.

FWIW, hybrid MBRs are most commonly found on Macintoshes and on PCs that have had Mac OS X installed on them ("Hackintoshes"). If your computer fits one of these descriptions (or did, in the latter case, since there's no evidence of a Hackintosh install there now), then that could explain a lot. If not, then I'm a bit mystified about how it got into the state that it's in.

### Post by srs5694 on 2011-04-30
Unfortunately, your story leaves several extremely critical details unspecified, the most important being what advice you followed from the Ubuntu forums thread you mention. (Please note my signature. It applies to *any* person's advice!)

My *suspicion* about what happened is this:


[list=1]
[*]You started with a conventional 4-partition MBR setup in Windows.
[*]In fiddling with your partitions in Windows, those got converted into a Windows [Logical Disk Manager (LDM; aka "dynamic disk")](http://en.wikipedia.org/wiki/Logical_Disk_Manager) setup. This is a common pitfall when new users try to prepare a 4-partition disk for a dual-boot with Linux. The key evidence here is the fact that your first three partitions are identified as LDM partitions by gdisk.
[*]The Ubuntu forums post that you followed suggested converting from MBR to GPT using gdisk; or maybe it was intended for somebody who already had a GPT disk, in which case your following the advice with an MBR disk would have automatically converted from MBR to GPT. I suspect this because several of your partitions have Name fields that are what gdisk assigns automatically when creating or converting partitions.
[/list]


In any event, your Windows configuration is pretty well trashed now; I know of no certain way to recover it. Therefore, I suggest you remove any critical data from the Windows installation by using Linux. (Try mounting /dev/sda2 and /dev/sda3 manually, if they don't mount automatically, and copy your data that way.)

Given that the Boot Info Script identified both /dev/sda2 and /dev/sda3 as holding NTFS data, there's a slim chance you'll be able to restore Windows to bootability by following this procedure:


[list=1]
[*]Launch gdisk ("sudo gdisk /dev/sda").
[*]Type "t" to change a type code. Specify partition 2 and give it a type code of 2700.
[*]Type "t" to change a type code. Specify partition 3 and give it a type code of 0700.
[*]Type "r" to enter the recovery & transformation menu.
[*]Type "h" to create a hybrid MBR. Specify partitions 2 and 3 to be included. Tell it to put the 0xEE partition first, use the default MBR type codes, set the boot flag on partition #3, and do not create an extra protective partition.
[*]Type "w" to save your changes and exit gdisk.
[*]Reboot and see if Windows boots.
[*]If Windows does *not* boot, use a Windows recovery disc to restore Windows to bootability.
[/list]


I don't hold out much hope that this will work. If it does, you might consider a full conversion from GPT to MBR; but if you try that there are other pitfalls.

If you want to start over, you should first back up all your important user data. I recommend then doing "sudo sgdisk --zap-all /dev/sda" to completely wipe all partitioning data from the disk. If you don't do this, you're likely to end up with other problems because the Windows installer won't completely remove the GPT data, which will then confuse most Linux partitioning tools. Starting with wiped GPT and MBR data, Windows will install using a conventional MBR (assuming it's a BIOS-based computer). If necessary, use Windows' partitioning tools to shrink the Windows partition, but under no circumstances should you use it to create additional partitions; Windows sometimes creates the LDM configuration in that case, and this leads to trouble, as you've discovered. Don't use gdisk or sgdisk, since these tools automatically convert MBR to GPT, which will cause problems for Windows. An MBR-only configuration is the conventional and safest configuration when dual-booting Windows and Linux on a single disk on a BIOS-based computer.

### Post by srs5694 on 2011-05-02
Your partitions are seriously messed up, which is causing problems. In particular, it looks like you've got old GPT data interfering with new MBR data, which is confusing the Linux partitioning tools. You didn't do as I specified in post #7:

[quote=srs5694]If you want to start over, you should first back up all your important  user data. I recommend then doing "sudo sgdisk --zap-all /dev/sda" to  completely wipe all partitioning data from the disk.[/quote]

My suspicion is that your MBR data are now accurate, but I'm not positive of that. The safest approach is to follow my original advice, quoted above, and start over again. It's imperative that you clear all the MBR and GPT data, and the simplest way to do that is with sgdisk, as I described.

As to choosing GPT or MBR for moving forward, the decision depends on the firmware on the motherboard, whatever's supported by the installation media, and how close to the "bleeding edge" you're willing to live. A Google suggests that the HP DV6 may support either BIOS or UEFI modes, but I'm not positive of that. (Windows requires GPT when booting in UEFI mode and MBR when using BIOS mode, so the firmware type and partition table type are linked if the computer boots Windows.) If I'm right, you should be able to select either mode in the firmware setup utility, but I can't provide explicit instructions on this; you'll have to consult the computer's documentation. If you're lucky, you'll then be able to restore Windows using your restore discs, but it's conceivable that they only support one mode or the other. You might want to call HP's tech support on this score, or at least see if you can find some explicit documentation in a manual or online. Since I don't own an HP DV6, I can't offer more than speculation and generic advice.

On the Ubuntu side, unfortunately UEFI support is still pretty new, and there are two or three current threads here about ongoing problems installing Ubuntu in UEFI mode. I've done one UEFI test install of the 11.04 alpha 2 and I had no problems; but that was just a test install with no dual-booting involved. OTOH, if your Windows recovery disk is hogging all four primary partitions, as it seems to be doing, you'll have to delete one or convert one to a logical partition (using [FixParts](http://www.rodsbooks.com/fixparts/)) before you'll be able to install Linux, and that particular hassle wouldn't exist if you were using GPT. Overall, my preliminary conclusion is that Ubuntu is probably better off in BIOS  mode at the moment, especially for non-experts doing dual-boot  configurations.

### Post by srs5694 on 2011-05-03
FWIW, GPT is superior to MBR, and we'll all be forced to use GPT before  too long because of MBR's disk-size limits (2^32 sectors, which works  out to 2 TiB, given 512-byte sectors). The problem at the moment is with  Ubuntu's GRUB issues with (U)EFI. The GRUB setup tools just tend to  produce flaky configuration files, and Ubuntu's installer makes it easy to create partition tables or mount points that just don't work. See [this thread](http://ubuntuforums.org/showthread.php?p=10762958) for a discussion of some of the GRUB issues, including a resolution (albeit one that requires constant vigilance, since sooner or later the automatic tools will try to overwrite the working configuration).

Ironically, the GPT/BIOS combination works better than the GPT/UEFI combination under Linux. Unfortunately, GPT/BIOS is an unworkable combination under Windows, at least for a boot disk; and of course the latest batch of motherboards is pushing UEFI pretty heavily.

I expect that Ubuntu's UEFI problems will be at least mostly fixed by 11.10, or 12.04 at the very latest. The developers are likely to be flooded with bug reports from people using the new boards -- or start using the new boards themselves!
