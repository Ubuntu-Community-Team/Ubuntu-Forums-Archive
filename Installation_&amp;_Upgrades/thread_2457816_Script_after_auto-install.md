### Post by TheFu on 2021-02-10
Use ansible.  Stiff like this is exactly why ansible exists.  I have an ansible setup that pushes my ssh keys, bash aliases, and does 50 other things on new systems like purging network-manager, nano, and some Canonical-added packages that I find unhelpful.
[https://www.ansible.com/resources/get-started](https://www.ansible.com/resources/get-started)

Don't get too bogged down. Ansible is 100% F/LOSS and you don't need to subscribe or pay anything.  The only need is to setup ssh from another system into the machine(s) you want to configure.  Ansible supports ad hoc commands as well as roles/tasks for stuff we do over and over.

For example, after my weekly patching for all the systems here (which can be done using ansible), I want a list of systems that need to be rebooted.  Ubuntu creates a file, FILE="/var/run/reboot-required", which tells me - this system needs to be rebooted if that file exists at all.  Honestly, I have a bash script that loops over all the servernames here, D, and runs this command:
```
  ssh $D "ls -l $FILE 2> /dev/null"
...
======
Working on romulus
-rw-r--r-- 1 root root 0 Jan 30 11:11 /var/run/reboot-required
...

```
Fairly simple coding. ssh is setup with key-based authentication, but ansible has a way to pass a username/password too.  I've never used that.
Been meaning to reboot romulus for a few weeks. It has a GPU problem, so I have to be in the same room and completely shutdown the system or it won't boot.

With ansible, 
```
~/.ansible/mydomain.com/tasks$ more reboot-required.yml 
---
- name: Check if Reboot is REQUIRED
  command: /bin/cat /var/run/reboot-required
  register: cat
- debug: var=cat.stdout_lines

```
I think that's the 'task" file - there's a problem with it. Ansible version upgrades sometimes break old scripts. Need to add it into a playbook and do a little troubleshooting.  

Another way would be to use the setup module:
```
$ ansible -a "cat /var/run/reboot-required" current
regulus | FAILED | rc=1 >>
cat: /var/run/reboot-required: No such file or directorynon-zero return code
hadar | FAILED | rc=1 >>
cat: /var/run/reboot-required: No such file or directorynon-zero return code
**romulus | CHANGED | rc=0 >>**


```
That works, but it makes ugly output for systems without the file, which is exatly the opposite of my desire.  I want systems WITH the file to complain loudly and those without the file to have a minimal output.  See the **rc=0**?  That means the file existed, but is empty.

Of course, you can create a bash script for this stuff too.  If you are building virtual machines, you can insert modified config files from the hostOS after the install, pre-boot, too.  Vagrant (I think that's the name) does that. 

There are lots of other tools for these things.  Cobbler if you want to shovel an image onto hardware, puppet, chef, salt-stack, rexify, for devops, and for 40 yrs Unix admins have created home-built tools for these same things based on rsh/rlogin/ssh.  Many of those old home-grown scripts have been replaced by the devops tools, which means there is a common language shared by millions of admins world-wide. A new admin doesn't have to learn some odd scripting created by someone 30 yrs ago. They just need to to spend 2-5 days learning, say, ansible, to be useful in the new environment.  Or if they already know Ansible, they will see things that are a little different, perhaps better in the new location and some things that aren't as good as their old job did it.  That cross pollination is good.

Now you've done it.  I have to fix that non-working Ansible task above now. Thanks. ;)

### Post by TheFu on 2021-02-11
> **hamstermandk said:**
>  What I was hoping for was something much more simple like "just add your bash script here and it works after the installation".
As long as a Tools can be implemented in the custom ISO directly it would be fine. Let's say you install the tool during the installation and run it with a config file. That would be great as well.

There are a number of ways to handle this.  If they are internet connected, then each system could have a script to be run at reboot (via cron) to pull specific files from known, trusted, locations, then run those. Inside the "pull" script, check if the last script existed and had been run already. Basically, it would be a way for self-updating, remote control, over the systems.  Lots of tools have something like this, but it is so easy to have a little DB that keeps track of patches and patch scripts, then marks each as completed when done. Just be certain it is 100% save to run over and over and over and over - does no harm.   BTW, that could be handed by ansible, but I'd just have a tiny perl/python/ruby/bash script handle it.  

Also, wouldn't hurt to have the script require positive user interaction for updates with a timeout that doesn't do anything. We hate when people (cough MSFT, Google, Canonical, ... ) force updates without any way to prevent/delay those updates. Wouldn't be hard to include a "don't ask again for 7 days" to the interactive script.

Or it could be as easy as a @reboot in the crontab that checks if a file exists and if it does, run it. The last thing that file does is either delete itself or make itself empty or just rename the file. Just another option.

Or it could be setup as part of the deployment userid login. At the bottom of the .bashrc, run another script if the file exists ... 

Lots of ways.
