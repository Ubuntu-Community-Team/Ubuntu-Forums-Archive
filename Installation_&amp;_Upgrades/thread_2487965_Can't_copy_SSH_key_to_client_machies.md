### Post by ActionParsnip on 2023-06-20
Juse rsync or scp to copy the file. Lots easier
```

scp $HOME/.ssh/publickeyfile remoteserver:/tmp

```

Obviously, change publickeyfile to your actual file and remoteserver to the actual servername. You can then SSH to remoteserver using our password, and run
```

mkdir $HOME/.ssh
chmod 700 $HOME/.ssh
touch $HOME/.ssh/authorized_keys
chmod 600 $HOME/.ssh/authorized_keys
cat /tmp/publickeyfile >> $HOME/.ssh/authorized_keys

```

### Post by TheFu on 2023-06-20
a) ssh keys are created on the client, not the server.
b) If you don't do anything funky first, you need just 3 commands.

Client:```

   $ ssh-keygen -t ed25519  # create a key using elliptical keys
   $ ssh-copy-id -i ~/.ssh/id_ed25519.pub username@remote  #   Push the public key from the client to the server
   $ ssh username@remote   # Test that the client can ssh into the server.
```
That's it.  No need to make it harder than this.
[Https://ubuntuforums.org/showthread.php?t=2433034&p=13916386#post13916386](Https://ubuntuforums.org/showthread.php?t=2433034&p=13916386#post13916386) has more details, I think.

There's lots of other things that can be done to make life easier, like setting up a ~/.ssh/config file so you don't know about non-standard ports or different usernames on different remote systems.

Obviously, on both the client and the server, you should install ssh + fail2ban packages.  Additionally, if you want more security, then you'll likely want to modify the sshd_config file on the server to limit which users have remote access and to force key-based authentication whenever going over the internet.

And if you've enabled ufw or set any iptables/nftables rules already, you'll need to open those from the subnet where your client sites. No need to allow everyone on the internet access.[https://ubuntuforums.org/showthread.php?t=2446413&p=13969099#post13969099](https://ubuntuforums.org/showthread.php?t=2446413&p=13969099#post13969099)

You can always start over by running **rm -rf ~/.ssh** on both the client and the server.

ssh-copy-id will create the ~/.ssh on the remote system and set all the permissions correctly.  Just don't screw with it. It also knows how to append keys to files correctly, so if you have multiple clients, the file on the server won't be incorrectly molested.
