### Post by The Keeper on 2006-09-10
I agree, I too use 32bit Ubuntu even though I have two AMD64's. The thing is; one of the major strenghts of linux are the repositories. If I go and install 32-bit apps manually, repositories become useless when these apps are concerned. I could be just as well be using Windows.

I'm tired of the fact that 64-bit CPU owners are treated like second class citizens. Soon even the low-end CPU's from AMD and Intel are exclusively dual-core and x86_64, single-core and x86-only CPU's are a dying breed.

You would expect that with increasing popularity of x86_64 CPU's, all applications would be recompiled for x86_64. This is not the case, most applications won't benefit from this recompilation and since x86_64 CPU's can run 32-bit applications just fine, there's no need to recompile. If distro maintainers refuse to support 32-bit applications in 64-bit distros and force users to manually install such applications, they are shooting their own leg.

OpenOffice.org in the 64-bit repositories is not a 64-bit app, but a 32-bit. All us, the regular users want is availability of 32-bit apps where their 64-bit counterparts have compatibility problems, such as Firefox. Even 64-bit Windows comes with both 32-bit and 64-bit versions of IE.

The likelyhood of AMD/Intel dropping x86 support in the next couple of decades is approximately 0.  Current processors support **all** of the x86 (aka: IA32) instruction set.  That includes all of the old real-mode 16bit stuff.

The history of the the 80x86, now called IA32, processors has been to gradually introduce features, never, **ever** remove them.  Even when it would make the processor much simpler.  Current x86-64 processors can still run code written for the 80086, the first, 16bit, popular Intel desktop processor.

They've managed to get pretty good at supporting IA32 while still being able to design fast processors.  Now, there's pretty much just an IA32 "emulation" layer in front of the backend stuff, which pretty much uses it's own instruction set.  Note that this is the case as far back as the first Pentium processor.

Do you know why the x86-64 instruction set is not known as IA64?  Because there's already an IA64 instruction set.  The processors supporting that don't have (good, native, fast) backward compatibility with IA32 code.  Practical upshot?  Almost nobody uses them.  Intel (and HP, IIRC) dumped billions of dollars of R&D effort into those processors, and they pretty much flopped.  AMD released an **extension** to the IA32 instruction set, adding 64bit support and some other goodies with processors which **also** executed IA32 code faster than the competition.  Result: All new x86 desktop processors support x86-64.

IA32 support is not going to go away.  Ever*.  You're completely safe installing the i386 version on your AMD64 processor.

* - Where "ever" means in the forseeable future.

### Post by RAOF on 2006-11-23
> **Kilz said:**

There never was 16bit linux.  The difference between 16bit real mode and 32bit protected mode on the processor is absolutely tremendous.  If you'd like one of those automotive analogies that seem popular, 16bit real mode is like a bicycle, 32bit protected mode is like a modern automatic car.  Although this analogy really doesn't emphasise how much better protected mode is over real mode.  Perhaps a more accurate analogy would be "16bit real mode is like a bicycle assembly kit".  Linux is **not possible** in real mode.

The difference between 32bit protected mode and 64bit mode, whatever that's called, is much, **much** less.  It's the difference between a modern automatic car and a modern automatic car with a tuned engine.  Having a larger word size does not magically make a computer better.  The only reason that 64bit mode is not **slower** than protected mode on normal tasks is that the the x86-64 instruction set fixed some of the glaring problems of the IA32 instruction set (in particular, it added more registers).  64bit mode uses more memory, and hence makes memory access on average slower.

And anyway, what I was responding to was "What if 32bit support was dropped from the processors".  It's **not going to happen**.

Finally, @tszanon:  I don't think you have a good idea of how development works.  The only difference in support for Ubuntu over the various archietctures is **closed source software**.  Oh, and wine, but you're only ever going to be running closed source software in wine anyway, so the point stands :p.  (Well written) Open source software runs on **everything**.  If a program runs on IA32 but not x86-64, that's a bug.  And will probably be an easy bug to fix, too.  There is no lack of 64bit development, because the (vast majority of)development isn't "32bit" or "64bit", or even "AMD/Intel".  The vast majority of development benefits everyone, IA32, x86-64, PPC, whatever.

What Kilz is asking for is a solution to run 32bit stuff on the 64bit platform, or at least make it easier.

The other difference in support is that the x86-64 forum has fewer regulars.  But that 'aint something the devs can change :).
