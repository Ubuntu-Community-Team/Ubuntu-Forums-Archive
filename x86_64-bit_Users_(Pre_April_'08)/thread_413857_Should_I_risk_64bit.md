### Post by x64Jimbo on 2007-04-19
I have been running 64 bit versions of Kubuntu on my desktop for about a year now, and I'm about to upgrade to the Feisty release. However, my reasons are different than most people's. I use my desktop as a workstation, and it needs to be as powerful and fast as possible. This is especially because I use it to back up all my DVDs to XviD format for playing them later either on the computer itself or over the network on the TV with a laptop running the video in through s-video. 
64 bit makes a noticeable difference in processor-intensive applications, so if you plan to use your machine for a lot of ballsy math stuff, go with 64 bit. Otherwise, for browsing, IM, watching youtube clips, and email (as most people use their computers) 64 bit is not necessary yet. Save yourself the (albeit small) hassle of having to tweak your system for compliance with 32 bit standards unless you really want/need the fastest OS you can get.

### Post by x64Jimbo on 2007-04-19
Let me be clear: 64 bit is only a minor inconvenience, and if you are feeling adventurous enough to try Ubuntu in the first place, you're probably adventurous enough to use 64 bit. However, if you get frustrated easily, you should probably stick with 32 bit until most of the apps out there transition over to 64 bit, then you won't have to worry about the tweaks/hacks. I use 64 bit daily and love it - the scripts out there that handle most of the troublesome items make it almost no trouble at all... Just know that 64 bit systems are more on the cutting edge, and you MAY have a few more snags getting things working. But if you enjoy a challenge, do it!

### Post by NickK1066 on 2007-05-03
> **Kilz said:**
> Compiling applications is something I have been doing recently. Im still not that good at it, but I'm learning. Can you please explain how I would use the gcc-autovect line. So far all I know is ./configure, make, and make install. I had no idea there were things that can be done to make the application run faster.

Firstly - this can be risky if you don't know what you're doing. Attempt at your own peril and you won't find any easy package management. Now with the disclaimer sorted.. and you feel the need to enter the bleeding edge - we'll continue.
Just check with the GCC dev mailing list to ensure someone hasn't borked the build or a critical bug has appeared in the test cases for the optimisation. Did I say we're at the bleeding edge ;)

As your probably aware there is different releases of the GCC compiler. In addition to this, within the subversion sources on the development branches there is an 'autovectorisation' branch which acts as a research/development area for ideas relating to improving loop performance by using the CPU's SIMD (Single Instruction Multiple Data) instructions such as SSE, SSE2, SSE3.
This is currently baselined on the GCC 4.3 sources.

So what I would do is use subversion (svn) to download the latest development version of this autovect branch of GCC. Now build it from bootstrap upwards, ensure you use a postfix identifier such as "-4.3-autovect" so that the compiler binaries get created as "g++-4.3-autovect" for example. This means you still have your older version of the compiler. Ensure your CLOCAL is set up too or every binary you run will output a warning!

Once the entire GCC compiler set and supporting libraries have been built.. we get on  to the fun bit :D

I use ./configure but name the "gcc-4.3-autovect" using the CC flags, use the same override parameters to point to the "g++-4.3-autovect", then I usually use (off the top of my head) the CCFLAGS="-m64 -march=athlon-fx -msse3 -ftree-autovect".

This means the application will be complied as 64bit with SSE3 for the athlon-fx (ie my X2 4400) and last but not least... enabling the compiler optimisation for autovectorisation.

I've done this for:
Latest Amarok  - goes like stink (and that's the FFT visualisations for example!)

Latest Sea Monkey - zoom!
