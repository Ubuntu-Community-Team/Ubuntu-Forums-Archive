### Post by mdmbkr on 2007-07-29
Hi,

I have three nearly identical systems, two of which are running 6.06 LTS with kernel 2.6.15-28-amd64-server, and one which is running Breezy Badger with kernel 2.6.12-10-amd64-k8-smp.

The Breezy system has a moderately large mysql database that I need to load on one of the LTS systems.  To do this, I used mysqldump which generates an approx. 5GB text file.  I then copied the file onto the LTS system and ran mysql < dumpfile.sql.

The problem is that when attempting to load the database, mysql will either error out ("sql syntax error"), segfault, or the machine will lock hard (i.e., have to use the reset button).

My first suspicion was bad memory, so I ran memtest86 for about a week, and saw no errors after about 70 passes.  All three systems have 8GB of corsair registered/ECC memory, and all three systems pass memtest86 with no errors.

Next I became suspicious of mysql and mysqldump.  When I examined the location of the dumpfile where mysql would crash, I saw corruption.  Initially I believed the corruption was caused by a bug in mysqldump, but this is not the case.  If I examine the dumpfile on the Breezy system (where it was generated), the corruption is not there.

So now I suspect there may be a bug in the kernel I'm using.  This suspicion is reinforced by the fact that the same corruption appears on either of the LTS systems running this kernel, but does not appear on the Breezy system running an older kernel on the same hardware.

The next step for me is to try booting an older kernel to see if the corruption is fixed.  But I'd also appreciate some input from the community on this issue!  Thanks for reading.
