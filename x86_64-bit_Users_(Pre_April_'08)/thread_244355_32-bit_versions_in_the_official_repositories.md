### Post by The Keeper on 2006-08-26
I searched about this but such general search words either turned up a ton of non-relevant threads or none when trying to be more specific.

Either way, I don't really understand why use of 64-bit linux distros or specifically Ubuntu in this case has been made more difficult than it should be. Why not include 32-bit versions of the packages that have known incompatibility problems in the official repositories? If not under main and restricted repositories, then universe and multiverse at least. Simply name each non-64-bit package as packagename-32bit or something like that.

Now there's this huge "How to get specific programs to run under Dapper Drake 64-bit edition" thread stickied which makes linux-newbies to scream in agony. Why aren't these same packages found in the universe and multiverse repositories a few mouse clicks away to be automatically installed by Synaptic or Adept?

We could have both 64-bit and 32-bit Firefox installed from the repositories and also have 32-bit flashplugin-nonfree package in the repository which would then work with the 32-bit Firefox. The same goes with all other 32-bit-only packages that are missing in the repositories and 64-bit packages that have know incompatibility problems such as 64-bit Firefox with plugins/extensions.

I'm sure there's a valid philosophy behind the decision not to include 32-bit packages in a 64-bit distro repositories, but it is nothing more than shooting your own leg for having regular users to go into great lenghts to get everything work like they work in 32-bit distros.

Not having 32-bit alternative packages available in the repositories not only force a good portion of 64-bit users to install 32-bit software once, but multiple times as updates are released. Without auto-update features these manually installed softwares become eventually insecure or are missing important features that have been since added, unless the user also remembers and bothers to update the software manually.

The point is that 32-bit software can be get to work with little trouble, but why do the regular users have to go through that in the first place when pre-built packages could be snucked into the universe and multiverse repositories from where installation would be straightforward and painless. It's not even a problem for pre-built packages to install 32-bit versions to different folders than 64-bit versions if also available.

I guess this has been discussed to death at some point earlier and all I am getting is a shrug with a link that I failed to find, but maybe I'll get an answer as to why it has been decided against this obviously end-user-friendly option.
