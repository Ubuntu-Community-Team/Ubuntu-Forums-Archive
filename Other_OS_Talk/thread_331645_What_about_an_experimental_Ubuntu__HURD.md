### Post by JasonCozens on 2007-01-17
I think one of the problems with the hurd is that it is still a single  threaded OS using preemptively scheduled timesharing   ([Towards a New Strategy of OS Design]("http://ww.gnu.org/software/hurd/hurd-paper.html")). So although most of the kernel is broken into translators and tanslators can be added by users on a single pc the advantage over existing kernels is not as great as it could be.
        
 I've been looking at the Hurd since December and have been wondering  whether it could be used with some ideas I've started exploring about  scheduling tasks across multiple processors. There are some notes I  wrote in 2006 at [http://developer.novell.com/wiki/index.php/OpenE_-_Lab_4]("http://developer.novell.com/wiki/index.php/OpenE_-_Lab_4")
        
I haven't spent much time on them yet but I'm thinking about starting a project (possibly on LaunchPad) to explore them further. I started thinking about these when I got a Nokia 770 and I was wondering how I could make it more powerful.
        
The basic idea is to schedule jobs with processors using a new protocol much the same way as a job exchange works. At the front of the job exchange is the job board and processors queue up to get work. When a job is put up on the board the processor nearest the front of the queue that can perform the task takes the job and removes itself from the queue. It then goes of to find the processor that advertised the job.
        
All the job exchange does is job scheduling. To implement this I intend  to use UDP. A channel is reserved purely for scheduling. The UDP packets are the job board. When a processor has some work to do it updates the  job board and broadcasts. The new protocol then ensures that the processor at the front of the queue gets the job. If this processor has fallen asleep then the next processor gets the job. The intention of the new protocol is to avoid collisions on the scheduling channel whilst responding to requests as quickly as possible. It is intended to work with heterogeneous tasks and with inexpensive components.
        
The problem has been working out how to break down the task and I think  that the Hurd might help with this.
        
I'd be interested in hearing if anyone is interested in getting involved with this.

[Jason Cozens]("https://wiki.ubuntu.com/JasonCozens")
