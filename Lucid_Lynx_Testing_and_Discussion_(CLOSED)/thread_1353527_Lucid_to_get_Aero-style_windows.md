### Post by BwackNinja on 2009-12-14
what's great about this change is that it would lead to window decorations that actually look nice - hence the need for rgba, so that we can have smooth edges. the problem is that applications aren't ready for rgba, which is why cimitan, author of the murrine gtk theme engine, didn't recommend just patching gtk, but instead individual programs, so that they would not experience the problems with this patch to gtk that we are experiencing now. beyond the 'omg, we can make this aero-ish', we have more flexibility in drawing, such as the ability to have rounded corners on menus.

a big area where there are problems is the notification area, where some applications fail and others work fine. another is xembed, which is the exact same situation, if you are embedding a window in another window, they have to have the same colormap. thus, programs like gnome-mplayer fail because it embeds another application window, and flash fails because its doing the same thing.

another big problem is applications that only pretend to be gtk, like firefox and openoffice.org. these applications will just look out of place with their inability to use rgba at this point. taking firefox as a lone example, a pure gtk firefox would be just trading one set of problems for another. there would be theming problems, at least being a lack of compatibility with previous themes, extensions that add themselves to toolbars, etc. would have to be reworked. all this deep down integration work would have to be done that would affect how the browser worked and hurt firefox's ability to grow to be even more featureful, extensible, faster, and memory efficient in the future.

making rgba compatible applications isn't hard. what's hard is making existing applications work well with it. even gnome-system-monitor, which has rgba by default, doesn't look good with it. the tab with the graphs, which are all custom widgets, does not look good at all. with a transparent background, the graphs still have an opaque and out of place one.

I was talking about for a user to do themselves, he *is* pushing for gtk to have rgba colormap by default. In an ideal world, you could just expect apps to just be fixed to work with rgba, but there are many useful applications that have very slow development, such as having only one developer with not so much free time. Rather than apps not being able to ship because of rgba, rgba would not be able to ship because of apps. Its not even something that they would just throw into a development release, but instead just have people volunteer themselves and try it because it breaks so many things. There's also the opt-out, just specifically setting an rgb rather than rgba colormap, which (I'm pretty sure) nautilus does for drawing the background in the packages in the ppa.

Putting this in ubuntu is like putting pulseaudio in ubuntu. Yes, it has existed for a while and it has great potential because it increases flexibility to a great extent, but until things support it, and support it well, you'll hear more complaints than praise. Good as an idea, but not necessarily good in practice.

RGBA's *only* benefit over RGB is transparency, and what makes it great is how its used. You can (with murrine and rgba) draw widgets without having to draw their backgrounds as well. [https://wiki.edubuntu.org/DesktopTeam/Specs/SpitAndPolish](https://wiki.edubuntu.org/DesktopTeam/Specs/SpitAndPolish) has a good synopsis on that. The whole "parent-relative" way of drawing can phase itself out, although will still need to stay for a while because of lower-end computers that don't support compositing and for people who refuse to use compositing but still want "transparency" even if (like the cake) it is a lie. The reason this transparency is being pushed now is that we have the client-side window decorations, which will finally give us smooth window borders using metacity (with compositing). The problem that still remains with that is if it *requires* rgba, then people will be left with ugly or unusable window decorations if they aren't using compositing.

I'd support a system-wide change to rgba to do whatever I could to help move this along, but this isn't something I'd expect to be able to be included for a long while. The drawing of custom widgets would need to be changed for a lot of applications, and often might need a mode for both rgb and rgba independently. This needs to be able to be done, fixes only; no hacks or workarounds.

Also, I've written some rgba gtk cairo apps, and they used lots of cpu with the patch and near none otherwise.

Sidenote: Sorry for my lack of capital letters in my last post, an update (probably from xorg-edgers) left me with an inability to type capital letters.
