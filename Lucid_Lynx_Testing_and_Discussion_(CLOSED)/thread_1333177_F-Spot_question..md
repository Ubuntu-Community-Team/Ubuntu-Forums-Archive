### Post by knarf on 2009-11-22
I use f-spot to manage my photos (about 13000 at the moment). I track the latest git version and have written a number of scripts to do things which f-spot itself does not do or does not do well enough. As to why I use f-spot the following is a short summary:

 - it is free software
 - even though it is built on mono which does raise a few questions it gives me the freedom I want: if mono - and with it f-spot - ever were to go 'boom' I still have free access to my complete photo collection, including all the tags
 - the sqlite database makes it very easy to create scripts which act upon photos based on tags
 - the interface and workflow are OK

Things which can be improved:

 - f-spot does get rather slow sometimes
 - import seems to only work once every other time: the first time it just hangs. Kill it and restart import and presto, it works. The next time it hangs again.
 - video is not supported, why not?
 - it is quite limited in editing capabilities, even more so after the tools have been moved into the sidebar (eg. no size constraint for crop anymore - this used to work)

What would it take for me to move away from f-spot to an alternative?

 - it needs to be free software
 - the program needs to support tagging and acting upon tagged photos in a way similar to f-spot
 - it needs to store all data in a freely accessible, open format which lends itself to scripting
 - it needs to have a simple, usable interface - no whiz-bang kde-like orgy of buttons and icons
 - it needs to be flexible - it should not try to force me into one specific way of organizing my photos
 - it should run on modest hardware. My main system is a Thinkpad T23 with a 1.2 GHz PIII-m and 768MB of RAM and a wheezy Supersavage videoprocessor. Trying to run something like blueMarine on that system is not recommended...

Would I move away to an alternative which has less features but does not rely on mono? As long as the above conditions are fulfilled I might.
