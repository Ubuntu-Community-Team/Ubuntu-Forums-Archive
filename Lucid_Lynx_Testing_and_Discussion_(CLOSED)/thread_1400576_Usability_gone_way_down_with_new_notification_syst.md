### Post by Keyper7 on 2010-03-02
> Well, it was Empathy causing the notification there is no way to generate a bug-report through the GUI for the notification daemon.

```
ubuntu-bug notify-osd
```

or, if you didn't know the package name,

```
ubuntu-bug
```

for symptom-based reporting. And before you say something in the lines of "this is an obscure command line I have no obligation of knowing", it is fully documented with screenshots in [the wiki for reporting bugs](https://help.ubuntu.com/community/ReportingBugs), which is the first link in the [official page about reporting problems](http://www.ubuntu.com/community/ReportProblem), which in turn is the first occurence for "ubuntu report bug" in Google.

> Empathy should be able to control what is done with their notifications.

If that was true, then it wouldn't be using a notification system in the first place.

> Even though you have both panels the "bug" is still there.

As well as the "bug" most applications with tray icons have, which is not having an icon if the user explicitly deleted the notification area applet from the desktop, or the "bug" of not having an alternative way of retrieving a minimized window if the user deleted the taskbar. I don't think filing a report for each program with those "bugs" would be very reasonable, though.

> Why would I uninstal notify-osd and install notification-daemon?
All programs in Ubuntu are changed to use notify-osd, no?

Technically, notification-daemon and NotifyOSD implement the same specification. Some patching was done, but as notification-daemon is still offered in the repositories, replacing one by the other should not cause breakage and [other users did it before](http://ubuntuforums.org/showthread.php?t=1396935).

> What are you talking about! I'm the original reporter, the Bug was answered by Omer the day after it was filed. It was changed to notify-osd bu Omer. What else should I do?

A more useful question is "what else the developers should do?". Your report offered little more than personal opinions and a non-standard scenario. To expect results you should present your case better.

> Yes it's impolite, the bug was written in frustration and as a use case that Ubunut/Canonical have totally ignored from the start.

First, impoliteness has no justification and in most of the cases is not helpful in getting your point across. Specially in a bug report where you are expected to be objective and technical as possible and keep personal opinions to a minimum.

Second, there is a lot of "use cases" that Canonical have ignored and will keep ignoring for all eternity, such as making Ubuntu user-friendly for a chihuahua to use. In a bug report you are supposed to convince developers that your use case *matters*, and you didn't do much effort in this direction so far.

> The MessageMenu is bloated and requires more clicks than the old way.

Ah, concrete arguments based on a standard scenario. Now we are getting somewhere. However, not quite there yet: "bloated" is a vague subjective term and you should bring your concerns to developers, not me.

> Just because text appears on a web page on the internet does not mean it's public. A search for 'notification' on the Ubuntu wiki does not even show NotifyOSD on the first page.

And why would you restrict yourself to the Ubuntu wiki when searching for information that could be on the wiki, Launchpad, or the Ubuntu official site just to name a few options? If you type "ubuntu notifications" in Google, you get the NotifyOSD wiki as the first result.
