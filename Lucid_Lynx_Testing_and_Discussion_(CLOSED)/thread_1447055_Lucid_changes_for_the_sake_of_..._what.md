### Post by saulgoode on 2010-04-06
> **uRock said:**
> I don't think the button change is that bad, after all Mac's window buttons are on the left, but to each his own.

If you are willing to entertain the concept that those who hold opinions differing from your own might still "see with an open mind", I should point out that in the OS X interface it is still easier for the user to access the application's menu than it is to access the application's Close button. The reason for this is that moving the mouse pointer all the way to the top of the screen requires no coordination and very little feedback -- the user merely "slams" the mouse a sufficient amount for the pointer to reach the screen edge; there is no possibility of moving the mouse too far and missing the target. Pointing the mouse at the Close button, however, requires a more deliberate action and interaction with feedback from the pointer on the screen.

Under GNOME, the application's menu does not reside at the top of the screen and therefore access to the Close button is equally as difficult as accessing the menus (if the GNOME menu is hid, it is actually easier to reach the Close button). Placement of the Close button on the left above the menu is facilitating what is literally the last thing one wishes to do when using a program at the cost of the user needing to be more cautious in avoiding an unintended exit when performing operations within the application.

### Post by Mr. Picklesworth on 2010-04-19
> **itsjustarumour said:**

And IMHO another reason why an otherwise extremely nice and solid release is being spoilt by whats rapidly shaping up to be a dogs dinner of a UI.

For what it's worth, I can correct you on the technical reasoning. The indicators at the top right all work through the StatusNotifier specification, so they can be displayed with a variety of front-ends. (Be it under Gnome, KDE or a text-based shell).

In our case, everything is displayed with a regular GTK menu bar. There are some advantages to that. One is consistency, which has a lot of far-ranging, generally very positive implications. Another is keyboardability (granted, broken, but theoretically better). Yet another is that the specification has a much more decisive name and use case than "system tray", so we may see fewer apps abusing it in the future.

Naturally, the menu thing poses a slight problem for the volume control, since menu items are horizontal and volume controls tend to be vertical. So, a slider item was added to the spec so this could be implemented, but it has no impact on the look / feel of the menu; it's a horizontal slider because of how menus are designed.

On the bright side, this could in theory allow for multiple sliders (permitting immediate control of different output devices). More importantly, this provides the volume control with the same benefit that is everywhere else in the design: we don't require the user to right click for some functions and left click for others. Everything, including Mute and Sound Preferences, is immediately discoverable by left clicking.

That would not be possible if the volume slider was vertical, because other options couldn't fit into that space.
