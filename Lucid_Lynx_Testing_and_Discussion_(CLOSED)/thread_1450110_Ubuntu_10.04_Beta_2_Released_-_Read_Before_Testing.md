Beta 2, the second and last beta release of the Lucid Lynx development cycle leading to Ubuntu 10.04, has been released. Refer to the links below for more information:

[list] [*] [Release announcement]("https://lists.ubuntu.com/archives/ubuntu-announce/2010-April/000131.html") on the [ubuntu-announce mailing list]("https://lists.ubuntu.com/mailman/listinfo/ubuntu-announce")
[*] [Technical Overview]("http://www.ubuntu.com/testing/lucid/beta2")[/list]

If you're new to testing Ubuntu, and are running Lucid for the first time with Beta 2, the tips below will be especially useful for you. Please take time to read them.

[list]

[*] If you're having a problem with Beta 2, or are curious about a certain feature, change or malfunction, you're most likely not alone. Please do a forum search before starting new threads. This is very simple with the "Search this Forum" function on the upper right side of the forum index.


[IMG]http://img718.imageshack.us/img718/2897/screenshotxs.png[/IMG]

Click it, type some keywords related to what you're looking for, and if you see relevant threads come up, please post to those threads instead of starting a new thread. Remember that the best way to provide feedback and help improve Ubuntu is to **file bug reports on the issues you're experiencing**, and improve existing reports if they've been filed. 

**Please don't post your feedback on Beta 2 straight to this forum**. Forums are not a good way to keep track of issues, and your post will not be read by the relevant people who can fix your problems. Instead, do a search to find if there's a thread related to your issue(s), and if not, start a new thread, check if others are experiencing the same issue(s), and make sure a bug report is filed (doing a forum search, and/or [a search on the bug tracker]("https://bugs.launchpad.net/ubuntu/") is a good idea). If not, and you need help with [filing a bug report]("https://help.ubuntu.com/community/ReportingBugs"), don't hesitate to ask for help!


[*] **If you install the Beta once it's released, you can upgrade to the final release without having to do a fresh install**. But there may be some drawbacks in certain scenarios; refer to [the sticky thread on common problems and frequently asked questions]("http://ubuntuforums.org/showthread.php?t=1343449") for details.


[*] The milestone releases (alphas, the beta and the RC) are snapshots of the package archive at their pre-decided release dates. **If you have the latest updates installed on your testing installation, you don't have to remove your current testing installation and install Beta 2**, unless you want to test installation-specific components (the installer, bootloader etc.), or your installation is affected by potential corner cases that may arise during the development cycle that require a reinstall to fix. Refer to [the sticky thread on common problems and frequently asked questions]("http://ubuntuforums.org/showthread.php?t=1343449") for further details.


[*] If you're just starting to test Lucid, please read the [technical overview]("http://www.ubuntu.com/testing/lucid/beta2") and see if you're affected by any of the known bugs, take a look at [this wiki page]("https://wiki.ubuntu.com/UbuntuDevelopment/UsingDevelopmentReleases"), and [the sticky thread on common problems and frequently asked questions]("http://ubuntuforums.org/showthread.php?t=1343449") for information on various testing methods and good practices. 


[*] Once you've started testing, **exercise caution when upgrading packages** - in special, avoid partial upgrades offered by Update Manager unless you know precisely why it's offering a partial upgrade. Regardless of which tool (APT, Update Manager, Synaptic, Aptitude) you're using to upgrade your packages, **always check the list of packages to be removed, upgraded and installed before each upgrade**. If in doubt, check the changelogs ("Package > Download Changelog" in Synaptic, or "aptitude changelog package_name", or at [packages.ubuntu.com]("http://packages.ubuntu.com")) to see why a package may be being removed. If packages are held back, wait until they're installable again. Refer to [the sticky thread on partial upgrades]("http://ubuntuforums.org/showthread.php?t=1343434") for further information.
[/list]

So how do you think the iso testing can be done if the isos are not produced (and tested) before their official release?

This is how I understand the work around a development milestone release:

1. Freeze the repos. Don't update any packages unless it's needed to fix a bug in the installer or live cd environment. This happened on april 1:st for beta 2, and was in effect until beta 2 was finally released on april 8:th.

2. Produce candidate isos and let people test how they work. If serious problems are encountered, fix them and make new candidate isos for further testing.

3. Once there are functional isos of all flavours (desktop/alternate, x86/amd64 etc.), rename these to "beta 2" and release them on the scheduled release date (april 8:th for beta 2). This is why the beta 2 isos were actually produced two days before they were finally released.

4. Unfreeze the repos to allow further updates until the next milestone release freeze (the next for lucid is FinalFreeze, which starts april 15:th).

### Post by ezsit on 2010-04-10
> and i want the very latest version because i want to ensure as much as i can that the installer want create me
sda5 and sda6 partitions again but sda2 and sda3 partitions as it should.

Partitions can be primary, extended, or logical. Primary partitions are limited to 4 on a single hard drive with a standard mbr that Windows, Linux, and a bunch of other OSes use and understand. sda1, sda2, sda3, and sda4 are your four primary partitions. sda5, sda6, sda7... are your logical partitions. An extended partition is a special type of primary partition that acts as a container for logical partitions, so an extended partition is designated as sda1-4. YOU are in control of what type of partitions YOU create during installation. Ubuntu "suggests" using logicals and defaults to using logicals because they will not use up your available primaries. However, there is nothing but the 4 primary partition limitation preventing you from using primaries. Just understand what you are doing.

The Ubuntu partitions created by Ubiquity are inside of an extended partition. You already have sda1 & 2 taken up if Vista, Server 2008 or Windows 7 is installed. Ubiquity must create 2 partitions minimum, so it creates an extended partition which must be numbered as well. Since you are allowed a maximum of 4 primary partitions on a system with a BIOS, one such designation is being set aside should that need arise. Thus you would have sda1 through 4 for primary partitions, an extended being actually a primary partiton, and sda 5 & 6 for the first logical partitions within the extended partition. If you had all four partitions taken up, 2 for Server 2008 and 2 for Ubuntu, you could do no more partitoning without destroying data. Ubiquity placing Ubuntu's partitions within an extended partition looks like a safety feature to me. The only time I have had Ubuntu placed on separate primary partitions was when installing to a bare drive. 

You may find this [article at Microsoft Technet ]("http://technet.microsoft.com/en-us/library/dd799232%28WS.10%29.aspx")helpful. And [here's another on Wikipedia]("http://en.wikipedia.org/wiki/Disk_partitioning").

Furthermore, the Windows disk manager does not properly recognize the extended partitions and therefore logical partitions for Ubuntu. For some reason (a bug?), it displays my hard disks with 4 primary partitions, but gparted shows the ubuntu partitions within and extended partition.

So be careful what you wish for.
