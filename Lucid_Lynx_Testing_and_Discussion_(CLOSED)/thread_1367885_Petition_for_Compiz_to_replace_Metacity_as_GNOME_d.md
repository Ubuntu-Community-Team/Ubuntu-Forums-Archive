### Post by DracoJesi on 2009-12-30
NOTE: Most likely in the wrong area, I honestly don't know if this will do any good but I figure if people agree, this the way to find out and perhaps it will lead to a different solution or get the message across.

Compiz has revolutionized not only the Linux desktop, but desktops everywhere.
Compiz not only provides enhanced effects and productivity features but provides a performance boost as well.

The GNOME desktop environment has changed somewhat over the years but has generally stayed the same, until now. GNOME Shell provides many new features and I personally feel it is a step in the right direction.

There is however one glaring problem that, if you're a Linux user, even a KDE user... you know exactly what I'm talking about.

GNOME Shell integrates it's new panel and overlay within the default windows manager, Metacity. This means that if you replace it with another windows manager you lose the panel. This really isn't acceptable, we gain so new thinks from the shell but one can argue you loose more capability without Compiz. Even if you argue you loose less, it's still allot.

I honestly don't understand the conflict. The GNOME team says that this is needed, I've seen people debate that but frankly I don't care if it's integrated into a WM or not, but why Metacity?

To keep compiz one of two things must happen. (excluding staying with 2.x)

a) The Compiz team would have to fork GNOME 3, either take the code for the panel and overlay and integrate it into Compiz directly (this would require a whole DE to be done and the Compiz team is already overworked) or they would have to replace Metacity with the current Compiz WM (well, with modifications of course.

or

B) GNOME could actually replace Metacity and make Compiz the default WM. There isn't anything you can do Metacity that can't be done with Compiz. If it weren't for the extra features, you wouldn't know the difference as they look exactly alike. If Metacity can be modified to include the Shell, Why can't Compiz? Does it not stem from Metacity, is it not designed to do everything Metacity does?

Metacity isn't bad but it's but it's time has come, it has done it's job extremely well but it's time for it's offspring to take over.

I prefer this latter solution because it brings with it many benfits.

Even though I worry GNOME is worried about make things right at the cost of customization, I do like GNOME, it's layout, it's practicality, It seems best for serious work but it's rather simple compared to KDE in many ways as far as polish. Compiz was the solution and added functionality as well, for a long time now the two have gone hand in hand and that's a tradition I think should be kept. 

I like GNOME, but if Compiz doesn't come along, many will switch... will I? probably not but I may stay with 2.x, Surely someone will fork it, and incorporate Compiz, even if the Compiz team doesn't have the manpower, it's simply a matter of time. This is reassuring in many ways but then what will happen to GNOME in the long term? Will it become a relic?, a memory?

Also, if the Compiz team is allowed to take care of the WM, this will make the GNOME teams workload much less, will lead to tighter integration of the two and, well, the possibilities are endless.

So alright, build Shell on top of a WM, but choose a better WM, choose Compiz.

I must be missing something because it seems so simple to me. 

Why Metacity?  

Why not combine the best of both worlds on an integral level?

So This Petition states the following.

*Compiz should replace Metacity as the default

*Shell should be built based on Compiz, not Metacity

*This isn't an attack on KDE, they should be allowed to use Compiz as well.

Also, a desktop switcher like in 2.x would be nice, even if it's not default, or is that asking too much?

### Post by Amaranth on 2010-02-10
Mutter and compiz actually work completely differently. Mutter is metacity + a compositor built on top of clutter. This gives them a nice scene graph for doing drawing which probably makes it easier to implement effects. However, it also seems to cause it to run rather slow.

Also, while mutter is open for extension development I don't think gnome-shell is. Gnome-shell is implemented as a mutter plugin and it's the _only_ plugin. So if you want to add effects to gnome-shell you need to rewrite most of it anyway as adding effects to mutter requires tossing out gnome-shell as it exists today.

So the real debate would probably have to be "compiz vs mutter" and would still only be for people who don't want to run the default GNOME desktop.
