### Post by berndth on 2010-01-08
> **Robin Nixon said:**
> The main design change appears to be that left hand pane has been substantially increased by moving the "Places" heading up to the navigation bar, and extending the pane down through the status bar.

The navigation bar used to be window-specific. Now, with the possibility of split-view mode, it becomes pane-specific. The original implementation moved the bar around depending on which mode one was in (see [http://www.youtube.com/watch?v=yqBDEi17l2A]("http://www.youtube.com/watch?v=yqBDEi17l2A")), but a clear grouping of widgets that belong together was considered better than widgets moving around on the fly. That grouping also applies to the status bar ([http://git.gnome.org/browse/nautilus/commit/?id=04c26296fb78b6de1a5bf933e543734ce18bd062]("http://git.gnome.org/browse/nautilus/commit/?id=04c26296fb78b6de1a5bf933e543734ce18bd062")).

> **Robin Nixon said:**
> 
I suppose the tabs also got dropped down to the bottom because there is no matching "Places" tab within the top of the left pane any more.

I wonder whether it's all change for the sake of change. If not I'd love to see the ergonomic user testing report that suggested these to be a usability improvement.

I know, guessing is more fun than looking at the ChangeLog. As for tabs, that would be [http://http://git.gnome.org/browse/nautilus/commit/?id=c6ee7553fd029239eff50d32dbcb57223ffa185e]("http://http://git.gnome.org/browse/nautilus/commit/?id=c6ee7553fd029239eff50d32dbcb57223ffa185e").

That is because build-dep is for getting the *build dependencies* of the Nautilus source package. On the other hand, aptitude show would give you the dependencies for the running Nautilus binary package after it is built.

If you haven't built many packages on your system, build-dep will pull in a lot of development packages, gcc, make, etc. Naturally, they're useful for any other software development you may want to do :)

Since you're on the current development version of Ubuntu, there isn't much finger crossing involved. The beauty of Debian is that the system to build the packages you receive is automated. The build server looks at the source package's build deps, installs those and builds the final package. So, the apt-get build-dep command you gets your system the exact same tools that were there to build Nautilus in the first place.

First nautilus will be updated between now and release so you'd need to do this a couple of times, and you will want to update nautilus. (rather than keep the current

When you build the source as a package set upon install it will be seen as a re-install, immediately afterwards the repo version will be seen as an upgrade.
You can if you wish give it a higher minor version number, then when nautilus updates you'll be informed 

Only the nautilus package needs to be installed, the others can stay at repo installs (even if you up the 'minor' version.

I might be inclined to not remove the line, just change from _BOTTOM to _TOP

You should build as a user, install as root

simple way 
If you want to follow those instr. and use debuild then
```
sudo apt-get install devscripts
```

(navigate thru the postfix with arrow and enter keys, no need to set anything from default options 

If you just want to use dpkg (debuild does a few extra things, 
```
sudo apt-get install dpkg-dev
```

Then either way 
```
sudo apt-get build-dep nautilus
```

```
mkdir nautilus && cd nautilus
```


```
apt-get source nautilus && cd nautilus-2.29.91
```


Make your edit and if desired open /nautilus-2.29.91/debian/changelog and change on the first line 
nautilus (1:2.29.91-0ubuntu1) lucid; urgency=low
to
nautilus (1:2.29.91-0ubuntu2) lucid; urgency=low

Then back at terminal run either 
```
debuild -us -uc
``` 
or 
```
dpkg-buildpackage -rfakeroot -D -us -uc
```

5 min or so later just install the nautilus package, leave the rest, see screen ( screen shows debuild which creates a new .dsc and diff.gz and a build log - not really needed here.

Note
if you leave the ver. number the same and are unhappy then just update back to repo package.
