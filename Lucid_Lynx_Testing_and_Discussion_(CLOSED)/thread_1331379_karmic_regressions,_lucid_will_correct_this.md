### Post by seeker5528 on 2009-11-23
> **Regenweald said:**
> You know what I mean Seeker! :D

Yeah, but I still felt it needed to be said. ;)

My biggest frustration is that what many people view as problems with audio in linux goes back to a single bad assumption. The assumption that hardware is incapable of mixing streams, so for example if you want to play a game with sound and still have an audible alert when you receive email, it couldn't be done. For those of us who used Soundblaster cards, that was never true, and even for some of the fairly common on board audio these days, it's not true, but because developers assumed it to be true, applications were developed that grab exclusive access to the hardware whether it made sense for that particular application or not, sound servers were developed and required, by KDE in particular, that grabbed exclusive access to the sound card, so those of us who had hardware that was capable of mixing streams were limited in many circumstances to the same set of limitations as those who didn't.

In the mean time dmix was created to provide the software equivalent of the hardware mixing, and for most hardware that was not capable of mixing it was enabled by default, the KDE devs got rid of ArtsD and came up with Phonon, which provides a fairly basic set of multimedia capabilities to the developers working on Multimedia applications, but plays well with others, so for those applications where it makes sense the developers can use ALSA directly or some other alternative and Phonon will not get in the way.

So now you have Pulseaudio coming along, which does provide some desirable, "modern" features, but regresses to the assumption that it should have exclusive access to the hardware.

That's my one big gripe with Pulseaudio.

It's my personal opinion that if Pulseaudio was designed to play well with others and extended off of the capabilities of ALSA in a more organic fashion instead of grabbing exclusive access and doing everything on it's own, then some of the complaints people have about it would not exist.

Later, Seeker

### Post by seeker5528 on 2009-11-23
> **RAOF said:**
> Something that *I* run into all the time is plugging in my USB speakers.  Thus, Pulseaudio Works For Me™, but ALSA does not.

Blanket statements rarely cover all situations. Even on windows introduction of USB audio some times does not produce desired results if you expect it to work with your on-board audio.

If the audio is not working out of the box with ALSA alone, then that is a bug and should be fixed in ALSA.

It's been a long running and common complaint with Linux audio that 'If I have application X open, I don't get sound in application Y'.

> I'm don't think that "all the problems of linux audio exist because of software mixing" is a true statement, either. :)

There is a smiley there so I'm not completely sure how serious that reply is, but.... That's not what I said, I said many of the problems (many is not all) resulted from the assumption that hardware did not provide the mixing, and it was fairly common that people had hardware that didn't, but soundblaster cards always provided the hardware mixing, so creating software and/or sound servers based on the assumption that they needed to grab exclusive access to the audio hardware was starting with a false assumption and artificially limiting to those of us who had hardware that was capable of hardware mixing.

Dmix is the ALSA answer to the problem for people who have hardware that does not do the hardware mixing. If it is not working for a particular combination of hardware and driver, then that should be considered an ALSA bug and fixed in ALSA. Dmix is not the end all/be all of software mixing, it could be better, but if something is going to come along and replace it, it should actually replace instead of tucking the mixing functionality inside a sound server that grabs exclusive access to the audio hardware and expects all audio output to go through the sound server, whether the software is designed to use the sound server or not.

Putting sound servers aside for a moment, there still exist some software that grabs exclusive access to the audio hardware, but these are primarily things that have not been updated to take advantage of the current set of ALSA capabilities and suggested practices, or some legacy stuff that still uses OSS, both of which are getting fewer and fewer these days.  

If Pulseaudio was developed without the assumption that it should have exclusive access, the people that are all gaga over it could use it for everything, the people who want to use it for all but a few things could, the people who only want to use specific features with specific programs (the network capabilities for example) could, the people who just want to kill Pulseaudio and have it stay dead without jumping through hoops or being required to uninstall it could do that, not assuming it should have exclusive access implies there should be something in the GUI interface somewhere providing the option to start it, stop it, or set it to run automatically at login or boot up.

Phonon seems to be able to provide applications the capability to adjust their volume independently, while still remaining friendly to applications that are not using Phonon, so I don't know why there seems to be a belief that Pulseaudio needs exclusive access to the audio hardware to provide that type of functionality. And I don't know why there is a belief that Pulseaudio should handle things that ALSA already does instead of either working with the ALSA devs to improve the built in functions or replacing those functions in a more direct fashion then building Pulseaudio on top of that.

If there is a feeling that Dmix is not good enough, it's a plug-in, it can be replaced directly with another plug-in that provides better mixing to all ALSA applications. It seems to me, if Pulseaudio can disable re-sampling in ALSA in favor of something Pulseaudio uses internally, the same should be able to be done as an ALSA plug-in that gets used by all ALSA programs.

Not having been designed in a friendly manner from the start it may be too late now even if the desire was there on the part of the developers, but one could hope. ;)

Later, Seeker
