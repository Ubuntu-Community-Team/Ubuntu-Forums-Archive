### Post by knarf on 2009-11-28
> **BbICEP said:**
> It's better to have an access to the file system instead of waiting for import which takes ages and then emulate file system with tags.

This works fine as long as your pictures fall within one category. As soon as you start 'tagging' (whether by using the file system or some tagging app) your pictures according to their contents you'll run into problems with gThumb or any other app based on similar assumptions. Multiple tags could be emulated using filesystem links but that soon gets unwieldy and tedious and still does not give the flexibility of tags.

Think about it: a filesystem is a hierarchical database which can be extended to something resembling a network database using links. It is not feasible to extend it to something resembling a relational database as that would lead to an explosion of links and still lack the tools to do arbitrary queries. For many tasks a relational database is better than a hierarchical or network database. Photo management is one of these tasks.

Maybe gThumb could be extended to support tagging in a sane way. It would offer the flexibility and functionality of f-spot combined with the speed, small footprint and filesystem-based functionality of gThumb.

Folders are totally unuseable for my collection. How would I select a series of images portraying eg 'two specific horses photographed on a specific location in winter' ? Not by using folders. With tags this is a simple query or tag search or dragdrop away. Or 'all images which have anything to do with harvesting silage' (as I recently used to document this process) ? Think 'Windows 95 shortcuts, "can not find file blahblah.jpg, did you mean bluhblugh.jpg?"'... hard links are somewhat better but less flexible (not across filesystems, not between directories on most systems, etc).

### Post by Gina on 2009-11-30
What language an application was written in is of no interest to the user!  What matters is performance and particularly, ease of use.  A consistent user interface throughout a system is important in this respect.  IMV speed comes second to usability unless it's painfully slow, but then, that is still covered by usability.  An app isn't user friendly if the user has to wait too long for a response.

Depending on the application, a slower programming language does not nesessarily result in slower performance.  In many cases, the software is waiting for input from the user.  Where a lot of processing has to be performed which might result in a noticeable pause to the user, the code must be more carefully designed for performance.  In some cases a change in language is appropriate just for that procedure.  eg. a library of assembler routines used by a higher level language.  I've often used this practice to achieve consistency and smooth running of an application when I was a systems designer and programmer (many years ago now, but the principle still applies).

So don't knock an application just because it, or part of it, was written in what you consider a "slow language".  It's the design and coding that's important, leading to something that should be a joy to use.

You run this risk any time more than one camera (with similar naming patterns) is used. The import day has nothing to do with the folder the picture ends up in as that is decided by the EXIF date in the picture. The risk is not big but is does exist.

Fortunately f-spot does check for existing filenames and acts accordingly as can be seen in FileImportBackend.cs:126

```
public static string UniqueName (string path, string filename)
        {
                int i = 1;
                string dest = System.IO.Path.Combine (path, filename);

                while (System.IO.File.Exists (dest)) {
                        string numbered_name = String.Format ("{0}-{1}{2}", 
                                                              System.IO.Path.GetFileNameWithoutExtension (filename),
                                                              i++,
                                                              System.IO.Path.GetExtension (filename));
                        
                        dest = System.IO.Path.Combine (path, numbered_name);
                }
                
                return dest;
        }
```As you can see this method returns the original filename if no file already exists in the given directory. If it does exist it adds a number and keeps increasing this number until it hits a unique name:
