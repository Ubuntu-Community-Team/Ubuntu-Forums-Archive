### Post by Sophont on 2008-09-17
> **imgkg said:**
> why can't they just make one stable robust version per year or more rather than to release half baked versions every 6 months

Feel free to only upgrade once per year. :-)

What makes you think that yearly releases would be much less half-baked (accepting for the sake of argument that this is currently the case - though I would actually disagree)?

I guess you assume that a yearly release would have a longer testing period? Possible - but that wouldn't necessarily mean better releases overall. Most of the testing happens when release date gets close and more people actually use it. Many if not most problems derive from certain hard-/software combinations that are not present on the machines of the main developers.

Also yearly releases would mean that many enhancements take longer to reach users - which is no less of a problem than bugs that weren't fixed yet. To the end user it doesn't really matter if the software doesn't work because a) a bug wasn't fixed in time or b) it wasn't supported yet at all.

Every release is a compromise between having more/improved features and less bugs.

And the vast majority of the packages come from upstream. Distributions select and sometimes patch and in a few cases actually add specific packages - but Ubuntu is mostly Debian testing (AFAIK) and Debian is mostly a packaging of a large number of individual projects.

But back to disagreeing with the half-baked. It's a diverse hardware world in the wild and people select any funny combination of software packages with a zillion combinations of config settings and in some cases user mistakes or faulty hardware. So there will always be some who have more trouble than others. But it's the people with problems that are most likely to post - sometimes creating the perception that the releases are worse than they really are.

### Post by carusoswi on 2008-10-08
Collin:
I went into Launchpad to file a bug report, but did some browsing first and stumbled across the solution.

Check out a post by Sokraates on 10/7/08 (sorry, don't have the link).
His solution is to comment out these lines in your /etc/network/interfaces file:

"iface XXX inet dhcp"

To do this, open a terminal, type: sudo gedit

When the editor comes up, use file-open, then navigate to your /etc directory, then your /network directory, and double click to open /interfaces.

Look for the line in quotes above, where XXX represents the name of our interface (eth0 or wlan0).

Use '#' to comment out those lines.

In my case, I found "iface wlan0 inet dhcp"  and I typed a single character so that it now reads "#iface wlan0 inet dhcp" (without the quotes).

I saved the file, rebooted, and all is well again.  I'm typing this via my Ubuntu wireless connection.

Login - My login screen was changed back to the default as well. It's not the end of the world, but why did it do that? For an "upgrade," there should be no need to meddle so much with existing user settings. Yeah, some things need to be changed regardless of the user setting .... But the splashes? Come on.

Desktop - Thankfully, the desktop did not get changed in the upgrade process. Same background and panel/bar settings. For some reason, the Compiz Fusion I was using in Hardy did not get upgraded/updated, so I had to manually reinstall it. Performance is noticeably slow using Compiz Fusion under Intrepid. The rotating cube effect that was smooth as silk in Hardy is now jerky and jittery under Intrepid. Is it a problem in the graphics driver? A problem in Compiz? Is it something that will get hammered out in an update to Intrepid? I'll wait and see.

Applications could not access ANY mic at all. Sure, I know there's a config file somewhere where I can add a line or edit a line and be good-to-go. Maybe it would have worked better from a fresh install. I don't know. But if Linux vendors want to get a competitive edge on Windows, they're going to have to work harder on dummy-proofing things so people don't have to hunt for solutions on forums and tweak config files.

Wireless LAN - Worked great! Detected networks, connected to networks (tested both WEP and WPA2 connections), and registered reasonable signal strengths. No weird problems with DNS resolutions or anything. And a big plus ... My Acer has a little light built into the WLAN button to let you know if it's on/off. The light didn't work in Hardy, but it works in Intrepid.

Applications - A lot of applications had to be reinstalled or reverted back to default settings to work properly (I'm mostly referring to the applications that you get from the Ubuntu repositories). Why was this? Really, maybe a clean install would work better for me next time.

System Resources - Ubuntu has never been piggy with the RAM. It's ridiculous ... I have 4GB of RAM on the laptop, but Ubuntu rarely uses more than 600MB. Oh well, the laptop has to run Windows as well, so the RAM is definitely needed for that. And the CPU consumption was just fine. No piggy background processes eating up cycles (unlike Windows).

Peripherals - I didn't test the memory card reader slot under Intrepid. But I do hope it works. Otherwise, most of the special function buttons and such on the laptop seemed to work fine. For example, the sleep, shutdown, brightness, and volume buttons all worked fine. Even the little odd-ball touchpad buttons on the laptop were all lit up and working.

All in all, I'd say Intrepid is a good step forward. Lots of things are better than Hardy. Unfortunately, there were enough things wrong (Sound Manager is my #1 gripe) to make me see that my laptop actually lost gross functionality under Intrepid.

So I dusted off the Hardy CD, and regressed back to 8.04.1 ... As soon as the very talented, intelligent folks working in the labs gets these problems hammered out, I'll be first in line to go Intrepid!



Installation - Went smoothly.
