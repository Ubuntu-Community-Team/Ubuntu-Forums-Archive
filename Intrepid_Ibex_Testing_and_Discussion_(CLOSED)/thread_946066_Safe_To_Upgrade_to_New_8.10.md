### Post by crazyness003 on 2008-10-14
> **russ5811 said:**
> In windows, you have to do a clean install of a new OS version to get the best results.

Is this the same with Ubuntu?

Yes, it is the same as Ubuntu. But please be advised, the upgrade process works wonders compared to the windows one. But in my experience its always best to do a clean install, since there may be some directory shifts, driver-details, and other system-critical changes that can occur.
Now, these subtle changes usually dont affect the average "lemme get my email, write a document and not have trouble with my drivers" user. Meaning, a regular user need not worry about these changes. But, if your gonna get curious, mess around with "hidden" properties, and compile, then a clean install is always a better choice.

It all boils down to this:
1. Back up (I cannot stress this enough. Even if you dont plan on doing-system wide changes, crashes occur, HDD go bad, and s**t happens.)
2. Do an upgrade. Use the feature, and if you see that it needs improvement, let someone know.
3. If the upgrade didnt work...Do a clean install. 
4. Done.

### Post by psyke83 on 2008-10-14
> **crazyness003 said:**
> [[COLOR=#76482C]**wgrant**[/COLOR]]("http://ubuntuforums.org/member.php?u=57283") made an excellent point. The only problem is that few (to my knowledge) users ever stay within the packaging system and not use 3rd party stuff.

Yes, but it's very easy (and highly recommended) to reign-in third-party packages before dist-upgrading.

The process I follow (in Synaptic):
[LIST=1]
[*]Go to Preferences/Repositories -> Third-Party Repositories. Uncheck ALL third-party repositories and close dialog.
[*]Reload the repository listings.
[*]Click on the category Status -> "Installed (Local or obsolete)".
[*]If there are any unwanted packages listed (that aren't a part of the regular installation), you can uninstall them normally.
[*]For the remaining packages listed, click on a package and select Package -> "Force Version...", and select the current release (i.e. Hardy). 
[COLOR="Red"]N.B.: If Synaptic asks to remove other packages during this process, then cancel the operation[/COLOR]*.
 
[*]Repeat step 5 for each package until all packages are downgraded, then apply changes.
[*]If some packages cannot be downgraded, it's better to remove them if possible.
[*]Optional: Click on the section "Not installed (residual config). Select all, right-click and select "Mark for Complete Removal", then Apply.
[/LIST]

* Sometimes you cannot easily downgrade packages unless you follow the proper order (to prioritize libraries over binaries). You can avoid this problem by using apt-get, e.g.:

```
$ sudo apt-get install <package1>/hardy <package2>/hardy ...(etc.)
```

Providing that you haven't installed a lot of software manually, these steps should ensure a much smoother upgrade.

IBM Thinkpad R31, 1 gHz Celeron, Intel Graphics, 512 mb memory runs 8.10 O.K. only with some workarounds and cautions:  

1. Trackpoint mouse causes crazy reactions when pointer gets near an edge.  This happened when some code from Debian was put into Feisty.  My fix is to use a USB mouse.

2. I've got 20 gb so there is room for two 6.6 gb installs.  One is Intrepid Alpha 5 NOT UPDATED which works O.K. for an Alpha.  Some bugs but functions.

3. Other install is this updated Beta.  CD Live Beta and Alpha 6 won't boot as is on the Thinkpad locking up tight just when the Gnome desktop should display.  Launchpad bug #277344 which is marked "low" importance, i.e. not likely to get fixed.

Install will, however.  Install doesn't use compiz.  I'm not brave, there's the other install which still works.  The resulting install locks up just like CD Live.  No surprise.

4. CD Live will boot only if I watch for X windows to start up, screen flashing, and quick do Ctrl-Alt-F1 to get a command prompt line.  This is time depent and I may have to Ctrl-Alt-F1 a few times.

Quick enter the following:
sudo apt-get remove compiz
sudo apt-get remove compiz-core
Ctrl-Alt-F7 to return to full screen.

If this works, it proves the bug is in the compiz code.  Compiz is entirely for "eye candy" fancy effects and, my words, plays tricks with the graphics hardware and software.  There are so many different hardware/driver combinations out there that the tricks that look pretty on one set break another.

5. If you try install, and booting the install gets the same lock up, then try booting in rescue mode.  
Select root command line prompt.
sudo apt-get remove compiz
sudo apt-get remove compiz-core
exit
resume normal boot.

That works for me.  I turn off appearance visual effects anyway.

6.  Another way to tell if compiz is the culprit is to try CD Live Xubuntu, which doesn't use compiz.

If Xubuntu is not flashy enough for you, try Kubuntu for lots of eye candy.  It doesn't use compiz either.

I haven't timed it, however my impressions are the later Ubuntu's like Hardy 8.04 and Intrepid 8.10 are a little faster than previous Ubuntu's.  Not like Microsoft Vista.
