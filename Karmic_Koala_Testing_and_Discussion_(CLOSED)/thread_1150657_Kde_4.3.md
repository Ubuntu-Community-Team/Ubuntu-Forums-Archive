### Post by SteveMcQwark on 2009-05-19
Most of these problems sound like packaging issues. To solve these, first fix your system with synaptic(more directly effective) or sudo apt-get -f install (less directly effective, but good if you don't have synaptic. If anyone knows another way to get to a non-broken system without synaptic, please pitch in)

Then follow these instructions: (they worked for me)
```
sudo apt-get clean
sudo apt-get dist-upgrade
sudo dpkg -i --force-overwrite /var/cache/apt/archives/*.deb
sudo apt-get dist-upgrade
sudo apt-get install plasma-widget-folderview
```

### Post by Zorael on 2009-06-23
> **jordan420 said:**
> hey guys. is anyone else having a problem with screen blanking in kde 4.3 beta2? if i set it up to shut off the screeen in in powerdevil it wont turn back on and i have to hard reset. not even an x restart will throw me out of it.
Intel graphics with xorg-edgers ppa enabled, correct? Sarvatt has been helping me debug this (or rather, the other way around :3), and a workaround is to set KDE to "do nothing" upon closing the lid or whatever event you had set it to turn the screen off at, and modify **/usr/share/acpi-support/screenblank** to look like the following. Backup first.
```
if [ `pidof xscreensaver` ]; then
        su $user -c "(xscreensaver-command -throttle)"
        if [ x$LOCK_SCREEN = xtrue ]; then
                su $user -c "(xscreensaver-command -lock)"
        fi

[b]#elif [ `pidof dcopserver` ]; then
#       dcop kdesktop KScreensaverIface lock[/b]

fi

**# xset dpms force off**

if [ x$RADEON_LIGHT = xtrue ]; then
    [ -x /usr/sbin/radeontool ] && radeontool light off
fi
```
You can't have it lock the screen either, or KDE will evoke dpms and stuff goes bad.

Basically, (for me) when KDE does dpms ("turn display off" or "lock screen"), it will hang as soon as I close the lid. Opening it up just greets me with a black screen. Furthermore, using xset to force dpms (**xset dpms force off**) is recoverable from if you force it back on (**xset dpms force on**) before some 50 seconds have passed; after that, something happens and it hangs. So even with KDE told to do nothing, the script that's run when you close the lid will still call xset and evoke dpms that way, which will result in a hang (50 seconds later, for some reason).

Hopefully a fix in the driver itself will pop up soonish.
