What I tried to say (rather cryptically) is that "commercial game support" isn't something that happens easily. So technically, yes we do have commercial game support. We have *very good* support for Windows games, if you take into account that normally (without Wine) that support would be 0.

In fact, with every little improvement to its user friendliness, Ubuntu is making *leaps* towards commercial game support. Those improvements attract users, and users expand the market for software development. It takes a long time because Windows has a ginormous market share and commercial game developers tend to be firmly set on their internal reusable code. This can be helped with developers putting out successful Linux releases to prove the validity of the platform, which happens at an acceptable rate. Smittysmit,

This isn't directed at solely at you, but you got me thinking with your comment on tarballs. Although the development release is somewhat more complicated to use, and perhaps what I'm going to say should not apply to development testers, you shouldn't need to install from tarballs on an Ubuntu system.

Almost every piece of software you'll ever need as a casual user is contained the repositories, or packaged elsewhere in .deb format if you know where to look. Most users resort to installing from tarballs because they read a HOWTO on some random website explaining the compilation process of a piece of software, when in fact that very software is already packaged and available in our repositories. Even today, some users still insist on installing the Adobe Flash plugin from the tarball or compiling their NVIDIA driver from source, when there are excellent packages available in our repositories to automate the installation of this software (as well as providing additional integration into Ubuntu compared to upstream code, and keeping it maintained through repository updates).

For casual users, compiling and installing from tarballs* should be discouraged, as it completely bypasses the packaging system that Ubuntu uses. I simply can't understand why people would want to compile and install from a tarball if there are packages available, because the resulting installation from tarballs leaves a complete mess of code on the user's system that cannot be tracked or maintained easily.

P.S. If there is no choice but to compile from tarballs, there is a very quick way to package the resulting binaries of a tarball into a debian package - **checkinstall**. Everybody should know about this excellent piece of software. Oh, and you guessed it, checkinstall is available in the repositories; no need to grab the tarball :P

*I'm not discouraging new users from learning how to compile software and experimenting with their systems in general - but I would encourage them to learn how to keep their system running properly and tidily.
