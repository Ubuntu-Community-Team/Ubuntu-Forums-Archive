### Post by kornelix on 2009-08-21
I have been reading and trying to understand grub2, and I have one fairly fundamental problem with it. I run several Linux distros on one PC, for application development and testing with the several distros.  I also have Windows Vista on the same PC. Up to now I have used legacy grub which I can re-install from a CD if I lose it, and the one simple config file is on the boot partition with grub.

My understanding (?) of grub2 is that the (many) configuration files reside in the linux file system that was used to install grub2. If I install another distro, I will have to install grub2 again and re-make any changes to the grub2 configuration files. There seems to be no way to install grub2 and edit its config files without a running Linux system that includes grub2. If I install another distro that installs legacy grub it seems I am really hosed. I suppose I could do some editing to move stuff between the old and new config files, but there should be a more elegant way. Stated as plainly as possible, I want to install a new distro without losing my boot setup - just add a new entry. I doubt if that is possible.

Can someone give me an outline of how best to manage these issues?

BTW, the documentation from the grub2 web site is pathetic. Someone in the Ubuntu community at least tried to clean that up a bit, and I am grateful.

### Post by ranch hand on 2009-08-21
There are definately some rough spots in the function of grub2.

A lot of folks are more worried about how it looks but I figure they are dual booting.

Multi boot brings out a lot of problems.

You can get all of your OS' booting and then either enter them in 40_custom (which will put them at the bottom of your menu) or in a file you create (say 06_custom) which will put them at the top of your menu.

If you are brave you can take away the executable function of all files in /grub,d after 05_debian_theme and your 06_custom file and then that is all that you are going to have on your menu.  To grab new OS' or update the kernal on existing ones you have to return them to their default and run update-grub.

I just let it build the entire menu below my custom entries

### Post by kornelix on 2009-08-22
SOLVED (somewhat, I hope)

1. copy /boot/grub/grub.cfg  to  /etc/grub.d/07_custom
2. edit 07_custom: remove trash, add /dev/sdaN refs, add custom titles, etc.
3. set all scripts in /etc/grub.d following 07_custom to no-execute permission
4. run update-grub
5. after installing a new OS using grub2: copy 07_custom, add menu entry for new OS, disable following scripts, run update-grub
6. after installing a new OS using grub1: try to avoid having grub1 installed since it cannot boot to ext4 OS with grub2, then reboot to OS with grub2 and do step 5.

So now I am using grub2 almost the same way I used grub1. Life is simpler that way.

### Post by dinxter on 2009-08-22
no holes, maybe a new upgrade of grub-pc might re add executable permissions to its scripts but i'm not sure. in theory you can just have one executable script in grub.d, containing your entire desired grub.cfg between
cat<<EOF
# all of my grub.cfg goes here
EOF
and this will generate the exact same grub.cfg on update-grub

another way i've been tinkering with to get menu renaming, delete menu entries, and various other things is to have custom scripts after the default ones that trawl through /etc/grub.d/grub.cfg.new (this is update-grub's temporary file) making alterations. 
so on running update-grub all the os-probes and so on are still run but then we get to rewrite whatever we like before it becomes the new grub.cfg proper
