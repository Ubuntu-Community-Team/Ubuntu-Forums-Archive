If they do not understand html 5 media they should ignore the video, audio, and source tags - and render the fallback div.

Malformed content is rendered if sent as html, thus often executing the XSS. XML parsers however require strict well formed content or nothing gets rendered.

Therefore, any my default browser MUST properly handle xhtml and advertise the ability to properly handle xhtml to web servers.

I use to use epiphany as my main browser years ago when it was gecko based. To summarise, I'd be more than willing to give a webkit based browser a shot, but it MUST

* Support fine grained configurable script blocker
* Properly support xhtml 5 media (preferably with GStreamer backend so my fluendo plugins can be used)
* Properly support application/xml+xhtml mime type and advertise that it does.

### Post by FunkyRes on 2009-09-07
> **Reiger said:**
> Well again there are complications with that. YOU know that you have a fall back element in video/audio. But does the browser know that?


The browser doesn't care. It sees a div tag with a paragraph inside it, it needs to render that - regardless of whether or not it acts on the parent element.

Internet Explorer does it that way.
Google Chrome does it that way.
FireFox does it that way.

When coming across a tag a browser doesn't understand, the browser should ignore the tag - but should render child nodes it does understand.

> 
It all comes full circle to the point that you should not serve HTML 5 for the time being.


wrong.
html 5 media tags were specifically designed so that fall back content could be used by browsers that do not understand it.

html 5 browsers will ignore the fall back child elements if and only if they know how to handle the specified media.

```

<grtew>
   <tytre src="someurl" />
   <p>I am a paragraph</p>
</grtew>

```

Sincr grtew and tytre are not defined, no browser should attempt to do anything with them. Every browser though should render the p node, even though they do not understand the parent node.

It's the same with html 5 media

```

<video controls="controls">
  <source src="/media/foo.mp4" type="video/mp4" />
  <source src="/medua/foo.ogv" type="video/ogg" />
  <p>Your browser does not support html5 media</p>
</video>

```

Every browser that does NOT support html5 media should render the paragraph. Browsers that support html5 media but do not support either of the video types should render the paragraph.

The only browsers that should not render the paragraph are browsers that support html5 media AND support one of the provided codecs.

The only browsers I've come across that do this wrong are the gtk-webkit browsers, epiphany and midori.

It's a gtk webkit engine bug. Those browsers are doing the wrong thing. Period.
