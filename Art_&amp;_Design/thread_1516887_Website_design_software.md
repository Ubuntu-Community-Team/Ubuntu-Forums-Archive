### Post by aklo on 2010-06-24
You design a website using programs like photoshop, fireworks / GIMP (Free). 

Design is getting the look and feel of your website down before you actually start CODING/DEVELOPING. This saves time since designing is easier than coding, you could draw boxes to represent various placing of your contents faster than you can type "<div>"


Start coding your layout using programs like NVU, Dreamweaver. They have a window to let you see where are the placement of various elements while you are coding so it give you a feel of how your webpage is coming out visually. It also has intellisense (Auto completion/suggestion) for css and other codes so it is easier and faster for professionals and newbies.

There is 1 CONS about these web authoring/developing softwares or IDE. The preview window sometimes display the layout wrongly. I used Dreamweaver i'm not sure which version but in that version, a valid css page will look out of place in the preview window but look ok when you view it using a browser...so I don't trust these programs when it comes to the layout of my site 

Ultimately you want to get your skills up to the level of not needing any of those web IDE for your layout except for a graphic editor. 

1. You can code comfortably using xhtml/css using purely a text editor with no intellisense.
2. Knowledge of IE6 css bugs, how IE6 displays some css differently from newer browsers. 
3. CSS short hand.
4. Code without using tables unless for displaying tabular data.
5. Not using deprecated tags E.G <font>

### Post by bigboy_pdb on 2010-07-22
I know it's been said, but [w3schools]("http://www.w3schools.com/") is a great place to learn from. As you feel more comfortable with writing HTML and CSS you should also look at their standards on the [W3C's website]("http://www.w3.org/").

> **bigseb said:**
> if one really wanted to go all the way what would be best?

What do you mean by "all the way"? There are a lot of interesting things that can be done, but just because they can be done doesn't mean they should be done. For example, you can try to process credit card transactions, but it would be best not to do that without first having a lot of knowledge and experience with site development, web servers, and security.

It would probably be best to start off with a site that doesn't process data on the server for a number of reasons:

[LIST=1]
[*] You'd have to know how to write client code anyway (HTML, CSS, and JavaScript).
[*] You'd have to set up a web server.
[*] Much more design and thought would be required.
[*] You'd have to learn about programming and databases.
[*] Your site would be far more likely to have bugs, which is unprofessional.
[/LIST]

I'm not saying that you shouldn't learn any of that. I'm just suggesting that it would be better to wait until you're very comfortable with client side concepts (HTML, CSS, and JavaScript).

If you need to build a site quickly and that is limited in what it can do, then visual site editing programs (such as Dreamweaver or KompoZer) might be fine, but they have limitations. Writing code in a text editor (such as gEdit or VIM) limits you to what a web browser can do, whereas using a visual site editor (such as Dreamweaver or KompoZer) limits you to what it was built to do.

With respect to using JavaScript, using a library, such as [jQuery]("http://jquery.com/") will make writing JavaScript easier. There are also user interface components, such as [jQuery UI]("http://jqueryui.com/") and [jQuery Tools]("http://flowplayer.org/tools/index.html") [(which has a component that can be used to create scrolling images)]("http://flowplayer.org/tools/demos/scrollable/gallery.html").

Here are also some additional hints:
[LIST]
[*] Test your site in multiple browsers, IE6, IE7, ..., Firefox, Opera, and Chrome.
[*] Always start your HTML file using a !DOCTYPE tag.
[*] Use IE conditional statements (which appear as comments to other browsers) to adjust your styles for IE6 if need be.
[*] Do not use JavaScript and CSS attributes in your HTML tags. Instead use ids and classes to add the content through external files. Doing this separates your content which makes it easier to read and debug.
[*] Use XHTML instead of HTML. This allows you to use XSL to transform your site contents if you need it and it allows you to embed XML based languages (such as SVG) into your page.
[/LIST]

To get an idea of what I'm talking about you might want to view the source code from [my site]("jdfortier.ca"). I also have a [list of technologies]("http://jdfortier.ca/pages/about/site.php") that I use on it. You can also look at how jQuery is used in the [art sections]("http://jdfortier.ca/pages/art/drawings.php"). Also, an example of server side processing is [my maze generator]("http://jdfortier.ca/pages/programs/maze/"). The data gets passed to the server, which passes back an image for the maze. Doing this requires PHP and the GD library.
