### Post by BradwJensen on 2007-11-03
**--Okay:**
So far I've made a "[Dark Mockup]("http://bradwjensen.deviantart.com/art/Ubuntu-Theme-Mockup-Dark-67903127")" & "[Light Mockup]("http://bradwjensen.deviantart.com/art/Ubuntu-Theme-Mockup-Light-69993148")" in Photoshop of a Ubuntu/Mac theme because i find the mac theme to be clean/simple/to-the-point; while the Ubuntu theme already has its thing going like the address bar.. lol

This would happen automatically when one shrinks a window too much..

(3)
I also added a Search text box on the window itself to replace the search button and make in simpler/faster to understand/use..

(4)
Merged the back/forth/up buttons because it not only saves space -- but anyone who's ever used a computer or will ever be shown how to use a computer will easily understand/know what they are and how they work.

### Post by gilgongo on 2007-11-03
Looks very nice.

I'd like to know how much the search is actually used by people. Currently, it searches the contents of the directory, which I find a bit pointless. If a search input field should be shown anywhere, I think we might take the convention that exists on the web right now: top right hand corner of the desktop screen itself, and not par of the window at all. We might also have a configurable setting to have that search always search the entire filesystem, or the active window (if people prefer the current behaviour).

As to showing/hiding various things like the icons, I think that's a preference rather than a feature of the actual windowing design. Personally, I like your approach of not having the (in my view pretty useless, since I use keystrokes) icons along the top. But that may not be for everyone. 

Incidentally, one thing I would really like Gnome to do is re-cycle windows (not found that preference if it exists). Currently, if I open (say) /home/gilgongo, leave that window open, then go to Firefox or something, then open /home/gilgongo in another way (say, from the "Places" menu) I get TWO identical windows displayed. That's annoying. I've sometimes found I've generated over 20 identical windows when I'm working on something, and some of them are showing old file listings - very annoying.

### Post by avik on 2007-11-17
One thing that's annoying when the notifications or information are on the desktop is that they are blocked by any maximized window.  As I'm using a 1024x768 resolution, that is a real problem.

With compositing window managers like Compiz, we're seeing some great interface changes that are not necessarily innovative, but helpful nonetheless.  This includes a widget layer, as well as widgets that are always visible.  Things like the volume bar when changing volume with a keyboard key already pops up over the current window to display pertinent information without be obtrusive.

Unless we have another new shift in design, like the shift from regular to compositing window managers, we can't change the interface so drastically.  Realistically, the best we can do in the immediate future is to try mocking up some interfaces with Glade, and see what we get.  I'll definitely try that.

Good ideas, though, everybody.

### Post by sicofante on 2008-02-02
@BradwJensen:

First of all: big thanks for your nice work.

I think your ideas are pointing in the right direction, both visually and functionally. I'm not especially in love with docks and I don't think removing "as much as possible from the desktop" is the right motivation, though. It can easily lead to remove essential items or put them in the wrong place (that's what I think about using a dock for the main menu or the trashcan, for instance).

These are the pros I find in your designs:

1- Merging the menu and the title bar. As a matter of fact searching for someone doing this is what took me to this thread. This is a good idea because the text in the two bars do usually not share the same area so they can use the same bar. No reason for the two bars, actually. The fact that it's not in the Gnome/GDK structure should not stop you or others from pushing for it. (Additionally, the fact that this is not Windows-like nor Mac-like should be appealing for those NIH paranoid devs too... This solution might be a Uniquely Invented Here thing. ;-))

2- Using a Windows-style launch-menu/taskbar/system-tray. I've been using Ubuntu for a while now and I believe my eyes fall easier to the bottom than go to the top. A second issue is that while the top panel in default Gnome is nice for putting shortcuts, two panels take just much too space from smaller screens (and these shortcuts can be put in a short favourite apps menu, as in Windows). This might not be an issue if you have a 20" or bigger monitor, but it definitely is for smaller screens and laptops.

3- The addition of 1 and 2 brings a very interesting side-effect: if you maximize  a window in this layout, you get a "Fitts's friendly" top menu for your application. Mac's global menu can't be beaten in small screens, and since small screen users tend to maximize every window they work with, this layout gives you that fast Mac-like menu for every maximized app. When using large screens, the Mac global menu has been questioned (quite rightly, IMO). In these scenarios this layout can give you the best of both worlds: you'll get an infinitely high menu when you maximize your window and a close-enough-to-the-window menu if your window is far from the top of the screen. Brilliant! (I know this maximized window behaviour removes the functionality for the two top corners, but I believe it's a reasonable trade-off.)

4- The colours both in the dark and light mockups are perfect IMO. I think there's a lot of people tired of the shiny designs in Windows, Mac and Linux. It's about time we go back to smoother, easier to the eyes, less flashy designs. Thanks a lot for that. Because I sell computers to the video industry, I know these guys like black themes (and the people behind [UbuntuStudio]("http://ubuntustudio.org/screenshots") know this too), so that's a good one for that industry. Lots of people prefer lighter themes though, so I guess your two choices are just right. (However, adding a few pale colours to the options wouldn't hurt.)

5- The layout for Nautilus is very nice and to he point. A configurable toolbar (a la Firefox) should be addressed by the Gnome devs though. The search bar functionality suggestions in this thread are very valuable too.

And there are a few cons too:

1- In my opinion docks are pretty (actually, very pretty) but flawed from a usability standpoint. Bruce Tognazzini has [explained]("http://www.asktog.com/columns/044top10docksucks.html") that very clearly. However, your proposal includes a classic taskbar too, so this is just half a con.

2- As Tog would put it "The trash can belongs in the corner." (I'm referring to the light mockup here). Actually, the Gnome trash can is the best one in the industry by far, just because of that. And I would say the main menu belongs in the other corner too. So: instead of using a two-stories high taskbar, make it only one and move the trash can to the corner. That will bring the main menu to the corner as well and only one line of minimized apps, which is the way to go if you believe in Fitts.

3- I firmly believe the x button for closing a window should be put anywhere but the right corner of the window. Especially when maximized, closing a window shouldn't be such an easy target and definitely not close to other non-destructive buttons like minimize/maximize. Put it somewhere it won't hurt that much (I can't make a thoughtfull suggestion right now though).

All in all, I like your design so much that I hope someone at Canonical takes the plunge and pushes the current Gnome design to allow for the merged menu-title bar. (I would also like to see an option for the global menu bar, but if a choice must be made, I'm definitely with you.)

I cross my fingers.

And by the way, I also think mailing lists are the wrong way for discussing development issues.

I had a couple of thoughts on user interface for windows.  I drew up a basic sketch, below.  

To solve the problem of the "close" or "x" button being too close to the maximize and minimize buttons the Max and Min buttons can be moved to the left side of the top corner of the window, next to what you have as the drop down/general button.  In turn, the "X" close button can be isolated in the upper right corner.From a usability perspective it reduces the risk of accidentally closing a dock when you want to maximize it, and (IMHO) would make a nice cluster of three highly functional buttons (Drop Down, Max, and Min) all in the upper left corner.  This would also differentiate Ubuntu from both OSX and Vista (esp. considering the "basic ideals" mockup I saw last bore a striking resemblance to Vista's buttons; [https://wiki.ubuntu.com/Artwork/Incoming/Hardy/Alternate/BasicIdeals](https://wiki.ubuntu.com/Artwork/Incoming/Hardy/Alternate/BasicIdeals))

### Post by GrouchoMarx on 2008-04-11
Good job on these mock-ups. I have to say that although I generally prefer lighter themes over dark ones, your dark theme does a much better job than your light theme at naturally partitioning the window into segments. Your light theme, to my eye, relies too much on solid, high-contrast lines. I think that with more scrupulous use of gradients, shades, and low contrast lines you could make the lighter theme more elegant.

I also think that the slider in the bottom right is unnecessary. Generally a person sets their icon size preference, and then there is no need to change it ever again (at least none that I can think of or have experienced). It does not need to take up GUI space.

With regards to the address bar: I think that it should be (relatively) flush against the first three buttons. With the sidebar closed, the address bar would seem to float at an arbitrary point 1/3 of the way across the window. Also I think that its size should remain static. I suspect that if you were to see it dynamically changing sizes, it would be very distracting. That space would be empty anyways in order for the field to expand.

The fact that file menu and the window title are in vertical alignment will be difficult to implement because window managers and GTK menu bars are currently rendered by different applications. It will also cause problems because the menu bar needs to be arbitrarily long, as does the title. How would the theme look with a long menu list, and a short title? Where would the title start?

Also you presuppose that the user will have compositing running, and therefore have shadows. This is not necessarily the case. Whereas your dark theme might look good without shadows, I'm not so sure how the light theme would look flat against the background. This is also something to consider.

How would the theme look without tabs? Without the side pane? Without the status bar? With multiple (three or more) toolbars? Remember that the layout of the GTK (or other) widgets cannot be an integral part of the aesthetics of the theme, because each application will have a different widget layout.

All in all, however, the sense style is pretty good. Maybe you could create a prototype using GTK and the pixmap engine, which would allow you to overlay your graphics onto the pixmap framework.
