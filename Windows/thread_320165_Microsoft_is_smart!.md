### Post by SuperMike on 2006-12-16
Regarding how hard it is to install stuff in Ubuntu. Well, for the most part, you're wrong. There's the apt command, and then there's the items on the menus for Synaptic Package Manager and Add/Remove Packages.

I will admit that some stuff doesn't automatically configure itself, or give you a GUI window to configure it, and you may be having to do that or download another tool that lets you do that. But don't you worry. We're working on it. A lot of this is changing and improving constantly. Microsoft began attempting to focus on user-friendliness back in the 1980s, but the Linux community, in my opinion, didn't focus on user-friendliness until GNOME and KDE came out, and I think that was in the late 1990s, and didn't really, in my humble opinion, get it /right/ until RedHat 8 came out, and that was around 2002. Many developers who write things for Linux today are focused more on the intelligent, informed, power user or geek -- that will always be there, but more developers are focusing on newbies too than ever before.

There's also a mentality out there, you must realize, that feels that if they make Linux too cute and too user friendly, it will not be used because it will be like a video game console or something. If you want cute, use Windows.

And then there's the cutting edge stuff. In the Windows world, they may not put cutting edge stuff out so early and may polish those pearls before putting them out. In the Linux community, often these pearls are put out early so that the community can provide feedback and improve it rapidly. Unfortunately on many of these items you may have to compile them by hand, and download and compile all their dependencies, and then configure these things by editing little files. Sorry about that, but that's the disadvantage we face when we get the advantage of trying stuff early before it is based for newbie user consumption.

Still, you had to pay for your copy of Vista, and I get my copy of Ubuntu for free (although I donate to the cause each year because I value the cause). A lot of hard work goes into the most popular Linux releases. Countless hours of programming time and hair pulling. And they do it for peanuts (sometimes) and get few accolades.

As for sudo, if you don't like it, there's a way around that, but it's risky because it means that you can get into a careless mode where you may not think about the carnage you or any of us can bring to Linux at the command line. In fact, I won't mention it here and you can hunt in these forums for how to change your root password on Ubuntu password.

As for leaving DOS on Windows, sorry, I haven't done that. At my office, where I am forced to help Windows users occasionally, I'm still doing commands like:

ping <host>
nbtstat -a <host>
type %windir%\system32\drivers\etc\hosts
type %windir%\system32\drivers\etc\services | find /i smtp
find /i myword c:\myfolder\*.*
ipconfig
telnet <server> <port>
ftp <server>

...and so on.

I encourage this nice dialogue with you, but I think that if you're wanting a defense of why Ubuntu is better than Windows, I'm prepared to give you information that I hope can persuade you and others.

Hope this helps! :)

HD is low-cost, bandwidth isn't. And then there's the security question: Windows programs tend to "static-link", meaning that they come included with all the libraries they need to work. Linux programs mostly use dynamic linking, so you only ever have one copy of a particular library. If you have a Windows program which includes "libfoo1.2", and a patch is released for a security flaw ("libfoo1.2.1"), then your program is still going to be using its vulnerable library.

With Linux programs, you've got one copy of the library, used by all applications that need the library. You update it to libfoo1.2.1, and all your programs are no longer vulnerable.

There are reasons for most of the differences between Windows and Linux.

Note also what I learned at InstallShield Windows Installer Developer class in Schaumberg, IL, USA a few years ago and which is still true today. You see, there's a thing called the Shared File Count in the Windows registry that installers and uninstallers are supposed to update properly when they operate. When this is ignored by developers, and sometimes it is, bad things happen when someone goes to uninstall their application. For example this scenario...

(a) User installs Quaker 15 (I made that up). It uses Xame, a shared DLL used by many new video games (again, made that up too).

(b) The installer for Quaker 15 fails to update the Shared File Count on Xame in the registry.

(c) User installs Bario Brothers At The Races, which also uses Xame and properly updates the Shared File Count in the registry.

(d) The user goes to uninstall Quaker 15 at a later date. It sees the Shared File Count for Xame has only a dependency of 1 instead of 2, so it assumes itself, and it rips Xame off the computer.

(e) Bario Brothers At The Races breaks because Xame is ripped off the hard drive!


Now take for instance the same thing on Ubuntu Linux. Apt handles all of this for us and it never has screwed up my system.
