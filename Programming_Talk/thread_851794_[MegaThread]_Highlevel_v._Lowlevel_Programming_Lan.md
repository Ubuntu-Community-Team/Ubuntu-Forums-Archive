### Post by ceclauson on 2008-07-06
Different programming languages are good for different things.  The choice of what language to choose should probably be dictated by what you're interested in doing, because getting involved in a project that you're excited about is the best way to learn to program.

This is a quick rundown of programming languages with a short blurb on what they do and what they're good for.  FYI, I'm currently a computer science major whose been doing programming as a hobby off and on for a something like five years now (mostly with games), and am currently learning how to program for the web.

Assembly/C/C++ - These languages write native machine programs, programs that are run by the processor directly.  The advantages of these programs is that they offer a high degree of flexibility (communicating directly with the machine and the hardware) and the possibility of very fast, efficient programs.  Typical applications for these languages might be operating systems (Linux is written in C with a little bit of assembly) and computer games (often in C++), where speed is critical.

Java - Isn't run by the machine natively, but is interpreted for it.  This usually translates to a loss of speed and flexibility, in order to communicate with the hardware, or do anything at all, it has to run on top of a "runtime environment" (itself written in C/C++).  The advantage is that any computer that has a runtime environment can run a Java application.  The Java runtime environment can run on any platform, and is free.

.NET (C#, VisualBasic .NET) - Same concept as Java, but a Microsoft product.  The .NET runtime, unlike the Java runtime, is only fully operational on Windows (there's a project called Mono to get it working on other platforms, but its not done yet -- if it ever will be).  Moreover, .NET development tools tend to be expensive.  On the other hand, I personally think that .NET is a more programmer-friendly platform to write for than Java.

The languages above are pretty much the only ones I can speak on with a lot of authority.  JavaScript is added to web pages to create dynamic content that can be run in a user's web browser.  Perl and PHP can be used to create server programs that users can run and interact with remotely from a web browser.  To do this, you'll need to have a web server installed and running (i.e. Apache).  Both Java and .NET languages can be used to run server apps as well, but I don't have a lot of experience with that. EDIT: To make a point about this functional aspect and not to seem like I'm chickening out or anything although I do not believe playing your Euler-game is in any sense meaningful... first sketch:

- Generating all valid colorings is of course a trivial recursion. Stick this into a lazy list so that you can just grab a new coloring at will. (I suspect simple greedy approach is pretty good, though 3-coloring is NP-complete)

- With appropriate triangle representation (this may be an optimization point), create functions that rotate/reflect the coloring.

- for each new valid coloring from your lazy-list, take all its possible reflections/rotations. These are valid and distinct.

- Create a hashcode for each coloring. Numbering colorings is easy, it's a 64 digits in base-three. Stick this into a hashset. You can bypass actually creating representations of rotated/reflected triangles by just reading the triangle structure in different orders and creating these hash numbers.

- now, each time you again pull a coloring from your lazy-list, just check it against the hashset before doing anything. If it's already there, it can be generated through rotations/reflections from something that came before, and its rotations and reflections as well.. transitiveness.

Finally you just count the size of your hashset.

Optimizations include possible interesting properties of the triangle representation wrt. the symmetry operations, and maybe early cut-off in the colorings search in the lazy list...

But, from my limited knowledge, I would recall the usual idea of language as a system to represent the world. A language has to fit somehow to the world it represents (yes, a bit what the "First" Wittgenstein's Tractatus says...), anything outside its scope will be not only out of the language's range, but also pointless to be expressed in that language.

A natural language's object (let's not talk on "constructed languages" like Esperanto, as it is a hard topic to define what is a contructed language really is) is the real world to which it has contact, so the whole system is "designed" to express the world using a particular set of meanings. Considering that meaning is defined with respect to the "difference" between all words/smallest semantic-capable units in a language (meaning is always "relative": "dog" and "cat" both refer to four-legged mammals, so the semantic "weight" is in the difference), it's all reduced to what "differences" the language users can distinguish in the world they're in. This is remarkably shown by how colors are named in different languages (and how difficult is to translate them... that's why we end up using hex values).

OK, let's go to maths. Maths' world is an abstract human-created one by logical methods. But logics (and maths) are formal: you can distinguish a 1 from a 2 not because of the difference in their real objective content (none: We know what a dog is; but, what's a 2?... and more important: who cares?), but the difference in their formal value given (we know how much or what a 2 represents... but in an equation an "x" can also be the same as a two!). Then we have, for example "+", which defines a relationship (again, formal)... so, "1 + 2" has meaning only as an abstraction, that is useful to "simulate" some real phenomena, also expressable in a natural language, like one dog that walks to other two dogs and now we have a group of three dogs. Notice how maths just declare things, you only can describe with maths, but not tell how it should or could be. (e.g. If some said "You two boys, walk to that other guy", you could model that in a mathematical way as "1 + 2", but the English sentence has given the world a new order, not just describe).

So, "computable" is, IMO, just a nice way to say "formally descriptable", and usually, related to numerical values (maths).

But, yes: the problem is that programming is also formal and also relies in numerical values... because the programming "world" (i.e. what a computer can do) is also an abstraction, based on logics. The main difference, to me, is that a computer is also an existing object, so it also is part of the world a natural language can express. That's why we can give an order to a computer through C's fopen(), no matter if the internal way to open a chunk of 1's and 0's in a hard disk is based on Magentic Fields Theory and then, binary bits processing. For the human-computer "interface" (as we called it in that Linguistics thread some time ago), it was an imperative statement semantically the same to a "Make me a sandwich".

This hybrid nature is the reason I think why you can't design a whole program just using maths on a paper: yes, you can express the algorithm **using maths**, but there always be some cheating by using sentences  in a human language: Let x = 0... That's not math. It's English/<your preferred natural language> with math notation: "Let 'x' be '0'" (and not something else like 'hello!', "difference" arrives again!). But then x + z = y[i] is maths as it describes a numerical relationship). "Let x = 0" is not computable, (another proof: that statement always yields a boolean true value... by definition, not because any logical deduction), but it does makes sense in a program (a lot of sense).

### Post by worksofcraft on 2010-09-18
If you just want to run an algorithm to find a solution then you are often advised to use a high level language like Python and not a low level language like C++. Here is one I prepared earlier in both of those languages:
```

#python
#!/usr/bin/python

class shifter:
	def __init__(this):
		this.Shifter = 1
		this.getbits()

	def getbits(this):
		this.bit1 = 1 << 20
		this.bit2 = 1 << 21
		# use up to and including the most significant bit
		this.Mask = this.bit1
		if (this.bit1 < this.bit2):
			this.Mask = this.bit2
		this.Mask = (this.Mask << 1) - 1

	def xorbits(this):
		return bool(this.Shifter & this.bit1) is not bool(this.Shifter & this.bit2)

	def shift(this):
		nextbit = this.xorbits()
		this.Shifter <<= 1
		if (nextbit):
			this.Shifter += 1
		# finish when cycle starts repeating
		return (this.Shifter & this.Mask) > 1
		
Counter = 0 # count iterations
S = shifter() # create a shift register
# Proving that the cycle will always return to it's start point
# is left as an exercise
while S.shift():
	Counter += 1

print 'pseudo random sequence was', Counter, 'iterations.'

```
v.s.
```

//C++
#include <iostream>
using namespace std;

struct shifter {
	unsigned Shifter;
	unsigned bit1, bit2;
	unsigned Mask;

	shifter() {
		Shifter = 1;
		getbits();
	}

	void getbits() {
		bit1 = 1 << 20;
		bit2 = 1 << 21;
		Mask = bit1;
		if (bit1 < bit2) Mask = bit2;
		Mask = (Mask << 1) - 1;
	}

	inline bool xorbits() {
		return ((Shifter & bit1) != 0) != ((Shifter & bit2) != 0);
	}

	bool shift() {
		bool nextbit = xorbits();
		Shifter <<= 1;
		Shifter += nextbit;
		// finish when cycle starts repeating
		return (Shifter & Mask) > 1;
	}
};

int main() {
	unsigned Counter = 0; //count iterations
	shifter S; // create a shift register
	// Proving that the cycle will always return to it's start point
	// is left as an exercise
	while (S.shift()) ++Counter;
	cout << "pseudo random sequence was " << Counter << " iterations." << endl;
	return 0xDEAD;
}

```
