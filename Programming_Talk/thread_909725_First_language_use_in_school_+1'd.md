### Post by nvteighen on 2008-09-06
I see that here, under the hood, the discussion seems to be what programming is: To tell a computer do something or to describe processes using computers as a tool?

If you think programming is "to tell a computer to do something", then, clearly your "real stuff" will be at C level: as the goal is the computer and everyone wants as more control to achieve his/her goals, you'll need pointers, manual memory management, etc, because you'll have to learn to "tame the beast" first and then, after you know what the computer is about, go ahead and use high-level languages.

In the other hand, if you think programming is to describe processes, you'll want a language that let's you express those processes in the best way, regardless what tool you use to simulate the process. For example, you could write a cake recipe in Python or Lisp, but in C? If you wanted to do that in C, you'd have to use pointers (which are nonsense in the non-computer world) if you need a dynamic list of elements just because you have to allocate memory in the heap ("allocate", "memory", "heap"... all concepts that are nonsense in the real world).

OK, I believe programming is the second, not the first, because of the way programming languages work, as languages. Programming languages serve for human communication, not for human-machine: the machine only understands machine code. Why do we write webpages in HTML, even if at the end they get translated to machine code? Or why do you use C instead of Assembly? Or why don't we code just in machine code, anyway? Because, intuitively, you want to understand what **you** are doing, not the machine. Do you care if the processor is a Motorola 68k or an Intel x86 to describe a prime numbers generator?

(This is all related to the fact that humans work with symbolic codes to gain the most possible abstraction from the objects. The most "high-level languages" are, of course, the natural languages as they can not only describe but also modify the reality or refer to fictions, because they work not with objects but with the concepts of given objects already abstracted in our minds)

Of course, this doesn't mean that low-level programming should disappear. No: as you use a tool, you have to also understand how the tool works, and low-level languages are great when you want to describe a process involving elements on a computer, but are useless when you're describing a process that could be executed from elsewhere and you're just using a computer to automate it.
