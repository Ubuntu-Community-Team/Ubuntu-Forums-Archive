### Post by Simian Man on 2011-12-14
All major programming languages are "Turing complete" which means that ANY problem that can be solved with one language can be solved with all of the others too.  So it's not a question of what can one language do that others can't but of how easy languages make certain things.

You do have to set your sights much lower.  Any one of the programs you listed would take hundreds of man-years to complete.  The choice of programming language is the least of your worries.

I recommend picking one language to learn (I'd recommend Python, but it doesn't really matter that much), one small goal and accomplish it.  After that, you'll be in a much better position to judge what you want to do going forward.

### Post by CptPicard on 2011-12-15
> **kurotensai said:**
> Look man, no disrespect, but these r my goals. Can i achieve them  with Lisp or not? Can lisp do everything c++ c and asm can or not?

Well, if you're looking for a clear cut yes/no answer, it's certainly a "no", and this coming from a guy who loves Lisp. Happy? :)

Of course, the basic fundamental problem is once again this idea that "C can do anything" (in the sense that it compiles down to native and thus there are no technical obstacles to building "anything" *directly* on top of some architecture) and that then you'd have to compare HLLs to the same standard to see "what they can do" -- and that this is the gold standard to strive to in programming language choice.

The thing with Lisp is that it's a bit like what Latin is for linguists -- they might not use it on a daily basis, but they like it for what it is and the ideas it contains. Sometimes elegant simplicity is remarkably powerful in the abstract ideas sense, and that does not have to have anything to do with some particular (hardware) implementation you have in mind. It's just that once you've seen the idea in the first place, you'll be able to emulate that idea in other languages using the tools you're given.


> 
 I only wanna learn Java n objective c in the future so i can have my programs runnin ob mobile phones n macs, 4 the rest i only wanna have to use one programming language.

Interestingly, there is this claim that any system that is sufficiently complex ends up containing some kind of Lisp because that is the only way to allow for the needed runtime genericity...

> **gardnan said:**
> Well, to be completely clear, you could not write an operating system in Lisp, as it generally requires either an interpreter or a target language, and it would be difficult to the point of infeasibility to write an intensive 3D  game in Lisp, so I would say that, no Lisp cannot achieve your goals.

Well, check out SBCL, the best CL implementation around these days. It's a native-code compiler/runtime that is quite capable of giving the programmer much choice in how much he wants to leave to the runtime library and how much is compiled into the actual binary of the function.

It is quite conceivable, although it would be hard to do, to imagine writing an SBCL image that runs directly on hardware, providing "operating system services" straight from the REPL. It would be a kind of modern-day Lisp machine in software...

Also, just grab your OpenGL bindings and start hacking away at the 3D game. There is nothing that stops you from even calling malloc directly from Lisp if you need that kind of memory management control. Certainly you'll need to pay some attention to not getting unexpected GC stutter, but IMO it would be remarkably cool to try this, as SBCL allows you to so seamlessly transition between almost C-like low-level code to even run-time-interpreted "scripting" Lisp for your high-level game mechanics.

As a linguist, I endorse this view. Just that I hate Latin and prefer to use Ancient Greek for that matter, but the argument stays the same ;)

Now, onto the OPs questions:

Be wise. Learn lots of languages. I have the intuition that you're young (that's what I get from analyzing your written language... you know, linguists do that stuff :D), so don't close your mind down. Learn Lisp, but also Python, Perl, C, C++, something as arcane as Forth, ASM, etc. and experiment with things. It's true that for writing an OS, you'd be primarily using low-level languages and fighting your way through BIOS interrupts, but in order to do that and, most importantly, grasp the real difficulty of the project you have in mind (you always have to be realistic about whatever project you embark on, not for demotivating yourself... actually, understanding in advance how hard something might become helps you avoid being demotivated at the first obstacle!). For that, you need experience and the more and more different experiences you have in a subject like programming, the better.

A bunch of folk here usually suggest starting with some high-level programming language because they have semantics that work much more like natural languages, being more or almost completely decoupled from the underlying platform you're working on. We usually recommend Python for a couple of reasons, but there are these "minimalist" Lisp dialects which might be of your interest: Scheme and Clojure. I'd rather recommend Clojure, because Scheme is become a huge mess of incompatible implementations, while Clojure is a nice Lisp implemented on top of the JVM. Learning one of both makes it quite easy to learn Common Lisp, which is way more "practical" but way more complex.

However, as I said one should always seek to broaden oneself's experiences. No serious competent programmer knows just one language nor languages of just one kind. This is like the linguists who are constantly seeking to learn, even if just bits, about different languages in order to understand language in general.

### Post by Vaphell on 2011-12-22
from wiki references
[http://bc.tech.coop/blog/060118.html](http://bc.tech.coop/blog/060118.html)
> "In all honesty, the biggest reason we're not using GOAL for next-gen development is because we're now part of Sony. I can only imagine Sony's shock when they purchased Naughty Dog a few years back, hoping to be able to leverage some of our technology across other Sony studios, and then realized that there was no way anyone else would be able to use any of our codebase. :)

Sony wants us to be able to share code with other studios, and this works both ways - both other studios using our code and vice versa. Add this to the difficulty curve of learning a new language for new hires, lack of support from external development tools (we had our own compiler, linker, and debugger, and pretty much had to use Emacs as our IDE), etc, means that there are clearly a lot of other factors involved. Note, however, that these issues aren't really technical problems, they're social ones.

I can definitively say that the investment in GOAL was worth it for our PS2 titles, despite the initial setup time and maintenance. Our productivity gains were huge, and were more than worth the time investment. This time around, however, the circumstances aren't quite the same. If we were still an independent studio, I'm almost positive we'd be extending GOAL for the next-generation of development. As it is, we are looking into alternative approaches (custom preprocessors, assemblers, linkers, you name it) - but all of these approaches fall short in many ways of the unified language and environment we had with GOAL.

That said, if there was a serious effort on the part of the game development community to develop and standardize a language for game development, everyone could gain the benefits without suffering the drawbacks (lack of code-sharing, learning curve, etc). And if there's enough community support, it would only be a matter of time before some really high-quality commercial tools came out to work with the language."
