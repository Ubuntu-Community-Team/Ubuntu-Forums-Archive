### Post by SledgeHammer_999 on 2009-08-11
[php]#include <iostream>

class my_class
{
  public:
  int test();
};

int my_class::test()
{
  int num;
  num = 5;
  return num;
}

int main(int argc, char *argv[])
{
  my_class* inst = NULL;
  std::cout << inst->test() << "\n";
  return 0;
}[/php]

I thought that I had to make a new instance of the class with the "new" keyword. Furthermore the pointer is NULL and points nowhere. Why doesn't this crash but instead it prints out "5"? Are my previous assumptions wrong?(obviously yes)

### Post by dwhitney67 on 2009-08-11
It does indeed look odd.  The program probably works because you are returning a hard-constant that sits in memory.

Try altering your class definition/implementation as follows, and surely you will get a segfault:
```

class my_class
{
public:
   my_class() : value(5) {}
   int test() const { return value; }
private:
   int value;
};

int main(int argc, char** argv)
{
   my_class* inst = 0;
   std::cout << inst->test() << std::endl;
}

```

### Post by Habbit on 2009-08-11
The delicate point is that you are seemingly dereferencing a NULL pointer, but, with most compilers, the particular code you wrote actually doesn't. Let me explain: in most C++ compilers, the code generated by your program would be similar to this: ```
#include <iostream>

struct my_class { };

int __my_class__testV(struct my_class * const this)
{
  return 5;
}

int main(int argc, char *argv[])
{
  struct my_class * inst = NULL;
  std::cout << __my_class__testV(inst) << "\n";
  return 0;
}
```
As you can see, the fact that the selected class method is being resolved statically, and that the method itself does not use the "this" pointer,  allows you to call a method on a NULL instance (!). However, this could have failed either: [LIST]
[*]On some debug compilation, where one could instruct some compilers to perform NULL checks
[*]On a different version of the compiler, maybe, if it tries to access some info from the instance pointer
[*]In particular, when invoking virtual methods, because the compiler will insert code to check the runtime type of "inst" and fetch the appropriate v-table
[/LIST]
