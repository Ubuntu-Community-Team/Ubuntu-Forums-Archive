### Post by r-senior on 2013-06-06
> **cwblanch said:**
> I have something I've been putting off, re-organizing my music, but that seems like a big project.
Big projects have small components. For example, you could start off with a little Python script that just extracts the ID tags from a single MP3 file. There is lots to explore and learn -- opening files, splitting strings, adding command line arguments.

Then you can expand it to work with a directory -- so you get to play with reading through files in a directory, checking the directory exists, etc.

Then more features -- create directories based on artist or album, move files, etc. Error handling.

Perhaps move on to a GUI? Find out how to get album art from one of the web services. Or take your basic MP3 info script and see how to make a Nautilus extension that pops up a little window?

So it's lots of non-trivial but achievable steps where you learn something new each time and have some "product" that keeps you motivated. This is why "Hello World" programs exist -- small steps to get something very simple working before you move on. It's difficult to keep motivated if you just see a big problem ahead of you.

Cheesehead mentioned Dive into Python and there is an example of extracting the ID tags in there:

[http://www.diveintopython.net/object_oriented_framework/index.html](http://www.diveintopython.net/object_oriented_framework/index.html)

But don't just take that as a recipe. Play with it. Look at the MP3 file specification and see why it works. Try and do it differently, show other tags, etc. At points on this journey you'll want to stop and take stock of what you have. Perhaps rewrite parts and discard others. This is how you learn about programming, reusability and structure. Learn how to use version control to make feature branches that you can throw away. Look at automated tests, run your program through pylint. Create modules. Learn how to package it into a .deb.

It doesn't even matter if you get bored of it and have a better idea. The things you learn stay with you and it's the accumulation of those things that makes the next project easier.

### Post by r-senior on 2013-06-07
The main concerns about an IDE for a beginner are that:

1. The IDE hides the process of putting projects together so the beginner becomes dependent on it
2. Code completion and templates prevent the structure and syntax of the language sinking in
3. Over-emphasis of debuggers, i.e. relying on a debugger to see what a program is doing

Just make sure that you can write Python using the lowest common denominator of a text editor and terminal.
