### Post by tribaal on 2006-10-31
Hum... pretty wide question here!

Maybe the first thing to do is simply to look for a simple open source project you like as a user, but that does not behave exactly the way you want. From there on, just download the source, get a good reference manual for whatever language it's written in, and get your hands dirty. Play with it. 
Of course, if you need to learn a new language, it's possible that by the time your fix is ready, a few versions of the app will have rolled by, and maybe an "official" dev will have fixed your problem already. But don't be discouraged, you'll have learned valuable skills and a good understanding of the application's internals! It's not wasted at all!

Then either think of something else to fix and hack it, or find the project's bugtracker, and start working on a bug. 

It will take a little time to actually submit a patch to the main dev tree (that gets accepted and published), but it's the only way to go...

I cannot stress enough that the application choice is really important, because you'll have to spend quite some time just analysing how it works before you change it. It's a necessary step, even for star programmers.
As a rule of thumb, when hiring a professional programmer in a commercial company, the manager usually considers the new hire as "adapting to the project" for a period that can be between one month to maybe six months...
It's pretty much the same for open-source projects. Just don't despair!

Another thing to remember is that open source is usually very open to questions. Don't hesitate to ask how a particular piece of code works to other devloppers, as long as you respect etiquette and usages (like forum lines of conduct) :)
Some projects have a "mentoring" structure set up, where an older coder "coaches" a new coder, too.

Hope I didn't discourage or scare you with my lenghty post :)

Have a lot of fun ;)

- trib'
