Easiest way is to find something you don't like, and fix it, then submit your patch to that project's development team (they usually have a website dedicated to such things). If the developers like your patch, they might add it to the main development trunk. If not, and they tell you why they don't want it (be prepared for criticism!), then you can go back, fix your mistakes, and re-submit, or just find a different project.

Alternatively you can just create your own project from scratch (not completely from scratch, mind - chances are there are some libraries and toolkits available for you to use), get a demonstration available for people to try out, and announce it somewhere with a call for others to get involved :)

### Post by Auria on 2007-08-18
The best place to start is to actually pick a few projects you like. Start by downlaoding the source code, building it, learning everythnig about it. Hang around its mailing list/forum/irc channel some time to learn how the devs work. Then look a bit at the source code. Try to understand how it's put toghether, try writing small changes. Learn their coding conventions. Follow patches posted on the bug tracker and try understand what they do. If you have questions youcan't figure out, ask them, often on irc - just don't be annoying and do research by yourself. When you understand the code enough to not be compeltely lost, look at the bug tracker, pick a bug that does not seem too hard, write a patch, submit it to the bug tracker.

voila :)

hope it helps - just my view of it, someone else will say something different (you'll soon discover entering an existing project witha large codebase and multiple programmers is very different from programming your own small projects)

### Post by pmasiar on 2007-08-18
First: **don't start new project!** - try to "learn the ropes" how projects work by helping some existing project. 

Now same thing but slower:
It is good to know that every project is eager to enroll new smart developers. Keyword is "smart" - you need to prove that time teaching you is not wasted. Basically, you need to be able to self-learn from existing documentation and by asking **smart** questions - ie. not questions which can be answered by trivial google search, and showing that you are listening to answers and learning from them.

OK, now this might look like tall order for you right now, because your lack of skills. How to learn programming, if I don;t know what to program? Fear not, we have solution! Good folks at [http://www.pythonchallenge.com/](http://www.pythonchallenge.com/) created series of puzzles (30+) levels, which you can solve and learn on the way, with forums for every question where you can ask (they will hint but not solve it).

Before or after that, you can go to wiki in my sig and check training tasks I collected from all over, good for learners. If it looks like doing your homework - well yes it is.

Learning programming is very like learning karate: [do your kata exercise](http://www.codekata.com/2007/01/kata_kumite_koa.html) before you are ready for kumite. :-)

Oh and I almost forgot (everybody in forum is waiting for my shameless Python promotion :-) ): in opinion of many (but not all), Python is the best first language to learn. So try it, wiki in my sig has many good links.
