### Post by ssam on 2012-06-19
See if you can fix a bug that effects you in a program you use. Dont be intimidated by huge projects, you dont need to understand the whole thing before making any change. most projects are far more likely to accept a bug fix, than a new feature from a new developer.

Some projects have a some information for new developers, and/or a collection of bugs that would be easy for a new dev to fix.
[https://live.gnome.org/GnomeLove](https://live.gnome.org/GnomeLove)
[http://www.libreoffice.org/developers-2/](http://www.libreoffice.org/developers-2/)
[http://kernelnewbies.org/](http://kernelnewbies.org/)

The script ack-grep is really handy for finding things in a large code base (it uses grep to do a recursive search, but knows to ignore binary files, and version contol system files).

### Post by fallenshadow on 2012-06-22
Start learning by programming with some small projects to get some good experience!

Its up to you as to what you want to work on.

I did things the opposite way around and I started by launching my own big project and took over development/leadership of Xwii for a few years too!

The problem with that approach is that progress on my own project is slow because I didn't spend time just learning with small useless projects for a while. So in the end its probably better starting small then going big when you feel ready.

By going big I mean starting a big project or maybe joining a big well established Open Source project.
