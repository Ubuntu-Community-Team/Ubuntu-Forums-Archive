### Post by vambo on 2007-08-12
Since it's your programme, I'd forget about the core. As you've compiled for debug, run the programme with gdb. 

```
gbd <myprog>
``` 

from there you can used gdb to run (step, next, etc) to get to the point(s) of the crashes. I'm sure it will be far quicker than deciphering a core - that's a last resort IMHO.

### Post by Jessehk on 2007-08-12
My guess would be that you're *free()*'ing memory that either has not been *malloc()*'ed, or a NULL pointer.

Do what vambo suggested and run through it with gdb. :)

### Post by runningwithscissors on 2007-08-13
Since the backtrace is entending into libc, it is likely that you're passing some illegal value to a library function.

Just double check your code for that, or rebuild the libraries you're using with debug support, so that you can step through the library code to pinpoint the exact cause of failure.

### Post by jodelson on 2007-08-13
Thank you all for the help.
I can say I am now one more real case of someone who solve a problem with the help of this forum, so I came back here to post the final solution.

1- Following the clue by **vambo **I paciently started running the program with GDB step by step to try to find the line of code with the error. Took some hours with no result as the program is quite huge (over 3000 lines of code), error only happend when processing a huge instance of input data and at random phases in the algorithm as it is based in some probabilistic decisions;
2- with the clues by **jessehk **and **runningwithscissors **I went back to trying to read the core dump and after the clue by **fct **I got to the final spot:
2.1- found the libc6-dbg debian package and installed it. In the description of this package they gave the clue to set LD_LIBRARY_PATH environment variable, so that I could explore the facilities of the unstripped shared library to backtrace with names in a debugger;
2.2- I run the program once again, got the core, entered gdb, got the backtrace and the line in the code was spotted precisely !
**Jessehk **was write. It was a free() command ! I should have done a search in the code looking for the function free() as I did not remember anymore this old free() statement left coded for months. This is the reason why when I read the **Jessehk **clue I thought I had no free() function call in my code anymore as I stripped all them (except that one) in a certain day in the past when I came across the fact that a function end automatically free all memory allocated.

THANK YOU VERY MUCH!
