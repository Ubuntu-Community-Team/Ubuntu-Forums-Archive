I don't know much about Python (never really tried it) but it seems to be quite an easy language, that will allow one to concentrate on the real problems: architecture & algorithms. This is my best bet for many wannabe programmers out there.

For someone determined, I'd recommend C++ though. Granted it's very hard. IMHO this comes from the fact that it mixes both low & high level concepts, which are *both* important to master for any serious programmer. If you don't mind learning and not getting immediate gratifying results, then go on with C++. Expect to have hard times though.


My final advice would be:

1) learn the mid level concepts (decisions, loops, procedural programming, some OOP is nice too). Python seems perfect for that.

2) temporarily forget what you just learn, go for assembly. It will teach you how the machine really works.

3) learn basic C++, it will allow you to mix (1) and (2) and hopefully you'll learn a bit more.

4) learn high level stuff (you should master GoF patterns). This is pretty language-agnostic.

5) stick to C++, learn state of the art OOP, and meta programming.

Once you get to (5), being an average programmer with language X (not expert, mind you) should take no more than the week that is needed to understand the language's syntax.

### Post by bigboy_pdb on 2007-08-17
Programming is neither a democracy nor a meritocracy because it is not an organization and it is not a system of government (it is the process of writing computer code).

When one guru says one thing, another might say another. Every language has its advantages and its disadvantages. What I've found is that scripting languages often have a simpler syntax and more higher level constructs built into them. So when Bruce Eckel states that Python allows you to get more done this isn't surprising granted that scripting languages are meant to allow you to code faster. Lower level languages allow the programmer to have more control over their environment and are designed to have less abstraction (for example in C and C++ the programmer controls memory management, but a number of other higher level languages have garbage collection).

The opinions of all programmers matter. Someone who's a guru and has been programming for a long time might not remember how difficult it was for him/her to learn to program. Also, some experts/gurus learned to program using languages that are no longer commonly used. Learning a language that is more common means that there will be more support, tutorials, books, and people who know the language that are available to help a person.

I agree that more information is necessary than just what language you recommend and why you think it should be used. In order for a thread like this to be useful it is important that people mention their programming background along with some suggestions. So I guess that I'll start.

I have a University degree in Computer Science. I know how to program/script using VB, Java, C, C++, Prolog, Scheme, SML, batch, JScript, VBScript, PERL, and bash (and I also know ASP, ASP.NET, and a little PHP). I first learned how to program using Turing and VB back in high school.

The programming concepts that I think a person should learn about are:
variables, conditional statements, loops, functions, recursion, objects/classes, reading from and writing to files, error handling, memory management, manipulating numbers, manipulating strings, regular expressions, and threads.

Learning a programming language isn't enough to write good programs though. A person should also learn about:
Run time complexity, proving that a program terminates, proving that a program is correct, proofs in general (of a logical, mathematical, or computer science related nature), algorithms, design patterns, debugging strategies, logic, and mathematics (subjects in algebra and discrete mathematics are closely related to computers).

With respect to learning a language, I think that it's a good idea to learn both a high level language and a low level language. C and C++ are good low level languages that should be easy enough to find information on, and Java is a high level language that has a good amount of available information and tutorials on it (I've never used Python). Learning to script using bash is a good start for someone with little background. Once you learn how to program using one language it should be easier to move to another language because most popular languages have common elements (such as conditional statements, loops, error handling and so forth).

### Post by samjh on 2007-08-17
No offence, but this poll is meaningless.

The key to choosing the first programming language is what the learner wishes to accomplish at the end of it.

My own experience is that those who learn very high level languages first without exposure to lower level languages, tend to be able to write code quickly, but design their software poorly; while those who learn lower level languages first tend to be slower at producing code, but design their software better (ie. more maintainable, extendible, etc).

The best thing IMO, is to learn at least one very high-level language and one lower-level language.  It doesn't matter which one first, as long as you get enough exposure to both.  Nice combinations are C++ and Java, C and Python, Java and Ruby, and so on.

### Post by CptPicard on 2009-02-16
This is the interesting "importance of low-level concepts in high-level programming" question, the side of which I take is probably well known.. :)

Now, disclaimer -- I do think someone should learn C at some point to become a well-rounded hacker. I mean, I too started "real programming (tm)" with doing 3D rotated cubes in C with inlined asm when I was in our equivalent of high school, so I've had the stack/heap stuff etc at the back of the head for such a long time that it would really be interesting to get to see how it would all look like if I started from scratch using a different methodology...

The "what does it actually do" question is valid for those so inclined, but IMO in learning and understanding formal language definitions it is important to just understand the language as defined and then reason with it from there. In mathematics for example it is a good skill to be able to just work with the theory you're given, instead of questioning its validity from first principles the first moment you meet it. Learning will be slow going like that.

The reason why I think you got nowhere with Java is not that "how to implement Java" is not taught, but the problem is rather that the static-typed OOP model as used in Java is just simply not all that great as an abstraction especially for learning. I find "having to think of everything as objects/interfaces" seems to just mostly trip up beginners in the early design phase of the program... it sure took me a long time to learn the esoteric art of object-oriented design before I could actually build anything in Java. A lot of the architecture patterns you use are such that you just have to pick them up somewhere -- they are unnatural enough that you would have a hard time learning them on your own.

As soon as you understand the meaning of a type system and the type substitutability ideas of polymorphism, and the thing that there are "objects" that are grouped to "classes" which obey the rules of the type system, you're good to go, and IMO implementation plays little role in that.

Then there are languages such as functional languages and their associated design issues where being too intellectually married to the machine-view of things is just actively harmful. I really needed to unlearn trying to see it as "an implementation" and all that, and just "read the manual" about symbols, environments, apply and eval... once I got how things really are defined, things made sense. And when you're able to think more independently "from the stack", things like closures and continuations make more sense... (and the latter are actually theoretically speaking a very broad "computation state" concept)...

### Post by JordyD on 2009-07-10
Whether a language is worth it depends on what you want to do, the past two posts (I think it was two, I didn't count, it was the ostrich head and purple background man anyways) summed it up nicely, but I figure I'll add my two cents:

**Assembly**: pretty much as low as you can go. It's much harder but gives you most control
**C**: very low-level language. Gives you a lot of control. Not quite as much as Assembly, but it's much easier.
**C++**: C with classes pretty much sums it up. It can go higher-level than C, but is still very low-level. CPU-intensive games are often written in this or mostly in this because it's close to C in speed and close to C# and Java in OOP support, although not quite at either level.
**C# & Java**: high-level languages. Good for big GUI apps where speed isn't important. For example: OpenOffice made a poor (IMO) choice for programming language. It's written in Java, and Java is slow. It should have been written in C++, which can still go pretty high-level.
**Python**: Good for things you might use Java for, but even higher-level and much easier. It's used for one of my favorite text editors: [Scribes]("http://scribes.sourceforge.net/"). This was in my opinion a good choice because Scribes does not need much speed, but it allowed rapid development of it, since Python is much easier to develop with.

That's pretty much the spectrum as I see it. There are of course other languages, but they mostly fit into those categories. At the top is faster, lower-level, and harder. At the bottom is slower, higher-level, and easier.

Often times beginners start with Python or Java and make their way down to C++. I don't know many that go down as far as Assembly unless they're programming for hardware. You would not want to program, say, Firefox in Assembly (the *pain*...).

### Post by QIII on 2009-07-11
I started programming in the days when we fed a pile of punch cards into a machine connected via a telephone handset in an acoustical coupling device to a University mainframe somewhere, when available "memory" was measured in bytes and processor frequencies were measured in Hertz.  The modern concept of "concise code" would have been an unmanageable monstrosity then.

I would say that "programming" is largely language agnostic - with a glaring generalized exception.  You can scrape the metal with assembly, or you can avoid getting even close to a computer with Microshaft's .NET "technology".  The latter, of course, is the glaring exception to my "language agnostic" rule.  .NET may make it easy to pump out software like stamped metal car parts, but it has probably done irreparable damage to the programmer's understanding of his craft by completely insulating him from any real interaction with the computer itself. 

What is and will always be more important than any specific language is a combination of the following (and this list is in no way exhaustive):

1.  Understanding, at a very low level, what it is that a computer really does.
2.  Understanding, at a low level, what it is that the computer needs to be instructed to do.
3.  Understanding how to identify a real world (or theoretical) problem.
4.  Understanding how to identify the desired end resolution of the problem.
5   Understanding how to bridge the gap between the problem and the resolution.
6.  Understanding common data structures, including primitive data types, abstract data types, arrays, lists, linked lists, stacks, heaps, queues, hashes, tables, trees, sets etc.
7.  Understanding a broad set of algorithms necessary for 5 using the items in 6.
8.  Combining the above into a generally workable solution that can be tested, rearranged, tested again and rearranged through enough iterations to be sure that it is sound.

All of this can be learned in most languages -- with the notable exception of .NET, which abstracts everything to the point that it is unrecognizable as real programming.  I don't mean to offend.  I am just calling it as I see it.  I make an obscene amount of money using Microshaft "technology".  Eating a t*rd is not so bad when it is washed down with a fist full of green.  I admit that I am a mercenary for the Evil Empire during the day.  But I still do actual programming on the side.

"Hard" and "easy" to learn are relative.  A "hard to learn" language may be hard because it forces one to deal with the necessary details rather than leaving them to the tender ministrations of an intermediary compiler that pats you on the head and says it will take care of everything.  But that doesn't mean that the "hard stuff" can't be learned in an "easy" language.  It is a matter of the manner in which we develop new blood.  If we let them be sloppy and ignorant, they will be sloppy and ignorant.  

Java, for instance, might be seen as a softer and fluffier little brother of C++, but the same elements of genuine understanding can be found there.  Whether it is fast or slow is not a good measure of its relative value as a programming language.  It is slower than C++ because it runs in the JVM.  The benefit of that is that Sun paid the price to make the JVM work across platforms so that Java applications could find a home wherever the JVM resides.  Compare that to the days when we had to code and compile C against specific hardware and platforms.

A better question than "what language" may be "which concepts" would be the first to learn.

Just my (rather tardy) two cents.
