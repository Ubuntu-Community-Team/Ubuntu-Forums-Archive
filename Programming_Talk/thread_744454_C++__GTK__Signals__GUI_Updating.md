Don't fork a process to update the GUI, instead use threads and note that gtkmm is NOT thread safe, which means 
that synchronization problems "like race conditions and deadlocks" will occur if multiple threads work on the same job, like updating GUI.

so you can try to synchronize your threads with the GUI thread, good luck with that ! :)

or, make sure that only one thread of execution handles GUI updates, and other threads "worker threads" dispatch jobs to the GUI thread, this is the best way to handle multi-threaded GUI apps "as far as i know".

> 
#include <sigc++/sigc++.h>
#include <sigc++/signal.h>

> 
g++ -c myfunction.cpp `pkg-config --cflags sigc++-2.0`

(Those tick marks are *single back-quotes*.)

### Post by heikaman on 2008-04-03
ok, this is how to use threads and dispatchers in gtkmm

```

#include <glibmm/thread.h>

using Glib::Thread;  
using Glib::Dispatcher; 

Thread* worker_thread; // a thread
Dispatcher dispatcher; //for the worker thread to dispatch jobs to the GUI thread.

//somewhere in your initialization code you specify the function that gets called when the worker thread dispatches a job
dispatcher.connect(sigc::mem_fun(*this, &my_class::on_dispatch)); 
//you start the worker thread to do the processing
worker_thread=Thread::create(sigc::mem_fun(*this, &my_class::thread_function), true);



void thread_function(){
	//do the processing until there's no more work to do
	while(jobs){
		//processing
		//finished processing tell the GUI thread to update the GUI
		dispatcher.emit(); //this call doesn't block, and you can pass args i think, don't remember
	}
}


void on_dispatch(){
	// the GUI thread updates the GUI.
}
```

that's all. :)
oh.. and read this FAQ [http://www.gtkmm.org/docs/gtkmm-2.4/docs/FAQ/html/index.html#id2552475](http://www.gtkmm.org/docs/gtkmm-2.4/docs/FAQ/html/index.html#id2552475)
