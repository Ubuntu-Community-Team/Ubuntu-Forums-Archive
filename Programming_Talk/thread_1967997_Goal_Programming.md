### Post by 11jmb on 2012-04-30
> **DaviesX said:**
> I just gave up learning that, I feel really comfortable to write c for instance. And I intend to learn assembly for X86 too, Can you please give any suggestion about it ?

I must ask what your reason is for wanting to learn assembly. 99 times out of 100, a good C compiler will produce as good (or sometimes better) assembly code than most engineers could write.

If your interest is purely academic, you might have a better learning experience with a RISC. I believe that most universities prefer to teach RISCs (Once upon a time, my architectures class was taught with MIPS) because you can cover all of the same concepts in a simpler context.

also, +1 lisati.

try compiling some basic programs with:
```

gcc -S myfile.c -o myfile.s

```

and looking at the assembly. you can assemble/link myfile.s in the same way that you would link an object file

```

gcc myfile.s main.c -o a.out

```

### Post by 11jmb on 2012-04-30
> **CptPicard said:**
> the "which language to start with", tends to deal with expressing computable things in general given some language

This is, in general, language-agnostic. Sure, some languages make it easier to express computability than others, and some languages certainly have more expressive power than others, but "thinking like a programmer" comes with time and experience, rather than use of a specific language.

On the topic of what language beginners should learn first, I think it is better to think in terms of domain. The sooner you can sink your teeth into problems that you find interesting, the better IMHO. DaviesX expressed curiosity in low level programming, so C->asm is an excellent starting path. If a beginner expressed interest in desktop apps, as another example, Python is an outstanding recommendation. I also agree that if a beginner has an interest in programming in general, Python is the perfect starting point as well.

### Post by CptPicard on 2012-04-30
> **11jmb said:**
> This is, in general, language-agnostic. Sure, some languages make it easier to express computability than others, and some languages certainly have more expressive power than others, but "thinking like a programmer" comes with time and experience, rather than use of a specific language.

Well, just consider Malbolge vs. C; I guess we could agree that we can't say it's language-agnostic when considering which language communicates "general" programming-related ideas better, even though both are Turing-complete just fine (well ok, at least Malbolge Unshackled provably is :) ).

So, programming language really carries ideas in it, and we can say that some languages do this better than others. Personally I've always felt that if I had to pick one language that best represents anything and everything in programming with the least amount of extra effort to get to any constructs/ideas in the space of all programming "stuff", it would be Lisp -- after all, it's a very minimal, abstract syntax tree of "any" language, with just the constructs you absolutely must have to build on top of. And you can build on top of those things in a very intellectually instructive ways.

### Post by 11jmb on 2012-04-30
I'm suggesting that a beginner learn with the best tool for a specific job, Malbolge was designed to be the worst tool for every job :)

I certainly understand your point, that some languages are better at describing high-level concepts than others. As an easy example, look at the hoops you have to jump through in Java or C# with interfaces and delegates in order to emulate first-class functions, a core concept in many higher-level languages. In fact, 16 of the 23 design patterns so beloved to Java programmers were found by Norvig to be "invisible or simpler" in lisp.

I am also in agreement with you (I think, you haven't said it explicitly) that no language will expand your horizons like Lisp.

Where I disagree: I think that a beginner ought to be applying his/her newfound programming knowledge to a domain of interest as soon as possible. Solving problems is what makes programming fun :) That's why I don't believe in a static answer to "what language should I learn?"
