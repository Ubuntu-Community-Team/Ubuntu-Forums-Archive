### Post by CptPicard on 2009-02-09
If you are nearly a programming n00b I would have suggested that you look into a less rigid language such as Python. The problem with Java is that before you can do much, you need to actually have quite a good grasp of how to model things in a static-typed OOP language that actually forces all functionality to be a member method.

And in order to learn all this design pattern stuff, you need to code first. So you end up in a nasty chicken and egg problem in your learning curve (this was probably one of my biggest stumbling blocks in my own learning over 10 years ago).

Anyway, at least use a proper IDE, it holds your hand nicely and speeds up API discovery with its hints... there is a lot of typing in Java without one.

### Post by michael_1234 on 2009-02-09
You're in a very large pool of people, plenty of folks with the same questions as you,plenty with answers to those questions.

Books feel good in the hand, but there are plenty of on-line starting points, as well.

[LIST]
[*][http://www.google.com/search?q=the%20java%20tutorial](http://www.google.com/search?q=the%20java%20tutorial)
[*][http://www.google.com/search?hl=en&q=new+to+java&btnG=Search](http://www.google.com/search?hl=en&q=new+to+java&btnG=Search)
[/LIST]
As far as you current book-list, you've got some that delve a little too soon into the deep end.   The Java Tutorial and Head First Java are good starting points.  I've heard positive things about "Thinking in Java".   Patterns are good to learn (essential, actually) to help re-inforce OO concepts, but the OO part has to be understood very, very well before getting into the GoF book ("Elements of Reusable..").  Also, it's just C++ & Smalltalk in the examples (doesn't/shouldn't matter, though -- implement these yourself in Java for practice).  Instead, start with the Head First Design Patterns book, which also teaches programming in Java at the same time.

After a firm foot hold on Java, read (and/or re-read) Effective Java. Don't be a professional programmer in Java without it.

Puzzlers and Pitfalls will tax your skills once you're feeling over-confident.

I'd highly recommend Netbeans (and I don't mean to start an IDE war...) for a new developer. I wouldn't recommend one of the "learning IDE's", which are a good idea in principle, but in practice you stick with the same IDE for-ever once you're comfortable with it.  Eclipse is bang-your-head-against-the-wall counter-intuitive until you know what you're doing (then it's fine... mostly).  

You'll also have to get familiar with Maven2 and/or Ant for building your projects, but then we're just going down that can-of-worms that modern development has become....

Good luck,
-m

in general i have to say that java is a poor language. it's extraordinarily verbose and bureaucratic, and it's abstracted away from the hardware so you won't learn much about how a computer works either. 

the company i used to work at paid me to do the scjp (sun certified java programmer) a while back. i think even after i had studied relentlessly for a month and had passed with a good grade i could do less with java than with python, where i had only spent an afternoon learning it. 

you can however get a lot of jobs writing java code. it's bureaucracy allows and requires companies to structure large projects, so they can charge more for the work they do, so it's very popular.

when it comes to an ide, i'd recommend netbeans. for java you really need an ide, otherwise you'll go crazy typing stuff like ArrayIndexOutOfBoundsException, and eclipse is dog slow and very buggy. for this reason, eclipse is the ide most used in companies to write java code.

sorry to be so negative here after you've already spent the money on the books, but i'd really recommend trying to sell the books or return them and instead learning either C (if you want to lean more about computer hardware) or python/ruby (if you just want to write software). java is rather badly suited for both tasks.

### Post by michael_1234 on 2009-02-11
> Is this by any chance book that had about 20+(I think) patterns like Builder, Factory, Visitor, Command etc? I read such a book while I was studying year in England(one of the courses centered around it) and it was extremely useful(atleast for me). Been trying to remember what the book was called so I could get my hands on it myself!

Yes, this is the classic, far-too-often referred to text "***Design Patterns: Elements of Reusable Object-Oriented Software***", aka the GoF book. ( [http://en.wikipedia.org/wiki/Design_Patterns_(book](http://en.wikipedia.org/wiki/Design_Patterns_(book)) ).  Good book at the time (15 years ago).

But, it is far from the only book out there on the topic. About ~10 years ago (and recurring in waves every few years) the topic of patterns became once again all the rage; & periodically, a slew of new books get published -- all claiming to be somehow "patterns" related. Some are terribly gawd-awful poor, some are simply idioms or at best [suggested] best-practices.

The only Java book that I ever liked which covered the GoF territory (since the GoF was mostly C++ and some smalltalk) was "Design Patterns Java Workbook" (Steve Metsker).  I used that when I taught a Design Patterns class many, many years ago  (pre-dot-com boom), as the course switched languages from Smalltalk, then to C++, then eventually to Java.  At the time, it was useful since many people were new to Java.

Btw, the other ("classic") series of books on the topic were the POSA books, [http://www.cs.wustl.edu/~schmidt/POSA/](http://www.cs.wustl.edu/~schmidt/POSA/)

(Note that my references are quite dated, but the nice thing about patterns is that they (inherently) age quite nicely, unlike programming languages.)
