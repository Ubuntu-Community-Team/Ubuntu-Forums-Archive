I would agree with Python being the way to go if you just want to get up and programming quickly. ### Post by Wybiral on 2008-03-04

Not really. C didn't teach me anything about registers or stack frames or what's actually going on "under the hood" of the C compiler (when I type "x * 2" I don't need to know that GCC is turning it into a left bit-shift or even an addition, I just care that it works). Then when I learned assembly, even that didn't teach me anything about the microcode layer that the assembly is running on or the actual circuit implementations of the hardware. But, in fact, I don't care about any of that. I care that my code can solve my problems, I don't care how it gets there (unless that's part of my problem, which it usually is not).

Your notion of C efficiency is built on layers of abstraction from the microcode, to the assembly being generated, all the way up to the optimizations that the compiler is performing behind your back. There's nothing wrong with those abstraction levels, and there's nothing wrong with adding another one, such as "Python/Perl/Java/etc" as long as it gets the job done with less work (and more maintainability).

I care more about writing efficient algorithms and data structure than I do about writing platform/language specific code.

Doing it will teach you what not to do. :) I'm no Java programmer, but I tried building a huge string in Ruby by contatenating many smaller strings. After the program took hours to run (mostly due to intense swapping), I realized that I needed to respect the fact that Ruby strings are immutable. I don't know what I would've done if I were forced to use Java in that situation. Java requires that you declare an array's size when declaring the variable, and there wasn't a good way to determine the size of my eventual array ahead of time.

@slavik, I see it's this discussion again :) I'm certainly one of those people who consider CS to be a "graduate-level major" and that yes, it should require a hard science, namely, math (which is not  strictly speaking a science as it is not empirical though). I never considered CS Master's programs to be "programming schools" -- I am personally a surprisingly weak programmer considering the time I've spent on academics :)

But I also disagree with the need to go deep down to the engineering side of things even when you're a CS graduate. Of course it would be a bit extreme to claim that I wouldn't need to know how a CPU works on the assembly level, but even for graduate-level CS, the more important part is the ability to work with the "abstract machines" you define -- you first define your automaton you use to compute, then you figure out its limits, write an algorithm, then analyse it and try to figure out whether there is a better way. This is at the point CS becomes math, as most of the time you spend 90% of the time analysing your *problem* instead of "programming".

So... for someone who is doing some other science and just using Python(?) to crunch numbers so that there's no need to use pencil and paper, yes, understanding the high-level abstractions is sufficient. If there are deeper algorithmic issues involved, I wouldn't expect him to even start tackling it -- there is someone who does both this science and CS and is able to write a library for the problem.
