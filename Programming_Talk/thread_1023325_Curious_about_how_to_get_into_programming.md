### Post by nvteighen on 2008-12-28
To get into programming you have to learn what it is: to describe processes in a certain language and simulate that process in a computer. You have different languages because each is specialized for some kind of task, so you have languages that are the best for controlling the computer's behaivor (therefore, describing a non-computer process gets harder) like C, C++, Assembly and Forth, for example. Those are called "low abstraction level languages" or simply, "low-level" languages, because their basic elements are too near to the computer's internal working. Other languages, instead, separate themselves from the computer's elements and offer you basic elements much nearer to human concepts like lists; these are the "high abstraction level languages" ("high-level") like Python, Perl, Ruby, Java, Lisp languages, etc. All of these serve for general-purpose programming: you can do everything with them, no matter how hard it can be (Turing completeness). But, other type of languages consists in those that restrict themselves to a certain domain like HTML, SQL, XML and similar. 

The best is to start with a high-level language, because it hides away the need of controlling the computer's state and also because their built-ins are much more natural for a beginner to work with. Python is very nice because the language itself is very easy, allowing you to learn what programming is without having to learn a complex language at the same time.

But you must keep in mind that, to be a good programmer, you need to have more than one single language in your toolcase. There are tasks where Python will be useless and you'll need C or even Assembly and others where Common Lisp will make it much better.

### Post by CptPicard on 2008-12-28
> **shadylookin said:**
> the concepts are what's important and they transcend the language wars.

Fully agreed with the fact that it's concepts that count... however, it is interesting to note concepts' relationship with languages and the so-called language wars. Higher-level languages genuinely introduce higher-level concepts, they are not just syntactic variants of lower-level languages. Interestingly, the most avowed low-level language guys may actually know that concepts are transcendental, but because they only know concepts present in their own languages, they refuse to accept the existence of other concepts in higher-level languages, but resort to Turing-completeness in their argumentation instead (that is, higher-level languages are just "syntactic sugar" on top of the low-level languages, as if the higher-level concepts did not exist at all).

### Post by CptPicard on 2008-12-28
It is interesting to consider though the immediate counter-reaction of the C crowd and how it relates to the way you slightly cut corners in your definition of a closure here.

In C, a function is just "barely" a first class object through the ability to pass a function pointer. You can dynamically select a function to be run on a set of arguments. However the definition cannot be ad hoc right there at the spot in code. Most importantly, we do not have closures, that is, *captures of the set of bindings visible at the definition of the function (lambda)*.

So we have multiple forms of abstraction here... runtime function selection through a simple function pointer that C is able to, inline anonymous function definition it is not capable of, and then the actual closure; the set of names *outside of the definition of this function* that we have access to. It's the last part that C totally lacks and needs a lot of implementation to mimic.
