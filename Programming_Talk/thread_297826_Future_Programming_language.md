### Post by deepwave on 2006-11-11
Programming languages... eh?  Took a third year university course devoted to that one question alone.  Never fear, my answer will not take you a semester to read. :)

Every programming language is designed to do one type of work very well.  So it depends on what you want to do.  Fortunately there are a few languages that most people use and know:

Java (nice syntax, object-oriented, great for the web, awful for low level)
C (good for short low level programs, fast, terrible for larger things)
C++ (like C only good for larger programs, can be a pain to debug, most used)
Python (object oriented powerful script language, overall nice and fast)
Perl (very fast, quick for scripts, UNreadible)
PHP (web applications)

These are most popular.  Get a handle on these, since these are the standards for industry and academics.  All these belong to a class of languages called procedural languages.  Some other languages need a complete different way of thinking.  These include: Prolog (logic based, used for AI) and Scheme (functional).

For someone going into university, I propse you take the Python (since you know it) -> Java -> C ->C++ route.  If you plan on making web programs try PHP.  Play around with Perl too, it is ugly but very useful.

Hope that helps.

### Post by hod139 on 2006-11-12
I think a lot of people forget that computer science != computer programming.  Computer science is a **science** and it does not depend, nor should it, on a programming language.  It is the systematic study of computation.  One topic of the many is programming languages (and even that is theoretical which should start at lambda calculus).  More important topics are abstract concepts like algorithms and data structures, grammars, program design, architecture, and theory of computation.  Then there are field specific topics like parallel/concurrent/distributed systems, databases, artificial intelligence, graphics, etc. 

A lot of early computer science students think computer science is just computer programming and get blind sided after taking the intro courses. Subjects like data structures and algorithms (DSA), theory of computation/complexity theory, and discrete math require a fair amount of mathematics and DAS in particular is considered the class that eliminates **many** computer science students.

For a take home message, computer programming applies to a specific language while computer science does not.  Getting back at your original question, I would say don't worry about a language.  Pay attention in **math** and you should be fine.  Once you understand all the concepts, learning a specific language's syntax is less important.

### Post by ArtechnikA on 2006-11-21
Programming languages are tools.
Like any tool, before you can pick the right one, you have to know what you want to do.  You've heard the phrase "When your only tool is a hammer, every problem looks like a nail" ?  Same thing.

If your vision of the world is user-interface elements, Java and VisualBasic are far and away the most commercially popular ways to make presentation-layer stuff.  I don't think I'd want to write a compiler in VisualBasic, tho.  (You -could-, I'm sure.  Once upon a time, *everything* was written in some assembler or another...  These were the days before it took 1GB memory to host a simple test editor.  It was the days before there was 1GB memory on the *planet*! But I digress...)

For systems work, C and C++, sometimes Fortran with realtime extensions, and some larger languages like Ada come to mind.

ALL (useful) languages do the same thing: read, interpret, transform, and produce DATA.  (Properly used, they typically transform data to information as well, but that's a fine point.) Everything else is implementation detail, favoring one tradeoff over another.  Once you can abstract what languages DO and how they WORK, you'll be in a much better position to simply get on with the task of producing results.

I have found that way too many programmers (perhaps most...) get too sucked into algorithm design and lose sight of the real issue, which is THE DATA.  A blinding-fast elegant algorithm that produces the wrong answer is USELESS!  Never lose sight of the DATA.

(To this end, I really suggest that at some point, you study a declarative language like Prolog, in which there's really no such thing as an algorithm - or a data structure, for that matter.)

It could be argued that the reason we have som many different kinds of languages is that there are so many different ways of looking at data.  An accounting program, that has to keep track of billions of dollars, accurate to the penny (try that in your standard 4-byte floating-point representation!) has a very different view of data than a signal processing program that is digitizing a 2GHz radio data stream at 5 million samples/sec, which is a very different view than a compiler, that is turning text into computer instructions.

It's always about the data first, and what you can do with it second.

I'll have to disagree with you here.  Good (correct) algorithms will never produce the wrong answer.  Part of computer science is understanding how to design fast and correct algorithms.  Proving that an algorithm will always produce the desired result is a very difficult task (e.g. loop invariants in designing procedures.)

>

Prolog is not a declarative language, it is a logic programming language.  Prolog = Programming Logic.  It is based on First-order logic, and as such a user has to define a set of rules.  These rules can be used to write algorithms (you can write quick sort for example).   Lastly, there is a data structure, prolog has lists.  Using lists, you can write more complicated structures like trees etc.

### Post by achron on 2006-11-22
Languages, schmanguages (not a real word I know)...  To reiterate an earlier post, higher education really IS about learning to learn.  Programming languages are, after all, different dialects of the same base language.  And it's NOT a programming language, it's a language of rational/logical thought.  As long as you can be organized, can decompose a problem into steps needed to solve it, and carry it through, you'll do fine.  The more tools you learn to use, the more adaptable you become, and the more different tasks you can perform.  Specialists are good, but I don't need 10 specialists in Java, or C++. I need two or three staff that can get into the gruesome details of any language.  The other 8 I need to be flexible enough to adapt to what the current customer wants or likes or requires.  Nothing is worse than a carpenter that only owns one hammer, or a programmer that only knows one language.

I've probalby interviewed upwards of 400 people for developer-type positions in the past 20 years across some 7 to 10 languages...  it's not terribly important to me what language you know at this moment, unless I have a burning need on a project for that particular language, that's all just syntax. Prove to me that you can think, and that you're in the field because you find it interesting or exciting, then you'll score extra points.

"I like programming because I can make alot of money" is a death knell if I am interviewing you. If you're chasing money, you probably won't stay for more than two years, and it's not worth my time training you to the specifics of the industry.
