### Post by SaintDanBert on 2010-10-22
Consider this:

Over here is a folder filled with email files -- from evolution, from thunderbird, whatever.

Open each email file and format it into a pretty document of some sort.
My preference is PDF, but HTLM works as well.

The idea is arrive at a folder filled with email mesages in a format that is suitable for an archive or historical record.

As a programming project, you will need to create the engine to find open and read the email files.  You will need an engine to open and create the pretty output files.  You will need glue to tie the two engines together.  Your engines might be two command line programs while the glue is a shell script.

If you create your parts well, you can then take known-to-work engines and insert them into a user interface so that your tool appeals to the GUI folks.

### Post by Some Penguin on 2010-10-23
Off-hand, I don't think I've ever had a interviewer ask me to demonstrate a previously written project, and it's not something I've asked candidates to do, either.  If you want to do a project to prepare, do it for what it teaches you, not necessarily to have code to show.  You may be asked to show coding prowess by having to solve one or more problems that they give you in a limited time.

What you -should- be prepared to demonstrate varies, but would normally include --

- familiarity with basic data structures (arrays, lists, maps, sets, trees and the like)
- familiarity with basic algorithms (sorting, searching)
- familiarity with algorithms / systems particularly pertinent to the potential employer (e.g. in the search space, experience with inverted indices e.g. Lucene is relevant.  For server-side application development, Apache Tomcat is a pretty common platform.)
- ability to work in teams of appropriate size (which are likely to be somewhat large for large companies); i.e. social skills actually matter somewhat, as does writing intelligible code and being able to read that of others
- speed of and willingness to learn new things
- initiative, i.e. self-starting
- useful domain knowledge; e.g. if you want to intern for a pharmaceutical company doing protein-folding simulations, chemistry is relevant

As noted above, a lot does depend on where you're looking at.  If you're looking at working for nVidia, they probably care much more about highly performant, tightly optimized low-level code.  If you're looking at Genentech, they'd care about some familiarity with such things as feature-selection algorithms.  Microsoft's operating systems group would appreciate awareness of computer and kernel architecture.  And so forth.

### Post by SaintDanBert on 2010-10-23
More and more, "programmer" job interviews ask for some sort of "portfolio". As a university instructor in computer science, our program encouraged students to accumulate their several projects into a portfolio to show during job interviews.

I agree about knowledge and skills covering data structures and associated algorithms. I know of companies that require various "tests" of programming knowledge. For example, Google(R) required Ken Thompson (co-inventer of Unix(tm) ) to pass a test before he could post code to the production web site(s).

Be careful about what you add to your portfolio from your current or previous employers. Get permission whenever possible.  If your not currently employed, portfolio projects are a great way practice existing skills and learn or refine new skills.
