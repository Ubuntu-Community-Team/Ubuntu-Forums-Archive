### Post by kyleddm on 2009-08-17
:confused: Hi guys, I'm building a somple program that uses third-party libraries, .a and .so. I am using the latest version of KDevelop with an automake project. I know the libraries are installed, I did it myself and I use the -L command to allow the linker to see the directories where the libraries are located, but I still receive this message:
 
"ld cannor find -l(library)" where (library) is any non-standard library I am using. In my project's case I am using third-party TCP libraries called ComPP as well as a software package called MOOS which allows me to use a central database to pass information between modules. If I place a space between the -l and the library name, I get an error stating that there is no rule to make the library needed by the executable. I tried the KDevelop forums but apparently after a week of waiting, they must have better things to do than help users of their software. If anyone knows how to fix this problem, it would be much appreciated.
 
Thanks

### Post by nvteighen on 2009-08-17
Is this at runtime or compile-time?

If at compile-time, please send us the error message and the generated Makefile.

If it's at runtime, in order to have the linker know where the libraries are, you have to either copy them to /usr/local/lib (/usr/lib is reserved for libraries installed through packages) and then run ldconfig (as root). Or set LD_LIBRARY_PATH (IIRC) to wherever the libraries are located.

### Post by stroyan on 2009-08-17
You misunderstand what the "-l" option expects.
It wants only the part of the library name between "lib" and ".a" or ".so".
Instead of ```
iDVL_LDADD = -llibMOOS.a -llibMOOSGen.a -llibMOOSUtility.a -llibfltkvw.a \
-llibComPP.a -llibSysPP.a
``` use ```
iDVL_LDADD = -lMOOS -lMOOSGen -lMOOSUtility -lfltkvw -lComPP -lSysPP
```

Be careful when linking with archive libraries.  You need to put them in an order that has all the symbol dependencies of the earlier libraries resolved by symbols provided by later libraries.  If the second library in a list needs a symbol from the first library in a list then the linker will not necessarily include it.  It is possible to group archive libraries so that the linker will loop over them searching for required symbols.  That is less efficient.  You would do it like this-
```
iDVL_LDADD =-Wl,--startgroup -lMOOS -lMOOSGen -lMOOSUtility -lfltkvw -lComPP -lSysPP -Wl,--endgroup
```

The pthread functions are in "-lrt".
The semaphore functions such as semget and semop are in libc, "-lc".
It is really odd that your linking is not getting libc functions.
I see that you link with-
LIBTOOL = $(SHELL) $(top_builddir)/libtool
It is much more common to link c++ code using g++.
It provides all sorts of libraries, .o files, and directives that c++ code needs.
You can see what it does using "g++ -v main.cpp" with a simple program.
