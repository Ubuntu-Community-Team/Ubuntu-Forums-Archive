### Post by bilijoe on 2008-08-13
I'm a bit "old school", but I think the most important computer language of them all is good old "C".;)  I'm not a major fan of the "++" extensions, though if you don't have the insight or expertise to implement object oriented functionality on your own (a great learning experience that will lead to a better understanding of objectifying program features than any degree of expertise at simply using the object oriented aspects of C++), it's not a bad way to go.  

I don't know how they are teaching things these days, but having done some teaching myself, I know that it's a lot easier to "do" or to "demonstrate" computer related concepts than it is to "teach" them (something about the nature of mechanizing concepts, I suppose:confused:).  So, however you are being taught, keep firmly in mind that the language the code is written in is "**_C_**".  The "++" part is one method of implementing constructs that make it easier to incorporate objectification into programs written (read that "coded") in C.

Just think of an "object" as a black box, with a well defined interface, that performs some specific function (or group of closely related functions).  The language is C.  Object oriented thinking is a subject unto itself, and I think it's a fundamental mistake (albeit one made by many, many educators) to teach "C++", especially as an introductory language.  C should be taught as a programming language, and thoroughly understood by the student.  Then the basic concepts of object oriented thinking should be taught, as a language independent subject.  Finally, the two should be married in a course aimed at using the ++ part of C++ to implement the object oriented concepts.

I also think that it is a mistake to teach computer programming by teaching a programming language.=;  You use a language to write code to implement a program.  Programming has no real connection to any language, and a well designed program can be coded in [virtually] any language.  

Programming and coding are two separate disciplines, and should be taught as such.  In fact, I would argue that "programming language" is a misnomer.  The language of programming is all the notes, diagrams, flow charts, action diagrams, decision trees, etc., that are used in laying out a program.  C, BASIC, COBOL, FORTRAN, and all the more modern "languages" would, I think, more accurately be referred to as Computer Codes.  

Once you have designed the actual program, the next step is to "code" it, so a compiler or assembler can turn it into the string of ones and zeros that is the actual application (or driver, etc.).

This is just my opinion--I know many educators would disagree.  But I learned to program back in the days when you were either self taught, or you chose a different profession, and the above is a relatively accurate description of the approach I used to teach myself how to get a computer to do what I wanted it to do.  Apparently, it worked rather well, as, during my career, I was called upon to write many highly complex (often classified) applications wherein inaccuracy, bugs, and failure were not options.  These programs were used in situations involving National Security, Government and Military Intelligence, and Federal and International Law Enforcement, and I was one of only three programmers in the country, trusted to do this kind of work.  

My methods may well be out of date now, but I still see students graduating with Computer Science degrees, who cannot produce anything except bloated, redundant, error prone programs.  Not because they aren't intelligent people (many of them go on to teach themselves to be excellent programmers), but because the academic community *still* hasn't learned how to teach people to efficiently and effectively get computers to do [exactly] what they want, or need them to do.

Hang in there though.  Take the ++ stuff with a grain of salt.  C itself is an elegant language, with tremendous power and flexibility, and when you get to the point where you find yourself describing it as elegant, you will have arrived.  Good luck.

### Post by nvteighen on 2008-08-13
Coming from C and learning Python, I can really tell you that dynamic typing is really great, when combined with strong typing. Javascript's dynamic-weak discipline is confusing, but Python's dynamic-strong gives you the flexibility of dynamic typing + a reasonable hard type safety.

There's no way to have a square root of a string in Python, unless you convert it to an integer or float, so a beginner wouldn't be able to pass a variable of the wrong type to the wrong function. Javascript, in the other hand, may allow you to take the square root of "hello" (NaN, of course... but that's a runtime error and a newbie will have a little more work in figuring out what's happening).

Static typing is only useful when combined with weak typing. A static-strong sounds almost as a nightmere to me... Static-weak gives you the type safety and some needed freedom to not cast everything. C (static-weak) is relatively easy to use (in this matter), if you're aware that its data types are just byte lengths, so you can store a 'A' into a long long int and the number 45 in a char. But of course, that would require the programmer to be more skilled and sometimes, implicit casts may get you into obscure runtime errors difficult to debug.

So, for a beginner, I agree with pmasiar. For an experienced programmer, this shouldn't be an issue, I think.

I'll try, but it's hard to explain. I don't know the correct terminology, but I can explain it in practical terms.

```

;;;; A closure in Common Lisp (Heavily Annotated)

;;; Define a new function, make-counter, that takes a single argument.
;;; LET bindings are like variable assignments.
;;; LAMBDA expressions create new annonymous functions.
;;; Since the last expression in a Lisp function is returned, the new anonymous function
;;; is returned after calling MAKE-COUNTER. But wait! The
;;; variable I is not declared inside the function!
;;; It's value is "captured" and is maintained for every subsequent call.
;;; INCF increments a variable (like ++i).
(defun make-counter (limit)
  (let ((i 0))
    (lambda ()
      (if (>= i limit)
          nil
          (incf i)))))

;;; Create a new counter that counts to 5
;;; and assign it to the COUNTER symbol.
(defvar counter (make-counter 5))

```

The COUNTER symbol is now bound to an anonymous function
that was returned as a result of calling MAKE-COUNTER.

The FUNCALL function takes a function and it's arguments (none in this case), and calls it.

```

CL-USER> counter
#<CLOSURE (LAMBDA ()) {B085C05}>
CL-USER> (funcall counter)
1
CL-USER> (funcall counter)
2
CL-USER> (funcall counter)
3
CL-USER> (funcall counter)
4
CL-USER> (funcall counter)
5
CL-USER> (funcall counter)
NIL

```
