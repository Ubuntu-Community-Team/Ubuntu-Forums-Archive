### Post by nvteighen on 2009-07-30
Hmm... C++ can be really demotivating if you're a beginner. You need to learn too much stuff in there to be able to have something interesting. It's a complex language that will always be getting into your way meanwhile you're **also** trying to learn how to program (which is also a difficult thing).

Programming is language-agnostic, but you still need to learn and use a language in order to program. That's why some people, including myself, think using high abstraction level languages ("high-level languages", for short) are much better suited for this. As their base concepts are nearer to human concepts, these make your path to learn how to program much softer... That's why the first language choice is critical, because you'll have to learn two things at the same time.

Once you've learned what programming is, you'll be able to move on your own, learn different languages and different paradigms. In your path, you'll learn what's actually programming and what's just some language's "way to do it" and progressively see things beyond the language you might be using at that moment.

Of course, some languages make some stuff easier to see than others. That's why learning C after, e.g. Python, is IMO very important and a must for any programmer-wannabe. C will teach you the details of low-level programming the best. C++, with its "middle-level = high-level + low-level mixed together" approach, does this much worse... I tend to think C++ should be learned when one's really mature on programming and had acquired some amount of experience.

Now, how to learn a programming language? **Nvteighen's Method(tm)** :D is: Get some learning resource, follow it until but also doing things by your own. Don't get frustrated if the stuff you do by your own doesn't work; they will in a future... but leave the knowledge ripe in your mind and keep challenging yourself, researching as most as you can to get them working. Doing your tutorial's exercises will help you to get the language's basics; your "personal exercises" will make you learn by doing and therefore, learning by experience. When you notice your "personal exercises" start going well, that you're grabbing knowledge from other places besides your "tutorial" (e.g. the language's documentation, some API, etc.), leave the tutorial and take off. You'll learn the rest of the language when you'll need it for some specific project.

That way, by inductive learning, you'll learn by your needs and you'll learn better. The drawback is that this method is a bit slower at the beginning than "deductive" methods (learning something by heart and then force some situation where you'll need it in order to practice), but IMO it's a much stronger and secure way... Basically, it's the same way modern Language Schools are teaching languages nowadays, even dead languages like Greek or Latin (ok, it's annoying to see your teacher coming to you, handing you a text of Cicero you don't understand a word of in your first lesson and start teaching you Latin from the text itself... but slowly you see the results :p).

### Post by ve4cib on 2011-10-02
> **CptPicard said:**
> Mine isn't, as far as I know, at it angers me that someone would suggest C and C++ are the same language :(

I think the presence of the "/" character implies that whoever wrote it knows that they are not the same language.  If that's how you choose to interpret it that's fine, but be careful about putting words in other people's mouths and then getting angry about it.  It's really a pretty trivial thing to get angry about.

Technically speaking C is a subset of C++ (or C++ is a superset of C, depending on how you want to word things), and given that you can compile any C program using a C++ compiler it is not entirely unreasonable to refer to them as "C/++" or "C/C++."  The "/" character implies that there is a distinction, but that the two languages are so closely related that -- in many respects -- they are the same language.

Also, there are many programmers out there who use a very mixed blend of C and C++ in the same program.  A lot of "high performance" C++ is written like this; it's a C program, with a small handful of classes where it makes sense to use them, but limits their use to save on the extra pointer dereferencing and constructor/destructor calling that is unavoidable in "pure" C++.  Given that such programs are essentially part C and part C++ I think any of the "C/++" terms are perfectly acceptable for such instances.


Anyway, back to the OP's question...

I find the best way to keep yourself motivated to learn something new is to try to make sure you have an interesting project to work on.  If you're just blasting through example programs in a textbook or on a website you will get bored easily.  But if you've got an end project you want to work on I would seriously consider starting working on it ASAP.  Accept that you will probably rewrite large sections of it as you learn more, but if you're working on something you're interested in doing then it's a much less onerous task.

There's also the technique of forcing yourself into a routine.  Set aside an hour or two every day (ideally at the same time) as your "coding time."  Set an alarm, and force yourself to sit down and bang out code and read examples for that period of time.  Sticking to a routine may help you stay motivated, since you've got your own private schedule to keep.

Finally, if you find that you're just not enjoying C++, or just "not getting it" don't worry about it.  Programming is not the kind of hobby everyone will enjoy, and if you discover that it's just not your thing nobody will think less of you for it.
