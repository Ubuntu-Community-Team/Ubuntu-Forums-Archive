**What language to learn first?** 

I have strong opinion (supported by experts) that Python is the best first language, and maintain wiki [LearnPython.pbwiki.com](http://learnpython.pbwiki.com/HowToStart) with best free tutorials I found on internet during couple years of searching and tested them on beginners. **Important part of learning programming is solving problems** so wrote up few tasks and posted links to more. See links below.

**So is Python the only language?** 

Hell no! Any competent programmer need to learn couple more languages: 
- bash shell on Linux to automate your tasks
- C to understand how CPU works on "low level", when you have full control - and full responsibility. C is harder to learn, because Python solves for you (and hides) many important problems (for small speed penalty). 
- Java or C#.NET to get a job. After C, they are quite boring, but have **huge** libraries, which is both blessing (you don't have to code it) and curse (you will spend **years** learning them). If you are smart and lucky, you can get non-boring job (some startup?) using more fun language.

More languages to expand your thinking (it's like weightlifting for brain):

- Lisp: with Fortran, oldest language still in use. Very different: After Lisp, you will think differently about what programming is. (Fortran is very same :-) )
- [Forth](http://en.wikipedia.org/wiki/Forth_language): my favorite "different" language: very close to the metal, still very easy to expand in any direction you need. Catch? Absolutely no protection from any errors at all. 8-) Expert programmers are 10 times more productive, 'just average' have hard time to do anything. So you may want to challenge yourself.
- SQL: for databases, Prolog, Haskel, Erlang - there are many non-trivial languages, very different from C++/C#/Java. 

If you know only languages from 'C family' (which should be Algol family but nobody remembers Algol anymore) you are blub programmer (see below), thanks for saying it upfront so I can ignore anything you say, to save my time.  

**How to start/which framework for web applications?** 

Start with trivial 3 line app (and 7-line webserver) in my wiki. Rails for Ruby has a competitor in Python. because Python community is bigger and more diverse, two competitors: Django (simpler, best for beginner, but 'one size fits all') and Turbogears (little more complex, but gives you a lot more flexibility).

**Want to start a project?** 

Any non-trivial project will take **years** to complete. You need skills and credentials to recruit a team: you can gain both if you take part in any existing project. All projects are interested in new developers, with one condition: only people who can read FAQ, self-starters need to apply. Is fine to ask any question (stupid question is only the one you did not asked and wasted week to figure it yourself). It is not fine to ignore provided links and leech time from your mentors. 

If you join existing similar project (for a year), you will learn their workflow and tricks. This will let you:
- learn from their experience (instead from your own mistakes)
- gain skills in handling big projects you may lack
- gain name recognition for your good work
- which would allow you to recruit people to help you with your own project: you will need it, if your project might easily be 10 man-years.

so "wasting" a year will help you to avoid stupid design decisions, and might allow you to  recruit say 3 more developers, so you finish your game in 1+3 years total, instead of 10 years doing it alone.

### Post by CptPicard on 2008-09-05
> And even if you only know one language, you're not a Blub programmer if you only solve problems that are best suited for that particular language.

The problem here is that the concept really includes the idea that if you just stick to the one language, it is very hard to tell whether something else would be suitable for the given problem, and plus the mindset that comes from using a single language -- simply not "getting" *why* some feature not present in that single language is meaningful in the context of the problem at hand.

Then there is the general belief that I tend to subscribe to that languages are not completely problem-specific -- some languages just simply are more suitable, because of higher abstraction capacity -- for the general case of "problem-solving by programming", but that has been discussed to death already :)

Summary of discussion (and lets continue in "love/hate" thread, in context):
- if I run code rarely, performance penalty does not matter
- if my code has other bottleneck, like I/O speed, database access, web server response: penalty does not matter.
- If in numeric calculations I use SciPy (bindings for same C libraries) for matrix calc (instead of generic Python objects), penalty is minimal.
- After profiling, with minimal effort I can convert code (bottleneck only, not all!) to Cython, which has Python-like syntax but compiles to C.

But in every case, agility of development in Python increases 10 times, and penalty is close to by ignored. Remember, in 85% cases, the most important speed of your code is speed to the market.
