### Post by CptPicard on 2009-01-24
Well, guys... speed is not really the issue. Concepts of the language are. Language shapes your thinking -- this is true of natural languages, and it is also true of formal languages such as programming languages -- some people have claimed this effect does not exist, but at least it does for me, as I see the programming language as some kind of "abstract algebra" I calculate about the problem domain with. This is why having a variety of languages with different kinds of approaches available to you is important. 

You may still have to hack up your implementation in C for speed reasons, but it's not as if writing C itself is some sort of pinnacle of programming prowess... it is your ability to see ways to approach your problem that matters, and higher-level languages tend to give you ideas that you can try to downscale/apply in something less flexible (which can be sometimes hard, depending on what you are trying to replicate).

The jury really is still out on (static-typed) native-compilation vs. VM-JIT-runtime recompilation... Java in particular has the problem of nearly everything being on the heap at the end of a pointer, so compiler has less ability to pull stuff down into registers. But it's not really *that* bad for most purposes -- the optimization thread where Cracauer kindly provided a C++ implementation was interesting. My impressions of that thread was that in particular the last big improvement into the C++ variant's runtime was indeed due to a data structure change; before that, Java was holding its own. But I have not got myself to writing a same kind of hash for the Java thing... 

If one really insists on being a speed kiddy dismissive of higher-level languages, well.. Haskell is a strong contender, and very abstract. Pure-functional stuff with strong type info (type-inferencing plays together real nice with pure-functionality btw so you don't need to be repeating your type annotations all the time) can be compiled into pretty fast stuff.

Also, SBCL compiles Common Lisp into asm that approaches C-like speed, but those functions are coded usually in a rather C-like imperative manner, too. There is a lot to be said for the ability to pick and choose your abstraction level at will, though.

From my perspective though, C is essentially a portable abstraction layer on top of assember. Nice syntactic sugar and admittedly does a good job at it (that is, there is nothing *wrong* with it, but people just needlessly associate mystical qualities to it). But as far as differences between HLLs and C go, HLLs most certainly are *not *just "C libraries" -- you can put together as much C as you like in terms of building libraries, but you're still putting together "more and more C". Until you actually build enough library to have built a whole another language, can you start seeing effects that are bigger than the sum of all the parts.

Claiming that something is "better" is a much more vague term than saying than some language has semantically stronger (or more) concepts in it than some other language. Personally I see that stuff as not being subjective at all -- it is a completely objective matter to observe that C consists of some basic control constructs, functions, and a bunch of primitive types plus a struct (and barely the concept of array). Other languages contain those concepts, and then other concepts, some of them which are hard to implement in C without resorting to essentially writing a new language.

I really do not understand how you can claim that it is a subjective matter that I can't, for example, create an anonymous function in C and return it with its associated lexical scope.  For a simpler example just take the basic idea of a virtual function -- it took quite a lot of hacks to the base language to get it turned into C++. You can get close to it by stuffing function pointers to structs, but there you are already hinting at a lot of concepts that can make use of added semantics to the language itself -- constructors for building these object-structs etc.

In any case, the understanding of object-oriented semantics from somewhere else strongly informs how to go about it in C, when you choose to go the OOP route.

My point?  Yeah, coding for human readability is a pain sometimes but going too much down the computer scale and you chop off what your user can do.  Sometimes the pain of having to massage the data is worth it when it provides a modicum of flexibility down the line.
