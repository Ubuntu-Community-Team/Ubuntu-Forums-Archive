tail -f /var/log/messages
```

Now, keep an eye on this, and plug in your HDD. Within a few seconds, you should get a bunch of output. Post this here for a clue to what could be wrong.

Please also post the output of ```
lsusb
``` with the HDD connected.

### Post by prshah on 2009-10-11
> **brianmc said:**
> I power-cycled the HDD, it came up, and now Ubuntu sees it to[o].

what would cause one USB HDD to be unmounted/vanished when I plug in another?

Onboard USB "hubs" generally service a pair of USB ports. However, there are power / bandwidth limitations which can be affected by the plugging in of anther USB device (even an externally powered one), and cause the hub to "blink", losing connected device information.

This will happen in Windows too, and it is usually identified by the message "unrecognized USB device" (and lots of grief thereafter). 

However, I have found that it only happens in the cheaper USB, similar to those in the VIA chipset. It rarely, if ever, happens on original Intel motherboards and chipsets.

In this case, when the hub "blinked" and disconnected your HDD, your HDD's partitions were marked as "dirty". A dirty partition will _not_ be automatically remounted, unless it is connected (and presumably checked) and "safely removed" from within Windows.

You can use the "force" parameter to force mount a "dirty" partition, but this can only be done from the command line. Once force mounted and then unmounted, it will continue to work as before, ie, with automounting.

The force mount command will be similar to```
sudo mount /dev/sdf1 /mnt -o defaults,force
```

This can then be unmounted with ```
sudo umount /mnt
```

### Post by prshah on 2009-12-28
> **brianmc said:**
> How do I pinpoint the exact details of this obviously sub-standard USB hardware 

You can find out your USB chipset with the terminal command```
lspci|grep -i usb
```

Note that it may not entirely be the fault of the USB chipset; a dodgy power supply, some blown capacitors, etc can also be a factor.

However, I strongly believe this to be a (VIA) chipset problem.

Note that onboard USB ports are "paired", ie, 2 ports are served by one "hub". So maybe you will not have a problem if you do not use both "side-by-side" USB connectors; ie, using one port in the front, and one on the back should not give you such a problem.

Or just replace it with a PCI-USB addin card. They are cheap and pretty reliable. Remember to disable the onboard USB if you use an add-in "daughter" card.

Presumably everything gets no further because the disk is formatted with XP filesystem. 

The last presumption is wrong. XP's filesystems (vfat, ntfs) are handled out of the box. 

I get USB reset errors typically when either the hard disk is damaged (with a very distinct, repeated, "click" sound) or when the USB is not supplying enough power.

For example, most 2.5" external enclosures come with a "Y" cable; in this case, the two paired ends of the "Y" are to be connected to the computer's USB ports, and the other end to the enclosure. If you connect only one to the computer, there is not enough power to make the HDD "go".

Mounting is secondary to the HDD being recognized; when you connect an external, look in the dmesg (or /var/log/messages) for a line similar to "sdc (sdc1, sdc2, sdc5, sdc6)" which will mean that the HDD and it's partitions have been recognized. After this, the OS will automount it, or in case of a dirty filesystem, it can be manually mounted. Without this, the HDD is not recognized and cannot be mounted.
