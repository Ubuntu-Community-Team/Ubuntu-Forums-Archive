### Post by sudodus on 2023-04-20
If you know this and have tried it already, please bear with me (that I assume you ran only the commands listed in the original post).

The command line
```

sudo dd if=/dev/zero of=/dev/sdF bs=1M count=1

```
wipes the partition table which means that you will see the drive, but nothing in it (when using standard tools).

You must create a new partition table with at least one partition and put a file system into it, if you want to use the drive to store your backup. **gparted** is a good tool for this purpose.

You must create a new partition table with at least one partition and put a file system into it, if you want to use the drive to store your backup. **gparted** is a good tool for this purpose.

+1, exactly this.

MSFT has been telling people that they have a "D: Drive" and a "E: Drive" for decades.  Well, they've been lying to the world ... or at least using factually incorrect terms.  Every non-floppy disk storage device used by MS-Windows uses a partition table that contains at least 1 partition and inside that partition, there is a file system.  

Linux is no different, though you can shoot yourself in the head (or foot or chest) by ignoring every recommendation to have a partition table and at least 1 partition inside it, then formatting that partition with a file system.  

While you are at it, it would be smart to add a unique label to the file system.  It will make mounting it inside the /etc/fstab file easier.

There are lots of different tools that can accomplish these things, but like sudodus, I would just use **gparted** if I were on a GUI desktop system.  

If on a server, I'd use **sgdisk** for most of the early work, then use LVM tools to setup PVs, VGs, and LVs, then I'd use **mkfs** to format each LV with a file system and finally, I'd use **e2label** to provide a unique label to each file system. If I needed some swap, I'd use **mkswap** on 1 LV.  Linux servers don't have GUIs, so gparted isn't an option.

And just to be 100% clear, I'd use GPT, not MBR/MSDOS for the partition table. There are many reasons.

### Post by sudodus on 2023-04-21
Try to list already collected data
```

sudo smartctl -a /dev/sdf

```

You should get a lot of output. When I run it on an external SSD connected via a USB adapter, I get 96 lines. You can look at the whole output and see if there are some failures but there are also general data about the drive and how much is has been used. If you get no output, then something is wrong. Compare with the other drives.

There should be an overall-health self-assessment. In my case with the SSD seen as /dev/sdc,

```

$ sudo smartctl -a /dev/sdc | grep overall-health
SMART overall-health self-assessment test result: PASSED

```
