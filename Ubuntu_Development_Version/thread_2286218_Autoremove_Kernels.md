I just remembered I had changed mine here is the the change I made to /etc/kernel/postinst.d/apt-auto-removal


```
#!/bin/sh
set -e

# Author: Steve Langasek <steve.langasek@canonical.com>
#
# Mark as not-for-autoremoval those kernel packages that are:
#  - the currently booted version
#  - the kernel version we've been called for
#  - the latest kernel version (determined using rules copied from the grub
#    package for deciding which kernel to boot)
#  - the second-latest kernel version, if the booted kernel version is
#    already the latest and this script is called for that same version,
#    to ensure a fallback remains available in the event the newly-installed
#    kernel at this ABI fails to boot
# In the common case, this results in exactly two kernels saved, but it can
# result in three kernels being saved.  It's better to err on the side of
# saving too many kernels than saving too few.
#
# We generate this list and save it to /etc/apt/apt.conf.d instead of marking
# packages in the database because this runs from a postinst script, and apt
# will overwrite the db when it exits.


eval $(apt-config shell APT_CONF_D Dir::Etc::parts/d)
test -n "${APT_CONF_D}" || APT_CONF_D="/etc/apt/apt.conf.d"
config_file=${APT_CONF_D}/01autoremove-kernels

eval $(apt-config shell DPKG Dir::bin::dpkg/f)
test -n "$DPKG" || DPKG="/usr/bin/dpkg"

installed_version="$1"
running_version="$(uname -r)"


version_test_gt ()
{
    local version_test_gt_sedexp="s/[._-]\(pre\|rc\|test\|git\|old\|trunk\)/~\1/g"
    local version_a="`echo "$1" | sed -e "$version_test_gt_sedexp"`"
    local version_b="`echo "$2" | sed -e "$version_test_gt_sedexp"`"
    $DPKG --compare-versions "$version_a" gt "$version_b"
    return "$?"
}

list="$(${DPKG} -l | awk '/^ii[ ]+(linux|kfreebsd|gnumach)-image-[0-9]/ && $2 !~ /-dbg$/ { print $2 }' | sed -e 's#\(linux\|kfreebsd\|gnumach\)-image-##')"

latest_version=""
previous_version=""
for i in $list; do
    if version_test_gt "$i" "$latest_version"; then
        previous_version="$latest_version"
        latest_version="$i"
    elif version_test_gt "$i" "$previous_version"; then
        previous_version="$i"
    fi
done

if [ "$latest_version" != "$installed_version" ] \
   || [ "$latest_version" != "$running_version" ] \
   || [ "$installed_version" != "$running_version" ]
then
    # We have at least two kernels that we have reason to think the
    # user wants, so don't save the second-newest version.
    previous_version=
fi

kernels="$(echo "$latest_version
$installed_version
$running_version
$previous_version" | sort -u | sed -e 's#\.#\\.#g' )"

generateconfig() {
    cat <<EOF
// DO NOT EDIT! File autogenerated by $0
APT::NeverAutoRemove
{
EOF
    apt-config dump --no-empty --format '%v%n' 'APT::VersionedKernelPackages' | while read package; do
        for kernel in $kernels; do
            echo "   \"^${package}-${kernel}$\";"
        done
    done
    echo '};'
}
generateconfig > "${config_file}.dpkg-new"
mv "${config_file}.dpkg-new" "$config_file"
```
I had changed that file about a year ago. So it has tested well for me at least!

### Post by Cavsfan on 2015-07-13
I have not been able to retain more than the one kernel (the latest that is installed) for nearly a year now. This happens on every Linux system I have except Arch Linux.

Here is the thread I opened about it [http://ubuntuforums.org/showthread.php?t=2251168](http://ubuntuforums.org/showthread.php?t=2251168)

And here is the bug I have filed [https://bugs.launchpad.net/ubuntu/+source/apt/+bug/1440608](https://bugs.launchpad.net/ubuntu/+source/apt/+bug/1440608)

Every time I either delete the kernel manually like this is the most recent one I deleted on Wily:
```
sudo apt-get purge linux-headers-3.19.0-20 linux-headers-3.19.0-20-generic linux-image-3.19.0-20-generic linux-image-extra-3.19.0-20-generic
```

For some reason **/etc/kernel/postinst.d/apt-auto-removal** would be triggered when either the kernel is deleted manually or via autoremove (which does not always get all 4 parts of the kernel) and a reboot is required.

After that reboot **/etc/kernel/postinst.d/apt-auto-removal** has modified **/etc/apt/apt.conf.d/01autoremove-kernels** to contain the newest kernel and the 3rd one back that was previously removed.

Then the 2nd most recent kernel apears that it needs to be autoremoved. If you go ahead and do that another **/etc/kernel/postinst.d/apt-auto-removal** is triggered and reboot is required again.

The **/etc/apt/apt.conf.d/01autoremove-kernels** file will then contain the names of the 2 most recent kernels as it should. The only problem is that the 2nd most recent kernel has already been deleted.

The only workaround to keep two kernels is to manually edit **/etc/apt/apt.conf.d/01autoremove-kernels** and change it back to contain the last 2 kernels. Then you will have 2.

If this is a problem to you add your name to that bug. I do not believe that **apt-auto-removal** should be triggered except once after a new kernel is added; not when a kernel is deleted manually or autoremoved.

But it does exactly that.

Just for simplicity let's say you have kernel 1 and 2 installed and 3 comes up in the updates.

If you are using the cli method of updating you will see this:
**/etc/kernel/postinst.d/apt-auto-removal** gets triggered creating **/etc/apt/apt.conf.d/01autoremove-kernels** and at the time the new kernel is installed,
it contains kernel 1 and 3 so after you reboot it will want to autoremove kernel 2. 
So, you autoremove kernel 2 thus kicking off another **/etc/kernel/postinst.d/apt-auto-removal** 
 So, you autoremove the 2nd to last and you wind up with one kernel.

I always purge the 3rd old kernel after a new one is added so maybe this is why I noticed this. But simply purging the kernel triggers **/etc/kernel/postinst.d/apt-auto-removal **
which should not happen.
