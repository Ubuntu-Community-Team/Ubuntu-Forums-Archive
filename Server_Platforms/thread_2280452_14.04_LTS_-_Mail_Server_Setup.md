### Post by SeijiSensei on 2015-05-30
Is this machine directly on the Internet or behind a firewall router?  If the latter, you'll need to forward ports 143, and 993 if you use SSL, back to the interface on the server that connects to the router.

If the machine is directly on the Internet, make sure whatever firewall you are running allows TCP connections to those ports.

### Post by SeijiSensei on 2015-05-31
Let's try a test.  From a machine outside the network, try "telnet server_name 143".  
```

$ [color=blue]telnet myserver 143[/color]
Trying 192.168.10.10...
Connected to myserver.example.com
Escape character is '^]'.
* OK Dovecot ready.
[color=blue]100 login username password[/color]
100 OK Logged in.
[color=blue]200 logout[/color]
* BYE Logging out
200 OK Logout completed.
Connection closed by foreign host.

```
The blue text represents user input. Replace "username" and "password" with legitimate entries on your server.

I find my Android smartphone to be quite useful for testing like this, since I'm coming in from the provider's network.  I use the [JuiceSSH]("https://play.google.com/store/apps/details?id=com.sonelli.juicessh&hl=en") client which supports telnet as well.

### Post by SeijiSensei on 2015-05-31
> **mlang2 said:**
> using my mobile and the JuiceSSH client, I managed to get logged in ok:

From another laptop on my internal/home network using same server name as above which resolves ok to external IP:
```
Trying xx.x.xxx.xxx...
telnet: Unable to connect to remote host: Connection timed out
```

Like many home routers, yours cannot connect to its public address from machines behind the firewall.  Since you can connect with your phone, your configuration is fine; it's your router that's the problem.

Even one of my clients with a supposedly business-class service from Comcast and the router Comcast provided has this problem. We spent some time tearing our hair out over this before discovering the problem.  I believe cursing soon followed.

They're in the process of changing providers.

So for connections from inside the network, use the internal IP address.
