We've recently implemented 4 new Ubuntu servers to replace an older MySQL 5.0 install on a 32bit windows server, the new master runs 4 disks raid 10 with ext4 filesystem, the three slave machines have single SSD drives also using ext4. All systems are UPS backed up and use NUT so the master lets the slaves know when power is cut and to shut down smoothly, this has been tested.

Read speeds have seen a huge improvement but there is a noticeable speed decrease when it comes to writing to the master, mysql has been configured, though we have set

I believe I've tracked the issue to write barriers reading older posts here and other places, but my main experience is with Windows and I'm cautious of changing the fstab file to included barrier=0 so a couple of question

- Is it as simple as backing up the fstab file, make the changes and reboot, and if there is an issue boot in recovery mode and put the old fstab file back ?

### Post by gottaa on 2012-06-01
Posting an update, made the write barrier change, which had zero effect, what I have worked out so far

- MyISAM much faster than InnoDB (might change the table to MyISAM)
- Using MySQL Transactions really improves speed of writes if you can get the code organised to do all the updates in one lump
- Innotop is useful at watching innodb actions
