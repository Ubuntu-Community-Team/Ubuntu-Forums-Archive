### Post by conjur3r on 2009-01-30
That's normal.  The kernel doesn't actually free up the memory it has paged but uses some sort of algorithm to determine which areas can be paged next time it is needed.  This is to keep most commonly used items in faster accessible memory.

To get a realistic amount of memory it really is using, look for the highlighted value in the following example:
```

# free
             total       used       free     shared    buffers     cached
Mem:       4055000    3983408      71592          0     188368    2882020
-/+ buffers/cache:     [COLOR=Red]**913020**[/COLOR]    3141980
Swap:      2000084       5356    1994728


```

### Post by aaron.d on 2009-01-30
while I agree it is "normal" to some degree ([http://ubuntuforums.org/showpost.php?p=5061778&postcount=5](http://ubuntuforums.org/showpost.php?p=5061778&postcount=5) ... this is one of the most posted issues, and, it would seem, the least searched for), I dont agree that you should be swapping aggressively (you should be doing very little swapping OUT and no swapping IN).

You can check this by using vmstat, try:
```

$vmstat -SK 3 100
``` 

where:
"-SK" = unit size 1024bytes 
"3" = 3 second update period
"100" = run 100 updates

This will start to print line by line snapshots of what the various resources in yo compy are up to.

under the header:
```

---swap--
si   so

```

Is your info of interest.
"so" is not so bad (pun intended), as linux is just basically doing this to give you back usable RAM for real applications by shifting out rarely/never used cached libraries etc.

Really pay attention to si (swap in FROM memory = generally bad).
If both numbers are doing some dancing around, you probably have some issues.
I would expect "si" (swap in FROM disk = generally bad) to be 0 on any well balanced system.

### Post by tubezninja on 2009-01-31
I'm not so sure that the manato (the OP) is talking about normal linux resource management here.  Note he's saying that the memory usage is spilling into the swapfile and causing CPU throttles.  That's not normal caching behavior. Caching is meant to speed up i/o, and it makes no sense to cache into swap file on a disk when that will only slow things down.   I have servers with 4GB+ of RAM, and while they utilize every last bit of that ram for buffering and caching, they've almost never touched the swap volumes.

There's two quick ways I know of to find out for sure.  The first is to install phpsysinfo and see what it tells you about RAM usage.  If most of it is cache, then it shouldn't be a problem, but if increasing amounts are being used by the kernel & apps, then you probably have a memory leak.

You could also try forcing the OS to free up any caches and see if that causes available RAM to increase.

```
sudo sync 
sudo echo 3 | sudo tee /proc/sys/vm/drop_caches 
```

If this doesn't free up any RAM, then your probably right that there's a significant memory leak in one of your apps.

Have you tried using ps to get a list of processes?  You can sort by memory usage this way:

```
ps -eo pmem,pcpu,rss,vsize,args | sort -k 1 -r | more
```

I'm attaching a snapshot from one of my reference servers of what a "good" memory utilization should look like.  This server runs e-mail, MySQL, Apache2, PHP, serves as an NTP server, and acts as a drop site via ssh/sftp for large volumes of data.  Uptimes are often 60+ days at a stretch, and Kernel + apps pretty much stays as displayed here.  Even 800MB at startup as the OP mentions seems pretty bloated.
