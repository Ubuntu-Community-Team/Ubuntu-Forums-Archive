### Post by koenn on 2012-07-10
> but I am clearly missing something, probably something very simple.
yes. Routing.
If a computer needs to see an other computer on a different subnet, it needs a route to it.
The easiest way to do that is to have their default gateway know all the required routes.

So, 
- your ubuntu server will be routing - you've done that with the net.ipv4.ip_forward=1 ; so OK

- your ubuntu server will be default gateway for the 2 subnets; you can set that by dhcp. Note that each subnet needs a default gateway (aka default router) in its own subnet (~ the ip addres of the NIC it's attached to)  -- EDIT: I see you've handled this in yopur dhcp conf; OK then. 

- you need routes, ie, you need to tell the server how to route between the subnets. something like
  * net 192.168.5.0/24 via 192.168.5.1   (or:: via ethX)
  * net 192.168.10.0/24 via 192.168.10.1 (or: via ethY)

you can use 'ip route ...' or 'route add ...' or so. 

you'll also need a script that sets those routs again after a system reboot


that's about it. I haven't looked at your other config files; do the routing first, then troubleshoot if necessary 


- make sure your firewall rules don't get in the way (I haven't looked at them), turn them of until you get the routing right.

```

up route add -net 192.168.5.0/24 gw 192.168.5.1 dev HIGH
```

### Post by SeijiSensei on 2012-07-12
No, you just need to two I gave you.

> Would it make sense it put a route in each destination has a gateway of the other subnet?

If both subnets use the Linux box as their default gateway, then no, it should know how to handle the routing with just the routes you have already.  Packets for the .5 network will be sent via HIGH, and ones for .10 will be sent via LOW.  Anything with a address not in these networks will be masqueraded and sent out via WAN.
