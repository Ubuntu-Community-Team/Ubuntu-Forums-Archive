### Post by 1clue on 2017-07-06
I haven't done it in awhile but some variants of syslog allow remote logging. You would need another box on the LAN which has a properly configured syslog willing to take the stream from your web server.

The thing is, a web server's logging can in some cases be more traffic than the web server's normal output. You will want to be sure you have good bandwidth between your log server and your web server.  IMO it's not worth it unless you have a security concern and suspect some black hat will break in and mess with your system, then cover up the logs.

Be careful again!

Sshfs adds a layer of encryption, which costs you cpu time (encryption) and network time, so it's less efficient than a regular syslog server.

I don't know the particulars of sshfs since I haven't done it on a production system, but if this were pretty much any other shared filesystem there would be additional overhead involved in maintaining the mount and ensuring files aren't being written by multiple computers. In some cases NFS can bring your system to its knees simply because of a bad network cable.

### Post by 1clue on 2017-07-06
Your web application should log failed logins in the application log. If the app doesn't define its own logging then that will end up in the tomcat logs, but regardless failed logins should be logged by the app. If you wrote your own app then you can put in whatever message you want.
