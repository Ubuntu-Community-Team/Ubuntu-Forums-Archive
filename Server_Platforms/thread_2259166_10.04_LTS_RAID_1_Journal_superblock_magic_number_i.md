### Post by MAFoElffen on 2015-01-03
This is what I would do...

md1 is coming up without errors. That is your root.

You said md0 is just swap, right? Instead of wasting hours trying to recover that, there is nothing there that you need to recover? Swap is temporary virtual memory paging. The question your should ask yourself is why do a mirror on volatile space? When you do that, it carries overhead, uses more resources and is slower. Server is not Desktop, so it does not write anything there which needs to be retrieved on a bootup (no hibernation or sleep).

So, if you remove that and convert that to a regular Linux swap partition, it would use less resources and work faster. It would work even faster if you move that off to a small separate disk. If you did decide , for some reason that you wanted that in RAID, it would be faster to remove those and add them back in, instead of You could do all that a lot faster than recovering it. If it is volatile and nothing to recover, why recover it? Just implement a replacement.

Does that make sense?

### Post by MAFoElffen on 2015-01-03
Just a perspective--

-md1 is what you want to keep/recover.
- Boot from a LiveCD and assemble md1. 
- Mount md1
- remove md0
- use fdisk to remove sda1 and sdb1.
- use fdisk to add sda1 and sda2 as type linux swap.
- use blkid to get the UUID's of at least one of the two linux swap partitions (you can redirect the output to a file)
- open the /etc/fstab file and the file you redirected to, to cut-and-paste the UUID, to direct the swap mount to it. Write for both sda1 and sdb1, and comment one out (backup for the what-if's...)
- chroot into your installed filesystem
- Install grub and update grub. What I would do is Since a RAID 1 mirror, install it twice-- first to sdb, then to sda. That way if one of the 2 goes south, you have a better chance of coming back up, next time...

### Post by MAFoElffen on 2015-01-03
Follow this logic--> A RAID container a container > That container is further partitioned (there may be LVM between these layers) > in that partition is a filesystem > In that filesystem, the filesystem has a filesystem type...

Following that logic:
The RAID has to be assembled first for anything to see what is inside of it
```

# mdadm --assemble --scan

```
Or
```

# mdadm -As &#8211;auto=yes &#8211;run /dev/md1

```

```

# Here=/mnt
# mount /dev/md1 $Here
# for i in /dev /dev/pts /proc /sys /run; do mount -B $i $Here$i; done
# chroot $Here
# df
# ls

```
Then do:

```

# grub-install /dev/sdb
# grub-install /dev/sda
# update-grub
<cntrl><D>
# umount $Here

```

That sounds like it's not mount for a filesystem issue inside the Array. Like the journal is still locked from a premature shutdown (wasn't shutdown gracefully).

After you assemble the RAID and before you try to mount it, please run fsck, to try to repair the filesystem.
Sort of like this: [http://linuxexpresso.wordpress.com/2010/03/31/repair-a-broken-ext4-superblock-in-ubuntu/](http://linuxexpresso.wordpress.com/2010/03/31/repair-a-broken-ext4-superblock-in-ubuntu/)
