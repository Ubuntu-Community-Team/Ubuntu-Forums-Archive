### Post by Dale_Walsh on 2014-09-29
OK, I've confirmed that ubuntu's 14.04.1 Server LTS mail installation has serious problems which I attribute to the lack of package maintainer communication.

I built postfix with an embedded cyrus-sasl authentication service and have confirmed that SMTP authentication does work by implementing the following settings in main.cf:
smtpd_sasl_auth_enable = yes
smtpd_use_pw_server = yes


Disabling embedded authentication services (smtpd_use_pw_server = no) and using saslauthd should give the same working results and it doesn't, the issue is confirmed with saslauthd itself and the lack of real logging doesn't help find the problem.

in previous versions it placed the saslauthd files in /var/spool/postfix/var/run/saslauthd/ but postfix expected to find them in /var/run/saslauthd/.

I guess the two projects are maintained by different parties who don't communicate with each other because now it places the saslauthd files in /var/run/saslauthd/ but postfix expects to find them in /var/spool/postfix/var/run/saslauthd/

Great consistency in software configuration.

It's a waste of my time to submit anything to ubuntu as the postfix and saslauthd documentation needs to reflect the actual directories used and these need to be carved in stone to prevent this type of installation nightmare, the directory defaults in the documentation do not reflect the actual directories being used and it appears that no one really tested the software to prevent this kind of problem in the installation process.

The time to write all of this up and submit it is far more time than I have to waste, hopefully this final post will be easily found for the next person who runs into this problem.

Another reinstall from scratch, stopping saslauthd, moving /var/run/saslauthd to /var/spool/postfix/var/run/ and placing a softlink in /var/run/, starting saslauthd solves the problem of SMTP authentication (with or without SSL/TLS).

For me, installing clamav and amavisd as an lmtp service makes better use of the spam service without wasting resources processing outgoing mail and at least installing these packages doesn't alter the postfix configuration allowing you to implement it as needed.

lf you place the softlink in /var/spool/postfix/var/run/ authentication doesn't work and gives the following warning "postfix/smtpd[1491]: warning: SASL authentication failure: cannot connect to saslauthd server: Too many levels of symbolic links".

Rebuilding postfix to use /var/run/saslauthd also seems to solve the problem but the location move and softlink is a quicker and less painless solution.

MY OPINION:
Server software selection/installation should give you a working mail solution out of the box as it does for other services, instructions on alternate installation methods such as the use of mysql/postgres etc... should be provided or easily accessible and syslog should place smtp logs in /var/log/mail.log (it currently does) and pop3/imap logs in /var/log/mailaccess.log (currently does not) to make it easier to debug the incoming/outgoing mail services.
