### Post by Legend1222 on 2010-07-22
I have setup a 10.04 server for testing purposes in my server room. This is replacing an 8.04 server and doing essentially the same job, functioning a a dvr. The 8.04 was doing this for five cameras, the 10.04 is being tested with 25 cameras. Its a ton of data.

Everything was fine with the server. I was very impressed with how well it handled the load. Watched it like a hawk for two weeks straight. Filled up the 2TB raid array no problem, and then started circular recording... no issue at all. So I forgot about it. Let it run on its own, unwatched, and load testing for a week. 

Just looked at it again to find the load at 33. Saw it shoot up to 88. Just to login to the box is taking about 2 minutes. I was able to determine that the problem is a process called "flush-9:0". It will take anywhere from 55% to 99% of the CPU on a given basis. With the other processes on the machine, the cpu is pretty much stuck at 100%.

I can find very little about this process on the internet. I found one thing that seemed to claim that it was an ext4 bug, I found another claiming that its the new way the kernel handles writes to the disk (or something like that), but either way, suddenly the machine is unusable now. Again, it was fine a week ago, and nothing has changed. It obviously has to do with the volume of data, because when I kill the DVR software, flush-9:0 drops to 0% (oh by the way, there are also  processes flush-1:0 - flush-8:0... ). I'm also wondering if this is a side effect of Linux software raid.

I made sure that all updates are installed, updated the kernel, and rebooted. As soon as the machine came back up, that flush process started again and shot the CPU load to 57%.

I really need to know whats causing this. My first instinct is to reformat the raid array to ext3 and see if that eliminates this problem process. If its really something in the kernel now, I'd call this a large bug.

Any information / help would be great.

### Post by Legend1222 on 2010-07-22
> **arrrghhh said:**
> If you're doing serious data transfers, why aren't you running hardware RAID?


Off topic, but I'll answer.

Under normal circumstances, I'd agree with you. However, I am using software RAID for a few reasons:

1. This is a proof of concept place holder to justify getting a better rack mount server. A bit of backwards logic, but thats how this place runs.

2. Cost. Try to find a hardware raid card for cheap, or even a reasonable amount of money. Can't be done. Everything is hardware assisted raid, which really just translates to proprietary software raid (aka Fake Raid)

3. This server is old/ancient and re-purposed. It only has a PCI expansion slot, so theres going to be a bottle neck.

4. There was no reason for it. There was no performance issue. Until this flush BS started, the CPU load was low, the memory load was low, and the hard drive through put was fine. There was nothing to be gained (performance wise) by hardware raid. I could have added more cameras before this flush BS started.

5. I've never used Linux software raid in a production environment before. Just screwed around with it a little bit here and there. Since this is essentially a testing machine, I figured good time to get some experience with it.

So, there ya go. Five reasons off the top of my head.
