Ok , from what I can see it looks like you are confusing the fact that  you created a site profile for calcium and incorporating it into the  path.

When you create the file /etc/apache2/sites-available/calcium . That  simply gives Apache a point of reference for the data to load about that  site, it has no effect on the path, where the path is effected is when  you declare the webroot directories in the calcium file.


To get to what I THINK you want try doing the following

create a record in your hosts file something like this
 
calcium       serverip

so then if you went to

[http://calcium/Calcium40.pl]("http://calcium/cgi-bin/Calcium40.pl") you would get your results, that you are looking for.

However if you went to [http://your-ip/Calcium40.pl]("http://your-ip/cgi-bin/Calcium40.pl") it should return a not found, since the cgi-bin directories will not be the same.

This may get you where you want, I am a little confused as to what exactly you are trying to accomplish. If you simply want a calcium sub-directory, you would maintain the regular www root and only change the cgi-bin path, then place a subdirectory calcium and associated files under the www root.

No you are absolutely correct you can do this just fine. I will give you a brief example tailor it to your needs though.

Say we want to create site calcium as a subdirectory of the main site and we want to be able to run our perl scripts directly from the root folder (which is actually the sub directory)

First we must create a site for this to do this we will create a profile for the site in /etc/apache2/sites-available (which you have already done).

In this case we will name the site calcium

so we should have our /etc/apache2/sites-available/calcium file looking something like this. (please note I am using the default web root of /var/www change this to fit your set up). I am also using the /home/calcium/cgi-bin directory for the perl scripts (keep in mind if you use encrypted home folders this may be a problem so you may wish to change it to a different folder)

```
<VirtualHost *:80> ##names the virtual host as anything:80 thus [http://localhost:80](http://localhost:80) will follow this profile
        ServerAdmin webmaster@yourdomain
        ServerName Calcium ##Just for giggles only will change anything if you add a dns record for this server. If you do not add a dns record just comment this line out.
        DocumentRoot /var/www ##Sets our web root to whatever we want, we can change this so long as the files/directories are in place and we have permission to readthem
        <Directory />
                Options FollowSymLinks
                AllowOverride None
        </Directory>
        <Directory /var/www/>
                Options -Indexes FollowSymLinks MultiViews
                AllowOverride None
                Order allow,deny
                allow from all
        </Directory>

        ScriptAlias /cgi-bin/ /home/calcium/cgi-bin/ ##Sets our cgi-bin to our custom directory assuming a user was added solely for these perlscripts (which doesn't really matter because they will run as the www-data user anyway. This is the downfall of perl.
        <Directory "/home/calcium/cgi-bin"> ##These permission are set so the scripts are executable.
                AllowOverride None
                Options +ExecCGI -MultiViews +SymLinksIfOwnerMatch
                Order allow,deny
                Allow from all
        </Directory>

        ErrorLog /var/log/apache2/error.log

        # Possible values include: debug, info, notice, warn, error, crit,
        # alert, emerg.
        LogLevel warn

        CustomLog /var/log/apache2/access.log combined
</VirtualHost>

```



Now that we have this set up we will need to place our files properly, to be able to install them into a subdirectory you will create /var/www/calcium in this case this will house the web site's files.

You also must create /home/calcium/cgi-bin.

once that is in place and permissions are correct you should

```

sudo a2ensite calcium
sudo invoke-rc.d apache2 restart

```

This should give you the results you're looking for

[http://localhost/calcium](http://localhost/calcium) will refer to /var/www/calcium
[http://localhost/calcium/cgi-bin](http://localhost/calcium/cgi-bin) will refer to /home/calcium/cgi-bin
[http://localhost/](http://localhost/) will refer to /var/www

if you change the webroot to /var/www/calcium
[http://localhost/](http://localhost/) will refer to /var/www/calcium

Hopefully this makes more sense.
