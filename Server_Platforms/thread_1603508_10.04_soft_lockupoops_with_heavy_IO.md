Kernel:
Linux cluster3 2.6.32-25-generic #45-Ubuntu SMP Sat Oct 16 19:52:42 UTC 2010 x86_64 GNU/Linux

Hi all.
My first shot at Ubuntu on a bunch of new servers.
As the title says there is something going very wrong under load, the triggers appear to be heavy I/O - triggers have been DRBD initial sync, rsync from one to another, etc.
Anything that causes a lot of I/O seems to bring this about.

I have 4 of these servers, all identical hardware and it is happening on all of them.
I started by installing XenServer on them then creating 2 VMs both running 10.04.
Everything went fine and without any issue, I set DRBD up as usual and about an hour into the initial sync of a 1.5TB partition the secondary started spamming messages like:
```
[88348.237793] INFO: task kjournald:22543 blocked for more than 120 seconds.
[88348.237798] "echo 0 > /proc/sys/kernel/hung_task_timeout_secs" disables this message.
```
Now, the process could be any process running... I have seen some of the drbd processes listed as well.
Nothing appeared to work, no command issued would actually do anything.
The other VM on the server was fine.
I was forced to reset this VM and restart it, and it went back to doing it's thing quite merrily.
About 10 minutes later the other VM on a totally different machine did the same thing, also running 10.04.
Once again, no command issued at the console would actually work and I had to reset that VM.
Thinking I had some squirly issue with the servers I installed Debian Lenny directly on two of them, no Xenserver, etc.
I set DRBD up on them as usual and let them sync - not a single issue.
Realizing this had to do with heavy I/O I generated as much I/O as I could without an issue while they ran Debian Lenny.

I blamed XenServer and installed 10.04 as I did Lenny, without any virtualization.
Set up DRBD on both machines and within 10 minutes of their initial sync the secondary node started spamming messages as above and I was forced to hold the power button to reset the server.

I was able to replicate this on 4 different servers so it's unlikely the servers are at fault - they worked with Lenny just fine.

So I went googling....
I found the following that matches the issue to a T:
[https://bugs.launchpad.net/ubuntu/+source/linux/+bug/588046](https://bugs.launchpad.net/ubuntu/+source/linux/+bug/588046)

I want to keep the systems as vanilla as possible and not start compiling my own kernels or monkeying with packages.
Does anyone have any ideas as to what I can or should do until it's fixed OR if there is a workaround?

### Post by dtfinch on 2010-10-22
I've run into this error often (regular servers, not VMs), but only on ext4, not ext3, though I haven't tried ext3 against since 4 became the default. And I haven't seen it happen to kjournald, just other processes that call sync() while some heavy copy operation like rsync is running.

[https://bugzilla.kernel.org/show_bug.cgi?id=14830](https://bugzilla.kernel.org/show_bug.cgi?id=14830)

You might try lowering the io priority of rsync and other long-running io heavy processes. Like "ionice -c3 rsync ...".

In a bind, I've used "kill -STOP pid" to pause a heavy process in order to let something else finish and "kill -CONT pid" to resume it.
