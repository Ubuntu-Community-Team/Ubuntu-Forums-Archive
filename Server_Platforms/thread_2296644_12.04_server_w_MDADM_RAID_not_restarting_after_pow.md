### Post by tgalati4 on 2015-09-28
Clean out the machine and reseat everything.  It is common for a long-running disk to get "sticky" after a sudden power loss.  The grease cools off and the drive's motor is slow to get it up to speed.  So try a few reboots until the drive heats up and starts spinning up within specification.

It's possible that the degraded drive has failed completely.  Print out its SMART parameters and see why it is degraded.  Of course if 2 drives are degraded in a RAID5 then the pool will not mount at all.  So it's possible that you have 2 bad drives.  One you knew about and another one that failed due to the power outage.

### Post by tgalati4 on 2015-09-29
Your Use Case points out the problem with a RAID5 array.  All of the disks are of similar age.  Like light bulbs, when one fails, the others are soon to follow.  For a drive that has been running for a few years, the bearings wear out and the grease becomes like glue when the drive cools down.  You could try heating the slow drive with a hair dryer to get it up to temp (110F or so).  The higher the temperature, the lower the viscosity of the grease and better chance to get the drive spinning.

The key to reviving this RAID5 pool is to get the n-2 drive to spin up and be counted.

Examine the SMART parameters of each disk in the pool.  I bet the others are on their way out as well.

The recreate is the last option but in this case it might be the only one. But that process depends on putting the members in the same order as they were in the array.

You think you can figure out the partition order? I see 3 disks are Seagate and 3 are WD. Does that help? Also the ports where they are connected?

Ubuntu assigns the drive letters in order how BIOS reads the disks on boot. Usually starting from port SATA0 onwards. If the board has two controllers one of them will be read first. If it has only one, the disks will be assigned letters in order from lower to higher sata port. That's why it's good practice to connect the disks in specific order. For example, why your boot disk is not sda? You start with the boot disk and then continue with the raid data disks. That can also help the system to try booting from the correct disk in a multidisk system.

Let me know if you can figure the disk order... I don't know what to do apart from the above. The metadata looks to have lost too much info for reassemble.

I was actually searching around for these kind of situations and found a very detailed post about recovering an array with various options. The guy made a test array and broke it on purpose to try various types of recovery. So that got my hopes high that we could actually save the data, or most of it anyway (depending how the array failed some small data corruption might be impossible to avoid).

As per the raid6 vs raid10 question, I was thinking about that too and I would go for raid6. Like you said yourself, not only that it offers more storage but most important it offers real 2 drive tolerance. raid10 might keep working with 2 drive failure but it depends on the drive positions in the setup. The minimum is at least 1 drive tolerance while 2 drive is not guaranteed.

I don't want to confuse you further, but just to mention that if you are considering redoing the server you can also think about ZFS which is a type of filesystem/setup made especially for storage. Long time it worked only on Unix systems like BSD, Solaris (not on linux), but now since few years ago it is available for ubuntu too. It also has 2 drive tolerance which is called raidz2 and lot of people prefer it for big storage. There is lots of info online and from what I've read it seems better designed than raid, although the benefits might not be too big. Managing it is not too complicated and you should be able to learn it easily. Basically once you create the storage pool(s) you manage all the rest same like standard ubuntu. It is standard ubuntu only with zfs package added so you can create your storage pool.

Whether you want to try to recreate this array or not depends on you and how recent backup you have. If you are OK with some data loss and don't want to waste time, simply redo the new server setup and that's it. But on the other hand trying to save this array will be good practice...
