### Post by endotherm on 2010-09-02
credentials are always stored on the client, not the server, so the issue is on the client end. my recomendation is that you create a unix account for your windows user, and use 'smbpasswd' to register that user with samba. then the user themselves can access the share without the need for alternate credentials. your smb.conf will have to be set for "security = user". 

another thing to try, is instead of creating shortcuts (how did you do that btw?) that you map network drives. these will allow you to add alternate credentials. I have several automatic mapped drives on my win7 boxes that work just fine, though I don;t really have to use alternate credentials as my desktop user is also a user on the fileserver.

### Post by endotherm on 2010-09-02
linux does not integrate the user accounts database with samba, so just because your user exists as a unix user, does not mean that samba recognizes them. 
you can register the unix user 'userX' with samba by entering this:
```

sudo smbpasswd -a userX

```
you will then be prompted to enter your password, and confirm it. after that, your unix user will also be a samba user.

### Post by Gantlett on 2010-09-25
OK guys I found the answer:
What I neglected to mention was that the Windows user logs in to the computer without a password. So the username was identical to the Linux user, but the passwords didn't match. 

I created a password for the Windows user to be identical to the Linux user. From that moment on, the shared folder did no prompt for a password anymore.

Now when I think of the error message that Windows showed, it kinda makes sense: it would immediately say "Wrong password" in the password prompt window. 

So this seems to be a stupidity in the way Windows is designed: the Windows username and password have to match the Linux ones. 

Anyway, case closed :) :popcorn:
