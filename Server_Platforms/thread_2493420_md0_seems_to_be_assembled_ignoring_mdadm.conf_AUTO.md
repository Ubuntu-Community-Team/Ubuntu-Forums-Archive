### Post by darkod on 2023-12-15
The first array created is always md0, then md1 etc. From my experience ubuntu uses md127 is some cases and more as a random value.

In any case I see you do not have any ARRAY definition in the #definition of existing arrays in your mdadm.conf. If you want it to use the same mdX all the time, you should actually create an ARRAY definition there and not use AUTO. Basically don't let the system automatically selecting things for you (like the array name) when you should be in control.

For the ARRAY definition you will need the uuid of the array itself (NOT the uuid of each separate member, and NOT the uuid of the filesystem). You can find this either with 'mdadm -D' or 'blkid'.

After you create the ARRAY definition in mdadm.conf, do the 'update-initramfs -u'.

### Post by darkod on 2023-12-18
After rereading post #3 I still fail to understand.

You have an array in the server and you want to add an older array to copy data over. Is that correct?

If that is so, then you want the opposite of any auto operation. You want to manually assemble the "old array".

Lets say your production array is assembled by members sda1, sdb1, sdc1. And your "old array" after adding the old disks is on sdd1, sde1, sdf1.

If memory serves me right, what you would do is something like:
```
sudo mdadm /dev/md1 --assemble /dev/sdd1 /dev/sde1 /dev/sdf1
```

If your old array is still good and all data on it good, it should have no issues assembling with the above command. After that you can mount it temporarily and copy data off it.

Depending how long you need it for, I actually prefer the manual assemble as opposed to any AUTO statements.

And for any array you want to auto-assemble on boot, definitely put an ARRAY definition in mdadm.conf and delete the 'AUTO -all' line.

### Post by darkod on 2023-12-19
I still think you are doing it wrong. You say you created two new arrays on the new disks. Why their ARRAY definitions are not in mdadm.conf?

Instead of focusing so much on 'AUTO -all' I would rather focus on the following which I think is true: no arrays will be assembled on boot unless they have an ARRAY definition in mdadm.conf. So just put definitions in mdadm.conf for the arrays you want autoassembled (the new ones), and don't for the ones you do not (the old ones). It is as simple as that.

So what I think you should do is:
1). Remove the AUTO -all line from mdadm.conf.
2). Add ARRAY definition in mdadm.conf for each new array you created on the new disks (you want them to assemble automatically on boot, right?).
3). Execute '-update-initramfs -u' for the system to process the above changes. In this situations on boot only the new arrays should be assembled even if the old disks are present too.
4). Assemble and mount the old arrays manually as per needed to copy data.

You think you are protecting your old data but if you are messing too much with mdadm actually you elevate the risk something to go bad. I believe the above recipe should serve you for getting your new arrays running and copying data from the old ones.
