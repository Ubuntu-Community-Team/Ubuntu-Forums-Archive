### Post by SeijiSensei on 2012-02-26
You need to add an iptables rules that intercepts outbound traffic and pushes it through the proxy like this:

```
iptables -A PREROUTING -i eth1 -o eth0 -p tcp --dport 80 -j REDIRECT --to-ports 3128
```

That takes requests originating on the LAN for port 80 on remote servers and redirects them to the proxy.  If you already have a set of iptables firewalling rules established, just add this one to the list.  If you have no firewalling rules, add this command to the file /etc/rc.local so it will be run automatically at boot.  If you try running this command from the prompt, you'll need to use "sudo" since only root can modify the iptables ruleset.

I would avoid trying to proxy SSL (port 443) requests until you have lot more experience with Squid.  Trying to proxy port 443 will generate "man-in-the-middle" warnings from browsers since the certificate chain cannot be verified.  There are [advanced technologies]("http://wiki.squid-cache.org/Features/SslBump") to handle this, but I'd start slowly and leave the port 443 requests alone.

### Post by collisionystm on 2012-02-28
In my similar situation...

I did not want to block websites however I wanted to log what sites were being visited. I did not want to configure each client machine or give them any indication of what I was doing.

I used webmin, squid and dansguardian on an ubuntu 10.04.4 server. I use UFW by the way.

I simply installed squid, left default. Started the service. Install dansguardian, configured it and started it. Tested that it allowed webtraffic by first setting my own browser manually. After testing I just changed my firewall rules to forward all http traffic through dansguardian so it will be logged.

Here is the rule I placed in my before.rules in UFW.

*nat
:PREROUTING ACCEPT [0:0]
-A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080
COMMIT

80 is HTTP, 8080 is my dansguardian port.


You can use the same thing in regular IPtables but just letting you know this works fine in UFW as well. Good luck!

EDIT:

I later started blocking high traffic websites in my bind config.
facebook, pandora, myspace, twitter...etc. Now if someone tries to access one of these it just redirects to the loopback.

```
http_port 192.168.2.1:3128 transparent
```

I think the problem is here.  I believe that REDIRECT will send the packets to port 3128 on localhost (127.0.0.1), not to 192.168.2.1.

I just checked the configuration of one of the transparent proxies I manage, and I have just

```
http_port 3128 transparent
```

which has the proxy listen on all interfaces.

Give that a try and see if your problem goes away.
