### Post by Lars Noodén on 2015-10-17
Instead of .htaccess, it should go in your web site's configuration file if you have access.  The use of .htaccess should be avoided if at all possible, it spreads out the configuration too much for one, but unfortunately there are many outdated tutorials out there using .htaccess from days when multiple, unrelated sites shared the same Apache 1.3.x server.  You should find your site's configuration file in /etc/apache2/sites-enabled/ in one of the files there.  

About the regex, what are you trying to do?   If it's a general redirect of all URIs then you'd probably want something like this:

```
 RedirectMatch 301 (.*)$ https://$1

```

### Post by Lars Noodén on 2015-10-17
> **maclenin said:**
> With the regex - I was trying to match either *[www.domain.com](www.domain.com)* or *domiain.com* and pass either string to https:// via *$1*


In the method you are using, the host name never comes into play only the path.  If you use [RedirectMatch](http://httpd.apache.org/docs/2.4/mod/mod_alias.html#redirectmatch), then you can redirect *w**ww.domain.com* and *domain.com* to one or the other, but not both. 

```

 RedirectMatch 301 (.*)$ https://*domain.com*$1

```

That will redirect either to the HTTPS service on *domain.com*.  As far as I know, that is the recommended way: choose one target and send both there.  

If you want fancier, to keep either *w**ww.domain.com* or *domain.com* in the URI, then you need to use the extremely powerful rewrite engine in mod_rewrite using [RewriteCond](http://httpd.apache.org/docs/2.4/mod/mod_rewrite.html#rewritecond) and co.  

```

RewriteEngine On
RewriteCond %{HTTPS} !=on
RewriteRule ^/?(.*) https://%{SERVER_NAME}/$1 [R,L]

```

The [Redirect](http://httpd.apache.org/docs/2.4/mod/mod_alias.html#redirect) directive can start with the HTTP status code:

```

Redirect 301 / https://domain.com/

```

### Post by SeijiSensei on 2015-10-18
If all you want to do is redirect plain HTTP requests to HTTPS, the method you presented above always works for me:
```

<VirtualHost *:80>
ServerAdmin xxx
ServerName domain.com
Redirect / https://domain.com
</VirtualHost>

```
[By default Apache will generate a 302]("https://httpd.apache.org/docs/2.4/mod/mod_alias.html#redirect") (temporary redirect) response before redirecting.
```

xx.xx.xx.xx  - - [18/Oct/2015:14:57:14 +0000] "GET / HTTP/1.1" 302 323

```
but you can force a 301 using the "Redirect 301" syntax.  In practice I don't see any difference.  Maybe browsers cache the 301 redirect as the target URL but not if they receive a 302?  Do search engines differentiate between the two and no longer bother to spider the original host after a 301?
