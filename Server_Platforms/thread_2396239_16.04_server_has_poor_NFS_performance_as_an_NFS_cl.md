### Post by clarkgr on 2018-07-12
I have one Ubuntu 16.04 server that is NFS mounting a file share from a storage array.  This particular Ubuntu server is the only server (out of 38 servers) showing NFS performance problems.  I've spent quite a bit of time searching for posts here and on other forums for clues.  I have found many posts complaining about NFS performance problems specifically in 16.04.  None of the solutions apply to me.  The NFS mount is defined in /etc/fstab and all block size and other options are specified per instructions from my storage vendor.  I've been running nfsiostat.py to get latency statistics and to test potential fixes.  So here's the weird thing; the latency problem drops to almost nothing once I fire up a tcpdump process.  It doesn't even matter what I am dumping.  For now I have a tcpdump process running that produces no output such as "tcpdump host 1.2.3.4" 

Obviously, this is not an ideal solution for a production server.  There is still some marginal latency but it's not significant.  I've set up hundreds of servers to NFS mount file shares from storage arrays and I have never seen this before.  Why would running a tcpdump process clear up the latency issues?  This problem only affects this one 16.04 server.  Any clues on where to look next?
