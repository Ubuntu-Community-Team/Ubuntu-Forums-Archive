### Post by disabledaccount on 2011-01-07
I can see that you have Raid10 with 3 partitions: (edit:)sde5[3] sdc5[1] sdd5[2] - that means the array must be started as "degraded". If something will go wrong with assembling degraded array during boot then Grub wont have access to your Root device.
What results comes from: >mdadm -D /dev/md1

I'm not completely sure how much difference it makes, but in case of problems with grub I go different way:
- mount /root device
- mount /boot device inside /root (as it would be mounted normally)
> grub-install --root-directory=</path_to/mounted_root> --recheck <--nofloppy>

...and it allways worked fine for me.

### Post by gagou7 on 2011-02-07
Sorry for my bad english, I'm French...

I don't think the problem is the raid array. If something wrong with your raid array, you will have an error like "error: no file found" at boot but not a hard reset.

I have a similar problem where grub2 freeze and reboot before menu display. I tried many different things and maybe my solution can work for you.

I used [Super Grub Disk]("http://prdownload.berlios.de/supergrub/super_grub_disk_hybrid-1.98s1.iso") for booting (option "detect OS") on my computer and make the change below.

First, uninstall grub, grub2, grub-pc,... with "--purge" for removing config files.
You have an raid array, make sure your "/etc/mdadm/mdadm.conf" contains  array's definitions, if not, do this "mdadm --examine --scan | sudo tee -a /etc/mdadm/mdadm.conf".
Now, install grub2 and do a "grub-update", "update-initramfs -u" and "grub-install" on one or more disk, like you want.

For me, it's here that does all difference. I know it's not very clean to modify "grub.cfg" directly, but you can try like me and if it doesn't work, regenerated a new one with "grub-update".
Edit "/boot/grub/grub.cfg" and comment (just two lines) the line with "terminal_output gfxterm" and "insmod gettext".

You must verify three things too:
- the first two  "set root='(X)'" must be with X your 'root' device  (in my case "set root='(md0)'" where md0 is a raid 1 array with two disk and contain /etc,/boot,/usr,...). In your case, md1 is the '/' device.
- the other "set root='(X)'" (in "menuentry") must be with X your device containing boot directory. In your case, md0 is directly the boot directory, so you don't have to put "/boot/" prefix for "linux" and "initrd" command.
- all "search --no-floppy ..." must have the uuid corresponding  to the disk set in line before.

Finally, I had a problem with "fd0 read error" on boot, simply disable floppy in your BIOS.


If it doesn't work, simply edit "grub.cfg" file and put many "echo '1'", "echo '2'", ... after each line. Then, just reboot and see after which displaying number grub hard reset. It's like that I have debugging my problem and found two instruction that freezed my grub loader.

Hope this help you.

### Post by YesWeCan on 2011-02-19
It works! :D My RAID 10 is fully functional and boots fine with sda missing and is now spending the next 3 hours resyncing. The problem I had was that my Ubuntu will not boot a degraded RAID by default. So it is necessary to add '**bootdegraded=true**' to the grub kernel command options. So obvious ;)
Always handy is the Ubuntu Server Guide: [https://help.ubuntu.com/10.04/serverguide/C/advanced-installation.html](https://help.ubuntu.com/10.04/serverguide/C/advanced-installation.html)


:idea: My humble suggestions for those with the cycling reset problem:

Grub appears to work like this. It is mostly installed outside your RAID 1 arrays, in both the MBR and the sectors immediately after the MBR. While booting it seeks out files it needs in the /boot partition. If it cannot find the right files it will throw its toys out of its pram.

The key symptoms I have read are that booting off any disk works provided the sda is present somewhere. This means all the grub installations at MBR are good. But remove the sda disk and the resetting happens before even a grub menu appears. So this strongly suggests that grub is failing to find any or all of the correct /boot files. It needs these files before it can generate a menu.

But how is it possible for the /boot to be only good on sda if they are in a RAID1 array? Good question. Well two observations may support this possibility:
a) With a RAID 1 array mdadm only needs to read one disk. I did a quick experiment and noticed mdadm always reading off the same disk. So even if you boot off sdc I think mdadm will still read /boot off sda. Only if sda is missing will another /boot be read.
b) If, somehow, you have modified the /boot files only on sda mdadm won't necessarily know and won't resync.

This is an odd theory but so are your symptoms. You should be able to check this by stopping the RAID 1 array and mounting the /boot partitions separately and doing a diff -R on them.

In any case, for good measure, you may want to rebuild your /boot files from scratch.
Umount the RAID1 so you have the original /boot files.
Update them: run 'update-initramfs -u' and 'update-grub'
Rebuild a fresh RAID 1 and copy the boot files to it. Then remount it at /boot.
Do another 'update-grub' so it knows about the RAID 1.
Do 'grub-install /dev/sdn' for n=a,b,c...
Reboot to check all is ok.
Try booting without sda. Tip: save yourself an unecessary resync by not adding 'bootdegraded=true' to the grub command...if it boots grub ok and starts the ubuntu boot it is probably fine. It will fall into the Busybox prompt. If you let it complete the boot in a degraded state it will insist on a resync when you reconnect the missing disk. Which is very boring :P.

Hope this helps.

[edit] Of course, if it is the case that Grub will boot a RAID 10 directly, you can save yourself a lot of trouble and delete your RAID 1 completely.
