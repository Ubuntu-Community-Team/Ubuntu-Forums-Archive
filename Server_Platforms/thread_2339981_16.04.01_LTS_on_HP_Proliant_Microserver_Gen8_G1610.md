How you use and set up your server is of course your own decosion...

But I would go a step back. If you used the bios raid setup, I would delete the raid arrays you created and leave the disks as simple AHCI mode, no raid from the onboard controller. Linux works quite great with its own software raid and that is not dependent on HW.

One thing you need to know and plan in advance, is that with SW raid there is no partitioning of the arrays. You create the arrays the size you want them, no splitting afterwards... ### Post by darkod on 2016-10-15
If you mean whether it's ok the OS disk to be sdb, yes it is. It doesn't matter at all. It doesn't have to be the first disk (sda).

And yes, LVM will let you combine any physical disks (devices) into any logical storage you desire. So, 3x 2TB disks will give you 6TB continuous storage.

In fact, with the raid1 you still had to use LVM, because raid1 mirror of 2x 2TB disks will give you 2TB device, and then the other 2x 2TB disks would give you again 2TB device. To combine that into continuous 4TB you need to use LVM (mdadm arrays can serve as physical devices in LVM terms).

Also, here is another idea to think about: You can think about whether to continue running the OS from the SD card or put it on the HDDs. You can "slice" a small part of them (like 16GB or 32GB) and create one raid1 with 4x members (all the disks) and use that as your root partition for the OS. That way even if one disk fails, the OS keeps running. In fact, even if three disks fail the OS will keep running. Of course, the data won't be there with so many disks failed, but the OS will be there. :)

The rest of the disks can be partition into one small swap partition (again on all disks same size to keep symmetry) and one big partition to use as you see fit. It can be two raid1 arrays combined into LVM, or 3x partition into LVM for a big data partition, and 1x for backup.

### Post by darkod on 2016-10-15
That partitioning of the 2TB disk actually looks like some old installation or something... Yes, you partition the drives even if you want only one big partition. It would go something like this:
```
sudo parted /dev/sda
unit MiB
```

That will open parted prompt for /dev/sda and set MiB as unit to work with. Then continue by creating new msdos partition table and one big primary partition (unless you want swap partition on the disks too). The end of the partition you can calculate from the total MiB size of the disk that you can see with the print command.
```
print   (note down the total disk size in MiB)
mklabel msdos
mkpart primary 1 <end>
print   (check the partition you created looks ok)
quit   (quit parted prompt)
```

For <end> I usually use the total size of the disk minus 10-15MiB. I leave that at the end as spare because all 2TB disks are not identical to the last sector and in the future if one disk fails and you buy new one to replace it, it might be few MiB smaller. In such case raid will not allow you to add it to the array.

Basically you need to do the above for all 4 disks. That will make one big primary partition on them.

### Post by darkod on 2016-10-15
What I meant by 2x raid is this: You have 4x HDDs. If you do a single raid1 array, that means raid1 with 4x 2TB members which makes total of only 2TB usable. Too little...

So, you take disk1 and disk2 and make raid1 2x 2TB = 2TB usable
Then disk3 and disk4 and make different raid1 2x 2TB = 2TB usable

Then you use LVM to join those 2TB + 2TB = 4TB usable.

That's the way to get maximum of your storage using raid1. Lets avoid any confusion, I'm talking about 4TB total usable space, not about 4TB for data and 4TB for its backup. You can use those 4TB as you see fit, but that would be the total. If you want to, you don't even need to join the two 2TB arrays, you can use one for primary data and the other 2TB for backup. In that case you don't even need to use LVM.

If you don't use any raid, you can get more total storage but remember that any disk failure means loss of data.

### Post by darkod on 2016-10-15
So, if you want to create two arrays as suggested, that would be using sda1 and sdc1 for one, and sdd1 and sde1 for the other. Somethine like:
```
sudo mdadm --create --verbose --level=raid1 --raid-devices=2 /dev/md0 /dev/sda1 /dev/sdc1
sudo mdadm --create --verbose --level=raid1 --raid-devices=2 /dev/md1 /dev/sdd1 /dev/sde1
```

That will create two raid1 arrays of 2TB, md0 and md1. You can check their progress from time to time with:
```
cat /proc/mdstat
```

You can start using them even before the sync is fully finished, however you might wanna wait for it to finish completely first.

Now that you have md0 and md1, you need to prepare them for LVM and create one single VG named vg for example. You can also call if vg1 if you think you will make more in the future, your choice... Just ammend the commands accordingly.
```
sudo pvcreate /dev/md0
sudo pvcreate /dev/md1
sudo vgcreate vg /dev/md0 /dev/md1
```

That creates the VG using full size of both md0 and md1. You can check it out with:
```
sudo vgdisplay
```

Now you need to create logical volumes, one or more, inside that VG. If you want a single big logical volume, lets call it data for example, and the important thing is that you don't need to give it the whole space right away. LVM is easy to expand later, so you can start off smaller and grow as necessary. For example 2TB for start...
```
sudo lvcreate --name data --size 2T vg
```

That create a LV named data with size 2TB into the VG vg. After that the device name will be /dev/vg/data. You need to format it to be able to use it:
```
sudo mkfs.ext4 /dev/vg/data
```

After that you can mount it and use it...

You will need to add the array definitions into mdadm.conf so that the arrays are assembled at reboot, I'll post that tomorrow, gotta go now...

### Post by darkod on 2016-10-16
It is not a good idea to put mount points inside anyone's home folder. Especially for big shared data.

Any empty folder can serve as mount point, the important thing is to be empty. And you control permissions on that folder to allow/deny users access to that shared data. So, you can create the folder /mnt/data as you suggested, or even only /data. That's shorter to type anywhere where you need it.

And yes, it would be:
```
sudo mkdir /data
sudo mount /dev/vg/data /data
```

Note that creating it with sudo, the folder /data will have root as owner and strict permissions. You will need to adjust owner and permissions of /data so other users can use it.

For putting the array definitions in mdadm.conf, you first need to take a look at the UUIDs with:
```
sudo blkid
```

You will see all partitions members of the same array have common UUID, and another UUID_SUB which is unique. If you are confused by the blkid output post it and we'll help you figure it out. You use the common array UUID for the array definition in /etc/mdadm/mdadm.conf, like:
```
ARRAY /dev/md0 UUID=xx:xx:xx:xx
```

Note that in the above UUID I shortened it, it has 32 characters and in my mdadm.conf they are grouped in four groups of 8 chars separated by :

You need to do that for both arrays. After you save and close the file, you need to update the initramfs:
```
sudo update-initramfs -u
```

That should be enough the arrays to assemble after a reboot. Try a reboot and check it out. If they don't assemble automatically, don't panic, it's easy to re-assemble them again without re-creating them from scratch.

### Post by darkod on 2016-10-17
I think that would change only the group owner, not the user owner too.

You can check ownership and permissions with:
```
ls -l /
```

That will list all folders in the root. You can filter for your data folder with grep:
```
ls -l / | grep data
```

The permissions have three groups of rwx permissions which means Read-Write-eXecute... And the three groupings from left to right are user owner-group owner-all others.

So, for example, if you have a file/folder owned by root and permissions rw------- only root has RW and no one else nothing. Because for owner (root) you have rw-, for group owner (which might be root group too) you have ---, and for all others you have ---.

It is always recommended to give only necessary permissions, not to give RW to all without need. For example, if you want to control who logs in from the windows clients, and if you need them to use passwords, then you can put all users in one group, and set that group to own the share:
```
sudo chgrp <name> /data
```

Then make sure the group permissions are rwx (for folders the x means listing the folder content, and is needed).

That way all users belonging to that group will have RW inside that share.

### Post by darkod on 2017-11-01
It is a long and old thread and I don't have time to read everything again now. Just to remind you that disks should NOT be pulled out if you were using the built-in server raid (either HW or bios raid). Put them back again in EXACTLY the same slots as they were.

If you were using SW mdadm raid taking disks out is not critical like with HW raid. You can even put them into different machine and assemble the arrays there.

If you had the OS on a SD card and it failed, no big deal. The data should be safe and accessible. You can use Ubuntu Desktop liveCD (or liveUSB) to boot it into live mode (Try Ubuntu) and check the content of the disks. That is the purpose of putting your OS volume and your data separately. If the OS goes down, the data should still be safe and intact.

The problem with OS on flash memory is that usually that memory is not designed for high number of read/write cycles, like a HDD is. So it has higher probability of failure even though the ubuntu server OS should be writing that much data to the OS volume constantly.

In live mode mdadm doesn't automatically assemble the arrays. In fact the package is not even included in live mode, you have to add it each time you start new live session.

That should assemble all arrays it can find. After that if you need to activate the LVM:

sudo vgchange -ay
```

That should allow you to look into the LVM volumes. Be careful...
