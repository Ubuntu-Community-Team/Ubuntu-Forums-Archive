### Post by The Cog on 2017-02-14
This rule is allowing friends to access family. I think it needs removing:
```
-A FORWARD -d 192.168.244.0/24 -i tun0 -j ACCEPT
```

I suggest adding a rule that allows forwarding of responses to connections that have already established (like you do with the INPUT chain).
Then you only have to worry about who can initiate connections to who, and the reverse direction is automatically taken care of:
```
-A FORWARD -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
```

See this note on the OpenVPN client-to-client option. You my not need to explicitly allow family-to-family in your iptables rules, but may be able to configure that OpenVpn server to allow it internally:
[http://backreference.org/2010/05/02/controlling-client-to-client-connections-in-openvpn/](http://backreference.org/2010/05/02/controlling-client-to-client-connections-in-openvpn/)

### Post by darkod on 2017-02-14
I would do what you want another way... You use destination subnets in iptables but instead why not do it with source subnets. You know which traffic goes where and what you want to allow.

For example, you want 243 to go out only to the internet, right? Which is eth0. So that would be:
```
-A FORWARD -o eth0 -s 192.168.243.0/24 -j ACCEPT
```

That should give them access only to eth0. Also by removing the pushed route you block them any access to 244, both by routing and by iptables.

But for 244 you want access to both eth0 and tun0 (towards 243 clients):
```
-A FORWARD -o eth0 -s 192.168.244.0/24 -j ACCEPT
-A FORWARD -o tun0 -s 192.168.244.0/24 -j ACCEPT
```

Something like that should do it. (I think)

PS. From rereading the thread it seems you don't want 243 to go out to the internet. In such case ignore the first forward rule about 243 above. For 243 client to client only, it should work as it is.

You also don't need rules like:
```
-A FORWARD -i eth0 -o tun1 -j ACCEPT
```

You should do it with state RELATED,ESTABLISHED like The Cog said so. Once you have that rule active for the forward chain too, you don't need -i eth0 -o tun1 because no one needs to establish a connection from outside to the tunnel.
