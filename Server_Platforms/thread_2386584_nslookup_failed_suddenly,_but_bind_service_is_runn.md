### Post by darkod on 2018-03-08
When the dns fails, does it work for the clients or it fails too?

If it fails only for the server itself, then it is some problem that does not allow the server to query itself. Because if it works for the clients that means the dns service in general is working OK.

Also, if you have no special config in dns, you can set the server to use google dns instead of itself. If you don't have private zones that you want queried from the server, then using public dns like google or private one makes no difference.

I still don't see why you needed to put 127.0.0.1 in listen-on if you are using the 192.168.0.2 IP. On the other hand, if you are using 127.0.0.1 as nameserver then it has to listen on that one too.

I have no more ideas right now, when a failure happens usually you should be able to find something in the logs. You said you couldn't...

### Post by darkod on 2018-03-09
I can't find anything related to those error messages either.

I would start considering backing up all relevant bind config files, and then purging and reinstalling bind. After that you can redo the configuration easily using the config files. And checking them for any errors or wrong config too, not just copying them back.

If bind was installed directly by you, this should work.

If it was installed as part of a larger script to install zimbra or similar, you need to check what is the best way forward. And check zimbra forums for these errors too. You can't treat it as default ubuntu if you installed further software using preprepared scripts...
