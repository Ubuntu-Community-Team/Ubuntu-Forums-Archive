Agreed - your post reminded me of another cool thing about iPerf. Remember that the "server" (parameter **-s**) is actually the *receiving* part of the test, while the "client" (parameter **-c**) is the *sender*.

Now on the client/sender you can use the **-F /path/to/file** parameter to read the data-to-be-sent from some file, instead of generating the data on-the-fly.

So in the OP's case, he can set up iPerf's server/receiver part on the workstation:

```
user@workstation:/~$ **iperf -s**
```

and send for example 1000MBytes (parameter **-n 1000M**) from a file from the Linux server with 

```
user@server:/~$ **iperf -c <workstation's IP> -n 1000M -F /path/to/samplefile -i 5**
```

where **-i 5** will write out stats every 5 seconds.

Like this, you integrate the server's disk I/O performance (however, watch out for caching effects!) into the test, but leave any file sharing protocol overhead out of it.

Just to clarify on IPTOS_..., as I can see them recommended every so often, with some vague references to "routing performance".

These remain irrelevant as long as you don't go across a routing device with your SMB connection. They set the type-of-service bit flags in the IP header, and will only ever come into play IF there is a routing device in the path between source and destination AND it has QoS configured AND is respecting (and not overwriting) the ToS bits.

As long as you stay within the LAN (or VLAN at that), the traffic is only forwarded by switches, and these are blind to IP headers.
