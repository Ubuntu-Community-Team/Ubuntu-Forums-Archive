### Post by clubsoda on 2008-12-22
Linux/Debian/Ubuntu is running at a fever pitch of development currently and the changes are not minor. Automatic configuration is the vogue, and the mere presence of a configuration file from a previous installation can throw a spanner in the works.  

The time it takes to track down these rogue configuration files means it isn´t sensible to upgrade any more. The idea that you´ll lose all your precious settings when installing from scratch isn´t valid because hardware detection is improving too, so many of those custom settings are no longer needed.

IMO, of course.  I´m changing my strategy to preserve /home and ditch the rest in future.

### Post by Cracauer on 2008-12-22
I don't use OSes that force me to do a complete reinstall, and I stay pretty up-to-date.

When my home server/workstation was running FreeBSD it went for 7 years on whatever the stable branch was, up-to-date, without reinstall.  My FreeBSD-current development installation is probably even older by now.

Ubuntu and Debian seem to take it quite well, too.  Fedora not so much but I don't use that anymore.

I should mention, though, that I pull several critical components out of whatever distribution I use.  I have a Mozilla chain, a xine/mplayer/mp3 chain, SBCL/elisp/SLIME and a number of minor tools in trees separate from /usr.  I also start a number of services manually, I have a /etc/rc script that kills whatever apache, tftpd and dhcpd was started by the OS and then starts them the way I want.  I don't even bother to look at what the OS wants me to do to configure them.  But I have constantly reduced the number of these cases, for example I don't have to use a custom apache build anymore.  Things are getting smoother and smoother, both Debian (and hence Ubuntu) and FreeBSD invest some serious brainpower into making update conflict merges smooth.
