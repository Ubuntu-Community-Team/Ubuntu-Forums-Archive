I think now would be a good time to also point out the difference between the two main desktop releases available from Ubuntu:

- 6.06(.1) LTS:  This version is widely accepted to be the more stable of the two.  This one gets updated regularly with security updates and bugfixes but doesn't get all the flash new features right away.  Before it was released it was tested far more than the "6-monthly" releases, of which Feisty is one, which is why it has been guaranteed support for longer than any other release.  If you are looking for a very stable system which has been used by far more people than recent releases then go for this one.  This is what I am running on my primary home system and I am happy with it.  I test new releases on my spare machine (although I understand a lot of people don't have the luxury of more than one PC).  I would liken this release to Debian's "stable" version.

- 6-monthly "bleeding edge releases".  This includes Feisty Fawn.  These releases are the bleeding edge / give the users the latest and greatest ground-breaking desktop features releases.  These are released amazingly quickly especially considering the myriad of fancy new features released.  Many of these new features often include changes to hardware-related software, which is why I suspect so many people have problems.  In my opinion, considering how many new features are included in each release in this category it is truly amazing how well they do in each release.  I would liken this more to Debian's "unstable" update path.

I think Ubuntu is doing a great service to us by providing 6.06 LTS - a version that we can "fall back on" when the latest version doesn't quite work for us.  I would love for you to install this version and let the Ubuntu community help you get Feisty Fawn working if you are still interested in trying it out.

### Post by H264 on 2007-05-17
I would like to throw in one more thing...

Software is never _*Ever*_ perfect. especially with how complex things are now a days where computers have billions (trillions?) of transistors inside. Chances are that one of them might not be in the right place. Same thing for software, with billions, perhaps trillions of lines of code to make up an OS, chances are that one or more lines of code are not in the right place.

Also it looks like it is a kernel problem... the linux kernel is known for it's very rapid dev times, chances are that something did not get properly tested before being released. Somewhere I read that the next major release of the linux kernel is getting a complete overhaul, which is also not surprising because those are not uncommon either.

Keep this in mind too... If I want to make a USB device, lets say it is an external hard drive enclosure, I am confronted with the option to conform to the USB standard or not. If I conform to the standard, chances are that the OS already has generic USB drivers for that device I already made. however if I don't conform to the standard, I will probably have to make my own drivers for it, and limiting myself to how many different OS's I can reach with my driver.
External USB HDs are typically made according to the USB standards (I don't know of any that are not). However that are many other devices that have a standard, but don't conform to, because it works fine with their windows driver they made, and/or it just works fine in windows (can anyone say winmodem?).
I am not saying that the problem you are talking about is related to the company's not making their hardware according to standards, but I am saying that the biggest problems come from drivers not being available due to people not caring about standards.

Just keep those things in mind when you run into problems with software and hardware not working well together next time
