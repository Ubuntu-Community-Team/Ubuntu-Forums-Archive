### Post by DakotaBlue on 2007-11-13
The mythical (and real) grandmothers are not the model to use - they will always depend on others to "set up" their computer and to install new stuff.  If the "others" they rely on are Linux experts, they will have no issues.

"Out of the box" is also only one issue.  Continuing use is another.

The real model to use is the typical home and small business computer user. There is no "IT" department. There is no "nephew" or "grandson" who is the expert. There are just people who want to use computers for ordinary things.

The very thing that causes much of the security holes in Windows is the thing that has been a major contributor to its success - the default "owner" account has "root" privilege.  And, the software Microsoft and other have built on top of this model has allowed ordinary users to add-to and evolve their machines without having to learn much about the internals of the OS.  No user accounts, no passwords, no terminal command language... just start it up and it runs.  Just plug in your new device and it works.

Microsoft has a much more difficult challenge with system security than any UNIX derivative (including Linux).  Linux,(with a few exceptions) insists it is a multi-user environment, where users are never to be trusted.  This makes doing nearly anything seem difficult and complicated (and many times, the perception is accurate).  

Windows, OTOH, insists that users are to be trusted and just tries to prevent security breaches from outside the computer.  Of course, telling the difference between the local, trusted, user and the outside attack is extremely difficult when you allow the local user to do anything he wants to.

Take a very simple example. Imagine a user who installed the latest and greatest desktop ease of use distro of Linux 4 years ago.  He got his nephew to get it all set up and working.  His nephew went off to college.  Uncle Joe now wants to buy one of those new mp3 players everyone is using and use his computer to keep the music library and sync with his player. Uh oh.  Better get the nephew on the phone.  Unfortunately, the nephew has no hope of talking Uncle Joe through a bunch of sudos, etc. 

Same scenario with XP Home, instead.  Uncle Joe follows the mp3 players instructions, loads the CD, plugs it in, click on the SYNC tab on WMP, and off he goes.

This is why Linux will never replace Windows (or even OS-X) on the ordinary desktop.  Linux will never allow the start up and run model, especially where that auto-logs into the root account.  Windows has always done that for its "home" use distributions.  In fact, trying to restrict that some with Vista has led to howls of complaints.

Multi-user UNIX is not the ideal OS for home use, and never will be.

JMO.

### Post by DakotaBlue on 2007-11-16
It is about as constructive as I can make it. Here it is in summary: a part of Windows success is due to its "trusted user" model.  A part of Linux's problem with wider adoption is due to its "untrust-worthy user" model.  Except for security from exteral attack, there is diminished reason for continuing the "untrust-worthy user" model for single-user systems. UNIX was invented in the days of large corporate mainframes where 50 or more individual users would be logged onto one computer. ### Post by aysiu on 2007-11-16
What you have brought into this discussion is a huge red herring: **permissions and passwords have *nothing* to do with MP3 players working or not working.** You have yet to establish a connection between security models and the addition of hard drives or peripheral devices.

Right now, in Linux, you plug an *external* peripheral drive, and it's detected and shows up as a drive on your desktop. It doesn't require the use of the terminal or passwords.

The fact that Ubuntu doesn't make it easy for you to have direct access to new *internal* hard drives is an interface implementation issue, not a permissions/terminal issue. It would be very possible for Ubuntu devs to keep **the exact same security model** and allow you, once you plug in an internal hard drive, to get a prompt that says "This disk isn't properly formatted. Would you like to format it to be able to use it? Warning: this will erase the disk's current contents."

I've seen way too many Windows users screw up their systems and waste hours with keeping it slow, reinstalling, or having someone else clean up the mess to believe there are time savings in dealing with that over just implementing some good security and prompting people for their passwords every once in a while. On the Windows system I have to use at work, I deliberately make myself a limited user for everyday use and keep a separate administrator account for the very few times I need to accomplish administrative tasks. It is no accident that my computer has far fewer problems than my co-workers' computers. 

But practicing good security practices in Windows XP (I guess this has changed for Vista?) is not easy. Limited users sometimes cannot delete desktop icons (even when the desktop cleanup wizard prompts them to). Limited users cannot define keyboard shortcuts for applications in the Start Menu. Limited users cannot *gksudo* Explorer or certain Control Panel items (like Printers & Faxes). And when you use the *Run as...* command to run Windows Updates as an administrator, the updates do not fully install. Worse yet, if you're on a domain, you cannot use fast user switching. So anything that cannot be *Run as...* properly (Windows updates, Explorer) means logging out completely from one account and then logging into another account to make the change, then logging out and logging in again.

When I install software in Ubuntu, I go to Add/Remove, enter my password, click a few boxes, and then wait for the software to install. When my wife installs software in Mac OS X, she double-clicks the .dmg file, sometimes is prompted for her password, and drags the application icon to the Applications folder. I never hear her complain about having to enter her password to authenticate for software installations.

In Windows XP, if I'm using a limited user account (there is no *sudo* in XP, unfortunately) and double-click a .exe to install software, rather than getting a password prompt, I'm simply told I do not have permission to install the software--or, worse yet, the software appears to install, but I get an error when the setup.exe doesn't have access to certain system files. And if I run as administrator all the time, all hell can break loose, and it's very difficult to fix the system.

In other words, with Windows XP, you have to sacrifice security for ease of use or ease of use for security. In Mac OS X and Ubuntu (which both use *sudo* for the security model), you don't have to sacrifice anything. You can have ease of use *and* security.
