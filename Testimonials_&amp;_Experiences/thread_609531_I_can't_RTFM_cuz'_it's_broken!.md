### Post by keyboardashtray on 2007-11-11
In the middle of all of the glowing positives about my *buntu experience in the past few months, one glaring glitch that comes to mind is the help programs.

Ubuntu and Kubuntu come across as ultra-professional in almost every aspect I've seen so far except the Help applications, document installation and program document integration. How does this come across to the new user, when one of their first actions is likely firing up the local help? 

*The documentation is quality.* I'm not saying that the documentation is bad. What sucks is the method in which it is handled, which from my experience is full of bugs and loose ends.

Most of my experience is with Gnome, so I'll start there. I see multiple dead-end links, both when searching from within Yelp, and from links in programs, including some fresh Ubuntu install programs but especially downloaded programs. In fact, I'd say it's more the rule than the exception that the help link doesn't work from within a downloaded program.

Additionally, Yelp doesn't seem all that powerful of a help program. Part of that is Ubuntu's implementation of it: the table of contents could link to an actual table of contents, with some programs listed, instead of only an intro to Ubuntu (which, I'll grant you, is a well written intro - still.)

I'm a little more impressed by KDE Help Center, which on the surface looks like it has the potential to be exactly what a help program should be. A table of contents, a glossary, a search. Nice comprehensive manuals. However, dive any deeper than the surface, and you'll see that the default install, right off the bat, does not have a functioning search.

How does this slip by? We're not talking about some obscure preference option in the fourth tab of the advanced section of X program you might never use. We're talking about the search function in the help program. To me this seems a no-brainer.

Of course, I'm getting a lot of dead links to help in KDE too. Is the standard directory for help so screwed up? Is there some giant underground battle between Distros and DE's about where to put help files?

Either the documentation programs are a very low priority, or the system is inherently flawed. And honestly I'm leaning both. 

Low priority in that I've gone to Launchpad, Bugzilla, and the Kde bug center, and there are some bugs years (plural mind you) old dealing with this. Both fundamental and simple bugs - which in my opinion is just as bad in either case, because you'd think the simple ones, like a program pointing to the wrong location, would be easily fixed right away (next release), and that something fundamental (like the KDE search index, htdig nonsense that still didn't work for me) would be a high priority.

Or maybe it is a flawed system, that inhibits proper configuration. I've looked into the process for trying to index a document in Yelp and it looks pretty damn technical, having to write up a separate document for the given help file, and then I don't know how you're supposed to fire up scrollkeeper to actually index it. 

A million websites manage to have search engines for their content - massive amounts of content that constantly shifts and changes. Why can't a simple program handle that with a bunch of local files? 

Ideally, the help system should be able to integrate documentation from every application, from any independent installed documentation package, search all of it - more than just the titles. And point to more than just the manual or GNU info pages, and if its not too much to ask be able to incorporate documents from an end user in a friendly fashion. And maybe if this last point was met, and it was just really simple to work with, there wouldn't be a problem in the first place - maintainers would have an easier job just getting their documentation to install to the right place and their program to point to the right place.
