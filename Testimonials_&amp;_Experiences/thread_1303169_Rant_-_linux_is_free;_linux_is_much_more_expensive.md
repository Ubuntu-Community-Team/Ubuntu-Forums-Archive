### Post by e_james on 2009-10-27
I am 62 years old. My first computer programmes were written in Fortran for the ICL 1905 mainframe. After that I wrote hex code for 8060, 6502 and Z80. I sold an application in Z80 code to several clients. I then rewrote it in Turbo Pascal and sold it again. With the IBM PC I progressed from MSDOS 3.3 to Windows XP. In all that time I have been aware that there is much more to computer science and programming than I have managed to learn, but I have done well enough. I began studying Linux with Mandrake 8 and Suse 7. I like Linux; I like the idea of FOSS; but Linux is very hard work.

I think the the problem is in the instruction manuals. Apart from Fortran, I have had no formal education in programming. Everything I know was learned from learned from manuals and experiments. In the 1970s I believe that the mainframe manufacturers had entire departments writing instruction manuals. The first IBM PC came with manuals several inches thick. About 1987 I worked for a while with a mini computer called a Jacquard which had a word processor called Typerite. The Typerite manual was the best I have ever seen. When I wanted to know something new, I looked up the relevant page and followed the instructions. As I recall, I never needed to read any page twice. Somewhere around the time of the 386 cpu it became common practice to put instruction manuals on CDs and more recently, they may only be available via the internet. With the possible exception of the Suse 7 manual I can't recall any Linux instruction manual / howto which hasn't left out key points of information (including the Ubuntu Pocket guide).

I don't like pdf files. Pdf can be written to be read on screen complete with hyperlinks but almost every example is constructed to fit a printed page. If you want to read it comfortably, print it. I often make use of the Google facility to convert pdf to html.

Around about 1990 I became aware of Hypertext by way of an article on the Apple application Hypercard. I was really impressed. It was suggested that this was the ideal way to write instruction manuals. You read the parts you need to know and ignore the parts you already know. The hypertext makes it easy to expand any topic in as much detail as necessary while keeping the core points short and concise. And then I discovered the Wiki. At this time it seems to me that the wiki is the quintessential hypertext instruction manual but I have yet to find a good wiki for Linux newcomers. I have my own description for Linux manuals generally - Too much information, Too little explanation (TMI-TLE). Every time I start to read a promising source of information I run into this wall of incomprehension. It looks like english but it is apparent that the words have a specific meaning I don't understand.

The most difficult instruction manual I know is the mencoder man page. 8000 lines!! I think I could work with it if I could read it. If I could convert it to html it would help. I know there's a way to do it but there's that wall again. There's a web site for mencoder but it leaves out most of the explanation I need. (TMI-TLE)

What I would like to do is to contribute to a wiki by adding explanations where I think they would help. Most wikis I have looked at have some sort of invitation to contribute on the page but, when I think I might do that, I hit the wall again. It seems I need a week or two of formal training before I can be trusted to edit a page.

Every few days I get really frustrated with Linux. There's something I want to do and I am sure it can be done with linux but it seems to be unnecessarily difficult to do it. 

Example 1
I want a linux procedure to tell me if my usb sockets are usb2 or usb1.1. There's a utility called usbtree but I can't figure out where it is or how to use it.

Example 2
I have an IP camera and I would like to connect to it using linux. The bundled software works but it needs ActiveX in internet explorer. Apparently I need to understand RTSP and RTP.

Example 3
Why is WPA so complicated with Linux compared with Windows?

Example 4
What exactly is a virtual desktop?

Example 5
Apparently I can add tags to this thread. Why?

I have plenty more to say but this will do for now.

### Post by scratman on 2009-10-31
Just a quick note, if you open Terminal and type ```
man mencoder >> mencoder.txt
``` and press enter, it will export the manual for mencoder to your home directory as a text file.  This may not entirely resolve your issue, but makes it much easier to manage and manipulate.

Also, to identify which of your USB ports are USB 1.1 or USB 2.0, again go into Terminal and this time type ```
lsusb
``` This will give you a printout of all of the USB devices on your machine.  You can then identify which ports are 1.1 or 2.0.


For instance, LSUSB gives me this information...

```

Bus 002 Device 004: ID 046d:c016 Logitech, Inc. M-UV69a/HP M-UV96 Optical Wheel Mouse
Bus 002 Device 003: ID 1267:0103 Logic3 / SpectraVideo plc G-720 Keyboard
Bus 002 Device 002: ID 03eb:0902 Atmel Corp. 4-Port Hub
Bus 002 Device 001: ID 1d6b:0001 Linux Foundation 1.1 root hub
Bus 001 Device 002: ID 0ace:1215 ZyDAS WLA-54L WiFi
Bus 001 Device 001: ID 1d6b:0002 Linux Foundation 2.0 root hub

``` 

From this, I can discern that the ports in which I have my mouse, keyboard and 4 port hub are USB 1.1, whereas my wireless USB dongle is connected through a USB 2.0 port.
