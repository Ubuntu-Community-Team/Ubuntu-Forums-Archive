### Post by disasm on 2007-04-24
The problem you are having with grub is (most likely) actually a feature (really), just the streamline of the upgrade is the problem. With Edgy, fstab started using UUID's to identify the disk to boot, however the kernel had a horrible bug and couldn't reference a UUID from grub. Now, in comes feisty, and new kernel fixes everything, all is well, except it guesses the wrong UUID to replace in the kopt line and tries to boot the wrong partition.

Now you can say that it's bad that they would make such a dramatic change in an upgrade, but if you look at this from a logical point of view, this change prevents this problem from happening again. By using a unique identifier that doesn't change from master to slave or even machine to machine, we will no longer have the problems of new kernel loading the drives in a different order and being unable to boot. This was a great step in the way of seamless upgrades, and it's only going to get better as more and more of upstart is completed and added to the new ubuntu releases.

Also, it doesn't help to make a long dramatic show about why you aren't using linux any more. We hate to see you go, but the least you could do is let us help you fix your problem by stating it clearly with any error messages you are receiving on startup.

### Post by Tomosaur on 2007-10-03
> **scwinn said:**
> Thanks. But I am installed and running and have so many packages installed that it would be almost impossible. I wish that I could download the Ubuntu 7.10 image and then click "upgrade". But alas....

In Synaptic, click the 'status' button in the bottom left, then click 'Installed' in the left hand panel. This should list all packages installed on your system. Click file, then 'Save Markings As'. In the save dialog which options, click the 'save full state, not only changes', and then save this file as 'to_install'. Back up this file on a usb drive or something like that, then install your new Ubuntu version from scratch. On your new system, open up Synaptic, and choose File > Read Markings, then point it to your 'to_install' file. It will read all of these packages and mark them to be installed. Now just click the 'apply' button and it will download all of the packages you had installed on your previous version of Ubuntu :)

(Remember this won't work for packages not downloaded from the Ubuntu repositories - for example if you got the .deb file from getdeb.net, then you should just back that .deb file up and install it manually on your new system. Same applies for non-default repositories, so it's probably a good idea to back up your non-default repositories from the /etc/apt/sources.list . Do NOT just overwrite your new sources.list with your old one, just add the non-default repositories to your new sources.list.)

### Post by Tomosaur on 2007-10-03
> **AnimateDeath said:**
> You know, maybe I am wrong, but wouldn't a software developer understand more than the average person about an operating system?

Not necessarily, no. If the only software you develop is user-land software, then you'll probably never need to understand much, if anything, about how the underlying system works. This is exactly why we have languages like Java and Python, which are platform independent. A program written in Python or Java on Linux will run exactly the same on Windows, for example. The developer doesn't need to know, or care, what platform the user runs their software on.

On top of this - if the developer in question took a course, then that course may have been very focused, only teaching the person what she/he needs to know in order to develop software (which is not a whole lot, by the way, it's more akin to learning Spanish, or some other language you don't normally speak). Many software development courses do try to teach fundamental concepts about operating systems and computers in general, but that kind of thing is more suitable for people who are learning computer engineering (which is a very different subject).

Software development has more in common with language and art than it does with computer engineering. Software developers need to be creative and able to abstract ideas and concepts in order to create a solution to some problem. Systems engineers need to be able to focus and understand the minute details of a system.

Many people who don't concern themselves with such things generally tend to think of all computer-related activities as 'computers', but it's a lot more complicated than that. A software developer doesn't need to know how the computer works, just how software works, and depending on what the software does, the developer can just ignore the actual physical machine, and the operating system, completely (within reason, of course).
