### Post by 3rdalbum on 2011-07-17
Sorry, but it's poppycock.

Some time between Ubuntus 6.06 and 8.04, there were two bugfix updates that did break normal people's systems, but the problem didn't occur on the developer's system. These updates were retracted quickly when problem reports flooded in, but to prevent this situation happening again the Ubuntu developers decided to put a "proposed" repository in-place - where new bugfix updates go into Proposed for testing, and then pushed out as regular updates if no problems were reported.

This is what happens today.

You never get broken systems from security updates because they are so minor. It's only plausible to get broken systems from doing updates from Proposed ( a self-admitted "bleeding edge" repository ) when there is a contentious update in it; or when you've messed around with your system so much that an update hits a situation that would never appear on a normal Ubuntu system. An example of this kind of modification could include using chmod on system directories to allow users to access it; something which newbies sometimes do if they don't understand going to root.

System breakage might also happen because of PPAs that mess with vital components of the system; either a bad update in the PPA or a conflict between the PPA's modifications and the regular Ubuntu update.

Or, the breakage is a coincidence - the person's filesystem is damaged because of a hard disk failure, and at next reboot the filesystem cannot be mounted.

People using the Security and Updates repositories and no weird and wonderful PPAs/modifications should virtually never see broken updates. They are less likely to see them than they did in previous years. Therefore, Ubuntu's quality is better than it used to be.
