### Post by Methuselah on 2009-11-23
My first non-windows OS was FreeBSD.
Hardware incompatibilities brought me to the Linux world where I had greater choice.
The Linux distro world is a lot more varied than BSD so popularity brought me to Ubuntu.

Hardy has worked well but long term usage of Ubuntu is a pain due to the instability of the later releases and the stagnancy of the older ones.
The trade off for operating stability in Ubuntu land is stagnancy but eventually you will need newer packages.

However, with FreeBSD I had the ability to upgrade application software and still be left with a stable operating system.
Arch seems to follow a similar system and while Ubuntu is more convenient initially, Arch seems to allow for a less painful upgrade path in the long term.

So after using Ubuntu for a couple years, I have to say that the initial effort in setting up an Arch system might pay for itself.
Ubuntu will exact penalty for its initial convenience every time you have to upgrade.
And of course, if the release is buggy you'll find yourself in deep excrement.

Ubuntu is very good training wheels for someone who needs to understand Linux (if you can find a solid LTS release).
The support section is courteous and helpful with much tolerance for beginners.
However, at some point one might need to move on because one's needs have grown beyond what Ubuntu best caters for.
So "quitting Ubuntu" need not be an insult...more like a graduation with thanks.

### Post by Tamlynmac on 2009-12-08
I've struggled with the concept that someone would give up on Ubuntu after having problems with a single release. With a 6 month release cycle, why do some feel the need to upgrade immediately and place so much emphasis on upgrading? Especially, without assuring that said upgrade will enhanced their system. I've never upgraded immediately upon release of a new version and even then I test the upgrade before installing.

 For example, I use an inexpensive spare HDD to test new release prior to committing. I test all our systems and confirm upgrade performance/compatibility. Simple and yet effective. Should a system under test fail to be fully functional, I won't upgrade that system. I use a predefined list of testing criteria and the process effectively assures a successful upgrade. To date, I've had zero failures. But I still repeat the process on every upgrade. Significantly reducing frustration and further loss of an already receding hairline. :)

I also find it hard to believe a business would even consider upgrading any OS, until the new release was _completely_ tested. The upgrade should reflect some significant improvement(s). It would seem, that the cost investment in implementation mandates a payback in performance and/or security.

A wise man once told me "if it works, don't fix it". I understand that some truly enjoy state of the art and the newest releases of everything.
