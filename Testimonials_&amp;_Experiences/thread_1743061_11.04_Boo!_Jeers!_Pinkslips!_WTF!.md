### Post by BrendaEM on 2011-04-29
If people like Unity better each day, that would suggest that it is unintuitive in the first place.

When a menu pops over an application, the user is denied access to the area under the menu and the application becomes unaware that it has been overlayed. If the screen is in any way reformatted to accommodate the new menu, the redraw times can be fierce on 3D applications.

I am not against the development of interfaces that can work as touch interfaces, yet taking additional screen real estate is bad GUI design. The 2nd menubar on Gnome is pointless Dunsel as well.

Are you not smart enough to figure out how to make a OS work from a 24 pixel x screenwidth line, like it used to?

Not my post specifically, if developers generally do not read these forums, then all of Ubuntu will suffer for it. Part of user interface design requires that developers observe how the users interact with the interface.

I do not think that most graphic users would have wanted larger icons and an additional menu bar littering up the screen, taking up screen real estate.

The desktop as a UI metaphor worked because the user had instant no-click access to their latest project files. In later versions of Gnome, the user wasn't even given a link to their home folder, unless you turned the icons on in the editor. At one time, in Gnome, the user was thought not to require the capability to edit their own menus.

Is there anything else to be done can do to dis-empower and slow the user?
