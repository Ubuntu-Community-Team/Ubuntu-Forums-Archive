### Post by Mr. Picklesworth on 2008-01-29
The problem with tabs of that sort for a usability-focused desktop environment like GNOME is that lots of systems rely on *windows* as sources of information.

A good example is the window switcher, especially what we get with Compiz. The title and position of a window can tell you a lot about its contents.
Once something is open within an application-managed tab that no external software knows anything about, the window title means nothing! Finding your document again via the window switcher is now a hopeless memory game.

If Nautilus were to implement tabs, GNOME would first need to have a desktop-neutral library for applications to register open documents, tabs, windows etc. so the user can quickly switch to them with (for example) a panel aplet. Essentially, we would need an alternative to what window management happens to give us, with some extra goodies thrown in for good measure. Actually, it wouldn't be a bad idea even if Nautilus *didn't* go with tabs; using windows to tell things apart is, after all, an arbitrary hack. (A necessary hack that happens to not like application-managed tabs versus WM-managed tabs, but an ugly one nonetheless).

Now, split *panes* on the other hand are not replicating the window manager's (somewhat indirect) job of keeping stuff organized. I agree with that thought!
Having said that, similar is achievable with the rather nice Tree view for the side bar, but I do realize it's not quite what is being looked for.



You know what? I'm going to link you to [this guy's](mystilleef.blogspot.com) opinion. He manages a particularly convincing explanation for why Scribes does not have tabs.
[http://mystilleef.blogspot.com/2006/10/no-tabs-why.html](http://mystilleef.blogspot.com/2006/10/no-tabs-why.html)
[http://mystilleef.blogspot.com/2006/12/its-not-textmateyet.html](http://mystilleef.blogspot.com/2006/12/its-not-textmateyet.html)
[http://mystilleef.blogspot.com/2007/05/on-tabs-again.html](http://mystilleef.blogspot.com/2007/05/on-tabs-again.html)

If you need further convincing, use the editor to write Python; you will find its actual text editing features superior to any IDE out there even though it is merely an open-ended text editor.
"But there's no quick build and run button! How will I go on?!"
Open the terminal. *You're using a bloody scripting langauge, the terminal is hardly a stretch!*. Go the directory that your project is in.
#python main.py (Or "make run", or whatever).
Whenever you need to build, switch to the terminal (one click on the window list). Press the Up arrow key. Press Enter.
Boom: It runs. There's even some terminal output!
What's more? Your choice of debugger, and the program will not crash the editor. Impressive, no?

This demonstrates something very important that the GNOME developers realized early on and I hope to God they never forget: That monolithic applications are stupid. Programs should instead trust the other programs around them and the desktop environment itself to make other, slightly out of scope tasks easy and intuitive. I admit that I kind of like MonoDevelop, but really there should not be a such thing as an IDE as we know them today. Please, I challenge you, name one reason how it is easier to customize Anjuta to fit one's specific development tools and habits than it is to simply open a decent text editor with a few terminals, a debugger GUI and Glade in the background. The only thing an IDE does that is special is autocomplete, and any text editor can implement that within its scope. (Scribes, for example, does arguably better autocomplete than most other IDEs out there even though it just goes by previously typed words).

The latter gives you choice. It keeps things light. It keeps things organized, and it keeps things stable. Losing the GUI editor does not mean interrupting an SVN checkout, losing the terminal, killing the debugger session and erasing your work. Every application is a modular component that can be pulled out, organized, swapped with something else.
"But there *are* IDEs that keep those things as modular applications that crash individually!" -- Those are even more daft. They are reinventing the wheel for the sake of it, and as a result making a slower vehicle. That is the job of the entire operating system *below* that application. By trying to implement a miniature operating system, that application is going way out of scope, wasting piles of system resources and achieving nothing that can't be done better with individual programs.
There is nothing wrong with having little applications that talk to each-other as opposed to a single big do-all application. That is what D-Bus is all about. That is what Linux desktops should seriously be pushing at *hard* to get a foothold on mobile devices, because applications working that way means beautiful interoperability. Just look at the legendary status of the CLI if you need an example; command line applications are really well designed to interoperate.



Which I suppose brings me back to trying to explain how this relates to tabs. Well, it's more that tabs smell of the evil do-everything-at-once, "leave no task unturned" applications. Many people have a knee-jerk "needs tabs!!!" reaction to any application that doesn't have them, and I find that a bit worrying. I suppose in a file manager they could be done right. After all, there are other unhappy examples of tabs already out there, such as in Pidgin or gedit. Just so long as they are not done the Firefox way, which means that they shouldn't really be "tabs" in the way we have come to expect them. After all, we already have tabs: The window list applet in the panel.
Perhaps the breadcrumbs at the top of the Nautilus window could swap between being history and being the path split into buttons. That could replace the existing functionality where the search bar appears there, since Search should really be a Place, given that its functionality has nothing whatsoever to do with the active folder.

### Post by 23meg on 2008-01-31
[QUOTE=mivo]Even if a Gnome developer doesn't like the idea, why not offer the funcionality to those who do believe it to be an improvement? Choice is a good thing.[/QUOTE]

"Choice" costs too, in the terms explained in the article linked in post #31.

A common misconception is that if a feature is kept optional, its possible impact and cost on those who do not really need/want it is automatically reduced to zero. In Penningon's terms, an optional feature is just as costly as a default one, because it too has to be supported, work and make sense together with the rest of the software. This is especially important in a huge project such as a desktop environment, and with a centrally important piece of software such as the file manager. 

That said, if the design implications (cited all over the web, as well as in Mr. Picklesworth's posts, partly) are sorted out, and there are people who put in the dedication to produce the code **and maintain it to the standards set by the project**, then it would be possible to have the feature. It's just somewhat unrealistic to expect existing developers who have put in years of effort to lay down certain design principles and produced software that conforms to them to implement a feature that goes head on against those principles.

Users often take consistency and the "just works" effect for granted, and only think of them once they're crippled. Just tacking on features without properly evaluating their implications on the whole environment is an approach that does cripple them, and thankfully, it's avoided in GNOME.
