	WebKitWebView *web_view;

	if (argc > 1)
		url = argv[1];
	else

	window = gtk_window_new (GTK_WINDOW_TOPLEVEL);
	gtk_window_set_title (GTK_WINDOW (window), "Web Browser");
	gtk_widget_set_size_request (window, 800, 600);
	g_signal_connect (G_OBJECT (window), "destroy",
			  G_CALLBACK (gtk_main_quit), NULL);

	sw = gtk_scrolled_window_new (NULL, NULL);
	gtk_scrolled_window_set_policy (GTK_SCROLLED_WINDOW (sw),

	gtk_container_add (GTK_CONTAINER (window), sw);

	webkit_web_view_open (web_view, uri);
	gtk_container_add (GTK_CONTAINER (sw), GTK_WIDGET (web_view));

### Post by Mr. Picklesworth on 2008-04-05
I still do not see how Firefox's "RSS support" can be considered good.

Live Bookmarks are useless because they do not give or download previews. The point of RSS is having that information beyond the title, which a powerful feed reader will happily download and display. Then, when someone is not connected to the Internet, he can still read through all those RSS feeds, mostly with full content. It's also a nice way to parse information: Headline (+ Read? Updated?) -> Summary (+ Author, Date) -> Full Text (often just put in the summary with sites that don't rely on ads). That method is particularly fantastic for comic strips like xkcd; I make sure to flag all the great ones so that they are permanently there and easy to recall.

Does Firefox's RSS reader let you do half those things? No! It just gives headlines, with the option to go to them and nothing more. If it did do half those things, that there would be bloat; I would rather it did none of those things and was built the right way, by entrusting the job to other applications.
Furthermore, Firefox's RSS feed subscription is not integrated with the desktop via dbus, so getting those feeds into a half decent RSS reader is a painful and unintuitive process.

One crit for Epiphany, though: Its default toolbar layout is ugly. I have it arranged all into one bar and it looks great, but lots of people just see the default layout and rate the browser at first glance. Dual toolbars looks particularly bad with the Clearlooks theme, which I find interesting since that's GNOME's favourite theme!
Some fiddling with the default layout could be in order.

It really surprises me that people do not realize how GNOME's human interface guideline benefits the user experience. Programs do not have a lack of exposed options; they have sensible defaults and are developed by people who realize that options are expensive, so they make sure the program's interests never diverge. Where there are thousands of options, there is also (usually) no vision. The reason you can install Ubuntu on your grandma's computer and have it work for her is entirely because of that simpler, smarter, streamlined design. A nice example that is horribly common in KDE: "Use broken functionality #2, #3, or 1?" presented in a list box. These direction-less programs tend to become something more like two programs in one, where simply toggling options entirely changes the interface. No, this doesn't necessarily make it slower, but this does make an application more prone to error since there are so many more cases to worry about.

In an app designed with GNOME's philosophy, those types of options would be replaced by a single piece of functionality selected by default that, ideally, provides the three key points of those broken functionalities and (most importantly) is not broken. Unfortunately, this often leads, as I am sure most of you have seen, to a very large bug report full of complaining by people who want back their broken functionality. "Just add an option to turn it off or on", they say! Thankfully, the maintainers tend to have a bit more vision than that and find out some more key points, sometimes adding little check boxes to toggle particular details of the same functionality, but never actually outright switching a program's behaviour.

Speaking of diverging options, another problem that occurs is diverging styles. Firefox is less a problem than, for example, OpenOffice (GNOME Office for the win!), but it still has problems even with subtle changes, such as when programs start putting buttons in different places. It is a problem for accessibility, usability and learnability. (That's right: All 3!). Central to a good desktop environment is consistency, and knowing where to look for something regardless of the application in use is the key. GNOME does this well, and Ubuntu's striving to not just be GNOME with .deb certainly makes sense (and is good!), but it seems pointless that Ubuntu is even *using* a desktop environment when so many of its default applications are out of place.
