### Post by mssever on 2008-11-13
Since I don't know the LTS policy, I'll restrict my comments to non-LTS releases.

Issuing a point release after six months doesn't make sense, because by that time the next release is out. And once the next release is out, the devs shouldn't waste any more time on it than is necessary to provide security fixes. For example, who cares about Gutsy these days, aside from the security folks? Anyone who doesn't want to upgrade every six months should only be running LTS releases.

The proper solution--which has already been proposed--is to modify the packaging system to enable binary patches. That way, when an update is released you wouldn't have to download the entire package; you could instead download just the specific changes.

And when it comes to bandwidth, apt-cacher is a great tool.

### Post by gnomeuser on 2008-11-13
No, it is additional work to do, it puts pressure on our mirrors and it brings nothing installing updates does not do.

The direct harm this will create will be people waiting for .1 because "then it will be stable", causing decreased testing and thus increased instability making .1 the new .0 - creating the need for a .2 release to indicate that "now we are really stable". For a 6 month schedule this will eat away time from future development cycles, and the effort is best invested in making the upgrade painfree.

It makes sense for an LTS release, you have to ensure that those install on new hardware over 3-5 years, meaning you'll need backported drivers and installer changes which mandates the release of new isos. For hardware support reasons, not to reduce post-install update download size.

From a user perspective, it seems like a great idea at first. But it isn't.  It conflicts with the QA on the CD for the new release.  It simply would be impossible for the community to be doing QA for the new release, the point release on the LTS, and the point release for both previous releases at the same time. ### Post by Mr. Picklesworth on 2008-12-03
I think particularly buggy releases like Intrepid should have their images rebuilt once a substantial number of would be show-stopping system crasher bugs are resolved. Especially ones related to wireless drivers that bring down the entire system at random and consume RAM / hard drive space at alarming rates.

Why? Because if that crasher happens while installing, that user will never touch Ubuntu, or possibly Linux, or possibly even free software, Ever Again. If that crasher happens while installing, it is not impossible that the user's system could be bricked.
The other reason is because Ubuntu's release team seem to think they are immune to show stoppers and system crashers. If the distro is going on a super-serious rain-or-shine release policy, it **needs** those point releases.

### Post by mssever on 2008-12-03
> **Mr. Picklesworth said:**
> I think particularly buggy releases like Intrepid should have their images rebuilt once a substantial number of would be show-stopping system crasher bugs are resolved.
The question is, who gets to define what a particularly buggy release is? Gutsy gave me all sorts of trouble, and I was very anxious for Hardy to come out. I've had only minor trouble with Intrepid. In particular, wireless is awesome in Intrepid--for me, at least. Probably some people were/are very happy with Gutsy. Most of us brand a release "particularly buggy" if we happen to be impacted by an annoying bug. But when viewed objectively, that might not be the case.

So all non-minor bugs are annoying to some subset of users. Should Ubuntu then issue a point release for each of them because someone was annoyed by a particular bug? Remember, regular releases are more or less transient. There simply isn't time to do all the QA for point releases AND keep up the development schedule for the next release. People who need more stability/quality than regular releases provide shouldn't be running Intrepid--they should be running Hardy, an LTS release.
