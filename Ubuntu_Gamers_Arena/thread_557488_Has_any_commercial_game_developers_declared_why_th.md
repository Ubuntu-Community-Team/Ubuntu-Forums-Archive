### Post by cogadh on 2007-09-22
Basically, Id chooses to create their games based on open standards and using open APIs, like OpenGL, OpenAL, SDL, etc. Most other developers choose to use Microsoft's DirectX API, which does not have a Linux equivalent. Id does what they do in a effort to show that you can create AAA games that are cross platform without going to any extra expense. The rest of the game development world doesn't really care about that since there aren't enough people gaming on Linux to make any significant profit off of. They won't make a Linux compatible after the fact for that reason as well as the additional expense of porting the client from DirectX to some other open standard.

Your ideas are great, but the fact is that Linux represents less than 2 - 5% of the worldwide PC market. That's a little over 32 million PCs (on the high side). Compare that to the Windows PC market of over 650 million PCs worldwide, and you begin to see why the Linux market isn't considered worth it. Additionally, generous market estimates indicate that about 75% of all PC users use their PCs for some form of gaming (from casual to "hard core" gaming). That brings the number of potential Linux gamers down to 24 million worldwide (again, on the high side), while the Windows gaming market is still up at 487 million worldwide. Linux is still an insignificant niche market in those terms.

It's not a case of letting the developers know how many of us there are, they already know and it doesn't matter to them. If we want more gaming on Linux, the only way it will really happen is if the Windows market shrinks and the Linux market grows (a lot).

### Post by cogadh on 2007-09-25
Here's another way to look at it: 
In the U.S., nearly 66% of all households own at least one PC. In fact, the numbers work out to an average of 1.5 PCs per PC owning household. There are about 106 million households in the United states, which means about 70 million households have a PC and therefore there are approximately 105 million PCs in the U.S. About 5 million of those PCs run some form of Linux (or other non-MS OS).

Now, last year, the gaming industry reported that they sold 39.4 million units in PC games to U.S. consumers. If you assume that most of those households that own PCs bought more than one PC game in that year (let's say they all bought 2), then only about 20 million PC users actually bought a game in 2006 (20% of the Windows PC market in the U.S.). The number is actually probably a lot lower than than, as hard core gamers are far more likely to buy many games than casual gamers.

Based on the ratio of PC owners to number of games purchased and the percentage of PCs that run Linux, if games were made cross-platform compatible, it would mean that game makers would sell, _at the absolute most_, one million more units per year than they already sell. Divide that one million units between all of the PC titles made in a year and you understand why, on a per title basis, it is not worth it to the developers to make cross-platform PC games.

### Post by aysiu on 2007-09-25
A number of interested Linux users isn't enough. You have to give them the number of Linux users who do *not* currently buy their games but who *would* buy them if there were Linux ports.

The sad fact of the matter is that Linux users generally fall into three categories:
1. Those who PC game and use Windows to game (Linux for other stuff)
2. Those who PC game but using only games that will run on Linux
3. Those who do not PC game

Right now, groups #1 and #2 are paying for Windows games, so what incentive do the gaming companies have to port the games *natively* to Linux? Even if people don't have Windows and run Windows games through Wine or Cedega, they are still paying for the Windows versions.

So there is no financial incentive for game companies to port their games natively to Linux unless group #4 becomes more prominent:

4. People who *would* buy PC games that had native Linux versions but who do not currently buy them, because they're mostly Windows-only versions.

### Post by cogadh on 2007-09-25
Well, it is an excellent idea, but I don't think basing it on IP address would really work, since so many ISPs use DHCP to assign a limited number of rotating IPs to their clients. Additionally there is the problem of multiple machines behind a single public IP. For example, my own ISP assigns me a single IP through DHCP, but I actually run anywhere from 3 to 5 different machines behind a NAT router. Two of those machines run Ubuntu and they would both show up as the same IP, according to the Ubuntu update servers. 

Perhaps basing it on the number of unique machine names that connect for updates would be a more accurate method. There is still the chance of some replication (let's face it, someone must have a machine named "ubuntu-desktop" just like I do), but I think there would far fewer instances of that than there would be of IP replication.

This must be the day for people to completely miss my point. ](*,)

Wine is not simply a dependency that must be met, like a library for a native application. It is a complete compatibility layer that allows Windows executables to be understood by Linux and the Windows executables to understand Linux. Wine must be installed and configured before you can actually run any Windows executables, including the Steam installer or the application itself. Even if you could somehow make a deb package that would install Steam and therefore install Wine as a dependency, it still wouldn't work because you have skipped the Wine configuration step. There is no way you could automate that process since it will be different for literally every machine it is used on. Maybe (and this is a great big maybe) if Steam or its installer were recompiled with winelib, it might work, but that would require Valve to either take it upon themselves to do it or a release of the Steam source code.

Also, I believe I may have made this point already somewhere in this thread, the problem is not that developers do not make games "Wine compatible" or that they don't take any interest in Wine (they probably don't), the problem is that Wine is still beta software and no where near complete. Even if someone like Valve were to take an interest in completing Wine, it wouldn't help since the Wine project is not trying to replicate Valve's technology, they are trying to replicate Microsoft's technology. Those Microsoft APIs are often poorly documented so a lot of what they do is glorified guess work. It is really quite amazing that Wine is able to work as well as it does, given that they are trying to replicate Windows functions without ever having seen the Windows source code or having complete documentation of all those functions.. Sure, having Valve assist would mean that there are more programmers working on Wine, but they really would not be able to contribute any more than any other programmer who decides to help.

### Post by cogadh on 2007-11-01
You're right, DirectX and OpenGL are not equivalent. DirectX is actually several APIs; Direct3D, DirectInput, DirectSound, DirectPlay, DirectShow. OpenGL is just a graphics API. 

In terms of what is required to produce results, Direct3D (the graphics component of DirectX) does automate a few things that OpenGL does not (something that is currently changing). However, Direct3D and OpenGL pretty much are equivalent in terms of the results produced.

As for alternatives to DirectX, there is no single suite like DirectX that is cross-platform, but there are several individual APIs that together can do all the things that DX does; OpenGL, OpenAL, OpenML, OpenMAX, SDL, GGI.

Most of what I do isn't contributed to open source projects (I do what I can in that regard but so far I don't do anything like as much as I'd like), so I'm not the best person to answer that.

But in my experience it's about "scratching an itch". If something I use doesn't work the way I need it to, and I can fix it (because I have the source and the skills) then I will do. The fixing it bit is usually selfish (I need it fixed so I fix it) but the sharing it afterwards is easy. In doing so you get to know the code, so when somebody else says they have a problem which hasn't affected you, there's a better chance you know where to look so if you have the time you do. I rarely have the time to scratch the initial itch (thus gaining knowledge of the code) so I'm limited in what I can do. But I'm getting better at it now that more of my itches are related to open source apps.

So, to answer your question more directly:
> Is it the fame, the money, the merits, or something else?

Fame is nice, but I doubt it's often an initial driving force (maybe it becomes addictive when it starts to come, though).

Money might be relevant if the initial "itch" is work related. I doubt that many open source developers earn enough from it directly to call it a living though, 

Merits could mean anything positive, but there is sometimes merit in doing something just for its own sake, just because it should be done (or to prove it can be done). I'd love to have enough time to consider that possibility!

Something else: well probably! Everybody has their own reasons.

My personal interest in Linux gaming is somewhere in this mix. I don't play games much at all, but pretty much as a point of principle now I won't play them on Windows. I have lots of friends who do, though, so I'd love to prove that Linux is up to the competition. So if I can do something to help make that happen I will. So far I'm limited to messing around with the likes of Wine seeing what I can do, filing bug reports to WineHQ, etc. (Based on my comments elsewhere in this thread in future I will also start posting bug reports to the game developer's forums too where they exist.)

### Post by hollymcr on 2008-01-30
> **dmn_clown said:**
> *OT* RMS wants every bit of code under a license that is compatible with his own effectively making him a tyrant.  


I'm not generally an apologist for RMS, but licence proliferation isn't a good thing.

Open Source has a few good things in its favour. One is you can look at the code if there's a problem and fix it, and most open source licences allow that. Another is that you can take the good bits from one application and add them to another, but that's only possible if the licences are compatible. In general if two applications are released under different licences you can't mix code from one application with code from another. 

Whether GPL is the best licence (and in particular whether GPL3 is the best - personally I think it's too politicised) where possible it's best that we have as few licences as we can.
