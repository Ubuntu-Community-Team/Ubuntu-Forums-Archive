### Post by jdong on 2009-03-18
I've reviewed the infraction history in question and support Matthew's judgement in this matter.

I'd like to further add with regards to the forkbomb site that I agree that it is **not appropriate** to link to the forums.

While you gave a theoretical description of what a forkbomb is, I don't think it's opaquely clear what consequences it has. You are inviting your users to test various forkbomb implementations on their systems without explaining what they do. This can be incredibly risky in several ways: First off, of course the user would almost certainly lose any unsaved work at the time. Secondly, the IO pressure of the forkbomb (presumably diving into swap) is going to prevent writeback cache data from flushing to the disk in a timely manner, which can result in data loss. Combine that with the possibility the user uses barrier-less consumer disk write caches, even filesystem structure can be damaged by the type of hard reset required in this case.

I would've rather seen you demonstrate what a forkbomb does (in a Youtube video, textual narration, so on) so that the user doesn't HAVE to try it on their own system. Or suggested using a virtual machine with limited memory.

In addition, I feel I've sufficiently covered what a forkbomb does -- if you wanted to discuss how to prevent  it , or if it's worth preventing (what's the tradeoffs?), that would've been a good discussion too.

At any rate, in its current state, I do not feel comfortable in allowing it in a signature.
