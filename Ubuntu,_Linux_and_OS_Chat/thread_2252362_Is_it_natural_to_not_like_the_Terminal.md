### Post by SagaciousKJB on 2014-11-12
I agree that it's a matter of imprinting, people tend to be conditioned to thinking the command line is too "complex".

I think there's too much made about the complexity of the command line that belies the convenience of it.  In a lot of situations, I can change a system setting with a command more quickly than I could by opening a GUI and checking a box for it.  There may be a little bit of initial learning in that one must look up the manual for that command, but it's typically not that difficult to find out the right way to use it--in most cases there's even handy examples of it at the bottom of the manual page or at the very most a quick Google search away.  I think part of the difficulty in learning to use the terminal or command line interface on the computers of old was that instead of having access to an interactive encyclopedia of help, you had a reference manual.  There is simply a lot of reading involved and so sometimes that negates the convenience.

However in today's cases, most of the time you can go find the command that you need by doing a Google search and then just remembering that command later on.  Then there's the power of scripting...  What's so powerful about scripting is being able to make complex bash scripts, to me it's the "inline" scripting ability.  Think about if you had a folder full of video files, and they were all named "Episode 1" and consecutively, but you wanted a leading zero so they were sorted correctly alphabetically.  Depending on how many files you had, renaming them each one by hand would get old--there may be a GUI application to do a batch process of it for you, but you'd have to go find it, download and install it and try it.  With bash you could conjure up a command to do it for you.  Might not seem like such a big deal, but if you had to do this to hundreds of files I bet you'd more quickly figure out how to do that than go and manually rename them with a GUI--chances are you could find a quick way to do it and modify it to your needs on Google without even having to come up with it on your own.

I'm rusty but here, let's see...

```

find -iname '*episode* [1-9].mkv' | while read file_name; do echo mv $file_name $(echo $file_name | sed "s/[1-9]/0&/g") ; done

```

Then if you take the "echo" out the command will actually excute.  Keeping it in just shows you a list of what WILL be done...

```
kenny@laptop:~/test_dir$ ls
Episode 1.mkv  Episode 3.mkv  Episode 5.mkv  Episode 7.mkv  Episode 9.mkv
Episode 2.mkv  Episode 4.mkv  Episode 6.mkv  Episode 8.mkv
kenny@laptop:~/test_dir$ find -iname '*episode* [1-9].mkv' | while read file_name; do echo mv $file_name $(echo $file_name | sed "s/[1-9]/0&/g") ; done
mv ./Episode 6.mkv ./Episode 06.mkv
mv ./Episode 3.mkv ./Episode 03.mkv
mv ./Episode 2.mkv ./Episode 02.mkv
mv ./Episode 7.mkv ./Episode 07.mkv
mv ./Episode 8.mkv ./Episode 08.mkv
mv ./Episode 9.mkv ./Episode 09.mkv
mv ./Episode 1.mkv ./Episode 01.mkv
mv ./Episode 4.mkv ./Episode 04.mkv
mv ./Episode 5.mkv ./Episode 05.mkv
```

Then you can even go "Oh I need quotation marks to handle those spaces" and modify it accordingly.  Point is, I just wrote a "script" right on the command line.  All I had to do was look up the man page to sed.  Also if I had looked up the man page to "find" I would have found I could do almost the same using the "-exec" option.  But I like "while".

Anyway point is once you LEARN how to do that you don't really unlearn it and then using GUIs to do such things feels extremely limited.

### Post by sudodus on 2014-11-13
When you create a new program that is performing something, it is often easiest to create a CLI interface. Making a GUI which includes all the options and combinations of options can take [much] longer time than creating the original program (to make it do what it should do). And the size of the program will grow (or at least it will be depending on a lot of libraries for the GUI). It is also easy to combine programs via the CLI interface, to build higher level programs using the CLI interface of several components. So a CLI interface is useful, even when the program itself is seldom used directly by the human user.

For this reason, I think there will always be tasks that can only be performed via CLI, or are easier and faster with CLI (when you know how). But when you don't know how or don't quite remember, the GUI system can help you find what you want (and to do it in a convenient way).

At the Ubuntu Forums it is efficient to give instructions as CLI lines, that can get copied and pasted into a terminal window. There are many desktop environments, and the GUI way of doing things can be very different, which makes it harder, particularly to help someone using another desktop environment (than the helper is using).

lol It's not exactly simple, but it's far from algebra.

In a nuttshell, you're having "find" generate a list of files for you to process.  The "while" loop reads each line of this output over "standard input" ( that's what the | pipe is for ) and saves them as file-names into a variable.  You can then perform whatever process you wanted on the file list provided by find, in a batch process by referring to that variable in place of the filename.  In this case my suggestion was changing the names of the files themselves--I chose that example because I think it's something more common.  all the $(echo | sed...) line does is send the filename over standard input to the "sed" command which then substitutes all single-digits from 1-9 with a a leading zero, as to keep alphabetic order in file manager views.  Squeezing it all onto a single line just requires delineating each statement with ";".  If I told you I did operations like this quite frequently I'd be hard pressed to come up with another example of it, but it certainly comes in handy in situations where you know you *can* use it.

A much more simple example is if you just wanted to move all your .mkv files to a folder...

```
find -type f -iname '*.mkv' | while read filename ; do mv $filename some_folder/$filename; done
```

Better yet if you read the man page for "find" you can just use the "-exec" switch to do the same.

```
find -type f -iname '*.mkv' -exec mv '{}' some_folder/'{}' \;
```


Anyway I think there's a perception that graphical interfaces are more "technologically advanced", which I guess in a way they are, but the assumption seems to be made that because they're more advanced that they are better.  You can see the same thing at work today with graphical interfaces being replaced by voice driven interfaces.  The thing is these technologies have their advantages and disadvantages.

Think of entering commands as text as short hand, the graphical interface as something in the middle and verbal commands the long-hand.  Can you write out a list of instructions for someone to perform more quickly than you can verbally direct them to do so?  Probably, and through this you can see why it's useful to interface a computer through something that is more simple.  On the other hand, though things like a GUI or a voice interface "obfuscate" the instructions you're giving to a computer through more steps, they present the user a way to input such instructions in a much more human friendly way.

But we're a long way away from being in a world in Star Trek where Geordi can say "Computer find me a list of all personnel files logged during our stay at Allpha Seti 4, and cross-reference them via a logarithmic algorithm to ..."  There is no voice interface that can interpret those instructions, and you're basically telling a computer to write software for you.  It would be awesome, but actual voice command software is lucky if it can even recognize your commands correctly.  Imagine trying to write a whole program, dictating it out verbally.  It might be great to conduct simple tasks without requiring more attentive input of the user, but to perform complex tasks there's simply a gap in the level of usefulness you can get from a voice commanded system and one with a  graphical interface.

Now take that comparison but frame it in the current context of GUI versus CLI and you can see how a GUI frees the user from having to make certain configurations, but on the other hand will hinder the efficiency at which the user can input commands to the system.  It may be "easy" to select and rename files in a folder, but consider if you had to do this to hundreds of thousands of files.  You might need a couple of hours to develop a "command" or a "script" to instruct the computer on how to complete the task for you, whereas doing it manually via a GUI interface could take days.

Now this is just an example...  Imagine making configuration changes with text versus GUI.  In a GUI you'll need to find the options on a menu, tick the boxes one by one, etc.  With a CLI you can modify all the options simultaneous, even use template configuration settings, etc.

The point is...  The "find" command I demonstrated for moving your .mkv files, the equivalent to that in GUI would be for the user to manually select all of the .mkv files.  Already this opens itself up to user error, because what if you miss one? What if in the middle of the selection you don't remember to hold control and accidentally screw up the entire selection queue you made?  What if you don't even know how to select individual files via the gui and just move them one by one?

> **SagaciousKJB said:**
> lol It's not exactly simple, but it's far from algebra.

In a nuttshell, you're having "find" generate a list of files for you to process. The "while" loop reads each line of this output over "standard input" ( that's what the | pipe is for ) and saves them as file-names into a variable. You can then perform whatever process you wanted on the file list provided by find, in a batch process by referring to that variable in place of the filename. In this case my suggestion was changing the names of the files themselves--I chose that example because I think it's something more common. all the $(echo | sed...) line does is send the filename over standard input to the "sed" command which then substitutes all single-digits from 1-9 with a a leading zero, as to keep alphabetic order in file manager views. Squeezing it all onto a single line just requires delineating each statement with ";". If I told you I did operations like this quite frequently I'd be hard pressed to come up with another example of it, but it certainly comes in handy in situations where you know you can use it.

A much more simple example is if you just wanted to move all your .mkv files to a folder...

```
find -type f -iname '*.mkv' | while read filename ; do mv $filename some_folder/$filename; done
```

Better yet if you read the man page for "find" you can just use the "-exec" switch to do the same.

```
find -type f -iname '*.mkv' -exec mv '{}' some_folder/'{}' \;
```


Anyway I think there's a perception that graphical interfaces are more "technologically advanced", which I guess in a way they are, but the assumption seems to be made that because they're more advanced that they are better. You can see the same thing at work today with graphical interfaces being replaced by voice driven interfaces. The thing is these technologies have their advantages and disadvantages.

Think of entering commands as text as short hand, the graphical interface as something in the middle and verbal commands the long-hand. Can you write out a list of instructions for someone to perform more quickly than you can verbally direct them to do so? Probably, and through this you can see why it's useful to interface a computer through something that is more simple. On the other hand, though things like a GUI or a voice interface "obfuscate" the instructions you're giving to a computer through more steps, they present the user a way to input such instructions in a much more human friendly way.

But we're a long way away from being in a world in Star Trek where Geordi can say "Computer find me a list of all personnel files logged during our stay at Allpha Seti 4, and cross-reference them via a logarithmic algorithm to ..." There is no voice interface that can interpret those instructions, and you're basically telling a computer to write software for you. It would be awesome, but actual voice command software is lucky if it can even recognize your commands correctly. Imagine trying to write a whole program, dictating it out verbally. It might be great to conduct simple tasks without requiring more attentive input of the user, but to perform complex tasks there's simply a gap in the level of usefulness you can get from a voice commanded system and one with a graphical interface.

Now take that comparison but frame it in the current context of GUI versus CLI and you can see how a GUI frees the user from having to make certain configurations, but on the other hand will hinder the efficiency at which the user can input commands to the system. It may be "easy" to select and rename files in a folder, but consider if you had to do this to hundreds of thousands of files. You might need a couple of hours to develop a "command" or a "script" to instruct the computer on how to complete the task for you, whereas doing it manually via a GUI interface could take days.

Now this is just an example... Imagine making configuration changes with text versus GUI. In a GUI you'll need to find the options on a menu, tick the boxes one by one, etc. With a CLI you can modify all the options simultaneous, even use template configuration settings, etc.

The point is... The "find" command I demonstrated for moving your .mkv files, the equivalent to that in GUI would be for the user to manually select all of the .mkv files. Already this opens itself up to user error, because what if you miss one? What if in the middle of the selection you don't remember to hold control and accidentally screw up the entire selection queue you made? What if you don't even know how to select individual files via the gui and just move them one by one?

We should remove preconceptions of complexity or something being "newer" or "better" and consider that each interface has its place.
