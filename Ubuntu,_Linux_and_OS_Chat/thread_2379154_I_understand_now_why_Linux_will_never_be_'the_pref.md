grep -i " install " /var/log/dpkg.log.1 /var/log/dpkg.log
``` will show all recently installed packages with a date stamp of the action from which you may be able to see what was also installed at the same time.

To go further back in the archived logs you may also need to run ```
zgrep -i " install " /var/log/dpkg.log.*.gz
```

### Post by 1clue on 2017-12-11
Linux has always been a distro for experts, or for people who want something else and are willing to work for it.

Distros like Ubuntu have been sponsored by a commercial entity which maintains the distro and chooses a set of priorities or goals for that distro. As such, they have more of a controlling entity than any other Linux distro, but still that's a far cry from Windows or Mac OS.

Many or most of the users of Linux see this lack of a central controlling entity to be a strong point, not a weakness.

Windows and Mac OS X and other operating systems have market-driven design, where Open Source (and Linux) have developer driven design.

Market driven design is when a bunch of people have a financial stake in a product, and they add features that they think will sell more copies of the product. Microsoft Word comes to mind, where they at one point had a fantastic product with lots of extra features but still a solid word processor. Then they had to sell the next copy, and at hundreds of dollars a copy they needed some flashy features. So they added things that a tiny portion of the users might someday find useful. And again. And again. And again, until the once fantastic product became the very definition of bloatware, and locks you into a specific brand.

Developer driven design is when a bunch of people need an application for a specific purpose. Having found none, the people band together and start a project. The project goals are defined by the people who actually want to use the software. The programmers design features that they themselves intend to use. What comes out is a lean, extremely functional piece of software that works exceptionally well.  Example, Apache web server. It has been ported to almost every platform still used, including smart phones, Windows, Mac OS, Every flavor of UN*X that still has hardware to run on, and a good many other systems as well. It has a plugin design so people who need something special can add it without forcing that feature on people who don't want it. It has a permissive license which protects the software for others but is not too obnoxious about what links to it.


Long story short, I don't care if Linux is ever 'the preferred OS.' The preferred OS for who? People who live on Facebook? Why should I care what they prefer? If being the preferred OS means that its users must sacrifice control and freedom of use then I say the people who want 'the preferred OS' should go look elsewhere, and not mess up a good thing which was built by developers for developers.

I've been using Linux as my primary workstation since 1996. I've been using Linux as a main server platform since then as well, and have almost always had at least a half dozen boxes installed for work. Recently it's been a lot more. Linux may have some rough edges, but like any professional tool it takes an educated, practiced user to get the most out of it and the rough edges it has are either there for a reason or they hurt nothing.

### Post by 1clue on 2017-12-18
I advise not backing up apps that exist in the repository. Instead, backup a list of apps you've installed, without dependencies you don't explicitly need.

The thing is, usually when you have a system which is more or less up to date, the fastest way to get it back is to install it over again. The software need not be installed because you can get it back very quickly. Start with a base install, then copy the list of software you want over and 'sudo apt install' everything. If you're set up correctly you can do it all with one command.  In most cases, external software (non-free or simply not in the repo) can be retrieved with a license key or easy download.

If you backup your software you're simply wasting backup media. Backup the things that can't be easily replaced.

As well, if you think that just backing up the whole disk is a good idea then you're in for a surprise when it comes time to restore. A database rarely can be restored from a file copy when the database is running.
