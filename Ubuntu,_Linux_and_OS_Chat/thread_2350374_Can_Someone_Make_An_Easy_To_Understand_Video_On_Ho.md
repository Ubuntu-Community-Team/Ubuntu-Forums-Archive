### Post by ian-weisser on 2017-01-19
Canonical engineers run make-a-snap workshops on IRC and Google Hangouts, and [archived on YouTube]("https://www.youtube.com/user/UbuntuOnAir"). 
Snaps are not easy to make. You must *understand* and *solve *the issues involved. It's not magic command-fu.

Snaps are not perfect - they take up a LOT of storage. They run into permission issues when interacting with other applications. They are supported and upgraded only by the Snap maintainer.
This implies that Snappifying an application is not a one-time task. There's an implied commitment that the snappifier will also snappify updates regularly, and push security fixes. Perhaps for years.
Debian has been wrestling with these integration issues for 20 years, with much progress made. And much progress still to go.

Your initial "I am new to Ubuntu and I can't get all my programs to install" seems like a typical Windows power-user hitting the learning curve (your skills encourage you to do very unwise things). We have all been there, and we all learned the skills to make Ubuntu easy. You will too.

### Post by Impavidus on 2017-01-20
> **july8six said:**
> Besides turning the apps into Snaps or using the software center, is there a way I can get every single program I want to install easily?
There is no single standard recipe for installing stuff on Linux. There are many ways, some better than others.

The most common way, and the easiest and most secure, is from the official Ubuntu repositories. Software Centre, Ubuntu Software, Synaptic, apt, names change and there are various interfaces, but they all get software from the same sources.

Snap is new. Most of us have little experience with it. It solves some problems and at the same time creates new ones. There's no way around that.

The tools you can use to get software from the official repositories can be expanded by using PPAs: additional repositories. Do you trust them?

You can download .deb files and double click to install. Are you sure you can trust their source? Don't forget to check for updates later, they won't come automatically.

Sometimes you can download an executable ready to run. Are you sure it's no malware? No automatic updates here.

Finally there's source code. Download, if you're suspicious read the source code to make sure there's nothing harmful, figure out the dependencies, compile and install, then check for updates and repeat.

Depending on what you want to install, you decide what's the best method. There's no single easy and secure way. It depends on how the maintainers of that software package it for us and if you have the patience to wait for easy packaging of a new version. But the methods from the top of the list are usually better.

### Post by ian-weisser on 2017-01-24
> **july8six said:**
>  a .deb file is the same as a .exe and I would just have to double click to install.

.deb and .exe are programmed to provide a similar experience to users (double-click to install), but are very different in almost every other possible way.

.deb packages are compiled for a *specific release* of Debian or Ubuntu. They often contain binary blobs compiled against specific versions of separately-provided dependencies. Oversimplification: When you change the version, the blobs no longer match each other properly and the deb simply won't work.

This isn't a problem as long as you stay within a single distro's release. All the packages for, say, Ubuntu 16.04 are compiled against exactly the same versions of the same dependencies. All the binaries for the entire distro match. In fact, that's the original definition of a distro - a snapshot of uniform dependencies. To upgrade applications in this environment, you must often upgrade the entire system to a new snapshot. In Ubuntu, a new snapshot (release) occurs every six months...in April and October.
