### Post by DuckHook on 2015-03-19
This question (a good one, by the way) gets asked on these forums all the time and my stock answer is:

You don't have to give up Windows. I'm about as committed a Linux user as they come, and I still have a couple of versions of Windows loaded on virtual machines that I use once in a blue moon. The last time was months ago to root an Android tablet. There just are no Android rooting apps for Linux that I'm aware of, so Windows still has its uses. In my case, it may only be once a year. In your case, it may be once a week or even once a day.

Over the years, I've gravitated to recommending Windows in a VM over dual booting. VM technology has come a long way from its infancy and, coupled with the greatly superior HW today, it is now very viable to run the apps that one needs inside a VM. So many advantages: from the natural sandboxing that a VM gives, to snapshots of the entire Windows system, to the increased productivity from working on two OSes at the same time. Even a toy has its uses... ;)

### Post by buzzingrobot on 2015-03-19
My two, or three, cents:

1.  If you switch entirely to Linux, then the best approach is to use only Linux applications, avoiding Wine.  Wine adds complexity while limiting the range of Windows apps available for use.  If you need Windows apps, run Windows.

2.  Don't look for Linux clones of Windows software.  Look for Linux software that provides the capabilities you need.

3. Dual-booting is fine, but it seems many people jump into it without adequately understanding how to set it up correctly, or even without understanding the hardware components in their system. Almost daily on this forum new posts appear from people who have tried to install Linux in a dual-boot scenario only to find Windows is no longer offered as a boot option. 

4. I've found the most useful way to research the Linux compatibility of a specific hardware component *I'm considering buying* is simply to Google for it with the specific release of the specific distribution I'm thinking of using. If a consistent problem exists, people typically post on it.  As mentioned upthread, booting and running in live mode is the best way to check compatibility of hardware you already own.

5. Red Hat and CentOS:  CentOS is not a fork of Red Hat, and it is not a variant based on Red Hat, in the way that, for example, Linux Mint is a variant based on Ubuntu packages.  CentOS rebuilds unmodified Red Hat source, after removing Red Hat trademarks, art work, etc. CentOS maintains its own repositories, as well. When Red Hat releases an update via its channels, CentOS users get the same update from CentOS repositories.  Red Hat targets enterprise users, and the subscription fee provides two basic things: 1) Access to Red Hat support, at a level depending on the subscription; and, 2), access to the official Red Hat update channels, i.e., the ability to log in to the Red Hat servers and pull down *binary* update packages. (All Red Hat source code is released on its public servers. Bottom line:  Red Hat and CentOS run the same code in the same way and are administered in the same way.  The difference is in the business relationship with Red Hat and the support that provides that customers typically want. CentOS is community supported, meaning you're on your own and you can't call Red Hat to help you fix things.

6. Red Hat/CentOS long-term stability is real and legitimate, but it comes at an eventual price for typical desktop users:  Core libraries in an initial release of Red Hat are very often not updated to new versions during the lifetime of that release. (Bug fixes, yes; new features?  Probably not, although occasionally things are backported.)  Sometimes, they cannot be without essentially reengineering the entire release.  So, down the road, for example, if a user wants to install an application that requires a version of GTK3 or python that's higher than Red Hat's, they're usually out of luck.
