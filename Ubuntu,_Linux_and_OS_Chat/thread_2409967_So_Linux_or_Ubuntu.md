### Post by TheFu on 2019-01-09
Strictly speaking, Linux is just the kernel, which is shared by all Linux-based distros regardless of platform, CPU, size.  A 20K node super computer running Linux to solve complex problems is linux, just as much as a $5 Raspberry Pi Zero is Linux.

Ubuntu is Linux.  The "Linux" term includes all "linux" distros like Ubuntu, SuSE, Arch, CentOS, RHEL, TinyCore, Puppy, Mint, and the 500 others on distrowatch.

Different Linux distros concentrate on different aspects, but under the GUI, they are all basically the same after you deal with the package manager differences and any GUI.  Linux is Linux about 90% of the time.  The remaining 10% are those slight differences that each distro brings as a differentiator to their competition.

Basically, there are 4 "families" of Linux based on the package manager used:
* Redhat - RPM
* Debian - APT
* Arch - pacman
* Slackware - tgz

There are some specialized variants that do not fit into those families, like gentoo or TinyCore.  Each has a use.  Which is "best" depends on your needs and situation.

The families might have slightly different locations for some system config files.  For example, network setups in the redhat-family are in different locations than for Debian-based distros.
Debian and Redhat families are the most popular in companies.  

I've run about 20 different distros over the decades.  Around 2008, after using Ubuntu Desktop for a year, I began switching my servers over from RH to Ubuntu.  Since then, I've also Debian servers and CentOS servers, and TinyCore for online banking.  In general, a server here will be Ubuntu-based.

Since I don't care about the GUI much, for me, which distro to use comes down to a few considerations.
* trust
* ease of use, which is mainly about the package manager, 
* stability
* support/maintenance provided for acceptable timeframes
* security

For me, Debian was just a little too far behind to be considered. Yes, it is stable, but many packages were 3 yrs behind.
CentOS feels heavier than we want.  We run most of our server systems inside virtual machines.
Support periods of 5 yrs or more is something we like.  Lots of distros provide support/patching for 6 months, which is almost useless for a business.  New is the enemy of stable.  The Ubuntu LTS with 5 yrs of support lets us skip a release while still getting patches.  Much longer than 5 yrs will lead to totally stuck computers. It becomes a big problem when the team that setup the .  Ubuntu LTS aims for a stable feature set, but with new enough program versions that are stable.  It is a good trade-off for our needs, almost always.

I will admit that 16.04 and 18.04 underlying changes have delayed our acceptance 6+ months before we would use them.  We still don't have **any** 18.04 systems in production as we await multiple issues to be solved around systemd and netplan changes.  Systemd needs about 2 more years before it will be production ready, IMHO.  Netplan just needs to handle more edge situations that most people don't see. systemd and netplan aren't ubuntu-only. RHEL, CentOS, and most other linux distros have switched to using them.
For my few complaints, Ubuntu LTS is still my choice over the other options.

For a desktop, I'm running Ubuntu Server + openbox with customized acceleration shortcuts. I've used Xfce, Lxde, Mate, and Gnome, but they were all more than I liked. In the 1990s, I used fvwm and it was good when we had limited monitor sizes.

What is the big difference?  Anyone can make a distro with a little effort.  Sometimes it is just because they don't like the default applications installed by a distro they used before.  Sometimes is is for a highly specialized use, like performing security checks or being as small as possible, but with a browser for any browsing needs.  Chromebooks run ChromeOS in that way.  There are a few clone distros that can be run on any computer, similar to ChromeOS.  People with older computers can be very happy running those.  ChromiumOS is one.  CloudReady is another.

People coming from Windows think that installing an OS is a big deal.  It isn't.  You can try all the Linux-based distros that you want. It only costs time and disk storage.  If you use virtual machines, there is very little risk at all.  If you need to dual, triple, quad, (or more) boot - that means having more than 1 OS installed - then there are more risks to any existing OS on the machine.  Also, most Linux OSes only need 10G to try them out, so it isn't like Windows where 60G seems to be the minimal storage needed.  

Installing 18.04 into a virtual machine took 13 minutes.  Not much of a time commitment.

In the end, each different distro is about choice.  Everyone has slightly different needs, so there are lots of different distro options.

### Post by SeijiSensei on 2019-01-09
Do you have a running Windows system with 3 G memory or more?  If so, you can try out Linux by installing VirtualBox on your Windows system, then creating a "virtual machine" into which you can install a Linux distribution like Ubuntu or Fedora.

[https://www.virtualbox.org/](https://www.virtualbox.org/)

Install the Windows version, then download a Linux ISO file like this: [http://cdimage.ubuntu.com/bionic/daily-live/current/bionic-desktop-amd64.iso](http://cdimage.ubuntu.com/bionic/daily-live/current/bionic-desktop-amd64.iso)

When you run VB, you can create a virtual machine and point the installer at the ISO file.  Then you can play with Linux while sticking with Windows for as long as you want.

Depending on how much disk space you have, you can create a number of virtual machines and try out a variety of Linux distributions.

### Post by QIII on 2019-01-09
There are some very minor differences under the hood in some cases.

For instance, some distros do not keep anything in /sbin, but rather use symbolic links from there to /bin.  Others still use the older /bin, /sbin, /usr/bin schema.  This is not unlike the historical shift from /etc being really "etc." for anything nobody knew where to put, to being the location for many configuration files -- "**e**verything **t**o **c**onfigure".


All important, perhaps, if you are a Linux Admin, but not something an end-user would ever notice.

That's certainly nothing that the OP will ever need to bother with.
