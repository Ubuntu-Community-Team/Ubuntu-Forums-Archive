### Post by TheFu on 2020-04-24
Nobody here works for Canonical. 

Different software has different licenses. Some SW licenses don't allow redistribution, which puts a damper on including them as part of the distribution.  Of course, if we all paid $120 for a license every few years, that could be solved.  I'd rather just run a tool post-install for that.

What is a "proper" dev tool?  I've been developing software using the GNU libraries and compiler since the early 1990s. I don't want those tools installed automatically on all my systems, just the development systems. No reason to pre-install tools that helps hackers on my servers, m-k?

Complaints about drivers need to be sent to the driver vendor, not Canonical.  It is their fault, 100%.  My nvidia 7200GS lost nvidia driver support when I moved to 16.04, but fortunately, the F/LOSS driver worked on by a group of fans, provided to the world for free, lets me still use that GPU.  On another system, freshly installed, the F/LOSS driver doesn't provide the native resolution that my primary monitor supports, so I spent about $65 to get a newer, fanless, nvidia 1030.  That HW upgrade brought new hassles.

Developers are known to be picky about their setups.  I am.  I want control over what and where things are placed.  I want to configure all environment variables, since they control where things go and how they behave.  Developers are supposed to know these things and know to configure them as needed.  

But you can feel free to make a tool that does what you seek, create a package, gain traction with millions of users, and get it included in different distros.  Bet lots of others would love that!
