### Post by buzzingrobot on 2014-08-18
Keep in mind there is a dichotomy here:  Emphasizing the so-called traditional panel-and-menu interface may please many existing Linux users, but that does little or nothing to attract new users to Linux.   These days, the vast majority of people using computing devices are accustomed to the interfaces on phones, tablets and, even, Windows 8. Interfaces for Linux that allow them to leverage their current habits and skills will be more appealing.

Gnome *2* is gone. It's dead. Maintenance stopped in 2011. MATE is a light fork and modernization, easily installed on Ubuntu.

So, Canonicial is not going to resurrect a huge dead codebase, especially since that interface is completely at odds with their plans re: convergence and devices.  The Ubuntu MATE project is doing well and seems like a cinch to become an offical flavor down the road.  An alpha release is out and a beta is coming.

Gnome *3* and Unity are superficially similar, and Unity does use some Gnome components.  Out of the box, I find Unity more usable.  When I do use Gnome Shell, I add several extensions that change its character entirely.

### Post by buzzingrobot on 2014-08-18
I think the debate over Linux desktop interfaces is almost always framed as a winner-take-all scenario, as if a single design should be used by everyone.  Linux media -- a pretty sorry lot -- typically treats this as a life-or-death battle.  That grabs mouse clicks and ad revenue, but isn't especially enlightening, or even accurate.

I think the division is deeper. For some, Linux is an entirely different experience and (disruptive) culture, as well as a collection of software tools. Traditional ways of doing things in Linux developed over the last two decades, along with what are now considered traditional interfaces. Many folks value that and want to preserve it, even, apparently, if that means creating barriers to increased use of Linux (because it requires a learning curve and a commitment that users of other OS's may find unattractive and unrewarding).

On the other hand, if someone's primary interest is expanding use of Linux, then it makes a lot of sense to design systems and interfaces that present as few surprises as possible to people whose computing experience has been on phones and tablets.  (The times when it could be assumed that almost everyone cut their computing teeth on a Win95-style interface are over.)

Both routes are open, and there's no reason why all of Linux has to go the same way, in lock step.

### Post by sotiris2 on 2014-08-20
There is a popular misconception that Unity is touch centric. I do not think this is true ,at least for the version we have on desktop. For example to see all programs in dash you have to click a rather thin line containing "show xx more". If it was touch centric the line would be at least as tall as the icos themselves to allow easy touch. Ditto for the different lens versions, tiny icons. 

 In fact it is also a bit mouse hostile for these reasons as well. You could say that old school menus favored the mouse (though if the menu was not a huge list or it was multi-tiered keyboard navigation was as good if not better) but in unity using the mouse only logical only for the common set of programs you do have on your launcher. If you want to open up something you don't use often without using the keyboard... Well you 'll be playing find waldo with icons. In big widescreen screen its really bad ( I find it easier for example to locate something in 4 columns by 8 lines that the opposite). It's not very usable. So if you don't like to use the keyboard and you also are not used to launchers (or simply can't fit anything you want there) it will feel like a horrible UI. 

Funnily i realized writing this post that to say icon based application launch is due to smartphones is wrong. It's actually a windows95 (and nt/gnome2/kde/etc) phenomenon unless you have never seen a desktop piled with shortcuts. The main difference with unity's dash is that users can place the icons where they want them which can make it usable.
