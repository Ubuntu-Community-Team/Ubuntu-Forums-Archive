### Post by 1clue on 2013-08-27
Documentation:

Start using the product.  Read the directions on their WIKI or web site for everything you have to do, and look for places where the documentation doesn't match the real world.  Offer to fix it.

Or better yet at first, look for things that have a lot of questions on the forums, write a HOWTO post and try to make it look like a WIKI page.  If people like it you can either ask to have it added to the documentation, or somebody may do that automatically.  A couple people on this forum did that with the basic security howto.

If you do something like that a couple times, they may give you access to the regular documentation after awhile, and then you're writing docs.  Especially if you volunteer for it.

I mentioned this earlier only half seriously but now based on your CLI comments I want to recommend FVWM again, and hear me out.

FVWM is an old-school window manager that is still supported with regard to current calls to modern libraries, but it's essentially the same FVWM I used back in the 90s.

The disadvantages first, because they're big and there are fewer:
[LIST=1]
[*]The more tailored you want it, the more work you have to put in.
[*]Configuration is a text file.
[/LIST]

The advantages:
[LIST=1]
[*]It's EXTREMELY light weight by modern standards.
[*]There are lots of example configurations to start from.  Most look like the old-school TWM windows but you don't have to follow that model, it's just the default.
[*]The virtual desktops is especially handy.
[*]You can set up apps to open to specific sizes and places.
[LIST=1]
[*]You can get an arrangement of multiple terminals, place them to the pixel ON THE APPROPRIATE DESKTOP, script it and map it to a hot key.
[*]You can have a browser mapped to a hot key, place it to the pixel on whatever desktop you want.
[*]You can control the width of edge handles, scroll bars and the height/font of title bars.  I set the edge handles to 0 and used a modifier key to grab them, set the title to the smallest font I could easily read.
[*]You can control how many special buttons are in the window frame, where they are, what they look like and what they do.
[*]You can make widgets that float above the apps or anchor to root window, toolbars that are wherever you want.
[*]You can have complete control of the menus, including one that opens wherever the cursor is no matter which app is on top.
[/LIST]

[*]You can have all that open up on login.
[*]It's the most configurable window manager I've ever used.  The biggest problem is that if you want it, you have to configure it.
[/LIST]

If you're keyboard oriented, FVWM is really cool.  Everything can be mapped to a keyboard shortcut.

For the first years I used Linux, I never installed X because it wasn't stable yet, it was more of a curiosity than a necessity.  I used terminals for everything.

When I finally tried to use X for serious work, I went through everything available (a much shorter list than it is now) and settled very firmly on FVWM2.  That's what is still out there, but they dropped the 2.  I had it set up so I had 20 or 30 terminals set up, lots of configurations based on the different types of tasks I performed.  Some were for programming, some for debugging (yes, 2 separate things) and some for monitoring or working with large text files.  I had a couple to follow various log files.  All these things started up on login, changed to the proper directories and possibly started whatever task I wanted to run in each one.

### Post by monkeybrain20122 on 2013-08-29
> **bhattabhishek said:**
> +1 to Werne! 

I guess Rock Stable  in Linux world means Debian! Have you given Wheezy a try?

I find this misleading. Debian (Stable) is 'rock solid' only because it is old as fossil. :)   It probably won't even boot on newer hardware because the kernel is too old (e.g UEFI) so there are downsides. If you don't need new features and bug fixes for  your software then Debian 'stability' is for you, for the more adventurous it is very stale. You shouldn't just say it is "rock stable" without pointing out the tradeoff. In Debian talk "stability" just means being predictable, nothing more or less, it doesn't imply better performance or even less buggy (simply because bug fixes for many applications often come through version updates which you will almost never see in Debian except for a backport which is also eons out of date).

Debian 'stability' may or may not be desirable depending on how one balances the pros and cons. Don't know about OP but I did give Wheezy a try, and no thanks, I would rather take a little bit of risk than being so outdated. :) Debian testing is less old (but far from bleeding edge), and Debian unstable (sid) is definitely not stable.
