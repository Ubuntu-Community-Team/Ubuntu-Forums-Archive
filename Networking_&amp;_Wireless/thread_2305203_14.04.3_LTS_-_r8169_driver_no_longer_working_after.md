### Post by RedScourge on 2015-12-03
Howdy,

I had an MSI 790GX-GD65 motherboard and decided to update to something more recent on the same socket type, an MSI 970A SLI Krait edition. The old one uses RTL8111D built-in and the new one uses RTL8111E built-in. It seems to recognize everything just fine in all the troubleshooting commands I've run such as lshw, lspci, ifconfig, ethtool, etc, the only problem is it fails to actually send or receive any data. I changed NO configuration after the motherboard replacement other than updating the device name in /etc/network/interfaces, so this should work fine but doesn't. I have triple checked everything, and am at a loss

Further useful information:

I don't have a firewall on this machine as there's already a network firewall outside of it. I changed NO configuration, other than to update the device name from eth0 to eth2 in /etc/network/interfaces after udev detected the new motherboard ethernet controller.

I tried plugging in a PCI RTL8139D card, which was recognized by the 8139too driver, and it doesn't work either. Strangely enough, the Intel Pro 82541PI Desktop card works fine using the e1000 driver, and when I run Virtualbox, I have an OpenBSD guest which is able to use all the cards just fine, even though the host is unable to use them, whereas my 14.04.3 LTS and Windows 7 guests only work with the Intel network card. I also tried compiling the r8168 driver from RealTek's site, blacklisting r8169, etc as per other guides on this site, and that changed nothing; the host was still unable to use networking and the OpenBSD guest was still able to.

Any help would be appreciated, as I'm only posting here now because I'm at a loss. I'm tempted to go out and buy another Intel PCI card, but that does not sound like a sustainable solution, given that my other 14.04.3 LTS machine has never had a problem whatsoever with r8169 related cards, and both are minimally configured machines which don't even run a GUI, and basically just run virtualbox.


Here's my relevant system info, let me know if you'd like any more info or command output. Realize however that now that I have no Internet on this machine, I can't easily install new programs, I have to do everything by USB copy until this is fixed, and I can't do much to this machine during the day because the VMs running on it are mission critical:

The desired behavior is that they all be able to recognize each other, as they could before the motherboard upgrade.

The OpenBSD machine's IP on bridged eth2 is 192.168.6.3, and eth1 is our ISP-assigned static IP address, as it is the gateway for the entire network. ### Post by RedScourge on 2015-12-04
So, false alarm, consider this thread solved now. After much tearing out of hair, it seems that nothing is actually wrong with the kernel, the driver, the OS, or anything. It turns out that when I reconfigured the adapters, I seem to have assumed that the internal ethernet and external ethernet were backwards of what they truly, are, which started me on a journey of plugging the wrong cables in, reconfiguring /etc/network/interfaces wrong, and re-assigning the virtual machine ethernet adapters to the wrong physical devices, etc. Somewhere in there, because my desktop DNS is run on a different virtual host than the gateway, and because of all of these various things being backwards, and because of me eventually taking shortcuts of checking if Gmail loads as a test of connectivity rather than ping, things got really screwy, what with having my network gateway as a VM, and my DNS and DHCP on another VM server, also as VMs, and them not being able to talk to each other.
