since it kept segfaulting i figured that i would wait a little while, maybe it there was some issue with some package and someone would notice it, then fix it quick. but that doesnt appear to be the case. segfaults most commonly happen after resumption from sleep. heres a couple bluetooth configuration settings i made that might be of importance to note:

(change udev rules for the logitech device, this has been working for awhile)
```

~$ cat /etc/udev/rules.d/70-hid2hci.rules 
# do not edit this file, it will be overwritten on update

ACTION!="add|change", GOTO="hid2hci_end"
SUBSYSTEM!="usb", GOTO="hid2hci_end"

# Variety of Dell Bluetooth devices - match on a mouse device that is
# self powered and where a HID report needs to be sent to switch modes
# Known supported devices: 413c:8154, 413c:8158, 413c:8162
ATTR{bInterfaceClass}=="03", ATTR{bInterfaceSubClass}=="01", ATTR{bInterfaceProtocol}=="02", \
  ATTRS{bDeviceClass}=="00", ATTRS{idVendor}=="413c", ATTRS{bmAttributes}=="e0", \
  RUN+="hid2hci --method=dell --devpath=%p", ENV{HID2HCI_SWITCH}="1"

# Logitech devices
#KERNEL=="hidraw*", ATTRS{idVendor}=="046d", ATTRS{idProduct}=="c70[345abce]|c71[34bc]", \
#ATTRS{ID_VENDOR_ID}=="046d", ATTRS{ID_MODEL_ID}=="c704", \
ATTRS{idVendor}=="046d", ATTRS{idProduct}=="c70[345abce]|c71[34bc]"\
  RUN+="hid2hci --method=logitech-hid --devpath=%p"

ENV{DEVTYPE}!="usb_device", GOTO="hid2hci_end"

# When a Dell device recovers from S3, the mouse child needs to be repoked
# Unfortunately the only event seen is the BT device disappearing, so the mouse
# device needs to be chased down on the USB bus.
ATTR{bDeviceClass}=="e0", ATTR{bDeviceSubClass}=="01", ATTR{bDeviceProtocol}=="01", ATTR{idVendor}=="413c", \
  ENV{REMOVE_CMD}="/sbin/udevadm trigger --action=change --subsystem-match=usb --property-match=HID2HCI_SWITCH=1"

# CSR devices
ATTR{idVendor}=="0a12|0458|05ac", ATTR{idProduct}=="1000", RUN+="hid2hci --method=csr --devpath=%p"

LABEL="hid2hci_end"


```

also i added the second action line there as per some redhat article:

```

~$ cat /etc/udev/rules.d/97-bluetooth.rules 
# Run helper every time a Bluetooth device appears
# On remove actions, bluetoothd should go away by itself
ACTION=="add", SUBSYSTEM=="bluetooth", RUN+="/usr/sbin/bluetoothd --udev"
ACTION=="change", SUBSYSTEM=="bluetooth", RUN+="/usr/sbin/bluetoothd --udev"


```

i also did this cause i was having problems coming back from sleep.. which has been working for awhile now

```


~$ cat /etc/pm/sleep.d/10_bluetooth 
#!/bin/bash

. /usr/lib/pm-utils/functions

case "$1" in
    hibernate|suspend)
    service bluetooth stop
    ;;
    thaw|resume)
    service bluetooth start
    udevadm trigger
    ;;
    *)
    ;;
esac

exit

```
