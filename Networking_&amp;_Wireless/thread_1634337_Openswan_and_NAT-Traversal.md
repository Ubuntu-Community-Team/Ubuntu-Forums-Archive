### Post by Karl1982 on 2010-12-07
I think I've got the problem worked out.  I had a lightbulb go off this morning and realized it was more a routing problem than an IPSec problem.  Because I'm behind NAT and my gateway is not the IPSec host, my return traffic was getting lost on the inside end of the tunnel.

On the external ("road warrior") end, I set rightsourceip to an unused IP inside the NATed subnet.  After pinging the other internal subnet host over the tunnel again and watching the traffic from inside with wireshark, I saw that the pings were hitting their intended target, and then it was ARPing to find the MAC before it could reply to it.  I set up proxy ARP on the internal IPSec host, which *should* have fixed it... but for some reason, even though I enabled proxy ARP for lo, eth0, all, and default in /etc/sysctl.conf, loaded it by rebooting, and set a static ARP entry via *arp -i eth0 -s <virtual-ip> -D eth0 pub*, I couldn't get the IPSec host to respond to the ARP requests.

So what I did instead was used a different subnet entirely.  I set rightsourceip=192.168.253.1 on the external host.  This caused rightsubnet which was unspecified to default to 192.168.253.1/32 on its end.  I also set rightsubnet=vhost:%priv on the internal host.  Then I set a static route in the gateway to forward all traffic for the 192.168.253.1/255.255.255.255 subnet to the internal IPSec host.  The result of this was that the remote host had its own little private subnet on the internal host's network, and any return traffic was sent to the internal IPSec host to be sent back over the tunnel.

It was a lot of hoops to jump through, but it works, and I think it'll remain stable when the road warrior is a moving target.  The road warrior will identify itself to the internal network the same regardless of what its actual IP and subnet are on the far end.  I think the proxy ARP would have been an easier solution had I been able to get that going.  In any case, I should be able to apply this in a real-world scenario just fine.

As for my site-to-site VPN problem, I believe this will work with NAT on both ends with the IPSec hosts not being their respective gateways as long as I set a static route in each gateway for the opposite subnet.  The route simply has to point to the local IPSec host, and netkey takes care of forwarding it across the tunnel.
