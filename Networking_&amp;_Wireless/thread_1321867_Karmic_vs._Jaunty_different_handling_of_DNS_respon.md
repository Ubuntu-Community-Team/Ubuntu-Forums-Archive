### Post by appultaart on 2009-11-10
Hi,

Short: 
My observation: DNS requests are handled differently by my cable modem when sent from Karmic (9.10) vs. Jaunty (9.04), as seen by Wireshark. My question is, why? Is this correct behavior?

Long:
I have acces to the internet via cable modem (Motorola 5120E Surfboard), which is connected to router (Netgear). To the router, two PCs are attached (1x wired, PC-1 running Karmic 9.10 and PC-2 wireless, running Jaunty 9.04). Internet access is mostly fine from my network, but I observed some glitches/hickups in loading pages (esp. pages with many links).

I ran Wireshark as root, both from PC-1 and PC-2. Only internet connection I ran was 'ping' command, no browser/mail was started. 

Procedure:
**(1) **
```
ping 192.192.1.1
```This pings the router by its LAN-IP address, and both Karmic and Jaunty find the router, and Wireshark shows the"ping request - ping response" lines. No differences here.

**(2)** 
```
ping www.tsri.edu
``` This is a ping to a non-existing IP address, and no DNS server is able to respond with an IP-number. Both Karmic and Jaunty give as wireshark output:

No. * Source * Destination * Destination port * protocol * Info
1 * 192.168.1.4 * 192.168.1.1 * domain * DNS * Standard query A [www.tsri.edu](www.tsri.edu)
2 * 192.168.1.1 * 192.168.1.4 * 53682 * DNS * Standard query response, No such name
3 * 192.168.1.4 * 192.168.1.1 * domain * DNS * Standard query A [www.tsri.edu.KWGR614](www.tsri.edu.KWGR614)
4 * 192.168.1.1 * 192.168.1.4 * 50302 * DNS * Standard query response, No such name

So, again, there are no differences between Karmic, Jaunty. I think this is expected  behavior: PC (192.168.1.4) asks router (192.168.1.1) for DNS-lookup; router asks it to the cable modem (which you cannot see in Wireshark), the cable modem asks the Internet Provider, who tells "sorry, no such name" and all is returned to the PC.

**(3)**
```
ping www.ad.nl
```Now, I try to ping a dutch newspaper website. Here, big difference in output as seen by Wireshark.

Jaunty gives:
6 192.168.1.4 * 192.168.1.1 * domain * DNS * Standard query A [www.ad.nl](www.ad.nl)
7 192.168.1.1 * 192.168.1.4 * 53066 * DNS * Standard query response CNAME www6510.persgroep.be A 195.130.156.117
8 192.168.1.4 * 195.130.156.117 *  * ICMP * Echo (ping) request
(...and done, no more lines apart from Echo (ping) requests that remain unanswered).


Karmic gives:
6 192.168.1.4 * 192.168.1.1 * domain * DNS * Standard query A [www.ad.nl](www.ad.nl)
7 192.168.1.1 * 192.168.1.4 * 53066 * DNS * Standard query response CNAME www6510.persgroep.be A 195.130.156.117
8 192.168.1.4 * 195.130.156.117 *  * ICMP * Echo (ping) request
9 212.54.40.25 * 83.82.140.13 * 4477 * DNS * Standard query response CNAME www6510.persgroep.be A 195.130.156.117
10 212.54.40.25 * 83.82.140.13 * 4478 * DNS * Standard query response CNAME www6510.persgroep.be A 195.130.156.117
11 212.54.40.25 * 83.82.140.13 * 4479 * DNS * Standard query response CNAME www6510.persgroep.be A 195.130.156.117
12 ...
13 ...
...
3912... <stopped the Wireshark reading here>

So, first lines (6-8) are alike Jaunty, and I think are the expected behavior. But then, there are about 3900 lines of DNS replies that I do not understand. 212.54.40.25 is the DNS server, while 83.82.140.13 is the IP-address of the cable-modem (i.e., my "outside" IP address). Only thing that changes is the port to which the DNS response is sent: it goes from port #4477 to port #4999, then starts again at port #2056, etc. 

Huh!? Why? What happens here?

Question #1: why does the DNS-server send something to the cable modem, that I am able to pick up on the PC-1 running wireshark(should it not stay hidden?)

Question #2: my internet lag/hickup seems to correlate with these massive DNS replies from DNS server to modem. Is there causality?

Question #3: is this a bug in some package that I should report? Or, do I simply have wrong settings?

Thanks.


```
$/etc/resolv.conf:
# Generated by NetworkManager
domain KWGR614
search KWGR614
nameserver 192.168.1.1
```

NB I first thought this was an Ipv6 problem, a la bug #417757 ([https://bugs.launchpad.net/ubuntu/+source/network-manager/+bug/417757](https://bugs.launchpad.net/ubuntu/+source/network-manager/+bug/417757)) . To exclude this possibility, I disabled IPv6 on my Karmic machine (grub: place GRUB_CMDLINE_LINUX="ipv6.disable=1" in /etc/default/grub). The above-seen lines are with IPv6 disabled, but there is no difference in behavior when Karmic has IPv6 enabled or disabled... So, I have enabled IPv6 again in Karmic...
