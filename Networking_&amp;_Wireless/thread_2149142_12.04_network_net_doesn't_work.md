Although the additional entry in /etc/network/interfaces file should have worked as expected given the mode 'managed=false' in NM configuration, it didn't. So the first thing I'd recommend is to reset it to the defaults (delete everything except the first two lines) and let NM control the device. Dealing with multiple (and non-default) things may create unnecessary confusion.

Also, the blacklisting seems correct to me. Maybe you did not do 'update-initramfs -u' after blacklisting it and loading the older driver (or it didn't work for some reason). The standard method of installing the r8168 driver was to run './autorun.sh' from within the source directory, and it used to rename the r8169 driver instead of blacklisting it (or maybe in parallel to blacklisting it), so there was no chance of it loading and conflicting with the r8168 driver.

That said, like I mentioned earlier, it shouldn't be needed at all with kernel version 3.5 and later, as the native r8169 is good enough. And evidently, it is not doing any good except possibly adding to the problem by conflicting with the native driver. So I'd suggest to un-blacklist the r8169 driver and blacklist the r8168 instead.

The "Carrier : off" and DHCP related messages are also not good, but let's just make sure that the customizations are reset to default first. Then, if "Carrier : off" and DHCP (dhclient) related messages still don't change, check your physical connection (cable, ports, connectors).

I don't know what arduino is capable of doing or if it may affect system settings, so we may also consider a reset of BIOS (unless this is a EFI based installation) to make sure the NIC itself is at its defaults.

So in short, we are going to reset everything back to default (reset /etc/network/interfaces file, un-blacklist r8169, blacklist r8168, reset BIOS if the installation is not EFI based), then start from there if required.

If doing these doesn't change anything, post back -
```
nm-tool
lsmod
dmesg | grep r81
cat /var/log/syslog | grep -iE 'r81|eth|dhclient|network'
```

Not necessarily. A connection profile gets created only when you have a connection first, or you create one manually in NM. You may try creating one.

Bad thing is - still nothing. I tried to look up a few things in the syslog, but nothing relevant comes up.

As per dmesg, the link is simply down, which means either there is no physical connection (or a bad connection), or the interface is unable to initiate the communication for some reason. You should try resetting BIOS if you are absolutely sure the physical connection is ok. Sometimes power-cycling both the computer and the router also does the job when it is a firmware issue. So you should try that as well, just keep the gap between power off --> on long enough to let the device fully reset (usually 20 seconds to 4 minutes is long enough).

One radical change you may try is to disable NM and do a manual configuration for a test (most of the results I found while looking up for error messages were related to NM bugs). Let me know if you need help regarding that.

Apart from these, let's also take a look at ethtool output, but you'll have to install it first. So, if you have an alternate connection on the system -
```
sudo apt-get install ethtool
sudo ethtool eth0
```

If you don't have an alternate way to connect it to internet, then replace the apt-get command with -
```
apt-get install --print-uris ethtool
```
It will give you the download uri for the required package(s), which you can then use to download the packages on a different computer, or the windows part.
