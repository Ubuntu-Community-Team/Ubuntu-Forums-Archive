### Post by DuckHook on 2013-07-02
Hi squakie. Trust all is well with you and yours.

I don't use Samba much (all NFS here), so take the following with a grain of salt, but:

1. Make sure both are not set up to be DHCP servers. Otherwise, they will both attempt to serve out local IP addressess which will indeed collide (and both will assume that they can be the boss and set themselves up as, say, 192.168.1.1).

2. If your IP addresses are being served by a separate appliance altogether (for most people, it's their router), then make sure that they haven't been assigned the same IP address. This sometimes happens if server addresses are static.

3. What do both ifconfig outputs look like?

As for Samba settings, I've forgotten all that I ever once knew, but the man pages should help.

All the right results for all the wrong reasons.  

You can have a Samba server that is part of a domain and is also the holder of workgroup shares. The parameter *security = user* is the default security setting with Samba,  In other words saying nothing implies that the you want the setting *security = user*.  With it in or not in the smb.conf results in the same result.  

You can test both of methods of  configuration with this```

testparm -v | grep security

``` 

You should get something like this ```
	security = USER
```...it should be the same whether or not you have: security = user in the smb.conf file.


If you want to see all the parameters in use by smbd and nmbd you can use this ```
testparm -v 
```

### Post by capscrew on 2013-07-10
Let's start with a little protocol info.  Not mine, but IP addressing and DHCP.  The client (Pi or Zbox) is in control of whether the DHCP server is involved not the router.  

When one of those hosts boots up the OS looks to see how to configure the IP addressing.  If it is configured by the /etc/networking/interfaces file and there is  IP address information the host configures itself and becomes available on the network.  If it uses Network Manager and it is set for static and the information is also there the same thing occurs.  There is no request for configuration from a DHCP server and that is why it is called static configuration.

When one of those hosts boots up and the configuration is set for DHCP (automatic addressing) the sequence is like this:
[LIST]
[*]The client (Pi or Zbox) requests an IP address from any DHCP server by broadcasting
[*]The DHCP server (in the router) offers an address and the lease term from its DHCP pool
[*]The client goes through a selection process for the IP address offered 
[*]The client accepts and configures the IP address
[*]
[/LIST]

This dynamic process can only work if the DHCP server uses only the DHCP pool.  The DHCP server never uses IP address in the network range that is set aside for manual configuration.  There is no way for the DHCP server to know what IP addresses are configured manually unless it could query local DNS server.

This does not mean that other non standard methods can't be used, but it surely makes it hard to rely on the router if you can't control it via configuration.  If what you say is true, then the router is very non-standard.  The only reason we are doing this at all is because we can't use wired Ethernet and you are having trouble manually configuring wireless connectivity. 

Here is what I'm attempting to do[LIST]
[*]Configure a DHCP supplied IP address that is constantly the same (fixed)
[*]Set the lease time to 30 minutes 
[*]Observe whether the request for renewal will serve as a keep-alive
[*]
[/LIST]

The request for renewal is part of the lease of an IP address that is in the DHCP pool.  All the DHCP servers I know of allow you to reserve the IP address via the MAC address from the DHCP pool and set the lease terms to what ever you want.  The lease renewal is attempted at 1/2 the lease time.  This means if I can set the lease to 30 minutes the DHCP server will start to connect to client in 15 minutes and you won't have to rely on cron hacks using ping.  This should be doable.  If not then you will have to use cron and ping.

There is no problem with Samba.  So far the problem is with your router (or at least its configuration). One last though about ARP.  When you think about data traffic you have to think about it in layers.  From bit stream (L1)  to frames (L2) to packets (L3) and back down again .  Each layer has it's own protocols.  When you look only at layer two (L2) the protocol is Ethernet on a LAN (and possibly frame relay or Sonnet) on the WAN.  Ethernet is where the IP addressing is handed off to Address Resolution Protocol (ARP) for delivery to and from IP addressing.  You MAC address is the hardware address that the Link Layer protocol uses to translate software addressing (IP) to the bitstream and on to the ultimate hardware destination (MAC address).  This is all automatic.  There is no reason to inject anything into the ARP cache.  In fact it is to your benefit to allow the system to operate as normal and view the results while checking for a positive outcome.
