I'm going to interject my opinions here, b/c this is a forum, and thats what we do. First off, Linux and Ubuntu, as much as they've been doing to make strides in getting end-users able to slap it on and run it. (which i personally hate anyways.) It still requires a bit of expertise to use, or at least use to its ability.  The WinModem, still to this day notorious for being on of the single hardest devices to get running fully and successfully in Linux. Second of all, it is, like in all computer related issues, a trade off:

The reason Windows works so well is in primarily to the concept of "Plug n' Play" namely drivers of hundreds of devices are pre-installed so when you plug in a device it loads the appropriate drivers for it. Thats all fine and dandy except that you are left with hundreds, if not thousands of drivers to devices you probably will never use installed on your machine.  

With Linux you get more options and more control, the other side to this is that it is also more complex. In Windows you lose the complexity and gain ease of use, but with that you lose customization and control.  If you want to use Linux (any distro) exactly as you would use windows, namely stick the cd in, click next and wait. Then everything *just works*, you should not be using Linux at all. Now I'm not saying that you have to delve deep into your system just to use Linux but you're going to get more out of your system and Linux experience if you do. The drawbacks to some of the distro's like Ubuntu/Redhat/SuSE are that everything is being done in one package so that the largest user-base will have working hardware right out of the box. This still doesn't always work due to the open hardware market. If you really want your devices to work in Linux you may need to go farther into the Linux system itself. Compile/recompile the kernel manually. Build in the right drivers. Download new software and install it yourself. Those of us that use *nix heavily enjoy having those options available to us. But like I said, it is not for everyone. Time and time again people push Ubuntu like it is the "the all-saving distro of Linux", yes, I'll agree that it has indeed made great strides at getting Linux to the end-user desktop, but it is still based around the standard Linux model and therefore suffers and gains the same from all Linux. 

Also, you talk about it being useless without a network connection. This is not the case, if you had a specific task that you wished to complete and did not require a connection, either Windows or Linux this would be sufficient. The big drawback to that model is the ability to add/install new software, web-browsing aside. But you'd have a similar issue on Windows. 

To say the least, I don't think Linux is the best choice for you at this time. While it can be a wonderful operating system, it can also be a pain in the ***, which is why in-depth computer users such as myself love it. But I'll leave you with stressing that it is not for everyone.

I downloaded my drivers from here: [http://martian.barrelsoutofbond.org/](http://martian.barrelsoutofbond.org/) 

I guess you have decompressed the tarball. To compile (FYI) you will need both the linux-headers-`uname -r`and build-essential package.  You can either download these or install them from Ubuntu installation CD.  These make it possible to compile source code.

Anyway within the martian directory - compile and install the driver:
make
sudo make install

Although the driver is installed - you need to load it into the kernel -- unloaded kernel modules dont help anyone :) :
sudo modprobe martian_dev
(Rather than typing this everytime, if you want this module loaded into the kernel automatically at boot do the following:
echo martian_dev | sudo tee -a /etc/modules
)

Now that the kernel modules is loaded, you need to launch the user space utilities to be able to interact with the driver.
sudo martian_modem

You need to keep this running at the command prompt or you can run it as a background process with sudo martian_modem &

I would keep in running in the foreground for now (Which necessitates you opening another command terminal if you want to keep using a command prompt).  It will give you good debugging output in case something doesnt work and you can always kill the modem with Cntl-C if something isnt working quite right -- instant hangup!!

This creates an interface whereby you can communicate or write and read to the modem.  The "file" is located at /dev/ttySM0.  In most dial-up programs that you can use however /dev/ttySM0 is really not an option, and most expect the modem "file" to be at /dev/modem.  Ill have to look how I did it however I basically created a virtual link /dev/modem that linked to /dev/ttySM0.  Therefore I could use the /dev/modem file as my modem interface rather than /dev/ttySM0.  

There are some useful scripts in the /scripts directory.  I will have to take a look at those tomorrow, however that is basically it.  Maybe if I can get you running I will write a formal How-To.  Its fairly straightforward once you go through the exercise once.
