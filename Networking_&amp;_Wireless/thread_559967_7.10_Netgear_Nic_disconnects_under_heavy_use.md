### Post by noob12 on 2007-09-26
Were you running 7.04 earlier without this issue?

Can you save and later post the output of these commands after you get disconnected next time?

```

lspci -nn

sudo lshw -C network

ifconfig -a

tail -50 /var/log/kern.log

grep dhclient /var/log/syslog

```

See if either of the following commands helps
```

sudo /etc/init.d/networking restart

sudo /etc/init.d/dbus restart

```

If either works, it is most likely a DHCP/NetworkManager related issue.  In this case, post the output of ```
cat /etc/network/interfaces
```.  People might see something.  If you have no wireless interfaces you may want to remove NetworkManager entirely.

If you find that you really have to reboot or reload the driver, it is probably a driver or PCI-routing issue. You may get suggestions here by posting the output of ```
dmesg
```

Depending on how this turns out, you may want to file a bug on launchpad.

### Post by noob12 on 2007-10-08
This looks like the culprit.  This is a sign that the PCI IRQ routing at boot didn't work very well.

```

Oct 8 19:16:53 chuck-desktop kernel: [ 6257.088000] NETDEV WATCHDOG: eth1: transmit timed out
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: Transmit timeout, status 0c 0005 c07f media 10.
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: Tx queue start entry 4 dirty entry 0.
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: Tx descriptor 0 is 0008a03c. (queue head)
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: Tx descriptor 1 is 0008a03c.
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: Tx descriptor 2 is 0008a03c.
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: Tx descriptor 3 is 0008a03c.
Oct 8 19:16:56 chuck-desktop kernel: [ 6260.088000] eth1: link up, 100Mbps, full-duplex, lpa 0x45E1

```

This is probably related to the bugs here
[https://bugs.launchpad.net/bugs/85520](https://bugs.launchpad.net/bugs/85520)
[https://bugs.launchpad.net/bugs/76489](https://bugs.launchpad.net/bugs/76489)

I would try booting with the option **pci=noacpi** as a workaround.

I've pasted in below some instructions for how to set boot flags.

**Instructions for one-time boot flag entry**

When booting you'll normally see grub flash a menu of boot options up for a little while.  While it is up  with the default selection highlighted, press **e**, then use arrow keys to navigate to the line that starts with **kernel** and hit **e** again to edit it.  Put the flags at the end of the line, separated from each other and from prior material on that line by space or spaces.  Hit Enter to accept, then b to boot.

To confirm you got the flags entered properly, you should see them if you type
```

dmesg | grep 'Kernel command line'

```
after you boot.

The flags will only apply to that one boot.

**Making the flags permanent**

To make flags permanent (apply to all boots), you need to edit the file **/boot/grub/menu.lst** and find the kernel line corresponding to the boot menu entry you are using and add the options there.  Note that there will typically be some kernel lines that are commented out (preceded by # signs).  You can ignore those.

I suggest that you do not edit this file until after trying boot flags using the one-time per-boot method and establishing that they work and solve an issue.

You will need to be root or use sudo to start your editor.  

**Be careful editing the file**, because editing errors may get you stuck in a situation where you can't boot.   In most situations, you can fix minor errors with a one-time edit for one boot as above.  In the worst case you would have to fix this by booting from the LiveCD, mounting the hard drive, and editing the file.
