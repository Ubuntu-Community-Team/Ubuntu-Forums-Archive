### Post by TheFu on 2022-03-23
20.04 uses netplan. check that as the issue. The old interfaces file is deprecated.
That's where I'd start researching the root cause. Nothing else is jumping out, except the identical metric settings for all three. I'd check into making the metric priorities reflect the needs.

BTW, ifconfig and route are also deprecated. Use the **ip** command going forward for both. I've never seen lies from either if they are there, but ... never know.

### Post by SeijiSensei on 2022-03-24
Check /etc/sysctl.conf and make sure you have forwarding across interfaces enabled. 
```
# Uncomment the next line to enable packet forwarding for IPv4
#net.ipv4.ip_forward=1
```
The default on most modern distributions is to disable packet forwarding as a security measure.

### Post by The Cog on 2022-03-25
I suspect it's some issue external to your box: I can't see anything wrong with the ifconfig or routing table outputs. My next step would be to try the ping again and then quickly look at the output from "ip neigh" command, which will show the ARP entries. I hope you will see an entry for 10.31.0.100 against ens224, but probably INCOMPLETE, meaning that it is not seeing any ARP responses.

I would further try to prove it by running the command:
```
sudo tcpdump -nntl -i ens224
```
in another terminal and look for arp requests while you try to ping 10.31.0.100. From your description, I expect to see ARP requests being broadcast to 255.255.255.255 asking "who has address 10.31.0.100" but no response. But only the tcpdump will really tell you what is really happening. It is concievable that you are getting a good ARP response but that something else strange is happening. Again, tcpdump is the tool to see whats happening.
