### Post by varunendra on 2014-03-06
I couldn't get any obvious clues from above, but a few things you may try/test -

**1)** Try setting the speed to 100 Mbps/full-duplex -
```
sudo ethtool -s eth0 speed 100 duplex full autoneg off
```
If that does not help, reset it by changing 100 to 1000, and "autoneg off" to "autoneg on"

**2)** Writing a .conf file to apply some driver parameters. To see what parameters are available, see the output of "modinfo forcedeth". The ones that look interesting to me are - msi, msix and phy_power_down, and all these take value 1 (enabled) or 0 (disabled). For example, to try the parameter "phy_power_down=0" -
```
echo "options forcedeth phy_power_down=0" | sudo tee /etc/modprobe.d/forcedeth.conf
```
The change will take effect from next boot, or after a manual reload of the driver.

The parameters msi=0, and msix=0 usually help when no link is detected at all. That can be confirmed by either ethtool ("Link detected" value) or lshw (link= value).

### Post by varunendra on 2014-03-07
Taking a look at the /var/log/syslog and normal configuration settings (nm-tool, ifconfig, /etc/resolv.conf, route -n etc.) may give some clue if it is a configuration related problem (intermittent nature usually indicates a DHCP problem in that case).

But if those aspects are okay (for example, when you don't rely on DHCP and the whole configuration is manual and static), the physical connectivity is good as well, then it is hardware/firmware level problem. Doesn't mean the hardware/BIOS/firmware are faulty, just that they are not working nicely with Linux drivers.

Have you also tried the "msi" and "msix" parameters? If not, I'd suggest to try them all -
```
echo "options forcedeth msi=0 msix=0 phy_power_down=0" | sudo tee /etc/modprobe.d/forcedeth.conf
```

You may try once with the msi/msix pair alone, and if that doesn't help either, with phy_power_down as mentioned above. The above command will overwrite the .conf file everytime you run it, so no problem of it getting messed up if the commands are correct.

One more thing you should look at is the BIOS to see if there are any special power management (PME functions) or "IOMMU" options in it. It doesn't sound like an "IOMMU" problem, but is worth checking and test with altered setting if your BIOS has this feature.
