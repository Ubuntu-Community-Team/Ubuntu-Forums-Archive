### Post by #&amp;thj^% on 2021-01-06
In Ubuntu 18.04 and later, the usual arrangement is:

```
ls -al /etc/resolv.conf
/etc/resolv.conf -> ../run/systemd/resolve/stub-resolv.conf
```
Maintaining /etc/resolv.conf is generally the sysadmin's job. 
The only systemd component that actually updates it, is the tmpfiles snippet in "**/usr/lib/tmpfiles.d/etc.conf**", and it _only creates a symlink *if*_ **/etc/resolv.conf** does not exist.
Only if yours is different, which it's not, a semi nuclear approach would be:
```
sudo rm -f /etc/resolv.conf
ln -s /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf
```
I would _most certainly_ make a backup first.

### Post by SeijiSensei on 2021-01-08
If you know the IP addresses of the DNS servers you want to use, you can add them to /etc/systemd/resolved.conf in the DNS and FallBackDNS lines.  For instance, I have:
```

[Resolve]
DNS=192.168.100.100
FallbackDNS=97.107.141.13 8.8.8.8
[...]

```
The first is a local server on my network that runs BIND9.  The other two are a public server I maintain that runs BIND9 and the Google DNS server at 8.8.8.8.

If you take this route, run "sudo systemctl restart systemd-resolved" afterwards to restart the resolver.

### Post by #&amp;thj^% on 2021-01-08
I had an old record lying in my /etc/resolvconf.conf that injected 127.0.0.1 whenever network went up:
```

# /etc/resolvconf.conf
# Configuration for resolvconf(8)
# See resolvconf.conf(5) for details

resolv_conf=/etc/resolv.conf
# If you run a local name server, you should uncomment the below line and
# configure your subscribers configuration files below.
name_servers=127.0.0.1
```

So commented it out, then ran "**resolvconf -u**" and my issue was fixed. Currently using systemd-networkd, so dont't know for sure if this will help marco135.
