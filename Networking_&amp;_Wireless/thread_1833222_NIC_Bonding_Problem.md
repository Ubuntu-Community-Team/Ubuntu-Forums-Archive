Hi,

the product-ID 8168 often doesnt work properly with Kernel 2.6.38 and its module r8169. Try [this]("http://forum.ubuntuusers.de/topic/lan-karte-funktioniert-nicht/#post-3005217") Installation of the r8168.

```
modinfo r8168
filename:       /lib/modules/2.6.32-33-generic/kernel/drivers/net/r8168.ko
version:        8.024.00-NAPI
license:        GPL
description:    RealTek RTL-8168 Gigabit Ethernet driver
author:         Realtek and the Linux r8168 crew <netdev@vger.kernel.org>
srcversion:     F7DC35DC47F0220384E5AE2
alias:          pci:v0000[COLOR="Red"]10EC[/COLOR]d0000[COLOR="Red"]8168[/COLOR]sv*sd*bc*sc*i*
depends:        
vermagic:       2.6.32-33-generic SMP mod_unload modversions 586 
parm:           eee_enable:int
parm:           speed:force phy operation. Deprecated by ethtool (8). (array of int)
parm:           duplex:force phy operation. Deprecated by ethtool (8). (array of int)
parm:           autoneg:force phy operation. Deprecated by ethtool (8). (array of int)
parm:           rx_copybreak:Copy breakpoint for copy-only-tiny-frames (int)
parm:           use_dac:Enable PCI DAC. Unsafe on 32 bit PCI slot. (int)
parm:           debug:Debug verbosity level (0=none, ..., 16=all) (int)
```

The problem may be, that the installation script renames r8169.ko to r8169.bak, which will cause eth1 and eth2 not working. So rename the driver again:

```
sudo mv /lib/modules/$(uname -r)/kernel/drivers/net/r8169.bak /lib/modules/$(uname -r)/kernel/drivers/net/r8169.ko
sudo depmod -a
```

because r8169 includes both IDs:

```
modinfo r8169 
filename:       /lib/modules/2.6.32-33-generic/kernel/drivers/net/r8169.ko
version:        2.3LK-NAPI
license:        GPL
description:    RealTek RTL-8169 Gigabit Ethernet driver
author:         Realtek and the Linux r8169 crew <netdev@vger.kernel.org>
srcversion:     741D80D73655DA31EBC1103
alias:          pci:v00000001d00008168sv*sd00002410bc*sc*i*
alias:          pci:v00001737d00001032sv*sd00000024bc*sc*i*
alias:          pci:v000016ECd00000116sv*sd*bc*sc*i*
alias:          pci:v00001259d0000C107sv*sd*bc*sc*i*
alias:          pci:v00001186d00004300sv*sd*bc*sc*i*
alias:          pci:v0000[COLOR="Red"]10EC[/COLOR]d0000[COLOR="Red"]8169[/COLOR]sv*sd*bc*sc*i*
alias:          pci:v0000[COLOR="Red"]10EC[/COLOR]d0000[COLOR="Red"]8168[/COLOR]sv*sd*bc*sc*i*
alias:          pci:v000010ECd00008167sv*sd*bc*sc*i*
alias:          pci:v000010ECd00008136sv*sd*bc*sc*i*
alias:          pci:v000010ECd00008129sv*sd*bc*sc*i*
depends:        mii
vermagic:       2.6.32-33-generic SMP mod_unload modversions 586 
parm:           rx_copybreak:Copy breakpoint for copy-only-tiny-frames (int)
parm:           use_dac:Enable PCI DAC. Unsafe on 32 bit PCI slot. (int)
parm:           debug:Debug verbosity level (0=none, ..., 16=all) (int)
```
This will cause trouble imho because its possible that r8169 is loaded for both IDs. You may put the following into your /etc/rc.local before the phrase "exit 0":
```
modprobe -rf r8169 r8168
modprobe r8168
modprobe r8169
```
This could be the workaround: First load the needed module for the first device, second load the other one.

No guarantee! If it doesnt work "backup" the r8168.

If you know ho to do, you can alternatively remove the ID 10EC:8168 from the driver r8169.
