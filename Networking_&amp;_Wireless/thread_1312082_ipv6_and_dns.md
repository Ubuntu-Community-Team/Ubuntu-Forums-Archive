### Post by Brandon Williams on 2009-11-02
The sysctl setting works to disable ipv6, as you've already verified. When I set net.ipv6.conf.all.disable_ipv6=1, I cannot find any evidence of ipv6 being in use on my system. However, the name resolver still sends ipv6 (AAAA) requests. AFAIK, this behavior is incorrect, and it's the most likely cause of your problem (as well as the fact that your DNS server is broken, but you probably can't fix that). 

One commonly suggested work-around is to install a caching nameserver on your box, such as pdns-recursor. Another commonly suggested work-around is to use the opendns name servers.

Look [here](https://bugs.launchpad.net/ubuntu/+source/network-manager/+bug/417757) for more information.

### Post by Brandon Williams on 2009-11-03
> **sliketymo said:**
> For what its worth,ipv6 is defaulted to "ignore " in Karmic.

What exactly does this setting do? My interface has been given an ipv6 address despite this setting, and the name resolver continues to attempt to get an ipv6 address with an AAAA request first, _before_ attempting to get an ipv4 address. On my system, this does not cause any trouble, because I don't have a broken DNS server, but it will cause problems for many others.

FWIW ... I don't recommend using centralized dns servers (such as opendns), as this can have a negative impact on performance when you are viewing content that goes through a decentralized content delivery system (20%+ of the internet falls into this category). It's best for you to use a dns server that is as close to your real location as possible. That's why I recommend trying pdns-recursor first, and falling back to opendns only if pdns-recursor doesn't help. I haven't used either method myself, but I've seen both of them listed as recommendations.

### Post by Brandon Williams on 2009-11-03
I wonder if pdns-recursor is simply relaying AAAA requests onto the same bad dns server that was already causing problems for you. Try running tcpdump on the machine running pdns-recursor in order to figure out what exactly is happening with the requests.
```
sudo tcpdump -i any -nvvv port 53
```

You might also want to open a bug in launchpad related to the fact that gethostbyname is issuing AAAA requests, even though ipv6 has been disabled on your machine. If there are no ipv6 addresses, then the resolver should not send AAAA requests. In the end, that's your real problem.

### Post by Brandon Williams on 2009-11-04
> **redmk2 said:**
> Brandon,

A while ago I read an article that stated that slow DNS resolution was NOT IPv6 related in most cases. I can't quote the article as I did not save it.  I wish I had now.

I expect that slow DNS tends not to be caused by IPv6. There are a variety of other possible causes, including the one that turned out to be the problem here ... trouble with wins resolution.

Using tcpdump is an easy way to figure out whether IPv6 is really the problem. If you don't see any IPv6 (AAAA) request going out or you see relatively immediate responses to AAAA requests, then IPv6 is unlikely to be the problem.

That said, there is still a sizable minority of users who have home network hardware that does not recognize AAAA requests, and such hardware often drops such unrecognized requests without responding to it. When this occurs, the resolver must wait for the AAAA request to time out before it sends an IPv4 (A) request.

It is supposed to be true that the resolver will not send AAAA requests when IPv6 is disabled. This is why most users with broken DNS servers prefer to simply disable IPv6. If the system continues to send AAAA requests even when IPv6 is disabled, this method of "fixing" the problem will not work. In that case, switching to a DNS server that understands AAAA requests is the suggested work-around. If you go this way with it, I strongly recommend that you configure a local name server, rather than using opendns.

### Post by Brandon Williams on 2009-11-06
> **shredder12 said:**
> I followed this thread from the beginning and it was great to see that finally the problem was solved..
```
hosts:  files mdns4_minimal [NOTFOUND=return] wins dns mdns4
```
but i didn't understand what was the real cause of the problem
Could someone please briefly explain what does this line and all the options mentioned here really mean??

That line gives the order in which various methods of name resolution should be used when attempting to look up the IP address for a hostname. 'files' means to look in local config files (e.g. /etc/hosts), mdns4 means to use multicast dns (typically provided by avahi), wins stands for Windows Internet Name Service, and dns is the standard internet Domain Name Service.

This config instructs the system to try WINS before trying DNS, but there was a problem with WINS resolution on the user's network, causing the system to wait for a timeout period before trying DNS resolution. Removing WINS from the list solved the problem by disabling the bad name resolution method. Swapping the order of the two items probably would have been equally effective.
