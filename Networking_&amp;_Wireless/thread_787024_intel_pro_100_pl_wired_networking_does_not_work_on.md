### Post by jtrindle on 2008-05-09
OK, you are missing a "gateway" statement somewhere.

I forgot to ask you for your /etc/network/interfaces file, but that is where the change goes.

You should have a section there that starts something like:

iface eth0 inet static
address 192.168.1.72
netmask 255.255.255.0
gateway 192.168.3.2

If that *does* appear somewhere in your interfaces file, the problem is in your choice of IP address.

See, networks which start with 192.168 are "Class C" addresses, which means they only talk to other IPs with the same numbers in the first three positions, without routing.  This corresponds to the netmask I have set above with the first three positions set to 255.  This group is called a subnet.

The gateway must be accessible within the same subnet, or you must explicitly add a route to tell the computer that it can get to "192.168.3.2" via "192.168.1.72".  I'm not sure this will work without another machine forwarding between the two subnets.

So, assuming your gateway address is the router between you and the rest of the world, can you choose a static IP that starts with 192.168.3  ?  That would put it in the same subnet as your gateway.

### Post by .rdg on 2008-05-09
The problem is strictly in the ip addresses. Your selected IP 192.168.1.72 with netmask 24 (255.255.255.0) will never reach your gateway - it's just how networks work.

You should use ip address 192.168.3.x (perhaps 192.168.3.72 if it's not used by someone else) with the same netmask, so it's able to connect to your gateway (and dns) without any problems. Other approach is to change your netmask to /16 (255.255.0.0), which would allow your PC to connect to gateway.

From where did you get the IP address and network mask? One of them is wrong. The first approach I wrote about is safer - if your gateway uses netmask /24 (255.255.255.0) it would not be able to respond back to you with the second one.
