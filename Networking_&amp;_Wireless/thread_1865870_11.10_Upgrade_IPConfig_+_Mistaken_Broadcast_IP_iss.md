### Post by otlayi on 2011-10-20
My setup is for a web server, with multiple C class IP ranges aliased to a single eth0 interface.

After the upgrade, ipconfig shows only my eth0 device and none of its aliases (eth1/lo etc are there as expected).  However these IPs ARE online, netstat shows they're clearly bound to all services and (almost) everything about them works just fine.  Using curl via php I can't bind the interface to these IPs as was working before.

I changed my /etc/network/interfaces from having an additional eth0:24 to adding it as an entry after iface eth0 as "up ifconfig eth0 add x.x.x.x".  After that, it shows up in ifconfig and curl works.

I'm trying to avoid replacing all of my aliases with that (there are few of them).


Second problem, one of my C class ranges has 64 IPs, subnet .192, and starts at .130, every IP except for .143 is pingable no problems.  When I try to ping it from internally on the system it gives "Do you want to ping broadcast? Then -b"

My theory is that because this C class range isn't the primary on eth0, and thus doesn't have it's own gateway set, that Ubuntu is treating .143 as the end of the /26 incorrectly.

I moved the dns that was pointed at .143 so it's not actually being used presently except by stale dns, but I can't get it working.  I moved the eth0:## declaration up higher in the config (before any other IPs in it's C class), and it broke a lot of IPs on network restart, but keeping it where it is currently it's the only IP that went out.

I've had someone else look at the interfaces file and it's all correct, and has been working for quite some time without problems.  I'm assuming that these are both bugs in the latest upgrade, but haven't yet read of anyone posting something similar and/or any potential fixes.  It's not currently killing my system, but it's a bit annoying and has never had issues previously

Your second problem may affect the first problem.  I don't work with web servers so I can't speak to that.  But clearly you have not correctly subnetted your ranges.  The IP address is a boundary address of some subnet and when you ping that address it returns that (e.g "Do you want to ping broadcast? Then -b").  It appears that you are using /26 subnets (64 hosts).  The .130 is not the start of any /26 subnet and this may be your problem.  The nearest range would be 192.168.1.128/26 this starts at .128 and ends at .191.  One of the /28 networks however starts at the same .128 and ends at .143.  remember the the last IP address in the subnet is the broadcast address! > 

My theory is that because this C class range isn't the primary on eth0, and thus doesn't have it's own gateway set, that Ubuntu is treating .143 as the end of the /26 incorrectly.
Gateways have nothing to do with subnetting.  Ubuntu is indeed treating your misconfiguration as a /28 network with the ending boundary at .143>

FYI -- /26 network  4 subnets with 64 addresses (62 hosts)```

192.168.1.0 to 192.168.1.63
192.168.1.64 to 192.168.1.127
192.168.1.128 to 192.168.1.191
192.168.1.192 to 192.168.1.255

```

The first half of the /28 subnets ```

192.168.1.0 to 192.168.1.31
192.168.1.16 to 192.168.1.31
192.168.1.32 to 192.168.1.47
192.168.1.48 to 192.168.1.63
192.168.1.64 to 192.168.1.79
192.168.1.80 to 192.168.1.95
192.168.1.96 to 192.168.1.127
192.168.1.128 to **[COLOR="Red"]192.168.1.143[/COLOR]**
```

Note that the boundaries are all odd numbers and that they are the broadcast address.  The starting number is the network ID number.

```
The CIDR format is: <netID>/ <subnet bits> (e.g. 192.168.1.128 /28)
```
