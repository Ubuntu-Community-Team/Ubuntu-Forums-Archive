### Post by netztier on 2009-03-30
> **u_walker said:**
> hi,

my E220 is shows connection, but 9 times out of 10, firefox does not show the page.


You don't have a ping problem, you have a name resolution problem. 

Are there any other network interfaces up (although, looking at the output of ifconfig -a, I think that there aren't).

NetworkManager has a preference order (based on bandwidth) for the different NICs it manages, and it might therefore not update the routing table (check with **netstat -nr**) or name server configuration (look at the file **/etc/resolv.conf**) correctly.

So for a first attempt, deactivate/unplug your WiFi and LAN connections and bring up the PPP login via UMTS, and then look at the outputs of **netstat -nr** and **cat /etc/resolv.conf** again.

While NetworkManager logs in via UMTS, it might be interesting to look at it's output in /var/log/syslog; you can follow it "live" with running **tail -f /var/log/syslog** in a separate terminal window.


regards

Marc

Hm.. so then the name resolution problem seems to be intermittent. If you have web access trouble again, perform these steps to pinpoint the cause:

[LIST]
[*]check **/etc/resolv.conf** to see which DNS servers have been assigned to your PPP session by the ISP. If there's none, the something with the login process has gone wrong.
[*]ping both/all of these servers (not getting a ping reply does not necessarily mean that they are unreachable: they might be behind a firewall that denies ICMP).
[*]do a manual name lookup: **dig @<dnsserver IP> [www.somedomain.com](www.somedomain.com)** If you don't get a reply, then you should call your ISP and open a trouble ticket. 
[/LIST]

Well that is expected: Because with that dig command, you were not "digging" *your* ISP's DNS (which we know to be 4.2.2.3 and 4.2.2.4), but 195.186.1.162, which is the IP adddress of one of *my* ISPs DNS servers). 

It is common for ISPs to configure their DNS with some restrictions, so  they can be only queried from the IP address ranges they assign to their own customers - hence swisscom's DNS gives back a REFUSED when you query it from your UK based mobile broadband.

So, in your case, you should try this command:

```
user@host:~$ dig @4.2.2.3 www.google.com
```

or another example:

user@host:~$ dig @4.2.2.4 www.theregister.co.uk
