### Post by SillyPoint on 2006-12-01
I've just cut over my laptop (HP Omnibook 500, ca 2001) from WinME to Ubunbtu Edgy. Trying to get it integrated into my home network has made me realise the whole setup needs sorting out.

The network includes a WinME PC (super), an Ubuntu Dapper desktop box (ubuntu), the laptop (laptop), and my wife's Apple (valerie). Valerie connects to my Linksys WRT54G by WiFi, the others connect by ethernet cable.

Everybody can get out to the internet just fine. But they have problems seeing each other. They can all ping each other by IP address (192.168.x.x via DHCP). All four boxes show up by hostname in the network areas of Nautilus and Windows Explorer. Super can ping the other three boxes by hostname. But ubuntu or laptop can't ping anyone by name ("unknown host"). They _can_ ping themselves by name (at 127.0.0.1).

I've set "send host-name" in dhclient.conf, as suggested in other posts here. Domain name is currently null, although I've also tried (somewhat optimistically) using "WORKGROUP".

Samba is running on the Ubuntu boxes. (It behaves a little squirrelly, but that's a problem for another post). But I'm worried that Samba is getting involved in what ought to be a strictly Linux handshake.

The simple question that needs answering is: How is name resolution supposed to work between two Linux boxes connected by a DHCP-capable router? Pretend there's no Windows, no Samba, no Apple -- support for all that can be layered on top of the Linux solution, once it's working.

TIA for any advice.

SillyPoint

### Post by Vermyndax on 2006-12-16
Actually, I did finally get DNS and DHCP3-Server talking to each other.  It was fairly simple to do:

```

apt-get install bind9 dhcp3-server

```The goal is to set up a dhcp server that will assign IP addresses to local hosts on your network.  That's easy.  What wasn't so easy (but is now) is setting up dynamic DNS registration on the DNS server (bind9) so that when dhcp3-server assigns an IP address, it passes along this information to the DNS server to register the necessary records.

The key to getting them to talk to one another is a shared secret.  On Ubuntu Edgy, this is automatically generated by dpkg when you install bind9.  It's placed in /etc/bind/rdnc.key.  The file is secured by chowning bind:bind on the file to prevent others from reading it (key point, because dhcp3-server needs to read that file too but cannot because of ownership).

Now that you have them installed, you'll want to set up your zones.  You should at least set up two zones - one for the domain you're running at home and another for a reverse zone for your IP space.

For instance, if your local hosts are going to get an IP address in the 192.168.4.x space, you want to set up a zone called 4.168.192.in-addr.arpa.  The filename of that zone will likely be set up as 192.168.4.rev.

How you create these files is up to you, (I used webmin because it's more fun), but they should be placed in /etc/bind and linked with the named.conf file.

Here's my examples.

Reverse zone: /etc/bind/192.168.4.rev:

```

$ORIGIN .
$TTL 38400      ; 10 hours 40 minutes
4.168.192.in-addr.arpa  IN      SOA     denizi.red-abstract.com. postmaster.galaxycow.com. (
                        1165213989
                        10800
                        3600
                        604800
                        38400 )
4.168.192.in-addr.arpa. IN      NS      denizi.red-abstract.com.
$ORIGIN 4.168.192.in-addr.arpa.
$TTL 360000     ; 4 days 4 hours
100                     PTR     black.red-abstract.com.
$TTL 302400     ; 3 days 12 hours
110                     PTR     red.red-abstract.com.
199                     PTR     blue.red-abstract.com.
200                     PTR     black-vm1.red-abstract.com.
$TTL 38400      ; 10 hours 40 minutes
3                       PTR     denizi.red-abstract.com.
2.4.168.192.in-addr.arpa.       IN      PTR     yellow.red-abstract.com.

```Here's a forward zone named red-abstract.com.hosts:

```

$ORIGIN .
$TTL 38400      ; 10 hours 40 minutes
red-abstract.com        IN      SOA     miranda.red-abstract.com. jmiller.red-abstract.com. (
                        1165213240
                        10800
                        3600
                        604800
                        38400 )
red-abstract.com.       IN      NS      miranda.red-abstract.com.
red-abstract.com.       IN      MX      10 miranda.red-abstract.com.
$ORIGIN red-abstract.com.
$TTL 302400     ; 3 days 12 hours
red                       A       192.168.4.110
                        TXT     "31c499324746052f54bb3f5955b1f51cd3"
$TTL 360000     ; 4 days 4 hours
black                    A       192.168.4.100
                        TXT     "3149ba7fcd38895effa329cf71fbac0629"
$TTL 302400     ; 3 days 12 hours
black-vm1             A       192.168.4.200
                        TXT     "317ef24c1f92ad5dbcd6ac821ecf1ae0d3"
$TTL 38400      ; 10 hours 40 minutes
denizi                  A       192.168.4.3
gateway                 A       192.168.4.1
denizi           IN      A       192.168.4.2
$TTL 302400     ; 3 days 12 hours
blue                      A       192.168.4.199
                        TXT     "00adcc1eb43dc4eca6786e568af75ff99d"
$TTL 38400      ; 10 hours 40 minutes
www.red-abstract.com.   IN      CNAME   denizi.red-abstract.com.
ldap.red-abstract.com.  IN      CNAME   denizi.red-abstract.com.
webmail.red-abstract.com.       IN      CNAME   denizi.red-abstract.com.

```Note that in this zone file, the names that are listed with a TXT record and a garbled bunch of characters were dynamically registered by dhcp3-server when the host received an IP address.  During operation, these are matched against a .jnl file.  When you first create a zone, you should leave these hostnames and the TXT keys out of your file.  Only create a zone file for machines that have statically-set IP addresses.

Next, edit /etc/bind/named.conf:

```

// This is the primary configuration file for the BIND DNS server named.
//
// Please read /usr/share/doc/bind9/README.Debian.gz for information on the
// structure of BIND configuration files in Debian, *BEFORE* you customize
// this configuration file.
//
// If you are just adding zones, please do that in /etc/bind/named.conf.local

include "/etc/bind/rndc.key";
include "/etc/bind/named.conf.options";
include "/etc/bind/named.conf.local";

// Setting up dynamic DNS
// From: http://www.debian-administration.org/articles/343

controls {
        inet 127.0.0.1 allow {localhost; } keys { "rndc-key"; };
};  

 // restrict access:
acl "trusted-subnet" {192.168.4.0/24; };

// prime the server with knowledge of the root servers
zone "." {
        type hint;
        file "/etc/bind/db.root";
};

// be authoritative for the localhost forward and reverse zones, and for
// broadcast zones as per RFC 1912

zone "localhost" {
        type master;
        file "/etc/bind/db.local";
};

zone "127.in-addr.arpa" {
        type master;
        file "/etc/bind/db.127";
};

zone "0.in-addr.arpa" {
        type master;
        file "/etc/bind/db.0";
};

zone "255.in-addr.arpa" {
        type master;
        file "/etc/bind/db.255";
};

// zone "com" { type delegation-only; };
// zone "net" { type delegation-only; };

// From the release notes:
//  Because many of our users are uncomfortable receiving undelegated answers
//  from root or top level domains, other than a few for whom that behaviour
//  has been trusted and expected for quite some length of time, we have now
//  introduced the "root-delegations-only" feature which applies delegation-only
//  logic to all top level domains, and to the root domain.  An exception list
//  should be specified, including "MUSEUM" and "DE", and any other top level
//  domains from whom undelegated responses are expected and trusted.
// root-delegation-only exclude { "DE"; "MUSEUM"; };

```Note that the top of this file does an include statement to pick up /etc/bind/rndc.key.  That makes bind9 aware of the shared secret.

Now you need to tell bind9 to use your created zones in /etc/bind/named.conf.local:

```

//
// Do any local configuration here
//

// Consider adding the 1918 zones here, if they are not used in your
// organization
//include "/etc/bind/zones.rfc1918";

zone "red-abstract.com" {
        type master;
        file "/etc/bind/red-abstract.com.hosts";
        allow-update { key "rndc-key"; };
        notify yes;
        };
zone "4.168.192.in-addr.arpa" {
        type master;
        file "/etc/bind/192.168.4.rev";
        allow-update { key "rndc-key"; };
        notify yes;
        };

```Edit /etc/bind/named.conf.options to set the servers for your ISP to forward requests to (so you can actually visit websites!) as well as secure zone transfers:

```

options {
        directory "/var/cache/bind";

        // If there is a firewall between you and nameservers you want
        // to talk to, you might need to uncomment the query-source
        // directive below.  Previous versions of BIND always asked
        // questions using port 53, but BIND 8.1 and later use an unprivileged
        // port by default.

        // query-source address * port 53;

        // If your ISP provided one or more IP addresses for stable
        // nameservers, you probably want to use them as forwarders.
        // Uncomment the following block, and insert the addresses replacing
        // the all-0's placeholder.

        // forwarders {
        //      0.0.0.0;
        // };

        auth-nxdomain no;    # conform to RFC1035

        forwarders {
                68.87.68.162;
                68.87.74.162;
                };
        // prevent zone transfers:
        allow-transfer {none;};
};

```You must now restart bind9 to pick up any changes.

```

/etc/init.d/bind9 restart

```Now move over to /etc/dhcp3.  You'll be editing files there.

For starters, copy /etc/bind/rndc.key to /etc/dhcp3:

```

cp /etc/bind/rndc.key /etc/dhcp3
chown  root:root /etc/dhcp3/rndc.key

```When you copy over the rndc.key, it may retain the bind:bind ownership, hence the chown command.  I'm not that familiar with Linux's permissioning rules on move vs. copy (I'm a converting Windows admin), so I throw that in for good measure.

Now, you'll want to define the IP addresses to hand out from dhcp3-server and also how to register those against DNS.  You do this with one file to keep it simple, if you like.  Note that in this file, you will also include the rndc.key you just copied over so it can use this to register records.

Edit /etc/dhcp3/dhcpd.conf:

```

# Basic stuff to name the server and switch on updating
server-identifier           server;
ddns-updates                on;
ddns-update-style           interim;
ddns-domainname             "red-abstract.com.";
ddns-rev-domainname         "in-addr.arpa.";
ignore                       client-updates;

# This is the key so that DHCP can authenticate it's self to BIND9
include                     "/etc/dhcp3/rndc.key";

# This is the communication zone
zone red-abstract.com. {
    primary     127.0.0.1;
    key         "rndc-key";
}

# Normal DHCP stuff
option domain-name              "red-abstract.com.";
option domain-name-servers      192.168.4.2;

default-lease-time              604800;
max-lease-time                  720000;
authoritative;

log-facility local7;

subnet 192.168.4.0 netmask 255.255.255.0 {
    range                       192.168.4.100 192.168.4.200;
        option broadcast-address 192.168.4.255;
        option routers 192.168.4.1;
        option netbios-node-type 8;
        option netbios-name-servers 192.168.4.2;
    allow                       unknown-clients;

    zone    4.168.192.in-addr.arpa. {
            primary     192.168.4.2;
            key         "rndc-key";
    }

    zone    localdomain. {
            primary     192.168.4.2;
            key         "rndc-key";
    }
}

```Note that the DNS server in options should be set to the server you're installing bind9 on (this server, if you followed this to the character).

Restart dhcp3-server...

```

/etc/init.d/dhcp3-server restart

```That SHOULD be it.  Now when your servers connect to the network, if they are configured to obtain IP address and DNS information from DHCP, they will ask your new DHCP3 server for an IP address and the networking options that should be used on your network.  That's fine - but now you have the extra-added bonus of DHCP3 registering your new host with bind9 to provide lookup services to that host (forward and reverse) so you can find them on the local network.

Note there's one weirdness with bind9 here.  When it registers a DNS record, it will create a <zone-name>.jnl file.  If you need to do any restarts on bind9, it may refuse to do so.  If this is the case, delete /etc/bind/*.jnl and try again.

That's all.  Hope this helps.
