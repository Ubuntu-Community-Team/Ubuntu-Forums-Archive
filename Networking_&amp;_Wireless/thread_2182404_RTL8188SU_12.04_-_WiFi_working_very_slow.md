### Post by varunendra on 2013-10-23
> **mik2 said:**
> ```

Bus 001 Device 002: ID **0bda:[COLOR="#FF0000"]8171[/COLOR]** Realtek Semiconductor Corp. RTL8188SU 802.11n WLAN Adapter

```

Okay, so the driver you were originally trying is indeed a wrong one. Your device is quite new in the market and the correct driver at Realtek site is 8192du, which can be downloaded from this page : [http://www.realtek.com.tw/downloads/downloadsView.aspx?Langid=1&PNid=21&PFid=53&Level=5&Conn=4&DownTypeID=3&GetDown=false&Downloads=true#3102](http://www.realtek.com.tw/downloads/downloadsView.aspx?Langid=1&PNid=21&PFid=53&Level=5&Conn=4&DownTypeID=3&GetDown=false&Downloads=true#3102) (the RTL8192DU-VC package for Linux).

However, there is an **even newer version** available at lwfinger's git hub page, and is recommended version. Please follow this nice post with instructions to download and install it : [http://ubuntuforums.org/showthread.php?t=2153777&p=12688576#post12688576](http://ubuntuforums.org/showthread.php?t=2153777&p=12688576#post12688576)

After installing it, you should delete your current udev rules file as it contains some confusing entries which may get more confusing after installing above -
```
sudo rm /etc/udev/rules.d/70-persistent-net.rules
```
It will be automatically regenerated, with only desired entries this time.

One more clarification to a frequent and common confusion. The "Name" of the chip only represents a 'Family' of chips, not the exact chip itself. Which driver will work with it and which won't is decided by looking at the chip's Vendor ID (VID) and  Product ID (PID) combination. Your chip's [noparse]VID:PID is [0bda:8171][/noparse], and the "modinfo" for the module we are trying tells us, among other things -
```
alias:          usb:v**[COLOR="#0000CD"]0BDA[/COLOR]**p**[COLOR="#0000CD"]8171[/COLOR]**d*dc*dsc*dp*ic*isc*ip*in*
```
which means it supports your chip. Sometimes this support is experimental, in which case it is *supposed* to support a particular chip, but not guaranteed. However, the fact that this chip is already supported by a native driver means a newer one should have even better support for it.
