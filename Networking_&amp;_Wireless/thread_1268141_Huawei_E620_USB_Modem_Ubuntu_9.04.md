### Post by louigi600 on 2009-09-21
If your resolv.conf did not get updated after establishing I can think of 2 reasons for that:

1) pppd was not instructed to negotiate dns (need to check how your ppp conection was setup)
2) pppd is not updating the negotiated dns in resolv.conf (but unlikely since mine works and I'm using 904 too)

You can see if pppd negotiated one by looking in /var/log/messages .... see if you see something like this:
```
Sep 21 17:24:16 eng-prova pppd[15636]: Using interface ppp0
Sep 21 17:24:16 eng-prova pppd[15636]: Connect: ppp0 <--> /dev/ttyUSB0
Sep 21 17:24:19 eng-prova pppd[15636]: CHAP authentication succeeded
Sep 21 17:24:19 eng-prova kernel: [31143.901247] PPP BSD Compression module registered
Sep 21 17:24:19 eng-prova kernel: [31143.925572] PPP Deflate Compression module registered
Sep 21 17:24:22 eng-prova pppd[15636]: Could not determine remote IP address: defaulting to 10.64.64.64
Sep 21 17:24:22 eng-prova pppd[15636]: local  IP address 95.74.54.209

Sep 21 17:24:22 eng-prova pppd[15636]: primary   DNS address 213.230.130.222
Sep 21 17:24:22 eng-prova pppd[15636]: secondary DNS address 213.230.129.10
```

Not sure how you configured your ppp connection ... if you did it with the Network Manager (most likely if you are using ubuntu) then check ipv4 setting of your ppp connections and make sure it's set to automatic ppp .... if you configured it manually check that the option "usepeerdns" is set ither in the global options or in the specific peers settings.
