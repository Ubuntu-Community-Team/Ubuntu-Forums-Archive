> i've never had to fight with wireless before on linux, so can you tell me a bit more about what sort

A problem I experienced with Ubuntu 9.10 is installing multiple drivers for the wi-fi device. In my case it installed 3: a non-GPL one, a GPL one and a completely wrong one. You handle this by finding out which one you would like to use (use lswhatever to find hardware number and then look up the relevant drivers in google) and blacklisting the others so that can never be loaded.

Another problem 9.10 inflicted on me was not installing the file that the driver reads when loading to set various parameters (complaints about this were in dmesg.)

I have also seen a fair amount posted about IPv6 causing problems. And NetworkManager. And... But it can be hard to know.  Most posts here are from the helpful but uninformed giving recipes that they have tried. It is not always wise to follow them without understanding what is happening.

The difference between Ubuntu and a distribution that seek to be stable like Centos/Scientific Linux/RHEL is quite striking. Unfortunately, that stability is achieved by continually patching years old software and taking little interest in desktop programs.

### Post by honestguvnor on 2009-11-10
You still have not told us the actual chip. Only the series.

The rt2800sta module is the GPL one and it did not work for me or for others posting here but it presumably must work for some peoples hardware.

There is another module provided by Ralink called rt28XXsta where XX is the number of your chip. This has also been provided by Ubuntu but they have missed the parameter file. If you type "dmesg" you will see an error message along these lines from the loading module.

You should only blacklist the top module (e.g. rt2800sta) and not the dependencies because the dependencies might also be used by other modules. Unlikely in this case but better safe than sorry.

An example parameter file is part of the driver source code for your chip supplied by Ralink on their site. The driver will fail to compile because of changes in the recent kernel but putting the parameter file in the right place in /etc will enable the module to load cleanly.

If you search these forums using the name of your chip or module you will discover plenty of threads with people posting about similar problems to you.

### Post by peepingtom on 2009-11-10
Your Linksys USB Adapter has a wifi chipset made by Ralinkk.

There have been new developments in the drivers for Ralink wifi chipsets. Ralink makes many chipsets, and there are drivers for different versions. Ralink develops many of their own drivers for their many chipsets. There is also the rt2x00 community project to develop drivers that run on many different Ralink chipsets. Your wifi chipset has been incorrectly detected, and inappropriate drivers have been loaded.

You need to determine exactly what chipset is in your wifi adapter, in order to determine what driver is appropriate.

They were listed when you ran the lsmod (list kernel modules) command.
```
rt2870sta rt2800sta rt2x00usb
```

rt2x00 is the Open Source one, it's better. You should try blacklisting the other ones, restarting your computer and testing your connection.

Using the series of numbers listed next to your wifi adapter when you ran the lsusb command, search the forums and google to help identify your chipset.

To see how it complains with the drivers that are loaded by default:

Turn off your computer, remove your Wifi adapter, boot, log in, then open a terminal. Plug in your wifi adapter, run the dmesg command and see what it says, you can post the pertinent last few lines here, you'll know it's pertinent when they start referring to USB.

Search, you can solve this!

This code does not compile against the 9.10 kernel but does against older kernels. I do not know what is used in 9.10 but it is likely to have been modified to work with the new kernel. It is easy enough to apt-get the sources if you can work out the ubuntu naming schemes.

> and could you explain what you mean from your previous post about a non-GPL

In some of your dmesg outputs there is a reference to installing something.bin files. This is often part of the driver that is binary code that comes from the chip manufacturer without source code (non-GPL). The GNU folk tend not to like this and respond by writing complete new drivers that do not use any binaries from the chip manufacturers (GPL). These GPL drivers often do not work as well as the non-GPL ones because the chip manufacturer know everything about their own hardware whereas the GPL folk have to guess, test and deduce. But the GPL driver will get preference in many distributions even though it performs worse. In which case the chip manufacturer may respond by writing a full GPL driver that works because they do not want their hardware running with poor drivers.
