Explanation: Each line is another step from me to the destination.
So Line 1 is my router.
Line 2 is my ISP's connection to its customers
Line 3 is my ISP's connection to a backbone provider, etc.

When a slowdown occurs:
Ping each system in the list, starting at the top.

For line 1:
```
me@my-system:~$ ping 192.168.1.1
PING 192.168.1.1 (192.168.1.1) 56(84) bytes of data.
64 bytes from 192.168.1.1: icmp_req=1 ttl=64 time=0.688 ms
```
time = 0.688 ms, so the problem is not my router.

For line 2:
```
me@my-system:~$ ping 76.229.159.254
PING 76.229.159.254 (76.229.159.254) 56(84) bytes of data.
64 bytes from 76.229.159.254: icmp_req=1 ttl=254 time=9.58 ms
```
time = 9.58 ms, so the problem is upstream from there.

For line 3:
```
me@my-system:~$ ping dist2-vlan60.my-service-provider.net
PING dist2-vlan60.milwwi.ameritech.net (65.43.19.243) 56(84) bytes of data.
^C
--- dist2-vlan60.milwwi.ameritech.net ping statistics ---
11 packets transmitted, 0 received, 100% packet loss, time 10079ms
```
Here is different. The machine never responded to the ping (they are not required to). So try upstream to see of any others respond.

For line 4:
```
me@my-system:~$ ping 151.164.190.132
PING 151.164.190.132 (151.164.190.132) 56(84) bytes of data.
64 bytes from 76.229.159.254: icmp_req=1 ttl=254 time=23.12 ms
```
I got lucky on line 4. Many backbone machines won't respond to ping.
The time is still not bad here - so any slowdown is upstream from it. But it means that the problem is not on my network, and not anything on the ISP's network.

So now I know who to complain to. My choices are Me, my ISP, or All-the-other-internet-users. In this example case, I can complain to the latter. Obviously, if traceroute and ping find a problem with Me or my ISP, complaining to them might be more useful.
