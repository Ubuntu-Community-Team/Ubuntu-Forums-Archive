### Post by Tomosaur on 2007-05-23
Without fully reading all of this, if you think there's a GPL violation, you're supposed to report it to the copyright holder of the original GPLd code:

[GPL Violation Info](http://www.gnu.org/licenses/gpl-violation.html)

Of course, from what I've read of this thread there's some ambiguity about whether the mod author is or is not obliged to provide the source. Regardless of whether the requestor is perceived to be 'rude', the developer is **legally obliged** to provide the source code, if his or her work is a derivative of GPLd code (which I expect it would be, being a modification). There are no ifs and buts, he/she just has to do it.

I would suggest you guys either go through and figure out whether the mod author is violating the GPL, and if so, take appropriate action.

If the mod author is not using GPLd content, then he/she has no obligation to release the mod content - so even if you do receive the source code, you may not be able to get the mod to work without the mod content (you can try creating your own, which you're allowed to do). However, if the mod is a derivative of GPLd code, then the author needs to GPL that code, and provide it either pre-packaged with the binary of the software, or make it available under the same conditions as the binary (ergo, freely available).

Oh and Chris - perhaps you may like to remind yourself of this stipulation from the [GPL FAQ](http://www.gnu.org/licenses/gpl-faq.html#GPLRequireSourcePostedPublic):

> 
If a library is released under the GPL (not the LGPL), does that mean that any program which uses it has to be under the GPL?
    Yes, because the program as it is actually run includes the library.


If your software requires the use of GPL libraries, then your software should be under a GPL licence. However, I haven't looked extensively into the Dark Places engine, but I would imagine that this applies to any mod which depends on it to function. If the Darkplaces engine has libraries which you link to in your mod, then I'm guessing your mod is bound by the GPL.

Oh yeah - if you're distributing the darkplaces engine, you also need to provide the source code to it, regardless of whether it has been modified or not. Telling people to 'go to the Darkplaces website' does not count as distributing, you need to host and make available the source code yourself.

### Post by LordHavoc on 2007-05-26
You are misunderstanding a great many things about a Quake-based game, the engine source (the only GPL part) is not specific to the game, it is simply an emulator that runs the game.

No demands can be made on the rest of the game as it is separately licensed, and accordingly it would be difficult to legally 'fork' as the Darsana developers have not applied an open license to it.

The comparison to Nexuiz in the forum thread is thus quite misplaced, as Nexuiz is a wholly GPL game (a rare breed indeed), Darsana is a closed work that happens to use a GPL engine to play it.

The Darsana game consists of these components:
1. DarkPlaces Engine - this is my (Forest Hale's) DarkPlaces project based on the Quake source code released by id Software, it runs the game but does not define any of the gameplay, it contains the QuakeC virtual machine (emulation for the "QuakeC CPU" as it were), it displays graphics, plays sounds, etc, but does not have any concept of how the game "Darsana" behaves.

2. Darsana progs.dat - this is the game logic written by Chris Page, it defines all of the behavior of the game "Darsana", it is a a non-native binary interpreted by the engine, it was compiled from a directory of .qc files (QuakeC language source) using a qc compiler (such as fteqcc, frikqcc, or the original qcc, or any other variant), this runs on any platform without porting work because it is non-native, it is also relatively easy to decompile (comments are lost, and constants are often misnamed, but the logic is preserved, so it can be maintained, albeit not perfectly, without the original source), frikqcc includes frikdec which decompiles progs.dat files back to .qc source.

3. Darsana Art - this is the rest of the game, the maps, models, sounds, images, and other things that make the game not look like a bunch of cubes moving around.

Chris lost his changes (around 15 lines) to the engine that replaced some of the menu art with text (to make it usable without the image-based menu items which he did not want to create for his game), so unfortunately he can't comply with the GPL demands on this component as he does not have the changed code anymore (due to hard drive failure), however the changes aren't very important.

Chris lost his progs.dat source code (around 17000 lines) which was based on the released Quake1 gamecode (also a progs.dat) which had no license included (id never slapped a license on it back in 1996, licensing gamecode was not a common practice back then, as Quake was the first game to have separate gamecode which was modifiable), so it is not under GPL and no demands can be made, additionally (as mentioned) his hard drive failed and he did not have backups of the game sources.  Chris is working on Darsana 2.0 with new gamecode.

The engine interprets the game code as data, and consequently the game code is not governed by the GPL, nor is any other data (the GPL only applies to things that link to the executable).

The only legal leverage involved in this whole situation is regarding the unreleased mofifications to my DarkPlaces engine but they simply consist of presenting a slightly different menu and loading from the "ddata/" directory instead of "id1/" as Quake1 would do, which do not seem terribly important to me.

I have made changes to detect missing game data that caused the menu to stop working at all in Darsana but that is simply the nature of upstream changes - Darsana simply does not include some images that the engine expects so it thinks data is missing.
