### Post by Tuna-Fish on 2007-10-07
One thing to note is that gaming is a very good example of the 80/10 rule (the program spends 80% of it's time on 10% of the code), only, in gaming it's really more like 98/5.

The game engine itself should be written so it'll be as fast as humanly possible, so c++/c is the best bet. However, most of game programming time tends to go to developing the content, which in anything reasonably interactive needs a truckload of coding. It will, however, only be executed relatively rarely, making it an ideal case for something like python.

The only safe bet for future is that no game is developed on just one language. At this day and age, doing so would just be utterly and unthinkably dumb.

And it's not just me thinking this way, the smartest game devs (the ones selling most games...) have already figured this out. For examples, look at Civ 4 and TES: Oblivion. Both are games with the core developed in something fast and content scripting done in python.
