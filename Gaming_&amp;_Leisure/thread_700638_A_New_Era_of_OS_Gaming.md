### Post by Ferrat on 2008-02-20
Well depending on game type but you need to be more specific I think when it comes to was is needed.. 

Programming has a lot of parts, specially if you don't use a "Game Engine" that's already done. 

"Game Engine" 
 - Grafic handling 
 - Physics even if just basic physics are needed
 - Sound
 - Input and output responses 
 - Read and write (as in what formats you use for models ect)
 - Network if any
and these are just basic stuff, and if you do use a finished game engine you still have scripting and coding the game to use the engine and connecting all the dots. 
There are a few good game engines out there that either cost money or is free but comes with legal restrictions, IMO you should take a look at Panda3D, it's a good engine, free and open source made and used by Disney, it's based on C++ making it very potent and with python bindings making it realtivly easy to use and learn, Disney created it for their MMO Toontown and used it for their MMO Pirates of the Carribean.
It's open source but any modifications must be sent to Disney so if they want they can implement them, also it's a fully working engine with most parts done and working and it can run natviely on linux, also works with 3D models created in blender (with an blender addon)

in grafics you got 
 - Modeling
 - Texturing
which are big jobs both of them and then you got level creation which is a b**ch if even if you create or have an level editor

It's a Major undertaking to create a game so before you start you need to put out guidelines, goals, mile stones, concept art, for an RPG I would recommend at least a basic storyboard, preferably a pretty detailed idea of the story, encounters and rules ect. to find out exactly what you need and don't need. 
Also if you really get anywhere with it don't be shy to ask for help ect. there are companies out there that want to help opensource and it never hurts to ask and you can use all the help you can get and make sure you got the hows and don't down, schedules and ways of communication ect. so you don't have two people doing the same thing or things that doesn't work together and so on. 

What type of game play?
What type of music?
What typ of models?
What typ of textures?

then figure out the best way and the easiest way to get where you want to go, for ex. a full 360degree 3D view rpg is cool but does it really suite your game? do you need areas generated at random or can you use more static environments, does the stones on the ground really need to be full 3D or is there any easier way that takes less comp. power? 

lots of these and more questions needs to be addressed even before you begin or you will have a lot of unnecessary work that will slow you down.

### Post by Jadd on 2008-04-10
Dude, I'm glad you want to get game programming experience, because you're going to get some. Sadly, it looks like you're going to get it the hard way. Game dreams like yours just don't get anywhere, unless I've totally misunderstood you. Believe me. I've had some, and they didn't get anywhere. Believe the dozen other people on this thread.
I'll guess you'll learn like I did how to be realistic. It's great to have big dreams, but start small. You will get to the huge stuff eventually, but don't try the motorbike stunts before you learn to ride a bike. Plus, **it's much more fun that way**. Trying something that's just way too hard will just frustrate you, you'll spend tons of time learning two dozen reasons why you can't do it, and have nothing to look back on and think: I did that!
Here's my advice: (I apologise if you've already done this, perhaps I've received entirely the wrong impression)
- Pick up some books and learn how to program.
- Pick a very small project which you know you can code and code it. You could make a 2D checker game with no AI, just hotseat mode. You'll get real experience and a finished product which you can add to your private portfolio. This will feel great.
- Pick another project, this time with moving parts, say, a Tetris clone. Give it a twist to make it original.
- Try an RPG with just two levels, and a very simple world and a very simple mission, say, deliver three parcels to their respective owners. This will make you learn a 2D game engine, how to load maps, how to load and save games, plus graphics skills, and the basics of drawing sprites. Give it a funny storyline, or one from your own personal life (make a game with your friends and family in it), and then you can show it off even if it's tiny. That will feel great too.
- After you've done that add some bad guys that chase you. This will teach you some AI, the hardest part in game programming, in my experience. Also a bit of level design.
- Make a basic 3D game. Say, Tetris 3D, with a square at the bottom.
- Take any of your games and add a multiplayer aspect to it, over the Internet and over a LAN.
- Now that you've got actual experience learning games, you really know how good you are, and what's doable.
When you want to make a sizable project,start making a game design document (see [this](http://www.gamedev.net/reference/list.asp?categoryid=23#121)). Then make a small demo or proof of concept demo. Then post a thread in a forum (hopefully ubuntuforums.org :)), and start coding! Even with a design document, I recommend you have several developers in the same area or at least time zone with plenty of spare time so that you can work at the same time.
If you have any questions, do ask. I'll be glad to help, if I can, and so will many in the Linux community. If you get stuck making a mini-game, I'm sure some-one will help you get unstuck. I hope to be seeing some code soon!
And my final piece of advice: keep dreaming! Dreams are good, seriously. Success requires dreams, and the better the dream, the better the results.
