### Post by estyles on 2009-01-30
The biggest reason I think, is that it takes effort to port a game to Linux and the market just isn't big enough.  The number of Linux users is small.  The percentage of those who are willing to pay for games is small.  And the percentage of those who will actually show up to buy any one particular game is not that large either.  And on top of that, the *perception* of the Linux market is that it is even smaller than it really is.  All reasons for game companies to not bother.  The effort put into a Linux port can more profitably be put into an expansion for the Windows version.

I think all it will really take is for one or two companies to put out a wildly successful Linux port of a game and show that it doesn't take that much effort (which it wouldn't if planning was put in to use engines that are as portable as possible), and that there's a profit to be made (which there would be if the game was lucky enough to find a niche to appeal to a large percentage of Linux users).  I don't know when or if that will happen because there are a lot of factors to fall into place, but when/if it does, I think that there would be a lot more Linux ports shortly afterward.

For now, though, I think the future of playing games in Linux is more dependent on Adobe making Flash not suck so hard.  (Granted, there are some good Open Source games available in the repos, including some very well-done FPS's - but I hate FPS's...)

### Post by xx.canes.rites on 2009-01-31
I took this comment from [http://www.freesoftwaremagazine.com/node/1591](http://www.freesoftwaremagazine.com/node/1591) posted by anonymous.. I thought it to be very interesting. Here's what he had to say.

I've been MMO gaming for years both on Windows and Linux (mostly FPS/flight sim games). I've used Wine and Cedega with mixed success (spent some coin on a Cedega membership for awhile). I am also a developer - I do integration work for a fortune 500 corporation on Unix/Linux as my day job.

I've finally come to the conclusion that as long as Linux gamers let Windows call the shots we will always have mediocre gaming on the Linux platform. Microsoft is not motivated to make it easy to interoperate with their system - in fact if you look at Steve Balmer's recent comments ([http://blog.seattlepi.nwsource.com/microsoft/archives/108806.asp?source=rss](http://blog.seattlepi.nwsource.com/microsoft/archives/108806.asp?source=rss)) if anything he would love to see Linux disappear altogether - or failing that, make sure everyone is on SUSE Linux which will provide a kickback to Microsoft - and even then you can be sure, from reading his comments, that integration with SUSE will always be a marginal affair if it impacts Windows numbers.

We can't settle for emulation anymore because that can be broken with every Windows service pack. We must make it financially viable for developers to port their products to the Linux platform - and I think that starts with tools. Microsoft has made game development easy with their directX APIs - there is no equivalent system for Linux - just a hodge-podge of competing systems that makes it difficult for new and established developers.

Here are my suggestions:

1. Build a common development framework and APIs that leverage OpenGL, and equivalent libraries for sound and user input devices into one package. Make it as easy to build games on Linux as it is on Windows.
2. Build a directX converter, that translates directX code to this new API. Make it easy for Windows developers to throw Linux users a bone.
3. Build a high performance online game engine and client that leverage the new framework - that is easy to install either by the user, or as a deliverable with games (and desireable for distros to include as a standard feature - so it becomes ubiquitous). Make it easy for new developers to get into the business on the Linux platform.
4. Release all of it under the GPL to the game development community -- and get some heavy hitters behind it (such as RMS and other respected people in the linux game world - Loki et al).

There are only two things holding back desktop Linux: an easy to install and upgrade distro that just works out of the box (and Ubuntu, Linspire and others are making huge inroads in that area), and a game platform that just works, and is native to Linux. All the other pieces are in place to gain parity with the Windows desktop. This is the 11th hour, and I think Microsoft knows it. Yet, as much as I look on the net, I don't see the projects we need to go the final mile...

### Post by Melcar on 2009-02-01
Once upon a time there was Loki Entertainment.  While most of their stuff revolved around the porting of Windows games, their end goal was to establish and jump start a native Linux gaming community that would stand on its own.  For this they developed tools to facilitate game making.   From installers to common API frameworks; everything that was needed to start a healthy Linux gaming community was made available by these enterprising people.  However, they were too ahead of their time; at that moment there was just no real interest in Linux gaming.  
Nowadays we do have a growing community that is interested in gaming.  Unfortunately, we have now reverted to porting and emulating, the very thing Loki tried to avoid and warned against.  
Software like Wine is simply horrible for Linux gaming as a whole.  Such software will simply reinforce the notion that games can remain the domain of a single platform.  What is needed instead is game development that is platform independent.  The tools for this are already there; they just need to be put to good use.
The big AAA developers will not bother making non-Windows games; they are already locked inside the high risk Windows gaming market, or even worse, are moving to consoles.  The only hope really is in smaller indie houses that are not locked into the mindset of huge blockbuster games and instead choose to make games simply for the love of game making.  Yes, such games are very "low tech" compared to most AAA titles, but you can't always have everything.

--- As gaming on *nix is very much a secondary priority, it doesn't get the attention it perhaps deserves. I say *perhaps* deserves as, lets face it, PC gaming has seen a bit of a decline in recent years.
--- There are some free professional-level tools out there but they're out of date (for instance, the older Quake/Unreal engines have been released under OSS licenses which is great but this partially explains why pretty much every game on *nix still looks like Quake 3 Arena).

- It *is* a bit of a jungle out there. Not just for developing games on *nix; for developing any complex software on *nix, there's a lot of hype around a new IDE, language, library every other week in Linux Land. There's also a lot of backwards-compatibility-breaking updates (see recent storm in a teacup over python 3.0) and project forks due to maintainers disappearing, disagreements within communities, etc. On Win, it's very much a case of if you've got the money, you buy the big, standard tools. If you don't, you choose an assortment of free/cheap tools. On *nix, you've got a lot more options and it isn't immediately clear which ones are best for your project. It's entirely possible to find the perfect match but from a preliminary feasibility investigation by a commercial developer, it's easy to get lost. Perhaps there is a business opportunity in just this: creating a company whose job it is to match the tools to their client's project.


Like many problems on *nix, you've got a chicken-egg scenario. The only reliable and practical way to make "X" issue better is to make *nix more visible, which means more popular but "XYZ" issues are in themselves a barrier towards establishing that popularity.

It's also a matter of priorities. *nix' commercial bread and butter has always been the server; convincing people to invest in the home desktop arena is difficult. Of course, MS' bread and butter was the business and in making a gamble on the home computer, it paid off big time but this little factoid seems to allude the big firms actually paying for a lot of the most important OSS work.

### Post by curvedinfinity on 2009-02-01
OpenGL is an industry standard graphics API specification.

[http://www.khronos.org/members/promoters](http://www.khronos.org/members/promoters)

The implementation of which is handled by the closed graphic drivers we have come to know and love.

First of all, there is nothing "broken" about OpenGL. It works great, and is easier to learn than DirectX. It has a very long capability list. That said, graphics "capabilities" are silly. "Shader 3.0" is a ******** marketing term. It has real graphics capabilities built into it, but these capabilities can _always_ be duplicated with other methods. In other words, the only companies that really should care about the latest graphics API capabilities are the ones that produce bleeding-edge graphics engines. However, those same companies have to support low end machines with less capabilities, so its a silly argument that DirectX is better because it has more "capabilities" -- you would be playing into graphics marketing spin.

Second, because the OpenGL specification is designed by committee, it takes longer to get new versions to market. This is offset by the fact that it favors no one. It is fair, and it is designed to support a wide range of devices. Linux is only along for the ride, since OpenGL is also the only thing that runs on most mobile phones, game consoles, and all Macs. -- At that, I have not heard much complaining about the graphics on the Playstation 3. It uses a modified version of OpenGL.

It is perfectly feasible for all of the game companies to support Linux. They already support OpenGL on their multiplatform engines, because they need to support the wii and Playstation 3. There is no shortage of OpenGL talent out there either. A game made with top-tier OpenGL specification, versus a game with top-tier DirectX specification will look nearly identical, so that's not the problem either.

The problem is that the business units of big game companies cannot justify spending approximately equivalent resources supporting another platform, when that platform has going on 2% of the users of Windows and perhaps %1 of a console. And that's all she wrote.
