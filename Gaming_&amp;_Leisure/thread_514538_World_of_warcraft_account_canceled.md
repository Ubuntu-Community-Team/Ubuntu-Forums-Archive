### Post by Sammi on 2007-08-01
[http://en.wikipedia.org/wiki/Wine_%28software%29]("http://en.wikipedia.org/wiki/Wine_%28software%29")
> **Wine** is a project which aims to allow a [PC]("http://en.wikipedia.org/wiki/IBM_PC_compatible") with an [x86 architecture]("http://en.wikipedia.org/wiki/X86_architecture") processor running a [Unix-like]("http://en.wikipedia.org/wiki/Unix-like") [operating system]("http://en.wikipedia.org/wiki/Operating_system") and the [X Window System]("http://en.wikipedia.org/wiki/X_Window_System") to execute programs that were originally written for [Microsoft Windows]("http://en.wikipedia.org/wiki/Microsoft_Windows"). 
...
 The name 'Wine' derives from the [recursive acronym]("http://en.wikipedia.org/wiki/Recursive_acronym") *Wine Is Not an Emulator*.
...
Rather than acting as a full [emulator]("http://en.wikipedia.org/wiki/Emulator"), Wine implements a [compatibility layer]("http://en.wikipedia.org/wiki/Compatibility_layer"), providing alternative implementations of the [DLLs]("http://en.wikipedia.org/wiki/Dynamic-link_library") that Windows programs call, and processes to substitute for the [Windows NT]("http://en.wikipedia.org/wiki/Windows_NT") [kernel]("http://en.wikipedia.org/wiki/Kernel_%28computer_science%29").To some extent, Wine is a open source implementation of the proprietary Win32 API.

[http://en.wikipedia.org/wiki/Compatibility_layer](http://en.wikipedia.org/wiki/Compatibility_layer)
> In [software engineering]("http://en.wikipedia.org/wiki/Software_engineering"), a **compatibility layer** allows [binaries]("http://en.wikipedia.org/wiki/Binaries") for an emulated system to run on a host system. This translates [system calls]("http://en.wikipedia.org/wiki/System_call") for the emulated system into system calls for the host system. With some [libraries]("http://en.wikipedia.org/wiki/Library_%28computer_science%29") for the emulated system, this will often be sufficient to run binaries for the host system.
 Examples include:[LIST]
[*]the [Linux]("http://en.wikipedia.org/wiki/Linux") compatibility layer on the [BSDs]("http://en.wikipedia.org/wiki/BSD"), which emulates [Red Hat Linux]("http://en.wikipedia.org/wiki/Red_Hat_Linux") using a system call translator and Red Hat libraries;[[1]]("http://en.wikipedia.org/wiki/Compatibility_layer#_note-0")
[*][Wine]("http://en.wikipedia.org/wiki/Wine_%28software%29"), which runs some [Microsoft Windows]("http://en.wikipedia.org/wiki/Microsoft_Windows") binaries on [Unix-like]("http://en.wikipedia.org/wiki/Unix-like") systems using a program loader and the [Windows API]("http://en.wikipedia.org/wiki/Windows_API") implemented in DLLs;
[*][Windows XP]("http://en.wikipedia.org/wiki/Windows_XP")'s use of compatibility layers to attempt to better run [Windows 98]("http://en.wikipedia.org/wiki/Windows_98") and [MS-DOS]("http://en.wikipedia.org/wiki/MS-DOS") applications;[[2]]("http://en.wikipedia.org/wiki/Compatibility_layer#_note-1")
[*]The [Classic]("http://en.wikipedia.org/wiki/Classic_%28Mac_OS_X%29") environment to run [Mac OS 9]("http://en.wikipedia.org/wiki/Mac_OS_9") programs in [Mac OS X]("http://en.wikipedia.org/wiki/Mac_OS_X").[/LIST]A compatibility layer avoids both the complexity and the speed penalty of full hardware [emulation]("http://en.wikipedia.org/wiki/Emulation"). Some programs may even run faster than the original, *e.g.* some Linux applications running on FreeBSD's Linux emulation were often held by advocates to perform better than the same applications on Red Hat Linux.
Hope this clears that up :D
