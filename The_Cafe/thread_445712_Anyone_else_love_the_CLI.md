### Post by Sunnz on 2008-11-21
I can't vote for neither, as I am also one of the people who uses both.

CLI is just more simple to use when you end up with a directory with many files, like /etc and you only need to edit one file. Having to scroll through that in a GUI would be a pain compared to tab completion in CLI.

Of course sometimes it happens that GUI turns out to be more efficient for some other task but you already have a CLI window opened, so what do you do?

This is where I really appreciate from open source Desktop Environments, all you had to do is to drag whatever file in the file manager to the CLI, and the full path is magically filled in for that file in the CLI for you.

That's what I prefer to do, I don't have to worry about what's in the GUI part of the system and what's in the CLI part. They are all there to help me to get things done. I don't even need to think about this most of the time, just use whatever I find the easiest.

That's what I call efficient, it is not about the fewest keystroke or shortest time possible to get one little task done; but how quickly can you apply your what you know to a given situation. E.g. if there something I already know how to do in CLI, then I'll just jump into the CLI and do it, there are no need to learn how to do it in GUI at that moment, and vice versa.

Overall I think using the CLI is a very important skill for people using a wide variety of computers. It tends to be very similar across different Linux and Unix systems and it doesn't matter what Desktop the system uses, that is, one can adapt a slightly different CLI on Linux and Unix systems than adapting different Desktop environments. CLI always works when the GUI breaks, and sometimes a GUI may not be available, for example if it is in a headless configuration; or if the computer is several hundred KM away and tunnelling a whole DE over a DSL link is just too slow.
