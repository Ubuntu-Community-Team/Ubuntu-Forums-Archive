### Post by geonik250 on 2008-09-05
I see that there is some confusion as to whether a quad processor is better than a core duo and if applications and under which circumstances are benefited by multi core architectures. I will try to shed some light into this debate using my knowledge on computer architecture. Bare in mind that this will be somewhat oversimplified for the sake of avoiding non essential details.
The basics. The CPU is a piece of circuitry having several millions of transistors on it through which electric current is conveyed. This piece of circuitry can be accessed from the outside world through the electrical contacts of the packaging. Now depending on what voltage is applied to the input contacts and the internal state of the circuitry we get appropriate voltage on the output contacts of the CPU. This is what program execution is about. Input voltages are the instructions and data fetched from memory and output voltages are the results being written back to memory. For convenience but mainly for having a formal mathematical representation we symbolize these voltages with 0 and 1. And yes that is the machine code or "assembly" if you like. The state of the circuitry is maintained in on chip constructs called registers. Think of them as very fast, directly accessible memory cell. It is on these registers that calculations are performed.
Now in the old days it used to be that if we fed the CPU with an instruction we had to wait until it calculated the result and wrote it back to memory in order to be able to start executing the next instruction. That was how 8086 and 80286 CPUs worked. In these architectures an increase in frequency yielded an exactly proportional increase in performance. At some point this became unacceptable since more performance was needed.
That is when the pipeline came along. What the pipeline or pipelining of instruction execution does is to divide the execution of a command in stages. Think of it as an assembly line in a factory manufacturing automobile engines. In a non pipelined factory one worker would be responsible for the construction of the entire engine by itself. In the pipelined factory an assembly line is used where each worker is responsible for only a small part of the engine manufacturing process, and relies on workers before and after him to complete the engine. The effect is that, at least initially, the more stages you have in the pipeline that much faster it is to produce the result.
Well not quite. The execution time of a single command is the same in both cases. If we had the exact same CPU but one version with and one without pipeline the time needed to execute a single instruction would be the same in both. After all it's the same circuit in the same frequency. What we gain is throughput. While an instruction is in a stage in the pipeline the next instruction is in the previous stage being also executed, like there are many engines on an assembly line being manufactured. It may take the first instruction the full time of the pipeline to be executed but the next one will come out after it in the time it takes to complete the last stage. It's just like in the factory. The first engine when the shift starts may take a while to complete but after that a new engine comes out of the assembly line every few minutes.
So if you split the instruction execution in let us say 5 stages you get a CPU that is 5 times faster. Great! That was the 80386 CPU.
The problem with the pipeline is that in order to have the performance benefit you need to feed it instructions on every clock tick. If we have no instruction available it will have to wait for one. It is like we have run out of parts for the assembly line. How can this happen in a computer? Well in many ways but the most easy to conceptualize is data dependency. Suppose you have to feed these two instructions in a CPU.
1. c=a+b
2. d=c*e
The variables are the registers in the CPU. How can you start executing the second instruction before the first one is completed? You can't. You have to wait to see the value for the c register before you begin calculating the value for the d register. This is disastrous for the pipeline. Something like this regresses the performance of the pipelined CPU to that of the non pipelined one. There are many techniques to overcome this but the one that is relevant here is hyperthreading (that is the term used by intel). (By the way we skipped a great many processor generations by going directly to hyperthreading. You now know there was a reason for intel to produce so many models until it implemented hyperthreading)
What hyperthreading does is provide two sets of architectural registers. That means that while there is one CPU there are two of each register.
There is a and a', b and b' and so on. What this does is that while we have only one CPU we can have two "virtual" CPUs. Each one has its own registers therefore has its own state and can execute a completely different program than the other. They share the same execution pipeline ("core") for the instructions but can operate independently. Now we have a solution to our problem with the blocked pipeline. If one "virtual" CPU finds itself in the predicament instead of leaving the pipeline empty the other "virtual" CPU can take advantage of it by putting in its instructions while the first one waits for the result of its own instruction. It's like we ran out of engine parts for the assembly line but instead of leaving it idle we feed it parts for the manufacturing of the automobile gears. The bottom line is the workers are kept busy.
The next step was the 2 core CPU. In that there are two pipelines("cores").So it becomes possible to execute two programs simultaneously. Still it is not exactly like having two CPUs in the package of one since there is contention for some resources. For example both cores compete for access to the memory and data bus, share the same cache memory and other stuff too.
The quad processors are a mix of these two technologies. There are two cores each with two hyperthreads. There are not 4 cores they are just called that way mainly i believe for marketing reasons.
Now how all this affects execution time of various programs is extremely difficult to measure in detail, especially when we refer to a real life computer running a multi-tasking OS like Linux. We can however draw some conclusions if we simplify the problem by seeing it from the side of the CPU. All the CPU sees is instructions coming from the data bus. It has no idea if this instruction is from program A or program B or one OS instruction. So it would not be far fetched to say that essentially the CPU operates like executing only one single program.
So suppose we have a program that executes on a single pipeline (core), single hyperthread CPU. It takes let us say 60 ms to complete. If we ran it on the same CPU but this time having two hyperthreads would the time be cut in half? Hardly. It depends on what the program instructions are but it is not expected to see a boost more than 10%. It all depends on the effort taken to optimize the program. If we are sloppy it will happen many times that an instruction is blocked and the extra registers will help. On the contrary if the program was written with care and did not stall the pipeline we would see no performance increase. (intel wants you to be a bad programmer to sell more core duo processors:)).
Now when it comes to two core CPUs there is another consideration. We can see performance increase not only by exploiting "badly" written programs, that stall the pipeline. We can execute instructions on the second core as long as they do not depend on instructions on the first core. The performance increase depends on how much of the program we can parallelize. If we can execute 50% of the instructions in parallel then the boost will be (50% of instructions * 2 cores) + 50% instructions single core = 150% meaning we have a 50% increase in performance. Now if could only parallelize 10% of the program what would we have? (10% * 2) + 90% = 110% meaning 10% increase in performance. (Formally that is called the [[COLOR="DarkGreen"]Amdahl's law[/COLOR]]("http://en.wikipedia.org/wiki/Amdahl_law") ).
Two cores together with hyperthreading yields the same improvement as hyperthreading does for single core CPUs.
In real life that  means that the performance increase depends greatly on how much in parallel a program can be executed. On the one end is the real hog that cannot execute in parallel at all so you have 0% increase and on the other hand you two separate programs that execute twice as fast on a two core CPU. Well even then you do not have a 100% increase since the OS needs to intervene to schedule and monitor the execution but you get really close to that. Considering that a well written OS, like Linux, tries to minimize its execution time you can see it as having to run a single program that can be parallelized by 99.9%.

### Post by 3rdalbum on 2008-09-07
A quad-core CPU running at 2GHz has four cores that are each running at 2GHz.

I would not describe it as "emulating an 8GHz single-core processor". A good analogy would be each core as a truck, carrying grain.

One truck can carry one tonne of grain in one hour. Four trucks can carry four tonnes of grain in one hour. Even with four trucks, it still takes one hour to carry one tonne of grain. Each extra truck increases the amount of grain you can carry, but it doesn't increase the speed that the grain is carried to the destination.

With four trucks, you can carry your daily quota of eight tonnes of grain, in just two hours rather than eight hours. But if the fourth truck had to stop for repairs or otherwise took longer on its last run, then you wouldn't have completed your quota until it arrived. Your other three trucks would be unable to help.

So, if you have a program that can split its task into four independent sections for execution on multiple cores, then each part will run faster. If each section takes the same amount of processing time, then you're right - you can get more done in a shorter amount of time. If the program can't split up its task into independent sections, then you'll just be hammering one core* without the other cores being able to help.

*Single-threaded tasks actually jump from one core to another, as a sort of "even wear" system.
