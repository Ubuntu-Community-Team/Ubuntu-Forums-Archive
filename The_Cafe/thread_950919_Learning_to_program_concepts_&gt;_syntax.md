### Post by Sinkingships7 on 2008-10-17
You already have the mindset to be a good programmer. You're thinking of different ways of doing things, and you realized (by yourself) that there is a big difference between programming and writing code. I'd say that's a big step in and of itself.

A rather primitive step to reaching what you're trying to accomplish is to find (or make) a small problem and solve it. It does not have to be a hard problem. It can even be something as simple as having the user input two numbers and add them together, printing back the result. The key is to solve the problem multiple times, using different paradigms, languages, and concepts. See just how many ways you can solve the problem, and hack it.

Be sure to visit the [Programming Talk]("http://ubuntuforums.org/forumdisplay.php?f=39") sub-section of this forum and hang around there.

After you think you have a fair grasp on that, I recommend to not language hop. Find a language and stick with it while you learn more. I suggest Python for its clean and easy-to-read syntax and its extensive support for multiple paradigms. It also enforces clean coding structure with its mandatory whitespace requirements.

### Post by MaxIBoy on 2008-10-17
Anything more advanced than a "do you want to do this or do you want to do that" program is going to require an algorithm. A good algorithm mimics the way you personally would go about solving the problem in your head. There are a number of ways to translate thought to code. Different ways work for different people.
[list]
[*]Get some paper (or a wacom tablet if you have one) and start drawing things out. For me, this only works for geometry/graphics related code. Other people can do the whole thing in flowcharts.
[*]Pseudocode vaguely resembles code but is much more similar to human speech. I do this for complex, multi-step stuff to help me see the "big picture." For example, I might just put in one line that says "calculate the damage done by player's bullet" instead of a set of five or six lines which would do the same thing. (If you're doing this a lot, it could be a hint that you need to move more stuff into seperate functions.)
[*]Pick whatever language feels most natural to write in (for me, that's Python,) and mentally solve the problem your algorithm must solve, writing code as you go.
[*]Write all the comments (with extensive details on the algorithms you'll use) before you write the code. This significantly increases the chances that your code will run the first time.)
[/list]
