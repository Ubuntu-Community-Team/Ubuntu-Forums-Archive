Under your icon theme 24x24 there should be a start-here or places directory but I think it looks for start-here in the apps folder first, you may have to copy the icon you want into that folder, not sure.

On a side note, if there isn't an icon-theme.cache it might not find it either. Here's how you create one.

```
sudo gtk-update-icon-cache -q -t -f -i /home/chuck/.icons/AwOken-934020

```  Of course you'll use the path to the directory that you are using. If you were using the gnome theme for example it would probably be ```
sudo gtk-update-icon-cache -q -t -f -i /usr/share/icons/gnome
```  The **-i** generates it without the image data & is quite a bit smaller, not sure if there is any advantage to creating it with that data or not, I didn't notice any difference & it reduced the file size by about 80%.

Since the default behavior when clicking a running icon is completely useless if there are more windows than one running under that app, this is a massive improvement. (Perhaps what turned Torvalds, since it was one of his complaints? = D)

This extension behavior really is closer to consistency than the default behavior (much as Windows Alt Tab and Workspace Navigator are.) The Favorites list is nothing more than the new way of doing desktop launcher icons. Window management happens in the content area of the Overview, not the left bar, which is for launching, or the right bar, which is workspace management. That consistency allows the dragging and dropping rightward (app icons to windows, windows to workspaces) to make sense. Retaining a fossilized element of window management embedded in the Favorites bar seems nutty to me; the icons are apps, running or no, not windows. (On the other hand, I look forward to the inevitable employment of Unity-style drag-and-drop of document icons onto app icons from the Dash search or even from the file manager, and this is still consistent with a strict treatment of the Favorites list as representing apps only.)

### Post by Copper Bezel on 2011-12-11
That's a good sign, certainly, depending on how that process is implemented. If the extensions we have activated are phoning home, and that information is being tabulated to show that a majority of extension users are making use of this or that extension, and those extensions most often used are simply included by default as an option in the future, I could see that as a useful feedback mechanism for Gnome's management of the workflow.

I would say that the Gnome Favorites bar is fundamentally a different thing in my mind from a dock. I never even considered the idea of having left-click launch a new window with Cairo, AWN, or DockBarX, but all of those docks put a focus on window management as an integrated feature of the dock, and DockBarX takes this to the logical extreme by putting the window controls on individual window list items and such. While it doesn't have any real window management tools built into the dock, Unity matches this way of thinking as well.

The thinking goes like this - an application icon represents a category of windows defined by their being run by that application. Clicking the icon is selecting that set. If the set is empty, a window is launched. If the set contains one item (one running window,) the item is produced. If the set contains several items, the user is queried as to which window is to be presented. 

Gnome Shell's Favorites doesn't provide that last step, so the whole metaphor is broken. It's really the same problem presented by the default Alt+Tab behavior, so maybe it's consistent, but if so, it's just consistently broken; one must right-click the icon or use a different keystroke to move between windows within a single group, which makes this behavior intuitively and functionally disconnected from window switching between applications. To the user, for whom the windows are presented as objects of equal hierarchical stature (toplevel) in the desktop itself, the distinction is arbitrary and completely counter-intuitive.

The workspace behavior is a much more versatile grouping mechanism (as it's user-defined and dynamic) but it's also the one favored by the Overview, because the content area of the Overview is the individual desktop. The user can't tick through running applications and have their windows represented as a physical group as he or she can with workspaces.

Contrast that again to DockBarX or Unity, where a window list on hovering the icon in DockBarX or a click on the icon in either *does* isolate the windows of a running application as a visually-presented group and allow them to be selected or closed.

It's complicated somewhat by the fact that web browsers and other applications (gedit, even!) often focus rather than create new windows when their commands are run. For Chrome, Epiphany, and gedit, I've changed this behavior by editing their .desktop files - without doing this, even the Dash Click extension wouldn't fix the problem. 

So I understand the extent to which these are more or less competing desktop metaphors, but I'd rather they not be competing directly on my own desktop, and I don't care for the alternative metaphor where just using two windows in concert from separate applications is an unnatural hassle. The apps-then-documents approach makes sense on something like an iPad (select the browser, then select the tab out of an Exposé) but that's an environment where there aren't any windows or desktops to contend with in the first place, because only one document can have the display at a time.
