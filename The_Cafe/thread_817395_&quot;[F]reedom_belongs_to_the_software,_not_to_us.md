### Post by Ozor Mox on 2008-06-03
> How does it free the non-programmer user from anything but cost?

Cost is actually the one thing the GPL does not aim to free the user from, since GPL software can be sold. The things it does free the user from can be stolen straight from gnu.org:

>     *  The freedom to run the program, for any purpose (freedom 0).
    * The freedom to study how the program works, and adapt it to your needs (freedom 1). Access to the source code is a precondition for this.
    * The freedom to redistribute copies so you can help your neighbor (freedom 2).
    * The freedom to improve the program, and release your improvements to the public, so that the whole community benefits (freedom 3). Access to the source code is a precondition for this.


Maybe not all of these things matter to all users, which is why so many people don't care what software they are running. But plenty of people obviously do care, and appreciate these freedoms.

Don't forget also that the GPL isn't the only free software license. The BSD license for example would give you near public domain freedom. The choice is of course down to the developer, as it should be.

### Post by Ozor Mox on 2008-06-03
50words, you are making the conclusion that because a user isn't a developer, that they cannot benefit from the software freedoms...

> * The freedom to run the program, for any purpose (freedom 0).

The user is free from vendor lock-in and DRM and can run the software on any platform architecture and OS it is built for, and will not be restricted in how they may use it.

> * The freedom to study how the program works, and adapt it to your needs (freedom 1). Access to the source code is a precondition for this.

Even if the user cannot program themselves, they are able to have someone they know, a paid programmer, or their business make a modification to the software for their own use, instead of being dependent on the developer to make the modification.

> * The freedom to redistribute copies so you can help your neighbor (freedom 2).

This is illegal using proprietary software, but possible and often encouraged with free software. Not a huge benefit to your average user, I agree.

> * The freedom to improve the program, and release your improvements to the public, so that the whole community benefits (freedom 3). Access to the source code is a precondition for this.

Again, the user themselves do not necessarily have to be doing this improving. Just because they cannot program, doesn't mean they won't benefit from the freedoms offered to those who can. They can reap the rewards in the form of a better application.

Of course, this is ignoring all of the freedom stuff, because I am considering this from the point of view of a user who just wants their program to work. As it happens, this is a *lot* of users! Still, I think there are plenty of benefits to be had for these users.

### Post by Samhain13 on 2008-06-03
I am inclined to agree with 50words that "freedom", for an end user, is more about the cost rather than the philosophy. But I cannot agree completely because "end user" can mean a lot of different things, and not all "software" are computer programmes.

Take a man who frequently surfs the Web. On one hand, we can say that he is a Web browser end user. Now, between the proprietary Internet Explorer (for which he paid in part when he purchased Windows) and the free Firefox, he has a choice. This man's choice is insignificant because regardless, he will be able to surf the Web anyway. Proprietary and free software make very little difference in this situation.

On the other hand, we can also say that this man is a World Wide Web end user. We know that the Web is built over a number of software technologies, many of which are "free". HTML is one technology that could have been made proprietary by its creator and restricted its use. If HTML had not been "free", we can imagine there to be a lot more formats being standard in the Web, with each format requiring its own client application to run on. And our example man would not have had a choice of Web browser because browsers, as we know them now, might not have even existed. That's a world of difference.

### Post by Jesse Weinstein on 2008-06-05
Things you can do with FOSS (Free/Open Source Software), that you can't do, no matter how much you pay, with proprietary software:

Benefit from independent review of the source code -- with proprietary software, independent review of the software is limited to treating it as a black box; and many things can be hidden in such a black box.   Independent review is important to users, programmers or not, because it ensures that the software does not contain malware, or do things that secretly benefit the vendor.  Even if a proprietary software vendor permits specific people to conduct "independent" tests, the reviews are not truly independent because the vendor can pick who can conduct them, and revoke permission to conduct them at any time.  Only with FOSS can you have truly independent review.

Sometimes, painlessly switch to a competing program -- If you are using a proprietary software program, and you hear about a competing program that you would prefer to use because it has better support, or it's developers intend to add features you want, switching to it will always involve some amount of pain and trouble, since, because the program you are using is proprietary, the competitor must develop their code independently,  and therefore introduce subtle differences and bugs.  If you are using a FOSS program, and you want to switch to another program with better support or a better roadmap, in some cases you can do that while still using the programs exactly the same way, because the source code for the two programs is identical -- only the support / principles / roadmap is different.  This is called a fork, and can't happen in the proprietary world, only in the FOSS community.

Don't have to "activate" it -- While not all proprietary programs contain "activation" requirements (i.e. requiring that you contact, by phone or network, the vendor to prove to them that you legally purchased the product), some very widely used ones do (like Windows XP, and Windows Vista).  No FOSS program contains such a requirement, and if such a requirement was introduced in a future version, you could painlessly switch to a version that did not contain such a requirement (see above), assuming (very likely) that some programmers chose to fork the program requiring "activation" to remove the requirement.

Use both old and new versions -- You might wish to do this in order to run software that only runs on the old version, and software that only runs on the new version.  While not all proprietary program licenses prohibit you from using old versions of the program after you have purchased a newer version, the Microsoft Windows XP Professional EULA does include such a clause.  It's unclear if you can buy two licenses, upgrade one, and still use the older version under the other license; but it's unambiguous that, if you simply bought one copy, and expect to be able to switch back and forth between versions, that is forbidden.  No FOSS license forbids this.

Rely on the "spirit" of the license not changing, even for updates -- Proprietary program licenses typically permit new licenses to apply to updates or add-ons to the program; and they make no restrictions on what those licenses can require.  The GPL, the most common FOSS license, requires that the licenses for any updates to the program maintain the spirit of the original license; while this is vague, it does provide reassurance that updates won't require onerous or objectionable terms.  You have no such reassurance with updates to proprietary programs.

I hope the above list will provide some more reasons why a non-technical, non-programmer user of software might prefer FOSS over proprietary solutions, beyond issues of cost or individual program quality.

Jesse Weinstein
(who registered for this board specifically to make this post)

P.S. The essay "[Comparing the GPL to the EULA](http://www.cybersource.com.au/cyber/about/comparing_the_gpl_to_eula.pdf)"(PDF) is a worthwhile read; it is a comparison of Microsoft's Windows XP Professional EULA with the GPLv2.  I got some of my ideas above from it.

There's just one barrier to your scenario that I see cardinals_fan, and that's the real world. I'm sure some companies do support BSD-licensed projects in return for making use of their code, but I'm equally as sure that there are plenty that don't, and have no qualms about taking this code and using it without giving anything back. The GPL is designed to prevent companies and individuals from taking advantage of the hard work of developers in this way, and I'm sure that's why it is so much more popular. If I was giving up huge amounts of my free time to write a free software project and released it for free, I'd be mightily hacked off if it turned out that someone else was making money off of it, and I'm sure I'm not the only one. But people don't mind writing software under the GPL, because they know that the software will stay free. As for the legalese in the GPL, I think it's lawyers that need to be blamed for that, not the license itself. It has to be able to stand up in court if tested, or it's as good as useless. The BSD license is indeed nice and simple, because it places no restrictions at all on the user, so it doesn't need to be complex.

As it happens, I like both the BSD and GPL for their own merits, but I think it's very obvious why the GPL is so much more popular.
