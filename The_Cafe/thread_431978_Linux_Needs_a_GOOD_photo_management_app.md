Q: What's different between the Windows version of Picasa and Picasa for Linux?

There are a few differences:

    * Picasa for Linux is currently available only in the U.S., with an English interface.
    * Google's Hello photo-messaging application is Windows only, so it’s not currently integrated with Picasa for Linux.
    * CD burning isn’t supported in Picasa for Linux. The button will be grayed out in the interface. The CD-ROM-burning library used by Picasa uses a Windows driver that’s not easy to support on Wine.
    * Similarly, the backup feature is not supported.
    * There’s currently no Export to TiVo® feature.

What the hell is all this about Picasa? 

Digikam, F-Spot, all these programs are absolutely fine - but really, *why* do you  need them? I mean, are they actually necessary for what you want to achieve?

I find the command line much more useful for dealing with photos. I take a *lot *. I'm often sorting through 1000- 3000 a week. Batch processing with GIMP and ImageMagick allows you to get on with more important stuff, like listening to music or messing about on the internet... 

But really, it's worth learning to do as much as possible through the command line, because it uses ***much*** less system resources. Even on a new machine, things can really slow down when you're trying to sort out hundreds or thousands of photos. For me, splitting the jobs up so that they are performed by different programs makes life much easier.

I have found Mirage to be the best image viewer and basic editor - it is, in my experience, much faster than Image Viewer or any similar one. I use Mirage just to go through each photo individually, rotate and delete, all with the keyboard, because it's quicker.

I use GIMP to edit any photos that need special attention, and batch process through command line to add effects to a whole folder. ImageMagick allows me to use the mogrify command to batch resize, and I'm pretty sure it could do everything else you want.

In my opinion, the best way to get what you want is to spend a week (or longer) working out what you can do automatically through the command line. Then, grouping photos into folders or sub-folders will allow you to batch-process the tags, indexing, resizing, emailing and uploading. The only trick is learning *how* to do exactly what you want. You could then map all the commands to shortcut keys and the right click menu, and just run them one by one when you want to. Sure, it'll take time to set up, but it will save you time in the long run...

If I could tell you just what to do, I would, but I'm still learning myself - although I *know* it can be achieved. The only things I'm not sure that can be done quickly and easily are EXIF data viewing/editing, spectrum analysis (I'm guessing a specialist program there), and timelines.

You *might* be able to use [Treeline]("http://www.bellz.org/treeline/") to set up a directory hierarchy like you're looking for, but I'm guessing you'll need to set up another shortcut key or right-click menu to get it to work efficiently for you for folders of photos, and you'd need to plan out exactly how you wanted it to work. 

This all sounds complicated, but my set-up is halfway there (my needs are only half of yours), and it works a lot quicker for me than F-Spot or Picasa would - *and* I get more done! Hope this is useful for you.

### Post by spillz on 2009-03-25
I share the dissatisfaction with each of the current photo management alternatives for one or more of the following reasons:

* closed source (picasa, adobe products etc)
* mono (f-spot)
* not gtk based (digiKam)
* sluggish or cumbersome with big collections (i.e. more than a couple of thousand photos)
* databases aren't always sync'd with filesystem (f-spot definitely has this problem)
* limited search/filter/sort abilities
* limited to directory based browsing (that's a clusterf**k once you have 10s of thousands of images)

To address this apparent hole in the offerings, I recently started developing my own photo collection manager using python, pygtk and various imaging tools. Here's an outline of the approach I'm taking (borrowing good features from all of the competitors):

* A program geared around collection management, not image editing. Thus, the focus is on tools to tag, organize and rearrange an image collection. Advanced editing will be via launchers to relevant programs.

* Emphasis on a snappy interface. Do heavy lifting on background threads or external processes to ensure the UI never blocks. Keep user informed of what is going on in the background with progress notifications etc.

* Keep metadata in the images where it belongs. To the extent possible, databases/indexes are temporary files that can be deleted without losing useful data.

* A collection is all of the images (jpeg, png, raw etc) in a directory and its subfolders. The program maintains a database of each image and a subset of its metadata for easy look-up. Each time the application starts, the program scans the entire directory for changes to the images or metadata and syncs with the database. This happens on a background thread, so the user can still do stuff while this is happening.

* The user sees a contact-sheet like display of the images in the view and can scroll through the entire collection. The collection can be sorted and filtered as the user sees fit. The view is propagated in real-time on a background thread, so the user should be able to interact with the program even while an intensive sort or filter is taking place.

* the filesystem monitor keeps track of deleted, changed and new images (courtesy of inotify) and updates the colection and view in realtime

* The program stores searchable user keywords in the image metadata

* The program offers the usual array of simple editing tools (crop, rotate etc) plus launchers for gimp, ufraw etc

* contact sheet thumbnails are stored in the ~/.thumbnails folder (as per the thumbnail spec, [http://jens.triq.net/thumbnail-spec/](http://jens.triq.net/thumbnail-spec/), but as that spec is a little broken I'm working on alternatives)

* The program supports batch operations on multiple selections, uploads and photo albums (i.e. sub-collections of images)

* The program will use a dating mechanism for matching duplicates (that is, any image with the exact same date will be considered a duplicate)

All of these features are in varying states of readiness (many not at all). I need another month before I open the program up some early testing by the community, so consider this a tease. :)

### Post by spillz on 2009-04-03
Does anyone know if there is a linux/freedesktop/some other standard on photo metadata and metadata precedence? Microsoft publishes this: [http://msdn.microsoft.com/en-us/library/bb643802.aspx](http://msdn.microsoft.com/en-us/library/bb643802.aspx). 

Even if there isn't a standard it would be helpful if someone could link to anything documenting the tags the more popular linux programs use.

From my googling so far, photo metadata handling is a messy grey area. By that I mean, there are several metadata standards (XMP, IPTC, EXIF) and sometimes there are duplicate ways of storing the same info. e.g. keywords can be stored in XMP.dc.subject or Iptc.Application2.Keywords (and some software even puts keywords in exif.image.usercomment) and different programs appear to make different choices. Even within one standard, different programs might prefer different keys. For example, a program might plausibly use any one of the following to maintain the date of a photo: Exif.Photo.DateTime, Exif.Photo.DateTimeOriginal and Exif.Photo.DateTimeDigitized.

### Post by Miguel on 2009-06-03
Hi guys,

I got a dSLR a couple of months ago (Nikon D60, in case you wonder), and I've found myself between a rock and a hard place. 

The first issue is space. My machine is a laptop (the T400 of my signature) with 250 Gb partitioned in order to also use Windows, so 7 Mb or RAW per image is quite a lot. Why RAW? Lossless 12 bit is one reason, but the main one is the ability to switch white balance and camera profiles **after** the shot. I do fiddle with those settings before shooting, of course, but as I'm learning I like to experiment*. Keeping both OSes mean wasting quite a lot of space that could be devoted to pictures. I'm definitely not killing the OS I use 95% of the time (ubuntu), and so far I'm using the windows partition to store pictures. God bless ntfs-3g.

Then there is managment/viewing. F-Spot is slow, or damn slow if used with the Jaunty intel driver. Furthermore, it sucks with RAW files, even with the plugins that allow RAW/JPEG grouping and editing via ufraw. What's more, in Nikon's case, F-Spot restricts itself to opening the basic jpeg embedded in the raw photo, so no 12bit lossless goodness. I don't really like how pictures are sorted, and would prefer a hybrid folders/tags system. digiKam for KDE4 seemed a little bit better, with better quick edit features, but I didn't try it for long. I'm quite tidy with my folders, so I could survive mixing eog and nautilus. 

And then there's the raw problem. First of all, half the raw formats are proprietary (yes, like doc or xls). Second, it just happens that (probably because nef is proprietary) the best raw converter out there in quality terms for Nikon is Nikon's own Capture NX. It's Win/Mac. It costs a "measly" 180$ (surely 180€). It's a memory hog, easily needing 1Gb of my precioussss DDR3 memory. It's slow. It has an awful UI. But the bloody program then dares to produce an outstanding image quality. 

Ufraw + GIMP would be good, but it happens that if I want the extra "oomph" or some saturated lego colours I need some colour profiles... that Nikon software generates on the fly for each raw image**. I've realised that there's no way I can get the same quality out of ufraw than from Capture Nx 2. No way. I don't really care about Capture Nx's noise handling, since GREYCStoration is awesome, and I suppose that wasting some time with layer masks plus unsharp mask would finally give me great sharpening results. If I want some smooth softish tones for a portrait, I get them in a second with Nikon's crappy proprietary software. If I want some jaw dropping colours, that's another click away. How can I do that in Linux?

I'm giving Bibble Pro a go, and after those 15 days I'll probably switch to Light Zone and see what I like more. Although both are prorpietary, I'm way more tempted to support some developers that publish on linux than Nikon for producing proprietary formats and bloated slow code.

Sorry for the long rant, but this really angers me.

PS: I might as well end up using View NX to batch convert all RAWs into tiffs and then sharpen/lighten/denoise as needed in GIMP. But I just don't want to boot into windows each time I take pictures.

* If your camera has "Shade WB", give it a go for sunsets. It's a must-have setting that not all cameras have. May be a tad too warm for sunset portraits, though.

** I've got one ICC profile for each picture profile on my D60, but it seems to change with the ISO and maybe also from shot to shot. For some reason, it's still not the same.

### Post by spillz on 2009-11-03
> **Nonno Bassotto said:**
> 
In any case, the real problem I find now is switching between photo management apps. Every such application adds tags to your images, and there are a lot of different systems to store this information: a database, a text file or inside the photo itself. This third alternative comes as IPTC, EXIF, EXIF2 or XMP comments. 

So choose your preferred tool wisely, as it could be a high time cost to switch later.

Some quick notes for anyone still reading. There are 3 standards to worry about:
Exif = the standard used by camera manufacturers for recording technical information about an image (e.g. shutter speed, geolocation etc). Can also contains some descriptive info. Stored in the image.
IPTC = the standard used by new organizations for describing images (Title, Description, Keywords etc). Stored in the image.
XMP = a more flexible system of storing technical and/or descriptive information about images. Can embed all of the information in Exif and IPTC standards plus much more (e.g. image editing instructions). Can be stored in the image or in a "sidecar" file.
(wikipedia has entries on all of these if you want to know more)

Using a combination of Exif/IPTC/XMP is pretty standard in many programs nowadays, but as noted some programs use a separate database. Reading and, especially, writing metadata to images is an art. This is particularly true for Exif where Manufacturers regularly depart from the standard fields (leaving so called "Maker notes" in the images, such as Nikon's color profiles). Historically, writing Exif and IPTC metadata to images was risky business and there were few good toolkits available (which explains the choice of older programs to use databases).

There is an emerging consensus to move toward XMP, a standard being pushed by Adobe, but an open standard nonetheless. DigiKam and F-spot can both suppport XMP to some extent. XMP is a superset of Exif and IPTC in the sense that you can copy across all of the data in the Exif and IPTC fields and put it into the XMP. XMP has the best internationalization support. Getting the Exif/IPTC data into the XMP isn't automatic, however, so you need a toolkit to do it and it raises a lot of issues about what to do when the IPTC/Exif fields get out of synch with the XMP. Not all formats can handle XMP, so sometimes you will need to keep the XMP data in a separate file, called a sidecar. This can cause problems if you forget to keep your sidecars with your images.

So here's what I recommend:
1. Don't use a program that stores important image specific information in a separate database even if that database is open unless it can also export that data to the images or XMP sidecars. (You don't want to be left with this task when the developer no longer has time to maintain the program).
2. Avoid using a program that regularly writes data to your images without informing you. You want your program to treat your originals as respectfully as a photo negative.
3. At a minimum use a program that can store descriptive information in Exif/IPTC. You should prefer IPTC here because it is a more flexible system for describing images. Not all images formats or programs allow you to store Exif/IPTC data, however, and you always have some risk of corruption, especially when writing Exif.
4. Ideally, use a program with XMP support. You may prefer to use sidecars because there is less risk of image corruption and it is guaranteed to work for every image, but the downside is needing to be careful to keep the files together. 

If you can't do 3, at least do 2. Exif/IPTC is going to be around for a long long time and tools to handle the migration to XMP are only going to improve.

### Post by Neobuntu on 2010-10-12
When I said to use ufraw; before Gimp, ufraw processes the Raw photo; in 16-bit, and outputs to 8-bit, for the Gimp. You would set the relevant 16-bit level (complex exposure correction, and much more); from ALL that your sensor "saw"; with ufraw. Once that's done, simply continue in Gimp, or Digikam (if you need to).

Your larger, temporary, transition file, can best be a ".png", among many others. You could use ufraw, if you prefer; and then go to Digikam. Then, proceed; just as you would edit JPEG's. Your transitional .png, is later saved; as finished, sharpened (per resize), JPEG output (to whatever task, or to tiff). Of course, you could just use DigiKam; for the whole, Raw to finished JPEG process. The point is, you have two Raw developers, to try, and they both develop the 16-bit Raw levels, as needed.

We could use a slightly more competitive NR, as a native program. Yet, what we have for NR, is good. Many times, the "Restoration" process in DigiKam is better, depending on the picture.

This all DEPENDS on the scene, and the camera. (After the Photographer, that is.)

Raw development takes a a lot of skill, at least to get the best results. You should load your "Black frame" photo (use the lens cap to make it) and also hunt a custom curve, for your particular camera. Even if you don't have a popular camera, you can still make a custom curve; but again, that takes skill. Results will vary.

Since many detailed aspects are adjustable with Raw, and as a beginner, you might take your cameras default JPEG output, along with the SAME Raw file, and develop the Raw to exactly match the JPEG, to learn. Then you could perhaps, make it match another cameras output. Such as emulate a Nikon color pallet, on a Panasonic photo. Raw is much more complicated, when beginning; but not as much, when you get your patterns set.

If you don't want any work. Just shoot JPEG. Some cameras are better than others; with their JPEG.
