### Post by 23meg on 2009-03-08
I file, follow up with and [triage](https://wiki.ubuntu.com/Bugs/HowToTriage/) lots of bugs, in Ubuntu and some upstreams. 

> **Kareeser said:**
> The problem is that multiple applications have their own bug tracking system. Some use Launchpad directly, Wine has bugzilla hosted on their site, others use their own, and you have to register for each of them, making it unweildy.

If you're using the Ubuntu-provided versions of applications, you can simply use the Ubuntu bug tracker to file the bug report in the Ubuntu package for the app. It will be triaged in Ubuntu and forwarded upstream if needed. 

This is arguably less efficient than directly reporting upstream, since there's a certain delay and bureaucracy associated with distro-level bug triage, but unless you are using the software exactly as upstream has provided it, you can't be sure whether the malfunction is rooted in the upstream code, or the Ubuntu modifications (if any) or packaging, or happens due to how the software interacts with other Ubuntu components, and you wouldn't want to "bug" the upstream developers with invalid reports.

> **oldsoundguy said:**
> I quit filing bug reports via launchpad as they pay no attention to anything other than the eye candy reports.  Anything serious gets ignored.

That comes across as a pretty harsh generalization. Can you point to any specific cases that have led you to this impression?

> **argie said:**
> I file bugs, but never upstream. It's too much hassle keeping up with all the bug trackers. I'm secretly hoping Launchpad has a feature where it informs the respective developers about the bug, because otherwise I've done nothing.

The fact that bug reports don't go straight from distributions to upstreams has to do with bug triage procedures and the highly varying bug tracking preferences of the thousands of associated upstreams, rather than bug tracker technicalities. 

You've not "done nothing". Your bug reports in Ubuntu are to be triaged and forwarded to the relevant upstream bug trackers if the bugs are actually in the upstream code. Short of upstream developers actively engaging in evaluating distribution bug reports, there's no magical way to go automatically from a bug reported at a distribution's bug tracker to a fully triaged and actionable upstream report. The process takes human judgment and labor.

> **scottuss said:**
> I would guess that generally a lot of devs who can get things done are subscribed to the Ubuntu project on Launchpad. Although I wouldn't know how to  find out which of those devs are upstream... you raise an interesting point.

The [upstream report for Ubuntu bugs](https://launchpad.net/ubuntu/+upstreamreport) can give you an idea of the current situation regarding upstream bug coordination.
 
> **aysiu said:**
> **And I shouldn't have to know**. Launchpad should be designed so that you can report a behavior and then they can attach it later to the appropriate package. I am not a developer or programmer.

Launchpad *is* designed like that. You can report a bug in Ubuntu without specifying a package. This will, however, make it less likely that the relevant people will actually get to your bug report and act on it within a reasonable time frame. But then [the Bug Squad](https://wiki.ubuntu.com/BugSquad) constantly works on the bug reports without any package specified and assigns them to the appropriate packages. You can help them to do this by following up with the bugs you've reported and actively engaging in having them triaged.

It's a good idea to use Apport, or the Launchpad integration built into most default Ubuntu applications ("Help -> Report A Problem") whenever possible. That way, the package will be picked for you automatically.

> **aysiu]Most of the time when I want to post a bug, it's already been posted. Instead of being able to add a "me too" I just get asked if I want to subscribe to the bug. They should have a "me too" option, so they know relatively how many people are affected by each bug. I've also seen quite a few of these bugs never get fixed even though they affect basic functionality[/QUOTE]

As pointed out in an earlier post, there is a "me too" feature now. Before that feature, you could simply subscribe to the report said:**
> debugging information[/url] for *your* particular set of hardware. If the bug has been badly described, you can improve the description. If the bug has been filed without a package, and *you* know what package it's affecting, you can assign it to a package. There are many simple ways in which you can improve an existing bug report and get it one step closer to the "Triaged" status, where developers can act on it.

[QUOTE=blueshiftoverwatch;6861146]How much good does posting about problems on the forums do? Do the devs ever look at these forums?

No good at all. Developers frequent the forums sparsely, but more importantly, forums are the wrong tool for tracking bugs.

Actually, bugs do tend to "magically correct themselves" (get fixed without an Ubuntu developer having to act on them), with syncs from Debian or new upstream versions, for example. But Colin Watson's point stands, that triagers should check themselves whether they have before asking.
