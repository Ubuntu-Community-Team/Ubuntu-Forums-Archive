### Post by peabody on 2008-02-22
If it was easy, it would have been done a long time ago.  There are deeply rooted technological reasons why rpm and deb don't mix.  It was never the Linux community's ideal to have competing standards, it just happened that way as both debian and red hat attempted to solve the same problem at the same time.

Red hat initially won the popularity contest, but debian stuck to their format as they had actually been at it longer than red hat and it had been a significant time investment, plus it mapped well to the way the debian community did things while rpm didn't so much.  And debian was a completely non-profit distro so they didn't care about changing to the popular format of the times for any sort of economic insentive.

Now the reverse is almost true...with Ubuntu's popularity deb is starting to become one of the more popular formats.

The reason it is so hard to mix packaging systems boils down to two reasons:

1) binary compatibility with newer shared libraries
2) dependencies

Packaging's a messy business.  Individual software packages depend on shared libraries.  This is so every software package doesn't have to include all of the libraries it uses.  This saves a ton of disk space and ram (believe you me) but it means that any system that installs packages has to track down these library dependencies and install those packages as well.

For point #1, the bad news is, in Linux, binary compatibility isn't always guaranteed, especially when it comes to glibc versions.  If the package you've got was compiled under a more modern glibc than what's installed on your system, you're out of luck.  This has historically never been regarded as a big deal because in the open source world, you're just a tarball and a compile away from getting the software to work on your platform.  In today's modern world of user friendly linux however, that's no longer considered adequate.

For point #2, packaging systems handle dependencies by the names of packages.  These are tied to the names chosen by the distro authors.  Debian got this right early.  Anything that had to be packaged for debian went through a strict policy.  It took a while sometimes, but when something finally made it into debian you were pretty sure lots of people agreed on how it was done.

The world of RPM screwed this royally.

Mandrake/Mandriva, SuSe and the rest of those rpm distros pretty much started life as a rebranded Red Hat.  But gradually they diverged in a number of ways too numerous to mention (major version numbers of libraries, names of packages etc.).  This lead to 'rpm dependency hell'.  For those who remember the days of trying to find rpms on rpmfinder.net and other such sites, pretty soon it became almost impossible to find something that worked on your specific distro version, let alone track down all of those dependencies yourself.

This is one of the reasons people still swear by distros such as slackware.  Forget packaging, just get out of my way and let me install my tarballs!  And you might as well, because that was practically your only option for some software on package-based distros anyway.  I still remember compiling gcc 2.95 and installing it locally on my redhat 7.1 machine just so I could compile mplayer so it would work (ah yes, the infamous gcc 2.96 release).  When you're compiling a compiler just so you can compile the software you need because the compiler on your distro is broken...well...something is totally and completely futsed!

It was from this apocalypse that I fled to the world of debian with its apt-get and debian packages that worked so well.  When Ubuntu showed up, I was overjoyed!  Here was somebody making the debian 'unstable' flavor, well, stable and they were doing everything they could to prevent a fork to boot!

The state of things are highly undesirable.  There are historical reasons for the way things are.

donaldYou can install Debian packages on Ubuntu and Ubuntu packages on Debian. Both are debs, so dpkg can handle it. However, lots of packages won't work. Why not? Many reasons. The most significant is that the core libraries on each system are not necessarily the same version. If you try to install a Debian version of Foo which relies on library libbar, it may not work if Debian and Ubuntu have different versions, which is a significant number of cases. In the foo example, you could try to run foo and get an error message like : libbar error: undefined symbol SOME_NEW_FUNCTION_IN_FOO; aborted. To further complicate things, not all packages include the same thing. An example is bootchart. On Ubuntu, there is merely the bootchart package. On Debian there is bootchart and bootchart-view. Ubuntu's bootchart contains all that is included in Debian's bootchart and bootchart-view. If you were using Debian's bootchart on Ubuntu, and you installed something that relied on what is in bootchart-view, you would have a broken system. Apt is only going to look for bootchart, when the bootchart-view package would be needed to. That also ignores that often they call packages different things. I believe that Fedora calls the standard C/C++ libraries something different than Ubuntu does, and further all development packages have a different name.

Further, think about why we have all these different packaging formats. They just didn't make them because they are cruel and take pleasure from making you suffer. There are good technical reasons for it. Each format is better a certain things without any being a clear winner. Deb packages tend to be much faster than rpms at calculating dependency trees. However, rpms have a delta rpm that can patch an older version in order to reduce network usage. The others have their own strengths and disadvantages.

Basically, don't expect packages to work universally until one distribution gathers up enough strength to conquer all the rest.

EDIT: There is also the Smart Package Manager that can install any package type on any system, but it tells you not to do so on their homepage.
