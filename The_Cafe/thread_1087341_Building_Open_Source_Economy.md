### Post by gerymate on 2009-03-04
I am thinking about micropayment/bounty/donation systems for free and open source projects. I think that it would be great to enrich the possibilities for users to affect development, and for developers to earn some money with their work put into free software systems like Ubuntu.

I have started to draft such an economy system for Launchpad here:
[https://blueprints.launchpad.net/launchpad/+spec/free-software-ecosystem](https://blueprints.launchpad.net/launchpad/+spec/free-software-ecosystem)

What do you think about such economy systems? Do we need it? Would it be healthy for free software? Would you use it? How would you implement it?

### Post by Hilko on 2009-03-05
I would develop open source software if i knew how to. The problem is not the programming itself, there's plenty of tutorials for that. I know c++, but the problem is how to get started. The wiki pages on development are not very helpful.
So the best way to get more developers, is to improve these pages with very clear instructions for a newbies first steps in development.

I'm sure this would increase the number of developers a lot more than just putting a bounty somewhere up high, out of reach for newbies.

Money helps ofcourse, but maybe not as much as you'd hope for and it will certainly not help new people get started.
One should be very very carefull in giving away this money. If it is earned too easily it will result in a lot of bad sloppy programming, with a lot of bugs to fix, wich will require even more money. question: where does the money come from ?

I'm sure the best developers are those who are not motivated by money, but just by interest/hobby. We could use more of those, so i think the money is best spent on marketing.

### Post by k2t0f12d on 2009-03-05
What ever happened to simply making an agreement with someone to do some work and then paying them for doing it?  It cracks me up when ppl get on about how free software demands this contrary and new economic strategy for making money.  It doesn't.  It takes the _older_ strategy where people are paid to do work instead of for giving others permission to possess/use copies of its fruit.

I don't think there is anything wrong with a bounty system per se, but I just wonder if there is any point in worrying about implementing it in ways intent on controlling the behavior of its participants.  A bounty system itself is based the premise of paying out to the first person who can deliver the goods, period.  If you want a system based on any capacity for honesty, just use a good old-fashioned employer-employee relationship.

Just be aware, so long as society is built upon a monetary system, people are going to do dishonest things to get money.  Don't be surprised when you can't think of anything non-Orwellian to prevent people from trying to cheat the system.

### Post by gerymate on 2009-03-05
The basic idea is relatively simple: connect bounties with bugreports. If a bug is solved, the bounty can be divided between contributors, according to their contributions. 

Measuring and feedback seems relatively easy on that.

It seems important to make it clear early: it is *not* en employment system, rather a way of making (and receiving) targeted donations. So there are no contracts and no binding (except for GPL and the like :-)).
