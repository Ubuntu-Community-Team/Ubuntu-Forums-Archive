### Post by aysiu on 2007-05-20
Closed source is not *evil*, but open source is certainly *good*.

I am not a programmer, but I definitely see the benefits of open source as an end-user:
1. No spyware/nagware/30-day trial crap
2. Software is almost always cost-free
3. Projects die only if there is no interest in them--not because some company has said "nobody can use our code."
4. The programs are more likely to be cross-platform. Not sure if this is really true, but it feels true. At least GIMP, OpenOffice, Audacity, Firefox, and Thunderbird are made for Mac OS X, Windows XP, and Linux.

[quote=WalmartSniperLX]And I know not everyone creates for money  I plan to be one of them :P But when it's a living, you must do what you must, right?[/quote] 
"Open source" does not mean "no profit."

According to [an article published only a week and a half ago](http://blogs.zdnet.com/open-source/?p=1043), Red Hat had a net profit of $60,000,000 recently (no specifics on which quarter).

[Christopher Blizzard, remarking on rumors that Firefox made $72 million in 2005](http://www.0xdeadbeef.com/weblog/?p=182) says: > There have been a couple of articles written about the amount of money that the Mozilla Foundation has raised through various means. This entry says &#8220;72 million&#8221; and has been repeated over on digg.com. I won&#8217;t comment on the dollar amount except to say that **it&#8217;s not correct, though not off by an order of magnitude.** I also won&#8217;t comment on sources of that money, except to say that some of the assertions that I&#8217;ve seen in the comments are pretty far off, both in terms of numbers and sources. Open source can make you a lot of money. The software itself may be free (or, in the case of Red Hat, extremely cheap), but there are other ways to make money off it--with paid support and advertising being the two main venues I can think of.

### Post by az on 2007-05-20
> **Lucifiel said:**
> 
Hmmm... now, just curious, then how do you uhm maintain a standard of quality within a software project if it's open-sourced? 

The software and the tools used to build it help the maintainer tend to it.  Revision control is the term for having multiple versions of an application in development.  Contributers can submit their code to one of the versions (or branches) of the project.  

So there can be a stable branch and a developmental branch.  As issues get ironed out in the development branch, it can either become the stable branch or large chunks of the code can be moved to the stable branch.

An argument as to why Free-libre open source is of better quality is that more eyes are looking at the code.  As well, not just anyone can contribute to the project.  Usually, the contributer has to get known by the project's community before gettig upload priviledges.  GPG allows for authentification, so that you know that when a patch is submitted and signed by a person, they are really that person.


> **Lucifiel said:**
> 
Or are there other reasons why close-sourced = bad? 

Closed source is closed source.  Proprietary software is, for the intentions of this conversation, typically closed source.

(I think the correct terminology for this discussion should be between FLOSS and proprietary and not open versus closed, since some proprietary software can be open source.)

Proprietary software takes away your rights.  Software freedom is what protects your rights.  Since software is code which is just a bunch of algorithms, some argue that it is wrong to:
1- prevent you from obtaining the software
2 - prevent you from running  the software for any purpose.
3- prevent you from seeing how it works
4- prevent you from being able to modify the program and redistributing those changes.

A computer is useless without software.  If someone else owns the software running in your computer, you are giving that person the rights to your data.  Computers are ubiquitous today and it is trivial to lose your privacy when using a computer.  That's another reason why software freedom is important.

To boot, proprietary software uses things like patents to try to take away things from the free software community.  If that system was fair, I would not have a problem with it, but the reality is, it's far from fair.


> **Lucifiel said:**
> 
Finally, how about a gaming project? Would it still be as terrible if it's close-sourced?

Don't ask me what is terrible and what is "just and fair"!  It's your decision to make!

Personaly, I would not expect software freedom from a games console, but I would from my PC.  Also, many game engines are FLOSS and benefit from the development model.  The actual content of the games (graphics, artwork, sounds) are not free and that's what you pay for.

> **WalmartSniperLX said:**
> 

Closed Source (CS) is often considered an evil because :

2) Usually the purpose of CS is for profit. 



That's completely wrong.  A recent european study measured the economic impact of FLOSS and reports that it is a multi-billion dollar industry. ([http://www.merit.unu.edu/publications/pressreleases/200701_flossimpact.pdf](http://www.merit.unu.edu/publications/pressreleases/200701_flossimpact.pdf)) IT accounts for about ten percent of the GDP of developed countries and FLOSS represents about 30 percent of that.

FLOSS is big business and there is nothing wrong with that.  And the free software ecosystem is a lot better off with these contributers.  Software freedom is compatible with commerce.  FLOSS provides opportunities for business that are inexistent in proprietary software.

From [the Copyright FAQ](http://www.copyright.gov/help/faq/faq-general.html#mywork): > **When is my work protected?**
Your work is under copyright protection the moment it is created and fixed in a tangible form that it is perceptible either directly or with the aid of a machine or device.

**Do I have to register with your office to be protected?**
No. In general, registration is voluntary. Copyright exists from the moment the work is created. You will have to register, however, if you wish to bring a lawsuit for infringement of a U.S. work. See Circular 1, Copyright Basics, section &#8220;Copyright Registration.&#8221;

**Why should I register my work if copyright protection is automatic?**
Registration is recommended for a number of reasons. Many choose to register their works because they wish to have the facts of their copyright on the public record and have a certificate of registration. Registered works may be eligible for statutory damages and attorney's fees in successful litigation. Finally, if registration occurs within 5 years of publication, it is considered prima facie evidence in a court of law. See Circular 1, Copyright Basics, section &#8220;Copyright Registration&#8221; and Circular 38b, Highlights of Copyright Amendments Contained in the Uruguay Round Agreements Act (URAA), on non-U.S. works.

### Post by forrestcupp on 2007-05-26
> **Praill said:**
> Yes it can be. After all if I write cout >> "hello world"; a program converts that to an assembly language which later gets converted to binary by the OS (i might not be entirely correct here but this is my general understanding). Therefore, if a program can convert it.. why cant a different one convert it back? The answer is it can.. and this is the method used to reverse engineer.

What you're talking about is a decompiler.  Decompilers have varying degrees of success.  Usually if a program is written in a virtual machine language, like Java, the bytecode has a lot of metadata that makes it fairly easy to decompile.  But languages compiled into machine code like c or c++ don't include metadata, and the compiler or post-compiler obfuscates it which makes it pretty close to impossible to get readable code.  You can get assembly code out of machine code, but if it was programmed in c++, it's hard to really get the mind of the programmer and understand how things were set up.  You don't get all of the variable labels and other things that make it readable.  You definitely don't get the comment lines, which is something programmers lean heavily on.

If you want a look at how software is really reverse engineered from the ground up, you can look through this online book:
[http://www.acm.uiuc.edu/sigmil/RevEng/](http://www.acm.uiuc.edu/sigmil/RevEng/)

I'll just tell you, it's not by having a program convert the machine code back into c for you.  It takes using the scientific method.  Figuring things out, observing how the original program works and responds, and predicting through your own code.  It's a lot of trial and error, and anything of any size takes a lot of engineers working together, and a lot of time spent on trial and error.
