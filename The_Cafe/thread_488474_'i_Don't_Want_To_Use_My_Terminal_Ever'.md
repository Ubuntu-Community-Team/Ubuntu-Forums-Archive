### Post by igknighted on 2007-06-30
I understand the trepidation that the terminal can cause, but honestly there is nothing to be frightened of.  It is just one more tool that is available at your disposal.  If you think about it, there are hundreds of window managers on linux (think: different GUIs), and although only 3-5 of them are really popular, thats a lot of different GUI commands and questions to ask users before you can assist them (before you even get into what distro).  The terminal is always the same, so i can just say run this command, end of story.

The terminal will never lose its following in linux because the whole theory behind a GUI is just slower than CLI in many tasks.  Think about how long it takes to open synaptic, find the programs you want to install from a list, install them and all the while the gui is slowing the process down.  On the CLI "sudo apt-get install item1 item2 ..." does it all in a tiny fraction of the time.  Searching for the app (if you don't know it already) is simpler too (apt-cache search).  You even get the added benefit of having all relevant info displayed (errors and everything) so if there is trouble you know exactly why.  Not that everyone should use the CLI for all installs, but if you know a little about how to you can save yourself a lot of time.

CLI isn't flashy and never will be.  It is simply functional.  And until a GUI can be equally as functional (I tend to think that this is an impossibility, at least with current GUI design theory), it will always be a central part of linux.  But this is good, because while it is not a requirement to ever use it, it can make your life much easier if you do try.

### Post by tgalati4 on 2007-06-30
Mac OS X has gui configurations that work because they have a limited hardware family to support.  But you can still used the command line in Mac OS X for certain tweaks that are not available in the gui.  These tweaks are helpful for performance reasons.

The same in Ubuntu.  Use the gui, but if you need to tweak something, don't be afraid to lift the hood (or bonnet or boot).

The command line is the least common denominator, and the quickest way to dispense advice in this forum.  Say a user has a slow-running computer.  What would the gui procedure be?

In the CLI:

>sudo apt-get install htop
>htop

Kill the process that is hogging your system (CPU % that is greater than 80%).  Highlight the offending process by clicking on it with your mouse, hit F9, then RETURN.  F10 to quit, or click on QUIT with the mouse.

Now the user has htop installed on their system and they have a rudimentary knowledge of examining running processes and killing them.

However, when I've tried installing programs that don't come from the repo's...I have to get into the CLI to do so (make file, etc).  It should be easier and more GUI for folks to install non-repo software w/o having to mess with the CLI.

I would personally qualify myself as a "power-user", since I know how to use the CLI, in both Ubuntu and Windows, as well as programming in general.  But, if the goal is to make Ubuntu easy enough for your grandmother to use, then I think activities should be categorized into 2 areas...

1) a common activity that the user WANTS to do...make a GUI for it so they can easily do it

2) a common activity that the user DOESN'T want to do (like maintenance)..make automation handle it for them

My reasoning behind this is that 1) average users don't want to have to see the CLI to do things they want to do (like check email), and 2) they don't like having to call their local computer "expert" to come "fix this damn thing" when it breaks.  

These are very ambiguous areas, and everyone has their own opinion on what the "average" user wants to do, because everybody thinks they're the "average user", or (especially for some Linux users) they think the average user is more technical than they really are.  For Linux in general, this is true, because Linux has always required more technical skill.

_But, Ubuntu isn't trying to be another Linux distro...it's trying to displace Windows as the dominant OS on the market (Shuttleworth's bug #1).  To do this, they must offer "dumbed down" GUI ways the average Windows user can use, not "sometimes GUI, sometimes CLI" ways the average Linux user is used to_.   So, that means no CLI for common activities, like installing software that doesn't come from repo's; an "Install Shield" type GUI instead of having to use the CLI to install non-repo software needs to be addressed.

Not saying the CLI needs to go away or get dumbed-down.  Keep it, because it allows power-users to do what they love to do (bypass slow GUI's and automate things).  Like others have said, keep the best of both worlds...dumbed-down GUI's for average users, and scriptable CLI for power-users.   I just think there's a few things that still need some GUI attention.
