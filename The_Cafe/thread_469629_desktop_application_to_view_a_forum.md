### Post by bonzodog on 2007-06-10
Whilst I like this idea, it kind of reminds me of posting to usenet back in the day.

However, there are significant problems with it.

One of these is posting to the forums - The forums let you post through the front end over the CSS.

There is currently no other way of doing it. To do this would require that a seperate feed/access point be setup on the webserver, much like google has with POP access to GMail. 

So, to get this program to work would require asking each forum admin to set-up a point of access for your program to interface with the Forums DB. Also, there is a permissions problem. You see, what you lot see now in the front end is not everything there is. Staff have their own, hidden, forum to talk amongst each other. With the web login, this is easy to hide.

What you need to do is talk to the people that write the Software - VBulletin, SMF, phpBB, etc, and ask if an idea like this has ever been thought of, and is it possible to write a DB plugin that would allow it.

If a small committed group of coders could pull this off, it would be cool.

I agree with both sides on this one.

Yes, Firefox is perfectly suited for the quick and painless task of Forums Browsing and Posting (FBP), however an alternative client would be well appreciated.

Consider: Yahoo mail can do what you want from a browser, but it's nice to have the features and functionality of Thunderbird. And I can read all the RSS feeds I want from FeedBurner or from the live bookmarks in Firefox, but it's nice to have a feed reader as well.

Forums, especially this Ubuntu Forum, would be somewhat simple to route into a Forum App, and it would be a good way to manage your different visited forums (clear up space in Firefox bookmarks), manage your forum IDs and PWs (or just IDs if you wanna talk security), and to store forum posts on the HD for later reference (much like a podcast is saved to the HD and you can listen to it later, offline).

Honestly, a forum app would be VERY welcome in my mind.

Considering the fact that many projects and individuals collaborate here, this could be a useful business tool as well as a good tool for the average joe.

### Post by iceportal on 2007-06-12
As for java vs asp.net, I would go for Java, as it'd be good for cross-platform use. This should be a standalone desktop app, and as such, we should use a cross-platform compatible language like Java or Python.

However, I don't believe direct access to the forums database would be a viable solution if we're expecting to use this app for multiple forums. There are many reasons for this:

1.) We would need a database account for every single forum we intend to view using the reader, and I severely doubt any admin is going to give us access to their database directly. Not to mention the fact that if a user wanted to use the software with a board we didn't have a SQL account on, they'd have to request and obtain the user/pass, and understand how to log in.

2.) There are numerous flavors of SQL that run different forums. Some are running MySQL, others running PostgreSQL, others MSSQL, SQLLite, etc. While their syntax is relatively the same, there are slight differences in how they're set up.

3.) Direct database access is not necessary for any of the tasks on the forum, as the forum software (phpBB, vBulletin, etc.) already has code in place to act as an abstraction layer between the web browser and the database.

All in all, it would be much easier to just parse the HTML output of the server based on a ruleset created for that specific forum software. For example, if phpBB and vBulletin have two different HTML layouts, we would detect which one a forum is running, then parse the HTML in that manner.

As for posting new threads or replies to threads, it would be as simple as posting through the normal HTML forms the forum uses by default, just that we'd be doing it through our interface.

Do you kind of understand what I'm getting at?

If we want this software to work with multiple forums and multiple forum formats, we'd want it to work simply with the HTML input/output of the site. This would be the most convenient and simple way to interface with the system.

Basic pseudocode:

```

MAIN{
get forum URL;
get forum username/password from user;
retrieve forum HTML based on URL;
parse HTML to look for string:
    if string = 'vBulletin' parse using vBulletin();
    else if string = 'phpBB' parse using phpBB();
    else return unknown;
once parsing complete, display forum to user;
if user.newThread, create newThread;
if user.reply, create reply;
}

vBulletin(){
figure out what part is title, body, post.author;
return forum title,body,post.author;
}

phpBB(){
figure out what part is title, body, post.author;
return forum title, body, post.author;
}

newThread(){
get title, body from user;
send to server (title, body, username) as post;
return;
}

reply(){
get title from forum;
get body from user;
send to server (title, body, username) as post;
return;
}

```

Note: That's PSEUDO code... Don't try to compile it. (Just in case there are inexperienced users out there, who would get frustrated when GCC reports 5 thousand syntax errors.)
