### Post by Mr. Picklesworth on 2008-01-23
Grr, tabs are the devil! We already have tabs and tab-like things handled by the window manager in cleverer, more consistent ways:

-Fluxbox allows you to group / tab windows (and it does so really nicely, in a way I dearly hope Compiz's group / tab plugin will adopt)
-Workspaces are like tabs...
-An enhanced window switcher, like in Compiz, lets you flip through just a group of windows

To each his own, though. I suppose it's fair that some people prefer solutions that already exist :P

In my opinion, though, tabs handled within applications themselves are usually bad. No two applications *ever* do them the same way; it is hard to predict how they will happen in different programs. As a result, "the average user" will have trouble learning them, and is rather likely to not notice them since they have no consistent metaphor across the desktop.
On the other hand, managing windows in workspaces is not handled by the individual applications, but by the window manager itself! Thus, every application can be tidily organized, the desktop environment can make standard assumptions that a window is one document or one project, and it ensures that we have only one tabbing metaphor instead of a bunch mixed up together. Much easier for people to learn ;)

I think tabbing is fitting for small screens, where the numerous little floating windows concept does not fit. On a large screen, however, floating windows are the way! What we need is cleverer systems to arrange those floating windows for comparison, such as the neat 3d desktop features we see in Mandriva Spring.

Granted, tabs work in many places, including diff tools and IDEs, both cases where lots of stuff is required to be opened.

I don't think the Nautilus developers should be "embarassed" at all. First off, they have to match up with GNOME's release cycle. Judging by the message now present on that web page, such is not the case here...
Secondly: Does it integrate with gio (or GNOME VFS)?

### Post by tonywhelan on 2008-01-25
I don't wish to denigrate the developers of Nautilus or Gnome. But when the gvfs version of Gnome eventually gets into place, I hope the Nautilus folk will be able to add some things that are just plain necessary and currently are not available.

Undo/Redo - this will I gather be possible under gvfs and that will be a big leap forward.   Windows Explorer (slow tho it is) can at least undo a series of file/folder moves/deletes. 

Split windows - I do a lot of drag and drop but Nautilus doesn't have a button to split the display into two windows. Doing it manually is just a recipe for RSI, and you can't even write a script to do it because Nautilus doesn't obey its own environmental variables in a consistent way (yes I've reported that).  

Copy To Folder and Move To Folder - even Windows has these. These are real time-savers and the "Send To" that some developers suggest just doesn't cut the mustard. There is a clear difference between Move and Copy and they should be presented 'up-front' not buried under a Send-To menu. Either as a right-click menu option or a toolbar button, or both

Editable toolbar - You should be able to add/remove buttons on the Nautilus toolbar to customise it for what you want to use a lot. Again Win Explorer has that.

The Nautilus Action Configuration tool is modestly useful - I use it to add functionality but that sort of config tool should should be built into the Nautilus screen itself, as a tool or option.

We all have very different activities and very different ways of working. To accommodate those differences, a file manager should be fairly customisable. Nautilus needs to have more facilities "built-in" but with more flexibility to show/hide selected facilities on the toolbar.
