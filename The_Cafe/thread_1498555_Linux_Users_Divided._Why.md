### Post by chessnerd on 2010-06-01
> **Legendary_Bibo said:**
> forks? like utensils? I have a very limited terminology for linux :)

A "fork" is when a group of users/developers decide to create their own version of a Linux distribution that is based on an existing distro. It is called a fork because it is like a fork in a tree. There is a split from the main branch, but it is still from the same branch. In other words, it's a new distro, but it isn't original.

For example, Ubuntu is a fork of Debian. Debian was around for a long time (since 1994!) and, in 2004, when the Ubuntu team started to work on their OS, they chose to base Ubuntu on Debian. So, they took the Debian distro, and then added their own stuff to it. Soon, they had a distribution that was based on Debian, but wasn't Debian. Later, distros like Linux Mint did the same thing to Ubuntu.

### Post by Meep3D on 2010-06-01
> **Calmor said:**
> I wouldn't want a "standard" Linux.  I like the option to choose from among a variety of things - GUI, bootloader, filesystem, etc.  I like the choice.  


What if Grub would only boot RPM distros and LILO would only boot RPM ones?  What if KDE would only run KDE apps and vice versa for Gnome (You do get issues even now)?  What if Ubuntu would only load EXT4 and Fedora only EXT3?  Is that really *choice*?

This is no more about choice than Microsoft making custom extensions to HTML and JS being about choice.  The reality is all a 'package' really is is a zip file with a manifest, just as HTML is a bunch of tags.  If you really wanted Ubuntu specific stuff then there is utterly no reason you couldn't add this to the packages you were distributing while still maintaining inter-distro compatibility.

Would it be considered 'choice' if half the Internet ran on TCP/IP and the other half ran on IPX/SPX?  Would web developers having to have a server presence on both sides be choice?  Would being unable to access certain sites because they were on the 'wrong' side be choice?  As this is the situation that the pro package mess crowd would implicitly have to support as a 'good thing'.

To re-summarise:
Application choice is good, protocol choice is bad.

To re-re-summarize:
Consistency is more important than correctness.
