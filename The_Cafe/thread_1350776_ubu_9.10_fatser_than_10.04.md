### Post by MaxIBoy on 2009-12-09
Many of these scores are due to an EXT4 regression. You know how early on, people were experiencing data loss with EXT4? The only way to solve this problem is to flush some buffer all the time, instead of some of the time. The problem is that EXT4 write speeds slow down A LOT with this fix enabled. 

This performance issue will not be fixed because of the data loss risks, EXT4 is stuck with this performance (30% worse on write speeds in some cases) forever. (Or at least until someone comes up with a clever workaround.) Anyone who has been following Phoronix the past couple months will know this.



The GPU-intensive benchmarks will improve when Mesa 7.7 is finally added to 10.04, so your game framerates will be back to normal or better than before with the open source drivers.

### Post by MaxIBoy on 2009-12-09
OK, I dug up the original article:
[http://www.phoronix.com/scan.php?page=article&item=linux_perf_regressions&num=1](http://www.phoronix.com/scan.php?page=article&item=linux_perf_regressions&num=1)

Relevant quote from the kernel hacker (the article strongly implies that this person wrote the fix himself: )
> "Hey, thanks for doing the digging :)

It is required for safe  behavior with volatile write caches on  drives.

You could mount with -o nobarrier and it would go away, but a  sequence like write->fsync->lose power->reboot may well find  your file  without the data that you synced, if the drive had write caches enabled.

If  you know you have no write cache, or that it is safely battery  backed, then you can mount with -o nobarrier, and not incur this  penalty.

-Eric"Almost all hard drives made recently have a few (between 8 and 32) megabytes of RAM so you can dump a bunch of data on them without waiting for the slow speed of the hard drive itself. But of course the RAM looses its contents if the power is cut off.

I personally used to experience this kind of data loss myself, where I'd be working on something in openoffice, then I'd realize I was about to run out of battery and hit "save" really fast... and then the entire file would be GONE. I can confirm that this change has solved the problem.
