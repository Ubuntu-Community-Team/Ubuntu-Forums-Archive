### Post by t.rei on 2010-08-25
Easy way to try firefox 4:

1. download the firefox beta for your os (linux ;) ):
[http://www.mozilla.com/firefox/all-rc.html](http://www.mozilla.com/firefox/all-rc.html)

2. unpack the archieve:
```
tar xjvf firefox-4.0b4.tar.bz2
```3. move the directory to a systemfolder:
```
sudo mv firefox /usr/lib/firefox-4-testing
```4. run it from alt+f2 or commandline or create a launcher:
```
/usr/lib/firefox-4-testing/firefox
```---
to remove it:
```
 sudo rm -r /usr/lib/firefox-4-testing
```

FoxTester is developed for testing different versions of Firefox, not to replace the default version. It has no interaction with the package manager and PPA repositories. It handles only ready-to-run binaries downloaded as tar.bz2 from Mozilla and placed on a common folder. The extension monitors the content of this watched folder, so whenever you download a new version, it becomes immediately available for installation through the context menu. Nevertheless, the extension also has a feature to make any version being tested as default and thus use your default profile, by extracting it to the /opt directory and diverting /usr/bin/firefox. You can easily revert this any time you want from the context menu as well. So in fact, you can for example make 4.0b3 your default browser, then revert to the version from the repositories, then install 4.0b4 and make it default. This means you can use it to "update" from one Mozilla version to another with just a few clicks. All you need is to download the new version from Mozilla, saving it on the watched folder and then install from the context menu.

### Post by chrisccoulson on 2010-08-27
> **plun said:**
> The best solution would be to just **add a pre-Firefox 4 to Mavericks repo for manual install** if a user wants it.

Its then much easier to make a SRU later when Firefox 4 is stable, IMHO...

There's no need, this is exactly what PPA's are designed for. Firefox 4.0 is still too much of a moving target for us to consider shipping it in the archive. Just last week we had to make major changes to the packaging layout and structure because of an upstream change, and we still need the flexibility to be able to do this on a regular basis.

Having to do that in the archive in a stable release is not acceptable.

In addition to this, the Firefox 4.0 beta releases don't get proper security updates. So, when 3.6 gets a point release to fix security issues, we have 2 choices:

1) Track the daily builds to pick up the security fixes
2) Backport the individual fixes to the beta milestone ourselves.

Option 1 has it's own set of problems, in that the dailies can stop working from one day to the next. Option 2 is a lot of work, and requires having access to private bugs.

This is why we already removed a lot of extensions from the archive in Lucid, and we have quite strict criteria for extensions we ship in the archive now. Extensions must generally fall in to at least one of the following categories:

[LIST=1]
[*]Not available as an installable xpi from addons.mozilla.org or other popular addon sites (eg, bindwood)
[*]The officially distributed extension contains architecture-specific components, which aren't built for all the architectures supported by Ubuntu (eg, enigmail)
[*]Has high popularity on addons.mozilla.org (of the level of extensions such as adblock). These extensions are generally well maintained and quickly ported to new versions of Firefox
[/LIST]
