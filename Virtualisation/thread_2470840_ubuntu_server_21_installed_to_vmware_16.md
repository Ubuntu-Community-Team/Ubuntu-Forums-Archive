### Post by guiverc on 2022-01-13
Ubuntu's main products use the *year.month* format, eg. [Ubuntu Server 21.04 which reaches it's EOL this coming Thursday]("https://lists.ubuntu.com/archives/ubuntu-announce/2021-December/000275.html") or more common in production use is Ubuntu 20.04 LTS Server.

Ubuntu also has *specialist *products that use the *year* format, eg. Ubuntu Core 20 which is intended for headless operation in devices, appliances or in the cloud (ie. server use).

The *year* format highlights the different product; ie. 20 is not 20.04; as the *year* products are *snap* package only (no `apt` or `dpkg` commands as cannot use *deb* packages).

There is no *snap* only server product for the year 2021; they are released only on *even* years.  If you meant 21.04 I'd suggest starting again with (or *release-upgrading* to) a product that will still be supported a week from today, or if you meant 21.10; do note 21.10 is not the same as 21.

### Post by TheFu on 2022-01-13
Often when networking of a VM stops working, it is because the host system networking changed.  While there are ways to support that, it isn't really for beginners.  The simple answer is to use the exact same ethernet connection for the host system every time and to avoid using dhcp.

Of course, we don't know which VMware hypervisor you are using. Last time I counted, there were 6 different options, each is different with different capabilities.  Are you using ESXi and vSphere, Player, Workstation, Fusion or something else?

When talking about networking, it is important to clearly state what is setup on the host and the VM. You say dhcp ... but don't say which system is using it.  Is the VM network configured for NAT, bridged, host-only, or some other less common inside the hypervisor settings?

Ubuntu 21 is ambiguous too. There were 2 releases in 2021 - one is supported and the other is not.
