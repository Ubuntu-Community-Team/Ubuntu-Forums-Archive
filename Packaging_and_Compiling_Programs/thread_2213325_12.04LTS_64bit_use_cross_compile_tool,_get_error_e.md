### Post by steeldriver on 2014-04-01
Running the tools from a Windows share on the host may just not work - Windows doesn't preserve Linux file permissions. As you can see, ./arm-none-linux-gnueabi-gcc is looking for its own *local* version of cc1.

As a first step I suggest you unpack the arm-none-eabi tools again inside the VM (in your Linux home directory for example). Don't just *copy* the files from the Windows host share because the permission bits will already have been lost.

Another thing to check is whether the tools you were given are 32 bit or 64 bit (and is that the same as your VM).

Strangely, I found a arm-none-eabi.tar.bz2 file and unpacked arm-2008q3 directory on my own system - so I must have played with this toolchain before.

### Post by steeldriver on 2014-04-02
You can use the **file** command to check the executable type e.g. on my system

```

$ file arm/arm-2008q3/bin/arm-none-eabi-gcc
arm/arm-2008q3/bin/arm-none-eabi-gcc: **ELF 32-bit LSB executable**, Intel 80386, version 1 (SYSV), dynamically linked (uses shared libs), for GNU/Linux 2.2.5, stripped
$ 
$ file `arm/arm-2008q3/bin/arm-none-eabi-gcc -print-prog-name=cc1`
/home/steeldriver/arm/arm-2008q3/bin/../libexec/gcc/arm-none-eabi/4.3.2/cc1: **ELF 32-bit LSB executable**, Intel 80386, version 1 (SYSV), dynamically linked (uses shared libs), for GNU/Linux 2.2.5, stripped

```

I can't answer the rest of your question - does /mnt/**hgfs** imply the files are on a remote Mac filesystem? I already suggested re-unpacking the archive in your Linux home directory.
