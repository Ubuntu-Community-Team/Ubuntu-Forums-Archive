### Post by TBABill on 2011-03-31
You've gotten guidance on most of your issues so I'll just offer a bit of info on a few steps I take after install to make sure the machine stays out of the way.
 
1. Connect via ethernet if disconnected during install, then update the system
2. Install any necessary proprietary drivers not installed during original install
3. Install Flash, codecs, Sun Java
4. Disable screensaver lock (so it won't require password)
5. Disable power setting of suspend while on AC power - set to blank screen instead (laptop)
6. Set wireless to connect automatically
6. Install themes, fonts, Chromium, Opera and other preferred apps
 
I'm sure there are more things I do on every install, but it's become a process for me no matter what distro I run. That's not necessarily the order I do them in, but close. And as a new user I'd recommend running 10.04 as a first choice. It's pretty much bug free for many users as it was based on Debian Testing, not Debian unstable. I know you had difficulty getting anything but the alpha to run, but that seems odd unless there are incompability issues, a bad download, a bad burn, etc. 
 
And as mentioned before, dual boot with Windows. I say it a lot - there's no wrong answer with what you need to run on your machine. You can feel Windows is great or horrible and still run it. Only YOUR productivity with that machine matters. 
 
Best of luck and welcome!!

gtk is a toolkit for building applications in several desktop environments. one of the environments in which it is used is the gnome desktop environment, which ubuntu uses by default. don't make too much of it, it's motsly technical stuff. for everyday purposes, it's almost impossible for anything you download from the software repositories to trash your system. 

now, these repositories are basically software markets, similar to the app store, but they're free. there are several of them enabled by default, and others that you can enable. the packages that can be found in repositories are packaged in certain ways. packaging means converting the source code for a specific application into binary form, ie. something that your computer can understand, since it only speaks in ones and zeroes. the package format that ubuntu uses is .deb (debian package). the other most widely uses package format is the .rpm format, which you can't install on ubuntu, unless you go to more extreme lenghts. 

you can acess the repositories in several ways. one is through the software center, and that is probably the best way to go in the beginning. another way, also graphical, is through the synaptic package manager (just open the dashboard and start typing synaptic). the third way is through the command line. both the software center and synaptic are front-ends for a command line programm called advanced packaging tool, or apt. if you stick with ubuntu, apt is gonna be you favourite thing. when you see instructions on installing packages using lnes of code similar to 

```
sudo apt-get install foo
```that is the command line equivalent of clicking install in the software center. with time, you'll come to see that using the command line is the easiest way of managing software in linux, but it's nothing that you necessarily need to concern yourself now. 

unity is a shell, or the end user interface for the gnome desktop environment. the shell used in the last several years was called gnome panel, and you can still use it in natty if you log out and choose ubuntu classic as you session.

### Post by JKyleOKC on 2011-03-31
Another point about compiling as opposed to using the Software Center or Synaptic or apt-get or ...

The classic method of installing programs in Linux was to compile them from source, and it's still possible to do so.

However, you can cause yourself heaps of problems by doing this when you don't have to. The packages in the repositories have already been compiled, and configured to run with the standard setup for the distribution. That is, Debian (on which Ubuntu is based) does things a bit differently from Red Hat, and neither is always the same way the original program source tools configure things. That means you may have to do lots of tweaking when compiling from source, that would not be needed when installing from the repositories.

Even more important is something you're probably familiar with from windows days, where it was called "DLL Hell" -- that is, not having the right versions of the libraries for a program to run, and having to search all creation to try to locate them. In Linux, the word is "dependencies" and it means the same thing. Packages in the repositories have, as part of their structure, a list of dependencies, and the package installer will download them automatically if they are not already in your system.

Incidentally, the heart of all the package managers is a program called "dpkg" for Debian Package Manager; it's a CLI program with lots of options. All of the others mentioned in this thread are front ends or wrappers to dpkg, that greatly simplify its use. Software Center is the most simplified of the lot. I prefer Synaptic, which can tell me much more, but apt-get is great for a quick CLI install and for housecleaning.

apt is basically a set of tools to handle the dpkg utility. it has numerous functions. sudo apt-get update updates the list of packages in the enabled repositories. sudo apt-get upgrade upgrades the ones you have installed. the gui verions are marking all the changes in synaptic and hitting apply, or just running the update manager gui. apt-cache search foo searches the repositories for the package you want. 

that should be enough to keep you busy for a while at least :D

btw, if you ever heard the argument that linux is much more secure than windows, the repositories are one of tha major reasons for it. all the packages there have been tested by the cummunity and developers, and only trusted developers have the right to place packages in the repositories. 

the thing is, unlike in windows, where you download programs from third party sites and install them through .exe files, which gives the programs a life of their own, meaning they can upgrade on their own, things in ubuntu are somewhat different. ubuntu bases the selection of packages in the repositories mostly on the packages in debian (that's the distribution ubuntu is based on), and from one of its branches, known as debian unstable. a month before a new release of ubuntu is put out, those packages are frozen. from then on, only critical updates, such as security and bug fixes are commited. this means that packages are relatively new, but not cutting edge. ubuntu does get around the problems by releasing new versions every six months, but you have to upgrade your system to enjoy the new stuff. 

one way to get arround that is by adding ppa's, or personal package archives. these are packages maintained either by ubuntu developers, but not included in the official repos, or by third party developers. a possible problem with them is that ppa's are not necessarily compliant with the software in ubuntu, and can lead to instabilities in the system. 

another way to get around that is to download software directly from the web, like in windows. this software will not be updated by ubuntu's package management system, but sometimes it does pay to download newer verions, depending on your needs. when you do that, you're looking for .deb packages. or you can compile software, which is not hard, but can be frustrating if there are dependencies (programs that the original program needs in order to funtion properly), which you have to resolve manually. 

but don't go too fast, take it a step at a time. in the meantime, you might wanna read this, it's a good book for beginners. 

[http://www.ubuntupocketguide.com/download_main.html](http://www.ubuntupocketguide.com/download_main.html)
