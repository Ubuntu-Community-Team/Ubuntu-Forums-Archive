### Post by DuckHook on 2013-05-11
The reports show that your system is not getting EDID info from your monitor, so cannot sense monitor modes. You did say this in your original posting, so my fault for not reading carefully enough.

Do you use a KVM switch? A non-standard cable? Important that you answer these.

If all physical connections are default and normal, then we can try the following four options in increasing order of severity:

1. Boot up with LiveDVD/USB to see if everything works.
2. Rename xorg and reboot to see if system recognizes monitor with clean settings,
3. Force xorg to work in specific modeline,
4. Change NVidia driver.

Problem is, choices 2 - 4 could leave you with a non-functioning GUI. Are you okay with command line tools in the event GUI is rendered unusable? i.e. using nano to change files, using rm, mv and cp to delete, rename or copy files, etc?

If yes, and if you have backed up your data so that nothing can possibly lead to disaster (at worst, you reinstall), then:

1. It's time to install read-edid from the repositories. Do:```
sudo apt-get install read-edid
```You can then try:```
sudo get-edid | parse-edid
```...but this will likely return nothing useful because we already know that xorg is not receiving EDID info. However, if you are curious, it won't hurt to try. If you do unexpectedly manage to get useful output, of course, you should post it.

2. Try LiveDVD/USB first. If you have downloaded and burned the Ubuntu Secure Remix DVD recommended earlier, that would be an excellent choice. Don't choose "install". Just "try Ubuntu". If you get a GUI, then from Live session, do:```
lspci -vk | grep -iA20 vga | grep driver
```Result will likely be *nouveau*. If so, then repeat above step #1 (you will have to download "read-edid" again because you will now be in a LiveDVD environment, which knows nothing about your real installation). If EDID info is available, do:```
cat /var/log/Xorg.0.log | grep -i modeline | grep 1920x1080
```...and if this yields a result, then write down the resulting line(s) very carefully (we will be using it in option 3 and a mistake would be--well--not good). Shut down LiveDVD session and reboot normally.

3. If you want to try option #2, then:```
sudo mv /etc/X11/xorg.conf /etc/X11/xorg.conf_backup
```Reboot and, if you get a GUI, run:```
cat /var/log/Xorg.0.log | grep -i modeline
```Make sure it is the full "modeline", not just "mode" this time, as this will show if EDID is being sensed. If you do not get a GUI, then <Ctrl>+<Alt>+<F1> to switch to a console and do:```
sudo get-edid | parse-edid
```and carefully write down the output, if any. In the event that no GUI shows up, you can rename xorg.conf_backup back to xorg.conf with:```
sudo mv /etc/X11/xorg.conf_backup /etc/X11/xorg.conf
```...reboot, and you should be back to your current state. If there are no results at all, then we must ponder next steps (but this is enough for one night).

### Post by DuckHook on 2013-05-13
KVM switches are known culprits when it comes to blocking info between monitors and video cards. In this case, it blocked the EDID info being passed from monitor to video card. Sometimes it blocks the wakeup command after resuming from suspend. Mileage varies with the KVM switch.

DVI to HDMI cable should usually not cause problems if it is a good one. The cheap ones will sometimes be wonky (there's your favourite word again!) because they don't properly map the signals from one protocol to the other.

Your reference link is a good one and following it will allow you to force the resolution properly. If you capture the modeline (without the KVM switch) using any of the processes I laid out, then you can define one in your xorg.conf and force your card into a mode that your monitor will work with. However, if you don't need the KVM, I would stick with the KISS principle. Simpler is always better.

Good luck and Happy Ubuntuing!
