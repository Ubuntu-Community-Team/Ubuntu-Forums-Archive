Showing the commands is good, but without the results, we don't know what actually happened. Which worked and which are failed.

A grub window isn't the same as a root terminal.  It is a grub terminal. Usually the fix I use for that is a fresh install of the OS, followed by restoring from my backups taken the night before. I have automatic, daily, versioned, backups. The install takes 15 minutes. Restoring the backups depends on the system, but for a desktop it is 15min or less. Servers can take longer because there are backup files outside my HOME to be manually integrated into the new OS install. Typically, just 15 min longer, so in 30-45 minutes, the server is back doing what it does.

There is a way to save output from commands to a file AND have it show up on the screen.  This is handled by normal Unix redirection and the 'tee' command.  Very helpful, if you do it in advance. Of course, we never know that we need the logs until there is an issue. That's the experience aspect.  Whenever a command throws lots and lots of output, perhaps that's the time to <ctl>-c out, run the command with redirection to a log file?  Live an learn.  More techniques for redirection: [https://github.com/jlevy/the-art-of-command-line](https://github.com/jlevy/the-art-of-command-line)  I find it useful to re-read this file annually.  Each year, there is something new that is useful which I wasn't able to understand clearly previously.

As for the BT stuff ....  is this laptop bleeding edge? I ask because intel wireless stuff usually ships with working drivers already in the kernel tree.  Might it be something else, like using a soft-lock on the devices to prevent use?  I don't use bluetooth - security reasons - but there is a command, **rfkill ** that will show the state of wifi and bluetooth devices *hard or soft locked*.
I think something like **rfkill list** will show the current status. Other rfkill commands can be used to enable/disable each device.

The non-booting needs to be explained clearer.  How far does the boot process get?  Others will have to help with details, since my boot issues are few.  I'm proactive to avoid those things by some of my preventive practices each week which make any boot issue a minor inconvenience unless there is a hardware failure.

>  Terminal offered to autoremove a big chunk of packages no longer required
I fear something in the autoremove list was a critical part of the OS. Whenever there is a long list shown, carefully look through the list for things important to booting and running the OS.  If in doubt, you can post the list here for someone to look over.

>  Posting from an extremely old machine with half the keyboard malfunctioning badly, apologies if responses slow. I need a hammer to get some characters.

### Post by KBar on 2022-01-12
Unfortunately, no and this is exactly what I was talking about. **apt** won't accept garbage inside parentheses. For its input, version numbers can only be specified with an equal sign.

You are going to have to resort to **sed** or a similar editor to process this file and make it input-ready for **apt**.
[LIST=1]
[*]Open an editor of your choice, and copy and paste only the part which contains the names of the packages affected 
[*]Copy this newly created list and paste it back as an argument to **cat** 
[*]Pipe it to **tr**: ```
cat kerneloops:amd64 (0.12+git20140509-6ubuntu4), libkf5quickaddons5:amd64 (5.86.0-0ubuntu1) | tr --delete ','
``` 
[*]Perform **sed** substitutions and pipe it back to **tr** once more for nice, clean format. 
[/LIST]

The whole pipeline will look like this:

```
cat test | tr --delete ',' | sed "s/([^)]*)//g" | tr --squeeze-repeats ' '
```

Removing extra spaces isn't necessary since shell will interpret any amount of them as a delimiter.

### Post by TheFu on 2022-01-12
You can take that list of "removed" lines and make a new temporary file
```
$ egrep -i removing /path/to/that-log-file > /tmp/step1
```
Then you can create a list with just the 2 field using any method you like ... awk is what I'd use
```
$ awk '{print $2 }' /tmp/step1 | tee /tmp/step2
```
Then you can feed that list from step2 into apt install
```
$ sudo apt install $(cat /tmp/step2)
```

It is an exercise for the reader to put each of those commands into a 1-liner. Unix has 50-100 text processing tools.
