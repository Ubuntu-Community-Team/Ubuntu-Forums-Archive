### Post by Seq on 2010-03-12
> **ericeclifford said:**
> I think the startup script is not running at the correct times, it is running to early or something.

There are different run levels right?

There are different run levels, yes, but Ubuntu doesn't really do that. 0 is shutdown, 1 is single-user (recovery), 2 is where Ubuntu usually runs, 3-5 are not used on Ubuntu (and typically the same as 2), and 6 is reboot.

Within each runlevel there is also an order. If your script depends on say, networking or dbus, you would want to make sure your script starts after that. You may just want to set your script to start after all critical system functions have come up (#99), and stop before they all go down (#1).
```
sudo update-rc.d FOO defaults 99 01
```

If you depend on networking have you configured your network to not require Network Manager? Otherwise you might not get have networking until you log in.

If your program is not a service, in that it doesn't stay running in the background (fire, run, exits and it's done) then you may want to skip the init script all together. The file /etc/rc.local is called on every boot. If you call your program from in that file it will run.

No, it just means you're half-way there. Ubuntu always runs at runlevel 2, your script is running at runlevel 2, so you don't need to worry about that.

Within a runlevel things are started in a certain sequence. Here is a small list of some of the items in my rc2.d directory:

```
$ ls /etc/rc2.d/ -1
S19mysql
S20postfix
S91apache2

```

The filename starting with with S indicates the service will be started. The number is the sequence. So mysql is started before postfix, which is started waaay before apache2. If I wanted mysql to start after postfix, I could rename that file to S21mysql. Or S50mysql, depending on how much later in the sequence I want it.

Now what may compound your issue is that mounting is no longer handled via sysv-init scripts, but via upstart init scripts. I'm not sure how those fit into the sysv sequence scheme since they can, I believe, run asynchronously. Your best bet would just be to set your script to S99 and you should have all internal mounts from your fstab by then.

No, keep your startup script as "/etc/init.d/Foo". The files in /etc/rc2.d/ are links to the correct scripts. The manual way to do this is:

```
ln -s /etc/init.d/Foo /etc/rc2.d/S99Foo
```

But update-rc.d is there to simplify this.
