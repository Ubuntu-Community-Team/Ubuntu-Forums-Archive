### Post by AlphaLexman on 2011-03-07
Linux and Windows utilize different parameters to define 'hidden' files. in Linux (and *nix) files begin with a dot '.' to indicate they are 'hidden'. In Windows there is a file attribute '+h' to indicate it as a 'hidden' file.

Linux (and *nix) does not read the Windows '+h' attribute, and Windows will not automatically assume a file beginning with a dot '.' will have the '+h" attribute.

Also, how an external (or any) partition is formated will have an [affect | effect]? on saving attributes, permissions and hidden files.

How is your external drive formatted? ntfs, fat32, ext3, ext4, other?

Almost any OS can read fat32, it takes special drivers (usually installed by default) to read ntfs in linux. However, Windows (XP, the last version I have used) requires third party software to read ext3, and can't read ext4. I don't know enough about Win 7 to make a statement.

In conclusion, any Linux file with a dot '.' before the filename will be determined a 'hidden' file on any linux machine. Any Windows file with the '+h' attribute will only be 'hidden' on MS machines and filesystems, i.e., fat32, ntfs.

### Post by AlphaLexman on 2011-03-07
> **Insomnolence said:**
> External HD is formatted as NTFS
On your external hard drive, you are always going to see the $RECYCLE.BIN and System Volume Information in Linux, no matter what, you cannot hide them from Linux, They have a Windows attribute. Even if you rename them by putting a dot '.' in front of the filename/folder, Windows will create a new filename/folder and you will see the new version after re-booting Windows and retuning to Linux.

However, if you created a file in Linux:
```
/media/externaldrive/[COLOR="Red"].[/COLOR]hidden_linux_file
``` even though it is on an NTFS drive, the first time you boot windows, the file will NOT be 'hidden', but because the drive is an NTFS drive you can add the '+h' attribute to make the dotfile 'hidden' to windows products.
