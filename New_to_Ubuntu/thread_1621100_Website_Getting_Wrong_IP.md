### Post by SeijiSensei on 2010-11-14
What you see is a consequence of these rules:

```
-A PREROUTING -d 173.2.167.83/32 -p tcp -m tcp --dport 80 -j DNAT --to-destination 192.168.1.173:80
-A POSTROUTING -d 192.168.1.173/32 -j MASQUERADE
```

The first rewrites inbound requests for 173.2.167.83:80 to have the private address of your server as the destination address.  Then the masquerading rule tells the router to rewrite these packets as if it were sending them rather than machines outside.  You could try commenting out the MASQUERADE rule and see what happens.  It might work the way you want, or it might fail miserably.

One solution I've used in this situation is to run Apache in "[reverse proxy]("http://httpd.apache.org/docs/2.2/mod/mod_proxy.html")" mode on the router.  

However I find it more convenient simply to forward the requests directly to the web server.  I use an [application-level TCP proxy]("http://quietsche-entchen.de/cgi-bin/wiki.cgi/proxies/TcpProxy") for this task.  I compiled a copy from the tarball, but the .deb package on that site might work with Ubuntu as well.  In some applications I call tcpproxy from xinetd so I can wrap it with more security restrictions.  For instance, at one site I forward inbound SMTP requests to an internal mail server and use /etc/hosts.allow and /etc/hosts.deny with xinetd to block mail from servers located in non-US/CA country-code domains.  xinetd invokes tcpproxy when requests matching entries in hosts. allow arrive and refuses to forward any others.

### Post by The Cog on 2010-11-14
My guess is that commenting out that MASQUERADE rule will break connectivity. I think a number of other things need to happen before you can do the routing without the masquerading.

Firstly, the router PC needs to have IP forwarding enabled. But I think you've probably already done that bit:
**echo 1 > /proc/sys/net/ipv4/ip_forward**
I think you can make this cahnge permanent by adding this line to /etc/sysctl.conf:
**net.ipv4.ip_forward = 1**

Secondly, you need to add a default route to the webserver, pointing back towards the router PC as the gateway. This will allow the replies from the webserver to find their way back out to the internet.

Thirdly, your forwarding rule drops packets in both directions. Since the current NAT is working, I guess (not sure) that the NAT statement is overriding this default DROP policy and allowing packets through. You might have to re-address this though, after removing the masquerade, and add a specific PERMIT, maybe two: one for incoming to the server and one for outgoing from the server.

### Post by alphacrucis2 on 2010-11-14
It looks to me like this one is the problem one:

```
-A POSTROUTING -d 192.168.1.173/32 -j MASQUERADE
```


To me that's saying in the postrouting chain that if the destination address is the internal web server then masquerade the source which you don't want. You do have to masquerade the internal addresses outbound so they get a public address, which I think is being done by the previous line assuming that eth1 is what is connected to the internet. IP forwarding must already be enabled otherwise you would have no connectivity at all through the router.

The order things happen in iptables/netfilter is shown by this diagram:

   --->PRE------>[ROUTE]--->FWD---------->POST------>
       Conntrack    |       Mangle   ^    Mangle
       Mangle       |       Filter   |    NAT (Src)
       NAT (Dst)    |                |    Conntrack
       (QDisc)      |             [ROUTE]
                    v                |
                    IN Filter       OUT Conntrack
                    |  Conntrack     ^  Mangle
                    |  Mangle        |  NAT (Dst)
                    v                |  Filter


You can't even get to the post routing chain unless the forward chain or conntracking has already allowed the packet through.

[http://netfilter.org/documentation/HOWTO/netfilter-hacking-HOWTO-3.html#ss3.2]("http://netfilter.org/documentation/HOWTO/netfilter-hacking-HOWTO-3.html#ss3.2")

### Post by The Cog on 2010-11-25
When you say you can't connect to the server from inside the home network, am I right in thinking that you can't connect to it if you try to connect to its public address (by its domain name or by its public IP address), but that you can connect to it if you connect to its private internal (192...) address? If so, this is to be expected, and is a by-product of using NAT. I think what happens is that because the server now sees the real IP address of the caller and because it is a local address, it replies directly, not via the router and therefore not back through the NAT translation - the client sees unexpected acknowlegements from the server's local address and never sees the expected ack from the public address.

You could locally configure the name->address in /etc/hosts on all the local machines.

Or you could install a local DNS server that gives different answers to internal clients for this name (I don't know how to do that). 

Or you could use a separate IP subnet for the server, so that the clients always have to go via the router to get to the server. You can do this without the need for a separate physical LAN by (for example) putting a secondary address of 192.168.2.171 on the router's eth1 port and hard-configuring the server to e.g. 192.168.2.2. This means that eth1 is running 2 separate subnets and routing between them. I think it should happily NAT those conections too.

To put the secondary address on the router's eth1, use a command like:
> sudo ifconfig eth1:2 192.168.2.1 netmask 255.255.255.0 

You will also have to change the default route on the server to 192.168.2.1 of course.

### Post by The Cog on 2010-11-26
You mean to change the server address so that the other clients have to route through the router (and therefore NAT) to reach it:

Actually, there are three steps, not just the two I posted:

1:
Add the new subnet to the router as a secondary IP address: 
sudo ifconfig eth1:2 192.168.2.1 netmask 255.255.255.0

2: Change the server's IP address from 192.168.1.173 to 192.168.2.173.
This is done by reconfiguring the IP address on the server. 
Its default gateway needs changing to 192.168.2.1 as well, during this change.

3: Change the NAT entry on the router so that it NATs the public address to the new sever's address instead of the old one:
sudo iptables -D PREROUTING -d 173.2.167.83/32 -p tcp -m tcp --dport 80 -j DNAT --to-destination 192.168.1.173:80
sudo iptables -A PREROUTING -d 173.2.167.83/32 -p tcp -m tcp --dport 80 -j DNAT --to-destination 192.168.2.173:80
