### Post by srs5694 on 2010-12-22
The best hope for recovery is using TestDisk. Failing that, try creating a partition (using [gdisk](http://www.rodsbooks.com/gdisk/); *do not* use parted or GParted for this task!) that begins at sector 2048 and ends at the last available partition. Chances are that will be your Windows partition (or at least part of it). You may then be able to recover files, and perhaps even make the partition bootable again; but the first step should be to access the partition from Linux to see if it's properly sized and accessible.

There are two entirely independent questions relating to *how* this happened: Why is the disk GPT rather than MBR; and why is the Windows partition missing? The disk could legitimately be GPT if the computer is EFI-based. If the computer is BIOS-based, then it *should* be MBR, and it became GPT because of user error or a bug, probably in Acronis but possibly in the Ubuntu installer. The same two possibilities (user error and a bug) are the likely causes of the missing Windows partition. It's possible that one cause was a bug and the other was user error, and there could be one or two independent bugs or user errors involved. All of this is largely academic at this point, though; the Windows partition is gone, and recovery doesn't depend on how it happened. Of course, if Acronis is buggy or so poorly designed that people can carelessly wipe out their disks, that's bad news for Acronis, and it should be avoided. I've never used Acronis, so I can't comment on it from personal experience. The Ubuntu installer is not that poorly designed, although it might be possible to make this mistake with it if the user were sufficiently persistent. It's conceivable there's a bug in the Ubuntu installer that would wipe Windows on a GPT disk, but I've never heard of such a bug.

Some other random comments:


[list]
[*][My Web page on missing partitions,](http://www.rodsbooks.com/missing-parts/index.html) which has been referenced earlier, is irrelevant to this specific case.
[*]If Windows must be made to boot again *and* if the computer is BIOS-based, the disk will have to be converted back to MBR form, or at least have a [hybrid MBR](http://www.rodsbooks.com/gdisk/hybrid.html) installed on it. Both tasks can be done with gdisk.
[*]The GPT backup data will almost certainly be useless in this case; those backups are useful if the data structures get damaged by some errant software (like improper use of dd), not for cases of user error or high-level program bugs like what caused the problems in this case. To be absolutely positive, though, download gdisk and run it. If it doesn't complain about damaged data structures, the backups won't be of any use.
[/list]
