Depends on what you do with your computer. Some tasks benefit a lot from the higher precision & ability to calculate larger numbers in a single operation.

On the other hand very few of the "normal" desktop tasks benefit from 64-bit computing. It's not going to speed up web browsing or word processing. For normal users the benefits are most likely to show in multimedia-related tasks that require heavy processing, like audio & video encoding and editing. And even then only if the programs you use are actually made in a way that they *can* benefit from 64-bit computing.

### Post by Paqman on 2011-02-09
> **neil_1 said:**
> so theres not going to be any harm if i install 2 gb more ?  ^_^

Certainly no harm, but unless you're going to push the machine quite hard you probably won't see much performance increase either.

If you do step up as high as 5GB RAM there's a couple few points to consider:
[LIST=1]
[*]Make sure you're still running in dual-channel mode. I assume with 3GB that you have 2x1GB and 2x512MB? For 5GB you could go 2x2GB and 2x512MB. If you have one of the newer i7 boards you might need to have your memory in triple channel instead of dual. Getting it right will significantly improve your memory speed.
[*]With 5Gb I would advise setting swappiness very low, or zero
[*]The package preload is an adaptive caching daemon that will help make use of your extra RAM. It can take a week or two to start showing full effects. All you need to do is install that package and let it do its thing.
[*]With that much RAM, consider making more use of ramdisks. Shifting the whole of /tmp into RAM is an obvious first step, and some apps (such as browsers) can then be told to put all their cache into RAM as well.
[*]I realise some of this may seem like gobbledegook right now, but it'll point you in the right direction. People here can help you with any of the above.
[/LIST]

### Post by Paqman on 2011-02-10
Ubuntu will actually run really well with 1GB. Probably 2GB would be better though. Anything more than that you're not likely to see much improvement.

> **neil_1 said:**
> so i would have to install a 2x1GB for dual channeling ?


Yep. They can be any size, as long as they're matched pairs. Most dual-channel boards have four slots ie: two pairs of two channels. As long as each pair of slots has matched sticks (ie: same size, same speed, and preferably same manufacturer) then you can do anything you like. For 2GB you could do 2x1GB or 2 sets of 2x512MB. For 3GB you could go 2x1GB and 2x512MB. You get the idea.

> whats swappiness ?

Swappiness is a setting in the machine that tells it how much much memory it should fill up before starting to use the swap partition on the disk. Default is 60, meaning the machine will start swapping once RAM is 60% full, which is a terrible setting for a desktop if you ask me.

[How to change swappiness.]("https://help.ubuntu.com/community/SwapFaq#What is swappiness and how do I change it?")

> dint get it ):

It's a clever little program that runs in the background and takes note of what apps you like to use. then it preloads them into memory in case you want to use them, so that they launch super quickly if you do.

### Post by mcduck on 2011-02-10
> **NightwishFan said:**
> I think the true meaning of the swappiness number is lost to anyone but the one who created it. I know higher = more swap used, however it most certainly is not a percentage. Though I will be honest and say there is really little performance gain in setting it to 0 or 10. Even on 100, the machine will probably never swap.

Swappiness is just one of the factors the kernel uses to decide how much and when swap should be used. Here's how it works:

*distress* is a value telling how much troubles kernel is having freeing memory. it starts from zero and grows up to 100 based on how many attempts the kernel needs to do until it's able to allocate memory for something.

*mapped_ratio* is approximate percentage of how much of the system's total memory is mapped.

*vm_swappiness* is a parameter used to configure how much kernel should prefer swapping versus dropping cached data.

The actual swap usage, *swap_tendency* is then calculated like this:
**swap_tendency = mapped_ratio/2 + distress + vm_swappiness**

The result is that even with swappiness set to 0, the kernel will still use swap if it needs to (if the distress value gets high enough). It's just less likely to do that when compared to having swappiness set to 100.
