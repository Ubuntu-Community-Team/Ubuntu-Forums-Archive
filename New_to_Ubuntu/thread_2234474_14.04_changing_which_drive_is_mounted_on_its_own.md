### Post by newb85 on 2014-07-15
You have indeed cloned the drive, including the UUID's for the included partitions.  The first "U" stands for unique, and your system needs them to be so to keep them straight.

Basically, this means you have two options. You can continue as you are, with duplicate UUIDs, and your system will continue to get them confused, and you will continue to be frustrated (as anyone would be).  Or, you can change the UUIDs on one to be unique (practically speaking--they only need to be different from any other HDs your system will access).  That second option means that if you use one intended as a backup as a drop-in replacement when something happens, you will need to change the fstab file to reflect the UUIDs of the partitions on that drive.

### Post by coffeecat on 2014-07-15
Yes, you originally created a byte for byte clone of sda on sdb. If you look at the output of blkid, you can see that the UUIDs are the same in the two drives which will cause immense confusion. Even worse, the output of mount shows that sdb1 has been mounted as your root partition, the clone is being used as the operating system. From what you say in your OP it would appear that sda and sdb are being used almost at random as the boot drive and from which the operating system is working. This would mean that not only are your personal files not now sync'ed, but that installed applications and  updates may very well be different on sda1 and sda2.

You need to change the UUID of sdb1 if you wish to keep sda as your primary drive. It would be safer to do this from a live session by booting a live DVD or USB. However, you need to be very careful which drive you are changing the UUID on. If you boot from a USB, and depending on how the HP BIOS works, this may appear as sda in the live session and your sda and sdb will be shuffled up to sdb and sdc. Or not as the case may be. It's all a bit unpredictable. All I can say is that you need to check out which internal drive is which and then decide which you need to modify. The command to change the UUID on partition 1 would be:

```
sudo tune2fs -U random /dev/sdX1
```

Where X needs to be a, b or c &#8211; you'll have to work that one out.

Also, you have two swap partitions with the same UUID. You need to fix that as well. The same caveat about sda/sdb/sdc applies. Modify the swap partition on the secondary drive with:

```
sudo mkswap /dev/sdX5
```

Substituting X with a, b or c as before. mkswap without any parameters will apply a new random UUID. The chance of this being the same is vanishingly small.

By the way, I'll edit your post to put your terminal output between code tags to restore formatting and to make it readable.

### Post by coffeecat on 2014-07-15
Assuming that you are happy to physically remove one of the drives, what you do is below. If you are not, there is another way of doing this, which I can detail if you're not happy.

[list=1][*]Remove sda, the primary drive.
[*]Boot up with the live CD/DVD.
[*]Open a terminal and run this command:

```
sudo fdisk -l
```

You will see output similar to what you posted earlier, but with only one drive listed. Make a note of whether that is shown as sda or sdb. 
[br][/br]

[*]Run these three terminal commands with swapoff first, but substitute X in the second two commands with either a or b depending on whether the single hard drive is showing as sda or sdb:

```
sudo swapoff -a
sudo tune2fs -U random /dev/sdX1
sudo mkswap /dev/sdX5
```

[*]Shut down.
[*]Put the sda drive back in.
[*]Boot up.
[*]Let us know how you get on. Run:

```
mount
```

... and check the first line to be sure that it says "/dev/sda1 on / ...." If so, the problem is probably fixed.[/list]

You will need to look in your home folder in sdb1 to see whether there are files there that you need to transfer over to your sda1 home folder. I can't help you with updates and anything that you might have installed to sdb1 - you'll simply have to redo that.

No, not bad. If that's what you want to do - keep an updated clone of sda on sdb - then that would be a sensible thing to do. Although you said you cloned sda to sdb, it wasn't clear from your OP that you wanted a continuously cloned system - I was concentrating on getting you out of the problem caused by the duplicate UUIDs.

You might want to consider alternatives before committing yourself. My instinct is that RAID is overkill for straightforward home systems, but others would have different views.

### Post by vanadium on 2014-07-17
Sledgehammer indeed. It does not make sense to have a raid setup to protect you from data failure on a laptop. Rather make sure you change the UUIDS of the partitions on one of the drives, and then use that drive for storage.
Do not bother about safeguarding the operating system. These come for free and are updated anyway every now and then. Do instead focus on safeguarding your data. In order to guarantee that, you *need* a copy, and preferably several, on a separate, removable drive or in the cloud. Your data lost is lost forever.
