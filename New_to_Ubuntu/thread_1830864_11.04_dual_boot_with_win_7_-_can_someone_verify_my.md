### Post by srs5694 on 2011-08-22
I'm the author of both gdisk and FixParts, so I can speak with some authority about how they should be used and what they do.

The gdisk "-l" command just lists the partitions on the disk without writing anything, so if all you did with gdisk was "sudo gdisk -l /dev/sda", as you showed, you didn't damage the disk with gdisk; however, oldfred is quite correct to say that you shouldn't be using gdisk on the disk. If you were to launch gdisk without "-l" and then use "w" to write the partition table back, your computer would become unbootable. It's possible to recover from such a mistake, but it's best if you never need to do so. Thus, if you don't have GPT disks, it's best to not use gdisk at all, and it's probably safest to uninstall the package to avoid the possibility of its accidental misuse.

The Linux fdisk program is another matter; I wrote gdisk as an fdisk workalike for GPT disks, since fdisk didn't (and doesn't) support GPT disks. The Boot Info Script output you posted includes fdisk output (or something very much like it; I've not studied Boot Info Script to know how it generates that output). You can use fdisk to partition MBR disks, should the need arise -- but I don't see that need at the moment.

FixParts can be used to recover from certain types of partitioning problems, but I don't see any evidence of the types of problems it's intended to repair, so running it is unnecessary at best; and since there's *always* a risk when running partitioning tools, it's best to not run FixParts unless new evidence comes to light or if you do something else that creates the sorts of problems that FixParts can fix.

That out of the way, my best guess about what's going on is that something has damaged the filesystem on the Windows partition -- perhaps repeatedly. This is distinct from damaging the partitions. Partition definitions are quite simple; they just specify where partitions begin and end, provide a type code, and sometimes some other data like a "boot flag." Filesystems are much more complex data structures that are stored inside partitions. They include all the pointers to specific files, filenames, ownership, permissions, etc. Comparatively speaking, that's a lot of information and it's pretty delicate stuff. If a filesystem is damaged, the OS may pause for a long time as it boots and it may insist on checking the filesystem. It sounds like this is what's happened to you.

One common source of filesystem problems is improper shutdowns. If you shut down the computer by hitting the power switch, that's a no-no in almost any OS; you should always select the shutdown option at the login screen, type "shutdown -h now" at a shell prompt, or whatever. It's also possible that a bug in the NTFS-3g drivers used by Linux has caused damage to your filesystem. If this was a one-time event you might be able to safely ignore it, but if the problem keeps recurring, you might need to take more drastic measures, such as backing up, reformatting, and restoring the partition; or configuring Linux to *not* mount it automatically, or at least to mount it in read-only mode. The best course of action if you keep having problems may depend on your needs and abilities -- for instance, do you need to write to the partition? Can you use a separate data-exchange partition and leave your main Windows boot [noparse](C:)[/noparse] partition unmounted in Linux? ### Post by srs5694 on 2011-08-22
One way to test for hardware problems is to run a [SMART](http://en.wikipedia.org/wiki/S.M.A.R.T.) utility, such as smartctl, GSmartControl, or the SMART tests in Palimpsest Disk Utility. Such tools read the diagnostic data that most modern hard disks maintain and report it to you, which can help you identify a drive that's on the verge of failing. If your disk is going bad, a SMART tool will help you spot that fact before you lose all your data, so you can get a new disk and copy your data to it.

The big caveat about this is that most SMART utilities present the results in a highly technical form that can be difficult to decipher. You may need to post your results here in order to have somebody interpret the information.
