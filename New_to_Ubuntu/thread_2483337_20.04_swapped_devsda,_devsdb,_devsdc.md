### Post by TheFu on 2023-01-28
Drive letters can move around. Before dealing with any device file that isn't KNOWN to be fixed, unique like LABELS or UUIDs, always verify that the device name didn't change this time from the last boot.  Most of the time they don't, but sometimes they will.  

My boot EFI partition and storage with the OS is on sdb almost always in one system. sda is an old HDD that holds backups.  Every once in a while those will swap around. If I have a USB storage device connected during boot, the USB storage might be sda and the others are shifted and/or swapped.  Post boot, the USB storage is usually sdc. But if I have other USB storage connected, all bets are off, perhaps sdd will be used, but I want to check it before formatting.

Of course, for my USB storage devices, they all have labels and I mount them using autofs and the label to specific locations as needed.  They also umount automatically, which is nice.

### Post by TheFu on 2023-02-08
> **maketopsite said:**
> Yes for example writing iso file from hdd to usb stick using dd command. I've become only one step from a disaster once.

Or just use the symbolic links that are generated at boot for each storage device.  Run 
```
$ find /dev/disk 
```
to see all the possible different names.  These are generated when the storage is seen.  UUIDs, LABELs, Path, and a few other UniqueIDs are each there for our use.  Don't use /dev/sdX.  Some examples:
```
/dev/disk/by-id/dm-name-hadar--vg-swap_1
/dev/disk/by-id/dm-name-hadar--vg-root
/dev/disk/by-id/dm-name-vg--hadar-lv--backups
```

If the storage uses LVM, then we have /dev/{vg-name}/{lv-name} links too.  I use these in the /etc/fstab file, since they actually provide useful information AND work.
```
$ find /dev/vg00
/dev/vg00/xubuntu22.04
/dev/vg00/xubuntu22.04-backup
/dev/vg00/ubuntu2204-srv
/dev/vg00/ubuntu22.04-srv-2
/dev/vg00/home
/dev/vg00/var
/dev/vg00/swap
/dev/vg00/root-00

```
Very useful and unique.  I named vg00 for the VG and the LVs all have names I chose.  Moving or cloning a running VG is 1 command. Very powerful.

That's the point, it **is** doing it automatically. Use the symlinks in /dev/disk/by-id/ instead of /dev/sd* and you are less likely to overwrite the wrong device when using dd to flash an image ...
And mount has been supporting using UUID= and LABEL= for several versions of Ubuntu, so there really is no excuse for not using those to mount the right device in the right place.

It does, if you use LVM, though I think it uses the /dev/mapper/xxxxxxx links, instead of the /dev/{vgname}/{lvname} links. There is a clear relationship between those, so I change the /etc/fstab to use the more descriptive LVM links.

For non-LVM stuff, like USB storage, I use LABELs in the fstab and with **autofs**.  Being a human, UUIDs are pretty useless to me.

### Post by The Cog on 2023-03-22
To summarise, /dev/sd* allocations are not stable, and can change on every boot. I believe they became unstable as part of changes to support hot-plug, and faster booting by using parallel hardware discovery.

/dev/disk-by-* is back-filled with symlinks after the raw device is discovered, using information discovered about each disk (UUID, LABEL etc). These links are created to link back to the current /dev/sd* device ID to allow for reliably finding a particular disk, e.g. in fstab. So disk-by-uuid (and similar) entries **always** link back to the correct disk device regardless of the order of hardware discovery. I guess they may also come and go as you re-format or re-label partitions, unplug drives and re-plug them in different slots etc.
