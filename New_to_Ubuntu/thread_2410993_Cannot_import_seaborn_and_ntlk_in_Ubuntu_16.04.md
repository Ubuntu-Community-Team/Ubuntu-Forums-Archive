### Post by Holger_Gehrke on 2019-01-23
Looks to me like you got python 3.6 **and** 3.5. pip3 installs stuff for 3.6 and 3.5 can't find it (it's probably looking in /usr/local/lib/python3.5). So you can either use python3.6 and have the modules installed or run pip3 with python3.5 to install the modules again in the places where the older version of python3 expects them to be.

Holger

### Post by Holger_Gehrke on 2019-01-23
For packages installed through apt, the command is 'sudo apt-get remove *packagename*' or 'sudo apt-get purge *packagename*' (the latter also removes configuration files generated by the program); so it should be 'sudo apt-get remove python3.7.2' if that's really the package name (it looks a bit unusual to me ...). For programs installed from source you need to check the README or INSTALL files included or the Makefile. A lot of programs provide a (pseudo-) target 'uninstall'; if that's defined in the Makefile you can possible just do a 'sudo make uninstall' in the directory with the source code to uninstall the program. Before removing Python 3.6.3 you should remove the packages you installed through pip. pip seems to install programs into directories based on the version of python that's running pip (pip itself is written in Python and can access the version information of the interpreter it runs under), so if you remove 3.6 before removing the pip-installed packages getting rid of those would become difficult.

Holger

### Post by monkeybrain20122 on 2019-01-23
Works here. I have python 3.5 (system) and python 3.6.8 (from source) the issue is when you invoke pip3 you have to be sure which one you are calling, I installed python3.6.8 from source in a special directory and use a script to set the environmental variables when I use it so there is no confusion.

Here is my script called python36.conf

```
export PREFIX=/home/monkeybrain/opt/python36
export HOME=$PREFIX #for spyder to create seperate config then system 
export PATH=$PREFIX/bin:$PATH
export LD_LIBRARY_PATH=$PREFIX/lib:$LD_LIBRARY_PATH

export PYTHONUSERBASE=$PREFIX
export PYTHONPATH=$PREFIX/lib/python3.6/site-packages:$PYTHONPATH
```

/home/monkeybrain/opt/python36 is where I installed python3.6.8
The second line is a hack so I can  install a separation spyder with its own config file so that it won't share config with the system copy

so to use python3.6 (I put python36.conf in a directory ~/Scripts)
```

$source ~/Scripts/python36.conf
$which pip3
/home/monkeybrain/opt/python36/bin/pip3
$ python3
Python 3.6.8 (default, Jan 14 2019, 11:58:32) 
[GCC 5.4.0 20160609] on linux
Type "help", "copyright", "credits" or "license" for more information.
>>> import numpy as np
>>> np.__file__
'/home/monkeybrain/opt/python36/lib/python3.6/site-packages/numpy/__init__.py'
>>> 




```

Without invoking the script

```
$which pip3
/usr/local/bin/pip3
$ python3
Python 3.5.2 (default, Nov 12 2018, 13:43:14) 
[GCC 5.4.0 20160609] on linux
Type "help", "copyright", "credits" or "license" for more information.
>>> import numpy as np
>>> np.__file__
'/usr/local/lib/python3.5/dist-packages/numpy/__init__.py'
>>> 



```

Note the different version  of python and different locations of numpy and pip3, the two numpys were both installed with pip3 for their corresponding versions of python3 (for python3.6.8, source ~/Scripts/python6.conf before running pip3 install)

The PYTHONUSERBASE environmental variable is for pip3 install modules

so if I source ~/Scripts/python36.conf

and then do pip3 install -U foo (no sudo)

it uses the python3.6.8 version of pip3 and the module would be installed inside the python36 directory, which will be separated from where python3.5.2 's pip installs modules. 

That way there won't be any inconsistency.

The common way to use multiple versions of python  would be to use something like anaconda, but I prefer this since anaconda installs a lot of junks I don't need and creates a dependency tree which makes the installation less flexible.
