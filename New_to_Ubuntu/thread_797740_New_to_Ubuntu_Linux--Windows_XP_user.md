Hi...

This is *most* operating systems, including Linux. Although, in Linux's case, it is a slower progression, plus there are distros out there that are specifically produced for the benefit of older machines.;)

A point that I should make clear to any one considering migrating to Linux with no prior experience (which I learned very well) is that Linux operates VERY differently when it comes to installing software and drivers! In the old days, one had to learn the art of compiling, make and make installing and Linux command line language in a terminal almost exclusively.

It's gotten better with programs like Synaptic and Adept but it's still not like Windows. There isn't the "point, click, install, it works" with anything that catches your fancy that Windows offers. With Ubuntu, you either install it from the repositories, which is easier, or you go the old fashioned "tar.gz" route as I've described above.

Other comparisons I have found:

1. Ubuntu has better memory management than Windows.
2. You can get a whole lot of software for free that under Windows would cost you thousands of dollars, although quality can vary.
3. The quality of technical support is about the same, although with Linux, it is mostly through the forums as opposed to one on one phone support.

Hope this helps...

> What do I look for in terms of system requirements?  Are commercial products also made to run on Linux (or Ubuntu)?

Some commercial products are made to work on Linux. For these, just focus on the system requirements that make sense to you. A lot of companies go into detail of what versions of libraries you need, but most of the time this is conservative enough so you can just ignore it - you're likely to have recent-enough libraries.

Why do you necessarily want commercial products? Linux is a platform, not just an operating system, and open-source programs can be just as good as the operating system.

Windows XP with antivirus is terribly slow compared to Ubuntu. A fresh Windows XP SP2 will beat Ubuntu in speed, as you're comparing a 4-year-old operating system with the latest of everything on Linux.

### Post by phread59 on 2008-05-19
I think I'll take a hack at one of your questions.

What is the difference between regular Linux and Ubuntu?

Well it's a tough one to answer.  There are basicly 2 major branches of Linux.  They are Debian and Red Hat.  The biggest differences are in how software packages are handled.  Ubuntu is a derivative of Debian.  Debian software will run just fine in Ubuntu.  Red Hat software although it is Linux will not run straight up in Ubuntu.  It uses a different file type.

The file type it uses is RPM or Red Hat Package Management (as I recall).  Red hat uses a different file type and must be converted to use in Debian.  The last bit of info is the way packages are handled.  RPM packages have a long history of poor dependancy support.  Whereas Debian does not.

What this all means is with RPM you don't always get all of or the correct dependancies for a given piece of software to work  or work properly.  When you install a piece of software such as Amarok (a very good audio player) you may need several other programs to get it to work.  They may not be very large but without them it won't work.  Synaptic will get them for you and install them very easily.  RPM's Yum package manager is not anywhere near as good.  So you need to look at whether a Linux Distibution (Ubuntu,Fedora ect.) is a Red Hat or Debian and make your choice accordingly.  I think Ubuntu is the best out there.  others may disagree.  None disagrees that the Ubuntu support here in these forums is the best out there, period.  You can't be in a better place for a linux noobie.

### Post by ugm6hr on 2008-05-21
I'm not sure where you are getting your (mis)information from, but let me try to clarify...

Grub does not have problems with Dell computers.

Grub does not disable booting to Windows unless the Linux partition fails to install properly (unlikely), or you install Linux on a separate HD but put Grub on the first HD, or if you uninstall Ubuntu without preparing properly.

> That does seem like the best option as the install program should ask which drive to put the Ubuntu installation on. More then likely if I keep it simple like that rather than go to just a partition on the second drive, that should increase my chance of success, correct?

Let me try and clarify again.

Partitioning does theoretically put data at risk, and hence backups are advised before attempting anything like that.  Note I have never had a problem, but I would also not recommend doing this without a backup.

Installing on an already wiped partition, or on a second HD does not put data at risk *as long as you follow instructions correctly*.

If people have made a mess of dual-booting, then it is only as a result of either:
1. Partitioning error
2. Human error

Unfortunately, while the Ubuntu installer does ask you where to install itself, it names the drives as hda / hdb etc; so it is easy enough for the uninitiated to pick the wrong one (and wipe Windows for example).

It doesn't specifically ask you where to install Grub (unless you click on the "Advanced" button at the installation / partitioning stage).  Worse still, the location of Grub follows a different naming scheme (e.g. hd0,0 or hd1,0 etc), so can be incorrectly located due to user error.

These are the main hurdles to pass when installing Ubuntu - just so you are prewarned.

The Ubuntu installer also defaults to taking over your whole HD - hence some users wipe Windows completely.   When using a dual boot option, Ubuntu is often installed in a separate partition on a single HD with Windows, so the installer "guides" you through how to achieve this, which means that some of the defaults will not be correct for dual HD installation.

I know this all sounds complicated - and it is for those unfamiliar with partitioning, bootloaders etc.  This is why, unless your aim is to wipe your HD in favour of Ubuntu, a degree of knowledge is required prior to attempting a dual-boot setup.  In fact, installing any OS (including Windows) is something not to be done without some knowledge.

What I would suggest is that you decide where you are going to install Ubuntu (i.e. on hda or hdb), how much space you want to dedicate to it, and then report back here for instructions on which options to choose (and explanation why) when installing.

As some people have already mentioned, Wubi allows you to install Ubuntu within Windows.  I don't know a lot about it, but it might be worth doing a bit of research on if you are concerned about partitioning.  I believe that it also has the publicized uninstall option, but can't vouch for its reliability.

I am not an expert on these things, but I think that way that IDE and SATA identify themselves as Master / Slave on an IDE connector is different (former uses a type of switch, latter uses position on ribbon cable only).  If possible, either use 2 ribbon cables to make both Master drives (and keep CD/DVD as slave, with new SATA as 1st Boot device), or make the new 500GB as Master with 80GB as Slave.

I would recommend you use Ubuntu to do the partitioning - but it doesn't really matter.  Also, you will have to choose a file format for your storage partition - that is another discussion in itself - although it can seem "safer" to use NTFS from within Windows.

### Post by ugm6hr on 2008-05-24
This is what I would recommend (based on what you said you wanted re: split HD in half):

1. Set new 500GB HD as 1st boot device
2. Boot Ubuntu 8.04 LiveCD
3. Start Partition Editor (System->Administration)
4. Select sda.
5. Create partitions as required (see below for approx sizes):
sda1 primary 25GB (plenty for /); sda2 primary 225GB (plenty for /home); sda3 extended, divided into sda5 249GB (for "storage") and sda6 1GB (swap)
6. Start "Install"
7. Select Manual install option
8. Mark the /, /home and swap as above (and format)
9. Leave other defaults (including hd0 for Grub)

Hopefully that should work!

Once you have installed - double check the Grub location by selecting the Windows HD as first boot, and see if it boots Windows.  If it doesn't - you can recover the Windows bootloader and reinstall Grub in the correct location (just ask again).
