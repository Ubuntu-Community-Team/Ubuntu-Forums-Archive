### Post by Rumtscho on 2009-04-04
I set up an old machine in order to learn Linux. I only have an old CRT monitor for that. The relevant system components are: 
- Monitor: Medion 1772xe 
- Video card: nVidia GeForce FX 5600 (has one VGA and one DVI output, I only use the VGA one) 
- Driver: nVidia 173 
- OS: Ubuntu 8.10 

After installation, the system starts in a 800x600 mode. Today I got around to installing the video card drivers, and after restart was greeted by a new resolution: 640x480 :mad: 

I spent the whole day in learning what the xorg.conf file is good for and tweaking it. First, I saw that my display won`t send a readable EDID to my graphics driver. Therefore, the graphic driver uses a default mode called "nvidia-auto-select", tries to determine the possible modes for the monitor and decides that there are only 640x480 and 800x600. I stopped the driver from doing the EDID check and added the line ```
Modes "1280x1024" "1024x768"
```, but this didn't change anything, because the driver doesn't find this mode, so it starts again at 640x480. I accidentally discovered that if I write something wrong into the xorg.conf, at startup I am greeted by amessage "xorg.conf unparsable, what would you like to do", andd when I select the option "start Ubuntu in low-resolution mode", it starts in 800x600, just as before drivers were installed, and the log file is quite different. 

I hooked the system first to my windows PC's monitor (a TFT, model is FSC C17-6); there I got the same 640x480. Then I tried it on my boyfriend's monitor (another TFT, Neovo F-417), and with an unparsable xorg.conf it started in 1280x1024 mode. With the PC running, I disconnected the TFT and connected the CRT, and it remained at 1280x1024. 

When the PC still ran under Windows, it displayed 1280x1024, 32 bit color, with the same video card and the same monitor, without need for any drivers. Please help me to get it running in this mode under Ubuntu. 

Here you can read my conf file and the various log files (irrelevant sections removed, header only included in the first one). I know they make the post quite long, but I don't dare trim anymore as I am not sure which information is relevant

### Post by Rumtscho on 2009-04-05
Thank you very much. I still had to change the xorg.conf, but you helped me localize the problem and thus determine the solution. 

The 180 driver used the autoselect mode again, which only got me 800x600 this time. But it wrote a bit more output to the log. Turns out the driver actually tried the 1280x1024 mode, but in a very narrow refresh rate range. 180 didn`t report "Not a valid mode" but "Hsync out of range". So I started Ubuntu in 1280x1024 again (using the F-417 monitor), then without restarting disconnected the monitor and connected the CRT instead. It continued in the mode used on the F-417, and per hardware button I was able to determine that the hsync in use is 60 (the driver had been trying the 30-37 range). Added to xorg.conf a HorizSync line with the range 30-75, and it worked like a charm! ):P
