### Post by JKyleOKC on 2011-01-12
Yes, I'm running Lucid these days. I made a stupid mistake and left my system an unprotected opening to the outside world, and consequently it got invaded early this summer. The worst of it was that I had my password in a file (in what I thought was the protected area) and as a result the invader got root access to my whole LAN. I had to reformat all of the machines and re-install everything from scratch, so rather than stay with the almost-expired Hardy, I went to Xubuntu Lucid.

My main machines have 2 and 3 GiB respectively, so that I can run VirtualBox to do all the Windows-only things that I need to deal with, but I have a couple more that are much older. They both use PC-133 memory sticks, which are hard to find any more; one has 256 MiB and the other has 512. Both of them are able to run Lucid, but the one with only 256 is painfully slow.

The reason for the slowness is that with so little RAM, the system has to "page" chunks of RAM back and forth to the swap partition on disk, and disk access is literally thousands of times slower than is RAM. With only 256 MiB, this swapping has to happen extremely often. You may have noticed that Linux always has more than 100 tasks running, most of them in the background and so usually not noticed (this machine has 166 of them going right now). The system switches between tasks around 10 times every second, and many of the tasks remain idle for minutes at a time, but as these switches happen, it's often necessary to page out a few blocks of RAM -- and all of that adds up to give you 100-percent CPU usage and very slow response, especially when starting a new program.

I would no longer recommend you try VirtualBox with these older boxes; I've found that it tends to slow things down even more when the RAM is less than one GiB, although when a box has 2 or 3 GiB of RAM and the processor is at least dual-core, then a virtual machine can almost equal the snappiness of a non-virtual installation. Just about any desktop you can buy today will meet those specifications, but that's only been true for about a year now...

As for partitioning, that can be fairly easy to get familiar with, especially with a "sacrificial" box that you won't mind doing repeated reformats and clean installs on. The "gparted" program you can download from the Software Center will show you a display of how a specific drive is partitioned, and you can change the partitioning by dragging a boundary, or an entire partition, in either direction. If you choose "manual" partitioning at the point during the installation process when it asks, you'll see and use the display. Don't try it on your main working machine, though, until you're comfortable with it!

Guessing RAM type from the location notches generally work, but you can get caught like that as well. ECC, Buffered and other 'exotic' RAM types share the notch pattern with other RAM sticks of the same type (EG, SDRAM ECC has 2 notches, in the same place as 'normal' SDRAM, but the actually connectors are offset). Trying to run normal SDRAM in a motherboard that only supports ECC (or vice versa) can lead to confused head scratching. 

The only reliable way to know what RAM a system needs is to fidn the model number, then go to the mnaufacturer and see what it supports. 

You can normally find the model number from lshw, but inkeeping with your genrally bad luck with computers Chris Edgell you dont get that. Annoying. You can try looking in the BIOS, with a lot of computers the motherboard type appears both on the BIOS initialise screen and inside the BIOS as well. 

If both of those fail, the eyeball method normally works, most manufactuers screenprint the model number onto the motherbaord.
