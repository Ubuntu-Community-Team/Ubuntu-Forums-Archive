Is your GRUB menu visible when you first boot up your computer?

Sometimes the GRUB menu is not visible at boot time, especially if you only have one operating system installed on your hard drive.

If its not visible apparently you can hold down a key for example the 'shift' key and this will make the menu visible as you boot up.

If it is visible, then you will be presented with a set of options for how you want to boot up.  Ideally you would read them and play around with it a bit, without making any changes at this current moment.  The Grub menu offers the options of pressing the 'e' key to edit lines, or the 'c' key to enter commands.  It also shows a list of boot options, the first one being the normal boot, the second is generally a 'recovery' mode, and often it has the Memtest (for testing your RAM memory for errors).

I think it best if you just find out if you can see the menu first before proceeding much further with explanations.

### Post by Mustard on 2010-02-11
Well, my memory on these things is not so good, as I tend to just reread whatever how-to's I google up as the need arises.

Basically the troubleshooting that we are doing at the moment is to determine if your filesystem has become corrupted and attempt to repair it.

Antonio suggested you run 'fsck /dev/sda1' from Grub, which I have never done before.  What little I know of fsck, is that it can only be run on a filesystem that is not mounted.

If it was me, I would be booting up from a liveCD and running fsck on main linux partition.

From the output you gave earlier using the fdisk -l command, which lists all the filesystems the operating can see, it shows your main ubuntu partition as /dev/sda1.  I have some doubts about Antonio's suggestion about checking the filesystem from the 'recovery' mode, because by the time you get into recovery mode, the partion /dev/sda1 would have already been mounted by the operating system.  This would mean that fsck would not work, because as I said earlier, fsck can only be run on an unmounted partition.

From my own experience it would be better to use a liveCD of some kind which runs the operating system from the CD, and doesnt mount the drive partitions.  From a liveCD you could open a terminal and type in fsck /dev/sda1 or sudo fsck /dev/sda1 if necessary and this would scan the hard drive, find errors and potentially repair those errors.  After doing this we would have a better idea of whether filesystem corruption is the problem or not.

If we get through all this process and we find that the filesystem was not corrupted, then we are back at square one, which is, no answer to the problem.  I guess you have to judge for yourself whether its worth the effort or not.

I know I dont have the time to coach you all the way through the process, so its really going to be a tag-team support effort, as someone is going to have to take over from me.

### Post by gsocker on 2010-02-13
I note there are a few "chmod 644" commands in your history. Due to how Unix/Linux deal with file permissions, it is possible to make your own files unreadable by you.

Given that your filesystem is clean, and that  I don't see anything in the history that would delete everything, this is most likely the problem. Especially given the output of ls -la:
```

   drw-r--r-- 35 luke luke   4096 2010-01-13 21:55 Documents
  
  drw-r--r--  2 luke luke   4096 2009-10-29 15:05 Photos
  
  drw-r--r--  4 luke luke   4096 2009-11-11 09:42 Pictures
  
  drw-r--r--  6 luke luke   4096 2010-01-25 15:08 Desktop
  
  drw-r--r-- 12 luke luke   4096 2009-10-03 11:56 Music

```Boot to a root shell in recovery mode, as described by earlier posters. 
Change to your home directory:
```
cd /home/luke
```Run:
```
find . -type d -execdir chmod u+x '{}' \;
```Make sure there is a space between the single qoute and the backslash.
What has happened: in Linux/Unix, the 'execute' permission is also used as the 'directory is accessible' permission. For chmod, '644' would remove the 'execute' bit, making your files inaccessible, although they still exist.
Your chmod commands must have been run in your home directory, thus making every sub-directory in your home directory inaccessible.
As you did not run 'chmod -R', you should only need to fix the top-level subdirectories; anything under these should still be OK.

When you are typing, in a command line. Lets say you want to get to a folder in Documents called "TestFiles". You can use tab-autocomplete to get there quicker.

What you would do, is type 
```
cd ~/Doc[tab]Test[tab]
```
Then, each time you press tab, it should insert the matching file/folder name for you, and you won't have to worry about typos. I find this very helpful for folders with spaces in them. That way you don't have to worry about escaping the spaces which is a headache.
