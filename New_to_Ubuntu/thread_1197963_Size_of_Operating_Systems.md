### Post by krow436 on 2009-06-27
Here are my thoughts as far as the reasons for this:
1. Bad coding: This results in achieving the same or similar results in a more time or space consuming manner.
2. Closed source: A lot of time and space is used making sure you can't play with as much as you'd like, or reverse engineer.
3. Legacy code: In an effort to keep backwards compatibility as complete as possible, M$ has only been adding to, instead of rewriting, their kernel for years.  I don't think I have to explain how that adds to the problem.
4. Hardware deals: Probably not the best name for this heading, but when M$ started with MSDOS, hardware vendors had no alternative but to go to them because they supported more hardware than any other DOS alternative, which was the main OS at the time, and people enjoyed and used that.  Now, there are alternatives.  Vendors want M$ to push your hardware to the limit so that you will have to buy new hardware, and they are completely willing to switch to supporting certain *nix systems instead of M$ if they won't play along.

I personally think a clean kernel rewrite would do wonders for Windows.  Only keep legacy features that are necessary, and do a better "Compatibility Mode" that can be downloaded separately for people that don't want to fill their hard drive with older stuff.  As far as the hardware vendors, most people are to stuck in the Matr... *ahem* Windows and aren't ready to be freed.  They would find a very sharp drop in revenue by switching over in the current software market.

### Post by Paqman on 2009-06-29
Shared libraries.

On Windows, apps tend to come pre-packaged with everything they need to run. That's handy, but can lead to a lot of duplication when you've got a lot of apps.

The fact that Linux has package management built-in gives it a huge advantage in installing and removing packages elegantly. If most of he dependencies of a package are already satisfied by stuff originally installed for other apps, a Linux system will only download the extra bits that it needs to complete that package's dependencies. That means a fast download and minimal footprint on your hard drive.

### Post by 3rdalbum on 2009-06-29
Windows Vista comes with (apparently) gigabytes worth of drivers. All those drivers are written by hardware manufacturers who don't have any contact with eachother. There is no code reuse.

Linux comes with loads of drivers, but they are written by kernel developers who have contact among eachother. If one is writing a device driver and realises that a part of the driver would be useful to use in other drivers, they can put that piece of code into the kernel, and other developers can call that code in their drivers. Result: Each driver takes up a small amount of space due to code reuse.

Mac OS X is currently bloated because of inefficient coding. The Rixstep website dissected programs in the default OS X install and discovered that they have been built with an enormous amount of duplication of resources - for instance, if an icon is used in three different windows of a program, the icon will be stored inside the program three times. And a lot of program resources are duplicated a lot of times without any apparent reason.

Rixstep released a program to strip out the duplicates and save space. That's how Apple is cutting 6 gigabytes from its default Snow Leopard install: It's fixed the problem of the programs having the duplicated resources.

Paqman (the poster above me) is also correct: Windows programs are static-linked to a lot of the stuff they need, and the installers for Windows programs tend to include absolutely everything they need. Whereas on Linux, when you install a program, it checks to see whether you've already got a copy of its dependencies. If you don't, it will get the package manager to download the dependency in a system-wide place where other software installers can find it. Take a look at iTunes - the installer takes up 50 megabytes (or more?) because it includes a full copy of Quicktime, regardless of whether or not you already have Quicktime!* The nearest Linux equivilant, Amarok, takes up 15 megabytes and will automatically pull down any multimedia libraries that it needs.

*In fact, I was really annoyed once. I wanted to upgrade Quicktime and iTunes on a Windows computer, to the latest versions. So I downloaded the Quicktime installer and waited ages for it to download. Then I installed it. Then I downloaded the iTunes installer and waited ages for it to finish downloading. At one point during the install it started up its own embedded, full Quicktime installer to tell me that I already had the latest Quicktime... and I realised that I'd wasted my time downloading the Quicktime installer separately!
