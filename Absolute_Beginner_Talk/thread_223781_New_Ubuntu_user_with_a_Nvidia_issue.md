### Post by Iandefor on 2006-07-27
> **l.j. said:**
> Thanx for the reply ran the code black screen on reboot had to restore backup
xorg file  :(   the saga continues :confused: 

```
 sudo -s -H
cd /etc/X11
cp xorg.conf xorg.conf.backup
sed -e 's/nv/vesa/g' xorg.conf > xorg.conf.new
mv xorg.conf.new xorg.conf
exit
``` 
Is your best bet for getting to a graphical interface quickly. The GeForce Go 420 works with the VESA driver. One thing, though: if you have a widescreen, it'll get stretched and look like crap.

EDIT: If you want to know more about what this series of commands does, PM me and I'll break them down for you. Just an open-ended offer from a slightly experienced noob to one who seems rather more new :).

### Post by The Noble on 2006-07-27
There is an explanation for the 24 bits deal

> 
9-24-99 Peter Hatch

What's happening here is that X servers run "32" bit mode using only 24 bits of color depth. So, if your X server is running at 32 bpp then what happens is VW gets a pixel value from X and *thinks* it's a 32 bit pixel--which it isn't. When VW attempts to perform bit shifting operations on the pixel value those operations are done under the assumption that the pixel value is actually a 32 bit pixel--which it isn't. The 32 bit pixel operations are actually performed on a 24 bit pixel which is a Bad Thing.

There's got to be a decent design fix for this, but in the meantime if you're having this problem then run your X server in 24 bpp mode. You don't lose color depth and the VW image classes will all work fine. This is a limitation for *all* 32 bpp RGB systems that I know of. The exceptions are those systems that run in RGBA mode (thus filling in the blank 8 bits at the end of the pixel), but I've got no real data on those systems....

This is an exerpt from a page on 32bits actually displaying 24, so in reality, go with 24 bpp in xorg.conf.
