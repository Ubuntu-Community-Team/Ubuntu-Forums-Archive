### Post by Rui Pais on 2007-06-16
hi,
You /etc/network/interfaces is incorrect.

It sets eth0 as static and as dhcp simultaneously (set one or the other, not both!!). 
I have read reports of network failures when it contains interface specifications that didn't not exist.
Try this:
```
sudo mv /etc/network/interfaces /etc/network/interfaces_BAKUP
```
```
gksudo gedit /etc/network/interfaces
```
and in the new empty file just copy+paste this:
```
# This file describes the network interfaces available on your system
# and how to activate them. For more information, see interfaces(5).

# The loopback network interface
auto lo
iface lo inet loopback

# The primary network interface
auto eth0
iface eth0 inet static
address 192.168.1.101
netmask 255.255.255.0
gateway 192.168.1.1

```


Your resolv.conf do not show OpenDNS IPs. 
Some router have problems solving DNS addresses on Linux (when the same ones are used by the same equipment on Windows they work) 
Usually OpenDNS always work under Linux so they are a goos test to see if its a dns issue or not.

Have you done this:
> If your settings get revoked after reboots, or after periods of inactivity, you can try this:

    $ sudo cp /etc/resolv.conf /etc/resolv.conf.auto
    $ sudo gedit /etc/dhcp3/dhclient.conf
    # append the following line to the document
    prepend domain-name-servers 208.67.222.222,208.67.220.220;
    # save and exit
    $ sudo ifdown eth0 && sudo ifup eth0 
as indicated in the [link]("http://www.opendns.com/start/ubuntu.php") I posted?

Do the chances on this 3 files, reboot (to be sure that hardware get correct ips) and report back, please.
Good luck.

If you only have 1 or 2 boxs on your network, there is no reason to use DHCP assign address. It's slow and waste resources. Static IPs are simple to deal, faster, and you remove one variable of your problems equation. 

Another rare situation is the inability of Linux to handle correctly some hardware when wake-on-lan is not set.
This case i have seen reported to be a problem with RTL8168/8111 cards, but the symptoms are more or less the same that you describe (changing among OSs prevent the correct working of net card). Maybe it afects other cards too...
A simple workaround [can be found here]("http://gentoo-wiki.com/HARDWARE_RTL8168#Troubleshooting"). Simply enable "Wake-on-lan after shutdown" on MS Windows device manager.

I helped a user that had an issue identical of what you describe at your [post  #23]("http://ubuntuforums.org/showpost.php?p=2830144&postcount=23"), where avhi-daemon ignores dhcp IP assigned and set to 169.254.X.X. That not being on the same subnet that the router, make connection unavailable.

You do have a very quirky issue, and it does appear to be at a low level.
I'm not sure if I caught what hardware you are running, 
I have a thought, perhaps its down to IRQ sharing.  This is in the BIOS of your computer.
Just to explain a bit more.
When you boot your pc, all the hardware, or at least the important stuff gets assigned a number, also called an interrupt.  There were limitations in the past, and often devices used to share the same IRQ numbers (modems would be on IRQ 3 or 4, etc) 
If you can get into the bios of your machine, normally by pressing F2, or insert, or enter when it 1st switches on (right at the start) when the keyboard lights flash once.
There are many different settings in the bios and you can rather stuff up your machine if you change the wrong one, but you are looking for a list of PCI devices with probably IRQ11 or AUTO after them.

I'm thinking that windows may be better at managing these resources than perhaps ubuntu is?

If I'm wrong on this (and, it's a bit of a long shot to be honest)

1)You have a wired connection to a computer that is dual boot with windows.

3) When the problem manifests itself you cannot "ping" your router (192.168.1.1) if you can't ping the address, you won't get onto the internet.
