### Post by QTRFy on 2007-11-01
This maybe a FireFox problem.  From:

[http://kb.mozillazine.org/Error_loading_some_websites](http://kb.mozillazine.org/Error_loading_some_websites)


The problem may be with IPv6 ("Internet Protocol version 6"). To disable IPv6, change the preference network.dns.disableIPv6 from "false" to "true" . Here are the steps:

   1. Type about:config in the address bar, press Enter.
   2. Find network.dns.disableIPv6 in the list.
   3. Right-click -> Toggle.
   4. Restart Firefox/Mozilla Suite and try again. 

If this doesn't work, re-enable IPv6 by resetting the preference to "false".

### Post by JohnSGruber on 2007-12-16
I had the same problem with Windows XP. The difficulty seems to be with the Fidelity domain name servers for retail.fidelity.com and their interaction with IP version 6 AAAA records. If your Ubuntu system doesn't do IP version 6 (the newly developing version of the internet protocols) you shouldn't often have this problem, but see below.

I was able to recreate this problem on gutsy by adding an IP V 6 address with global scope to my ethernet interface, Otherwise I only had the problem with Windows XP. If I made the change to firefox about:config network.dns.disableIPv6 suggested by QTRFy I can get to the site even with this new IP V6 address added.

Note well that after addressing the problem by doing what is necessary to disble IP V6 (depending on your version of your operating system, or after changing the Firefox item) you must wait for bad information cached in the appropriate domain name servers to expire before the fix will take effect. With either fix you can still have the problem if you are sharing a domain name server (for resolving domain names for you) with someone who is sending IP V6 queries for myfidelity.members.fidelity.com or myfidelity.members.retail.fidelity.com as that will put the bad information into the shared domain name servers' cache for you and others to get as well. I use my adsl modem for resolving domain names and simply turned it off and on when I tested in order to empty the cache. Other people may be using a domain name server provided by an ISP and will simply have to wait.

I was able to resolve the problem on XP by removing the IP V6 driver. I'll try QTRFy's firefox change on XP, too, not that I'd expect anyone here to care about that situation.

Gutsy normally seems to try to avoid sending IP V6 queries (AAAA queries) even though IP V6 is installed by default because it checks to see if an interface is configured with a more global scope IP V6 scope than LINK. Windows Vista probably avoids the problem in a similar way.
I don't know about older Ubuntu versions, but I suspect that they may issue the queries that are problematic for fidelity's domain name servers more easily than gutsy does.

I also suspect that other Ubuntu programs will run into this problem depending on how they look up the IP addresses. Ping seems to behave like firefox with the network.dns.disableIPv6 true fix in. I suspect it does an old-style lookup.

You may be able to circumvent the problem by waiting until the appropriate cache is empty. Then from a terminal window ```
 ping myfidelity.members.fidelity.com
```
If you have waited long enough you should see from the top line of the ping that an address has been found for it (no ping responses will be seen, probably due to fidelity security policy--that's ok). That ping command puts good information in the cache. After doing this, if I moved fast so the cached data remained unexpired, I could bring up the website without applying either fix above.

No matter what I do with IP V6 I can get to EERIE radio, so I'm guessing that it is a different problem. Sorry cygnis1.
