### Post by p_quarles on 2007-07-21
Brief answer is that packages (via the magic of the package management software) get their various parts installed in the correct places in your filesystem. 

With binary files (and I haven't seen many of these, honestly), you would have to deal with that yourself, as well as making sure the code fit with the architecture of your system, verifying correct installation, giving the executables permission to run as such, and (if needed) making menu items. Source code is actually a *lot* easier to deal with.

### Post by p_quarles on 2007-07-21
Actually, most source code is packaged as a tarball (archive of directories), meaning that in the vast majority of cases, no moving of files is needed. Plus, the "make install" command can be configured by the programmer to take care of file placement, much like a wizard.

I've never actually gotten any pre-compiled binaries. My point about them is that every system architecture reads binary code slightly differently. High-level programming languages like C or Perl, though, are exactly the same on every platform, meaning that your compiling program is much more likely to create the correct binary code.

My other point was that --unlike in Windows-- the executable status of a file in Linux isn't dependent upon the file extension name. I.e., there's no equivalent for ".exe". In order to make something executable, you have to go through a process that's every bit as complicated as compiling from source.

So, I would say your list is mostly right, I would just reverse the first two. I'm sure it's possible to download and place binary files, but my understanding is that the order of 1s and 0s is so machine-specific that it would be be a huge headache getting it to work unless it was compiled on a machine nearly identical to your own. But, IANAP(rogammer).

### Post by p_quarles on 2007-07-21
On your first point, yes, you can tar any directory structure. If I wanted to give you my installation of of Apache, say, I could tar it, send it over, and you could untar it and have the same exact installation. That just doesn't guarantee it would do anything, and it would probably be buggy if it did (mainly because of it how it interacts with other applications and daemons on my system). 

The absence of default executables also makes linux a lot more secure -- i.e., if you get an e-mail that disguises a script as a .jpg, clicking on it won't do anything. 

To my understanding, you're absolutely correct to say that packages are basically binary files. The main difference is that the package manager knows quite a bit about your local system, and is able to tweak the configuration of those binaries to work like they're supposed to. 

archive = not necessarily compressed
tarball = one particular and very efficient archiving protocol (tarball is the nickname for the files; tar is the name of the protocol)
package = quite a bit more difficult to put together than most archive files, but yeah

Conceptually, you're right, these things are all pretty similar. The practical difference is that they all use different software procedures (i.e., you can't unzip a .tar.gz file, and you can't dpkg a set of binaries).
