### Post by macogw on 2006-12-21
> **Pobega said:**
> Where did you learn to use make, I need to be schooled on that function.

To build from source, you should read the readme first, but usually you just cd to the directory where the source is, type "./configure" and let it do that.  Errors? fix your dependencies.  No errors? Type "make" let it go then "make install" and finally "make clean" cleans up temporary files.

Trying to get my wireless card to work with WPA encryption was an annoyance, so I learned from that, but I don't know what taught me the most.  I don't think I really know a whole lot right now.  I just know a few tricks, part of which is from reading and seeing others' suggestions in here all the time.  I only know how to fix stuff that pops up when you start out.  For any new problems I run into, Google!  But everything's rather stable so I don't really have any issues once it's going.  Maybe the biggest thing was attempting (and failing) to dual boot my mom's computer.  For some reason, Grub thinks Ubuntu is on hd1 and the supposed-to-but-not Windows is hd0.  If I leave the Grub menu as that I get "OS not found".  So I had to learn to boot from the live cd, mount my drive, chroot in, and fix Grub.  I've had to do it a few times (any time a new kernel is in the updates it has to be done.  I'm glad my mom didn't install the updates because she would have been stuck).

To above people:  I <3 vi.  It's all I use to edit system files now.

### Post by Littleweseth on 2006-12-22
THe hardest lesson so far : Do not ever, ever, ever, ever, ever attempt partition wrangling when you've just gotten up! It's worse than a dozen Chuck Norrises inside your hard drive (Chuck Norri?)

You'd be surprised how different mkfs.ext3 /dev/sda1 and mkfs.ext3 /dev/sda2 really are :) Also don't try mkswap /dev/hda1 (interesting effects) or chmod +rx -R /etc (messy and painful to clean up, creates a million security holes.)

The root lesson : phear the root terminal and think before you whack return :p
