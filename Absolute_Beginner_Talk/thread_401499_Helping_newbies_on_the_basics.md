[quote="thenme"]As a complete noob I would just like to say, anything that could be developed to help me get a better grasp on the inner workings of GNU/Linux (ubuntu) would be of great help.[/quote]

That brings up another question: how detailed should the explanations be? Should they explain everything that you're doing to fix your system, or just tell you what to do?

Example, should it say:

> Open a Terminal (Applications >> Accessories >> Terminal) and type:
sudo rm -r a_folder
The "sudo" allows you to gain root (or Administrator) access. Since you don't have permissions to delete this folder, this is required.
The "rm" stands for "remove"
The "-r" means recursive, which is "remove this folder and everything underneath it"
The "a_folder" is the name of the folder you wish to completely remove.

or:

> Open a Terminal (Applications >> Accessories >> Terminal) and type:
sudo rm -r a_folder

Ideally (IMO), it would initially display the second one (since many people just want to get their computers working) and have an arrow below it for "Explain What Exactly I'm Doing" for people that wish to grasp more of what's going on.

And I completely agree with thenme about the simple titles. Users should literally be able to step through the guide, one option at a time (when possible, a simple "yes" or "no") to find their answer as quickly as possible.

EDIT: I'm getting ahead of myself again, but I think it would also be cool to have the guide adjust based on their hardware. Many users don't know the exact specifics of their graphics card, modem, etc. When it gets to a point that, for example, relies on knowing their graphics card, the user could download a simple script that would pass it on to the guide and direct them to the right page (the Device Manager can seem pretty confusing to people at times)

Hey guys, thank you for so many replies.
Well, my idea kind of splits into two - decision trees and explanation graphs.
An example explanation graph would be something that would show how the kernel integrates into the system (the kernel, filesystems, modules, etc.), which would allow the user to understand what they are about to do more clearly etc. 
Another one is decision graphs - simply link people to a problem and then they can find out what is really wrong. 

This might seem just too much to do sometimes for a small number of people around, but my idea is to create something like a Wiki where users could actually see a graph, and then simply post comments and questions and so on. This would allow them to go through the problems and find all they need. Every command would be described in detail and so on, so the newbies get every single thing instead of us having to repeat it all over and over to them again!

Basically, my idea for a site is to have a customly built wiki (I could start working on it, if anyone is willing to support me). An example of what a developer would do:

1. Create a decision tree
2. Add the keywords and link them to both wikipedia and the wiki itself (to find related trees and diagrams and questions).
3. Publish it

What the user would do is:
1. They come to the site
2. Search for their problem
3. Find their problem and look at the decision tree
4. Something in the decision tree does not match up, and the user is not very happy about that (for example, they are in a feedback loop, or they have some kind of extended problem).
5. They can simply click on that specific point, and add their comment.
6. Somebody answers their question/comment and then the developer can modify the decision tree (or the user ;-)), and here you have a very in-depth decision tree.

This would mean that the developers and helpers in Ubuntu support places such as forums and IRC wouldn't have to go through the same problem over and over every day, asking people same questions such as "Did you install build-essential?" and so on.

Same decision tree would be nice for a type of flow chart, where it would show instructions to users and they could comment there too (and as well develop into another decision tree?).

So if anyone is up for the idea, I'd be happy if someone would start the design of how the application/site would work itself, so it would be much simpler to implement and understand what is happening.
