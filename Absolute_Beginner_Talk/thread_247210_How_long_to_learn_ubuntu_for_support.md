I've been using Linux exclusively for about 9 months, with very little experience before that.  I'd be able to help someone setup a new system and configure it, and I'd be able to troubleshoot problems for a casual user.  But, I would not be able to support complex, low-level problems.

### Post by jimcooncat on 2006-08-30
Pick yourself a niche, and learn that. Try to find a subject that is interesting to you and covers an area you suspect will be popular with your users.

For me, it was ssh to securely administer remote computers. For a secretary, it might be OpenOffice styles and templates. Telecommuters, freenx. Stay away from overly complex subjects at first, like setting up postfix or web servers.

And get the basics under your belt as well. Knowing the layout of the file system and working with user accounts, groups, and permissions will make you a valuable resource with little learning on your part. Get to know how to keep the system updated and how to work repositories.

Get some backup support for those things not under your belt yet. The forums and IRC are nice, but if you're doing this for a job then you could use some commercial support for the questions you can't answer. 

Cruise around the forums and look for topics which match your target of interest. Lots of questions you can take a stab at answering yourself after doing some research: google, ubuntu docs, /usr/share/doc/*, man pages. By helping others with their real-world problems, you learn a lot (and give back to the community).

### Post by nrwilk on 2006-08-30
> **ShagzModo said:**
> -Try to use apt-get instead of the symaptic package manager for installing.

Actually, if you're going to do that, you might as well use aptitude.  It's better than apt-get for one very important reason.

With apt-get, if you install some software which requires dependencies, apt-get also installs those dependencies automatically.  aptitude does the exact same thing.


When you decide that you no longer want that software, and you use apt-get to remove it, apt-get leaves the dependencies on your system even though no software needs them.  However, aptitude always remembers which packages installed what dependencies, and if you remove a package that depends on several dependencies, it will remove them as well (as long as no other software depends on them).

The catch is that aptitude only can do this with packages you installed with aptitude.  It cannot do it with packages you install with apt-get or with synaptic.

Also, aptitude has a nifty text-based package manager, equivalent to synaptic, you can run by doing:
```
sudo aptitude
``` 
Also, aptitude's commands are virtually equivalent to apt-get's, so anytime in the forums when you see a person saying to do this:```
sudo apt-get install <package_name>
```instead, you can just do this:```
sudo aptitude install <package_name>
``` 
I do agree, though, that you should learn to use the command line to install your software, and not a GUI package manager.

[Here]("http://ubuntuforums.org/showthread.php?t=164082&highlight=aptitude") is a thread where Aysiu gives links to other threads where people outline the superiority of aptitude in greater detail.

Hope this helps!
