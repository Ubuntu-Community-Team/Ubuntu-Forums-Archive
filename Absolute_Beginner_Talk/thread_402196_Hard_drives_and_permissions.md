### Post by LaurelLynn on 2007-04-05
Ok,

The UUID is throwing me for a loop. Normally fstab entries start something like:

/dev/the-device-name  /the-mount-point

Note: Anything you do from this point runs the risk of cutting you off from the drives completely so backup fstab and keep a Live CD handy (In fact backup up all the drives is a good idea too).

You could try changing the home partion first, to be on the safe side:

Edit fstab and comment out the UUID entry for home. Then add

# New Entry for /dev/sda2 :
/dev/sda2  /home ext3 defaults 0 2

Run:

$ sudo umount /home
$ sudo mount /home

That should work.

If it does work for home, you could try it for root, but you won´t be able to umount /root, you´ll have to reboot  (have Live CD ready in case it doesn´t work).

### Post by LaurelLynn on 2007-04-05
Ok thenme,

your main problem seems to be that you can´t run any commands because you can´t read  ¨/¨.

Two ways around that:

1.  Reboot and choose the Recovery option from the Grub menu.  That will get you a root prompt ¨#¨ from there you have permission to do anything **unless** there is an error with the drive and ¨/¨ is not mounted properly.

Mount permission take precedence **over** shell permissions. Since most command are simply names, for programs in the bin directories of  ¨/¨. Without being able to read that partition, even root can´t run most commands. 

Which brings us to solution 2.

2. Reboot to Live CD. From there you can use all the Gnome GUI tools to mount the partition with write access. From it you can use sudo to run the commands you need to fix your harddrive install.

Now for that, I suspect an error is occuring when it goes to mount the root partition.
Because your fstab points to the UUIDs listed, those strings of numbers have to be absolutely correct. Right now, you can get to a command prompt, because the shell is in the kernel which GRUB is loading.  All the other commands are on the drive mounted (or not mounted) by fstab, which seems to have failed.

That is why in my previous post I recommened changing the fstab entry to point to the device name instead of the UUID. The comments in your fstab already state:

/dev/sda1    is supposed to be mounted at    /
/dev/sda2    is supposed to be mounted at    /home

The usual entries for that should be:

/dev/sda1  / ext3 defaults,errors=remount-ro 0 1
/dev/sda2  /home ext3 defaults 0 2

The use of UUID is not specifically wrong, but I have only seen it used once or twice in over 20 years. I have no idea how your system got setup that way to begin with, but it could be the reason for the problem (especially if  you replaced your hard drive with a new one, UUID is drive specific).

Hope that helps.

Laurel Lynn

### Post by LaurelLynn on 2007-04-05
Dear thenme,

The /etc/fstab entry just  tells the Ubuntu installed on your hard drive where and how to mount drives.

When you boot from the Live CD it has it´s own method of mounting, so it is not effected by changes to the hard drive /etc/fstab. So the Live CD will still be able to read it and run commands no matter what is in the hard drive fstab.

The place where there is danger is if you start changing the options, or accidentally use the device name of another partition.

If you make a change to fstab from the Live CD, reboot, and it doesn´t work, you can go back to the live CD and change it back. So long as you remember what was already there, which is why I always comment the first entry out instead of deleting, you can **put** it back.

Now if you tried what I recommended on the /home partition, and it did work. Then the same thing should work for the / partition.

If not I´m running out of ideas, and unless someone has another you will probably have to reistall Ubuntu.   You can do so without erasing the home partition, but just in case, you´ll want to back up the home partition so that you don´t loose any user files (i.e. your stuff).

Laurel Lynn

### Post by MattM32 on 2007-04-08
I had the same problem and solved it too. But my solution was a bit different.  I found [this]("http://lists.centos.org/pipermail/centos/2005-September/051844.html") on the web.  and basically change the line in fstab to this:

```
/dev/sda1	/media/x_drive	vfat	noexec,rw,gid=<group name>,umask=002	0	0
```

Where <group name> was my user name.  I don't quite understand the umask bit, but its supposed to allow write access to the group.  It works.
