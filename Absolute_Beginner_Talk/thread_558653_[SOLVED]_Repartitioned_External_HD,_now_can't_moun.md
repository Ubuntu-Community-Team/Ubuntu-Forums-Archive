### Post by jw5801 on 2007-09-24
Hiya!
You don't happen to have a Windows install lying around that you could boot it with do you? Try mounting it there and see what happens. Otherwise we can follow the implied instruction and run a check on it:
```
sudo fsck /dev/sdb1
```EDIT: On second thoughts, I'm not entirely sure that fsck has ntfs error checking support. Might have to search for a Linux one. Or use Windows.

Also if you're mounting via the command line, the mountpoint must already exist. So try:```
sudo mkdir /media/backup
sudo mount -o force -t ntfs-3g /dev/sdb1 /media/backup
```
As for the order of the command it's just convention to use *command options target result* bash doesn't care either way. Also "force" is an option specific to the ntfs-3g driver "man ntfs-3g" will tell you about it.

As for your fstab, since you don't have a row for this filesystem already you'd need to create a new one, something like:
```
/dev/sdb1 /media/backup ntfs-3g user,auto 0 2
```at the end should do it. Make sure, once again, that the /media/backup/ directory exists else it will have nowhere to mount to.

If you still get errors with the ntfs-3g driver then try the default driver (use the same command but drop the "-o force" and use "-t ntfs" instead of ntfs-3g).

### Post by jw5801 on 2007-09-24
> **swarup said:**
> nouser,defaults,noatime,auto,rw,dev,exec,suid 0 2

These are all just mount options.
nouser: says that the drive may only be mounted and unmounted by root
defaults: uses the default options set at the top of fstab
noatime: don't store the last access time (atime) tag on all files, this makes reading files faster as a write does not need to be done after every read
auto: mount whenever detected
rw: read and write capable
etc.

### Post by jw5801 on 2007-09-24
```
/dev/sdb1 /media/backup ntfs-3g **user,auto** 0 2
```
If the above is your line in fstab, the bold bits are the mount options. The format runs as follows:
*device mountpoint filesystem options dump fsck*
Ther last 2 fields (0 and 2) in this case, determine how a couple of boot processes look at the filesystem, pretty much everything will use "0 2" and the root partition uses "0 1"

### Post by jw5801 on 2007-09-25
Sorry, you were meant to replace the "drive:" with whatever drive Windows saw the partition as, so in this case E: The "/F" is simply a commandline option that tells CHKDSK to fix any bad sectors it finds. Windows uses a "/" to denote commandline options where Linux uses a "-". Therefore the command is:```
CHKDSK E: /F
```
As for the reported capacity, 21,048,483,840B = 20,555,160kB. This is because data volumes work in base 2 rather than base 10 like everything else. So the prefix kilo implies 2^10 (1024) rather than 10^3 (1000) as is normal. That's a bit of a mathsy sciencey explanation so I hope it makes sense. So yes you are correct, running CHKDSK without explicitly directing it to the partition you want it to look at will default to checking the root drive of Windows, ie. your internal drive. Point it in the right direction and we may see something more conclusive!
