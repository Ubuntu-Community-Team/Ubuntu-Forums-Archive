### Post by Morbius1 on 2012-04-11
You are getting advice that's pulling you into different directions so I would just like to post some observations:

** There's nothing wrong with "wins support = yes" if what you want to do is set that Linux machine up as a WINS server. But if you do:

* It must always be running when the others are running because it's the mechanism that's being used by everyone else to resolve names to ip addresses.
* It must have a static ip address.
* All of the other machines will have to be modified to point to that WINS server as part of it's networking scheme.

** Name Resolve Order

* The default is lmhosts wins host bcast
* lmhosts and host either don't exist or have nothing samba related in them by default so they are ignored.
* wins doesn't exist on the network by default either so it should be ignored but there may be a problem here. 
* bcast is the only one that works by default. Both Linux and Windows uses bcast but it's listed last in order in Linux. Have a hiccup in wins ( where it actually tries to leave the lan in order to resolve names ) and it never gets to bcast.

I usually recommend placing bcast first and placing wins last but that depends on how you are setting up your network.

### Post by ewangr on 2012-04-11
Well, I "fixed" things, but not in a manner I would consider particularly friendly to the "non-technical".

Step 1 was to make the changes suggested in the preceding message (turn off the WINS server and set up bcast as the first option). Step 2 was to go ahead and set fixed IP addresses for my Ubuntu machine and the Windows machine downstairs. Step 3 was to use the fixed IP address in my /etc/fstab. Step 4 was to chown my /media mapping from root to myself. I had set 777 permissions, but realized the owner "conflict" was probably trumping the directory permissions.

That has finally given me a drive that loads and stays loaded. It "shouldn't" require someone to do this if it worked with DHCP on both machines under Win 7, but I'm more concerned about making it work than fixing the underlying problem at this point.

My suspicion is that something in the stack was not liking my use of the 192.168.10 subnet instead of 192.168.0, and was trying to send the resolution to the latter. Since the cable modem is setup to that subnet as it comes from Time Warner, the resolution call goes to the cable modem, which passes it for a DNS call, and then when it fails passes me to the RR "suggestion" service. When I hit the next call, it is still "timed out", and goes to the router instead, which then sends me to the correct machine and drive. The variability would then occur based on how quickly I was trying again after a failure.

I'm thinking this is enough of an edge case that trying to debug would require a developer with a similar setup, and so is something I don't think would lend itself well to launchpad. But it would be nice if someone eventually addressed this.
