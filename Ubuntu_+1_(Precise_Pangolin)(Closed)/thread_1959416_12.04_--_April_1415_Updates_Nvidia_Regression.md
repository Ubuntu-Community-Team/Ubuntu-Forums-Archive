mkdir $HOME/Downloads/nvidia-drivers && cd $HOME/Downloads/nvidia-drivers && if [[ $(uname -a | grep "x86_64" | wc -l) == '1' ]]; then wget download.nvidia.com/XFree86/Linux-x86_64/295.20/NVIDIA-Linux-x86_64-295.20.run; else wget download.nvidia.com//XFree86/Linux-x86/295.20/NVIDIA-Linux-x86-295.20.run; fi

sudo service lightdm stop

sudo chmod +x NVIDIA-Linux*295.20.run
sudo ./NVIDIA-Linux*295.20.run

sudo reboot now

That message is normal. Have you rebooted? NVidia installer builds another kernel module. It's better to reboot and load it than to simply restart lightdm.

Also: Make sure you don't have nouveau loaded. The following command should output nothing.
```
lsmod | grep -i nouveau
```
Check if your nvidia kernel module is loaded. The following command should output "nvidia" and digits. Not nvidiafb or anything else...
```
lsmod | grep -i nv
```
...and that the kernel module is the same version as the driver (in this example, we downloaded 295.20). The following command should show you a line with the module version.
```
modinfo nvidia
```

If still you're stuck at console, run dmesg and cat /var/log/Xorg.0.log and paste the error message here. Unless you card does not support a modern driver and needs NVidia 173, I can think of no reason for it to fail.

Ok, NVidia 295.40 can lead to problems in some hardware models (as pointed out by Dino and the article on Phronix). But we do have people with other hardware that had stability / performance issues too since Apr. 14/15th. Therefore, they might (or not) be affected by NVidia (depending on the hw - we don't have a clear and complete list of affected hw) and/or some of the other GL related updates that day.

In my case, reverting to 295.33 and 295.20 didn't fix it. Some people mentioned improvement but not to normal status. Other believe it fixed it for them. We don't have enough people testing it.

Following up an advice by mc4man (thanks!), I have downgraded some nux packages and my PCs feels a lot snappier and reliable, even on NVidia 295.40. I do not advice anyone to go for it though, one can easily break dependencies.

I think our best approach here would be if people could revert to a previous NVidia driver release, so we can see if it fix everyone's problems. Because if it does not, as was the case in my PCs, we'll have enough reason to investigate the other packages. Otherwise, if we trust this is exclusively NVidia-related, we are assuming a risk of receiving an updated driver from NVidia (which is needed, of course), but also not dealing with some other (potential) GL-Related bug prior to release.

Until we investigate it properly and gather enough information
to understand what are the causes for all the similar reports (compiz/unity/nvidia/OpenGL/acceleration/3D/etc) we received in a 3-day interval (which is highly unusual), it makes total sense for us to debug it together in a single thread.
