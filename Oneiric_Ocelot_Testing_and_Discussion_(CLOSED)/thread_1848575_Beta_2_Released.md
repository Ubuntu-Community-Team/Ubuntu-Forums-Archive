Root should not be enabled and a reboot in case of issue is not uncomfortable.
If you want more info take a look at this thread on askubuntu:
[http://askubuntu.com/questions/16178/why-is-it-bad-to-run-as-root](http://askubuntu.com/questions/16178/why-is-it-bad-to-run-as-root)

This post sums up main points against root enabled:

> Running as root is bad because:

Stupidity: Nothing prevents you from doing something stupid. If you try to change the system in anyway that could be harmful you need to do sudo which pretty much guarantees a pause while you are entering the password for you to realize that you are about to make a possible big/costly change.

Security: It has been mentioned already quite a few times in this question but basically it's the same thing, harder to hack if you dont know the admin user's login account. root means you already have one half of the working set of admin credentials.

You don't really need it: If you need to run several commands as root and you are annoyed by having to enter your password several times when sudo has expired all you need to do is sudo su or sudo bash or whatever your fav shell is and you are now root. Want to run some commands using pipes? Then use sudo sh -c "comand1 | command2".

You can always use it in the recovery console: The recovery console allows you to try and recover from doing something stupid or fixing a problem caused by an app (which you still had to run as sudo :)) Ubuntu doesn't have a password for the root account in this case but you can search online for changing that, this will make it harder for anyone that has physical access to your box to be able to do harm.

### Post by effenberg0x0 on 2011-09-24
About the root discussion:

I see that most people usually create a single user in Ubuntu PCs. I have seen that even in servers. I don't think that is wise.

I have an old habit of creating a secondary account in every machine as a safety measure. If, for some reason, I can't login with my primary account, I can generally use the secondary one and fix things. This secondary user account obviously is a sudoer, so he can fix everything. No need to reboot / go to recovery prompt or have a root account enabled. 

I can't remember a situation in which I had messed up so much that this strategy was not enough to fix things or in which my only way out was to get to recovery or chroot from a live disk, etc. I run sshd (key+password) on desktops, on unlikely ports enabled only to the local network, so even if console gets screwed somehow or I lose physical access to the machine, I can use primary or secondary login from any other PC, smartphone, etc.

As long as one can manage to keep a secondary account safe (monitor  login attempts and auth results vs date-time, basically), I believe it's  a good safety practice.

As another security precaution, you can have a script running chown/chmod in home, etc, sbin, other vital folders every n minutes/hours, etc via cron. I have fix_permissions.sh checking / fixing weird permissions every 60 minutes. 

Another strategy is to keep a very small Linux OS in a partition in the HDD. It may help a lot if the main OS becomes unbootable, which honestly is incredibly rare. Anyway, you can log on it, mount the other partitions, fix stuff, etc, like you would using a Live CD/USB. I do that on servers. 

OBS: In the case of desktop PCs, some new motherboards, like ASUS, make it even easier. They offer the possibility of quick booting to a Linux distro installed in its flash memory. You can get to a root console from there, mount everything, chroot, etc. 

Being it a distro installed in HDD partition or in motherboard flash, it is a good thing to prepare this distro, customize it, create some quick crisis recovery scripts, etc, so its ready when you need it. 

Anyway, if one implements all or some of these measures, I think it is virtually impossible to get locked out the system these days.
