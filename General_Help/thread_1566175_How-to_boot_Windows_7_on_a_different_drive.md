### Post by Jordan Uggla on 2010-09-02
Windows 7's bootloader does not assume that it's being loaded from the first drive like Windows XP and previous did. Because of this it doesn't need drivemap, at least not normally.

For whatever reason, it seems like your Windows install is an exception to this general rule, and due to this bug: [http://bugs.debian.org/cgi-bin/bugreport.cgi?bug=584846](http://bugs.debian.org/cgi-bin/bugreport.cgi?bug=584846) drivemap in grub 1.98 is not compatible with Windows 7. This has been fixed since the release of 1.98, and the fix is available in Debian unstable and also Maverick.

If you are willing to try installing grub-pc and grub-common from Maverick, [http://packages.ubuntu.com/maverick/grub-pc](http://packages.ubuntu.com/maverick/grub-pc) [http://packages.ubuntu.com/maverick/grub-common](http://packages.ubuntu.com/maverick/grub-common) , then I expect you will be able to boot Windows using this entry in 40_custom:
```

menuentry "Windows 7 (loader) (on /dev/sdc1)" {
	search --no-floppy --fs-uuid --set d06a05bd6a05a0f8
	drivemap -s (hd0) ${root}
	chainloader +1
}

```

### Post by Jordan Uggla on 2010-09-04
> **retooh said:**
> Windows boots fine when the drive it's on (sda or sdb) has boot priority in the bios, so not sure if drivemap is even the answer to my problem.

Most BIOSs present the drive that you boot from as the first drive, no matter where it is physically. The only important difference I can think of between chainloading Windows from GRUB on another drive and booting the Windows drive directly is that the Windows drive will be "second", at least from the bootloader's point of view. That's why I expect that it will work with drivemap.
