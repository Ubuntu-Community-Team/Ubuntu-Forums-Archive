### Post by QIII on 2018-05-11
Is that the purpose of your testing, or is the VM where you plan to store the files permanently?

I ask because a VM is ephemeral.  You can break it easily, decide to get rid of it, etc.

Your "permanent" files would best be saved in a shared folder you have access to from the VM.

I have a Windows VM that I use for Windows development for my clients that use Windows.  All the "hard" files are kept outside of the VM.

### Post by HermanAB on 2018-05-11
Install Guest Additions in the VM.  With that, you can readily access files on the host.  

Leave your files on the host.  Don't save data in the VM.  That way, you can replace the VM at any time, without having to worry about the data.  It also makes it easier to backup your data, since you only have to backup the host.  It also ensures that the VM will not 'grow' and use more and more disk space.

### Post by QIII on 2018-05-11
^^ A bit more explicit explanation.

I have a Windows app (which I developed and maintain) that I use for my client marketing purposes.  The source code is on the host.  The mySQL database is on my server, which I reach with a .NET add-on for ssh.  Nothing is really in the VM but Visual Studio and the .exe after I compile it.  (I use KVM, by the way.)

Windows breaks?  Meh.  Blow the VM away and reinstall or go back to a snapshot.  Nothing lost.  I back up the host regularly.

As a direct answer to your original question, you can add the Guest Additions, add a shared folder, put the files you want to transfer there from your host and pull them out to your VM.  You could add the Guest Additions and set your VM up to use USB and transfer files that way.  You could ssh from your host to the VM.  Basically, you can use all the methods you would use to transfer files exactly as you would if they were two physical machines.

But it's still probably not a good idea to do it that way.
