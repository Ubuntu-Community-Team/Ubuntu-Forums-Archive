### Post by JKyleOKC on 2011-09-06
If you can find a copy of it anywhere, read "Software Tools" by Brian Kernighan and P. J. Plauger. It may be out of print, though -- and if you find something by the same authors called "Software Tools in Pascal" run, don't walk, away from it!

The original book is a concise introduction to "the Unix way" of programming, and Kernighan (with Dennis Ritchie) co-authored the original book on "The C Programming Language." In "Software Tools" all of the code is in a language called RATFOR, for "RATional FORtran" and the reason is that, at the time it was written, Fortran was the lingua franca of the entire programming community. RATFOR, however, converts Fortran itself into a sort of C clone, and all of the samples can convert into C almost directly. It's fairly obvious, in hindsight, that K&P translated the original C code into RATFOR to make the examples available to a wider audience!

The Pascal version, however, changed everything over to Pascal syntax, which is just similar enough to C to make it almost impossible to keep things straight when you try to work in both at the same time. For your purpose of getting a better grasp of C, stay as far away from Pascal as you can (and I say that as a person who currently programs almost exclusively in a Pascal dialect, after years of working with C).

### Post by JKyleOKC on 2011-09-11
> **RyanRahl said:**
> I was wondering what are the disadvantages of learning c++ without learning c first. Is it that hard to program in c if I learned in c++? And why would I program in c instead of c++?I'll answer the second question first: The main reason to learn C for anyone who wants to debug or modify Linux system programs is because that's the language in which most of them are written.

Now to the original one: C++ is really sort of "C with training wheels" and hides a very large part of the language's simplicity within its class libraries. C itself is just one small step above actual assembly language, which means that you are much closer to the actual hardware. For instance, it has no "string" type. Instead, it uses an array of individual characters to take its place, and a pointer to that array with which to access it. The effect is the same, but the array actually duplicates what happens right down at the machine level. The "string" type of C++ hides that implementation detail.

There's nothing wrong with such hiding, once you're familiar with what's happening behind the curtain. Object oriented programming has a lot of advantages when you're creating applications. However when you're at the system level, way below where applications run, you need to be as close to the hardware as you can reasonably get.

And yes, it would be hard to grok many snippets of C code if all you knew was the abstraction that C++ provides. For instance, the heart of a "copy" program in C is the single line ```
putchar(getchar())
```but it must be nestled inside a function named "main" and the source must also include the header program "stdio.h" because the getchar and putchar functions aren't defined by the language! The same program in C++ would probably be quite different, depending on how many of the available bells and whistles you wanted to include.

EDIT: Note that I greatly oversimplified that code snippet, to make a point. In practice, it's really necessary to use an intermediate variable to hold each byte, so that the program can detect an end-of-file condition and break out of its loop. Nevertheless the idea that I'm trying to express remains true.

### Post by JKyleOKC on 2011-09-12
> **RyanRahl said:**
> Would it be correct to assume that software programmed in c is generally faster than software programmed in c++ or even python beyond that?Yes, it would, except for one thing: it's possible to write bad code in any language. A **well-written** program in perl, python, or even bash, can easily be faster than a poorly-written one in C or assembly language.

At one time, it was standard practice for systems programmers to do everything in assembly language to achieve the greatest speed. Finally someone managed to prove that well-written code in a language such as C could out-perform the **average** assembly language version, simply because the additional detail required when coding in assembly language made it more difficult to make the code as good as the higher language could achieve.

With today's hardware, the question becomes much less important in many situations. I was amazed to discover how much of my favorite Linux distribution is implemented through bash scripts, for example. The whole "Unix way" that I've mentioned elsewhere is based on writing tiny programs that do just one thing but do it extremely well, and then pasting these programs together via pipes or shell scripts to make them parts of bigger applications. It works, and saves an immense amount of time in getting from an idea to a functioning program.

EDIT: By the way, bash is every bit as much a "real programming language" as C++ or C. Check out some of its reference manuals if you don't believe me. It's what the theory folk call "Turing complete" which means that it can do anything that any other programming language can do, but most of the features that make it so are not at all obvious in its everyday usage! Don't sell it short...
