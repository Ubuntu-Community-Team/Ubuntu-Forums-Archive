### Post by CatKiller on 2020-01-08
> **joebuntu99 said:**
> Issue started after doing "sudo apt upgrade" which updated kernel to kernel-lowlatency-5.0.0.38.
I subsequently did "sudo apt autoremove" before rebooting - not sure if this is relevant, I thought apt was "smart" enough to not allow me to mess things up like that.

It is. Your issue is unrelated to the upgrade process, except that you're now using that kernel version.

The message is what it says it is: an indication that there's a bug. Atomic operations should not be scheduled.

Unless you made some configuration change that fiddled with the way things are scheduled - which you could reverse - you can use the Grub menu to boot from an older kernel until you get a new one.

Also it's worth being aware that 19.04 will end support this month. You should get onto 19.10 as soon as you can.

### Post by CatKiller on 2020-01-08
> **joebuntu99 said:**
> If I run the following command will I safely remove the newer kernel?

```
sudo apt remove linux-image-5.0.0-38-lowlatency linux-headers-5.0.0-38-lowlatency
``` 

Yep, that should do it. It may well also remove related kernel packages. 

> Does it matter which kernel is loaded when I do this? (I assume not since my understanding is the kernel is loaded to ram on boot?) 

I don't think it matters, but the older kernel's working better for you, so you might as well boot into that one. 

> I understand I would have to put a hold on the kernel packages to prevent apt upgrading them again. 

That would be necessary for most packages, but the kernel is handily controlled by a metapackage that simply depends on the latest version. If you remove the linux-lowlatency package (or the HWE one, if that's what you're using), you shouldn't get any new ones. Then it's just a case of reinstalling that package once there's a newer, hopefully non-problematic, version. 

It might also be worth testing the -generic branch of that version to see if it has the same issue for your bug report. The lowlatency kernel is more likely to be running atomic operations, but is only used by default with Ubuntu Studio, so it might have had less testing.

### Post by CatKiller on 2020-01-20
> **gh0zt362 said:**
> Ok , I didn't know if the system would download and default to an older kernel if you removed the current .  This bug we're discussing is the kernel not loading modules correct ?   

I don't think it is a kernel bug, actually. Systemd would be my first guess. It's just exposed by some change in that new kernel. I've been using the lowlatency kernel for years with no issue; I was using the 5.0 branch when the thread was started, and I've since been bumped up to 5.3, since I'm using 18.04 with the HWE stack. I never experienced the issue that you both had, which makes me think it's a different component of your 19.04 installs. 

>  If this is the case I manually download a previous kernel then what ,  I've never installed a kernel by itself . I open a terminal in the download folder than ?

Nope, none of those things. You use your package manager to install software. You might find it easiest to install Synaptic if you don't have it already. 

The easiest way would be to install the linux-generic metapackage. That will give you the latest kernel but without the lowlatency stuff. From the error message that seems less likely to trigger the issue.

If that doesn't help, using ```
sudo do-release-upgrade
``` would be the next-best option, to upgrade to 19.10. Once you're on 19.10 you can try the lowlatency kernel again. 

At another point in the release cycle it might be worth messing about with removing the newer version and the lowlatency metapackage to keep yourself on the earlier version, but that will interfere with the upgrade that you should really be doing: 19.04's end of life is on Thursday.
