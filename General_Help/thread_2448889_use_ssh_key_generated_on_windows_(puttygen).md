### Post by TheFu on 2020-08-17
Win10 has ssh built-in.
Use **ssh-keygen** directly just like on Ubuntu.
Then you'll have to manually **scp** the public key file to ubuntu and append it to the **authorized_key** file in ~/.ssh/
Directory and file permissions must be correct on the ubuntu side. Normally, **ssh-copy-id** run from the client handles this, but msft didn't bother providing that command too.

Normally I'd run these 2 commands on the client:
```
   $ ssh-keygen -t ed25519
   $ ssh-copy-id -i ~/.ssh/id_ed25519.pub username@remote
```
Obviously, the username@remote needs to be for local stuff.  ed25519 keys are better.

Win10 also doesn't have **ssh-agent** setup. I don't know if that program was ported too. It will keep track of the time ssh keys were unlocked so you don't get hassled for those credentials too often.

### Post by TheFu on 2020-08-17
On Windows, the cmd.exe prompt should put you into c:/Users/{username}.  Forward slashes replace all \ uses on Windows in every way, so don't worry about usng \ or \\, just always use / everywhere.
All that means that the location of the public key will be  .ssh/id_ed25519.pub on Windows.

I only know these things because a Windows user needed help yesterday and someone else on the meeting knew these things.  Watched the setup work perfect.  The only downside is that cmd.exe doesn't support select/paste like an xterm does.

I don't know if the key can be used by putty or not. sorry. The putty select/paste isn't too bad.
