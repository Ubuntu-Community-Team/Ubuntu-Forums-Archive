### Post by SamD42 on 2009-12-07
I've had this issue on two computers i've installed Karmic on so far.

The only workaround i've found so far is to directly edit grub.cfg

sudo chmod +w /boot/grub/grub.cfg
gksudo gedit /boot/grub/grub.cfg

Then where it says

if [ ${recordfail} = 1 ]; then
  set timeout=-1
else
  set timeout=5

change the first set timeout from -1 to whatever value the second is.

so:

if [ ${recordfail} = 1 ]; then
  set timeout=5
else
  set timeout=5

Then save and exit

This workaround will need to be applied every time update-grub is run.

More info can be found in the grub 2 basics thread.

As a warning /boot/grub/grub.cfg is not meant to be edited, which is why the permissions must be changed before the file can be, be aware of that. I also don't fully understand the potential effects or ramifications of making this particular change to the file, I just know that so far it has worked ok for me. 

I'm still looking for a proper fix myself, and would feel a lot more comfortable about suggesting this workaround to others if someone more in the know could chip in.

### Post by Leppie on 2010-01-15
> **graphov said:**
> This can be also done with less brute force in /etc/grub.d/00_header file and it will be automatically applien on update-grub calls.
it's not really "brute forcing".
commenting out the last if statement in 00_header is a workaround that works and sticks. remember to not comment out the normal "set timeout" command (this would result in what you already have, no timer):
```
#if [ \${recordfail} = 1 ]; then
#  set timeout=-1
#else
  set timeout=${GRUB_TIMEOUT}
#fi

```
after receiving a grub2 update, you can try to uncomment it ;)

### Post by pgolec on 2011-06-15
I had the same problem with Maverick Meerkat 10.10 after deleting old kernels.

This fixed it:

[https://bugs.launchpad.net/ubuntu/+source/grub2/+bug/462888](https://bugs.launchpad.net/ubuntu/+source/grub2/+bug/462888)

sudo grub-editenv /boot/grub/grubenv unset recordfail
