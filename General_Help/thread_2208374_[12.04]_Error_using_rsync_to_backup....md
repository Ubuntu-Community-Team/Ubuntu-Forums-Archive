### Post by papibe on 2014-02-27
Hi again, bc.haynes.

I suspect permission problems here.

Where is the destination? Is it an external disk? your user has permissions to write there?
If by any chance there's a file not owned by you on your home directory, rsync won't be able to create them with other owner than you.

Use this command to find files that are not 100% owned by you:
```
find -not \( -user youruser -and -group youruser \) -exec ls -l '{}' \;
```
Another alternative could be there are files that you own that don't have read permission to even you. Use this command to find them:
```
find -not -readable -exec ls -l '{}' \;
```
This command will also find sym links to files that no longer exits.

Another idea: there's no risk on removing the dry run option. If rsync fails, you just running again. The source remain untouch. Also, actually running it may print the exact files which are giving you trouble.

Hope it helps. Let us know how if that helped.
Regards.

BTW: -a implies -r and -X, so you can just use -avxPhn

### Post by sudodus on 2014-02-28
I suggest that you start with something simple, and make it work. Then make it more complicated step by step until you have a complete backup script.

Change the ownership and permissions of the partition on the pendrive until it works.

For example:

1. Copy a file to the pendrive with the file browser.
2. Copy a directory (with some files)  with the file browser.
3. Copy a file to the pendrive with **[FONT=courier new]cp file target[/FONT]**
4. Copy a file to the pendrive with **[FONT=courier new]cp -p file target[/FONT]** # preserving the permissions
5. Copy a directory (with some files) with **[FONT=courier new]cp -r directory target[/FONT]**
6. Copy a directory (with some files)  with **[FONT=courier new]cp -pr directory target[/FONT]** # preserving the permissions
7. Copy a directory (with some files)  with **[FONT=courier new]cp -av directory target[/FONT]** # preserving the permissions
9. Copy a directory (with some files)  with **[FONT=courier new]rsync -av directory target[/FONT]**
...

Notice the difference between **[FONT=courier new]rsync -av directory target[/FONT]** and **[FONT=courier new]rsync -av directory[COLOR=#ff0000]/[/COLOR] target[/FONT]**

See ```
man cp
``` and ```
man rsync
``` for more details about the command options.

### Post by Dennis N on 2014-02-28
> Can't figure this one out.
Any more ideas? 

1) I would not use sudo with rsync. The man page states it is not needed, so I would try without sudo to see if that is part of the problem. My rsync commands don't use it.
2) If all your files are regular files or directories from your home directory, then options **-rtui** should be enough to back them up and also show what is being copied.

Also, you should know that the source can be multiple sources: 

If D1, D2, D3, are three folders you want backed up,

rsync D1 D2 D3 D4 

will back them up to D4. 

D1/ will send the contents of D1 and not the folder itself.

### Post by Dennis N on 2014-02-28
> I checked and the pendrives are formatted ext4. I used two pendrives.

One last thought:

After formatting them ext4 (with gparted I presume?) these pendrives would be owned by root. Did you also change owner and group of the filesystem to your username? Otherwise, you cannot save anything to these drives, nor could rsync when run in your username. 

New flash drive after the filesystem is formatted ext4 with gparted:
```
dmn@Zotac:/media/dmn$ ls -l
total 20
drwxr-xr-x  3 root root 4096 Feb  8 09:13 9a537b36-f577-4617-9366-a652c87f0454

```
At this point, you are an 'other' and have r-x permissions. No write permission. You can't save any files on them.

The command itself is good. Here's what the output code you see means:

Each line starting with **[FONT=Courier New]>f+++++++++[/FONT]** indicate a new file is being created  on the destination, followed by the path to the file relative to (meaning inside) the destination folder. 
Each line starting with **[FONT=Courier New]cd+++++++++[/FONT]** indicate a new directory is being created  on the destination followed by the path to the new directory relative to the destination folder.

A line starting with **[FONT=Courier New]>f.st......[/FONT]** indicates a file was updated because of size and time difference. You won't see this until you have some files that are being updated (replaced by newer versions).

Lots of weird directories are being created, which must reflect what is inside /home/ben/
