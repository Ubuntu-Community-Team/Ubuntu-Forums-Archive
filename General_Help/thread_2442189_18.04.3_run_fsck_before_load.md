### Post by TheFu on 2020-05-01
Systemd broke the old **sudo touch /forcefsck** method.  

The way we are supposed to do it now is from the Advance Boot menu (hold SHIFT down), choose Recovery Mode, then fsck from the menu. That will be hard without a RIBLO/DRAC, I suppose.  

But any file system going read-only probably has HDD/SSD issues. Run the smartctl tool to see the SMART data for the HDD. [https://www.cyberciti.biz/tips/linux-find-out-if-harddisk-failing.html](https://www.cyberciti.biz/tips/linux-find-out-if-harddisk-failing.html)  
The recovery mode menu: [https://www.tecmint.com/wp-content/uploads/2018/09/Select-fsck-Utility.png]("https://www.tecmint.com/wp-content/uploads/2018/09/Select-fsck-Utility.png") 

You can also tweak the file system tuning parameters to force an fsck using the file system tuning tool. For ext4, tune2fs is it. As this is an advanced tool, best to point someone planning to use it at the manpage. The -c option should be it, but I've not used this method myself.  My systems have remote console access at boot.

Since the systemd team took over the fstab, I'm not certain that the tunefs method works anymore. It is dependent on those last too numbers at the end of every line for a mounted file system. Vaguely remember something about the systemd-mount system ignoring those settings.  [https://www.linuxuprising.com/2019/05/how-to-force-fsck-filesystem.html](https://www.linuxuprising.com/2019/05/how-to-force-fsck-filesystem.html) has the systemd and grub options in much more detail.

Really wish there was a command that could be run to force the fsck on the next boot. Alas, I don't know of any.
