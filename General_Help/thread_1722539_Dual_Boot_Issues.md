### Post by 3Miro on 2011-04-05
I think this is the problem:

```
### BEGIN /etc/grub.d/30_os-prober ###
menuentry "Windows Vista (loader) (on /dev/sda2)" {
    insmod ntfs
    set root='(hd0,2)'
    search --no-floppy --fs-uuid --set 1ab430f2b430d24d
    chainloader +1
}
### END /etc/grub.d/30_os-prober ###
```

It should be (hd0,1) and not (hd0,2). Basically, Vista is on 1 and XP is on 2.

The Grub auto-detection script is far from perfect, you should probably report this as a bug on Launchpad. 

I had an issue once with Ubuntu not detecting my Gentoo install, so I had to add a manual entry into Grub2. 

Go to terminal and type:
```
gksudo gedit /etc/grub.d/40_custom
```
then add the following to the bottom of the file:
```

menuentry "Mine: Windows Vista (loader) (on /dev/sda1)" {
    insmod ntfs
    set root='(hd0,1)'
    search --no-floppy --fs-uuid --set BCB82D30B82CEA9A
    chainloader +1
}

```
If it doesn't work again, try making the BCB82D30B82CEA9A lower-case or remove the search like like this:
```

menuentry "Mine: Windows Vista (loader) (on /dev/sda1)" {
    insmod ntfs
    set root='(hd0,1)'
    chainloader +1
}

```

### Post by JKyleOKC on 2011-04-06
That "boot.ini" listing from your /dev/sda2 is part of the XP boot loader that got written to the drive when you were trying to install to the external HDD. The grub entry that references (hd0,2) is telling grub to go to the sda2 partition to boot Vista, and the Windows boot code is then finding the boot/ini file and displaying its menu -- which is the XP menu you are seeing.

Changing the "hd0,2" for grub over to be "hd0,1" simply corrects the error and lets grub find the correct Vista partition, which is apparently only 3 GB in size -- far too small to be the whole Vista system, but it may "chainload" to sda2 during the boot process.

I would definitely make certain that everything is backed up before making the change, though, since if sda1 is really the factory-restore partition it might automatically do its thing and wipe the drive entirely!

The boot.ini file is located in the root folder of drive C: on XP; I don't know whether it's still used in Vista or not but if it is, your results.txt listing indicates that the recovery disk didn't do its entire job correctly because it failed to replace this file, although the report shows that the system on sda2 IS Vista, not XP -- and also does not show any system on sda1, which makes me even more suspicious that this might be a recovery partition. You might examine these two drives from Ubuntu; boot.ini is a plain text file that you can open with gedit to view.

### Post by oldfred on 2011-04-06
When you run windows repairs it repairs the partition with the boot flag. So you may have just repaired the recovery partition. as I agree with JKyleOKC that sda1 is too small to be your full install.

Also sda1 has just these files. In win7 it could be a windows boot/recovery partition or more  likely a vendor recovery partition that restores the system to as purchased & erases everything.
   Boot files/dirs:   /bootmgr /Boot/BCD

Your sda2 has the above two boot files plus it has the executable that is always in the main partition:
/Windows/System32/winload.exe

And in sda2 you have remains of your XP boot files. If you for sure do not have XP you can delete these or copy somewhere as sometimes they come in handy for XP repairs.
/boot.ini  /ntldr /NTDETECT.COM

I would move boot flag to sda2 with gparted (right click on sda2 and manage flags) or with disk utility, click on partition and edit partition to manage flags. Remove boot flag from sda1 if it does not do it automatically as you can only have one per drive.

Then rerun the Vista repairs. If you run the auto repairs it may install the Vista boot loader to the MBR but you then can test that it works directly. But then you also have to reinstall grub2's boot loader to the MBR.

### Post by oldfred on 2011-04-06
From a liveCD or your Ubuntu install you can drill down to the windows root  in sda2 and see the boot files & folders.

Be careful not to delete these files/folders:
  /bootmgr /Boot/BCD /Windows/System32/winload.exe

Note: if your XP install is on a drive connected to this computer you may need these files. Windows does move all boot files from second installs to the first install of windows. But it is better to have the boot files in the second windows install and use grub to choose which to boot.
Then you can cut & paste to somewhere as a backup - these files:
/boot.ini  /ntldr /NTDETECT.COM

You can just delete them, they should be in your actual XP install and boot.ini varies(needs editing) with every install. They are also in XP install disks, so you can find them if need be.
