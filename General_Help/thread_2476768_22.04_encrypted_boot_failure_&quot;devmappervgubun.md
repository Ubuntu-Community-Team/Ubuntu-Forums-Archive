### Post by TheFu on 2022-07-06
FYI,  "dev/mapper/vgubuntu-root" isn't a proper absolute path.  The leading / is absolutely critical.  The error image has it, so that is not the issue.

I've never had boot issues on LUKS encrypted systems, but if I did, I'd fall back to my backups, follow the restore process, which in my world, starts by doing a fresh, minimal, install + ssh, on the system.  As you know, whenever there is encryption involved, all sorts of tiny hardware issues can create a situation where the system doesn't work. This is because a small storage issue can completely jumble access to the entire system. That's unlikely, but often used boot areas do get read a bunch.

If I were to troubleshoot this, I'd begin by booting the same release of an Ubuntu DE/Desktop as the system in "Try Ubuntu" mode. The exact DE doesn't matter, since we just want to mount the encrypted storage, look around and see what is accessible and what isn't.  To accomplish this, I'd use the cryptsetup tool to "open" the partition(s) that are encrypted, then  ...

run ```
sudo lvchange -ay
``` so the "Try Ubuntu" OS will search and discover any PVs, VGs, and LVs that it can.  If no LVs or VGs or PVs can be found, you've just discovered the problem.  By default, to make partitioning cleaner, the Ubuntu installer uses LVM whenever we choose LUKS encryption.  Most of the time, we can think of an LV to be the same as a partition, just a crazy-flexible, partition, that can be modified while the system keeps running.  But if /dev/mapper/vgubuntu-root" isn't found, game over.

From there, if all is well, we can use a GUI file manager to look around.  Ideally, we'll setup a chroot for the non-booting OS (there are plenty of how-to guides) and use ```
journalctl -b -1
``` to seek any issues.  The "-1" is how many boots ago, we want to look for HW issues. I'd probably NOT use the chroot initially and just look through the text logs.

Whenever there are booting issues, we need lots and lots of information, so running the "Boot-Info" script on the Ubuntu Forums github page would be really helpful. Google finds it easily.

### Post by TheFu on 2022-08-31
My mistake.  vgchange, not lvchange. Sorry.  But it will only work AFTER the encrypted container is opened/unlocked.  The vgchange command, with the options above, tells LVM to look on the available storage for LVM headers and make all VGs and LVs available to be mounted.

If you don't have vgchange or lvchange installed, then the booted OS can't access LVM stuff. You need an OS with it.  The /dev/mapper/ stuff in the errors means LVM is expected, so I'm 99% certain it is there unless you specifically did something odd.  Checking the "Encryption" box during installation automatically uses LVM.
