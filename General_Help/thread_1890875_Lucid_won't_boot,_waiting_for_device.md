### Post by drs305 on 2011-12-04
There are two possible issues.

Grub 2 at one point was installed on the partition rather than the MBR. As boot files move, it's possible for the system to break. Additionally, you may have a BIOS limitation that is affecting things.

First, boot a LiveCD, mount your Ubuntu partition, and install Grub to the MBR to make sure everything is correct:
```
sudo mount /dev/sda1 /mnt
sudo grub-install --root-directory=/mnt /dev/sda
sudo umount /dev/sda
```
Don't use "sda1" in the second command, which would install Grub 2 to the partition instead of the MBR.

Next, reboot, but interrupt the boot and enter the BIOS setup. Find out what your reported drive size is. If it reports 137GB or less instead of 500GB, let us know.

Resume the boot, but at the Grub menu press 'c' to get to the grub prompt. Run these commands (up to the #) and provide us with the answer to the questions after the #.

```
ls (hd0,1)/ # Do you see 'vmlinuz' and 'initrd.img' ?
ls (hd0,1)/boot/grub # Do you see a lot of *.mod files ?
```

Press ESC and attempt to boot. If it boots successfully, run:
```
sudo update-grub
```

### Post by drs305 on 2011-12-04

I think you are going to need to get a more recent CD or make a bootable USB. Once you can get to an Ubuntu Desktop, you can 'chroot' into your real installation, purge grub (Legacy) and grub-pc (Grub 2) and then reinstall Grub 2. The procedures are listed in the 'Chroot' link in my signature.

Another, probably better option, would be to create a Boot Repair CD - it will work with Grub 2 and should be able to repair your system. 

The question you haven't answered yet is whether the BIOS can see your entire drive. Many BIOS's, especially older ones, only see the first 137GB of the drive. 

If this is your case, your option is to change a BIOS setting to allow it to see large drives (if this is available), update the BIOS, create a separate /boot partition within the first 130GB of the drive, or shrink the Ubuntu partition to less than 130 GB. The last option will force the boot files to remain within the readable area of the drive. If you shrink your drive, make sure you backup any important data first. You can always mount your Ubuntu partition from any LiveCD and copy/backup your data files to another drive or partition.

### Post by drs305 on 2011-12-05
There are at least 3 options.

1. Try turning off the UUID feature. Remove the **#** symbol in the following line. Save the file, update grub and reboot. 
This change removes the UUID from the 'linux' line. The 'search' line remains, and if you still have problems we will proceed to option 2.
```
gksu edit /etc/default/grub
```
Remove the # symbol from this line:
> **[COLOR="DarkRed"]#[/COLOR]**GRUB_DISABLE_LINUX_UUID=true

Also, the "vga768" that was displayed in your original RESULTS.txt is deprecated in Grub 2. When you reinstalled Grub 2 this entry in the 'linux' line should have been removed and I did not include it in the custom entry. If you need a specific resolution, edit this line in the same file by removing the **#** and set the desired resoltuion:
> **[COLOR="DarkRed"]#[/COLOR]**GRUB_GFXMODE=**[COLOR="DarkRed"]640x480[/COLOR]**


2. If turning off the UUID feature in the 'linux' line didn't fix it, the next option is to make a custom entry that always uses the most current kernel and eliminates the 'search' line:

Open gedit as root and create a new file in /etc/grub.d:
```
gksu gedit /etc/grub.d/06_custom
```
> 
#!/bin/sh
exec tail -n +5 $0
echo "Adding 06_custom" >&2
# This file provides an easy way to add custom menu entries. Simply type the
# menu entries you want to add after this comment. Be careful not to change
# the 'exec tail' line above.

menuentry 'Ubuntu 2.6.32' --class ubuntu --class gnu-linux --class gnu --class os {
	recordfail
	insmod ext2
	set root='(hd0,1)'
	linux	/vmlinuz root=/dev/sda1 ro  splash quiet splash
	initrd	/initrd.img
}


Save the file, and then make it executable and add it to grub.cfg:
```
sudo chmod +x /etc/grub.d/06_custom
sudo update-grub
```

3. A third option would be to skip creating custom file but remove the 'search' line from the menu by editing the system file that adds the 'search' line. You can use this option rather than #2 if you don't want a custom menu.

Keep the change in Option 1 which turns off the UUID. Edit this file:
```
gksu gedit /usr/lib/grub/grub-mkconfig_lib
```
Find this section. In Grub 1.99 it's around line 155. It will be somewhat different in 1.98:
> 
**[COLOR="DarkRed"]#[/COLOR]**  if fs_uuid="`"${grub_probe}" --device "${device}" --target=fs_uuid 2> /dev/null`" ; then
**[COLOR="DarkRed"]#[/COLOR]**    echo "search --no-floppy --fs-uuid --set=root ${fs_uuid}"
**[COLOR="DarkRed"]#[/COLOR]**  fi

Add the comment # symbols, save the file and run:
```
sudo update-grub
```

Omitting the 'search' line is not the perfect solution, as the system will not automatically set the $root variable, but if your Ubuntu partition number doesn't change this should not be a problem.

### Post by drs305 on 2011-12-12
> **morningsunshine said:**
> 
one questions on #1. isn't there a potential issue in case the disk id changes from sda3 to sda5 for example? Will that be easy fix then?

It's possible it could be a problem in the future but it's the lesser of evils. 

In the future, should your disks change and you forget to run update-grub before rebooting, all you would need to do is press 'e' at the grub menu and edit the 'linux' line - changing the "root=/dev/sdX" to the new value. Once booted, run update-grub and it should automatically enter the correct value.
