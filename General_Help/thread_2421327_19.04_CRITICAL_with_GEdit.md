What is the output of the command **find ~/.config -user root -ls**  ?

Also, does gedit work despite the messages?  Often running GUI programs from a terminal will cause messages in stdout/stderr to appear that you wouldn't see when launching them from the desktop or launcher.

### Post by TheFu on 2019-06-20
kpatz is correct about editing using a GUI editor and sudo.

If you only use Linux desktops and always want gedit to be the editor used, then add this line to your ~/.bashrc - top or end, shouldn't matter

```
export EDITOR=gedit
```
Now you can edit your personal files using gedit as normal and when you want to edit system files, then use **sudoedit /path/to file**.  gedit will be used.   <---  UPDATE: corrected, thanks to 1fallen

Clean, safe, 1-time setup and forget it.

If you have servers, then you can set the EDITOR to be whatever editor you prefer. I think nano is the default for sudoedit.

### Post by TheFu on 2019-06-20
1fallen - Thanks for the typo catch! corrected to prevent too much confusion.

The rule of thumb not to use sudo with GUI tools is really just because GUI tools tend to write to config files when ever they like. CLI/Shell tools, don't do that.  sudo -H {editor name} is file, but people forget the -H. The -H changes the HOME directory environment variable to match the userid's (root) expected HOME.

**sudoedit** runs the editor as the current userid, so _shell escapes don't matter_. The user is  just themselves while editing.  Anyone editing system files **is** a sysadmin, qualified or not.  It is just the file copy and copy-back that are run with elevated permissions.   However, sudo has a built-in feature to block shell escapes.  From the sudoers manpage:
```
Preventing shell escapes
     Once sudo executes a program, that program is free to do whatever it
     pleases, including run other programs.  This can be a security issue
     since it is not uncommon for a program to allow shell escapes, which lets
     a user bypass sudo's access control and logging.  Common programs that
     permit shell escapes include shells (obviously), editors, paginators,
     mail and terminal programs.
....

```
There are some caveats to allowing users to sudoedit files.

I'm aware of the root-user-group rwx display from the ls command, but cannot interpret which permission is wrong or correct.The find command showed files and directories under your .config folder that are owned by user "root" and group "root".  This was caused by running gedit with sudo, you were running as "root" and not "macro", so when gedit created or updated those files it created files owned by root, which leave your normal account unable to write to them. This is why you get the permission denied errors when running gedit without sudo.

The files in your home directory should be all owned by you (well, your user account and group account), which in your case is "macro".  The chown command will fix this.
