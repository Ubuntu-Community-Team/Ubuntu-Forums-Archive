The red lines should not be there.
The blue line is missing a ${root}.
There is nothing wrong with the green line, but I suggest to remove the "(hd0,2)", since  it defeats the purpose of the 'search' line.

To fix the "blue and green" problems use:


```
drivemap -s (hd0)[COLOR="Purple"] \${[/COLOR]root}
chainloader  +1
```

in your windows item.

I'm not sure what is causing the red lines to appear.  You might be missing a ">&2" at the end of the echo line in your custom scripts. Please post your full custom scripts if you need further help.

### Post by meierfra. on 2010-01-09
> I thought it would output the string when I ran sudo update-grub, 

Try this:


```
#!/bin/sh -e
[COLOR="Red"]echo "Adding Ubuntu" >&2[/COLOR]
cat << EOF
#Linux Entry
menuentry "Ubuntu Linux 9.10, 3.6.31-17-generic" {
    recordfail=1
        if [ -n ${have_grubenv} ]; then save_env recordfail; fi
    set quiet=1
    insmod ext2
    set root=(hd0,3)
    search --no-floppy --fs-uuid --set 33b15938-6146-48f9-9ed0-445abe2063b5
    linux   [COLOR="Blue"]/vmlinuz [/COLOR]root=UUID=33b15938-6146-48f9-9ed0-445abe2063b5 ro   quiet splash
    initrd   [COLOR="Blue"] /initrd.img [/COLOR]
}
[COLOR="Red"]EOF
[COLOR="Red"]echo "Adding Memtest" >&2[/COLOR]
cat << EOF[/COLOR]
#memtest86+ entry
menuentry "memtest86+" {
    linux16    /boot/memtest86+.bin
}
[COLOR="Red"]EOF
[COLOR="Red"]echo "Adding Windows XP Professional" >&2[/COLOR]
cat << EOF[/COLOR]
#Windows XP Entry
menuentry "Windows XP Professional x86" {
    insmod ntfs
    set root=(hd0,2)
    search --no-floppy --fs-uuid --set 262c1f752c1f3eed
    drivemap -s (hd0) \${root}
    chainloader +1
}
EOF
```


The red changes should output the string in the terminal when running update grub.

The blues changes make sure that Grub 2 will always automatically load the newest kernel. So you won't have the edit your custom script after kernel updates.
