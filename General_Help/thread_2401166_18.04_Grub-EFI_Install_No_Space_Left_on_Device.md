### Post by oldfred on 2018-09-16
I have a Asus Z97-AR and similar issues.
[https://bugs.launchpad.net/ubuntu/+source/grub2-signed/+bug/1787692](https://bugs.launchpad.net/ubuntu/+source/grub2-signed/+bug/1787692)

Bottom line maybe just reformat your ESP - efi system partition, edit fstab with new UUID,  and use Boot-Repair's advanced options to totally reinstall grub & latest kernel. I did it manually see below.

I have several emergency flash drives that worked for booting, one with rEFInd & another with a full install of Ubuntu.  UEFI entries from my main working install in my SSD, sda and ESP - efi system partition on sda were not being recognized, only my emergency boot flash drives.
Realized finally that something in ESP was causing UEFI not to see sda1 partition.

I think I resolved them, but went thru a lot. Not sure what I did that actually worked.
I reinstalled latest UEFI from Asus. Erased all UEFI entries with efibootmgr, disconnect drives so UEFI would not see anything. But it may have been just refommatting sda1, my ESP.

I tried running dosfsck or fsck.vfat and it said it removed dirty bit.
sudo fsck.vfat -t -a /dev/sda1
But on reboot it was back to being corrupted and UEFI would not boot.
So I just backed up sda1 & totally reformatted sda1 using another install. 
Also had to update UUID in fstab with new UUID as it would not boot with old incorrect UUID.
sudo mkfs.vfat -F 32  /dev/sda1

But that left ESP, blank with no /EFI/ubuntu or /EFI/Boot folders or files in the folders. 
So I totally reinstalled grub rather than restoring backup.
sudo apt purge grub-efi-amd64  grub-efi-amd64-signed
sudo apt-get install grub-efi-amd64-signed
sudo apt install shim-signed grub-efi-amd64-signed linux-signed-image-generic
sudo update-grub

Not sure if install added UEFI entry or I ran this, as I ran this multiple times but error on UEFI not seeing sda1, then erased entry.
sudo efibootmgr -c -w -L ubuntu -l "\EFI\ubuntu\shimx64.efi"  -d /dev/sda -p 1
