### Post by oldfred on 2009-11-27
It should need the drivemap command to work. But drive number may vary depending on which drive you have booted first per your BIOS. I have found on my system grub2 does not seem to follow the BIOS but the actual order plugged into the SATA ports, but boots from the first one listed in BIOS.

I have also seen it as this:

copy to : /etc/grub.d/40_custom 
menuentry "Windows  (loader) (on /dev/sd[COLOR=DarkRed]b5[/COLOR])" { 
    insmod ntfs 
    set root=(hd[COLOR=DarkRed]1,5[/COLOR]) 
    search --no-floppy --fs-uuid --set [COLOR=DarkRed]4a6077fc6077ed57[/COLOR] 
    drivemap -s (hd[COLOR=DarkRed]1[/COLOR]) ${root} 
    chainloader +1 
}

You will have to adjust the red bits to your drive, partition and UUID. This should work for all versions of windows

### Post by Crafty Kisses on 2009-12-01
I had the same issue, Windows7 64-bit didn't want to boot with Grub2. I seen somewhere else what oldfred suggested and it worked perfectly, which was:
```
menuentry "Windows (loader) (on /dev/sdb5)" {
insmod ntfs
set root=(hd1,5)
search --no-floppy --fs-uuid --set [COLOR="Red"]4a6077fc6077ed57[/COLOR]
drivemap -s (hd1) ${root}
chainloader +1 
```
Remember the parts in red you need to change it to your own UUID, you can get that by running:
```
sudo blkid
```

### Post by bforbes on 2009-12-26
I fixed this problem by just installing grub2 into the MBR of the Windows 7 drive, and putting that drive first in the boot order. The Windows drive seemed to then be (hd0), no drivemapping required. Although, in my computer, the windows drive IS actually plugged into the first SATA port.

Ubuntu still booted fine.
