Wrong order:
```
sudo apt autoremove
sudo apt-get autoclean
sudo apt-get update 
sudo apt-get upgrade
```

I typically do the autoremove and autoclean AFTER patching, not before.

```
\df -i /boot
\df -Th /boot
```

Directories don't get full. File systems do. It may seem like a subtle difference, but it is important. If /boot is part of the / file system, not a separate file system, there isn't much to be done. File systems usually are connected to partitions.

I feel the pain.  When we install software manually, we seldom think about upgrades in 2 yrs, so we make mistakes.  In the future, know this is the preferred order of installation techniques:
[Https://ubuntuforums.org/showthread.php?t=2468553&p=14066847#post14066847](Https://ubuntuforums.org/showthread.php?t=2468553&p=14066847#post14066847) 

If you must install manually, place the software under /usr/local/ into its own directory structure and be certain to include /usr/local/ in your backups.  Take careful notes about any external dependencies and OS packages installed through the normal APT system.  This will help in 2 yrs.  I keep the notes in the same directory under /usr/local/ ... so it gets backed up daily with my normal backup stuff.  Any changes to the software, say I add a module, will be added to those notes and picked up in  the overnight version of the backups.  By retaining 60+ days of versioned backups, I'm likely to uncover and fix any issues well before the next upgrade is needed.

Of course, nobody thinks of these things when they haven't experienced the challenges already.  I certainly didn't.  Took me about 12 yrs before I started doing any real backups, even after spending $300 for a tape backup solution years earlier.  Tape was just too hard to use. Then I lost 80% of my data due to a number of issues, including not having backups.  You'd think that would teach me. It didn't.

By some magical luck, I'd run a backup (not versioned), the prior week and was able to figure out exactly what the cracker changed, figure out how they got in and close that method.  It took being cracked for me to automate the backups, though they weren't versioned.

Around 2008, I finally spent the time to learn how to do daily, automatic backups. I was responsible for running servers for 3 small companies, so NOT having backups simply wasn't acceptable.  I was rotating 10 separate versions of backups and they were taking forever.  Each system was 45-90 minutes to backup and I has 12 systems which meant the backup window didn't allow the needed time to get everything completed.  I was backing up full virtual machines, which appeared like 10-40G blobs of binary data. This was stupid.

After struggling for a few years, trying to find ways to make those backups faster, I finally figured out the solution. in 2010 and have been using it, optimizing it ever since.  Backups for each VM/Container take 10 seconds to about 3 minutes for nearly all the systems.  1 system take 25 minutes, but it has lots and lots of tiny files which are harder for any backup tool to check for changed data. My backups begin at midnight and are generally finished by 12:10am.  That one system that takes longer to backup just takes more time to finish.

You could say that I have backup religion now.  If it isn't clear, backups are a great way to make performing an OS upgrade much less painful, at least the way I do it.  It is all in the design of the backups and the expected restore techniques. My full restores all begin by installing the OS, fresh.  Usually the OS will be the same version as was there, but that isn't mandated. A new release can be used for the most part.  Sure, a few things will have issues post install and data restore, but most things will be fine.  How-To restore backups:  [Https://ubuntuforums.org/showthread.php?t=2474107&p=14091927#post14091927](Https://ubuntuforums.org/showthread.php?t=2474107&p=14091927#post14091927) explains the restore order, which is very important so the system comes up and "feels" like it did before  with the data and settings carried over.  Of course, those things need to be included in the backup data.

If you get someone else to do this, beware they can't be just an intermediate-level Linux person. Much of what I've posted requires a deeper knowledge to understand the subtle aspects of what is being said.  It takes a little time to read, then it can take some research to understand each piece, then it can require sleeping on it a few days for all the tidbits to "click" together.  But once it is done, it is really very simplistic and obvious.  Overall, we decouple our settings and data from the OS and the storage layout.  Those can change and barely impact the restore process.
