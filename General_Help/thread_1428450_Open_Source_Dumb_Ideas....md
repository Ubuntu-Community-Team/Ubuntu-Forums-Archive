### Post by Hamma Hedd on 2010-03-12
On one hand I am supremely grateful for escape routes from the corporate moron mentality of Microsoft.....

But on the other hand there are a fairly thick and intractable range of really stupid ideas that are forced upon people. Here is one of them.

SORT BY DATE:

Well there are some folks who have developed work arounds for Microsoft and their idiotic filing system for retards...

And they have SORT BY FILE SIZE - and the freebee programs actually show the size of the files etc...

But in Ubuntu, some clever group decided to SORT BY DATE and SORT BY FILE (?) SIZE.

Two problems:

For archival purposes, lets say today is 01 of January 2010. I import / download a file that was created on the 15 May 2005.

The file size sorting method, will sort by DATE, that is the actual DATE of the ORIGINAL FILE CREATION, and not the date it was "created by copying" onto my hard drive.

This makes it a pain and anti-intuitive to find files.

So the file sorting by date must be changed from sort by (original file creation) date, to "DATE CREATED" and "DATE MODIFIED" and "DATE MOVED".

It must be qualified by "CREATED" means the original date of "CREATION" in it's true sense. "MODIFIED" must  mean that date the original file was modified (edited etc)., and the "DATE MOVED" must demonstrate the date of the file being created (copied / moved) in that folder.


The second issue of ingrained stupidity is the "SORT BY FILE SIZE" - I mean this is good for sorting files, but in respect to FOLDERS, - using a head count of the AMOUNT of files within the folder is not the same as the actual size of the folder in terms of megabytes (or whatever) of the folders.

I want to see the dual listing for folders to be "25 files 650MB" - and for the sort criteria to be in the actual file SIZE.
