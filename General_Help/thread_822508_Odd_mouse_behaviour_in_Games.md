### Post by Xianath on 2008-06-19
I think it's related to multi-head setups and more specifically the way virtual screen size is calculated. Check these out: [http://www.nevercorner.net/forum/viewtopic.php?id=581](http://www.nevercorner.net/forum/viewtopic.php?id=581), [https://bugs.launchpad.net/ubuntu/+source/libsdl1.2/+bug/14044](https://bugs.launchpad.net/ubuntu/+source/libsdl1.2/+bug/14044) and [http://gentoo-wiki.com/HOWTO_Advanced_Mouse](http://gentoo-wiki.com/HOWTO_Advanced_Mouse) (Uncontrollable mouse in SDL apps)

I was able to solve this on my box by using the Gentoo workaround + making sure metamodes did not include offsets (i.e. changed them from 1440x900_60+0+0 to just 1400x900) + only using relative screen positioning (i.e. Screen1 above Screen0 as opposed to Screen1 +0+900). Give it a whirl and let us know if it works for you too.

Cheers,
Peter

### Post by kaempfer on 2008-06-25
Thanks very much. I've been seeking for a solution for weeks :)

Simply adding these lines to my xorg.conf did the trick (from the gentoo how-to)
```

   Section "Module"
       ...
       SubSection  "extmod"
         Option    "omit xfree86-dga"   # don't initialise the DGA extension
       EndSubSection
       ...
   EndSection

```

### Post by detyabozhye on 2008-07-22
Problem with imwheel is it will only take events after they go through X (it's on the same level as xbindkeys). If you can set up the mouse (with evdev or without) so that you will get unique events in xev for each mouse button (except cruise buttons) you will have more than half of the problem fixed by then. Unfortunately, I have not had any success with this using anything other than evdev. Also, I haven't tested it on Hardy yet since I'm still running Gutsy on my home machine. Once I upgrade to Hardy, I might come up with something.
