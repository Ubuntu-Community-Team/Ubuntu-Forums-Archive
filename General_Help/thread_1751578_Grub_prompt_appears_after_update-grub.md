### Post by drs305 on 2011-05-08
When you ran the dpkg-reconfigure command it wrote the address of computer "a" into the Grub files, so that is the one it is looking for.

From the grub prompt, with the drive in "a", try booting from the command line:

```
set prefix=(hd0,1)/boot/grub
set root=(hd0,1)
insmod (hd0,1)/boot/grub/normal.mod
normal

```
When you enter "normal" your Ubuntu menu may appear and you can select the menuentry to boot.

If that doesn't work, try the following:
```
set prefix=(hd0,1)/boot/grub
set root=(hd0,1)
insmod (hd0,1)/boot/grub/linux.mod
linux (hd0,1)/vmlinuz root=/dev/sda1 ro
initrd (hd0,1)/initrd.img
boot

```

Once booted, run:
```
sudo grub-install /dev/sda
sudo update-grub
```

### Post by drs305 on 2011-05-11
> **l07 said:**
> 
Should I now run
```
sudo grub-install /dev/sda
sudo update-grub
```
?

I think that is just going to try to boot off the most recent kernel if you haven't changed anything, so you will need to alter the grub defaults.

If you are still booted into your normal system, first run:```

uname -r
```
This will tell you which kernel works. You will want to make that one the default. So you can update-grub, but then make sure to set the working kernel as the default. As a minimum, make sure it appears somewhere in the G2 menu so you can select it.

Pre-Natty, the best way to make it the default is to use the exact title rather than a number in /etc/default/grub's "DEFAULT=" line so it doesn't matter what position it's in. You could also make a 40_custom entry with that kernel so it always appears in your Grub menu.

Once you have done that (and updated-grub), you could try working with the newer kernels (purge and reinstall), play with kernel options, etc., to see if you can get one to boot.

If you have questions about how to do this, just ask.

### Post by drs305 on 2011-05-15
Update:
We have determined that the problem is a very old BIOS which recognizes only the first 8GB of the drive. Notice from the last RESULTS.txt all the bootable files are within the first 8GB *except* for grub.cfg.

Since grub.cfg is outside 8GB, the previous post wouldn't work because 'normal' would try to read grub.cfg, which it wouldn't be able to find. This was how the BIOS issue was confirmed, plus checking the drive size reported by the BIOS.

The OP has several options:
[LIST]
[*]Enable large drives/LBA in BIOS, if it offers this.
[*]Update the BIOS.
[*]Create a small /boot partition at the start of the disk.
[*]Put /boot on another device for booting purposes.
[/LIST]
