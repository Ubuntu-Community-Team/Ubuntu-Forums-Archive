### Post by SuperSonic4 on 2009-12-09
BIOS is independent of, and loaded before, any OS on the system. It is contained on a chip on the motherboard and is referenced whenever the hardware is booted up.

It does accept keyboard input but mainly for changing config settings like boot order, time and date and any OC options.

X is part of the OS, a layer on top of the shell if you like and that handles input and output from the OS

### Post by lisati on 2009-12-09
As I understand it, there's what might be called a chain of command, like a hierarchy, with varying levels of sophistication. Although not necessarily intended as a technically accurate outline of how all the parts fit together, the way I envision it is something like this:

At the very bottom level, there's direct access to the hardware. The next level up (in terms of sophistication) is the BIOS, which is a bit like the "Built in operating smarts." (yes, I know that's **not** really what the letters stand for, but can be helpful for understanding what it's for!) 

Next up is the drivers that the OS provides. These can either access the hardware directly or use the BIOS.

The next level up can be other components provided by the OS or designed to work as part of the general software environment, e.g. startx. 

At the top level is the software which the computer user runs.


Hope this helps someone somewhere.

this is the correct way to look at it. it's all levels of abstraction. 
if you want a good example of the concept, study up on the ISO's OSI model for computer network stacks [http://en.wikipedia.org/wiki/OSI_model](http://en.wikipedia.org/wiki/OSI_model)

the hardware sends electrical signals, which the bios interprets into binary data, and notifies any one who cares, that a signal has been recieved. a kernel level program recieves that signal and the binary data, and uses the driver to make sense of the input (converting it to raw yet meaningful), and notifies any applications that the data is avaliable. the applications then encode the raw data against their native encoding scheme (ascii, ebcidic, unicode), and use it for whatever purpose they desire. all hardware access moves through BIOS controlled channels like the system bus, northbridge and southbridge.

### Post by doas777 on 2009-12-09
one of the core concepts of *nix, is that it is portable. older computer systems (before x86 took over the world), had to be written almost entirely in assembler code specific for that platform. an DEC machine and a Alpha machine, and a MIPS machine could not use the same assembler, so you had to rewrite it from scratch.

Unix was special because it used only a tiny peice of assembler to form the C compiler, and the very bottom of the stack (where it touches the hardware). everything else is written in C. that way, you only need to write a small amount of native assembler to port unix to a new hardware platform. the rest of it just compiled right up.

This concept of encapsulation and abstraction is what Lisati was mentioning. when I write a python script, I need not care what platform I am on, or even what DE/WM my user employees. all I need to know is that they have python 2.6+ and GTK installed. all else is unimportant to me, as I can be reasonably sure that my program will execute correctly while relying on the unknown lowerlayers underneath python.
