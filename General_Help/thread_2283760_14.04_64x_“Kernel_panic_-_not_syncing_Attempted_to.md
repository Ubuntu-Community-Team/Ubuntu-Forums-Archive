### Post by oldfred on 2015-07-05
For removing older kernels I prefer to use synaptic. But if you have a full drive issue you may not be able to install and have to use command line.

       Determine your current kernel:
uname -a
-s is similate so you can review first:
sudo apt-get -s autoremove
sudo apt-get install synaptic
In synaptic search for linux-image to choose to delete old ones
Using synaptic
[http://ubuntuforums.org/showthread.php?t=1283521](http://ubuntuforums.org/showthread.php?t=1283521)

      
 cd /boot
ls vmlinuz*
sudo ls -l /boot/vmlinuz*
sudo apt-get purge  linux-image-[version]-generic linux-image-[version]-generic
Multiples, just be sure not to delete your current kernel:
sudo apt-get purge linux-image-3.8.0-{XX,XX,XX,XX,XX,XX}-generic
Example:
sudo apt-get purge linux-image-3.8.0-{17,18,19,21,22,23,24}-generic

If you upgraded and have old kernels from previous installs, they are not in dpkg so will not be deleted. Then you may have to manually delete those.

### Post by oldfred on 2015-07-13
Your install is in BIOS boot mode with MBR(msdos) partitions and then LVM or logical volume partitions overlaying or inside the physical partitions.

If you have a newer UEFI hardware system, you can boot in either UEFI or CSM boot modes.
       CSM - UEFI Compatibility Support Module (CSM), which emulates a BIOS mode 

If you then boot Boot-Repair in UEFI mode it will want to try to repair your install in UEFI mode. You should always be consistent with new UEFI hardware. Always boot in UEFI mode or always boot in BIOS mode.

I do not know LVM, but if a standard partition I would suggest fsck. But not sure how to mount your LVM from live installer.
You have to at least do this, I think Boot-Repair automatically adds it, if you have run it.

 sudo apt-get install lvm2

This is a standard partition entry for fsck, but with your LVM you need to change from the physical partition to the logical. Probably /dev/mapper/ubuntu--studio--vg-root
 #From liveDVD/Flash so everything is unmounted,swap off if necessary, change example shown with partition sdb1 to your partition(s)
#e2fsck is used to check the ext2/ext3/ext4 family of file systems. -p trys fixes where response not required
sudo e2fsck -C0 -p -f -v /dev/sdb1
#if errors: -y auto answers yes for fixes needing response, also see man e2fsck
sudo e2fsck -f -y -v /dev/sdb1

Your sdc drive is using exfat, but you show no Windows. You may need Windows to run chkdsk on it periodically. It also has a less standard  4096:4096:msdos where normally you would have gpt partitioning and 512:4096. And if just Linux then Linux formats are better.

### Post by oldfred on 2015-11-08
With gpt partitioning you need the ESP - efi system partition for UEFI boot or the bios_grub partition for BIOS boot. 
But your drive was MBR(msdos) which is only for BIOS boot.
So do not boot Boot-Repair in UEFI mode. Reboot in BIOS mode and try running repairs.

No bootable device sounds like you are trying to boot in UEFI boot mode or secure UEFI boot mode.
