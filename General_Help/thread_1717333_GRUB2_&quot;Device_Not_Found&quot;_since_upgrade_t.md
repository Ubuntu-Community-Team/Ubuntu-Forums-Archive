### Post by lithopsian on 2011-03-29
Perhaps trying to access the drive too early?  Or just in a bad order?

Try with a rootdelay and see if that helps.

You can also try running with quiet turned off so you can see just what is happening, although I'm pretty sure where it is failing.  One of the last steps in initramfs is to mount the root partition and if it isn't found after a certain amount of time (default 30s?) then you get this message.  The error will probably occur just after you see "Waiting for root file system ...".

It could also be a problem with the function used to detect that root is up from initramfs.

### Post by oldfred on 2011-03-29
You have a problem on sdb that may take it a while to resolve and that may be the entire issue.

$MFTMirr does not match $MFT (record 1).
Failed to mount '/dev/sdb1': Input/output error 
etc...

I would do as it says and run chkdsk on sdb1. If errors are found you have to rerun chkdsk until there are no errors as it does not fix them all on one pass.

Vista/7 CHKDSK
chkdsk [drive] /f /r
chkdsk C: /r
/f : Fixes errors on the disk.
/r : Locates bad sectors and recovers readable information.
Note If you specify the /r option, the /p option is implied. When you specify the chkdsk command without arguments, the command checks the current drive with no options in effect. Rerun until no errors.
[http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/chkdsk.mspx?mfr=true](http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/chkdsk.mspx?mfr=true)

It may also be that you use the /dev/sda5 rather than UUID.

### Post by garvinrick4 on 2011-03-29
You have a boot partition in sda1 and operating System in sda5 lets try:
With live cd and internet working.
sudo mkdir /media/boot
sudo mkdir /media/root
sudo mount /dev/sda1 /media/boot
sudo mount /dev/sda5 /media/root
sudo mount -o bind /dev /media/root/dev
sudo mount -o bind /dev/pts /media/root/dev/pts
sudo mount -o bind /proc /media/root/proc
sudo mount -o bind /sys /media/root/sys
sudo cp /etc/resolv.conf /media/root/etc/resolv.conf
sudo chroot /media/root
apt-get update
apt-get purge grub grub-common grub-pc
apt-get install grub-common grub-pc
grub-install /dev/sda
update-grub
exit
sudo umount /media/root/dev/pts
sudo umount /media/root/dev
sudo umount /media/root/proc
sudo umount /media/root/sys
sudo umount /media/root
sudo umount /media/boot
sudo shutdown -r now
You are purging and reinstalling grub and installing in mbr in sda with boot partition mounted.

### Post by oldfred on 2011-03-31
No, bios-grub is only for gpt with BIOS systems.

I guess the manual boot needs some thing more for a separate boot partition. Also most reinstall of grub2 or old grub legacy leave off the extra instructions for the separate boot partition as it is uncommon for standard desktop installs.

You are really booting sda1 as that is where the kernel is.

This may be better:

set prefix=(hd0,1)/boot/grub
set root=(hd0,1)
insmod normal
(If that fails try:  insmod (hd0,1)/boot/grub/normal.mod)
insmod (hd0,1)/boot/grub/linux.mod
linux /vmlinuz root=/dev/sda5 ro
initrd /initrd.img
boot

The boot files are in sda1, so those references must be to the boot partition (hd0,1) but the root=/dev/sda5 has to be to the actual install.

General menuentry Construction Rules:

    * The first line must start with menuentry and end with {
    * The area between the quotation symbols is what will appear on the GRUB 2 menu. Edit as desired.
    * The last line of the menuentry must be }
    * Do not leave empty spaces at the end of lines
    * The set root= line should point to the GRUB 2 /boot location ( sdXY )
    * The root reference in in the linux line should point to the system partition.
          o If GRUB 2 cannot find the referenced kernel, try replacing the UUID with the device name (example: /dev/sda6 ).
