### Post by Gannin on 2009-01-22
For you and me, installing software on Ubuntu usually isn't a big deal.  If we want to upgrade OpenOffice, we know to dig around until we find the .deb version.  We're conversant with Add / Remove Programs and can find newer versions of programs on getdeb.net.  This isn't necessarily the easiest or quickest way to install new programs or update existing programs that aren't updated through Update Manager, but we can do it.

However, a lot of this doesn't make sense to new users that are trying out Linux and only have Windows experience so far.  Here's a short example from an article I just read where a Windows user decided to try Linux for two weeks.

"Ubuntu's biggest Achilles heel is software installation and updating. Installing some software was simple, but installing others was so baffling as to be nearly incomprehensible. The same holds true for updates; I ultimately gave up on even trying to update OpenOffice.org."

I think if Linux is ultimately going to succeed as a mainstream desktop OS, instead of just a netbook OS, it needs truly consumer friendly software installation.  Something where all software is installed in the exact same way regardless of what program it is, and all programs are updated in the same way.  Something truly consumer friendly.

I know there are some that think the way software is handled now is good enough, but it's really not good enough for the average consumer.  Linux really needs to come together and come up with a new way for handling software.

### Post by 3rdalbum on 2009-01-23
Linux runs on more than just x86 processors, kiddo. You can't make one binary that runs on all the platforms that support Linux.

Windows only supports one platform - x86. That's why 64-bit Windows is still in such a shocking state; no source code is ever available for any programs so they can't be recompiled for 64-bit. That's why the Itanium version of Windows sucked so hard, because you couldn't get anything to run on it.

It's also important to note that neither Windows nor Mac OS X actually has a universal method for installing software. Windows has a shockingly large selection of binary installers that all sorts of companies have written, plus the ".msi" files. Mac OS X has binary installers, disk images that you drag 'n' drop the contents of into your Applications folder, and apt-like solutions like fink. And you can compile programs with GCC.

What Linux *does* need is some standard GUI'fied compiling system. I believe XFCE still uses its excellent build system that feels like a binary installer but actually compiles the environment; if something like that could be made for installing any compatible source package, and registering the resulting binary with the system's package manager (and getting dependencies from the package manager) then we'd really be onto something good.

### Post by mb_webguy on 2009-01-24
Well, there are three basic methods of installing software on Linux other than from a distribution's repositories: deb, rpm, and compiling from source.  Since you can already convert between rpms and debs using alien, it shouldn't be difficult to create a program that takes as input a deb, rpm, or directory containing source code, checks for dependencies and installs them as necessary, converts the package to the proper format or compiles the code, and then installs the software.  It would basically just be a script or GUI front-end for executing existing commands.  You could throw in checkinstall (and the rpm equivalent thereof) to add compiled software to the package database so it can be removed with a package manager.

This would make it possible for simple double-click installation of pretty much any software on any system.  The problem with this, of course, is that the conversion process between rpms and debs is not perfect because of the differences in Debian and Red Hat systems, and so this method shouldn't be used for important system software (which should be installed through the distribution's repositories, anyway).  If someone were to take the initiative to create such a program, and it were included with each distribution, then it seems to me that this would solve this issue.  You'd have one consistent way to easily install any software on Linux.

So all we need is for someone to volunteer to write the program.  Once that's done we just have to convince every distribution to include it.  Sounds simple enough, right?
