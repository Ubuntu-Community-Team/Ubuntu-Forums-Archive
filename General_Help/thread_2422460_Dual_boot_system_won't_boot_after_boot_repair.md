### Post by oldfred on 2019-07-08
Do you have two installs of Ubuntu & two installs of Windows?
And on MBR drive BIOS installs and on gpt drive UEFI installs?

If you have UEFI boot on sdb, you need to have boot/esp flag on sdb1, so it is seen as the ESP - efi system partition. It looks like it is FAT32 and shows /EFI/ubuntu & /EFI/Microsoft folders.

Generally best not to have both BIOS & UEFI on same system. But since separate drives, you can dual boot, but only from UEFI boot menu. You may have to turn on/off UEFI or BIOS/Legacy/CSM mode, but mode newer UEFI now know that entry in UEFI menu needs correct mode.

Your UEFI boot entry also seems to have grub.cfg in ESP that boots sda3 on MBR drive. So is Windows on sda an old BIOS install that is not used?
Windows only boots from MBR with BIOS and Windows only boots from gpt with UEFI.
But Ubuntu lets you use the MBR drive with UEFI (perhaps it should not to avoid issues).

Turn on esp boot flag on sdb1, and see if it boots. If not boot Ubuntu live installer in UEFI mode and run Boot-Repair (after sdb1 is seen as esp) to see suggested fixes.
Make sure Windows has not turned fast start up back on, as it will do that with updates and then grub will not boot it until you turn it off again, buy booting directly using UEFI.

### Post by oldfred on 2019-07-08
While maybe possible to boot the same install with both BIOS grub & UEFI grub, they eventually get out of sync.
And your fstab now shows the mount of an ESP - /boot/efi, so now configured for UEFI boot.
But it did not show an original install comment line, so probably was BIOS before you ran Boot-Repair.

But you also show a Windows entry in the ESP - /efi/Microsoft which only is for UEFI boot and would only work on the Windows in sdb. Did the Windows install in sdb work before or was it just a backup/copy? Report is not showing the typical rest of Windows boot files in sdb, so not sure it that ever worked or not?

Do you have good backups of both Windows & Ubuntu. And install disks?

Do you want newer UEFI boot or older BIOS boot? For both systems (recommended)?
For BIOS boot you need to temporarily install a Windows boot loader to the MBR of sda & fix Windows. Boot-Repair may install a BIOS boot loader in advanced mode, if it can see Windows (not hibernated), otherwise you need your Windows repair disk or installer with repair console. And if Windows just starts to boot you may need f8 to get into internal repair console or use Windows repair flash drive.
For UEFI boot you may need to reinstall Windows either to sdb, or convert sda to gpt as Windows only boots in UEFI mode from gpt drives.

If both HDD, best to have Windows on one drive and Ubuntu on other drive. For BIOS boot of Ubuntu on gpt drive you also need a tiny 1 or 2MB unformatted partition with bios_grub flag.

Even if you want to stay with the now very old BIOS boot better to keep sdb as gpt and have both ESP (for UEFI boot) and bios_grub for BIOS boot, so you can configure it either way.
But Windows requirements control partitioning BIOS/MBR or UEFI/gpt for Windows drive only.

### Post by oldfred on 2019-07-09
With MBR, Windows often deleted from partition table entry for ext4 partitions. Do not think that is an issue with UEFI/gpt.

You will have to also reinstall grub for UEFI boot and edit fstab with entry for ESP (full reinstall of grub may add it), and it is likely that UUID for / will also change, which then also requires fstab edit. 

Again can be done, but often reinstall quicker & easier, especially if /home is on sdb drive.  You should export list of installed apps to make it easier to reinstall them and any files in /etc for system that you may have edited. (I edit grub & exports, but not much else).

       Converting to or from GPT - must have good backups.
[http://www.rodsbooks.com/gdisk/mbr2gpt.html](http://www.rodsbooks.com/gdisk/mbr2gpt.html)
gdisk to convert from gpt to MBR
[http://www.rodsbooks.com/gdisk/mbr2gpt.html#gpt2mbr](http://www.rodsbooks.com/gdisk/mbr2gpt.html#gpt2mbr)
