### Post by TheFu on 2013-05-01
I had a problem with not enough storage for /boot too.  Until I made plenty of storage - 500M over the currently used amount, installing new kernels never worked.  Ended up writing a tiny script to **apt-get purge** all but the last 2 kernels.  

Don't forget that running out of inodes is just as bad as running out of sector space.  **df -i** vs **df -h**.

### Post by majorbullet on 2013-05-02
OK partially solved.

The changes i had made to /etc/lvm/lvm.conf caused /usr/share/initramfs-tools/hooks/cryptroot to throw the errors.  I noted that as I updated grub, there were additional lines error lines at the end of the terminal output:
```
readlink: unrecognized option '----'
```

These same type of errors were reported in bootinfoscript:
```
=============================== StdErr Messages: ===============================

xz: (stdin): Compressed data is corrupt
lvdisplay --  Volume group name - has invalid characters
lvdisplay --  Skipping volume group -
lvdisplay --  Volume group name - has invalid characters
lvdisplay --  Skipping volume group -
lvchange --  Volume group name - has invalid characters
lvchange --  Skipping volume group -
hexdump: /dev/mapper/-: No such file or directory
hexdump: /dev/mapper/-: No such file or directory
lvdisplay --  Volume group name - has invalid characters
lvdisplay --  Skipping volume group -

```

It dawned on me that the errors were reported 6 times => the same number of LVMs in my system.  Uggghhh.
So reverted /etc/lvm/lvm.conf back to standard 
```
command_names = 0
prefix = "  "
```
and reinserted /etc/cryptab.  Now update-initramfs runs to completion without errors and no errors during update-grub.  the size of the initrd.img is almost back to the size of the archived version.

```
12796 -rw-r--r--  1 root root 18008465 May  1 23:41 initrd.img-3.2.0-39-generic
   77 -rw-r--r--  1 root root 18103131 Mar 28 20:27 initrd.img-3.2.0-39-generic.baculabackup

```

But, when I reboot, the system still drops into shell (initramfs).  Looks like I need a bit more work. But good to know that the lvm.conf changes have an impact on scripts that read from lvm tools (vgdisplay, lvdisplay, etc); and hooks/cryptroot does.

### Post by majorbullet on 2013-05-03
Thanks FuMan,
lvm boot has been supported for quite awhile: [http://askubuntu.com/questions/76095/what-is-the-use-of-boot-lvm-based-in-partitioning](http://askubuntu.com/questions/76095/what-is-the-use-of-boot-lvm-based-in-partitioning)
As for cryptroot, it was included during my install as part of cryptsetup package.  It is not needed if you do not have/ want encrypted drives.

Now as to my troubleshooting and solution:
All of the problems that I had after the full snapshot recovery and the update failure stem from the changes that I made to the /etc/lvm/lvm.conf file.  The information was was staring me right in the face in the diff file but I just did not see it/ could not believe that changing logging of LVM could cause my issues.

The use of the of either or both
```
command_names = 1
prefix = "  -- "
``` caused scripts that rely on lvm tools (lvdisplay, vgdisplay) to throw errors.  Therefore the hooks/crytroot failure during the update-initramfs runs.

Once I fixed this update-initramfs completed with no errors, but the new initrd.img's were still dropping me into shell (initramfs).  There was no lvm module listed from
```
cat /proc/modules
```.

So I tried to load the modules only to receive error message that 'could not find /var/log/lvm2.log'.  Yes, the same log file that I had activated through /etc/lvm/lvm.conf.  Same error when i tried to run /sbin/lvm from intitramfs.  So I reverted back to the original configuration for lvm.conf (logging through syslog, commented out /var/log/lvm2.log), ran update-initramfs, ran update-grub, crossed fingers and rebooted.

Sure enough, system is now booting on initrd.img-3.2.0-39-generic and initrd.img-3.2.0-40-generic with no problems.
