Both issues happen only at boot time, and since this machine normally runs 24/7 except immediately after certain updates or when troubleshooting they're not show-stoppers, but I greatly prefer to have my systems working the way I've configured them rather than going off in other directions!

Anyone have suggestions as to what configuration files are likely to be involved?

### Post by tredegar on 2011-01-22
**gdmsetup** can set up autologin for you (System, Admin, Login Screen)
You need to "unlock" it with your password.

Networking not working may be a problem with ubuntu's "upstart" controlling the boot sequence. Sometimes it does not get it right, and tries to start things (like networking) too early in the boot process, so they fail.

So I'd put **sleep 20; service networking restart** into **/etc/rc.local** on the line before the final **exit 0**

But maybe you have your **/etc/network/interfaces** file misconfigured. Please post it.

### Post by tredegar on 2011-01-22
> The problem with that is that when I click "unlock" it blinks orange momentarily, but the unlocking dialog never shows up to ask for the password. 
Something is broken. Do you see errors when you run gdmsetup from a terminal and then click the unlock button?

Your interfaces file looks fine.

Your rc.local file is odd here:
```
# enable WAN and forwarding...
/etc/init.d/networking stop
sysctl -p
/etc/init.d/networking start
```
Be aware that you should now be using **service networking stop** and not calling things through the traditional init.d scripts
Try this to see what I mean: **sudo  /etc/init.d/networking start**
```
Rather than invoking init scripts through /etc/init.d, use the
service(8) utility, e.g. service networking start

Since the script you are attempting to invoke has been converted to an
Upstart job, you may also use the start(8) utility, e.g. start networking
networking stop/waiting
```
