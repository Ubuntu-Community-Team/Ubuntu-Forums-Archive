### Post by SeijiSensei on 2012-12-14
That looks like the output from iptables.  That's a UDP packet being sent to the broadcast address, 255.255.255.255, and intended for port 19540.  Is something listening to that port?  Perhaps you are running a BitTorrent client with DHT enabled?  What machine on your network has address 192.168.0.89?  That's the origin of the traffic.

You can probably exclude these from the log by writing a rule to ignore them like this

```

iptables -A INPUT -p tcp -d 255.255.255.255 -j ACCEPT 
iptables -A INPUT -p udp -d 255.255.255.255 -j ACCEPT 

```

That assumes this is valid traffic.  If it is not, then replace ACCEPT with REJECT.

Put this up toward the top of your iptables ruleset.

### Post by SeijiSensei on 2012-12-14
If you use ACCEPT, then yes.  What is likely happening is that packets that don't match any rules are being logged.

Is the machine that is logging the requests also the one with IP 192.168.0.89? If not, try to figure out why that machine is broadcasting packets to the all-ones address.

You can look up the services associated with a port number in /etc/services.  For instance, port 1900 is listed as supporting the "[SSDP]("http://en.wikipedia.org/wiki/Simple_Service_Discovery_Protocol")" service.  That file only lists officially "registered" ports, though.

### Post by JKyleOKC on 2012-12-15
To the best of my knowledge, the netfilter part of the kernel (that acts on the rules in iptables) does not log anything at all by default. It needs a special rule that jumps to the target "LOG" in order to write anything to the log files.

Check your rules to see if you have somehow gotten such a rule into place. If you're using UFW or GUFW, checking the firewall's status should tell you. In any event, you can find out with the command ```
sudo iptables -L |grep LOG
```If it returns anything, then you can remove the rule and all should be well. How to do that will depend on the firewall configuration techniques you use, but we can tell you how once we know the status.

### Post by JKyleOKC on 2012-12-15
It means that your set of rules for iptables is explicitly logging almost every packet that it drops or rejects. Those rules are **not** part of the default rule set, so they must be specified by some configuration program such as Firestarter or UFW. However they are not the logging rules that UFW and GUFW create. Do you know what such program you have in place?

While we could disable the logging action as you asked in the first place, it would be much better to determine just why so many "unknown input" packets are running around your network. I had a similar problem here, with packets addressed to port 7100 appearing every second. It turned out to be from a weather applet that my wife was running on her machine, that "called home" once a second to ask for the latest weather information to display in the task bar of her system. I solved it by adding a rule to my iptables set manually that would simply reject the packets without logging them. You could do the same if you want, provided you have administrator access to the rule set, but we would have to know just how your rules are being configured in order to give you the correct wording for the new rule(s)...

As others have said, there might be problems needing correction, on the machines that are originating these packets. Heavy network traffic of this sort can slow everything down.
