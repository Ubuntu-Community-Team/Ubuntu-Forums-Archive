### Post by srs5694 on 2011-04-29
First, when posting text-mode program outputs, please enclose them between [noparse]```
[/noparse] and [noparse]
```[/noparse] tags. That preserves legibility and, in the case of long output like that from Boot Info Script, makes it easier to scroll past the post as a whole. (Editing your original post to include these tags will help future readers of this thread.)

As to your problem, it seems that you've accidentally wiped out your Windows installation; it's no longer there -- or at least, its partitions aren't. Your disk now uses a GUID Partition Table (GPT) setup, which is an up-and-coming partitioning system that's uncommon on PCs except those that are based on the new (U)EFI firmware rather than a traditional BIOS. Your disk has just the two Linux partitions you described; there's no hint of a Windows partition on it. I don't know how you came to have this configuration; maybe you erred in setting things up, or perhaps there's a bug in the 11.04 installer that trashed your disk. One possibility is that you started with a formerly-GPT disk on a BIOS-based computer; then a combination of known bugs in the Windows and Ubuntu installers might have done the deed.

Since you say your Windows 7 install was "semi-fresh", you may just want to start over from scratch:


[list=1]
[*]In Ubuntu, install GPT fdisk (gdisk) by typing "sudo apt-get install gdisk".
[*]Type "sudo sgdisk --zap-all /dev/sda". ***This will completely wipe the partition table on the disk!*** This step is necessary because if you just try to install Windows, it will do an incomplete job of wiping the GPT data, leaving the Ubuntu installer thinking that the disk is blank. (That's one possibility for what happened the first time around....)
[*]Reboot into the Windows installer and install Windows.
[*]Reboot into the Ubuntu installer and install Ubuntu. Select the advanced/manual partitioning option and be sure that the installer recognizes the Windows partition(s) before you proceed.
[/list]


If the problem recurs, I recommend booting a live CD after each step and typing "sudo sfdisk -d /dev/sda > parts*n*.txt" (where *n* is a digit so you can keep each version separate), then copying parts*n*.txt to a removable disk such as a USB flash drive. That will keep a record of the partition table changes, which will be useful diagnostic information at worst and might enable you to recover the disk at best.

If you've already got valuable data on the Windows partition or if you've invested enough effort in customizing it that you don't want to start from scratch, your best bet is to use [TestDisk](http://www.cgsecurity.org/wiki/TestDisk) to recover your existing partitions. You may need to start with "sgdisk --zap-all /dev/sda", as above, to wipe the GPT data and enable TestDisk to recover the partitions as MBR partitions.

### Post by srs5694 on 2011-04-29
Sef, there is no evidence of a Windows installation on /dev/sda; the bit of the Boot Info Script output you quoted mentions ext4 on the data partition and an unknown filesystem on the swap partition. (I don't recall offhand if the Boot Info Script identifies swap as such or reports it as unknown; but I do know that it correctly identifies NTFS, so clearly the swap partition is not a mislabeled Windows installation.)

/dev/sdb has a FAT filesystem on it, but it's unpartitioned. I'm guessing that's a USB flash drive. (I recommend removing such disks when installing Linux or running Boot Info Script, unless you're installing to or from the flash drive.)

Also, the fdisk output you requested is redundant with the Boot Info Script output and is useless in any event, since the disk uses GPT, which fdisk doesn't understand. Windows cannot boot from GPT on BIOS-based computers, so unless the computer uses UEFI, the disk should be in the more common MBR format for Windows to work. As I wrote in my earlier post, one possible cause was a disk that was originally in GPT form (used on a Mac or with GPT for some other reason) being re-used for installing Windows and then Ubuntu; this procedure might easily result in precisely the problems described here.
