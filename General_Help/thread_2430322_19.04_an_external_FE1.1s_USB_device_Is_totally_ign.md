### Post by Holger_Gehrke on 2019-10-30
Since these devices normally are just webcams in disguise (and should be covered by the uvcvideo driver) there are several things to try:
[LIST]
[*]Just start a program that will display images from a webcam (guvcview, cheese, ...) and check whether you can select it as an input source
[*]Unplug the device, wait a moment, plug it back in and then check the messages the kernel left about the last few seconds with 'dmesg'
[*]Unplug the device, run 'lsusb', plug it back in and run 'lsusb' again. Compare the two sets of output. Does a new device show up in the second set ? Then get details with 'lsusb -v -s <bus>:<device>' (substituting the numbers after 'Bus' and 'Device' for the new device in the second output). This should give you a better idea of how the system sees the device.
[/LIST]

Holger

### Post by Holger_Gehrke on 2019-10-31
Have you tried opening the endoscope as an input in guvcview ? The output from dmesg looks almost exactly - including the failures to initialize - like the initialization of my webcam which does work with guvcview.

The usb id at the end of the first line of dmesg (058f:3822) is exactly the id on the first line of the second lsusb ('Bus 002 Device 022: ID 058f:3822 Alcor Micro Corp.'), so you might want to try 'lsusb -v -s 2:22' or better 'lsusb -v -d 058f:3822' since the numbers for bus and device will change every time you unplug and re-plug the device while the device-id stay constant.

Holger
