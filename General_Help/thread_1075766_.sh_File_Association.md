### Post by mb_webguy on 2009-02-20
A properly written shell script should have a shell specification line at the beginning of the file, along the lines of "#! /bin/sh".  If the script is written for the sh shell, then you can run it with sh, bash, dash, or any other sh-compatible shell.

If the script is written for a specific shell other than sh, however, it will only run in that shell.  The best way to run a script is to make it executable, then run it by simply calling it by it's name.  Therefore, if you have a script on your desktop that's called "sun_java_wireless_toolkit.sh", the command to run it (once you make it executable) would be "~/Desktop/sun_java_wireless_toolkit.sh".  And you should be able to just click on it.

Just because something ends with ".sh" does not mean that it **is** a shell script - that is just a convention that indicates to people that it **should be** a shell script.

The actual contents of the file could be anything - and the "magic number" of the file will indicate to Linux what application to use to run/launch it.

People should not get trapped by the DOS/Windows convention of filetypes identifying what is used to process particular files, that does not apply in the *NIX universe.

If a major software project distributes a file named "something.sh", and it's *not* a shell script, then they obviously don't know what they're doing, and I don't want their software on my system.  Linux doesn't require file extensions, but the sh file extension denotes a shell script.  If a file can be executed and has the sh extension, it's pretty much either a script or a misnamed binary.
