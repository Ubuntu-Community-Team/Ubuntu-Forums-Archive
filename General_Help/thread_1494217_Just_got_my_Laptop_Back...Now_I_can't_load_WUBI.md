### Post by bcbc on 2010-05-27
This bothers me c:\boot\grub\core.img - it's on your windows and I'm not sure why it's there. Windows isn't case-sensitive and you also have a "c:\Boot" directory, so this may be causing a conflict. Strangely windows boots fine - but I believe this could cause issues in the future with windows.

I'd boot under a linux live cd, mount /dev/sda2 and delete /boot/grub/core.img (directory and files).

There's also this which is probably caused by the above > =================== sda2: Location of files loaded by Grub: ===================


??GB: boot/grub/core.img

As far as recovering wubi goes - I can't see a problem with it from the script. What I would do is backup the virtual disks (c:\ubuntu\disks\root.disk is the important one) _to a completely separate directory_ (so it doesn't get removed when you uninstall). Then uninstall and reinstall wubi. After that simply copy the backed up root.disk over the new root.disk. That's probably the easiest method if your problem is to do with the booting mechanism.

You can also try and verify that the virtual file is ok by mounting it (from an ubuntu live cd) as described here: [https://wiki.ubuntu.com/WubiGuide](https://wiki.ubuntu.com/WubiGuide)

Here's an excerpt adjusted for your install.
> How can I access my Wubi install and repair my install if it won't boot?

Boot the Ubuntu Desktop CD, or another LiveCD, then mount the windows partition:

sudo mkdir /win
sudo mount /dev/sda2 /win

sudo mkdir /vdisk
sudo mount -o loop /win/ubuntu/disks/root.disk /vdisk

Now the content of the virtual disk will be visible under /vdisk. 

To check the filesystem you can use:
sudo fsck /win/ubuntu/disks/root.disk

All your ubuntu files are on your c:\ubuntu\disks\root.disk That's why you need to back that up first. The uninstall and install of wubi will create a new root.disk, but you can replace it afterwards with your old one (provided you stored it somewhere safe first i.e. an external drive or any folder not going to be deleted in the uninstall).
You should back this up anyway.

Second, when you say you looked on /, did you loopmount your root.disk file? Just checking, otherwise you are really looking at the install CD's root directory. I find it strange that something Best Buy did could damage a kernel file within a virtual disk without damaging the virtual disk itself.

I'm not really sure what else I could suggest. I sometimes chroot into my installs and run updates when they're not booting properly, but it doesn't sound like your ubuntu is actually loading, so I don't believe this will help you.
