### Post by DuckHook on 2023-02-25
Anything can be done with enough contortions, but I think the question you need to ask is: is it safe? I would never allow someone to access my LAN for backup purposes.

What I would do instead would be to ask the requester to set up her own standalone box in a properly configured DMZ outside my firewall, then set my router to forward only very specific IPs to very specific ports on that machine. Even then, I would have to have high confidence in the requester's tech skills to not unknowingly get pwned. I don't want my ISP suspending my account when it turns out that the backup machine is using my IP address for spam, DDoS attacks and as a malware relay.

Even with a standalone backup machine, I would not consider any other connection but a properly configured VPN.

If you really, really want to take the risk of connecting LANs over the internet, then the only method worth considering is a VPN. I would additionally use SSH on top of that. But unless you take special measures segmenting the two LANs that are involved, you will have full access to the other person's LAN and he to yours. The details of such a setup would be considerable and technical but it's doable.

### Post by TheFu on 2023-02-26
"Easy" is subjective.
Duplicity can do it ... so all the tools based on that should be able to as well.  Look for the scp/sftp method.  You'll need to get scp/sftp setup between the two systems which is fairly easy as far as most client/server methods are concerned, but some people just get confused.  What should be 30 seconds somehow becomes 3 yrs and they don't get it working.

If you are already making local backups using duplicity with encryption enabled, then why not use rsync over ssh to clone that data remotely?  Just be certain the backup files ARE encrypted.   [https://www.linux.com/training-tutorials/backing-linux-duplicity/](https://www.linux.com/training-tutorials/backing-linux-duplicity/)

Whatever you do, it is generally best to have the storage used using a native Linux file system, so ext4 would usually be desired.  If you are using zfs, there's a built-in remote backup method called zsend with a few very popular scripts to manage that available in github. The one from Jim Salter is one of those.

If the files are encrypted before you transfer them, then even using cloudy storage is possible. That's a judgement call.  Really what you want is assurances that someone will put your data in overnight mail if you ever need it back.  Personally, I'd never use cloudy services for any data that I didn't want shared around the world, since they all seem to lie, but duplicity has strong encryption, so if that works for you, great.  15GB should be enough for saving only the most important data that your wife would killing if lost - wedding and baby photos, along with critical financial information (which I keep inside a password manager, encrypted).

I'd never put unencrypted data into cloudy services.

### Post by TheFu on 2023-03-08
Using USB flash media completely changes my answer!

You can use LUKS encryption with it.  A how-to guide for that:
* Creating Part1: [Https://youtu.be/vk9Z2_rEUak](Https://youtu.be/vk9Z2_rEUak) (nerds should find this very funny)
* Accessing Part2: [Https://youtu.be/ELEVo6SbYl0](Https://youtu.be/ELEVo6SbYl0) seems to be it.
* Text version: [Https://baldnerd.com/add-a-drive-to-linux-and-encrypt-it](Https://baldnerd.com/add-a-drive-to-linux-and-encrypt-it)

For backups on flash storage like a thumbdrive or SDHC media, you'll need to use f2fs as the file system.  Just install the f2fs-utils package and format that file system inside the LUKS container.

You'll never want just 1 flash storage device.  Get 2 and swap out which you use every other week.  Perhaps leave one at work for a week between swaps locked in your desk.

I'd rather see you with (2) 64G microSD cards than (1) 128G thumb drive.  Flash storage has a terribly finite lifespan, so expect it to fail.  

Or get a 240G m.2 SSD and put it into a $12 USBc metal enclosure.  Then you can treat it like any SSD you know and expect 3-5 yrs of lifespan.  I'd guess that would be $45-ish for the m.2 240G SSD and enclosure.  Plus, performance will be far better than any flash drives.  If you really want great performance, get an NVMe m.2 and a USB 3.2 enclosure which supports upto 10Gbps xfers.

### Post by TheFu on 2023-03-17
> **Robbyx said:**
> Thank you for your hopeful answer. 

My USB 3 ports have never worked for devices I connected to them.  


That isn't good.  Something is very, very, wrong.  I had an issue around 2011 with a USB3 hub. The vendor created a firmware patch for the hub and it has worked perfectly since that time. I did add a PCIe USB3 card to that system - it was so old that USB3 didn't exist.

> **Robbyx said:**
> 
"I notice no performance loss, mine are plugged to computer port though."
 
I am wondering how you were able to plug the USB 3 ports into the computer port (Does the computer port have another name that I might recognise?)

If you use **lsusb -t**, all the ports and connected devices should be shown in a tree with the bus, port, and device number. With this information, it is possible to trace the PCIe slot used all the way to the device.  Sometimes USB plugs fail, so try a different one.  Incompatibility with USB3 usually only happens when a computer is very old. A $30 current PCIe card with 2-4 USB3 ports can solve that.  While with disk storage, you won't see 5Gbps (that's the theoretical bus limit), spinning disks can get to the 150Mbps range easily.  SATA SSDs connected via USB3 should get 400-550 Mbps and NVMe SSDs connected via USB3 should get 3000 Mbps easily.  USB3.2 goes to 10Gbps. My Ryzen systems have 3 USB 3.2 ports built-in. They have a red USB tab, not the blue tab usually seen in USB3.0 ports.  All newer USB standards are backwards compatible ... er ... 99.9999% of the time.  Boot flash drives seem to be an exception on some systems.

> **Robbyx said:**
> 
Is flash storage via a PCIe bus hugely superior to USB 3 (I assume that using the PCIe is not as mobile as USB 3 ie I assume I can not easily unplug the PCIe storage and take it home.)

All USB storage has the problem that the connection isn't as solid as eSATA.  But USB is almost universally available on all computers today.  eSATA provides the complete SATA command set and performance, just for external devices. It is important for storage that will be permanently connected.  

> **Robbyx said:**
> 
I found an article that claims the PCIe bus supports tens of thousands of parallel command queues and I doubt if USB 3 can do that. Am I right to assume the PCIe bus is for running the os, apps, and storage, whereas USB 3 is primarily for ops not involving the need for parallel command queues?

USB has queuing issues, sometimes they are major and destroy performance.  USB3 is great for portable storage, especially backups.  I've been burned by loose USB connections. Not just the disconnection, but also corrupted data.  For that reason, I avoid USB for storage, except when there's no other choice.  My systems do have eSATA-PM ports available, so I don't have to use USB for primary, external, storage.  I do have some external USB3 storage connected, but it is used for backups only.

Everyone has different experience with USB. Some people don't use any internal storage on their MX Linux systems. They boot and run from USB3 storage 100%.  They decided that having everything on a physically small flash drive that was fast enough was a good compromise for their needs.  It also provides them with freedom to think of the computer and all their data/settings as being completely separate things, since there isn't any internal storage at all.  If I were going to run that way, I'd put an SSD into a USB3.2 enclosure and run that way.  I have a small USBc SATA-SSD enclosure. It is quite nice and fast enough. The m.2 SSD that fit inside aren't the normal 2280 size, but the smaller 2242 size.  "SSK Aluminum" makes cheap USB 3.2 enclosures. Just be certain to get the type you want with the local interface you need.

For backups, I'd get a 2TB, $40, externally powered, USB3 spinning HDD from a well-known vendor. I wouldn't get SSD, not for backup data.

### Post by TheFu on 2023-03-17
Er ... most people do simple performance tests of their storage.  There's a KDE storage testing program that creates output very similar to all the screenshots we see posted in storage reviews online ... like on Amazon. I don't remember the program name.  I usually use rsync with some random files I have lying around.
```
time rsync -ar --stats --info=progress2 {source} {target}
```
If you are testing copying 1 file, then just make 1 file the {source}.  If you are testing copying 500,000 files, then find a directory with that number and total size for your test.

Of course, **fio** is a good tool for testing. So is **bonnie++**, but the output and different scenarios are beyond what 95% of users want/care to see.

Also, realize that performance will depend on the slowest part of the entire path that the data traverses.  That could be flash storage or network or a bad controller seeing resets every 2 seconds.  Even within the same computer, using storage connected via different buses can matter greatly since 1 bus may be oversubscribed with data for the GPU. Storage on that same bus will be slower.  It comes down to  ... you never know until you test it in the real-world setup with real-world data.
