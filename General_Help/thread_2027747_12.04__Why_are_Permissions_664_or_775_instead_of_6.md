### Post by L a r r y on 2012-07-16
In all of my experience with Permissions, I am better acquainted with the numeric values assigned to R, W and X, which are Read, Write and Execute.

If you can read a file, you see its existence, you can read its contents.

If you can write a file, you can create, edit, and delete.  Read permissions are almost always present if you have write permissions.  I say "almost" just to cover the possibility that someone has created a system whereby you can write, but cannot see what you wrote! 

If you can execute a file as a script or program, you can run the script or program.  Read permissions are almost always present if you have Execute permissions.

Read-only has a permission value of 4, write a permission value of 2 and execute a permission value of 1.

Read-and-execute is 4+1=5, read and write is 4+2=6,  Read, Write and Execute have a permission value of 7 (4+2+1=7).

Linux Permissions have three sets of permissions on each file:  Owner, Group and everyone else.

The Owner is the creator or importer of the file, his Group may contain other users who will be limited by Group permissions on Owner's file, then there is everyone or everything else, who neither are the owner nor a member of the owner's Group.

It is this system of Permissions that keep Windows viruses at bay, for if you download an executable script, you must first give it executable permissions.  It can't just be downloaded and self-execute on your Linux system.

**Now, having laid out the way Permissions work, my past experience has always been to see Permissions of 755 and 644.**  (Owner has read-write-execute, Group has read-execute, everything else has read-execute [755] or same again minus the execute flag [644].)

UBUNTU 12.04 is customarily dishing up 775 and 664 permissions, giving the Group the same permissions as the Owner.  I am forever changing these settings to the customary 755 and 644.

Another way of expressing 775 is rwxrwxr-x, and 664 as rw-rw-r--.

Can anyone shed any insight on this?

Thanks.

The most direct answer is that the default *umask* is now 002 where prior versions were 022.  But that really doesn't tell the whole tale.  @steeldriver is correct as to how Debian style distro user and group entities interact, but that is only part of the story too.

Starting at the beginning; the linux kernel creates permissions of 777 on all ***directories *** and 666 on all ***files***.  The *umask* then filters the settings to what the system administrator wants.

Why the difference between directories and files?  The execute bit is used differently on directories than on files.  The execute bit allows traversal of the directory (decent into the subdirectory).  Therefore you can control access to directories with the execute bit settings on directories.

On files you have the action of the permissions correct, but maybe not so much the philosophy quite right.  Here is where the UPC notion is important to understand.  Debian has made the decision that all users by default will use their private group when creating directories or files.  This means that user ***bob*** will always by default have ownership of ***bob:bob***.  Also by default the user *bob* is the only user that has write permissions, regardless of whether the permissions are 664 or 644.

So why the change?  Because Debian/Ubuntu users are more likely to use the system with multiple users where the group is not private (e.g group = domain_users).  This is more like the way UNIX (in my case SUN Solaris) is used.  Since it doesn't matter to the single user (or shouldn't) then the setting of 775 for directories and 664 for files is better suited to multiple user environments.

I have shared directories outside of */home* that I control by the group rather than the user.  For example all of my Windows shares (samba) are at */smb*.  The group is *smbusers*. All users connecting to the various shares are a member of the *smbusers* group.  This includes the anonymous guest user which is configured as smbguest.

All users create files and directories with the group being *smbusers*.  This is accomplished by setting the group ID bit (sgid).  This is done by setting the permissions on the /smb share as 2775 (yes, there are 4 bits of permissions).  All files and directories below /smb inherit the group you set.  You need to set the group with *chown* or *chgrp*.

Once this is set the /smb file system acts like UNIX or BSD with sgid while the /home directory still has User Private Groups where the group permissions mean nothing.

The take away is this: the file system is more flexible than you think.  There can be both single user systems and multi user systems.  Linux can accommodate them both on the same host.
