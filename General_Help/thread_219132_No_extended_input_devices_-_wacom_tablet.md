I don't know if this is related or not, but I ran into the same problem about a year ago and finally figured it out today.  Depending upon the version of GTK+ that you are using, you may need to specify --with-xinput=yes when you build GTK.  I never had to do this in the past, but when I updated to the most recent GTK, my tablet stopped working.  Today I stumbled upon this and it fixed my problems.

### Post by tboss on 2008-02-25
Before attempting to build GTK+ from source, here are a couple more things to check:

1) Is the correct device specified in xorg.conf?  Typically, /dev/input/wacom is a soft link to something else, like /dev/input/event0 or /dev/input/event3.  Look in /proc/bus/input/devices.  You are looking for a series of lines that look like this:

I: Bus=0003 Vendor=056a Product=0011 Version=0203
N: Name="Wacom Graphire2 4x5"
P: Phys=
S: Sysfs=/class/input/input3
U: Uniq=
H: Handlers=mouse1 event3
B: EV=f
B: KEY=1c43 70000 0 0 0 0
B: REL=100
B: ABS=10003000003

The line "H: Handlers=mouse1 event3" is the one of interest.  That tells you that the correct device to put into xorg.conf is /dev/input/event3.  Or that /dev/input/wacom should be a link to /dev/input/event3.  What happens to me frequently is that when my computer is rebooted, the Wacom tablet is no longer attached as the same device.  Sometimes it's event3, sometimes event5, etc.  I have to verify this each and every time I reboot my computer.  If you don't attach and detach a lot of USB devices, this won't happen to you.  But it happens to me a lot because I connect and disconnect a lot of USB devices.

2) Does the tablet show up correctly in Xorg.log.0?  Try this:
grep XINPUT /var/log/Xorg.0.log

You should see a series of lines that look like this:

(II) XINPUT: Adding extended input device "Eraser" (type: Wacom Eraser)
(II) XINPUT: Adding extended input device "Pen" (type: Wacom Stylus)
(II) XINPUT: Adding extended input device "Cursor" (type: Wacom Cursor)

If you don't see those, there is something amiss in your xorg.conf file.

3) If all the above looks correct, let's see if the way GTK was built is the culprit.  NOTE: This fixed my problem, but the problem was self-inflicted.  I updated GTK / GLIB from source (for something else I was working on -- don't ask....)  When you build GTK from source and use the default configure options, it does NOT support XINPUT.

To prevent clobbering already installed software, let's build GTK+ and GIMP from source and install them into their own directory (for this example, let's use /opt/gimp).

Building gtk+:

./configure --prefix=/opt/gimp --with-xinput=yes
make
make install

Building gimp:

This can get a little tricky, because you want gimp to use the gtk that you just built.  I created a script (called myconfig.sh) that looks like this:

----------------------------------------------------------------------------------------------------------
#!/bin/sh

CFLAGS=-O2
CXXFLAGS=-O2
PATH=/opt/gimp/bin:$PATH
GTK_PATH=/opt/gimp/lib/gtk-2.0
LD_LIBRARY_PATH=/opt/gimp/lib
PKG_CONFIG_PATH=/opt/gimp/lib/pkgconfig
export CFLAGS CXXFLAGS LD_LIBRARY_PATH PKG_CONFIG_PATH GTK_PATH

./configure --prefix=/opt/gimp
exit $?
----------------------------------------------------------------------------------------------------------

Next, build and install gimp:

./myconfig.sh
make
make install

Finally, when you run gimp, you again need to make sure it's picking up the GTK libraries from /opt/gimp and not from somewhere else on your system.  To do this I created a script that looks like this:

----------------------------------------------------------------------------------------------------------
#!/bin/sh

GTK_PATH=/opt/gimp/lib/gtk-2.0
PATH=/opt/gimp/bin:$PATH
LD_LIBRARY_PATH=/opt/gimp/lib:$LD_LIBRARY_PATH
export PATH LD_LIBRARY_PATH GTK_PATH

exec gimp $@
exit 1
----------------------------------------------------------------------------------------------------------

IMPORTANT SAFETY TIP: The above just rebuilds GTK itself, it doesn't rebuild gtk-engines or QtCurve and a bunch of other stuff upon which GTK relies.  You might get some weird error messages when you start gimp, like warnings that it can't find libqtcurve.so.  But gimp should essentially work, at least well enough to see if your tablet is now working as an extended input device.  If it fixes the problem of the tablet not showing up, then at least you know that the GTK build was the culprit.  Then we can go about creating a more elegant, permanent fix.

### Post by spayced on 2008-02-27
I believe I just got mine working (for now) pressure sensitivity and eraser are working. Wacom Graphire 1 (usb) here.

 First I did all the tests on the driver that some threads have suggested. Everything looked good. So In a bout of inspiration opened xorg.conf and found this line.

[INDENT]Option        "Device"        "/dev/input/event" [/INDENT]

Then I decided to open device manager  (System, Administration, Device Manager) in order to look for something that resembled that line. 
Mine was hidden in Graphire, Wacom Graphire. 
On the advanced tab, a line next to input.device said "/dev/input/event2" I replaced that on my xorg, restarted x, 

[INDENT]Option        "Device"        "/dev/input/event2" [/INDENT]

and nothing happend. 

Since I'm insane I decided to do the same thing again expecting different results. I browsed device manager again, this time the line next to input.device said "/dev/input/event3"  Which is different. I have a migrating usb device? Anyways I just took that, plugged it in, 

[INDENT]Option        "Device"        "/dev/input/event3" [/INDENT]

and it works now. So bizarre but that's what I did, I hope it helps you, and I hope mine continues to work.
