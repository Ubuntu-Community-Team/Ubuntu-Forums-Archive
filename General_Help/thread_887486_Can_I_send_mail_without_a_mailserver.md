> **G1ZmO65 said:**
> Thanks cszikszoy, Could you explain a little more of what that command does for me? and "checkinstall" too?

Checkinstall is a useful program that is *sort of* a hack.  It's a very useful hack though.  What it does is creates a deb package of whatever software you are trying to compile from source.  When you compile things from source, it often puts files all over the place.  If you ever want to remove this program that you compiled and installed from source, good luck tracking down all of the files.

So, when you install some source with checkinstall, it creates a deb, and allows you to easily remove it because it shows up in synaptic.

```
sudo apt-get install checkinstall
```

There's currently a bug with the repo version in Hardy, but it's easy to get around.

To use checkinstall, go through compiling source files as normal, except for the last step.

```

./configure
./make
##last step is different##
./sudo checkinstall -D --fstrans=no make install

```

the -D flag makes a debian package, and the --fstrans=no turns off filesystem translation (this is needed because of the bug in the 8.04 repo version of checkinstall I was talking about).

From there, checkinstall will ask you to fill out some information about the "package", like a summary and version number.  This is just for the information on your system, so it really doesn't matter what you put here.

There's more information about it on the debian administrator's website, here: [http://www.debian-administration.org/articles/147](http://www.debian-administration.org/articles/147)

### Post by Subban on 2008-08-13
*Ok, I wrote this up before lunchtime but the server rudely went down and the forums broke before I could post it, so this is late and Brian_P has given you more great advice. I though this information could still be helpful so here it is.....*

> Thanks brian_p, I made a little script to test the email sending which worked but my original intention was to regularly email myself some sort of server status message. Any idea how I'd go about this?
I would probably use the command 'tail' to snip off XX number of lines from whichever log file/command was relevant to what I was checking, perhaps the output from 'dmesg' command as a general check, for specidic services then what ever log files were relevant in /var/log and perhaps its sub folders (and possibly other locations app dependant)

> This is my script:

```
sendEmail -f paul@mydomain.com -t paul@workemailaddress.com -u Ubuntu Server Status -s mail.mydomain.com -xu myusername -xp mypassword -m Here is the message from the server
```

Well, thats just a command rather than a script. A script is a file containing one or more commands which are run in a sequence, you would then call the script by its name parsing any data to it necessary if required.

If you want to run this on a schedule it is likely you would be better putting this lot into a script and calling that from cron, rather than run a longer sequence from the crontab directly.

Your cron entry might look like this:
```
59 23 * * * /home/paul/bin/serverstatus  > /dev/null 2>&1
```
Which would run the script 'serverstatus' from /home/paul/bin at 11:59pm daily.

Then in your home folder, I am going to assume its /home/paul so change any references if its wrong. do the following:

```
mkdir bin
```
we are making a folder that bash will automagically add to your $PATH

```
touch serverstatus
```
this makes the empty file called serverstatus

```
chmod +x serverstatus
```
this will make the file executable, so the system can run it.

```
nano serverstatus
```
open a console text editor to start editting the file, change nano to your prefered editor eg. vi, vim, emacs etc

Next, enter this script. Please note I am writing this off the top of my head and don't use sendEmail to test it.
```

#!/bin/bash
#
# small script to email tail of dmesg to check server status
#
# 
# Here we run dmesg and pipe it into tail, which snips off the
# last 5 lines and saves them into /tmp/ss.txt

dmesg|tail -n 5 > /tmp/ss.txt

#
# next, this is basically the line you posted above, but the last
# part is changed, instead of supplying a specific message we tell
# it to read the message from the file /tmp/ss.txt
# you can test this line seperatly by calling it from a command line
# to check for errors in case I got it wrong.

sendEmail -f paul@mydomain.com -t paul@workemailaddress.com -u Ubuntu Server Status -s mail.mydomain.com -xu myusername -xp mypassword < /tmp/ss.txt

```

It is trivial to change the log file output that is mailed if required, dmesg is used as an example. you can also append extra information to the file before emailing it, in the following code line, if you changed the > into >> it will append to the end of the file and not overwrite it, so you can add extra information into the file to send in one pass.

```
tail -5 /var/log/somefile > /tmp/ss.txt
```
instead of the line
```
dmesg|tail -n 5 > /tmp/ss.txt
```

You can print your own lines into the files using 'echo' and piping that in, perhaps to add headers to the email between the sections. such as:

```
printf "/n"
echo "Dmesg output" >> /tmp/ss.txt
echo "--------------"
```

I've thrown a printf in there as something else you can look up, it has more powerful options to layout the text as you want.

I hope this at least helps a little.
