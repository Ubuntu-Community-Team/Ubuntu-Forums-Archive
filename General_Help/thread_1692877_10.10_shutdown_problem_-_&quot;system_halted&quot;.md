### Post by el_Paraguayo on 2011-02-22
I'm running Mythbuntu 10.10 on a new build machine.
 
Everything seems to be running pretty well so far apart from this one issue. It won't powerdown when I try to shutdown the machine.
 
I've got the nVidia drivers installed (and haven't tried fixing the splash screen yet) so I get the "Ubuntu 10.10" text and the coloured dots when shutting down. 
 
The screen says "System will now halt" and you can then hear the harddrives power down but the display and CPU fan stay on. The only way to shutdown is by holding the power button in.
 
Interestingly, the system reboots just fine - (I've not tried suspend/hibernate yet).
 
I've looked at a number of solutions:
 
*1) Myth backend process*
 
Having looked through some logs I noticed that the mythbackend process was being stopped but respawning almost instantly. I can fix that my changing the myth-backend.conf file but that didn't change anything in terms of the shutdown working.
 
*2) Boot options*
 
[This thread]("http://ubuntuforums.org/showthread.php?t=1597567") suggested adding "acpi=force" to boot options.
 
That didn't work for me.
 
Another thread suggested "acpi=off". No dice.
 
At one stage I did get to see the screen say "system halted" (I think this was from removing the "quiet splash" options) - but the same symptoms as before (i.e. drives power down, monitor & CPU still on).
 
*3) Blacklisting wireless drivers*
 
[This thread]("http://ubuntuforums.org/showthread.php?t=1603642") suggested some wireless drivers could be the issue and they should be blacklisted. 
 
Given that I'm not running wireless I didn't see any issue in blacklisting the drivers. No joy.
 
*4) BIOS power settings*
 
My BIOS gives me the option of S1 and S3. S3 is the default. Switching to S1 made no difference.
 
*5) Deactivated Nvidia drivers*
 
I tried deactivating the proprietary drivers and hoped this wasn't the answer as I need them to run XBMC - shutdown still not working.
 
 
So... I'm all out of ideas now.
 
I've reported this as a bug so if this sounds similar to what you're experiencing please say it affects you too - ([https://bugs.launchpad.net/ubuntu/+bug/721706](https://bugs.launchpad.net/ubuntu/+bug/721706))
 
 
If anyone's got any ideas on helping me debug this, I'd be eternally grateful.
 
System info:
 
Motherboard: Gigabyte 870A-UD3 (rev 2.1) 
CPU: AMD Athlon II X2 245e 
CPU Cooler: Scythe Shuriken Rev B
GPU: MSI M210-ND512H 
Memory: 4GB (&#65279;&#65279;&#65279;4GB kit (2GBx2), 240-pin DIMM, DDR3 PC3-10600) 
HD1: 160GB 7200rpm 
HD2: 2TB 5400rpm

### Post by robert shearer on 2011-03-02
irq assignment ouch. :(

> Some BIOS will list the IRQ configuration at boot time. If you're quick with a camera or know how to use the keyboard Pause button to suspend the processor, you can get a look at the list of IRQs assigned by the system.

> Conflicts on a shared IRQ for onboard USB and a slot in the motherboard are common, especially if the card is a sound card or internal modem. Check the BIOS IRQ assignments, if possible, and do not put cards in slots that are shared with on-board devices. All BIOS are different so you will have to look for something like "PnP/PCI Configuration", "IRQ Resources", or "PCI IRQ Assignment". Consult the manual for your motherboard.

Also check in bios for something along the lines of 'reset (irq) configuration' so as to allow the bios to re-sort assignments automatically when new hardware is installed.
 This is usually a one-off setting and re-sorts irq on next boot then maintains that configuration until next time the setting is activated.

Do research this in the motherboard manual as I am mentioning a generic setting that may or **may not** be present for your hardware.
 You don't want to reassign irqs randomly as you may end up with worse conflicts or even [B]boot failure.
[/B] :(

Are there usb devices mounted and if so what happens if these are unmounted and removed before shutting down... ?

### Post by craigchambers on 2012-06-07
I have the cx88 modules issue caused by my Hauppauge WinTV Nova S2-HD and have found a fix that works for me on Ubuntu 12.04...

> **natdan said:**
> Only thing that worked at the end was what was explained here:[https://bugzilla.redhat.com/show_bug.cgi?id=630420](https://bugzilla.redhat.com/show_bug.cgi?id=630420)


Using the commands in the script above, I created a SysVinit script that I added to run right before S90halt

```
#! /bin/sh
### BEGIN INIT INFO
# Provides:         unloading cx88 kernel modules
# Required-Start:
# Required-Stop:
# Default-Start:
# Default-Stop:      0
# Short-Description: fixes shutdown by unloading cx88 kernel modules
# Description:
### END INIT INFO

do_stop () {
        killall pulseaudio
        rmmod cx8800 cx88_dvb cx24116 cx22702 cx8802 cx88_vp3054_i2c cx88_alsa cx88xx
        exit 0
}

case "$1" in
  start)
        # No-op
        ;;
  restart|reload|force-reload)
        echo "Error: argument '$1' not supported" >&2
        exit 3
        ;;
  stop)
        do_stop
        ;;
  *)
        echo "Usage: $0 start|stop" >&2
        exit 3
        ;;
esac

```

Create the file as above using your favourite editor.  e.g...
**sudo vi /etc/init.d/fixshutdown**

Paste the script above into the editor, then save it.

Make it executable...
**sudo chmod +x /etc/init.d/fixshutdown**

Use update-rc.d to place the file in the rc0.d (shutdown) folder.

**sudo update-rc.d fixshutdown start 89 0 .**   <== (don't miss off the ".")

From now on, the script runs right before 'halt' is called, and my PC switches off :-D
It may be worth checking where 'halt' is called in your /etc/rc6.d folder (as I say mine is at S90).

To remove the script run...
**sudo update-rc.d -f fixshutdown remove**

(OK, so it's not an upstart job, but I'm not really up on them!)

Once that was done, I re-added the "splash" and removed the "acpi=force" entry from my grub settings without breaking my shiny new fix.

HTH,
Craig
