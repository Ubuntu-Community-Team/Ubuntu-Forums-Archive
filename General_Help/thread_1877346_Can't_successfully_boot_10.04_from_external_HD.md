### Post by oldfred on 2011-11-08
Script looks normal.

For whatever reason you installed to sdc, but it is sdb now. But since Ubuntu/grub use UUIDs that does not matter as the search overides settings of drives in grub & fstab is based on UUID.

I think it is BIOS issue but have not found the correct setting that may cause it. There was an issue with old systems that would not boot from a partition that had boot files above 137GB. I think some newer systems have BIOS legacy settings that work the same way. Make sure BIOS is set to LBA, disable quickboot, and possibly AHCI mode?

You also can just try shrinking your / (root) as there was an older bug in grub2, that it could not find files with large separation or in large / partitions. I normally suggest a 20-25GB / partition and use the rest of the drive as /home anyway. 

Please try the shrink and if that does not work, then reinstall grub2 to the MBR. I have had this work for about half of those I have suggested it to.

### Post by oldfred on 2011-11-09
With the larger drives, partition in advance using gparted and then use manual install or Something else to choose which partition is / (root) and which is /home. If a partition already exists as swap it will automatically find it. You also do not have to [COLOR=DarkRed]fully[/COLOR] partition in advance. I did not with my 650GB drive, and now keep adding another 20 or 25GB / partition for each new release of Ubuntu. I have all my data in two 100GB data partitions, one NTFS for sharing and one ext3 for Linux. My old install of XP is on another drive which I now use very little.

GParted partitioning software - Full tutorial 
[http://www.dedoimedo.com/computers/gparted.html](http://www.dedoimedo.com/computers/gparted.html)
Screenshots of using gparted
[http://www.howtoforge.com/partitioning_with_gparted](http://www.howtoforge.com/partitioning_with_gparted)

For the Total space you want for Ubuntu:
Ubuntu's standard install is just / (root) & swap, but it is better to add another partition for /home:
1. 10-20 GB Mountpoint / primary or logical beginning ext4(or ext3)
2. all but 2 GB Mountpoint /home logical beginning ext4(or ext3)
3. 2 GB Mountpoint swap logical

Depending on how much memory you have you may not absolutely need swap but having some is still recommended. I do not hibernate (boots fast enough for me) but if hibernating then you need swap equal to RAM in GiB not GB. And if dual booting with windows a shared NTFS partition is also recommended. But you usually cannot create that as part of the install, just leave some space. Or partition in advance (recommended).
One advantage of partitioning in advance is that the installer will use the swap space to speed up the install. Thanks Herman for the tip.

### Post by efflandt on 2011-11-09
I have had this same issue ("Error: file not found" and "Grub Rescue") with a couple of computers, one which might be expected with a PC from 2004, but the other totally unexpected with a PC from 2010 (Dell XPS 8100 in my sig).

It is especially unexpected in the later case of the new PC because I currently run 64-bit Maverick from a primary partition at the far end of its 1 TB internal drive (I only recently installed 11.10 on separate internal SSD). It will boot Linux fine installed near the end of a 160 GB USB hd (with grub in its MBR).  But it throws those errors if I attempt to boot ubuntu at the far end of a 500 GB USB drive. 2 laptops from 2006 (one a Dell) have no trouble at all booting the same 500 GB (or 160 GB) USB drives. BIOS is set for RAID even though no drive is configured for RAID because Win7 will not boot with the other setting.

Strangely if I boot the internal drive into a regular grub prompt from the menu (not grub rescue) I could see the root directory of the USB drive, and maybe contents of boot, but could not see contents of grub directory (hence, file not found).  Yet from grub on the SSD, I had no trouble seeing anything on the 1 TB internal drive (my Linux partition is about 900 GB out).  And if I make the SSD (sdb) the boot drive, it boots Linux fine on sda.

It is still a mystery to me why some computers have no trouble at all booting far out on an internal drive, but refuse to boot less far out on an external drive (DOS partition table with ext3 or 4 primary Linux partitions) or exactly where that cutoff point is.

So if you buy a new big external drive it may be a good idea to initially create partitions(s) for Linux lower on the drive, just in case you run into this issue, which is quicker and easier than trying to move an existing partition up out of the way later.
