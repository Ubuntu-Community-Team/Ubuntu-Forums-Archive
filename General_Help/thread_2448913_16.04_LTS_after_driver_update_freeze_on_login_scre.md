### Post by TheFu on 2020-08-16
resolv.conf is just a text file that tells any Unix-like OS where to go for DNS.
```
echo "nameserver 1.1.1.1 
nameserver 1.0.0.1 " | sudo tee /etc/resolv.conf
```
will create one with clourflare's DNS servers. Copy/paste those 2 lines (it is critical that 2 lines are used), then run them on the impacted system.

Should be root:root 644 permissions. When I tested it, that happened automatically.
If systemd or resolvconf is trying to control name resolution on the system, after 5-10 minutes or after the next reboot, it will be replaced with those.  Because I run a caching, local, DNS server, I don't use either systemd or resolvconf. 

If you cannot **ping 1.1.1.1 **, then the problem isn't just the DNS setup, but networking too. People often confuse DNS problems with internet problems, unfortunately.

### Post by TheFu on 2020-08-16
Editing system files is a standard skill to learn. There are multiple ways.

Use sneakernet with a flash drive?
Boot from a "Try Ubuntu" flash drive, mount the storage, find the file, edit it with sudoedit?  Paste the 2 lines inside?
But it won't help if you can't **ping 1.1.1.1** successfully.

apt autoremove wouldn't remove network drivers or video drivers.
Unless the system is has been on sale less than 6 months, it is unlikely any needed intel drivers aren't automatically included. In general, don't go looking for drivers - only use those provided through Ubuntu software management tools.

There are some issues with new Intel 2.5Gbps NICs.  Others trying to get those working have ended up just buying a new NIC with a well-supported Intel PRO/1000 chip.

BTW, I don't do unattended patching for a number of reasons. Been burned a few times. Once a DBMS server was taken down for a few hours - seemed that one package decided mariaDB wasn't a suitable replacement for mysql, so it removed mariaDB which was running about 10 DBs for production systems.  Never again.
