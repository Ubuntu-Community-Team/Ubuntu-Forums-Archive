### Post by ajgreeny on 2017-01-04
What version of Ubuntu are you using?
Are you using encryption or LVM partitions, as otherwise a separate /boot partition is not necessary.
Most importantly, **DO NOT DELETE ANYTHING FROM BOOT USING THE *rm* TERMINAL COMMAND.** Always do that with the package management system or you will end with a very unstable system.

I am fairly certain that with that small amount of free space you will not be able to solve this problem with terminal command ```
sudo apt-get autoremove
``` but it's worth trying before we go any further.

If it is unsuccessful show us the output of terminal command ```
dpkg -l linux-image*
``` to see what kernels are installed and once we know that we can start removing the old unwanted kernels with **sudo dpkg -P** commands.

### Post by ajgreeny on 2017-01-06
**Oh dear!**

Autoremove will have failed as there is insufficient headroom, so don't worry about that at this point.

Having used **rm** to delete some files from /boot already you are probably going to have problems as I don't think dpkg will work to remove kernels associated with those missing files and you will get an error message saying it can not complete as files it's trying to remove are not found.

This is precisely why it is so important not to start playing around with files in system folders if you don't know what you're doing.
If, however, you know the names of the files you removed or can find the command in **.bash_history** in your home folder, you may be able to create new same-named empty files with ```
sudo touch /boot/*filenames*
``` which will give the package manager some files to work with; it will not care if those files are empty or broken as long as they exist.

I do hope that helps you.
To see where we have got to so far in terms of the /boot free space can you show the output of command ```
df -h /boot
```
We can then go on from there and see what is the best way to move forward.

### Post by oldfred on 2017-01-06
For current install ajgreeny is correct, never use rm to delete system files. But as with any rule, there may be exceptions.

If kernels are from old install and you upgraded before housecleaning then you may have to use rm as they will not be in the new installs inventory of installed applications for dpkg to manage.

But kernels & related programs are not just in /boot, but several other places and that is difficult to find & houseclean.
So best to recreate filenames or reinstall (newer) kernels & use dpkg to fully/correctly remove.

### Post by ian-weisser on 2017-01-06
> **asb3 said:**
> I'm sure I'm not the only one who has done this.  The software updater should either remove the old versions automatically or be more explicit about what the user has to do.

You are not the first person to unwisely use rm on package-provided files.
Repairing the damage is easy - simply read the 'file not found' error message carefully, and create a dummy file in the correct location for the package manager to remove.

The software updater does remove old versions automatically (new in 16.04). However, it's logic does not include the corner case of four half-configured kernels at once. It also won't work around 'file not found' failures. Note that future versions of Ubuntu can boot directly from the LVM partition, so separate /boot partitions are already on the path to obsolescence.

The dpkg error, as written, does try to give as much information as the system knows. dpkg and apt pre-date LVM and encryption and their tiny /boot partitions; the developers did not foresee the growth of tiny /boot partitions, and few users have seemingly cared enough to contribute changes or improvements. Any community member who wishes to improve the error messages, or (better) to improve the software so it doesn't fail, is welcome to join the dpkg and apt developers and contribute their time.

### Post by ian-weisser on 2017-01-12
That's right. initrd.img is a file, not a package.

### Post by oldfred on 2017-01-13
The links are not normally used. So you can probably ignore that error. When next new kernel is installed links should get updated.

They are a simple way to refer to current & 2nd most current kernel if manually having to boot from a grub rescue. 
Or if you use 40_custom to boot another install you can use the link to always boot the most current kernel without having to go thru multiple boots into each install and updating grub in each install.
