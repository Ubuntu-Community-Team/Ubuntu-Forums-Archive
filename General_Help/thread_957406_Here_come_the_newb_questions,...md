@ jerome1232:
Of course it's 'hard to replace', simply because there is no such complete suite on any other operating system. You have to use lots of different programs with different interfaces which are not very consistent between each other. That's something Apple has done very well.

And then there's wine, which in many cases (and increasing daily) makes dualbooting with Windows superfluous.

@Corelogik:
Audio Hijack sounds nice, but first you have to consider the nature of a Linux system: it is completely open. You can do everything you want and the system doesn't try to impose barriers on you or hide something. Administrator privileges are of course required, but you can for example directly edit every aspect (like RAM content, access to system interfaces, ...) of an open process via your proc folder.
With this in mind, accessing the system's audio sources is trivial.
The downside (for the moment, that is) of all this: it rather quickly gets technical, and the (command line) tools for accessing the system resources are, while often very sophisticated, not user (/newbie) friendly at all.
Now the good thing about Ubuntu is that it tries to improve this issue and include GUI tools, so that many of the capabilities users ask for are already there in the default install. You can try ecasound or dbmix, though that's just what I happened to pick up and can't elaborate precisely what they do...

### Post by jerome1232 on 2008-10-25
This is a basic rundown [http://www.psychocats.net/ubuntu/installingsoftware.php](http://www.psychocats.net/ubuntu/installingsoftware.php)

An overview of the Linux File hierarchy might be insightful to you as well
[http://tldp.org/LDP/Linux-Filesystem-Hierarchy/html/index.html](http://tldp.org/LDP/Linux-Filesystem-Hierarchy/html/index.html)

This is my version of installation methods

**Debian Packages** - come as .debs, they are actually what synaptic downloads and installs. Programs generally goto /usr, system wide settings are in /etc, user settigns are in ~/ as hidden files (ctrl+h to see them)

**Binary Installers** - equivalent of .exe in Windows. Extension doesn't actually matter but common ones *if* they are used are .bin, .run, .sh. You make it exectuable (right click, permissions, executable) then double click it. 

**Binaries** - once again equivalent of a .exe in Windows. Usually doesn't even have an extension but common ones are again .bin, .run, .sh. These are programs that are generally packaged in an archive (.tar.bz2 or .tar.gz) You simplly extract them to where you want them and run them by double clicking on the executable file.

**Source Code** - usually comes packaged in a archive, you extract them and compile them. There is Usually a README or INSTALL file that tells you what to do.

### Post by bryonak on 2008-10-25
Not much to add there...
~/ corresponds to /home/[your current account], the same as /Users/[your account] in OSX. The rest is in jerome's second link (long read though)

A quick comparison to OSX and Windows:

.deb is about the same as .pkg - double click (I prefer single click, depends on your settings... the choice thing again ;)) to open, one click to install, in OSX it would "move itself to Applications".
The main difference is that the program isn't contained in the .app file as a whole, but integrates itself into the system's various dedicated folders (binaries, config files, libraries, help docs, image files, ...)

Binary installers are pretty much the equivalent to Windows installers (.exe), where the author of the program decides what goes where and what dialogues he wants to present you during installation. Rarely used.

As jerome said, Linux tries not to rely too much on file extensions (executable JPEGs in Internet Explorer, anyone?), so the filesystem itself keeps track of whether a file should be executed or viewed (you can change this in the file's preferences, of course depending on your permissions).
After installation, GUI programs are usually available in the menu.
For command line tools (like ndiswrapper), there is no 'list' of installed applications. The most common way is that you 'just know' what you just downloaded and execute it in a terminal. Of course, you can search for it in Synaptic/aptitude to see if it's installed or not. Typing 'man [application name]" in a terminal is often useful too.
If you plan on using the shell, you should quickly get comfortable with the autocomplete feature, it makes life a lot faster and easier.
If you want to watch the binaries, visit /bin or /usr/bin, though the view there is rather boring ;)


As for your wireless issue, I suggest starting a new thread in the networking forum.
EDIT: you might also want to try out ndisgtk, a GUI for ndiswrapper
