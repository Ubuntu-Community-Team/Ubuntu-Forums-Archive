### Post by Simian Man on 2011-02-07
It really, really, doesn't matter what platform you code for.  By and large the only thing that you will run into is that some libraries and tools are only available for one platform or the other.  The important thing is just to learn to solve problems, not to learn a particular tool.  Once you can program well, moving to a new set of tools is a trivial matter.

### Post by Mark Phelps on 2011-02-07
Specialization has become so ingrained today that all the folks that I know that do hiring of computer programmers look for folks with a specific set of skills.  This is usually a combination of (1) tool set, (2) language, and (3) platform.

Having done programming myself for more years than I care to mention, and having served in various Leadership roles over the years, I know that ONCE you learn one tool/language/platform combination, any other is an easy transition to make.

But, I don't make hiring decisions -- others do -- and THOSE folks are obsessed with what I said in the first paragraph. From THEIR perspective, programming for MS Windows, using Visual Studio, is entirely different for programming for Linux using Python.  When I was hiring programmers, I got shut out again and again because a candidate was missing one acronym from the skill set!

Programming generalists, like all other generalists, have a very hard time finding jobs today. All too often, the resumes are screened by HR folks who are only looking for certain acronyms. To them, it doesn't matter if you're a whiz at one scripting language, if it is NOT the scripting language they're looking for.

SO, my advice, for what it's worth, is to decide what combination in the first paragraph you want to pursue professionally -- and do that.

### Post by Simian Man on 2011-02-07
> **Mark Phelps said:**
> Programming generalists, like all other generalists, have a very hard time finding jobs today. All too often, the resumes are screened by HR folks who are only looking for certain acronyms. To them, it doesn't matter if you're a whiz at one scripting language, if it is NOT the scripting language they're looking for.

There is something in that, but it's really more a matter of just having a working familiarity with a lot of technology, even if you aren't specialized in it.  For example if you are a great Python programmer, but Ruby is the hot new language, spend a week learning Ruby, then put "Three years scripting language experience with Python and Ruby" on your resume.  You are being honest, and giving yourself a good shot to be hired.  I know at least a dozen languages to the point I'd be comfortable putting them on my resume, but some I haven't spent *that* much time with.

In general learning the ins and outs of a new group's project is FAR harder than learning a tool similar to one you already know.
