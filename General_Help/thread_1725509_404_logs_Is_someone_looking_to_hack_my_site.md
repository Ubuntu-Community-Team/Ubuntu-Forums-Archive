### Post by SeijiSensei on 2011-04-09
Someone is running an automated script trying to find vulnerabilities in your Apache installation.  If they only get 404's they're not succeeding.  I get these attempts from time to time, but I don't run any of the things they're looking for like phpmyadmin.  If you are running something like that, make sure you either have strong password authentication, or limit access to the relevant directories to only IP addresses you trust.  Something like:

```

Alias  /phpmyadmin   /path/to/phpmyadmin
<Directory "/path/to/phpmyadmin">
    Order deny, allow
    Deny from all
    Allow from my.own.ip.address
</Directory>

```

There are lots of canned PHP applications that people install without paying much attention to the security implications of doing so.  Those are the kinds of people scripts like these are trying to find.

### Post by SeijiSensei on 2011-04-09
It's generally not a good idea to leave scripts that run phpinfo() lying around where they can be seen by public visitors.  The level of detail it provides might give some ammunition to a determined attacker.  If you do want to keep a phpinfo() script for reference purposes, give it an unlikely name like whoanellie.php or something like that, and make sure there are no links to that script in publicly-visible pages.  A good robots.txt file can help, too, though that only protects you from well-intentioned spiders like Google or Yahoo.

Are you running PHP on this server?  If not, none of the *.php type exploits can possibly work.
