### Post by TheFu on 2018-07-19
How is this helpful?  
It provides facts. Facts are critical.
It also shows process owner, memory and CPU used.  There is a method to delay the output.  The top manpage has that explained.

Would be good if the ASCII color stuff wasn't included so reading the summary in the header was possible. I'm sure there's an option to disable that.
Try: **top -n 1 -b**

So this provides a list of processes to research for common CPU hogging issues.  Consider running the top command a few times and providing "representative" output back here. You know what that looks like, or munin will have graphs so you can see the times when there are spikes. Munin isn't a desktop solution. It is a system solution.

Ok, some thoughts related to the provided output:

Surprised to see initctl in the list if this is an 18.04 machine.  Isn't initctl from upstart - pre-16.04?  I don't have it on my 16.04 desktop.  There is a systemd version of it, but that has a different name.

I run kodi, but only on dedicated kodi systems, so seeing it with browsers and email programs seems off to me.  Kodi using that much CPU if it is playing media definitely is an issue.  The GPU should be handling the playback with less than 3% of a CPU involved at all for most media.  If it isn't playing media, kodi has a terrible architecture and hogs the CPU. You've probably known that. It is because of the way the original xbox graphics system worked.  If kodi isn't being used, I'd close it.

I run thunderbird too. It spikes in CPU, but that goes away quickly, milliseconds later. It showing up near the top doesn't make sense, unless thunderbird has a bug, you allow HTML email (huge security issue) or it happened to be checking email at that instant. 

X/Windows will spike whenever a screen redraw is happening, but that goes away quickly. If it doesn't, then I suspect it to be a GPU driver issue.

For the machine you have, none of those processes should be using much more than 30% of 1 core for less than 1 second.  Obviously, if you are transcoding videos on it, that will take longer and use all the cores, but that isn't what we're seeing here.

BTW, I'm slightly jealous of your machine hardware.  Actually have parts for a Ryzen 1600 build in a cart on another tab. 

I don't use the other stuff you're running, but you are using Studio, so that's understandable.
