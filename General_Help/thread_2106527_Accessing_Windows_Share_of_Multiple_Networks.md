It happens because the default Samba package defaults to a 1 segment LAN setup.  If you are going to run Samba over multiple subnets you should run a WINS server.  See [**_[COLOR="Blue"]here[/COLOR]_**]("http://www.samba.org/samba/docs/man/Samba-HOWTO-Collection/NetworkBrowsing.html#id2583597").

We are dealing with the most Byzantine networking protocol that I know of.  There are so many packages and variations that it is hard to really be specific.  That being said, there are some rules of thumb.

The protocol is SMB/CIFS.  It doesn't matter whether it's Windows or Ubuntu.  Of course the implementations do vary slightly.  A Windows host that is acting as a SMB/CIFS client is basically the same as an Ubuntu SMB/CIFS client.  An Ubuntu host that is acting as a SMB/CIFS server (Samba) is functioning the same as a Windows host running the service known ironically as *"server"*.  I tend to think of all of these hosts as SMB clients or servers.  

To your question:  I believe the client routines do use the smb.conf file.  The samba-common package is installed by default when you install Ubuntu and it contains the smb.conf file.  Also most folks (including me) install cifs-utils (aka smbfs).  I even have some SMB clients that also have the package *samba *(the v3 server) installed.  Who knows.  There are so many packages which have overlapping files that it's near impossible to tell what is needed and what is not.>  

Moreover, as you had mentioned in your earlier post that Samba package defaults to one segment, while logging off and back in clears some kinda cache and Ubuntu can browse other networks.  In addition to that, Ive got Wireshark installed as well and once one of the subnet PC's are browsed and I switch to another subnet, no NETBIOS query is generated by the ubuntu machine. That is what is confusing that Ubuntu simply dont try to refresh its browsed entries.

As I see it; the default Samba setup uses broadcasts as a final method to resolve IP address to COMPUTER NAME.  The *nmbd *daemon can convert hostname to NETBIOS name to IP mapping.  Therefore it is important to have a properly configured /etc/hosts file on each host or local LAN DNS.  All of this is great for a single segment LAN.  Unfortunately the last resort bcast resolution doesn't cross subnets.  This is why you need WINS.  The WINS server will hold the mapping of IP addresses to COMPUTER NAMES and will supply the mapping across network segments. When you configure a WINS sever in SAMBA the *nmbd *daemon is disabled. >

In the end my advice is to never muck around with the nsswitch.conf file.  Make sure you have functioning LAN side DNS (including /etc/hosts).  Select an appropriate host (always on) to run a WINS server.  Point all other clients (including Windows hosts) to the WINS server.  If the WINS server is also a SMB client it will find itself via bcast as both the SMB client and the WINS server are both on the same segment.

### Post by PowerBarry43 on 2013-01-22
Hi

An alternative method would be to create a script or launcher that runs the command

```
nautilus smb://{ip_address}/share
```

This would not require any DNS or WINS server as you aren't using any hostnames, just ip addresses.

You couldn't go through network places to access the share by doing this but you could mount the samba share locally with the commands


```

mkdir /path/to/mount-point # first time only

sudo mount //{ip_address}/share /path/to/mount-point -o user={smb username} -o password={smb password}
```

or you can just leave the command at the username and you will then be prompted for password.

Just another way to go about doing things ;)

hope this helps!
