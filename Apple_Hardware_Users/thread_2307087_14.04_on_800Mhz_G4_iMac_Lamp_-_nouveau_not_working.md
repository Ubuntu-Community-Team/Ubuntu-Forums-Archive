### Post by Peter_Saisanas on 2015-12-22
Hi,
Please provide a full dmesg log.
This will help understanding what the issue is exactly with nouveau and why its not working with your particular config.
According to the freedesktop nouveau page, NV11 should be supported (but obviously with ppc's, YMMV).

nouveau.modeset=0 is basically disabling KMS and probably wont enable the nouveau driver in the first place.
Its important to try and understand why KMS isnt working in the first place. Hopefully a full dmesg log without any nouveau module options passed will help diagnose.

Also, if you can provide an xorg log, that would be helpful.

Regards,
Peter

### Post by Peter_Saisanas on 2015-12-23
Yes, you seem to be having issues with nouveau not being able to extract a valid DCB table from the nVdia VGA GPU ROM (be it open firmware FCODE or x86 VGA BIOS).
The nVidia DCB table describes the features of the card to the nVidia OS driver, i.e. VGA or DVI connector type and count, CPU core clock, GPU VRAM type, amount, clocks and so forth.
Since nouveau is expecting the DCB table information before it can initialize the GPU, it seems nouveau is failing because it cannot extract the required info from the DCB table that it requires.

At this point, this is all theoretical.
Since you don't have time to experiment and it looks like you will be chartering the unknown, i would also take rsavage's advice.
I would just use OSX for now, but if your hell bent on getting linux on a 15 y.o. machine with nouveau and a modern graphics stack, its going to take time and experimentation.
Otherwise if nv is easier to get working, use this instead.

Ultimately, its looking like a lot of effort is required to get nouveau running on your hardware with experimentation required.
Are you sure you want to put this much effort for a machine with such limited resources?

If you are still inclined, keep reading.

There are two issues, either nVidia hasn't used a DCB table in such an early GPU and instead hard coded register values directly in the VGA ROM for GPU devices of this generation.
Or Apple in their infinite wisdom decided to "Think Different" and decided to do something non standard and not include a nVidia DCB table in their FCODE implementation of the VGA GPU ROM.

However, the good thing about nouveau is that you can force it to load up an external x86 VGA BIOS from a file.
If you happen to know the exact specs for the GPU you happen to have, i.e. it seems you have a GeForce MX400 with 64MB VRAM, try and download an equivalent exact type of x86 VGA bios for the GPU you have.
This will prove if the DCB block is present in this generation GPU.

Follow the guide here:
[http://powerpcliberation.blogspot.ca/2015/10/g5-and-g4-nouveau-modesetting-bug-and.html](http://powerpcliberation.blogspot.ca/2015/10/g5-and-g4-nouveau-modesetting-bug-and.html)

If this seems to get you further down the track, the next hurdle you may have is that KMS will attempt to parse a valid EDID table. Practically all modern displays that are connected via DVI or VGA have a i2c flash chip with EDID which describes the monitor parameters, (i.e. display timings and max resolution, colour calibration, etc). This is read via the DDC lines on a typical DVI or VGA interface. However since most laptops use a LVDS interface for the LCD panel, they do not have a separate i2c flash chip containing an EDID table. On a typical x86 laptop, the EDID will be in an ACPI table. You can force an EDID table from file as well but you will probably have to create one. As a quick experiment, you could try to connect an external VGA monitor to see if it works. If this works, create an EDID for the built in LCD panel and load from file.

The likely hood of getting this working with nouveau in the time you have available is slim to non existent.
Best to try OSX or the nv driver as rsavage has recommended earlier.

Sorry, i dont have an easy answer or workaround here!

If the internal monitor is on a native VGA interface, that would be a big surprise.
Typically most LCD panels are driven with a DFP (typically LVDS) interface.

Id say it is a symptom of the display driver getting confused about the available outputs of your GPU.
This is the sort of thing that for nVidia GPU's, the DCB is meant to describe.

The nouveau driver seems to rely on the DCB being present and valid, as would the proprietary nVidia GPU driver for windows or linux.
