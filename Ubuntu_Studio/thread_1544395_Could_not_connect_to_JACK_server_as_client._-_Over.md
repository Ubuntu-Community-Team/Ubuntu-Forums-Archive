### Post by AutoStatic on 2010-08-03
```
JACK is running in realtime mode, but you are not allowed to use realtime scheduling.
Your system has an audio group, but you are not a member of it.
Please add yourself to the audio group by executing (as root):
usermod -a -G audio (null)
After applying these changes, please re-login in order for them to take effect.
You don't appear to have a sane system configuration. It is very likely that you
encounter xruns. Please apply all the above mentioned changes and start jack again!
```So either untick the realtime option in the setup window of QjackCtl or add your account to the audio group and make sure your */etc/security/limits.d/audio.conf* file is there and has the following lines in it:
```
@audio   -  rtprio     99
@audio   -  memlock    unlimited
```

You could also run **dpkg-reconfigure -p high jackd** in a terminal and select Yes to enable realtime process priority.

### Post by Pablo_F on 2010-11-24
Hi, 

Let's see. I can't reproduce your exact problem but some thoughts:

Both pulseaudio and jack use the alsa driver. Lets' assume the alsa driver is OK with both cards (maybe a speaker-test will help to discard an alsa driver issue but by the time being let's think the alsa driver is fine, I think so).

aplay -l lists the cards and the so-called "devices" (and even the "subdevices" but i don't think you have to worry about subdevices). You have to worry about "devices" when using jack. aplay -l shows two different devices with analog audio ports of some kind for the c-media but you can only choose one device at a time in jackd (unless you use one device for capture and another one for playback, but then you would miss the playback ports of device 0)

The other card is different. See aplay -l for it. 

Pulseaudio is the default audio server and yes, gnome sound preferences is a pulseaudio front-end. It is no wonder that pulseaudio works because it has a different philosophy. It is focused on the desktop use and it deals with card-device-subdevice and what-not so that consumer cards, which are rather complicated, work out the box. 

On the contrary, jack is focused on audio production use and on other kind of audio cards, professional ones (some of them at least) which are not so tricky in these regards.

Well, now, you are running jack2 (version 1.9.6). I am not familiar with jack2 and this issue.  I have heard something of "jack audioadapater" I am not sure how this works but try "jackd audioadapter --help" "man jackd2" or something of the sort. Sorry, I can't test here with jack2. I can test at the weekend though. I will be back. Maybe someone can chime in aswell.

I think alsa_out should work with jack1. Maverick installs jack2, You can try to install jackd1 package which will uninstall jackd2 and maybe some apps, then reinstall the apps if some of them uninstalled automatically.

Anyway, maybe the best option is to join device 0 and device 1 of the c-media in a unique device, in an .asoundrc file. This is tricky. Begin here:

[http://alsa.opensrc.org/.asoundrc](http://alsa.opensrc.org/.asoundrc)

If you get stuck, I suggest you ask in the #jack channel, in IRC, or linux users mailing list, or alsa users mailing list. This can be done I am sure, just jack is not friend of these kind of cards, and then there is jack1 and jack2, with different approaches and not a lot of user documentation.

To sum up, you have several options:

Install jackd1 and try alsa_out again
Try some equivalent to alsa_out in jack2
prepare and .asoundrc file to convert the two devices in one device so that jack sees all the available ports in the new device.
Others that I don't know of. 
Ask the devs. 

Another tricky thing: When you have multiple soundcards, in each reboot alsa can assign different numbers to them. Now, the c-media is hw:0, tomorrow is the hw:1. There are easy ways to avoid this confusion but by the time being just be warned.

I hope this helps, Pablo
