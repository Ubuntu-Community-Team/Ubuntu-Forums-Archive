### Post by TheFu on 2015-03-01
Do not use UPnP - the security issues with that are endless. It should be disabled in your router - verify that.

a) don't connect any USB storage to a router. There are lots of known security holes and nobody is fixing them.
b) setup an ssh server on a static IP inside your LAN,
c) install fail2ban on that machine to prevent brute force attacks from being successful, 
d) setup key-based ssh authentication from any clients to that machine - internal AND external, 
e) use port translation from the public router IP:port to the internal server IP:22  - on the public side, using a random, high port, is helpful to keep logs smaller. If you use the default port (22/tcp), expect 10K hack attempts daily. 
f) connect any storage you want to the ssh server ... directly attached or network mounted, but have this "server" be the middle, not the storage directly. Use NFS if possible, CIFS is a last resort.
g) access the ssh/sftp/scp server from anywhere in the world, from any remote computing device you like.  If it has a network, then there is an sftp client (android, win, unix, etc.)

You might want to setup a dynamic DNS to make finding your public IP easier. Most routers have a tab to enable that. Only do backups when inside the LAN. There are free Dynamic DNS services ... or you can pay. I use DynDNS.org and no-ip.com. Easiest to pick one that is already in the router.

ssh is enough for:
 [LIST]
[*]   secure remote access to files via sftp
[*]    secure remote filesystem access via sshfs
[*]    secure remote CLI/shell access to systems with plain ssh
[*]    secure remote desktops via x2go/freenx
[*]    secure remote file replication with rsync (ssh is the default rsync protocol)
[*]    secure port forwarding of selected ports
[*]    secure remote editing with vim/gvim and other editors
[*]    pseudo-VPN with sshuttle
[/LIST]
ssh really is **THE toolbox** for remote connectivity. [http://blog.jdpfu.com/2014/09/23/you-don-t-know-ssh-about-ssh](http://blog.jdpfu.com/2014/09/23/you-don-t-know-ssh-about-ssh)

Oh - and I use ssh to tunnel all backups here - it is the default for any librsync-based tool like rdiff-backup.

ssh is about 10x easier to setup than openvpn.  If you want to use something like owncloud or seafile, only do those if you use a VPN too. They are not safe over the internet.

### Post by TheFu on 2015-03-01
How do you access it from anywhere?  WD provides the connector so you don't have to get a dynamicDNS.  It is foolish to believe that any networked device which can open ports on your router will NOT be something that could be accessed by the vendor's support team without your knowledge or approval.  

Does it check for and automatically update firmware? Then even if it doesn't provide that access today - it could tomorrow. If they insist that you have an online account to use the device ... keep looking for another device.

Of course, you can trade convenience for lack of security if you like.  We all do, daily.

Some background reading: 
* [http://www.theguardian.com/technology/2014/oct/22/apple-warns-of-new-icloud-threat](http://www.theguardian.com/technology/2014/oct/22/apple-warns-of-new-icloud-threat)
* [http://blog.kaspersky.com/internet-of-crappy-things/](http://blog.kaspersky.com/internet-of-crappy-things/)
* [http://www.techtimes.com/articles/14800/20140903/cloud-hacking-isnt-hard-think.htm](http://www.techtimes.com/articles/14800/20140903/cloud-hacking-isnt-hard-think.htm)
* [http://community.wd.com/t5/WD-My-Cloud/Insecure-Device-LAN-vs-Internet/td-p/798029](http://community.wd.com/t5/WD-My-Cloud/Insecure-Device-LAN-vs-Internet/td-p/798029) - some reading

Basically, if you allow access from the internet using only passwords, you've failed.  Use keys, always. If that isn't provided, then provide the access yourself.  Inside your LAN - I would be so worried, unless you have teens. Then I'd put them on their own subnet so the viruses can bounce around there and not on the adult computers.

Perhaps I'm paranoid, but sadly not enough, for the last decade. Turns out, if anyone can think of a way to hack something, then it will be hacked, eventually.  If not by bad guys, then by multiple governments [http://blog.jdpfu.com/2013/08/05/are-you-paranoid-enough](http://blog.jdpfu.com/2013/08/05/are-you-paranoid-enough)
In 2005, I helped design and deploy an email metadata capture system for use worldwide to comply with EU mandates - this was for a commercial company. We could have done much more.  We were the ISP to other ISPs.  One of my peers was deploying "deep packet inspection" tools on the network at the same time - never thought to ask "why" back then.

Be paranoid. Once stuff gets out, it is gone, never to be retrieved.  **The right to privacy really is the right to liberty.**

### Post by tgalati4 on 2015-03-05
If you reread the NAS suggestions, those options allow the security frameworks that TheFu and others have described.  Poking a pinhole can be accomplished using "port knocking".  You ping a specific port (say 23462 only known to you) and it opens another port (say 44386 again only known to you) and that allows the *ssh* connection (which is internally mapped to port 22 or to some other port that you specify).  When you log off *ssh* or drop the connection, then the pinhole closes.

The WD products are consumer products with typical consumer security problems--part of the collection of "Crappy Things".
