### Post by TheFu on 2014-11-09
After seeing another post here about fail2ban not working, I decided to double-check my log files to ensure fail2ban was working fine.  Been using it for years - every ssh-server gets fail2ban, even if they are internal-only access.  The sshd_config blocks remote root for most of my systems completely, but for the backup server, I allow remote root using keys only (PermitRootLogin without-password).

So ... there's 1 IP that listens on port 22 here (default for ssh).```

Nov  7 20:57:24 romulus sshd[21315]: message repeated 5 times: [ Failed password
 for root from 60.173.26.53 port 2265 ssh2]
```

```
$ egrep -c  'Failed password for root' /var/log/auth.log
314
```
These were all from the same IP address in China.  Fail2ban used to stop these automatically just by installing the package.

To make this 1 trouble-maker stop, I quickly put this 
```
/sbin/iptables -I INPUT -s 60.166.0.0/10   -j DROP
```
into /etc/rc.local and manually ran it.  Problem solved. Blocking Chinese ISPs is easy - they all have huge allocations.
[B]
Still - there is something broken in the fail2ban packages. This stuff used to work, by default. It doesn't anymore.[/B]

Or is it just my 10 installations?  Only 1 of these will likely ever get attacked enough to ban anyone.
