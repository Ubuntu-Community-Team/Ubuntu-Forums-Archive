### Post by Doug S on 2011-03-27
The first number is the number of packets.
The second number is the number of bytes.
When using the iptables-save and iptables-restore method, verses an iptables script, those two numbers should be set to [0:0], restting the counters.

### Post by bodhi.zazen on 2011-03-28
iptables-save and iptables-restore does not reset those numbers.

To zero them out, use

iptables -Z

than save or manually edit the saved file.

You might also want to look at the iptables-apply script, especially if managing your firewall from remote.

### Post by BkkBonanza on 2011-03-28
Unless you intend to keep count of bytes/packets...

In which case you need to add iptables-save to your shutdown script so the end counts are saved before shutting down. And perhaps use a cron for periodic saves in case you crash.

I'm not saying that's worthwhile, but just noting it's possible to do.

### Post by Doug S on 2011-03-28
I agree that my posting was misleading that [0:0] resets the counters. I did a test and it doesn't seem to matter. It can be [XXX:YYYY] and the counters will still be reset.
 
However, and I am just trying to improve my own understanding of things here, I think that iptables-restore does reset the counters.
 
Consider the following example (using my old old Ubuntu box (on local network only)).
First, take a look at the counters with "sudo iptables-save -c":
 
```
 
# Generated by iptables-save v1.3.3 on Mon Mar 28 07:18:24 2011
*filter
:INPUT ACCEPT [7:1047]
:FORWARD ACCEPT [0:0]
:OUTPUT ACCEPT [52:4368]
[52:4368] -A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT 
[0:0] -A INPUT -m recent --update --seconds 5400 --hitcount 3 --name BADGUY_SSH --rsource -j DROP 
[0:0] -A INPUT -p tcp -m tcp --dport 22 -m recent --set --name BADGUY_SSH --rsource -j ACCEPT 
COMMIT
# Completed on Mon Mar 28 07:18:24 2011


```
 
Second, execute this script with "sudo ./doug_ex2":
 
```
 
#!/usr/bin/env iptables-restore
# Smythies: simplified examples:
#               Normal: "sudo ./doug_ex2"
#               Test: "sudo iptables-restore -v -t <doug_ex2"
#
*filter
:INPUT ACCEPT [100:1000]
:FORWARD ACCEPT [200:2000]
:OUTPUT ACCEPT [300:3000]
#
-A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT
#
-A INPUT -m recent --update --hitcount 3 --seconds 5400 --name BADGUY_SSH -j DROP
-A INPUT -p tcp -m tcp --dport 22 -m recent --set --name BADGUY_SSH -j ACCEPT
COMMIT


```
 
Third, take a look at the counters again:
 
```
 
# Generated by iptables-save v1.3.3 on Mon Mar 28 07:28:30 2011
*filter
:INPUT ACCEPT [0:0]
:FORWARD ACCEPT [0:0]
:OUTPUT ACCEPT [0:0]
[0:0] -A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT 
[0:0] -A INPUT -m recent --update --seconds 5400 --hitcount 3 --name BADGUY_SSH --rsource -j DROP 
[0:0] -A INPUT -p tcp -m tcp --dport 22 -m recent --set --name BADGUY_SSH --rsource -j ACCEPT
