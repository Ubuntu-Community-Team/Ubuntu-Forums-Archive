### Post by OpSecShellshock on 2010-08-19
Took a quick look. Port 111 seems to be reserved for a service called sunrpc, which is used for web servers (sometimes used for NFS I guess?). I assume we're talking about a server of some kind since ports have been set to listen for http and ssh already, which would only be necessary if incoming connections on those ports need to be initiated from the web. Is this server also set up for NFS? That might necessitate the use of rpc (remote procedure call) portmapper, which is what would run on 111.

### Post by CharlesA on 2010-08-20
I just use iptables, iptables-save and iptables-apply.

Or in my case I have a script like so:

```
# Generated by iptables-save v1.4.4 on Wed May 26 10:07:13 2010
*filter
:FORWARD ACCEPT [0:0]
:INPUT ACCEPT [0:0]
:OUTPUT ACCEPT [0:0]
# Accept established
-A INPUT -m state --state ESTABLISHED -j ACCEPT
# Accept related
-A INPUT -m state --state RELATED -j ACCEPT
# Accept ping requests
-A INPUT -p icmp -m icmp -s 192.168.1.0/24 --icmp-type echo-request -j ACCEPT
#Accept Samba
-A INPUT -p udp -m udp -s 192.168.1.0/24 --dport 137 -j ACCEPT
-A INPUT -p udp -m udp -s 192.168.1.0/24 --dport 138 -j ACCEPT
-A INPUT -m state --state NEW -m tcp -p tcp -s 192.168.1.0/24 --dport 139 -j ACCEPT
-A INPUT -m state --state NEW -m tcp -p tcp -s 192.168.0.0/24 --dport 445 -j ACCEPT
#10 minute lockout if trying to bruteforce
-A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set --name  SSH --rsource
-A INPUT -m recent --update --seconds 600 --hitcount 4 --rttl --name SSH  --rsource -j DROP
# Accept port  22 ssh
-A INPUT -p tcp -m tcp -s 192.168.1.0/24 --dport 22 -j ACCEPT
# Accept loopback
-A INPUT -i lo -j ACCEPT
#-A INPUT -p tcp -m tcp --dport 22 -j ACCEPT
# Drop All
-A INPUT -j DROP
COMMIT
# Completed on Wed May 26 10:07:13 2010
# Generated by iptables-save v1.4.4 on Wed May 26 10:07:13 2010
*mangle
:PREROUTING ACCEPT [0:0]
:INPUT ACCEPT [0:0]
:FORWARD ACCEPT [0:0]
:OUTPUT ACCEPT [0:0]
:POSTROUTING ACCEPT [0:0]
COMMIT
# Completed on Wed May 26 10:07:13 2010
# Generated by iptables-save v1.4.4 on Wed May 26 10:07:13 2010
*nat
:PREROUTING ACCEPT [0:0]
:POSTROUTING ACCEPT [0:0]
:OUTPUT ACCEPT [0:0]
COMMIT
# Completed on Wed May 26 10:07:13 2010

```

Then I have this set in /etc/network/interfaces to apply them after the interface is up:

```
post-up iptables-restore < /etc/network/iptables

```
