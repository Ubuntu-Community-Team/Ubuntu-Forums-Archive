### Post by WinstonChurchill on 2010-06-19
First of all, I have to laugh at the fact that you still use the floppy drive...

You probably have other issues with the ethernet - yes, there do exist exploits that can compromise the actual firmware of devices, but these are incredibly rare, and are only effective against very specific hardware configurations, which makes them difficult to pull off, and not usually worth the time. 

Your logs do suggest that you were hacked, though; nothing for sure, but take a look:

From access-apache.log:```
 217.24.240.68 - - [17/May/2010:22:50:28 -0500] "GET /user/soapCaller.bs HTTP/1.1" 404 470 "-" "Morfeus F***ing Scanner"

```This is probably just a port scanner - it's not that exciting. The IP could be fake, but if the client expected a response from the probe, it would have to actually be his IP, or an IP he has control of.

From access-apache.log
```

[Mon May 17 07:47:36 2010] [error] [client 98.199.19.56] request failed: URI too long (longer than 8190)
[Wed May 19 23:17:45 2010] [error] [client 98.199.18.7] request failed: URI too long (longer than 8190)

```Anything that is "too long" in logs can usually be considered suspect: I don't run Apache, so I don't keep up with it, but it's possible that this client was attempting to exploit some sort of buffer-overflow - it's a shame Apache didn't log the URI he provided... Both of these IP's are owned by Comcast; since they are DHCP addresses, it is possible that these are the same person. 

From kern.log:
```

May 19 23:14:30 web-server kernel: [ 8004.634742] nautilus[2649]: segfault at 89fcbce1 ip 00f78a93 sp b47bfd90 error 4 in libgvfsdbus.so[f5b000+24000]
May 19 23:14:30 web-server kernel: [ 8004.636430] nautilus[2650]: segfault at 897cece1 ip 03e19a93 sp b40dcd90 error 4 in libgvfsdbus.so[3dfc000+24000]
May 19 23:36:24 web-server kernel: [ 1025.082939] MohFill[3000]: segfault at bee6effc ip 08049f91 sp bee6f000 error 6 in MohFill[8048000+5000]

```The last entry is more suspect - it occurred right after the "URI too long" message above; it could be that the hacker managed to get a shell at Apache's privileges and was trying to get a root shell - does MohFill run as Root, or SUID 0? 

Of course, the best sign that something is amiss is that Apache stopped working. Was the process still running and simply not responding to requests, or did it die? If it was still running and not responding, it is much more likely that something was going on... Of course, somebody who really knew what they were doing would have found a way to exploit Apache and keep it running, and probably would have deleted these log entries (even if they weren't a result of the hacking, they look kind of suspicious - if it was me, I'd not want anything hinting that something was amiss).
