### Post by MikeRitter on 2007-08-19
Updating the Ubuntu Linux file server.


When the Chronic Fatigue Syndrome Societies fileserver crashed.

In early 2007 I bought parts and build a replacement Linux file server for the Chronic Fatigue Syndrome Society. It runs Samba, FTP, SSH and Apache. Since it has been up and going it has been going 7 months non-stop and it has not even look like crashing. It was so solid you could have a direct hit with a ten tonne thermo nuclear device and it would still work. Linux is that stable.

The computer made from an ASUS M2NPV-VM motherboard and an AMD Athlon 64 CPU running @ 2.6 GHz with 512MB of ram. It has an integrated GeForce 6 GPU High Definition Video Processing display with max resolutions to 1920 x 1440 (@ 75Hz). However, the best features, by far, is that it glows blue in the darkness and you can make a visual inspection through its clear side into the inner workings of the machine. You can see that all the fans are going. The file sever has a SATA hard disk 160GB and a dual layer DVD drive. It was such a bargain at under $1000 AUD.

Since I had installed Ubuntu 6.06 LTS it was time to upgrade the software. I started the upgrade over the internet. My plan is to upgrade from Ubuntu Linux v6.06 to v6.10 over the Internet then go on to 7.04. You cannot go from 6.06 to 7.04 you must do the intermediate steps so the Ubuntu documentation says. Bugzilla stepped in and a crash report sent by way of the Internet. The first step in the upgrade process had invoked Bugzilla. It reported a problem over the internet. I continued to download and install 7.04 and tragedy struck.

There was a crash after it downloaded and started installing the software. I decided to reset the machine. The machine never came back up again and it was now a job to fix and recover the server.

To recover from such an error it is not all that hopeless. Firstly, you have options. Use a specialised recovery CD with loads of collected and good recovery software or just boot up with the live version of Ubuntu. Get into a gnome-terminal session. When you get the system up and going you just make the directory “mkdir /mnt”, then “cd /mnt” and make the directory “mkdir mydir” with these commands. Then you mount the crashed hard disk in that directory with “mount /dev/sda1 /mnt/mydir” and the disk mounts in that directory, but here’s the real move. Use the command “chroot .” to change the root directory to the mounted directory and the whole thing starts to look like the old system and you can start to look around at the disk.

Because the update crashed some way while installing downloaded update you could just issue an apt-get install update and have things download, but the problem with the sate of this machine is that some of the software was updated and some was not, in fact the amount of errors the apt-get install program handled was so many it just stoped and said there were too many to work with. My machine was broken really and truly. So, you can try and remove the offending software with the “apt-get remove <software package>”. It does work but again with my case the problem was too complex and too far-gone. You could have done it but you never could have discovered everything that was wrong with it and it would have taken forever.

I went to the Linux forums and even had someone on the IRC Ubuntu Linux chat room where at any one time there is between 100 and 150 people in the room. With this amount of brains to pick I can say conclusively if there was an earlier way of recovering the system I would have found out and done it. There are no simple answers this time.

The only thing to do was to wipe the disk and put Ubuntu 7.04 on the computer. A word of warning I decided not to delete the swap space and I found that when I had put Linux back on the system it still didn’t want to come up with the desk top, therefore I went back and for the second attempt I wiped out the swap partition and the file partition and installed Ubuntu Linux 7.04 from completely nothing. It worked the second time. So what happen to the files on the file server? Fortunately I religiously back up the files on DVD using an automatic cron job. It does this every Wednesday at 13:35pm. I did not loose a byte in the end and because people were only in the office on Wednesdays only one working day was lost. I had about 6 days to get the machine back on line again.

It took a little more time to start up Samba and VSFTP and SSH but it wasn’t all that much of a problem compared with the mess the file system was in. I estimate there was between 25,000 and 250,000 files on the system, so it would have been a massive job to discover which of the files the wrong versions were. My thanks to Paul Schultz of Adelaide for his words of advice and support during this time.

In the end not much time and data is lost and that is a good result, so here is to cron’s initiated weekly-automated backups and good IT practices. The machine is up and going again.
