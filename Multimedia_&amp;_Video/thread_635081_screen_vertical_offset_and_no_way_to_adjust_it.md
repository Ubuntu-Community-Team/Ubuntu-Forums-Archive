### Post by redsun82 on 2007-12-08
This problem's really buggering me...

A couple of weeks ago, I bought a Samsung Syncmaster 971P monitor, a stylish 19'' LCD, good looking and all BUT it has no control buttons (apart from the power switch and a button to cycle through some predefined contrast settings). Regulation is done by software, and you can bet the software is for Windows only :( And it does not work under wine.

Now, at the beginning I found that when I run at 1280x1024 (the native resolution of the monitor), there was an offset of some pixels upwards, so that there was a balck band on the bottom of the screen, and the top was chopped. I searched for the solution and found various people telling to

a) use the autofit button on the screen, which I can't do because there is none
b) adjust the screen via the control buttons, which I can't do because there are none
c) use xvidtune, which doesn't work because it does not let me move the screen downward, apparently it's already as low as it can be

I tried to fiddle with the values myself, only to find out that indeed the screen couldn't go lower by xorg configuration.

I switched for a while to windows to see if I could do something there. Same problem there. The program by Samsung di not even install, but I noticed that if I changed the frequency to 72 hertz the offset went away, while returning as before on 75Hz (and even worse on 60).

Went back to linux, started nvidia settings... to find out I did not have the option for 72Hz. After hours of fiddling, I finally added the following modeline in xorg, obtained from an online generator:

```
ModeLine	"1280x1024_72.00" 132.0 1280 1368 1504 1728 1024 1025 1028 1067 -hsync +vsync

```

and the problem was apparently solved. Indeed the offset went away, and the display looked also somewhat better than on 75 Hz. I was happy... until today. With no apparent reason, after I had logged in the screen "autofitted" itself (a rectangle with "autofit" written in it briefly appeared) and the damn offset was there again, and seemingly to stay. Now I really do not know what to do. Anyone has any suggestions?

### Post by redsun82 on 2007-12-12
In the end I (temporarily) resolved by creating another modeline with the gtf command for 73Hz. It seems that when the screen faces a setup (resolution + refresh rate) never encountered before, it autofits and comes out nice. But if for some arcane reason it looses its good setting, it does not autofit again afterwards, no matter how much you go crazy on it. It's like it has in an internal memory the settings encountered... too bad there seems no way to operate on that memory, at least not from linux...

Anyway, if it happens again, I'll go for 74Hz, and then I'll start with decimals...
