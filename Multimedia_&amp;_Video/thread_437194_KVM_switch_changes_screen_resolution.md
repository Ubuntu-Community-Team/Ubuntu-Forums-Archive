### Post by ohgod on 2007-05-08
Strange:  I used to have this exact problem with Dapper, but not with Feisty.

Anyway, what I usually do is manually edit the /etc/X11/xorg.conf file to remove all the resolutions I don't want (I only leave the one I want).  That kind of forces X to use the resolution I'd like it to.

For example, I use 1100x900 in my vmware installation, so my 'screen' subsection in xorg.conf looks like this:

```

Section "Screen"
	Identifier	"Default Screen"
    Device      "VMware SVGA"
    Monitor     "vmware"
    # Don't specify DefaultColorDepth unless you know what you're
    # doing. It will override the driver's preferences which can
    # cause the X server not to run if the host doesn't support the
    # depth.
    Subsection "Display"
        # VGA mode: better left untouched
        Depth       4
        Modes       "640x480"
        ViewPort    0 0
    EndSubsection
    Subsection "Display"
        Depth       8
        Modes       "1152x900"
        ViewPort    0 0
    EndSubsection
    Subsection "Display"
        Depth       15
        Modes       "1152x900"
        ViewPort    0 0
    EndSubsection
    Subsection "Display"
        Depth       16
        Modes       "1152x900"
        ViewPort    0 0
    EndSubsection
    Subsection "Display"
        Depth       24
        Modes       "1100x900"
        ViewPort    0 0
    EndSubsection
EndSection

```

Hope this helps.

Yikes.  Sorry to hear that didn't work.  

It's likely that X doesn't know much about your monitor, so it's using the lowest common denominator.  You might have to edit xorg.conf again and set the refresh rates that your monitor will support.  You could also generate modelines for resolutions & refresh rates that you're sure you monitor can handle.

Ubuntu is sniffing out the monitor during boot and not being able to access it because the KVM switch is locked to another computer. Hence, it's writing a new xorg.conf that isn't correct. A workaround at this point in time is to get your xorg.conf working properly with Ubuntu and then setting permissions on it so that it can't be overwritten. The following command will remove the ability of root to write to xorg.conf ...

sudo chmod 500 /etc/X11/xorg.conf

In future should you need to make changes to xorg.conf, then you'd have to give root permission to write to the file before you can edit it. 

Note: if you're using some other software that needs to write to xorg.conf, it too will not be able to do so, unless you change the permissions back. So use this workaround at your own risk. I don't know whether that will produce some other problems, but it should get you around this KVM problem.
