Since you have a safe backup of your original database, and have restored successfully from it, one idea might be to log into this original, pre-upgrade database that you restored and drop the offending column "autometadata" that the upgrade routine is trying to add a second copy of. Then log out of MySQL, restart the MySQL daemon, and start mythbackend from a prompt to see whether mythbackend can now successfully complete the database upgrade process. 

After logging into MySQL the first task would be to find the table in the old database that contains the spurious "autometadata" column.

The command to do this would look something like```
SELECT DISTINCT TABLE_NAME FROM INFORMATION_SCHEMA.COLUMNS WHERE COLUMN_NAME IN ('autometadata') AND TABLE_SCHEMA='mythconverg';
```

If this returns the table name, the next step would be to drop the offending column autometadata from the table:```
alter table TheTableName drop column autometadata;
``` 

Assuming this has returned no errors, exit MySQL, restart the MySQL daemon, and then start mythbackend to see whether it upgrades mythconverg successfully.

If the db upgrades, try mythtv. If not, or if the db has gotten bleeped up by this attempt, you can restore the original one from backup as you did before, and you're no worse off than being back at square one.

### Post by Panhead Bill on 2012-07-29
klc5555; Thanks for the info.  It lead me to Mysql and BD commands and after "Night-Cogitating" (sleeping)on it, Instead of using the last stored database backup, I went to all of the backups on a terminal:

/var/lib/mythtv/db_backups
la -a
 and all the backups were listed.

The one I was using: mythconverg-1277-2012726045614.sql.gz was the one that would not upgrade the schema.  I found a slightly older one that was at 1264 instead of 1277 and restored it instead:

/usr/share/mythtv/mythconverg_restore.pl --drop_database --create database -- filename /var/lib/mythtv/db_backups/mythconverg-1264-20120725185225.sql.gz

The older database was restored, and mythbackend then upgraded the schema to 1299.  VICTORY!
