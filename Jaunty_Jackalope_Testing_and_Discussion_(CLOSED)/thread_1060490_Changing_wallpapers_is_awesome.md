### Post by BwackNinja on 2009-02-07
@bruce89

The big deal about real transparency vs the way it is done now is highlighted _more_ by how changing the wallpaper fades now. Rather than being actually transparent, a "transparent" gnome-panel right now only takes a picture of the desktop periodically and uses it as its background, giving the *appearance* of transparency. This shows when using transparent cube in compiz (where the background of the gnome-panel doesn't go translucent like the rest of the background) and in the fade when changing wallpapers, because it lags behind the actual background (unfortunately, the same thing happens for conky). This stops it from looking as nice as it could.

A true transparent panel isn't too big of a deal either. There is a patch (not too long even) that makes the xfce4-panel use a hard-coded amount of true-transparency and that's been around for years. The biggest problem with a transparent panel though is that it breaks the notification area because right now it depends on having an rgb rather than rgba colormap. To work around this, you've probably seen KDE4 screenshots with a transparent panel, but with a black background for the notification area. You can put an rgb colormap window on a rgba colormap window, but not the other way around.

True transparency, rather than another fake transparency (the one you can set in compiz, the opacity of windows) does not also make the text of the window transparent like the one in compiz or any other composite window manager, so you still have readablity.

Unfortunately though, I wouldn't expect true transparency for gnome-panel until metacity compositing becomes more than just a gconf key.

Wow, I should try to be more brief, but at least I covered all the bases. :P
