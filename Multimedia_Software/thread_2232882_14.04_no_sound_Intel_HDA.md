### Post by soodvarun78 on 2014-07-06
hda driver(sound controller) has the intelligence to determine and load the driver of the codec which is present in your system.
Linux has a device table which mentions for which codec represented by (vendorid:deviceid) which module has to be loaded.

If you see nigel is getting the error, which  means controller driver(hda) is not able to determine the codec, hence not able to load the driver codec module.

### Post by Yellow Pasque on 2014-07-08
Your specific model is already quirked in the driver (I'm not sure why it's labeled as a K8 board, but that's the PCI ID) [https://github.com/torvalds/linux/blob/master/sound/pci/hda/patch_realtek.c#L1471](https://github.com/torvalds/linux/blob/master/sound/pci/hda/patch_realtek.c#L1471)
So, I would not play with model= string. In fact, you should remove the two lines you added to /etc/modprobe.d/alsa-base.conf

What I would try:
- Getting latest ALSA kernel module: [https://wiki.ubuntu.com/Audio/UpgradingAlsa/DKMS](https://wiki.ubuntu.com/Audio/UpgradingAlsa/DKMS)
- Disabling the codec in the BIOS, and re-enabling it on next boot
- Removing the model= and probe_mask options and trying to disable msi (options snd-hda-intel enable_msi=0)
- If none of that works, remove all customizations from alsa-base.conf and file a bug (since you said probe mask 8 did not work)
