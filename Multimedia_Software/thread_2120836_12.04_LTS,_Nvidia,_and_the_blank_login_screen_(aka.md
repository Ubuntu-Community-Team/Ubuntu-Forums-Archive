### Post by briansrapier on 2013-02-27
I finally (and foolishly) relented and decided to upgrade from 10.04 LTS to 12.04 LTS last week. Why? I can't recall now...

Overall, the installation went well with no problems to report. On first boot I was welcomed by the new greeter, logged in, and tried to become accustomed to the new Unity desktop. After fighting with it for a couple of days, I decided enough was enough and installed gnome-shell. 

The trouble started when jockey-gtk popped up telling me I should install the restricted drivers for my video card (Nvidia G86 GeForce 9400GT). I tried the default drivers at first, but on reboot found myself staring at a black screen rather than the expected greeter. 'No problem', I said, 'I'll just use the drivers from Nvidia since I had no issues with them in 10.04'. 

Right.

After attempting to manually install multiple versions downloaded from Nvidia (295.33 (including the custom patch), 295.49, 295.71, and, the latest, 310.32, I was still having the same issue. After digging some more I decided to try using the ubuntu-x-swat repo and, later on, the bumblebee repo as well. I didn't have success with either of them.

I went through the very comprehensive sticky ["Graphics resolution upgrade/Blank screen after reboot"]("http://ubuntuforums.org/showthread.php?t=1743535&highlight=nvidia+blank") and attempted many of the great suggestions found there, but still could not resolve my issue. I have tried the following options in GRUB:

```
nomodeset
xforcevesa
vga=794 and vga=795 (based on my card's supported resolutions)
```

Most of the posts/bug reports I've seen seem to date back to the April/May 2012 timeframe with little activity since. So either this issue has been resolved for most people and I haven't found the resolution or they've moved on.

At the moment I'm able to limp along using the default nouveau module, but it's performance is very limited. Basic websurfing and terminal use is fine, but bigger apps - word processing, spreadsheets, and <ahem> games - are lagging to say the least.

"You" could install from either. 

I recommend the packaged from the repo to users that are just starting out with Linux and/or want something low-maintenance. This assume that their card is covered by it and has no other problems. Usually self updates through normal system updates, but still an occasional boink from some kernel updates.

The binary, I recommend to more seasoned user's that can follow somewhat detailed instructions and are going to keep up with it. The usual drivers used to be all manual update- meaning that you have to restart the nvidia-installer from a commandline with flags to check for updates and to install an upgrade. Some kernel updates boink this driver also, but that is just part of the game with highscale video gpu's. I hear that the new nvidia installer installs their own update agent... but I haven't checked into that in detail yet to see if that is really true or not. (haven't had time for that yet.) 

Whichever way you go... After 20 some years with X and over 10 years using nVidia and ATI, I can say you will have to re-install or recompile your driver eventually.

You could try to remove the nouveau driver from the blacklist and see if it will still load nvidia. This is usually added to the blacklist, because "some" cards have problems with nouveau and nvidia being coexistent (they don't play well "together") on "some" hardware. This is not the normal case, but more a failsafe in case that problem exists.

If that is not such on your hardware, then there "is" an advantage to doing that. First the nouveau driver will then be the fallback graphics driver. Second, is that if it doesn't automatically fallback to nouveau and it instead locks up... Then all you have to do is to rename the xorg.conf file to something else and it will default back to the nouveau driver.

You can always comment it out in blacklist and test for yourself, then uncomment it out if it does affect you...

As for Bumblebee... If you get graphics working in high-res, are you having a battery drain problem or high-heat problem, where you really need it to switch to a low-power low-res GPU? If not a problem or issue...

Otherwise, what bumble does through scripting is, the same thing I explained above (manual), but through scripts, but also switches the GPU's through the a call from the script to BIOS. So it switches the GPU to nvidia- copies the backup xorg.conf file with the nvidia driver to the xorg.conf name. Then when switches, switches the GPU used to the Intel GPU- deletes the xorg.conf file (or copies an xorg.conf file backup configured to intel to xorg.conf). The driver still stays installs. The pointer (in xorg.conf) just changes.

### Post by bogan on 2013-03-01
Hi!,** MAFoElffen**,

You Posted: > I hear that the new nvidia installer installs their own update agent...  but I haven't checked into that in detail yet to see if that is really  true or not. (haven't had time for that yet.)According to Nvidia.com, drivers version 304 and later are DKMS compatible, and hence the kernal module will be auto-updated in the event of a kernal update.

I have found that I got offered the option to register the driver installation with DKMS about 50% of the times I installed 3.04 or later versions, and those were updated with the next kernal update. The others were not.

I presume that was because DKMS was deleted as a dependancy with 'apt-get remove --purge nvidia*', so one would need to ensure DKMS was installed before running the Nvidia .run file.

Extract from 304.51 Read-Me:
> The installer will check for the presence of DKMS on your system. If DKMS is found, you will be given the option of registering the kernel module with DKMS, and using the DKMS infrastructure to build and install the kernel module. On most systems with DKMS, DKMS will take care of automatically rebuilding registered kernel modules when installing a different Linux kernel.

Not all kernel configurations are supported by DKMS: for example, DKMS does not support kernels built with separate KBUILD source and output directories.
Note that versions of 'nvidia-installer' shipped with drivers before release 304 do not interact with DKMS.

Edit:
Excerpt from 304.32 nvidia-installer --advanced-options, last item: 

> --dkms
      nvidia-installer can optionally register the NVIDIA kernel module
      sources, if installed, with DKMS, then build and install a kernel module
      using the DKMS-registered sources.  This will allow the DKMS
      infrastructure to automatically build a new kernel module when changing
      kernels.  During installation, if DKMS is detected, nvidia-installer will
      ask the user if they wish to register the module with DKMS; the default
      response is 'no'.  Use this option to make the default response 'yes'. 
      This is useful with the '--no-questions' or '--silent' options, which
      assume the default values for all questions.With 304.32 already installed:
