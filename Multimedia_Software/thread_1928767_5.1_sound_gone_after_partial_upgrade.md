### Post by BicyclerBoy on 2012-02-21
Your setup strongly suggests you are encoding multi-channel audio to AC3 using the alsa a52 plugin..
Note this a52 definition does little unless you point vlc (etc) at the a52 plug or your surround51 definition has been modified to point at a52 plug.

Why do you have an expensive soundcard & then output AC3 over S/PDIF iec958 ??
You could do just as well with any cheapo mobo audio codec..

Your HT amp AVR could support bitrate 640K.

The std repo alsa plugin package does not have the a52 function as it uses libav AC3 encoder..

VLC/MythTV have the AC3 encoder built-in & can output digital pass-thru' directly..

Fix a52:
You have to get the alsa plugins source & then build the a52 plugin & copy it into the right place.
Unfortunately the alsa plugins are badly affected by the rapid/frequent changes in ffmpeg/libav abi/versions.

speaker-test -c 6 -r 48000 -D surround51

(play an AC3 soundtrack/testtrack)

So exactly how is your AVR HT amp connected ?

### Post by BicyclerBoy on 2012-02-22
As "speaker-test -c 2 -r 48000 -D iec958" works then:
- your connection is S/PDIF
- stereo PCM over S/PDIF works.

Your test with chan-id.wav is playing as stereo because you have not selected the AC3 encoder (or pass-thru') in vlc configuration.

To test AC3/DTS pass-thru' you must do 2 or 3 of these 3 things:
- invoke vlc to use iec958
- set vlc to output pass-thru' AC3/DTS over iec958; play AC3 track
- or set vlc to encode to AC3 & output pass-thru' over S/PDIF; play mulit-ch track. 

VLC must have changed its cmd line options..
Find an AC3 test-track...

Note:
You can **not** play multi-channel audio over S/PDIF without **encoding** to either DTS or AC3.
The alsa a52 plug is/was a real-time AC3 encoder that outputs to S/PDIF.
VLC/MythyTV & XBMC have built-in AC3 encoders..


```

cd /tmp
apt-get source libasound2-plugins
apt-get build-dep libasound2-plugins
apt-get install libavcodec-dev
cd alsa-plugins-*
./configure
make
cd a52/.libs
cp libasound_module_pcm_a52.la libasound_module_pcm_a52.so /usr/lib/x86_64-linux-gnu/alsa-lib/
cd /tmp/alsa-plugins-*
cd /tmp
rm alsa-plugins-* -rf

```
The above code (that you used) compiles the a52 plug with libav* libraries.
This may work with ubuntu 11.04 11.10..
I think this bit is wrong:
/usr/lib/x86_64-linux-gnu/alsa-lib/ 
and on a 64 bit setup should be
/usr/lib/alsa-lib/

### Post by BicyclerBoy on 2012-02-24
This link
[http://www.alsa-project.org/~clemens/xonar-models.html](http://www.alsa-project.org/~clemens/xonar-models.html)

states that the AC3 & DTS encoders only work with windows; so probably no real h/w encoder just licenced software/driver feature..

So you only have (2) options & one of them is broken...

The alsa a52 plugin is specifically coded to match a couple of ffmpeg/libav* revisions. The libav* a52/AC3 encoder has been greatly improved but now the alsa plugin needs re-coding to be compatible.

Some version of the libav* AC3 encoder is used in VLC, MythTV, XBMC & mplayer
Mplayer cmd options:
-af lavcac3enc  # AC3 encoder filter
channels=6 afm=hwac3 # AC3 pass-thru

For AC3 encoder you would need to find a mplayer compiled with unstripped libav*.
Normal distro libav*, ffmpeg, mplayer are stripped of any code that may be copyright/patented/licenced/non-free.

The ffpmeg code shows that they had the channel order of a52 AC3 wrong at one point.
