### Post by balaknair on 2009-01-17
Chances are it's not uninstalling the older versions completely or isn't getting the correct precompiled modules from the repos.

To uninstall the old drivers safely:

ctrl-alt-F1 to get to text console--> Login with your user name and password

Stop the Xorg GUI
```
sudo /etc/init.d/gdm stop
```

Remove the Old drivers with envyng-
```
sudo apt-get install envyng-core
sudo envyng -t
```

Use the envyng text based utility to uninstall the driver.


Install the driver and necessary packages
```
sudo apt-get install build-essential linux-headers-`uname -r`
cd Desktop
sudo sh ./NVIDIA***.run
```

**OR**

```
sudo apt-get install nvidia-glx-180
```
This will install either the 180.11 or 180.22 drivers(depending on the repos enabled), but 180.11 will work fine
180.22 is available in the Medibuntu repositories
Add the medibuntu repositories to your sources list
```
sudo wget http://www.medibuntu.org/sources.list.d/intrepid.list --output-document=/etc/apt/sources.list.d/medibuntu.list
sudo apt-get update
sudo apt-get install nvidia-glx-180
```
After it's done, restart the GUI
```
sudo /etc/init.d/gdm start
```

Things should be working at this point.

### Post by balaknair on 2009-01-17
This is getting curiouser and curiouser...

Did you download and install the system updates **after** installing the drivers?
The 180.22 drivers work great for most newer cards. You should however remember that during the install, it compiles kernel modules for the kernel you currently. Updating the kernel later causes the modules to not work as intended.

Solution:
At boot-time, at the GRUB stage, select the current kernel version(I think it will be 2.6.27.9-generic) recovery mode--> boot up to the recovery menu--> select root console--> sh NVIDIA* -K

This compiles new modules for use with the updated kernel.

### Post by balaknair on 2009-01-17
> **flameproof said:**
> Yes, coz I wanted to start with a clean system. Was that wrong?

I'm afraid so. The driver installer compiles a module specific to the kernel that is running during the install. So when you update the kernel, the module compiled will not work with the new kernel. This would also cause the graphics drivers to break when doing a version upgrade(like from Ubuntu 8.10 to 9.04). If you were using the driver version in the Ubuntu repositories, the package manager would update the restricted kernel modules as well as the kernel itself, and this would not be an issue. That's why I suggested you use the driver from the repositories if the PC is for office use(and gaming or 3D eyecandy is not an issue), since stability would be the main requirement. Updates to the kernel would be less likely to break the driver modules for installs using the package manager.

If you want to use the latest drivers, perform system updates first, rebbot and then install the drivers. At the last step in the driver install process, select 'YES' for automatic configuration of xorg.conf(by default 'NO' is preselected), and you will have fewer problems with xorg.conf settings as well. When you have to do kernel updates, use the *sudo sh NVIDIA* -K* command to rebuild the kernel module(you might have to use the 'Recovery mode' to do this as described in the previous post).

EDIT: when using the command *sudo sh NVIDIA* -K*, remember to cd to the directory where you have saved the nvidia driver package(the NVIDIAxxx.run file; eg: cd /home/flameproof/Desktop-->  sudo sh NVIDIA* -K )

Great to hear you've got the graphics problems sorted out :D

**64-bit** is going to become more and more common in the near future. Most issues with 64-bit are history now, and hardware on Ubuntu 64 bit works 99% out of the box in my experience. I use the 64-bit native version of Flash and Java as well, and I have to say it works just great(Flash 64 bit works better than 32-bit IMO).

I've been using 8.10-64 for a while now(since Nov 1 to be precise), and I've never had any problems with hardware drivers. So far everything has just worked out of the box(the only additional stuff I had to do was installing the nVidia drivers). 

**Printers**
In my experience, most printers also work out of the box with Linux. I've got an HP MFD, an Olympus photo printer and a Samsung laser printer here, all working fine(the HP won't work with WindowsXP, the driver CD is scratched and unusable).
It's mainly Brother and Lexmark printers that seem to have problems with Linux, mainly models that don't have their own processor and use the driver to use the PC's resources. To know if they have native Linux drivers for your model, check this page
[http://www.openprinting.org/printer_list.cgi](http://www.openprinting.org/printer_list.cgi)
Didn't find the MFC-6490 there though.

I'd recommend HP, Epson and Samsung for laser printers, HP, Epson and Canon for inkjet printers and HP's MultiFunction Devices. I've 'set up' a few under Ubuntu for my own use and for friends/family. Most people have been amazed at how it all works out of the box(especially those who'd been tearing their hair out trying to get them to work with Vista). 'Set up' in Ubuntu consists of just plugging it in and turning it on.

To sum it up: 
***Keep the 64-bit OS for now***(as you've already decided to do), but if you have more than 3GB RAM, using 64bit is the only way you'll get the full benefit of your system resources. Use the 32bit version if you have to(to fix something that simply doesn't work in 64-bit). If you have <4GB memory(RAM+ video memory, and for practical purposes, the limit is closer to 3.2-3.5 GB) both 32bit and 64bit can handle it.

The printer MFC6490CW drivers- the ***CUPS drivers should work OK*** in 8.10-64.
