### Post by Peter09 on 2010-05-04
So here we are again struggling with those newbies (and some who should know better) about all the bugs in Lucid and how Ubuntu is not fit for the desktop and how the development team have release a rubbish product.
 
While, I suppose, you could point a finger at the complainers I think we need to look at how we brand these releases. For instance, Lucid, is branded as a Long term release, and while it is true, it most probably will not be completly stable for a month or so, if not more. Anyone looking for stability on their desktop and using LTS will suddenly get an O/S straight from development!
 
So what should be done - well my thoughts are that all 6 monthly releases are branded development releases, people are more likly to expect disruption for that sort of release and support conversations can start with - yes well it is a development release...
 
An LTS release is as now, but instead of coming straight from the development team it is actually, the last development release + bug fixes. So this LTS would be Karmic plus bug fixes.
 
This means that an LTS has had at least 6 months in the field before being released as a stable O/S.
 
Also, when new users go to the website to download the O/S - it should be clear that they have a choice between a stable and a development O/S.
 
I think at present we run the risk of losing the stability branding which Linux should be well capable of.
 
Edit: Just to clarify, this would mean that Lucid would have been released at the same time as the new LTS (Karmic+bug fixes).

### Post by snowpine on 2010-05-04

Hi Peter, all releases are, by definition, "straight from development" if you install them on the first day they are released. ;)

This discussion comes up every six months... "Lucid is buggy," "Karmic is unstable," "Jaunty is the worst release ever," etc. 

The "S" in "LTS" does not stand for Stable, it stands for Support. LTS releases need to be bold and forward-thinking. If Canonical puts only stable, well-tested software into an LTS, then they are stuck supporting it for 3 years. By that 2nd or 3rd year, the applications would be quite outdated. Instead, the developers look into the crystal ball and ask "what choices will be best for 1 year from now (when Hardy reaches end of life) and 2-3 years from now (when Lucid is mature)?"

In other words, LTS releases don't start off any more stable than regular releases (you could argue they are sometimes less stable, for example Hardy released with a beta web browser), but become more stable over their 3 years of bug fixes and security patches.

I am not at all worried about Canonical's branding, however. They depend on the throngs of enthusiasts who download on release day to be their *real* beta testers, while those who have been using Ubuntu for a while and need rock-solid stability are still using 8.04 on their desktops and 6.06 on their servers (or have migrated to Debian, CentOS, Red Hat, etc.).

Im going to offer two perspectives. 

1) If we must have a release cycle, I concur with the OP that stability needs to be more of a focus for the LTS release. 2) While I realize this will likely not ever happen, why cant we use a rolling release for the development release? The development release moves along at a pace somewhere between Debian Testing and Arch, and a seperate branch for the LTS focuses on eventually committing a release (where the software is fixed like a release). So, say an LTS comes out every 3 years- 6 months before the LTS is due out as "stable," the devs take a snapshot of the rolling release at that moment (while the development version keeps rolling), decide if they want to incorporate any upstream software that would benefit the release, and then do nothing but bug fixes. Arch and PCLinuxOS stay rolling. Debian is similar, but with Debian the entire distro goes through a freeze period (except Sid which is brutally unstable for a new user) and the focus is not on ease of use for the user.

Just my thoughts..

An LTS release is as now, but instead of coming straight from the development team it is actually, the last development release + bug fixes. So this LTS would be Karmic plus bug fixes.
 
This means that an LTS has had at least 6 months in the field before being released as a stable O/S.
