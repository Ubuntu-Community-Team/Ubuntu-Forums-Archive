> **NightwishFan said:**

This argument can go both ways.

One could make the argument ; that because closed source software first has to be reverse engineered (have fun fuzzing and going blind on OllyDBG) to create an exploit, it's actually a little better off. You can download the source code for whatever open app you want, find where it's not managing memory properly, and bang there is your buffer overflow. Granted, this is a very simplified example that doesn't take things like ASLR or DEP into account. However, at its core it is a valid point.

Of course the reverse argument is there, the more people that can see the source code, the more like a 'good guy' will find the vulnerability and patch it before exploit code is released by the 'bad guy' or the flaw is even discovered.

I think the real strength comes in not with whether the weakness was there in the first place, since you can make the argument that open source programmers are by and large hobbyists, they are not getting paid, and there is much less quality assurance.  However, like I said, the open source community generally tends to be a little bit ahead of the curve in patching security vulnerabilities, although most respectable closed source vendors (with the exception of apple) do just fine on keeping up with the 0 days. 

In closing, I don't really think the proprietariness (yes I know that isn't a word) of a piece of software really has much to do with its overall security.
