### Post by quequotion on 2011-06-12
There is another way to upgrade.

The Ubuntu method is a radical departure from what's outlined in the Debian manuals.

I've been working on a better, more old-fashioned, way of doing things for a few versions now.

I apologise to double post, since I just attached these scripts to another thread, but I'd like to paste them here for exposure and discussion.

It is ridiculous how much trouble upgrading Ubuntu can be

```

#!/bin/bash

# Ubuntu Upgrade script for special cases
# COMMENTS IN ALL CAPS ARE NOT YET IMPLEMENTED

# Ignore user-added sources for phase one
for f in /etc/apt/sources.list.d/*.list;
	do
		mv "$f" "${f%list}list.ignore";
	done

# Set all main sources for Natty
sed -i 's/maverick/natty/g' /etc/apt/sources.list

# Clean apt's cache and update package lists
apt-get clean
apt-get update

# Add pin to ensure Natty packages (even downgrades)
echo -e "\nPackage: *\nPin: release v=11.04\nPin-Priority: 1001" >> /etc/apt/preferences

# Download packages from main sources
# COLLECT THE LIST OF "REMOVED" PACKAGES
# ENABLE TO RESTART AT THIS POINT IN CASE OF INTERRUPTION
apt-get -d dist-upgrade

# ADD A FINAL CONFIRMATION STEP, HERE THERE BE DRAGONS


# Do the main upgrade
# ENABLE TO RESTART AT THIS POINT IN CASE OF INTERRUPTION
# ENABLE ERROR AND CONFLICT RESOLUTION
apt-get dist-upgrade

# RESOLVE DEPENDENCY CHAINS

# This concludes phase one.

# Resolving dependency chains manually (not the only way):
# Force install packages overwriting old ones with:
# dpkg -i --force overwrite /var/cache/apt/archives/NEW PACKAGE NAME
# Then remove old packages with:
# apt-get remove OLD PACKAGE NAME
# Then reinstall the new package
# dpkg -i /var/cache/apt/archives/NEW PACKAGE NAME

# Make sure everything is done
# apt-get dist-upgrade
# dpkg --config -a
# update-initramfs -u -k all

```

Obviously, there's a lot still to be done. The most intricate part is resolving the inevitable dependency clashes and overwrites from changes in package name or structure.
Most often a new package wants to overwrite files in an obsolete package and the package maintainer has not added this conflict to the new package's dependency chain.
This method does minimise the number of packages slain on the upgrade altar, but apt still tends to remove a few user-selected packages without giving any good reason why.

```

#!/bin/bash

# Re-enable user-added sources
# DO NOT ENABLE APT-BUILD IF THERE IS AN OVERRIDING PREFERENCE SET
for f in /etc/apt/sources.list.d/*.list;
	do
		sed -i 's/maverick/natty/g' "$f";
	done

# Clean apt's cache and update again
# TEMPORARILY DISABLE 404 REPOSITORIES TO SAVE TIME
apt-get clean
apt-get update

# Download packages from user-added sources
# COLLECT THE LIST OF "REMOVED" PACKAGES
# ENABLE TO RESTART AT THIS POINT IN CASE OF INTERRUPTION
apt-get -d dist-upgrade

# ADD A FINAL CONFIRMATION STEP, HERE THERE BE DRAGONS THAT EAT YOUR DATA AND BLOW UP YOUR PC

# Do the user-added upgrade
# ENABLE TO RESTART AT THIS POINT IN CASE OF INTERRUPTION
# ENABLE ERROR AND CONFLICT RESOLUTION
apt-get dist-upgrade

# Resolving dependency chains manually (not the only way):
# Force install packages overwriting old ones with:
# dpkg -i --force overwrite /var/cache/apt/archives/NEW PACKAGE NAME
# Then remove old packages with:
# apt-get remove OLD PACKAGE NAME
# Then reinstall the new package
# dpkg -i /var/cache/apt/archives/NEW PACKAGE NAME

# Make sure everything is done
# apt-get dist-upgrade
# dpkg --config -a
# update-initramfs -u -k all


# REMOVE PIN FOR NATTY PACKAGES IN /etc/apt/preferences
# CHECK THUROUGHLY FOR ANY LEFTOVER MAVERICK PACKAGES

# This concludes phase two.

# Leftover Maverick Packages can be found in Synaptic
# Sort the package list by "Origin" and look under "Local"
# These packages are not available in a repository.
# Maybe they are hand-made, obsolete, temporarily unavailable, or discontinued.
# Some are probably obsolete Maverick packages.

```

This is the part that matters most to me, because I keep my system on the bleeding edge most of the time with a lot of ppa packages.
Ubuntu's upgrade method typically removes all ppa packages and disables all ppa sources without even telling the user what it's doing or why.
There were a few obsolete packages left over, but not many and nothing that was in the way.

I did everything in those scripts manually and had a pretty smooth upgrade experience this time, aside from some Gnome/Unity/Compiz/Nvidia issues.

[I]"About Nix
Nix is a purely functional package manager. This means that it treats packages like values in purely functional programming languages such as Haskell — they are built by functions that don’t have side-effects, and they never change after they have been built. Nix stores packages in the Nix store, usually the directory /nix/store, where each package has its own unique subdirectory. 

Multiple versions
You can have multiple versions or variants of a package installed at the same time. This is especially important when different applications have dependencies on different versions of the same package — it prevents the “DLL hell”. Because of the hashing scheme, different versions of a package end up in different paths in the Nix store, so they don’t interfere with each other.
An important consequence is that operations like upgrading or uninstalling an application cannot break other applications, since these operations never “destructively” update or delete files that are used by other packages."
