### Post by ericesque on 2008-10-18
Honestly, your best bet is to determine what you want to develop today. If there's some type of application or process you want to automate, tell us what that is, and we can recommend a language for that task. 

People get very hung up on learning the 'right' language. The reality is, the more tools you have in your toolbox, the better off you are. As you get into programming, you'll find that lessons learned from one language will benefit you in the next.

I'll debunk one more myth before I'm done with my rant here. In college I learned the 'basics' of a half dozen languages. I got to a point where I could read just about any language thrown at me - even without a reference for the syntax. But I had this nagging doubt in myself that all I knew was if/then and loops in a number of languages. I thought that I should really get to know one language really well in order to be a real programmer.  

Well the truth of it is, about 80-90% of programming is if/then and loops. The rest is seldom used and you'll never use any of it often enough to commit it to memory. So don't concern yourself with 'mastering' a specific language. Just learn a language that suits your current task.

### Post by Delever on 2008-10-20
It is not that C++ is hard to understand. For me it was amount of knowledge required to get it going.

For example, compiling, linking, different compilers on different platforms, platform-specific code, strings, std library, automake or make tools. When you get this going, you obviously want to reuse existing open sourced code, then you face the need to link libraries, learn about different types of libraries, again, platform specific things, then, selecting the way to handle certain things (like: threads, user interface, unicode).

You can remove much from list above if you target single platform. You can remove almost all of this if you are only **learning** programming. In my opinion, to learn basics, C is ok. To learn objective programming, C++ is almost ok too. To create something more, learning required grows much more than using other languages. However, with C you are closer to hardware and after several segfaults and head banging to the keyboard, you learn the real way computer is working. You learn to be careful about what you do, learn not to leave loose ends in code, and even write some comments *where they are actually needed*. After that, python it's so easy it's not even funny.

For me, C/C++ was deeper dive then anything else, but I find it rewarding. To be fair, I knew some other languages before starting (Python, Java), but they were no help with things I mentioned above.

Python is invaluable tool for writing small scripts, like, removing certain symbol from every line in text file, renaming lots files, etc., so it is easy to pick up, to make it do what you want.

C++ - more work before fun can start
Python - less work before fun can start

### Post by enigma_0Z on 2008-10-21
It, as many people have said, what is "best" depends on your application... a little like one user said, "what type and size of wrench should I use?" There are ratchets, open-ended, closed-ended, drill-powered, impact, etc. They all accomplish the same task (removing a bolt), but they do it in different ways and that makes them better for certain applications

If you're writing a game or application of some sort, I would too recommend either C++ or Python, depending on what you intend to do,  how quickly you want to produce results, and how much time you are willing to invest in learning your programming language of choice. However, if you are making a game or any relatively complex application, I'd recommend C++ or Python because they are both Object-oriented or OO languages. This little feature will save you time and frustration with most apps that you develop.

If you're writing a script to automate a task of some sort, I'd recommend Python, Bash (Linux shell), and then Perl. Perl, while probably the fastest, is also a pain in the butt to understand... it's an entirely different mentality than other "standard" languages, and takes some effort to understand and become efficient.

As far as capabilities, Python is easier to learn and seems to me to have a larger library of "end user" libs which provide a high level of usability. While there are more libs written exclusively for C++, the python libs are easier to use, even easier than their C++ equivalents (SDL anyone?). Python, in my experience, comes with a boatload of "modules" or libraries that add to it's functionality in useful ways. While you can accomplish the same tasks with C++ and a default install, it usually takes more work.

C++ does, however beat Python in speed by a margin (applications run faster), for the simple reason that C++ is closer to the hardware than Python. This speed comes at the cost of added complexity (working with pointers, etc. (YUCK! - I hated pointers))

Python's documentation, in my experience, is easier to read and understand than documentation for C++, but there is less of it, especially for C libs that have been ported to Python. For these, the documentation is typically a straight copy and paste and you have to fill in the gaps with what you know about python's quirks vs. C/C++.

Lastly, C/C++ allows you to do slightly more than python. Most of this is related to the fact that C++ has a lower level of access to the hardware than python. You can do some really neat low-level things in C++ that you can't do in python, but I have not encountered any high-level tasks that I could do in C++ that I couldn't do in Python.

All in all, Python offers the classes and (most of) the inheritance of C++ without all the stuff that makes programming difficult. Even at that, it's still pretty fast. Those two reasons are why I recommend python for most tasks.

### Post by Delever on 2008-10-22
For example, game Civilization IV, it's core is written with C++, however, upper layer is done with python, players also can extend it by adding mods and other cool things using python.

It makes perfect sense to use C++ (or C) where the biggest bottleneck is, and use python where speed is not so important.

To extend this example, suppose you have few different problems. One: produce as much frames per second as possible. Second: handle game logic.
First is better written with C++, yet second benefits from python. You may say that game logic still requires constant checks and evaluations of game state, but actually, compared to the rendering pipeline to be executed every second, those checks take little time.

Yet, i do not recommend to delve into the region of combining these two methods for your first code. Computers are fast nowadays, and python is fast too. If you don't believe, try it. There are thousands of ways to write inefficient and slow code in both C++ and python, so, what can you loose.

I suggest to do and actually finish something small at first. For example, take one of those well-known default games like tetris, snake, lines, and remake it with OpenGL. Then you will know more about time it takes and what you can do.

Look at it this way: to use OpenGL, you will need to learn to use it. The function names you call to manipulate OpenGL from python are the same to manipulate OpenGL from C++. The same is true with GTK+ library, and with many other things.

This illustrates that more clearly:

To draw a square with Python:

```

	# Draw a square (quadrilateral)

	glBegin(GL_QUADS)           # Start drawing a 4 sided polygon
	glVertex3f(-1.0, 1.0, 0.0)  # Top Left
	glVertex3f(1.0, 1.0, 0.0)   # Top Right
	glVertex3f(1.0, -1.0, 0.0)  # Bottom Right
	glVertex3f(-1.0, -1.0, 0.0) # Bottom Left
	glEnd()                     # We are done with the polygon                           # We are done with the polygon

```

To draw the same square with C:

```

    /* Draw a square (quadrilateral) */

    glBegin( GL_QUADS );                /* Draw A Quad */
      glVertex3f( -1.0f,  1.0f, 0.0f ); /* Top Left */
      glVertex3f(  1.0f,  1.0f, 0.0f ); /* Top Right */
      glVertex3f(  1.0f, -1.0f, 0.0f ); /* Bottom Right */
      glVertex3f( -1.0f, -1.0f, 0.0f ); /* Bottom Left */
    glEnd( );                           /* Done Drawing The Quad */

```

You see, syntax difference is not the hard part. Programming is harder part :). And knowledge won't be lost if you learn it with python.
