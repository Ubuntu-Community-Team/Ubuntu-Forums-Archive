### Post by nvteighen on 2009-04-23
Let's be fair: C++ is not gone

But it's not a good language to start with. Why? Too complex, too many features... a way more than a beginner needs. It mixes two layers of programming: abstract constructs nearer to human mind and internal computer workings that are just "mechanical" issues. The RAII pattern eye208 talks about is an example of this: you can use it properly only if you know how the call stack works... which is trivial, but it's an internal issue you shouldn't be tracking if you're coding something that works on base of abstract objects you're creating (e.g. a "task" object in a schedule application).

The best is to learn to program in a language that helps you to map your thoughts better into the code. It will ease your journey a lot as you'll be more focused on learning how to program and less on learning how to use the chosen language. And a language that avoids those computer implementation details is also needed at a starting stage. Python is suitable because it's simple and maps well to almost anyone... of course it's not the perfect language... and it may be the case that someone rather needs Perl or Ruby... but hardly C++ (again, **hardly**... maybe you can deal with it).

Read the stickies for more and better information than this post :)

### Post by eye208 on 2009-04-25
Here's why Python is not suitable for beginners:

C++
```
#include <iostream>

struct Demo {
	Demo() { std::cout << "Requesting exclusive access to resource X" << std::endl; }
	~Demo() { std::cout << "Releasing X" << std::endl; }
};

int main() {
	for (int i = 0; i < 2; i++) Demo obj;
	return 0;
}
```

Python
```
#!/usr/bin/python

class Demo:
	def __init__(self):
		print "Requesting exclusive access to resource X"
	def __del__(self):
		print "Releasing X"

for i in range(2):
	obj = Demo()
```

Both programs should do the same thing, right?

Here's what happens:
```
$ ./demo_cc
Requesting exclusive access to resource X
Releasing X
Requesting exclusive access to resource X
Releasing X
$ ./demo.py
Requesting exclusive access to resource X
Requesting exclusive access to resource X
Releasing X
Releasing X
$
```

If the "Demo" class actually did what it said, e.g. open a file for writing, the Python version would break. To fix it, you would have to resort to **manual resource management**, i.e. release X through an explicit method call or by triggering the garbage collector manually.

Some people here say that garbage collection makes things easier, but in fact it does not. It only makes things less predictable. In C++ object destructors will be called immediately and in the correct order. Manual resource management is not required. That's what RAII is all about.
