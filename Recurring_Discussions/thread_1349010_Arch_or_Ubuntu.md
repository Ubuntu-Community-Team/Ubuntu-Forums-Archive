### Post by murderslastcrow on 2009-12-07
Arch is faster for some people. Might be the fact that there are more Canonical-provided services and daemons included in Ubuntu, but in Arch you only choose what you need.

Also, Arch is a rolling release distro. This means no upgrades- you just get the newest stuff constantly. You're basically constantly using a beta release.

For most users, Ubuntu is good enough, and includes a lot of niceties that are a pain to install from scratch on another distro (out-of-the-way notifications, indicator applet, Ubuntu One, Software Center), but all in all if you're a regular Linux user it might appeal to you to experiment with a release that has more recent packages and focuses on being lightweight.

### Post by Pogeymanz on 2009-12-08
Basically Arch is more geared toward the do-it-yourself Linux users. It doesn't install X for you, therefore it has no favored DE or anything like that.

It follows suit from BSD in several regards. It uses BSD style init scripts which are much easier to configure than other Linux ditros'. This means that it is very easy to add a new default network connection and add or remove daemons or modules.

It also has ABS, which is similar to BSD's ports system. You can always compile an app from the repos with an easy-to-read-and-edit PKGBUILD file if you want to enable some option that is off by default in the vanilla binary version.

It's package manager is Pacman, which I find to be better with dependencies than Debain's apt. 

I know I'll start a flame-war, but I really hate how Ubuntu adds unnecessary dependencies to packages (mostly making everything require Gnome). This doesn't happen in Arch, as the devs try to keep all packages as close to upstream as possible.

It sounds like you would enjoy Arch based on your description of other distros you've tried.

### Post by kevCast on 2009-12-11
> **Pogeymanz said:**
> Hey, those BSD init scripts and ABS make Arch a lot more like BSD than most Linux distros are. At least, fundamentally speaking, you can compile software in the repos or get the binary package. Only Gentoo and Arch have package management like that in the Linux world.

Could you elaborate? I admit I'm a BSD newb, but it seems like distros like Arch and Gentoo are somewhere between Linux and BSD. And how is the ports system better? (Not being antagonizing- I want to know if I should switch!)
They are similar in that they both intend to be POSIX compliant. And that's where it ends. BSD has a much cleaner layout, a much cleaner code base, and every line of code has a lot of thought put into it. It's developers are meticulous in their coding. BSD is known to have incredibly clean code (if you have the time, look at the source code for OpenSSH. Or any NetBSD code. It's beautiful).

Linux is chaos. The kernel commits on a single day are staggering. Much of this has been controlled since git was implemented. Now every developer has their own sandbox to do whatever they want. Still, the amount of development Linux gets at one time is considerably more so than BSD.

Some BSDers would say quality over quantity, although that would make an assumption that the stuff pushed into Linus' git tree isn't quality. What makes the Linux development model work is the chaos, and everyone pushing what they want to see into the system, breaking it down into small parts. Intel works on incorporating their video drivers, IBM wants it to work on their supercomputers, Novell on their mainframes, etc. These small contributions lead to large and varying support. However, this does make it harder to maintain, and the potential for bugs greater than you'd see in a BSD variant.

There has been borrowing from both camps. BSD code was in early GNU/Linux code, as well as proprietary software like Microsoft's TCP/IP stack. BSDers use the GNU coreutils on their systems (although that's mainly out of necessity). More significant things have been taken from the BSD code base in my opinion, and some users are touchy about this. BSD's base is monolithic. GNU/Linux is broken into parts.

Gentoo has portage, inspired by ports. Here you start to see what I'm talking about as far as elegance. FreeBSD & OpenBSD use ports, NetBSD uses pkgsrc. To use either, you extract either a tar.gz from an FTP/HTTP mirror to /usr, or you can use CVS to stay up to date. So, let's say I want to install Firefox. I cd to /usr/ports/www/firefox. There will be a makefile and some notes. That's it. You change the make file to use compile options you want (if you want to). make && make install clean. Done. make uninstall or equivalent removes it.

Gentoo took that idea and implemented python scripts, make.conf, rsync, and a lot of other tools until it was more of a Gentoo thing than BSD. Although it was inspired by it, a lot of FreeBSD users would say it's over engineered, and some would even say it ruined the ports sytem. Gentoo users might say it's better (I'm not a fan of either. It's mainly opinion on these things).

Arch has some BSD inspired things in it as well: the rc.conf, BSD scripts, and the ABS. I haven't used the ABS, so no comment on that. Arch is much more bleeding edge than the BSDs, and the rc.conf is not the same used in FreeBSD. Arch uses a binary package manager as its main way of package management, using .tgz packages. It's very simple, and easy to use, however it's bleeding edgeness makes me hesitant to stick with it.

I don't think the ports system is better. I don't agree with the idea of foo_1 > foo_2 as far as package managers. I'm an aptitude fan myself. Some people detest it.
