### Post by ErikNJ on 2010-12-21
There is nothing wrong with standards (like "LSB") and there's no reason the standards have to stifle innovation.  It's also acceptable to adhere to a standard that allows for changes.  Simply, an application may require certain libraries to "work" but if the developers require a non-standard library, there's not reason not to statically link the library and include it within the entire application.

Also, sticking with the standards of putting application in their appropriate directories would be really nice.  If anyone wants to be a weirdo, feel free to install to /whatever-the-hell/weird and place the appropriate sym-links in the directories sane people would use (like /opt or /usr/bin/local).  ;)

For the most part, every distribution is using a lot of the same software (libraries) but possibly different versions of the software.  Ubuntu (non-LTS) and Fedora tend to be closer to the latest and greatest while Redhat or Debian (stable) tend to be a bit older.

What would be cool is if newer applications could easily be compiled with statically linked libraries at the user's choice - thereby negating the need to upgrade some library that could cause a bunch of other stuff on the system to break.  I'm also a Linux noob, so there may be an easy way to do this that I am wholly unaware of.

For the most part, a developer could create a DEB and an RPM and have most folks covered.  Have a look at what the latest Google Earth downloads are (.deb and .rpm).  The other requirement is LSB 4.0 (aside from system to system oddball variants - in my case this meant adding the 32 bit nvidia libraries to my 64 bit install).

### Post by lykwydchykyn on 2010-12-22
- There is nothing preventing an ISV from compiling static binaries and writing a custom install script/wizard to install their software.  Nothing.  Google Earth does this, for example.  Oracle does this.  There's no reason an ISV *has to* use the package management system or dynamic linking, nor is there any reason a graphical installer can't be created (Heaven knows we have enough toolkits running around).

 - If a company is going to go to the trouble (read: perceives adequate ROI) of porting its code to Linux, adding a couple extra build scripts to the end of the process is trivial.  It's not like the program has to be completely rewritten to support .deb or .rpm.  Heck, there are even [automated build services](https://build.opensuse.org/) out there to automagically build packages for multiple distros/package formats.  This is just not the impossible hurdle people make it out to be.

 - The package management system is *central* to what Debian does.  Think about what Debian is for a second:  they primarily take upstream sources (the kernel, X11, GNU, etc) and package them for easy install.  Their primary activity isn't software development, it's packaging.  Switching from .deb to .rpm would not be some trivial weekend project for a distro like Debian.  It would probably mean throwing out Debian entirely and starting from scratch. Same for Redhat or any other non-derivative distro.  Is that reasonable to ask of these projects, for the unproven assertion that doing so will increase Linux usage and ISV support?  It's like asking countries to abandon their native language in favor of English because you're pretty sure it would bring world peace.

### Post by Shining Arcanine on 2010-12-25
> **madjr said:**
> If BSD is so good then why wasn't it the popular one instead of linux?
oh wait let me guess, that darn bsd license.

BSD is an operating system that was developed by the University of California at Berkeley with funding from the US Department of Defense. In 1993, President Bill Clinton cut defense spending, which meant that funds for BSD development were cut. In 1995, the University of California at Berkeley disbanded its operating systems group. BSD was then forked and now has several successors, the most prominent of which is Mac OS X.

If you consider the various forks of BSD to constitute BSD marketshare, then it is clear that BSD is far more popular than Linux. Of course, no one thinks that way.

BSD was developed by PhDs and its modern-day forks reflect that. Each fork of BSD is developed within a single organization that has a common goal and vision, which creates a consistently designed operating system.

Linux was designed by an undergraduate student as a hobby and it largely reflected that until the Linux 2.5.x rewrite, which was done by professional developers working in conjunction with Linus Torvalds. While the Linux kernel has certainly improved, it is useless without a corresponding userland. Linux distributions are formed by having other people put together userlands for Linux. This has heavily fragmented the userlands used on different Linux systems. The majority of the fragments use the GNU userland. The GNU userland was meant for Richard Stallman's GNU operating system. The only reason why the GNU userland works with Linux is because the GNU kernel is not ready for public use.

Richard Stallman et al and Linus Torvalds et al have significantly different opinions of how an operating system should be developed, which has lead to both groups making changes that they know will cause problems in what the other group is doing without telling them. A recent example of this is the reverse memcpy() implementation in glibc, which was changed by the GNU people without notice, so any code depending on how it was designed broke:

[https://bugzilla.redhat.com/show_bug.cgi?id=638477](https://bugzilla.redhat.com/show_bug.cgi?id=638477)

With that in mind, the next time someone talks about something that you do not understand, try asking about it so you can avoid making an ignorant remark that you will ultimately regret.
