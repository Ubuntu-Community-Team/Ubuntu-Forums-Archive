### Post by Paqman on 2010-05-10
The only disadvantage of 64-bit is that programs will take up slightly more space in RAM than 32-bit ones would. But unless you're really low on RAM, who cares? 

64-bit used to be a minor hassle to run a few years ago. The lack of 64-bit Flash, Java, etc meant it was easier to run a 32-bit copy of your browser on a 64-bit system. Not a major hassle, but did require a bit of user input to set up.

Windows however was a different story. When XP 64-bit was launched support from hardware and software vendors was atrocious, and 64-bit users had a hard time. It's mainly for this reason that 64-bit OSes have been tarred with the "incompatible" brush, whether they deserve it or not.

### Post by lostinxlation on 2010-05-11
> **chappajar said:**
> 
FYI, wider registers are not slower by definition.
 
I think what he meant was access time of the general purpose register could be slower. The word lines of 64bit registers have to travel twice as long as 32bit counterpart and it means more R and C which results in slower access time. In the same way, more entries require taller physical footprint which makes longer bit line that also costs delay. Having said that, the critical paths could be somewhere else, maybe cache access, where there are much mreo entries than general purpose register, much wider line size, and Tag read has to decide whether the cache hit or miss.
 
 
 
There are always pros and cons on 32b vs 64b.
Off the top of my head...
 
1. 32b mode uses less part of the circuitry than 64b so that if critical paths are on 64b specific parts, 32b can run on faster clock than 64b. If the critical paths are on the common part of 32b and 64b, the clock speed would be the same.
2. 64b mode has more registers which helps less memory/D-cache reference. Mem/D-cache access always takes extra cycle than dealing the operands on register files, and it hits the performance.
3. 64b mode's more reigster entries make the context switch as well as trap handling take longer.
4. 64b mode can handle extremely large number by 1 instruction, while 32b mode requires multiple instructions to do the same.
5. Newer 64b mode has some new instructions which may help the performance.
6. 64b mode has a little longer instruction length than 32b which may result in slightly higher I-cache miss.
7. 64b mode eats more power which causes greater IR-drop, and generates more heat. Both could affect the timing and clock might get slow down.
8. And of course, 64b can use larger address space.
 
 
I haven't done much research on #5 and #6, so correct me if I'm wrong. 
 
 
Except address space issue, what comes in favor of or against performance depends on the nature of the program.

### Post by chessnerd on 2010-05-11
The only advantage of x86 is that it uses less RAM. 

My 64-bit install on my laptop starts out using about 360-400 MB while the 32-bit on my desktop uses only 220-280 MB at startup (about 120 MB more). If you have more than 1 GB of RAM this isn't much of an issue and you probably won't notice any performance difference. (On Windows this is more pronounced and 64-bit Vista and 7 use a lot more RAM (I think about 400 MB more) than their 32-bit counterparts.)

So, here's the deal: if you have more than 3 GB of RAM, use 64-bit. 32-bit only handles about 3.25 GB, so if you have 4 GB or more, you'll really want 64-bit to take advantage of it.

If your CPU supports 64-bit and you have, say, 2-3 GB of RAM, you should strongly consider the upgrade to 64-bit because of the computational improvements. The increase in RAM will be minimal.

However, if your computer only has 1 GB of RAM (or less) you might want to stick to 32-bit because of the increased RAM usage.

And finally, if your CPU doesn't support 64-bit, then you can't use it anyway and should stick with 32-bit.
