### Post by Zorael on 2010-03-25
I run KDE on my machines. I don't really have any issues beyond what's plaguing linux as a whole (hardware compatibility) and whatever bugs I run into when using development builds (lucid). Admittedly, I don't use scanners, nor email or address book apps.

I don't think I need to *limit* myself to either framework (Qt4/GTK), so I happily run Chromium next to Dolphin. I don't use Konqueror for anything else than to test webpages that don't load in Chromium for whatever reason, but I do use Kopete instead of Pidgin. That does mean I need to load the GTK libraries into memory whereas otherwise I could just cruise-control with shared memory of the Qt4 ones, but my netbook has 2gb of ram. And the space those libraries allocate pales in comparison to what Chromium will hog up over time. I don't use KPackageKit nor Synaptic; I like the terminal aptitude command.

The first bootup of KDE *will* be slow, as the tool that generates icon caches and populates the list of applications (and their supported mimetypes) for the menu and the runner (kbuildsycoca4) has to do it from scratch, and not just do an incremental run. Also, nepomuk might start indexing your files' metadata, which will also contribute to a feeling of the system being slow. Disable it if you don't foresee yourself tagging files.

The Plasma desktop shell is only as sparkly as you make it. If you remove everything but the panels and set your activity to be a folder view (instead of an empty workspace housing a folder view widget), it's pretty basic. Moreso if you remove the standard application menu panel widget and replace it with a classic one.

My favorite plasma widget is an on-off button (from Sam Rog's ppa, source on kde-look.org) that sends a STOP signal to all Chromium and Flash processes. So when I'm not using the browser, I can pause the processes making them use literally 0 cpu time. We all know Flash likes to sit and muse for itself, and some JS-heavy pages likewise. With that, they can't. Clicking it again sends a CONT signal.

### Post by Simian Man on 2010-03-26
Usually I don't like the "there is too much fragmentation" kind of argument, but I *would* like to see more homogeneity in the GTK development world.  It strikes me that most core GTK and Gnome developers write all of these great libraries in C that nobody wants to use in their raw form.  Instead, we have people writing bindings for C++, Python C# etc. and *then* application developers get to work writing GTK applications.

I realize that some people do write desktop applications in C, but these people are either stuck with legacy codebases that are, oftentimes, fairly stagnant and unmaintainable (Rhythmbox, Gimp etc.) or they are choosing their tools out of ignorance or for non-technical reasons (ie they mistakenly believe that only C will deliver performance or because it's the "official" Linux devlelopment language).

So we have people developing code in C, which is frankly an exercise in tedium and pain compared to modern languages, people writing and maintaining disparate language bindings, which are also not fun at all to work on, and then people writing GTK apps in several different languages.

I would really love to see them move away from C, and start using C# as the official language of GTK and Gnome.  It would take a long time to transition, obviously, as there is tons of old code, but it would breathe new life into the project and it would be the better for it.

Plus it might push away the BoycottNovell crowd once and for all which would be a blessing in and of itself.

Actually, GNOME is going through a phase out for deprecated libraries and unwanted dependencies for Gnome3 - libgnome, libgnomeui, libglade, bonobo, all Gtk-Deprecated-Symbols removed, freeing WebKit from Xulrunner, and the CORBA infrastructure replaced with a DBUS infrastructure.

According to their [release schedule]("http://live.gnome.org/TwoPointTwentyseven"), most of this has already been done over the last 6-12 months...
