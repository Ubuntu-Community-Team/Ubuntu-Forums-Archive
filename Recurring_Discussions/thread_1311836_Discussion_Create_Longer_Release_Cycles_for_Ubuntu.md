****************

So, here's my idea:
Because Ubuntu is almost ready to compete with Windows & Mac, the 6 month release cycle is out-of date. This is why: 

1. Ubuntu has lost most of its small bugs but may need some serious improvements that require a longer development cycle (cloud computing, for example). 
2. including the alphas, betas and RC there are about FOURTEEN ubuntu releases every year now. 
3. The Release Candidate needs much more time to be tested than the week it has now. Example: in the Netbook Remix final 9.10 are serious kernel problems for the most common netbook models (Asus 1000 & MSI Wind) which could not be fixed in time and are included in the final now (!). A serious Release Candidate needs at least one MONTH of testing! 
4. Home customers and companies don't want changes every 6 months. Think about Windows XP (which is almost 10 years old and STILL has 60% of the OS market!!) 

So let Ubuntu become the successor of windows XP by loosening the release schedule. This is what I propose: 

Release only one Ubuntu each year (in October), but with MORE Alpha releases so the total amount of Ubuntu releases per year is diminished from 14 to 8 or so. The release schedule could look something like this:

nov: Alpha 1
dec: Alpha 2
jan: Alpha 3
feb: Alpha 4
mar: Alpha 5
may: Beta 1
jul: Beta 2
sep: Release Candidate
oct: Final Release
Every 3 years: A Long Term Support Release

This way developers have more time for complicated features, and home users and companies get more stable, better tested releases they can rely on.

I am wondering what other people are thinking about this idea; so let me know, both by adding comments and voting :)

Lets all help make ubuntu better!

> **slumbergod said:**

I would not call that release note; its a damn long BUG LIST! You don't include that much bugs in a serious final release; it's just  cutting in your own hands! If your stuff is not yet of good quality; dont bring it to the market!

Many people seem to believe that increasing the length of the release cycle automatically and necessarily allows for more time for fixing bugs. Here's some simple math to disprove that assumption, from a slight re-edit of an old post of mine:

[QUOTE=23meg]Putting out an up-to-date distribution in *2n* months instead of *n* months doesn't give you twice the time to prepare and doesn't automatically make the distro twice as stable, because more new software and new technologies come into play at more or less the same rate in any given window of time, and to make the release stable, you have to put in an equal amount of work. In rough terms, twice the quality assurance effort is needed for *2n* months of development time instead of *n* months, because upstreams are busy pumping out new releases of their software, and hardware vendors aren't stopping either, so the kernel and drivers have to catch up.

If you release every 12 months, people miss the opportunity to use the latest versions of most software that comes and goes between the releases for a whole year, and that's a huge inconvenience. Make it 6 months and you cut the inconvenience to half in the worst case (there may be better cases because new libraries may not begin to be required by new software in a period like six months, so upgrading may be less problematic). Also, since "in-between" releases of existing software will not ship with a sparsely released distribution, the software will get tested less intensively, get less bug reports and feedback, thus improve at a slower pace. [/QUOTE]

As long as the work done per unit time remains constant, stretching the periods does not help.

### Post by Devport on 2009-11-02
There is no need for any full distribution releases at all. Software progresses gradually. Ubuntu consists of 1000`s of software packages and they are not aligned to a common schedule and since its open source never ever the majority of software will apply to a fixed schedule - people work on the software in their spare time.

_There should be distribution states from which a user can choose ( during installation or later on with synaptic ) :_

- **stable** : This state should be used by default targeting end user / production systems

Only packages would be installed that have been heavily tested ( e.g. they would be as stable as packages should be for a full release in Ubuntu`s current schedule ).

- **testing** : This state would be used by power users, geeks and testing volunteers

Those packages would be as stable as they are in the current Ubuntu beta/rc release phase. They might contain bugs.

- **development** : This state would be used by developers or curious geeks

Those packages would be under heavy development and might cause system instability.

- **possibly a LTS state** : Packages in this state would be rock solid and they would have near to 0 open bugs

Using this state system wide would enable all LTS support options.


States should be selectable distribution wide ( all packages ) and on a per package selection ( all packages depending on another package in a certain state receive the same state ) because developers would only choose the packages they work on to be in development state without the whole system possibly becoming unstable.

To prevent any possible problems threw updates any change done by a package may be reversible till the next update of the same package so that changes between states would be possible and safe.

Installation / Live media would be generated automatically regularly based on the stable or LTS  state.


Some advantages of this system :

- users would get feature / security updates faster ( not only every 6 months )
- users would get updates more regularly
- LTS wouldn`t mean having to cope with outdated software anymore
- the work for developers, maintainers and distributors due to bugs and development would be distributed over the whole year without those peaks due to full releases every 6 months
- no more decreasing release quality due to fixed release dates coming too fast to fix remaining bugs
...

Implementing this idea might involve a lot of work but the outcome might well be worth it.

### Post by wildweathel on 2009-11-05
Using older software only decreases bugs if someone else is finding and fixing them.  Longer cycles, in effect if not intent, make testing and debugging somebody else's problem.  It's fair to strike a balance between contributing upstream and shipping a finished product, but we must consider that balance.

If, instead of lengthening our cycle and putting old software in front of users, we "sharpen the saw," we can help both ourselves and upstream at the same time.  Some examples:

**- APT/dpkg literacy.**  What does **dpkg-divert** do?  (I just learned that myself)  How many people know how to **replace a package** with a development version **without breaking things** *and *so **that the original package** can be easily and **cleanly reinstalled**?  And how many know that stuff well enough to **explain it to inexperienced users**?  How can an inexperienced, yet motivated user gain this knowledge?  I feel that the APT/dpkg suite is the single most under-appreciated part of Debian/Ubuntu, even though it's so powerful.  **Every time I see a "sudo make install," I cry **a little.  I'm nowhere near fully APT (if you pardon the pun), but I hope to improve.  Let's at least point curious, motivated people in the right direction.

The problem is that we focus on package management for end-users, figure that maintainers and developers will figure it out and ignore the middle ground of power-users, testers, troubleshooters, and hobby developers.

**- Apport for troubleshooters.**   Apport is great at collecting data.  But, it then bundles it off and cleanly ships it away to Launchpad.  You don't have to touch it.  Fine.  But what if you *want* to touch it?  Apport should also support the use-cases of investigating your own system and troubleshooting through IRC/Ubuntu Forums, not just reporting bugs to Launchpad. 

** - We can learn a lot from ISO testing.**  You set up a controlled environment and do a controlled task relying on your experience and intuition to interpret how to interpret the system's behavior.  Will this sort of thing catch all bugs?  No.  Is it a very powerful tool for catching problems?  Yes.  Can it be expanded to software that isn't on the Live CD or is distributed through other release channels: SRU, backports, distribution upgrade?  Certainly.  We just need tools to easily create repeatable test environments, and community wisdom of how to use them.

** - New testers.**  We need to be able to bring new testers easily up to speed.  Some things only come with experience.  For everything else, there should be a curriculum of best practices.

I'm sure experienced testers can think of many more.  That just scratches the surface.  But let's not worry about changing the development schedule as much as we worry about using the best technology and technique.

### Post by kansasnoob on 2009-11-10
Ubuntu has to continue with 6month cycles because...I like it the way it is!

Quite honestly I wish there'd be more promotion of the benefits of multi-booting! It truly is the safest way to upgrade.

I started with Gutsy after about 2 months of distro-hopping and it was incredibly stable for me. Then I jumped on Hardy and for the first few months pulse audio just drove me buggy, so I decided to reinstall Gutsy.

After sleeping on the decision I decided why not multi-boot? I was already dual-booting with Win XP so I gave it a whirl.

I've been multi-booting ever since then, the most I've ever had was Win & 8 different Linux distros. Currently just Win, Jaunty (which is incredibly stable for me), Mint Gloria, and a Karmic that I just converted to Lucid (testing).

Later today I'm going to add a fresh install of Karmic to the mix and as soon as the new Mint RC rolls out it'll go on there too! It's incredibly easy to transfer data from one distro to another so, just for instance, if and when Karmic becomes stable enough I can transfer the data from Jaunty and use it's partitions for another distro.

This way I always have at least one stable OS to use for getting things done and I can play with the others at my leisure. It's also incredibly easy to mount and chroot a borked OS from a stable OS and try to fix it from cli.

It turns frustration into fun & learning!

Now, I know most people won't care to play around to that extent, but just imagine you have a plain-Jane default install of Hardy on a 80GB drive; one primary partition about 78GB and a small extended partition that contains a small SWAP.

Knowing that it's always best to have a primary first on the drive just shrink the Hardy partition to the left and create a new primary partition between the existing primary and the extended partition. Note: moving or resizing swap will throw you into UUID hell, but it can be fixed.

If the Hardy partition is less than half used you can make the new partition equal in size to the Hardy partition and very easily copy everything from Hardy to the new install just by going to Home, click on the Up arrow, and then copy-n-paste your entire <user name> folder(s) to the new install!

Of course if the old OS is using more than half the space you'd need to transfer things incrementally and then shrink the old partition and expand the new partition until you get-r-done. And there are a few things to consider like compatibility of browser profiles, all browsers and apps should be closed while transferring, and of course you should always check, double check, and triple check to be sure everything is transferred successfully before considering deleting the old primary partition and expanding the new partition to the left.

But I think you get the general idea. And, of course, anytime you're partitioning or installing you should always have a complete backup of everything! Only fools don't create backups! Power outages happen at the worst times!

### Post by lykwydchykyn on 2009-11-24
My perspective on this is that I came from a distro with (roughly) a 12 month release cycle (SimplyMEPIS).  It was a pretty stable distro considering it was made by one guy (though it's based on Debian stable, and was based on Ubuntu LTS for two releases).  

But I wonder if everyone calling for a longer release cycle has ever used something with a longer release cycle.  What we'd see in the MEPIS community was that, as the year wore on, more and more popular upstream projects (firefox, KDE, openoffice) would come out with a new release, and the forums are flooded with "how to I install the new <application>?".  

The community would back port some stuff, or publish extensive how-tos for those who want to upgrade, and of course all kinds of problems result from the upgrades, etc.

Also as the year wears on, you get a lot of "My <hardware> isn't recognized" threads, as people try to install on newer hardware where drivers aren't in the kernel or xorg.

By the 11th month, you get all kinds of people writing dramatic posts about how "MEPIS is stagnating!" and comparing its versions to what's in Ubuntu, etc.

New version comes out, it all repeats.

My point is, it's not perfect.  More time is not a panacea for all development problems.  

Part of the problem is that Linux is not yet mainstream enough.  FOSS developers are playing catch-up to the industry to keep hardware supported.  When the day comes that Linux distros maintain a decent market share, when hardware makers MUST support popular Linux distros to be viable -- then something like a two or three year release cycle becomes more feasible.  I mean, imagine if a piece of hardware has got to work with Ubuntu LTS to work for a big chunk of users.  Vendors would find a way to get support in AND make it painless for the user.
