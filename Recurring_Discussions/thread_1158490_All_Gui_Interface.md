### Post by samjh on 2009-05-13
The reason for Linux's complex file system structure is because of its Unix heritage.  It is a multi-user operating system, whereas most Windows are not (the exceptions are Windows Server editions), so it's necessary to distinguish between global files and programs, local files and programs, and per-user files and programs.  The current structure works quite well if you have a basic idea about the structure, with few variations between distros.

Linux is ultimately a server operating system.  Ubuntu and other user-centric distros have placed a pretty-looking blanket over the complicated innards, but at its core, there is no way to make Linux's internals as simplistic as those of Windows consumer operating systems.  Mac OS from X onward, suffer the same problems as Linux, except Mac developers have covered it with even a thicker blanket than Ubuntu/Fedora/et al.

As for moving away from CLI to GUI, I support the idea.  But a great number of Linux installations do not use any form of GUI.  Back-office installations and servers are accessed from remote stations and do not have graphical environments installed.  What then?  Admins need to use CLI commands and scripts.  The great thing about CLI is that tasks are easy to automate using scripts, which cannot be done when you have GUI-only utilities.  Having said that, I think that the CLI utilities should have good GUI front-ends.

### Post by samjh on 2009-05-13
> **brdystyls said:**
> Its not the file system itself, its the hard to navigate naming convention.Yes, they could be improved.

> As for the windows not being multi user this isn't true. I have hard multi users in almost every windows version.You heard wrong, or you don't understand what a multi-user operating system is.

Most Windows versions are not multi-user capable.  They have separate usernames, passwords, and private folders, but this doesn't make them multi-user operating systems.  Windows XP and Vista both need third-party software or a dubious "hack" in order to be multi-user capable.  Neither did Windows 98 and prior versions.  Windows 2000 and Windows Server editions are multi-user capable.

A multi-user operating system allows multiple users -- each with their own file system space and access privileges - to log into and operate a single machine simultaneously without interfering with each other.  This is not the same as remote desktop access (where the remote user has the same access as the local user).  Unix, Linux and other Unix-like operating systems have had multi-user capability built-in for decades.  For Windows, it is a relatively new concept.

### Post by issih on 2009-05-14
There is already far far less NEED to use the command line than you think.

There are gui tools that already exist to manage all sorts of things that are commonly manipulated via the cli. If you poke around the repositories you can nearly always find a little gui tool to manage things be it samba, or mounting drives or whatever else.

The problem here is that the cli is the most efficient way to do things once you know the commands, and it is certainly the most efficient way to communicate a solution over an internet forum. Consequently given that nobody here is getting paid, it is incredibly hard to imagine being able to coerce every user who may have useful advice into telling people to go get gui tool xyz to manipulate this that and the other, especially when the chances are that they, as someone who is an advanced user of that aspect of the OS, would choose to do it via cli and have no actual knowledge of the gui solution.

It is nearly always easier as someone who knows the solution to just do it yourself, than to teach someone else how to do it. This is why tech support people are paid by big software companies. Because they need to be willing to put up with the pain of explaining things 8 million times. The cli is basically doing it yourself, and if the user wants to know about what just happened they can ask. This is not an ideal solution at all, but an inevitable outcome of a user supported operating system.

If there was a workable solution I'd be all for banning cli explanations from the beginners forum, or at the very least requiring all cli commands to be explained clearly. Sadly there is no workable solution to achieve this kind of goal.

In essence unless a new user is willing to invest the time to learn what is going on, they have to accept that they will be mystified as to the innards somewhat. If they want to learn what is going on they can, there are lots of guides out there, and once you have the basics of understanding, you can often find the gui tool to solve your problem yourself, but there is no way to remove that initial hurdle of understanding.

Essentially the cli is far more powerful than a windowed interface for certain things (e.g. I am right now manipulating thousands of files and moving them about based on a set of criteria, all with a couple of commands) and therefore is not going anywhere. Unfortunately, this power makes it the ideal tool for troubleshooting and fixing things if you know what you are doing, and consequently those in the know use it. Asking them not to because it "seems old fashioned" cuts no ice, and as long as those self same advanced users are the people who come in here and offer support, the cli will rule.

There is no way out of this conundrum unless someone is willing to set up a linux tech support line that operates at consumer friendly rates...which I don't see happening.
