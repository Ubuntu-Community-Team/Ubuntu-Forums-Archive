OSX is phasing in a more granular UAC approach, too, and if they took security seriously on the whole, their strategy would probably be the best, from what I've seen.

### Post by Copper Bezel on 2011-11-15
No, you misunderstand. 

I'm not saying that someone is going to send you an mp3 and it's going to "do a virus," as the silly people say, because you're using Windows. I'm specifically talking about document files in one particular application that allows them to store user macros.

MS Office's macros are a (tiny) security vulnerability because they can embed user actions in a document. Since they can be triggered by whatever the user does after opening the file, a malicious macro in a downloaded document file has just enough freedom of motion to, say, steal your clipboard or something on the chance you might have something sensitive there. That's not even real malware - it's just a quirky possible security exploit. That's the "hole" part, and it's not a real problem. There are presumably countless similar vulnerabilities in programs that exist on Linux. It's an end-user app accessing user data. There's no "system security" or execution or privileges involved.

The "plug" part is the bit I find a little baffling. If you download a file and open it under Windows 7 in Word 2007 (I've not used 2010) it launches in "protected mode." The software is watching where the file came from, and implements a locked-down mode in response that doesn't permit editing or printing the file (and which the user can simply click through.) But it's curious that a *document* ends up needing to be confirmed as *trusted*.

I was just using it as a specific example to characterize the weirdness of Microsoft's security philosophy. You could also look at the old way of handling registry keys (as executable) or its lack of a global package installer or anything else. There's a general sense that specific exploits are blocked, but there's no general overriding best-practice system built into the OS (or in my initial example, in an unrelated MS product.) It's different from the architectural approach to security that Linux systems use, where the assumption is that the system is only going to give a particular class of thing the kinds of permissions that pertain to it.

And that's all aside the point of actual, end-product security, which I've already said is probably better handled in Windows, due to the granularity of the UAC. Linux's security approach is baked-in, but it's also designed around servers, having more to do with protecting the system against processes run by a large number of users rather than protecting user data from applications run *by that user*.

The ideal, of course, would be to simply enforce strict policies on end-user applications in a way that's really relevant to the desktop, which Apple is beginning to do, so that an individual application can't edit anything but its own config folder or narrowly defined shared resources without asking permission (by invoking a global open / save dialogue, for instance.)

The problem with these discussions is that they're often framed as is operating systems were wind-up toys instead of actively used systems.

Once you get users on an operating system, there are too many variables to have any kind of meaningful point of comparison as to what is more or less secure.

### Post by aysiu on 2011-11-15
Just as a practical example of why "security" built into the OS really means nothing, let's just look at passwords.

If you're an organization or company, even a small one, you have to deal with passwords. If you give users a complexity requirement, chances are they'll create passwords that are complex and then forget them and then have to constantly reset them. This will annoy both the users (who cannot log in to get their work done) and the IT department (who is constantly resetting user passwords). Or they'll just write down their passwords and maybe lose the slip of paper somewhere. Same deal with forcing users to change their passwords every month.

But then if you just let them create whatever password they want, they'll create stupid passwords like *password* or their username or the name of their spouse, etc.

Usually when it comes to passwords, you're trying to figure out what's a system that sets up reasonably difficult passwords while not making them so difficult as to annoy users or to hamper productivity. It's not just about "secure" or "insecure."

### Post by qyot27 on 2011-11-16
[quote=cbanakis]Simply, are there less security holes in Windows?[/quote]
The problem with this question is the lack of freely-available data to compare.  I'd feel safe assuming the review and fix process on *nix gives a fairly accurate assessment of the number of security holes (and remember, the parts of a nominal *nix system come from lots of different original sources, not just one).  Good luck trying to get that kind of information from Microsoft during the process.  You might be able to infer it some from the content of Windows Updates, but the actual holes won't be publicized in real-time (unless it's of the media-hype-machine-Y2K-level-hysteria sort), only reported after-the-fact with the Update to fix them.

There's also a qualitative element to this.  While it's true that a security hole is a security hole and therefore needs to be fixed posthaste, there are clearly cases where one hole is more severe than another.  To be an accurate comparison, you need to compare like with like - don't compare a couple dozen minor holes in *nix to one or two major ones in Windows.  That would be intellectually dishonest.
