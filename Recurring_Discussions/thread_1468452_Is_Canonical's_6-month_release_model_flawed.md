### Post by darrenn on 2010-05-01
This was asked in last weeks Distrowatch. 

[http://distrowatch.com/weekly.php?issue=20100426](http://distrowatch.com/weekly.php?issue=20100426)


> Dodging-the-rapid-fire asks: Is six months too short of a release cycle?

DistroWatch answers: This question generally gets aired (often rhetorically) around the time when Ubuntu and Fedora are due for a new release. If you're wondering whether six months is too short a time to put together a solid product, the answer is: no, it's not.

The quality of a release has less to do with the amount of time spent putting it together than such things as:

   1. The feature and bug-fix list
   2. Development model
   3. Quality assurance testing 

I've worked with software which received regular monthly updates and it worked very well. We hardly ever encountered a problem from performing an upgrade. For that matter, I've worked with software which received monthly updates which never rose above beta quality. The timing is much less important than the focus of the production team (the leaders, developers and testers). The OpenBSD project is, in my opinion, an excellent example of a group which follows a six month release cycle without sacrificing their security or stability. The OpenBSD developers have an interest in those things more so than being on the cutting edge.

Another complaint some people have is in regards to the lifespan of the product. Rapid releases usually make for short-term support. But if you're looking at support time, I think the question needs to shift from "Is six months too short of a release cycle?" to "Is six months too short of a release cycle, for me?" The Linux sphere has a wide range of products from rolling releases to six month updates to seven year support terms. If you're finding yourself struggling against the upgrade treadmill, there are plenty of alternatives. I'm sure one distribution can be found which balances the desire of new features against the desire for long-term support.

### Post by Mr. Picklesworth on 2010-05-05
> **snowpine said:**
> Two thoughts on this:

1) Rolling release is *by definition* unstable; I predict Ubuntu LTS will never be rolling release. 

2) Ubuntu is based on Debian Unstable (for the most part), which is itself rolling release... it is impossible to make a rolling release distro based on another rolling release distro; the logistics boggle the mind. ;)

I believe snowpine's thought is to keep the core stuff stable, as it is now but to more aggressively track upgrades for individual desktop applications, perhaps with a separate repository.

Of course, people can do this in a lot of cases with unofficial (and occasionally official) repositories dedicated to those applications, GetDeb and, occasionally, Backports.

One unfortunate issue in all of this is that Debian and Ubuntu apply a lot of patches to packages they ship, so fairly often the upstream stuff won't feel right, but let's pretend we've solved that issue. (After all, there is always lots of healthy discussion about reducing divergence from upstream and everyone wants to make it happen).

Keep in mind that you probably only care about having rolling upgrades to specific applications. For example, I doubt many people would be interested in downloading the absolute latest version of Evince (or Adobe Reader), but wanting the latest Blender, Wine or Firefox makes a lot of sense.

So, the existing Backports project can't / shouldn't backport everything and going that route could case some serious issues.
I think having a way to subscribe to official "upgrade channels" (or backports, or whatever) for specific packages would be a very neat thing to work towards.

But indeed, I think the underlying stuff should always be stable with a six month rhythm for major updates. There's an awesome reason why that is: app developers can actually aim for a tangible target setup that works for a lot of users. If the most popular distro went with a rolling release for everything, chaos would surely follow.
