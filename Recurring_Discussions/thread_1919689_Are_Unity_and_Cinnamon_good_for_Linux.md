### Post by rg4w on 2012-02-03
> **keithpeter said:**
> I remember why I didn't like it last time now. The redistribution rigmarole (installing a runtime &c).
Yeah, as a consultant I'd strongly prefer if they went FOSS, but they have a considerable investment in it and thus far none of us have been able to come up with a compelling business case to support such a move.  Would be nice, though.

> @rg4w: are there any restrictions on re-distribution of the runtime?None aside from "Portions (c) 2012 RunRev Ltd." in the About box, similar to most other tools of this nature.  No royalties, no complex licensing issues.

> Could I GPL an app made with this platform?A number of us who use LiveCode have discussed this at some length, and the consensus seems to be that although the engine required to run LiveCode apps is proprietary there's nothing preventing one from releasing the code they write for it under any license they choose, FOSS or otherwise.

In practical terms, it's not much different from making a FOSS app for Mac OS or Windows, where the app requires the proprietary system to run but the app itself is still FOSS. 

In fact, a few of us share libraries and components under FOSS licenses, usually MIT so folks can use them in proprietary works if they choose, but sometimes GPL or L-GPL.

> Would it be possible to distribute applications compiled and packaged with this system via Ubuntu Software Centre?Yes.  In fact, I talked with Ted Gould about this at last year's SCaLE and it seems that because LiveCode apps are generally self-contained (zero dependencies outside of things included in any GUI Linux system) it should be relatively simple to package those up for the USC.

I'd like to make a drag-and-drop tool for making LiveCode apps into .deb packages, but I've been up to my armpits in client work so I haven't yet had the time.
