### Post by Copper Bezel on 2011-04-23
Understand that I can't completely defend Unity, because I'm using Compiz and AWN with some XFCE parts right now in a Unity-but-faster-and-prettier arrangement; I haven't really used either shell, so I'm basing this on principles and videos, but it's also a fun game. = ) As you explain it, I'm getting a sense of how Shell is supposed to work; nonetheless, I think I can make a fairly damning argument against it and in favor of Unity, regardless. = )

[quote=JDShu]What is the set of rules? Whats the vision?[/quote]

* All focus on the active window or present task. 

[INDENT]The global menu and highly drag-friendly Launcher mean that the shell works as an extension of whatever application is focused. The global menu follows Fitts' Law by making the screen edge a part of the click zone for menu items. 

Dash fills the screen on netbooks and takes up a substantial amount of screen space on ordinary desktop systems, pulls up the file or application by search or click, including quick-access Favorites, and gets out of the way (which is of course no different from Shell, except in that these tasks are separated in Unity.) Click regions are large and all elements are highly visible.

The auto-hiding feature of the Global Menu is a mistake, but I'm really, honestly confident that the developers will come to understand that with time. It's completely unnecessary and unattractive, and it contradicts basically everything I'm saying here, primarily by requiring its weird flick-readjust motion.[/INDENT]

* Efficient use of screen space, esp. vertically. 

[INDENT]Maximized window title bars integrate into the panel so that there is no screen space lost to a panel of any kind (where Shell retains a nearly identical panel but does not integrate the menu or titlebar into it.)  Overlay scrollbars, stripped edge decorations, and the new resizing handle shave a precious fifteen or twenty pixels off the horizontal footprint of windows. [/INDENT]

* Visibility and efficient switching of tasks. 

[INDENT]The new systray discourages sytray items for apps that don't need them, reducing clutter. The Launcher auto-hides by default, but even if it's set as such, it's quicker to access than Shell's shell view and offers instant visual indication of all running applications with less visual interruption. Workspace switching and application launching is a click away from there. The extra widgets offer quick, visible access to core desktop productivity features - roughly the same ones as in Shell, but as separate widgets, following the focus rule.

This is where I feel Unity's choice to stay a bit traditional wins out. Shell's integrated Shell view brings up everything to do with window and workspace management at once; the view is cluttered, and the reality of rendering speed sets in. Unity's more conventional widget-based approach, especially when the Launcher isn't set to auto-hide, strikes an even balance between keeping all of these functions immediately accessible and keeping all focus on the present action, so that the shell is essentially *always* present.

Much as the systray discourages developers from creating clutter, the Launcher also discourages users from creating desktop icon clutter by integrating Trash, Places, and pinned applications in a vertical orientation.[/indent]

* An integrated "window management click zone."

[indent]The integration of the title bar and panel requires the window controls to be at the left for maximized windows and, thus, globally (to retain an intuitive consistency.) However, this positioning means that all task-management features stay as close as possible to the Launcher.

The maximize button is retained for legacy workflow reasons. Minimizing, on the other hand, is retained in the default experience because it performs a different function than workspace switching, backgrounding as opposed to sorting tasks, and it does so with much less effort (a single click as opposed to, in Shell, a hover, a select, and a drag that ranges from the upper left to the center-right of the screen.) Unity's window controls for maximized windows are also, like the global menu, laid out along the screen edge instead of crammed in between the content area and the upper panel.

Combined with the launcher for task switching and launching, all common window management functions can be accessed by clicking a screen edge on the left or near the upper left corner, which is not at all true of Shell.[/INDENT]

* Visual consistency and a fluid desktop experience. 

[INDENT]Compiz is more mature and much better at this kind of fluidity than Mutter. Like Shell and unlike Compiz running on the Classic desktop, Unity keeps the panel visible at all times. Unlike Shell, it uses viewports instead of workspaces and makes fluid transitions between. 

Integration of other Compiz plugins, like Scale and the viewport switcher in the Launcher, reinforce the spatial metaphor, where Shell's environment is more built around list metaphors, particularly in the spread windows and vertical layout of the workspace switcher. Since the metaphor of the desktop is itself spatial, this seems more immediately intuitive.[/INDENT]

* Touch friendliness for the future. 

[INDENT]Reducing the importance of menus and hover, broadening click targets, removing scrollbars, etc., while a reasonably streamlined desktop solution, all anticipates compatibility with a touch workflow. Of course, most of this applies to Shell, as well, but there are small signs of tablet-friendly choices that Shell lacks - horizontal instead of vertical orientation for category selection in the Dash screen makes for larger fatfinger zones, as do the screenedge alignments, and the Launcher's autohide behavior can be disabled. Ubuntu Tablet Edition could already exist as a couple of changes to Compiz and system font settings.[/INDENT]
