### Post by eems01 on 2007-04-29
I have had good success since dapper following (and now modifying) items from this original thread from scubajeff.  I apologize for some of the direct quotes but he conveyed it so well.

[http://ubuntuforums.org/showthread.php?t=78904&highlight=alps]("http://ubuntuforums.org/showthread.php?t=78904&highlight=alps")

This is my own howto that I rework with each release.  Don't ask me for specifics about why it works, it just has always worked for me on my Compaq Presario R3000?

1.  Find out what device handle the hardware:

	cat /proc/bus/input/devices

on my system, it shows up something like the following

I: Bus=0011 Vendor=0002 Product=0008 Version=7321
N: Name="AlpsPS/2 ALPS GlidePoint"
P: Phys=isa0060/serio4/input0
S: Sysfs=/class/input/input4
H: Handlers=mouse2 ts2 event4 
B: EV=f
B: KEY=420 0 70000 0 0 0 0 0 0 0 0
B: REL=3
B: ABS=1000003

2.  Found out the handlers for Alps Glidepoint device in step 1, on my machine it's event4. Use it in the following command to find out how to identify the device's description in udev.

Force EventX remember Eventid:

The reason why the eventX number changed is because "udev" control all the device naming during machine bootup. The way to get it fixed is to write our own udev rules. This is tricky! The very first thing to do is to identify the unique description of our Alps touchpad, type the following, substitute the "eventX" with your own number:

Code:

	udevinfo -a -p `udevinfo -q path -n /dev/input/event4`


You will got something similar to this:

Code:

Udevinfo starts with the device specified by the devpath and then
walks up the chain of parent devices. It prints for every device
found, all possible attributes in the udev rules key format.
A rule to match, can be composed by the attributes of the device
and the attributes from one single parent device.

  looking at device '/class/input/input4/event4':
    KERNEL=="event4"
    SUBSYSTEM=="input"
    DRIVER==""
    ATTR{dev}=="13:68"

  looking at parent device '/class/input/input4':
    KERNELS=="input4"
    SUBSYSTEMS=="input"
    DRIVERS==""
    ATTRS{modalias}=="input:b0011v0002p0008e7321-e0,1,2,3,k110,111,112,145,14A,r0,1,a0,1,18,mlsfw"
    ATTRS{uniq}==""
    ATTRS{phys}=="isa0060/serio4/input0"
    ATTRS{name}=="AlpsPS/2 ALPS GlidePoint"

  looking at parent device '/devices/platform/i8042/serio4':
    KERNELS=="serio4"
    SUBSYSTEMS=="serio"
    DRIVERS=="psmouse"
    ATTRS{resync_time}=="0"
    ATTRS{resetafter}=="5"
    ATTRS{resolution}=="200"
    ATTRS{rate}=="100"
    ATTRS{protocol}=="AlpsPS/2"
    ATTRS{bind_mode}=="auto"
    ATTRS{modalias}=="serio:ty01pr00id00ex00"
--->ATTRS{description}=="i8042 AUX3 port"<----

  looking at parent device '/devices/platform/i8042':
    KERNELS=="i8042"
    SUBSYSTEMS=="platform"
    DRIVERS=="i8042"
    ATTRS{modalias}=="i8042"

  looking at parent device '/devices/platform':
    KERNELS=="platform"
    SUBSYSTEMS==""
    DRIVERS==""

3.  The interesting part of the output is --->highlighted<---.

Then create a file: "/etc/udev/rules.d/55-alps.rules", with the "ATTRS{description}" part matching the output of udevinfo. File content:
	sudo gedit /etc/udev/rules.d/55-alps.rules
Code:

	BUS=="serio", ATTRS{description}=="i8042 AUX3 port", KERNEL=="event?", SYMLINK+="input/alps"


The number 55 is to make sure it get loaded before the system-wide udev.rules file.
After the above setup, the machine, when bootup, will create a symbolic link /dev/input/alps to the correct eventX of our Alps touchpad. 

4. So we have to change our xorg.conf to reflect this change. 

Code:

          sudo gedit /etc/X11/xorg.conf

The following sections are parts from my xorg.conf file that pertain to the ALPS touchpad _only_.  Everyones file is different and I can post the whole thing if it would help:

### Post by Runamok on 2007-07-29
Hello,

I have a Dell Inspiron 9300 with an Alps Glidepoint touchpad.  I am using Ubuntu Fiesty Fawn 7.04.  By default Fiesty does not recognize this, and instead gave me some default device drivers.  This caused my touchpad to work incorrectly.  It was too sensitive on registering taps as clicks, and the vertical scrolling did not work. 

But after an evening, I successfully solved my Alps glidepoint issues.  :popcorn:

Read on for the story...  I hope this helps someone else!

Like others in this forum, I could NOT get
 ```
cat /proc/bus/input/devices
```
to display ANYTHING about ALPS, and I was stumped.](*,)

However, this older ALPS page mentioned one very important item.
[http://doc.gwos.org/index.php/Alps_touchpad]("http://doc.gwos.org/index.php/Alps_touchpad")

  > First of all, if you have ever touched the file "/etc/modprobe.d/psmouse.modprobe", please remove it. Since it prevent the system from probing the Alps touchpad. On my Sony TR2, before removing this file, the system only found a PS/2 Generic Mouse, after removing it, the system found a PS/2 Mouse and a Alps Glidepoint.

I don't remember "touching" this probe file in any naughty way, but I decided that file had to go.  After deleting the psmouse.modprobe, and restarting the x-server [ctrl+alt+backspace] I was able to get Ubuntu to recognize AlpsPS2 ALPS Glidepoint!  

If deleting that file works for you, then your halfway there.  In the of output of the above code, please pay close attention to which event number is assigned to the ALPS device. You will need that event# later when we use the next page to edit the xorg.conf.

The following page helped me tweak and resolve my xorg.conf to use ALPS.
[SIZE="4"][http://ubuntu.wordpress.com/2005/11/15/fixing-my-alps-touchpad-with-the-synaptics-driver/]("http://ubuntu.wordpress.com/2005/11/15/fixing-my-alps-touchpad-with-the-synaptics-driver/")[/SIZE]

In summary, the above page mentions the changes that you need to make to your Xorg.conf.  However, don't just cut and paste everything.   The author fails to mention some CRUCIAL nuggets of info. 

One is that you must also add ALPS to the "ServerLayout" section of the xorg.  (Thats in the comments) 

Two, the author fails to mention that your specific Event# must be reflected in the alps device section of the xorg.conf!   Mine, was initially Event7, but after a full reboot it changed to Event3.  (Apparently Linux does not permanately assign event#'s to devices, and just doles out them out as it detects devices during bootup.  For instance, plugging in a USB mouse at bootup will shift all the numbers!  This might explain a couple issues I have!) 

After saving all the changes to my xorg.conf using my specific event# and rebooting the x-server, I could tell it worked because the touchpad input was different.  Slow, and still no verical scroll.....

Finally, the last comment on the above page (31. by Merrick) explains that you must consolidate the two sections that reference the input device for the ALPS into one.  After that last alteration, and consilidation everything works great!  Fast, with vertical scroll.

### Post by mrtwister123 on 2007-10-13
Hi All,

I've finally got my Alps touchpad working on my Fujitsu 6120 laptop (albeit on PCLinuxOS 2007). My problem originated from the fact that Linux (Ubuntu Fiesty) refused to detect my Alps touchpad device correctly and was detecting it as a PS/2 Mouse.

I was able to fix this by editing my /boot/grub/menu.lst entry for my installation by tagging i8042.reset i8042.nomux to the end of the kernel entry. Thus my main boot entry in my menu.lst file now looks like this:

title linux
kernel (hd0,0)/vmlinuz BOOT_IMAGE=linux root=/dev/sda5  acpi=on splash=silent vga=788 **i8042.reset i8042.nomux**
initrd (hd0,0)/initrd.img

This is an entry for a PCLinuxOS install, but the principle should be the same for Ubuntu (i.e. add the  i8042.reset i8042.nomux parameters to the kernel boot options).

This works on the Intel i8042 keyboard/mouse driver.

After getting the OS to recognise the Alps device, it's a simple case of configuring your xorg.conf file with the correct entries for synaptics. Further details for doing so can be found at [http://web.telia.com/~u89404340/touchpad/](http://web.telia.com/~u89404340/touchpad/)

Regards,
Mark
