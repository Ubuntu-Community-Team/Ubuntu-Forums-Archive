### Post by dioktos on 2007-11-01
I have two SATA devices that are changing order and names (sda->sdb, sdb->sda) AFTER boot.

Here is my setup: I have one SATA drive (Seagate Barracuda ES, ST3250820NS) connected to my motherboard (Asus P5B-VM) through the on-board SATA controller (JMicron 20360/20363).  This drive has two partitions: one is swap (loop-aes encrypted) and the other is ext3. These usually show up as /dev/sda1 and /dev/sda2, respectively.

I also have three drives (Seagate Barracuda ES, ST3320620AS) connected to a raid controller (3ware 9650SE-4LPML) and combined into a single raid 5 array. This array is not partitioned and contains a loop-aes encrypted ext3 file system. This usually shows up as /dev/sdb.

If I start in recovery mode, I get the following result:
```
user@medon:/$ ls /dev/sd*
/dev/sda  /dev/sda1  /dev/sda2  /dev/sdb
user@medon:/$ mount
/dev/sda2 on / type ext3 (rw,errors=remount-ro)
...
```
Which is correct.

If I let gdm start, and log in normally, I get the following result:
```
user@medon:/$ ls /dev/sd*
/dev/sda  /dev/sdb  /dev/sdb1  /dev/sdb2
user@medon:/$ mount
/dev/sda2 on / type ext3 (rw,errors=remount-ro)
...
```
Which is not only incorrect, but highly perplexing. This means that /dev/sda turned into /dev/sdb sometime AFTER boot and mounting of the root filesystem.

My /etc/fstab looks like this:
```
# /etc/fstab: static file system information.
#
# <file system> <mount point>   <type>  <options>       <dump>  <pass>
proc            /proc           proc    defaults        0       0
/dev/sda2       /               ext3    defaults,errors=remount-ro 0       1
/dev/sda1       none            swap    sw,loop=/dev/loop7,encryption=AES128 0 0
/dev/sr0        /media/cdrom0   udf,iso9660 user,noauto 0       0
/dev/fd0        /media/floppy0  auto    rw,user,noauto  0       0
/dev/sdb        /secure         ext3    defaults,noauto,user,loop=/dev/loop0,encryption=AES128,...

```

Any ideas on where to start looking?

### Post by dioktos on 2007-11-03
Unfortunately, while this works for root, it doesn't work for encrypted swap (since the swap UUID changes everytime swap is mounted, as the encryption keys are random) or the encrypted array (since the FS, and thus the UUID, doesn't appear until after mount).

Thanks for the hint, though: I've solved the problem for now by doing something similar. Since the swap header is not encrypted, I labeled it and mount it using "LABEL=", and I refer to the array as "/dev/disk/by-id/scsi-...." since udev makes reliable entries for that.

So, thanks again. I still wish I knew why the sda/sdb order was indeterminate...
