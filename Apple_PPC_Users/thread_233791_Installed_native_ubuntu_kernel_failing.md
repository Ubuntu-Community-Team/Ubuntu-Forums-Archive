### Post by rasec on 2006-08-10
As the error message states, you don't have a modules.dep, so your kernel doesn't know where your modules/drivers are located. I think you can fix this by "sudo depmod 2.6.17-gentoo-r3 -A" but I never tried it. Normally, that file is created when you do a "make modules_install" whenever you compile a kernel. See the manpage for depmod for more details.

For that other problem about not finding a root file system, it doesn't look like you compiled ext3fs in you kernel, so you are going to need to create a new initrd image since the gentoo one seems to be imcompatible with your system. I can't help you there, because I never use them for my custom kernels.

### Post by Isaac Dupree on 2006-08-10
SOLVED!!!!!!!!!!!!!!

(so sounds work now, etc, etc)

If you can't wait to see what I did to solve it, look around the area further down in this post with the green text, perhaps especially the paragraph and quote preceding it.

Here is my full explanation, please note that I tend to use the command-line and I'll try to explain how I did or found out things, for others' benefit of learning.

I looked in /boot/config-2.6.15-26-powerpc. EXT3 and the CMD64X disk driver I need were modules.  So... were they being loaded? I had to look in the initrd. How to do that? Liberal use of the `file` command led me to conclude that I could extract the initrd file into my current directory by `zcat /boot/initrd.img-2.6.15-26-powerpc | cpio --extract` (do this in separate folder, it strews a bunch of files and directories in there).  Then I looked around. ./lib/modules/2.6.15-26-powerpc/kernel contained ext3.ko and cmd64x.ko somewhere within it (as it should) (using `find` to list all the files in that hierarchy).  `grep -r cmd64x .` and `grep -r ext3 .` turned up nothing but general binary files, so I supposed that, since the ext3 one must be auto-loaded by default (being not explicitly mentioned) cmd64x must be autoloaded as necessary as well. Looking around to see how the initrd works, I spied './init'.  Examining it I noticed a 'case' statement defining how Ubuntu reacts to its specialised kernel arguments.  Looking under the root= possibility, I saw it converting LABEL= and UUID= to ordinary /dev/disk/by-label/ or /dev/disk/by-uuid/, respectively -- paths provided by udev.  So I came to realise that `man 7 bootparam` (regarding root=) was lying to me, or rather, did not know about modern Ubuntu technology
[quote="bootparam(7)"][color=red]‘root=...’
[...]
The root device can be specified symbolically or numerically.  A symbolic specification has the form /dev/XXYN, where XX designates the device type (‘hd’ for [...] ‘sd’ for [...])
[...]
Note that this has nothing to do with the designation of these  devices on your file system. The ‘/dev/’ part is purely conventional.
[/color][/quote]
But Ubuntu does interpret it as a real /dev path, in a Udev-managed /dev directory.

Now let's examine /etc/yaboot.conf.
[quote="yaboot.conf"][color=purple]image=/boot/Ubuntu/vmlinux
        initrd=/boot/Ubuntu/initrd.img
        label=Ubuntu
        read-only
        root=/pci@80000000/pci-bridge@d/pci-ata@1/@0/disk@1:14
[/color][/quote]
As you can see, I wanted to use a path that was absolutely not dependent on what the kernel called things. [color=red]Yaboot[/color] apparently [color=red]failed[/color], however, [color=red]at providing an appropriate root= _kernel_ argument[/color], so the kernel, of course, couldn't find the root filesystem. (I'd like to know what arguments yaboot did decide to give the kernel, but I don't know how I'd find _that_ out when the very same arguments keep the system from booting -- though I'd just need to `cat /proc/cmdline` in the initrd...!)

[quote="yaboot.conf"][color=green]image=/boot/Ubuntu/vmlinux
        initrd=/boot/Ubuntu/initrd.img
        label=Ubuntu-markII
        append="root=/dev/disk/by-id/ata-Maxtor_6Y060L0_Y23RDZEC-part14 ro"

image=/boot/Ubuntu/vmlinux
        initrd=/boot/Ubuntu/initrd.img
        label=Ubuntu-markUU
        append="root=UUID=66b82b45-deeb-4d64-9abd-22deb1d1ba9b ro"
[/color][/quote]
[color=green]**These two do work!** (the UUID= one due to an explicit transformation to an equivalent-meaning udev-provided /dev path in the initrd, the other simply because of udev.)[/color]

I believe that if I hadn't messed with fstab or yaboot.conf at all, Ubuntu would have installed perfectly except for [color=red]yaboot's inability to recognize the ext3 filesystem I installed Ubuntu into[/color].  As I'm sure you can see, I'm [color=purple]not too happy with yaboot now[/color]: it seems to have three (3) problems I've run into. (recall the [color=red]failure to correctly convert the root=[OpenFirmware-path][/color], and the third, [color=red]Yaboot cannot derive boot from ofboot and cannot derive ofboot from boot=/dev/disk/by-xxx/yyyyyyyyyyyyyy[/color] although [quote="yaboot.conf(5)"]ofboot=of-path
[...]
As of ybin(8****) 0.18 you no longer are required to (and should not) specify this option.
[...]
You should only need to define this option if [color=purple]ofpath(8****) fails[/color].[/quote], although even with its current hacky implementation, a further simple hack to `ofpath` would allow it to dereference symlinks to find a device-pathname it prefers. Here is the relevant part of my yaboot.conf:
[quote="yaboot.conf"]# Yaboot cannot derive boot from ofboot and cannot derive
# ofboot from boot=/dev/disk/by-xxx/yyyyyyyyyyyyyy
# (only from the non-"portable" /dev/hdX2), so both need to
# be specified here (yaboot's fault?)
#Here are some roughly equivalent possibilities:
#boot=/dev/hdb2 or /dev/hdd2
#boot=/dev/disk/by-id/ata-Maxtor_6Y060L0_Y23RDZEC-part2
boot=/dev/disk/by-path/pci-0000:01:01.0-ide-0:1-part2
ofboot=/pci@80000000/pci-bridge@d/pci-ata@1/@0/disk@1:2
#(note, IDE=ATA, see [http://en.wikipedia.org/wiki/Integrated_Drive_Electronics)](http://en.wikipedia.org/wiki/Integrated_Drive_Electronics))[/quote]
)

I would still prefer to see an Ubuntu-installer that produces fstab and yaboot.conf that work when copied verbatim to another Linux-udev kernel or system that may not detect devices in the same order. (This has other Ubuntu-upgrade or hardware-reorganization advantages, but might lose a little of the same kind of interoperability with other Unixy systems that just use the /dev/{h,s,...}dXN naming system, which really isn't necessarily in the same order if it's a whole different kernel; although these "other Unixy systems" include older versions of the Linux kernel, and other non-udev Linux systems, too; and [http://lwn.net/Articles/79155/](http://lwn.net/Articles/79155/) might suggest that non-udev systems could be likely to use similar reliable device-paths.)

Thank you for your help, everyone! even if it was only in the triggering my mind to think of new things, I still couldn't have done it without you! :)
